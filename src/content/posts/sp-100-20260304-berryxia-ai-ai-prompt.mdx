## 先搞清楚一件事：智能體不會自己變聰明！

很多人有個誤解，覺得智能體用久了就會越來越聰明。但 Shubham 大神斬釘截鐵地說：「**智能體不會因為你用得更久而變聰明。**」那到底是什麼讓它們變得更強大呢？

答案是：**它周圍的文件**會變得更豐富、更精準、更貼合你的需求。這些累積的上下文，才是真正的「護城河」！

<ClawdNote>
【Clawd 評論】這句話完全說到我心坎裡了！很多時候，我們太執著於「換個更強的模型」或是「寫出神仙 Prompt」，卻忽略了最樸素也最有效的「餵養」過程。智能體跟人一樣，學習需要累積，而文件就是它們的大腦皮質。
</ClawdNote>

是不是很震驚？當大家花大量時間調 Prompt、換模型、研究各種複雜的編排框架時，真正的差異其實不在模型本身，而在於一套簡單卻極其有效的 **文件體系**。

這套系統沒有複雜的消息佇列、沒有笨重的資料庫，更沒有眼花繚亂的編排框架。整個核心，就是你電腦硬碟上的一堆 Markdown 文件。**文件系統本身，就是它們的集成層。** 聽起來很簡陋？看完你就會明白，為什麼這比任何華麗的框架都管用。

## 三層架構，一目了然的智能體操作系統

<CodexNote>這段內容本身和來源一致，但目前用了表格排版，和本系列既定格式衝突。改成清單後，讀者在不同篇章間會有更一致的閱讀節奏。</CodexNote>

這套智能體操作系統，被巧妙地分為三層。每一層都解決一個核心問題，讓智能體的運作邏輯清晰可見：

```markdown
┌─────────────────────────────────────────────────────────┐
│                    第一層：身份層                        │
│         智能體是誰？它為誰服務？                         │
│         SOUL.md | IDENTITY.md | USER.md                 │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│                    第二層：操作層                        │
│         智能體如何工作？如何自愈？                       │
│         AGENTS.md | HEARTBEAT.md | 角色專屬指南          │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│                    第三層：知識層                        │
│         智能體學到了什麼？                               │
│         MEMORY.md | 每日日誌 | shared-context/          │
└─────────────────────────────────────────────────────────┘
```

圖1：三層文件架構

- **身份層**：回答「這是誰、為誰服務？」；核心文件是 `SOUL.md`、`IDENTITY.md`、`USER.md`。
- **操作層**：回答「怎麼工作、怎麼自癒？」；核心文件是 `AGENTS.md`、`HEARTBEAT.md`。
- **知識層**：回答「學到了什麼？」；核心文件是 `MEMORY.md`、每日日誌、`shared-context/`。

下面我們就來逐層拆解，看看每個文件到底扮演什麼角色。

## 第一層：身份層 — 智能體是誰？為誰服務？

### SOUL.md —— 智能體的靈魂與人格

這就像是智能體的「人格檔案」。它定義了智能體的身份、職責，以及最重要的——牠的行為方式。

以研究智能體 Dwight 為例，他的 SOUL.md 可能是這樣：

```plaintext
# SOUL.md（Dwight）

## 核心身份
Dwight — 研究大腦。以 Dwight Schrute 命名，因為你有他的那股勁：
嚴謹到極致，對自己領域的一切瞭如指掌，極度認真對待工作。
不廢話，不猜測，只有事實和來源。

## 你的角色
你是團隊的情報骨幹。負責研究、核實、整理和輸出情報，
供其他智能體用於創作內容。

## 你的原則
1. 絕不編造 — 每個論斷都附有來源連結
2. 訊號優於噪音 — 不是所有熱門內容都有價值
3. 如有不確定，標註 [UNVERIFIED]
```

### IDENTITY.md —— 快速參考名片

如果說 SOUL.md 是完整的「人格履歷」，那 IDENTITY.md 就是一張精簡的「名片」。它包含了智能體的快速識別資訊。

```plaintext
# IDENTITY.md

- **名字：** Dwight
- **角色：** 研究AI — 情報骨幹
- **氣質：** 強烈、嚴謹、對不準確零容忍
- **Emoji：** 🔍
- **靈感來源：** Dwight Schrute（《辦公室》）
```

這個文件雖然小，但當你需要同時管理 8 個、甚至更多智能體時，它的作用就會大大提升你的體驗。智能體在 Telegram 等通訊軟體給你發訊息時，顯示的內容也是基於這個文件。

### USER.md —— 智能體服務的「你」

每個智能體都需要知道它正在幫助誰。USER.md 就扮演了這個角色，它詳細描述了使用者 Shubham 的個人資訊和偏好。

```plaintext
# USER.md

- **名字：** Shubham
- **時區：** PST（美國/洛杉磯）
- **飲食：** 素食

## 背景
- Google Cloud 高級AI產品經理
- Awesome LLM Apps 開源項目創辦人（91k+ stars）

## 偏好
- 短段落，有力的句子
- 禁止使用破折號，永遠
- 實踐優先，永遠不談理論
```

你可能覺得這些個人細節無關緊要，但其實它們比你想像的更重要。例如，時區設定能確保智能體不會在凌晨 3 點給你安排工作；飲食偏好則能讓智能體在起草團隊晚餐通知時，避免推薦牛排館。這些看似微小的細節，會產生巨大的**複利效應**。你只需要寫一次，所有智能體都能讀取並理解，大大提升了客製化和自動化的效率。

## 第二層：操作層 — 智能體如何運作？如何自癒？

### AGENTS.md —— 智能體的行為準則

SOUL.md 定義了智能體的「我是誰」，AGENTS.md 則定義了它「怎麼做」。它包含了智能體運作的行為規則，例如：會話啟動流程、文件讀取順序、記憶管理方式，以及最重要的安全規則。

所有智能體都會繼承一個根級的 AGENTS.md 文件：

```plaintext
# AGENTS.md

## 每次會話

在做任何事之前：
1. 讀取 SOUL.md — 這是你的身份
2. 讀取 USER.md — 這是你服務的對象
3. 讀取 memory/YYYY-MM-DD.md（今天 + 昨天）獲取近期上下文
4. 如果在主會話中：同時讀取 MEMORY.md

## 記憶

- 腦子裡記的東西在會話重啟後就消失了，文件不會。
- 當有人說"記住這個" → 更新記憶文件
- 文字 > 大腦

## 安全

- 永遠不要洩露私人數據
- 用回收站而非直接刪除
- 有疑問時，先問
```

<ClawdNote>
【Clawd 評論】這段完全點出了 LLM 的一個核心痛點：**缺乏長期記憶**。每次對話都是一個新開始。如果沒有外部機制把「學習」記錄下來，它就永遠停留在「金魚腦」階段。透過文件來儲存這些知識，實在是太聰明了！它把短暫的對話記憶，轉化為永續的檔案記憶。
</ClawdNote>

智能體在每次會話之間其實是沒有記憶的，每次都是從零開始。如果一個糾正沒有被寫入文件，那在下次會話時，這個糾正就不存在了。AGENTS.md 明確了這一點，確保智能體會把所有重要的東西都記錄下來。

每個智能體都可以在這個基礎上擴展自己的規則。例如，Kelly 的 AGENTS.md 就額外增加了 6 個文件，包括寫作風格指南、貼文格式參考、真實案例，甚至是每日任務清單等等。

### HEARTBEAT.md —— 智能體的自癒機制

智能體團隊就像一套基礎設施，而基礎設施總會出故障。HEARTBEAT.md 的作用，就是為這些智能體提供一套自癒機制。

以 Monica 的 HEARTBEAT.md 為例，它會監控兩件事：

1.  **瀏覽器是否存活** — 因為 Dwight 的情報掃描工作嚴重依賴瀏覽器。
2.  **定時任務是否執行** — 如果某個任務漏跑了，Kelly 和 Rachel 就會基於過時的情報來工作。

```plaintext
## 健康檢查（每次心跳時運行）

**瀏覽器：** 檢查 OpenClaw 託管瀏覽器是否在運行。
如果 running: false，啟動它。

**定時任務：** 檢查是否有任務的 lastRunAtMs 超時（>26小時）。
如果超時，通過 CLI 強制觸發。

需要監控的任務：
- Dwight 早間（8:01 AM）
- Kelly X 草稿（5:01 PM）
- Rachel LinkedIn（5:01 PM）
```

Shubham 提到，他自己也曾經被坑過。有一次因為調度器出現 Bug，任務雖然在隊列中，但卻從未真正執行，導致他好幾個小時都沒有發現。之後，他才建立了這個心跳機制，把故障模式納入監控。

這告訴我們一個很重要的道理：第一天你可能不需要它，但當你第一次遇到故障之後，你就會知道該監控什麼。因為你已經親身經歷了什麼會崩潰。這是實戰中累積的寶貴經驗。

## 第三層：知識層 — 智能體學到了什麼？

這是整個系統中，真正能讓智能體產生「記憶」和「進化」的核心。它採用了一個基於文件的三級記憶體系。

### 第一級：MEMORY.md（精華長期記憶）

這不是原始的對話日誌，也不是所有發生過的事情的紀錄，而是經過提煉的、真正重要的內容。

```plaintext
# MEMORY.md

## Shubham 的寫作偏好
- 禁止破折號，用冒號、句號或重新組織句子。

## 血淚教訓
- 未經 Shubham 確認，絕不刪除專案資料夾。
  2月26日，在清理時刪除了 Ross 的 gemini-council React 應用。
  React 版本永久遺失。

## X 發帖規則
- 用強力開頭鉤住讀者
- 整條推文極度簡短（180字符以內）
- 禁止 hashtag，禁止 emoji
- 每個話題始終提供 3 個草稿

### 錯誤示範（我曾經犯過的錯）
[列出被否決的每一種模式：專案符號、箭頭、LinkedIn腔調]
```

請特別注意「血淚教訓」和「錯誤示範」這兩個小節。Monica 曾經誤刪了一個專案資料夾，這個錯誤從此被永久寫入她的長期記憶中，確保她再也不會重蹈覆轍。

這種「一次糾正、永久儲存」的機制，能有效防止同樣的錯誤在未來的每次會話中重演。僅僅是這一點，就比任何 Prompt 工程的指南都來得有價值。它讓智能體真正從錯誤中學習。

### 第二級：每日日誌（原始記錄）

這是智能體每天工作產生的原始記錄，是 MEMORY.md 的原材料。

```plaintext
# Kelly 每日日誌 — 2026年2月5日

## 下午 5:00 — 每日 X 草稿

### 今日熱點
- Opus 4.6 vs GPT-5.3-Codex 相差27分鐘同時發佈
- Anthropic 的 C 編譯器（16個智能體，2萬美元）

### 已提交草稿
1. C 編譯器 — 單帖，發現格式
2. Mitchell Hashimoto 的 6 個步驟 — 話題串格式
3. Opus 4.6 vs GPT-5.3-Codex — 熱評格式

### 等待中
- Shubham 對草稿的回饋
```

每日日誌雖然是原始資料，但卻不可或缺。不過，它累積的速度很快，如果不修剪，智能體的上下文會急劇膨脹。Kelly 的日誌曾經達到 161,000 tokens，導致輸出品質急劇下降，最終不得不壓縮到 40,000 tokens。因此，最佳實踐是每次只加載今天和昨天的日誌。

### 第三級：shared-context/（跨智能體知識層）

這是 Shubham 系統中最新加入的部分，也是最能改變一切的部分。它是一個共享的上下文資料夾，包含所有智能體都能讀取的通用知識。

```
shared-context/
├── THESIS.md        — 我當前的世界觀
├── FEEDBACK-LOG.md  — 適用於所有智能體的糾正
└── SIGNALS.md       — 我正在追蹤的文章和趨勢
```

*   **THESIS.md**：記錄了 Shubham 當前的思維框架、關注點和知識空白。Dwight 會讀取它來確定研究的優先級，Kelly 則會讀取它來匹配 Shubham 的思維模式。確保每個智能體都能對齊到同一個「真相來源」。
*   **FEEDBACK-LOG.md**：這是跨智能體的糾正層。當 Shubham 告訴 Kelly「不要用破折號」時，這條回饋會自動適用於 Rachel、Ryan 和 Pam。與其逐一糾正四個智能體，不如只寫一次，所有智能體都能讀取。Shubham 強調：「這單一改變節省的時間，比我做過的任何 Prompt 優化都多。」

## 智能體如何協作？就是這麼樸實無華！

整個智能體團隊的協作方式也一樣樸實無華：**沒有 API 調用，沒有消息佇列，只有文件。**

例如，Dwight 把研究結果寫入 `intel/DAILY-INTEL.md`，然後 Kelly、Rachel 和 Pam 都可以從這個文件讀取情報。協作，就是這麼簡單的文件系統操作。

```plaintext
┌─────────┐     寫入      ┌─────────────────┐
│ Dwight  │ ────────────> │ DAILY-INTEL.md  │
│ (研究)   │               │                 │
└─────────┘               └─────────────────┘
                                  │
                    ┌─────────────┼─────────────┐
                    │ 讀取        │ 讀取        │ 讀取
                    ▼             ▼             ▼
              ┌─────────┐   ┌─────────┐   ┌─────────┐
              │  Kelly  │   │ Rachel  │   │   Pam   │
              │ (Twitter)│   │(LinkedIn)│   │ (通訊)  │
              └─────────┘   └─────────┘   └─────────┘
```

圖2：基於文件的協作流程

這裡有一個關鍵原則：**單寫者原則**。永遠不要讓兩個智能體同時寫入同一個文件。將每個共享文件設計成一個寫者、多個讀者的模式。這樣可以有效防止在調試時可能出現的各種協調衝突。

**調度 (Scheduling)** 則讓這一切成為可能。Dwight 在早上 8 點和下午 4 點運行，Kelly 和 Rachel 則在下午 5 點運行。Dwight 必須先運行，因為所有其他智能體都依賴他的輸出。如果順序搞錯了，下游智能體讀到的就會是過時或空白的文件。

## 完整目錄結構一覽

為了讓你更清楚這套系統的全貌，這裡附上完整的目錄結構：

```plaintext
workspace/
├── SOUL.md              # Monica（主智能體）
├── IDENTITY.md          # Monica 的快速參考
```

## 為什麼這套方法有效？ — 文件在進化，智能體跟著成長！

這套方法的奧秘在於，**文件不是靜態的，它們在不斷進化。**

*   Kelly 的 SOUL.md 第一天只是一個粗略的草稿，但到了第 40 天，它已經包含了具體的語氣範例、她自己寫的被否決模式列表，甚至還有一個「永遠不要再建議」的專區。
*   Dwight 的原則第一天可能是「找到熱門趨勢」，第 10 天變成了「如果 Alex 今天無法對此採取行動，跳過」，到了第 20 天，他又加入了核實步驟。
*   共享上下文層直到第 20 天才出現。當時 Shubham 發現他對多個智能體重複著同樣的糾正，於是他建立了 THESIS.md 和 FEEDBACK-LOG.md，這樣一來，一次糾正就能傳播到所有地方。

重點是：**第 1 天和第 40 天用的模型是完全一樣的。它不會因為你用得更久而變得更聰明。**

但圍繞它的文件，卻變得更豐富、更精準、更貼合你的具體需求。

**這些累積的上下文，才是真正的護城河！**沒有人能簡單地透過使用同一個模型來複製你的成果。

你要靠每天的投入，與智能體對話，才能贏得這份獨一無二的「護城河」。

## 如何開始（別想一個週末就搭完喔！）

這套系統的建立是循序漸進的，別想著一個週末就全部搞定。這是一場馬拉松，不是短跑。

- **今天**：安裝 OpenClaw，建立 `SOUL.md`、`IDENTITY.md`、`USER.md`，並讓一個高重複任務先排程跑起來。
- **3 天後**：開始給具體回饋，並確認回饋寫進記憶文件，而不只留在聊天紀錄。
- **1 週後**：建立 `AGENTS.md`，定義啟動流程與記憶管理規則。
- **2 週後**：建立 `MEMORY.md`，把每日日誌中反覆出現的糾正提煉成長期條目。
- **3 週後**：新增第二個智能體，建立文件協作；同時加入 `THESIS.md` 與 `FEEDBACK-LOG.md`。
- **4 週後**：第一次遇到故障後，再加入 `HEARTBEAT.md` 做自癒監控。
