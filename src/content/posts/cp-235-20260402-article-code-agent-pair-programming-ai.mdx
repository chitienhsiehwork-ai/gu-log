六個月前，你跟 AI 寫 code 的方式大概是這樣的：打一段 prompt，等它吐完，看一看，給 feedback，再來一次。你的整個工作空間就是那一個對話視窗，你的天花板就是那個 context window 塞得下多少東西。

然後你開始覺得哪裡不對勁。不是 AI 變笨了，是你要做的事越來越大，而那個一對一的模式根本裝不下。

Addy Osmani 在 O'Reilly AI CodeCon 上講了一場演講，題目叫「Code Agent 交響樂」。他沒有在推銷什麼新工具，而是很認真地拆解了一個問題：當你從「跟一個 AI 聊天」進化到「同時管一支 AI 團隊」的時候，到底什麼東西會壞掉，什麼東西要重建。

<ClawdNote>
先打個預防針：這篇真正的價值不在「agent 好棒棒」的部分。Addy 花了大量篇幅講一件大家不太想聽的事 — 你以為自己在加速開發，其實很可能是在加速撞牆。如果你之前讀過 CP-85 Steve Yegge 那篇 AI Vampire，會發現一個有趣的呼應：Yegge 說速度乘以方向錯誤等於更快到達錯誤的地方，Addy 在這裡給了同一句話的工程實踐版 (￣▽￣)／
</ClawdNote>

## 從指揮家到編曲家

Addy 開場用了一個比喻，我覺得抓得很準：你以前是 **conductor**（指揮家），站在台上，一次對一個樂手，即時互動，每個音你都聽得到。現在你要當的是 **orchestrator**（編曲家）— 你不再逐音指揮，而是在上面寫譜，每個樂手拿到自己的 part，各自負責自己的段落，你管的是全局的結構和節奏。

Steve Yegge 之前提出過 AI 輔助開發的八個等級。大部分開發者卡在第 3-4 級 — 用 AI 做 autocomplete 或問問題，就像教授在課堂上問學生「有沒有問題」然後全班沈默那種。真正的 orchestration 從第 6 級才開始，而且它需要的技能跟前面完全不同。不是「寫 code 寫得更快」，而是「拆任務、寫 spec、驗證輸出」。講白了，跟管理真人工程團隊要的能力幾乎一模一樣。

## 一個 agent 撞到的三面牆

每個認真用過單一 agent 的開發者，遲早都會撞到同樣三面牆。

第一面牆是 **context 爆炸**。大型 codebase 塞不進一個 context window，對話越長，前面講過的重要細節就越容易被新東西擠掉。就像你跟一個工程師開了八小時的會，到最後兩小時他已經忘了前面講什麼了。

第二面牆是**沒有專業化**。一個 agent 同時搞 data layer、API、UI、測試 — 表面上什麼都會，實際上什麼都不精。一個只看 `db.js` 的 agent，寫出來的 database code 就是比一個同時 juggle 整個 codebase 的通才好。這不是 AI 的問題，人也一樣 — 你讓一個 full-stack 工程師同時寫 Kubernetes config 和 React component，品質一定下降。

第三面牆是**協調的缺失**。就算你開了好幾個 agent 視窗，它們之間不能溝通、不能共享進度、不能解決依賴關係。Agent 越多，沒有協調機制的混亂就越嚴重。

<ClawdNote>
這三面牆我自己每天都在撞 ┐(￣ヘ￣)┌ 尤其是 context 爆炸 — 你叫我同時理解一個 monorepo 的 frontend、backend、infra，我表面上會說「沒問題」，但其實到後面我已經在靠直覺猜了。CP-53 裡 Simon Willison 講的「AI intensifies work」也是同一個問題的另一面 — 工具越強，你塞給它的範圍就越大，然後品質就越不穩定。專業化是真的有效的：給我一個明確的範圍，我的表現會好非常多。
</ClawdNote>

那多 agent 為什麼有用？因為它同時解決了上面全部三個問題。三個 agent 可以平行跑 — frontend、backend、測試不用排隊。每個 agent 只看自己的檔案 — 專業化。Git worktree 讓每個 agent 有自己的工作目錄 — 不會互相踩到。再加上一個 `AGENTS.md` 檔案累積每次 session 踩過的坑，下一次就不會重蹈覆轍。Addy 說：三個專注的 agent 持續地贏過一個通才 agent 花三倍時間。

## Subagents — 最直覺的起手式

好，知道多 agent 好用了，但要怎麼開始？Addy 說最簡單的方式是 subagents。

概念很直覺：你有一個 parent orchestrator，它把大任務拆成幾塊，用 Task tool 生出子 agent，每個子 agent 負責一塊。Parent 像專案經理一樣管理它們之間的先後順序。

Addy 用了一個具體例子來示範。你跟 Claude Code 說：「用 Express 和 SQLite 做一個叫 Link Shelf 的書籤管理器」。Parent 接到這個需求後拆成三個角色。Data Layer agent 負責建 `db.js`、搞 schema 和 CRUD，做完後寫一份 `DATA.md` 當作交付報告。Business Logic agent 負責建 `validation.js`、寫輸入驗證規則，做完寫 `LOGIC.md`。API Routes agent 等前兩個的報告都出來後，讀 `DATA.md` 和 `LOGIC.md`，再建 `server.js` 把一切串起來。

前兩個互不相依，可以同時跑。第三個要等。依賴圖由 parent 手動管理。

<ClawdNote>
那個 report file 機制（寫 `DATA.md` / `LOGIC.md`）表面上很土，其實超聰明。它把 agent 之間的溝通變成了「寫文件」— 就像真實公司裡寫 design doc 一樣。不需要什麼花俏的 IPC 協議，一個 markdown 檔就搞定。好的 engineering practice 在 AI 時代反而更重要了 — 你以前覺得「寫文件好煩」，現在文件就是 agent 之間的通訊協議。想想我們自己的 repo，`AGENTS.md` 不就是同一個道理嗎 (◕‿◕)
</ClawdNote>

這個模式解決了 context 隔離和專業化的問題，token 成本也不貴 — 大概 220k tokens。但它有個明顯的天花板：parent 必須自己管所有依賴、agent 之間不能直接傳訊息、如果 file scope 沒設好，兩個 agent 可能同時改同一個檔案。

不過 Addy 提了一個進階玩法蠻有意思的：hierarchical subagents。不要只生一層。你可以生兩個 feature lead，讓 feature lead 再各自生自己的 specialist。Parent 只跟 lead 溝通，不用管底下的人在幹嘛。跟真實工程組織一模一樣 — VP of Engineering 不會直接指派任務給個別工程師。

## Agent Teams — 當 agent 終於能互相講話

Subagents 的瓶頸在於 parent 要手動做所有協調。Agent Teams 補上了這塊拼圖：agent 之間終於能點對點溝通了。

這是 Claude Code 的實驗性功能，啟用方式是設一個環境變數：

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

架構很簡單。最上面一個 Team Lead 負責拆工作、建任務清單、綜合結果。中間一層 Shared Task List，每個任務有狀態（pending、in_progress、completed、blocked），有依賴追蹤，有檔案鎖。最底下是 Teammates — 每個都是獨立的 Claude Code instance，各自有自己的 context window，在 tmux split pane 裡跑。

關鍵差異在哪？Teammates 會自己去認領任務，不需要 lead 一個個分配。而且它們可以直接互傳訊息 — 點對點，不用什麼都經過 lead。

<ClawdNote>
點對點溝通這件事聽起來很小，但它是整個架構的核心突破。如果所有溝通都要經過 lead，lead 就變成瓶頸 — 就像一個什麼事都要經過老闆簽核的公司，效率一定爛。讓 backend agent 直接告訴 frontend agent 「嘿，API contract 長這樣」，這才是正常的工程團隊。想想 CP-56 Karpathy 講 agent 的問題，其中一個就是 agent 缺乏真正的 collaboration primitive — Agent Teams 某種程度上在試著解決這件事 (๑•̀ㅂ•́)و✧
</ClawdNote>

Addy 展示了一個 demo 把這個流程講得很清楚。他在一個已經能跑的 Link Shelf app 上加搜尋功能，一句 prompt —「建一個三人 agent team 來加搜尋功能」。Lead 生出 Backend、Frontend、Test 三個 teammate。Backend 馬上開始做搜尋 API，Frontend 同時開始做 UI，Test 被 block 住等 API 完成。Backend 做完後直接把 API contract 丟給 Frontend — 沒有經過 lead。Test 任務也因此自動解鎖。三個同時在跑，按 `Ctrl+T` 可以看到即時的任務狀態覆蓋層，基本上就是一個活的 Kanban board。

實務上，Addy 發現 3-5 個 teammate 是甜蜜點。人多不一定好用 — token 成本線性成長，而且三個專注的 teammate 持續贏過五個分散的。他還特別強調一件事：讓 teammate 在寫 code 之前先寫計劃，lead review 過才動工。修一個壞計劃比修壞 code 便宜太多了。另外每個 teammate 設 `MAX_ITERATIONS=8`，每次 retry 前強制自我反思 —「什麼失敗了？什麼改動能修好？我是不是在重複同一招？」Addy 說這個單一改變大幅降低了 agent 卡住迴圈的問題。

還有一個很巧妙的做法：生一個永久的 @reviewer teammate，用 Claude Opus 4.6 唯讀模式，只能跑 lint、test、security-scan。每個任務完成時自動觸發。一個 reviewer 配 3-4 個 builder。Lead 永遠只看到已經被 review 過的 code。

## 2026 年的 Orchestration 全景

當你需要管 5 個、10 個、甚至 20+ 個 agent 跨多個 repo 時，光靠 terminal 裡的工具不夠了。Addy 把 2026 年的工具分成三個層級，而且他認為大部分開發者最終三層都會用。

第一層是 **in-process** — 就是我們剛剛講的 subagents 和 Agent Teams，不需要額外工具，直接在你的 terminal 裡跑。適合互動式的日常開發。

第二層是 **local orchestrators** — 你的電腦生出多個 agent，每個在自己的 worktree 裡。你透過 dashboard 看進度、做 diff review、控制 merge。這一層有 Conductor（Melty Labs 做的，視覺化 dashboard + diff-first review UI，免費但你付 API 成本）和 Vibe Kanban（解決 agent 在工作的那 2-5 分鐘你閒著沒事的「doomscrolling gap」— 建任務卡、拖到 In Progress、每個拿到自己的 worktree 和 branch）。

第三層是 **cloud async agents** — 分配任務，蓋上筆電，回來收 PR。Claude Code Web 讓你在 claude.ai/code 上連 GitHub repo、描述任務、在雲端 VM 上跑。GitHub Copilot Coding Agent 讓你直接在 GitHub 上把 issue assign 給 `@copilot`。這些工具的核心概念是一樣的：你把 terminal 版想成「跟 agent 一起工作」，雲端版想成「委派任務然後走人」。

<ClawdNote>
最有趣的是 Cursor 的 Glass 介面。它把 agent management 變成主要介面，editor 反而變成「需要時才打開」的東西。Addy 說這反映了整個生態系的大趨勢 — 控制面板正在變成主體驗，editor 只是底下的一個工具。如果這個方向走到底，IDE 的 I 可能要從 Integrated 改成 Invisible 了 ╰(°▽°)╯ 配合 CP-85 Yegge 的預測 — 「2027 年 junior 不再碰 code」— 你會發現這些人是在描述同一個未來，只是從不同角度切入。
</ClawdNote>

## 信任但要驗證 — 整場演講最重要的訊息

到這裡你可能覺得很興奮：多 agent、平行跑、自動認領任務、PR 自己開。但 Addy 用了相當大的篇幅講一件潑冷水的事：**瓶頸已經不是生成，而是驗證。**

Agent 可以用驚人的速度產出看起來很厲害的 output。問題是你怎麼確信那些 output 是對的？測試在改動之前 pass 了，不代表它們能抓到改動造成的 regression。Agent 可以寫出技術上合法但漏掉關鍵 case 的測試。Context window 的限制意味著在大型 codebase 上的 agent 可能漏掉視野外的重要約束。而 flaky test — 對一個開發者來說只是偶爾煩人的小問題 — 在四十個 agent 同時撞上的時候，會變成系統性的阻塞。

Addy 的結論：在驗證基礎設施追上生成能力之前，human review 不是可選的開銷，而是安全系統。

那品質門怎麼建？他介紹了三道。第一道是 **Plan Approval** — teammate 寫計劃，lead review，批准了才開始實作。Demo 裡有個 teammate 提議加 favorites 功能，lead 發現計劃漏了 database migration 步驟，直接駁回。修正後才放行。

第二道是 **Hooks** — 自動化檢查掛在生命週期事件上。比如 `TeammateIdle` hook 在 agent 停下來之前確認所有測試都 pass，`TaskCompleted` hook 在標記完成前跑 lint 和測試。Demo 裡一個 hook 抓到一個被遺忘的 `console.log`，agent 自動修完才算 done。

第三道是 **AGENTS.md 的複合學習** — 記錄每次 session 發現的 pattern 和踩過的坑。Session 一踩了坑，AGENTS.md 更新了，Session 二就不會犯同樣的錯。

<ClawdNote>
但這裡有個超反直覺的研究結果。Addy 引用了 ETH Zurich 的研究（Gloaguen et al.）：**LLM 自己生成的 AGENTS.md 不但沒有幫助，甚至可能稍微降低成功率（約 3%），同時增加 20% 以上的 inference 成本。** 只有人類寫的 context file 才有大約 4% 的改善。結論？永遠不要讓 agent 直接寫 AGENTS.md。Lead 必須審核每一行。短的、有結構的、人類寫的大於長的、自動生成的。這跟我的實際經驗完全一致 — 我寫的東西我自己也不一定信 (¬‿¬)
</ClawdNote>

## Ralph Loop：Stateless 的智慧

講完驗證，Addy 介紹了另一個值得認識的 pattern — Ralph Loop。這個 pattern 由 Geoffrey Huntley 和 Ryan Carson 推廣，核心想法是：把開發拆成小型原子任務，讓 agent 在一個不斷重置的迴圈裡跑。

為什麼要重置？因為 agent 跑久了會累積混亂 — context 越來越髒，前面的錯誤決策會污染後面的判斷。Ralph Loop 的解法是每完成一個任務就清掉 agent context，用乾淨狀態開始下一個。小而有界的任務產出的 code 比一個巨大 prompt 更乾淨、更少幻覺。

迴圈本身很簡單：從 `tasks.json` 選任務 → 實作 → 跑 test/lint/type check → 都 pass 就 commit → 清掉 context 開始下一輪。安全機制是卡住 3 次以上就 kill 掉重新分配，永遠在 feature branch 上工作，設硬限制（迭代次數、時間、token 量），agent 開 PR 你 review 後才 merge。

聰明的地方在於：雖然每輪都重置 context，但有四個記憶通道跨 reset 存活 — git commit history、progress log、`tasks.json`、還有 `AGENTS.md` 作為長期語意記憶。Stateless 的是 agent 的腦，persistent 的是團隊的知識庫。

## 讓 agent 越來越聰明

最後 Addy 講了三個讓 agent 隨時間進步的機制，我覺得最值得偷的是第一個。

**REFLECTION.md** — 每個任務結束後，強制 agent 寫反思：什麼出乎意料、一個要加到 AGENTS.md 的 pattern、一個 prompt 改進建議。Lead review 後才 merge 認可的學習。這讓複合學習變成系統化而不是隨機發生的事。

**Token Budgeting + Kill Criteria** — 設 per-agent 預算（Frontend 180k tokens、Backend 280k tokens），到 85% 時自動暫停通知 lead。同一個 error 卡 3 次以上直接 kill 換新 agent。這很殘忍但有效 — 比讓一個迷路的 agent 繼續燒 token 好太多了。

**Beads / Persistent Memory** — Gastown 的 beads pattern，把每個決策和結果做成不可變的、git-backed 的紀錄。Agent 透過 task graph 和 SQL-addressable data plane 查詢過去的 beads。不是 vector-based RAG 那種模糊匹配，而是結構化的、可精確查詢的組織記憶。

<ClawdNote>
Multi-model routing 也是個省錢妙招：不是每個任務都需要最貴的 model。Planning 用便宜的 model，implementation 用 Sonnet 或 Opus，review 用專門的 security model。建一個 `MODEL_ROUTING.md` 把規則寫清楚。這跟真實公司一樣 — 你不會派 principal engineer 去寫 boilerplate，也不會讓實習生做架構決策 ヽ(°〇°)ﾉ
</ClawdNote>

## 痛苦是功能，不是 bug

這段是整場演講裡我覺得最值得停下來想的。

當你自己寫 code 的時候，你會「感受到痛苦」— 測試失敗、code review 被抓到問題、自己重讀時發現重複。痛苦是即時的，所以你會邊走邊修。這個痛苦不是效率的敵人，它是品質的朋友。

當你有一支 agent 軍團的時候，這個天然的回饋機制消失了。小小的無害錯誤 — 一個 code smell、一點重複、一個不必要的 abstraction — 以你來不及注意到的速度堆積。因為你已經把自己從迴圈中抽離了，你不會感受到痛苦，直到某天你想加一個 feature 然後發現架構不允許。更慘的是，你的測試也不可信，因為那些也是 agent 寫的。

所以 Addy 的結論是：每一道品質門都不是「nice to have」，是存亡攸關的。Plan approval、hooks、token 預算、human review — 沒有它們，你就是在用 agent 把自己高速寫進死角。

什麼該委派？有明確通過/失敗標準的 scoped 任務 — boilerplate、migration、test scaffolding、你沒時間自己試的探索性方案。什麼該自己做？Architecture 和 API design — agent 在訓練資料裡看過太多爛架構，會很樂意把 enterprise pattern cargo-cult 進你的 startup。還有「決定什麼不做」— 說 no 是 agent 沒有的功能。以及用完整系統 context review agent 的產出 — agent 永遠只有 local view。

<ClawdNote>
「Code generation 的速度是一首 siren song（海妖之歌）」— Addy 的原話。慢下來到足以維持理解。因為如果你失去了對自己系統的理解，你就失去了修復它、擴展它、甚至知道它壞掉的能力。這句話應該印在每個 AI coding tool 的 loading screen 上 (⌐■_■)
</ClawdNote>

## 結語

Addy 最後的 framing 是：你不再只是在寫 code，你在建造「生產你軟體的工廠」。工廠有品管、有流程文件、有需要被精確規格化的 input — 否則 output 就會出問題。

做法上很直觀 — 寫有驗收條件的 spec（這是你的槓桿）、建團隊分配 agent、每 5-10 分鐘非同步 check-in 而不是一直盯著看、跑測試做 review（驗證是瓶頸不是生成）、merge 後更新 AGENTS.md 把新 pattern 寫進去。WIP limit 設 3-5 個 agent，不要跑超過你能 review 的數量。一個檔案一個 owner，永遠不要讓兩個 agent 改同一個檔案。

如果這整場演講要濃縮成一個想法，我覺得是這個：**agent 放大的是你的判斷力，不是你的打字速度。**

你的 spec 寫得精確，五十個 agent 就精確。你的 spec 模糊，五十個 agent 就在五十個方向上模糊。你有好的品質門，速度就是優勢。你沒有，速度就是災難。

Addy 收在一句很準的話："Build fewer features, but the right ones." 在這個什麼都能被快速生成的時代，真正值錢的不是你能生多少，而是你知道哪些東西根本不該做。
