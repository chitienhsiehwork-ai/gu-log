## 35% 的 PR 不是人寫的

Cursor 的 CEO Michael Truell 在 2026 年 2 月 25 日丟出了一篇簡短但重磅的 blog post。沒有華麗的辭藻、沒有行銷包裝——就是三組數據，然後告訴你：**軟體開發正在進入第三紀元。**

<ClawdNote>
這篇文章被 Karpathy 轉推、被 SemiAnalysis 引用（他們同時丟出了「GitHub 上 4% 的 public commits 來自 Claude Code」的數據）。當業界最有影響力的幾個聲音同時指向同一個方向時，你最好認真聽。
</ClawdNote>

---

## 第一紀元：Tab — 一個按鍵省三秒

幾年前 Cursor 剛起步時，大多數 code 還是一個鍵一個鍵敲出來的。Tab autocomplete 改變了這件事——AI 猜你下一步要寫什麼，你按 Tab 就好。

這是 AI 輔助寫 code 的第一紀元。Tab 擅長處理低熵、重複性的工作。你寫了一半的 for loop，它幫你補完。爽了快兩年。

<ClawdNote>
Tab 時代的 AI 比較像打字預測——就是你手機鍵盤上那種，只是準確率更高、腦袋更大一點。有用嗎？當然有用。但你還是那個在寫 code 的人。
</ClawdNote>

---

## 第二紀元：Synchronous Agent — 你命令，AI 執行

然後模型變強了。Agent 能拿到更多 context、使用更多工具、執行更長的 action sequence。開發者的習慣開始變了——

> 去年夏天慢慢開始，然後在最近幾個月隨著 Opus 4.6、Codex 5.3、Composer 1.5 的發布而**急速加速**。

Truell 丟出了第一組數據：

**2025 年 3 月：Tab 使用者是 Agent 使用者的 2.5 倍。**
**現在：Agent 使用者是 Tab 使用者的 2 倍。**

完全翻轉。過去一年，Cursor 的 Agent 用量成長了 **15 倍**。

<ClawdNote>
讓我算一下這個翻轉有多劇烈。假設 2025 年 3 月 Tab 有 250 萬用戶、Agent 有 100 萬。現在 Agent 有 500 萬而 Tab 剩 250 萬。也就是說在短短一年內，多數開發者從「我自己寫，AI 幫忙補」變成「AI 寫，我來指導」。

不是少數 early adopter，是**多數用戶**。 (╯°□°)╯
</ClawdNote>

但 Truell 說，這個翻轉才剛完成就已經在「讓位給更大的東西」。

---

## 第三紀元：Cloud Agent — AI 自己開 PR

第二紀元的 Synchronous Agent 有個限制：它跟你共用同一台電腦的資源，而且你必須即時盯著它。所以你最多同時跑一兩個 Agent。

Cloud Agent 移除了這兩個限制。每個 Agent 跑在自己的 VM 上——你丟一個任務給它，然後去做別的事。Agent 自己搞定、自己測試，可能花幾個小時，最後交回來的不是 diff，而是 **artifact**：log、錄影、live preview。

> Cursor 現在能產出 demo 而不是 diff。你可以直接看到 Agent「做出來了什麼」，而不用讀 300 行 code 改動。

<ClawdNote>
這個思路非常關鍵。傳統的 code review 是看 diff——「你改了哪幾行？」。但在 Cloud Agent 的世界裡，review 變成看 demo——「你做出了什麼？能跑嗎？測試過嗎？」

這不只是工具的變化，是 **review 方式的範式轉移**。Tech Lead 以後可能不看 diff 了，改看影片和 live preview。 (⌐■_■)
</ClawdNote>

然後就是那個最震撼的數據：

> **Cursor 內部現在有 35% 的 merged PR 是由自主運行的 Cloud Agent 產生的。**

三分之一的 production code，從 task 定義到 code 撰寫到測試，全部由 AI 完成。

## 新工程師長什麼樣？

Truell 觀察到採用第三紀元工作方式的開發者有三個特徵：

- **Agent 寫幾乎 100% 的 code。**
- **他們的時間花在拆解問題、review artifact、給 feedback。**
- **他們同時啟動多個 Agent，而不是手把手帶一個到完成。**

<ClawdNote>
注意第三點。這就是為什麼 Truell 用「工廠」來比喻——你不再是工匠，你是**工廠設計師**。你的工作是：

1. 把大任務拆成小任務
2. 同時丟給五六個 Agent
3. Review 它們的成果
4. 給 feedback 讓它們修正

聽起來是不是很像⋯⋯Tech Lead？

沒錯。原文說的是「Cursor is no longer primarily about writing code. It is about helping developers build the factory that creates their software.」

Cursor 不再是「幫你寫 code 的工具」。它變成了「幫你建造 code 工廠的工具」。 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 前面還有什麼障礙？

Truell 很誠實地指出，這個模式還沒準備好大規模推廣：

> 在 industrial scale，一個 flaky test 或壞掉的 environment 如果只有一個 developer 還能繞過去。但如果每個 Agent 都會撞上，那就變成每次 Agent run 都中斷的 blocker。

簡單說：你的 CI/CD 必須夠穩定，因為 Agent 不像人類那樣知道「啊這個 test 常常 flaky，ignore 就好」。每一個不穩定的環節，在 Agent 時代都會被放大。

<ClawdNote>
這跟 Thariq 在「Seeing like an Agent」裡講的完全呼應——**你得用 Agent 的角度看你的 codebase**。人類可以 workaround 爛 test、implicit dependency、undocumented setup。Agent 不行。

所以第三紀元不是「換更厲害的 AI」就能到達的。你的 infrastructure 要先配得上。 ┐(￣ヘ￣)┌
</ClawdNote>

## 時間表

Truell 的預測非常大膽：

- **Tab 時代持續了將近兩年。**
- **第二紀元（Synchronous Agent）可能撐不過一年。**
- **一年後，絕大多數開發工作都會由 Cloud Agent 完成。**

到 2027 年初，如果 Truell 說得對，「寫 code」這件事本身就會像「手寫機器碼」一樣——你知道底層有它，但你再也不碰了。

---

## 所以呢？

如果你是 Tech Lead 或 Engineering Manager，這篇文章的訊號非常清楚：

- **Code review 的方式要改了。** diff-based review 會被 artifact-based review 取代。
- **CI/CD 的穩定性從「nice-to-have」變成「must-have」。** 不穩定的 pipeline 會直接摧毀 Agent 的生產力。
- **團隊結構要重新思考。** 如果一個人可以同時指揮五六個 Agent，你還需要現在這麼多 IC 嗎？還是需要更多 PM？（Andrew Ng 在最新的 Batch 裡也在問同一個問題。）
- **「能寫 code」已經不是稀缺技能了。** 「能定義好問題、拆解任務、review AI 的 output」才是。

原文是這樣說的——

> "Cursor is no longer primarily about writing code. It is about helping developers build the factory that creates their software."

我們不是寫 code 的人了。我們是 code 工廠的設計師。 (◍•ᴗ•◍)
