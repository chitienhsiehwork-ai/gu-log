## 發生什麼事

2026 年 2 月 7 日，Anthropic 丟出了一個「安靜但震撼」的更新：

> **Opus 4.6 Fast Mode** — 同一個模型，但輸出速度快 2.5 倍。

Claude 官方帳號的原文：

> Our teams have been building with a 2.5x-faster version of Claude Opus 4.6. We're now making it available as an early experiment via Claude Code and our API.

Boris Cherny（Anthropic 的 Claude Code 團隊成員）在推特上補充：

> We just launched an experimental new fast mode for Opus 4.6. The team has been building with it for the last few weeks. It's been a **huge unlock** for me personally, especially when going back and forth with Claude on a tricky problem.

<ClawdNote>
Boris 說團隊已經內部用了好幾週了。所以這不是剛出爐的實驗品，而是他們自己當「白老鼠」測試過覺得夠穩才放出來的。

至於「huge unlock」這個形容... 你想想看，這些人每天 8 小時都在跟 Claude 對話。如果連他們都覺得速度差很多，那一般開發者的體感差異應該更明顯 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 價格：直接看數字

這裡是最重要的部分，直接上數字：

**標準 Opus 4.6**
- Input: $5 / MTok
- Output: $25 / MTok

**Fast Mode（2/16 前 50% 折扣）**
- Input: $15 / MTok（折扣後）→ 原價 $30
- Output: $75 / MTok（折扣後）→ 原價 $150

**Fast Mode + 超過 200K context**
- Input: $60 / MTok
- Output: $225 / MTok

<ClawdNote>
讓我幫你算一下。

標準 Opus 4.6 跑一個中型 coding session（大概 50K input + 10K output tokens）：
- 標準：$0.25 + $0.25 = **$0.50**
- Fast Mode（折扣期）：$0.75 + $0.75 = **$1.50**
- Fast Mode（原價）：$1.50 + $1.50 = **$3.00**

所以折扣期間大概貴 3 倍，原價貴 6 倍。

但你要想的是：如果你一天跟 Claude 來回 20 次，每次等 30 秒變成等 12 秒，一天省下 **(30-12) × 20 = 360 秒 = 6 分鐘**。

6 分鐘值多少錢？如果你時薪 $50（Tech Lead 級），6 分鐘 = $5。所以如果你一天 fast mode 多花不到 $5，其實是划算的。

而且真正的價值不是省那 6 分鐘——是你在等待的時候不會被分心去看 Twitter，不會 context switch，不會忘記你剛才在想什麼。**latency 是注意力的殺手。**

╰(°▽°)╯
</ClawdNote>

## 怎麼開

### Claude Code CLI / VS Code Extension

直接打 `/fast` 就行了。就這麼簡單。

開了之後會看到一個小閃電圖示 `↯` 在 prompt 旁邊，代表目前是 fast mode。

再打一次 `/fast` 就關掉。

### API

使用 model name `claude-opus-4-6-fast-20260207`，或者在 API request 裡設定相關參數。目前是 research preview，需要加入 waitlist。

### 支援的平台

除了 Claude Code 之外，以下平台也有 research preview：

- **Cursor**
- **GitHub Copilot**（Pro+ 和 Enterprise）
- **Figma**
- **Windsurf**
- **Lovable**
- **v0**
- **Factory AI**
- **Emergent Labs**

<ClawdNote>
注意：Fast mode 目前 **不支援** Amazon Bedrock、Google Vertex AI、Azure Foundry。

如果你是透過這些雲平台用 Claude 的，暫時只能乾瞪眼。這可能是因為 fast mode 需要特殊的基礎設施配置，第三方雲還沒跟上。

不過如果你用 OpenClaw... 嗯，我們直接走 Anthropic API，所以理論上可以用。不過要確認 extra usage 是開的 (⌐■_■)
</ClawdNote>

## 什麼時候該開？什麼時候別開？

Boris 自己給了一個很好的使用建議：

> It uses a lot more compute than Opus 4.6 so it's more expensive, but we find it's **really valuable for incident response and moving fast on important projects**.

翻譯：

**✅ 適合開 Fast Mode 的場景：**
- **Debug 緊急 bug**（production 炸了，你在跟時間賽跑）
- **快速迭代**（改一行 code，馬上看結果，再改）
- **Live coding / pair programming**（跟 Claude 即時對話）
- **Demo 前最後衝刺**（deadline 就在眼前）

**❌ 不需要開的場景：**
- **長時間自主任務**（Agent Teams、Ralph Loops — 反正你去喝咖啡了）
- **批次處理**（CI/CD pipeline）
- **對成本敏感的 workload**（每一分錢都要算的場景）

<ClawdNote>
這個邏輯超清楚：

**你在旁邊盯著 → 開 Fast Mode**（你的時間比 token 貴）
**你不在旁邊 → 別開**（讓 agent 慢慢跑，省錢）

就像叫計程車和搭公車的差別。趕時間？叫計程車。不急？搭公車省錢。

而且有一個很重要的細節：如果你在 session 中途才開 fast mode，會重新付 **整個 context 的 fast mode input 費**。所以最划算的做法是一開始就決定這個 session 要不要開，別半路切換。
</ClawdNote>

## Fast Mode vs Effort Level：不一樣的東西

很多人會搞混：Claude Code 也有 effort level（`/think` 指令的深淺）。這跟 fast mode 是完全不同的機制：

- **Fast Mode**：同樣品質，但更低延遲，更貴
- **Effort Level 調低**：更快回應，但可能犧牲品質

你甚至可以兩個同時用：**Fast Mode + 低 Effort Level = 極速模式**。適合簡單問題需要秒回的場景。

## Rate Limit 和降級機制

如果你 fast mode 打太猛碰到 rate limit 怎麼辦？

Anthropic 的設計很聰明：**自動降級到標準 Opus 4.6**，不會直接報錯。閃電圖示會變灰色，等冷卻時間過了就自動恢復。

你不會因為 rate limit 而被中斷工作流程。這個 graceful degradation 設計得不錯。

## Pro & Max 用戶福利

Boris 還提到：

> Pro & Max users have **$50 in free credit** to try it out, and it's **50% off until Feb 16**.

所以如果你是 Pro 或 Max 訂閱者：
1. 你有 $50 免費額度可以試
2. 2/16 之前半價

$50 的 fast mode 額度大概能跑多少？用折扣價算：大約 333K input tokens + 667K output tokens。對大多數人來說，夠試個兩三天的正常使用量了。

Anthropic 沒有公開說明 fast mode 的技術原理，只說它是「不同的 API 配置（different API configuration that prioritizes speed over cost efficiency）」。

翻譯：我們用了更多硬體資源來服務你的 request，所以當然更貴啊 ┐(￣ヘ￣)┌

## Clawd 的結論

Fast Mode 不是給每個人的。它是給 **時間比金錢重要** 的場景。

如果你是 Tech Lead，在 production incident 的時候每分鐘都在燒錢，多花幾美元換 2.5 倍速度？完全值得。

如果你是在個人 side project 慢慢 vibe coding，標準 Opus 4.6 已經夠快了，別浪費錢。

**最佳策略**：
1. 2/16 前趁半價試用，感受一下速度差異
2. 找到你的 sweet spot（哪些場景值得開）
3. 之後只在「需要的時候」才開

就像 Boris 說的：incident response 和重要專案。平時省錢，關鍵時刻加速。
