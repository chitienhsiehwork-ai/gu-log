## Automatic Caching — 終於不用自己記停車位了

以前用 prompt caching，你得在每個想 cache 的 content block 上手動加 `cache_control`。這就像去大賣場，你不但要自己記哪個走道放了什麼，還要在每個貨架上貼一張便利貼提醒自己「這個我常買」。

```yaml
system:
  - type: text
    text: 你是一個文學分析助手...（一大段 system prompt）
    cache_control:
      type: ephemeral
```

在 multi-turn 對話裡更痛苦 — 每次新增訊息，你都得決定要把 `cache_control` 放在哪裡。SP-73 那篇有提到，Claude Code 團隊花了大量工程時間就是在搞這件事。

<ClawdNote>
想像你每天上班都要重新跟保全解釋一次「我是這棟大樓的員工」— 出示證件、背誦員工編號、等他打電話確認。Prompt caching 就是讓保全記住你的臉：「喔你啊，直接進去吧。」但以前你得自己跟保全說「記住我的臉」，現在 automatic caching 讓大樓管理處統一處理了 (￣▽￣)／
</ClawdNote>

現在 Anthropic 推出了 **Automatic Caching**：你只要在 request 的最上層加一個 `cache_control`，系統會自動把 cache breakpoint 放在最後一個可 cache 的 block 上。

```yaml
model: claude-opus-4-6
max_tokens: 1024
cache_control:
  type: ephemeral
system: 你是一個會記住對話的助手。
messages:
  - role: user
    content: 我叫 Alex，我做 ML 的。
  - role: assistant
    content: 嗨 Alex！今天想聊什麼 ML 的話題？
  - role: user
    content: 我剛剛說我做什麼的？
```

就這樣。不用在每個 block 上標記，系統自己搞定。

### Multi-turn 對話的自動前移

Automatic caching 最聰明的地方是 **cache breakpoint 會隨著對話自動往前移**。你可以把它想成便利商店的「熟客模式」— 店員會記住你最近買過什麼，但記憶會隨著新交易不斷更新：

- **第 1 次請求**：System + User(1) + Asst(1) + User(2) ← 全部寫入 cache
- **第 2 次請求**：System 到 User(2) 從 cache 讀取；新增的 Asst(2) + User(3) 寫入 cache
- **第 3 次請求**：System 到 User(3) 從 cache 讀取；新增的 Asst(3) + User(4) 寫入 cache

你完全不用動任何 `cache_control` 標記，它就是自動的。

<ClawdNote>
這對做 chatbot 跟 agent 產品的人來說，根本是從「手排車」升級到「自排車」的感覺。SP-73 那篇 Thariq 提到 Claude Code 團隊要花大量精力維護 cache 前綴排列 — 現在 Automatic Caching 讓這件事的門檻低到「會呼吸就會用」的程度。當然，如果你是那種手排車開得比自排還快的老司機，explicit breakpoints 還是給你用的 ╰(°▽°)╯
</ClawdNote>

### 可以混搭

Automatic caching 跟 explicit breakpoints 可以同時用。比如你想讓 system prompt 獨立 cache（因為它幾乎不變），同時讓對話部分自動 cache：

```yaml
model: claude-opus-4-6
max_tokens: 1024
cache_control:
  type: ephemeral    # 自動 cache 對話
system:
  - type: text
    text: 你是一個文學分析助手。
    cache_control:
      type: ephemeral  # 獨立 cache system prompt
messages:
  - role: user
    content: 《傲慢與偏見》的主要 themes 是什麼？
```

但有一個小陷阱：automatic caching 會佔用 4 個 breakpoint slot 中的 1 個。所以如果你已經手動放了 4 個 explicit breakpoints，再加 automatic 會收到 400 error。這就像停車場只有 4 格，你已經停滿了，不能再開一台進去。

## 1 小時 Cache TTL — 花點小錢買個保險

預設的 cache lifetime 是 **5 分鐘**。每次 cache hit 都會自動刷新這個計時器（免費的），但如果 5 分鐘內沒有任何請求命中這個 cache，它就消失了。

五分鐘。

你去上個廁所回來、泡個咖啡、回個 Slack 訊息 — cache 就沒了。對人類來說五分鐘是眨眼之間，但對 API 帳單來說，五分鐘就是 cache write 跟 cache hit 的分界線。

<ClawdNote>
5 分鐘 TTL 就像便利商店的「限時優惠」— 結帳後 5 分鐘內回來買第二件半價，超過就恢復原價。聽起來很合理，但你有沒有想過，你的 agent 在「思考」的時候並不會送 API request？所以它思考 6 分鐘，cache 就過期了。恭喜，你剛剛的優惠券作廢了 ┐(￣ヘ￣)┌
</ClawdNote>

現在你可以選擇 **1 小時 TTL**：

```yaml
cache_control:
  type: ephemeral
  ttl: 1h
```

代價是 cache write 的價格從 **1.25x** 變成 **2x** base input price。

用 Opus 4.6 舉例（每百萬 token）：

- Base input：$5
- 5 分鐘 cache write：$6.25（1.25x）
- **1 小時 cache write：$10（2x）**
- Cache hit（讀取）：$0.50（0.1x）— 兩種 TTL 一樣

多付 60% 的 write 成本，換來 12 倍的 TTL。如果你的 agent 會在一小時內反覆讀取同一段 prompt，這筆交易划算得不像話。

## Cache Invalidation Hierarchy — 什麼改動會炸掉什麼

好，這段是整篇最重要的部分，但也是最少人認真看的部分。Phil Karlton 說過「電腦科學只有兩個難題：cache invalidation 跟命名」— prompt caching 讓你同時體驗兩個。

Cache 的前綴是按照 **tools → system → messages** 的順序建立的。你可以把它想成一棟大樓的結構：tools 是地基，system 是樑柱，messages 是每層樓的隔間。地基動了，整棟樓都要重蓋。

<ClawdNote>
或者更接地氣的比喻：這就像疊積木。你從底部抽掉一塊，上面全部塌。但如果你只動最上面那塊，下面的完全不受影響。所以最穩定的東西（tools）放最底下，最常變動的（messages）放最上面。很直覺，但很多人在設計 API 呼叫的時候根本沒想過這件事 (⌐■_■)
</ClawdNote>

**改 Tool definitions（名稱、描述、參數）：**
- ❌ Tools cache
- ❌ System cache
- ❌ Messages cache
- 全部重算，最慘的情況 — 地基動了，整棟樓重蓋

**開關 Web Search / Citations：**
- ✅ Tools cache 不受影響
- ❌ System cache
- ❌ Messages cache
- 原因：這些功能的開關實際上會修改 system prompt（Anthropic 在底層偷偷注入 search instructions）

**改 Tool Choice 參數：**
- ✅ Tools cache 不受影響
- ✅ System cache 不受影響
- ❌ Messages cache

**改 Extended Thinking 設定（開關、budget）：**
- ✅ Tools cache 不受影響
- ✅ System cache 不受影響
- ❌ Messages cache

<ClawdNote>
注意一個超反直覺的點：開關 web search 會炸掉 system cache。為什麼？因為 Anthropic 在底層會把 search 相關的 instructions 偷偷塞進你的 system prompt。所以表面上你只是開了個 toggle，底層其實是在改 system prompt — cache 當然要重算。SP-73 那篇 Thariq 說「tools 不能加不能刪」就是同一個道理。你以為你在調旋鈕，其實你在拆牆 (╯°□°)╯
</ClawdNote>

## 20-Block Lookback Window — 長對話的隱藏地雷

當你設定了一個 explicit cache breakpoint，系統會從那個點**往回看最多 20 個 block**，逐一檢查每個 block 的 hash 是否匹配。

短對話裡完全不是問題。但想像一個 30 個 content block 的長對話，你只在第 30 個 block 設了 `cache_control`：

- **前面都沒改，送第 31 個 block**：系統檢查 block 30 → 命中！只處理 block 31。完美
- **改了 block 25，送 block 31**：系統從 30 往回查 → 29 → 28... → 25（不匹配）→ 24（匹配！）。從 block 24 之後重新處理。還行
- **改了 block 5，送 block 31**：系統從 30 往回查 → 29 → 28... → 11（第 20 次檢查）。**到此為止，不再往前查了。** Block 5 的修改超出了 20-block window，結果是 **整個 prompt 都要重算**

這就像你在圖書館找書，但圖書館員只願意往前翻 20 頁目錄。超過 20 頁？「不好意思，全部重新登記。」

<ClawdNote>
20-block lookback 是那種「你不知道的時候覺得不存在，知道之後會回頭驚出一身冷汗」的規則。如果你的 agent 跑長對話（coding agent 很常見），又沒有在中間放 explicit breakpoints，恭喜你的 cache hit rate 可能比你 dashboard 上看到的低得多。解法很簡單：在可能被改動的內容之前多放幾個 breakpoint 就好。4 個 slot，省著點用但別捨不得用 ヽ(°〇°)ﾉ
</ClawdNote>

### 解法

在可能被編輯的內容**之前**放一個 explicit breakpoint。這樣系統在碰到 lookback window 盡頭時，會跳到下一個 explicit breakpoint 繼續檢查，而不是放棄。

你最多可以放 4 個 breakpoints，根據內容的變動頻率來分段：

- Breakpoint 1：Tools（幾乎不變）
- Breakpoint 2：System prompt（偶爾改）
- Breakpoint 3：歷史對話中段（可能被 compaction 修改）
- Breakpoint 4：最新的 messages 或 automatic caching

## Pricing 與最低門檻

### 不同 Model 的 Cache 定價倍率

所有 model 都遵循同樣的倍率結構，差別只在 base input price：

- **Cache write（5 分鐘）**：base input price × 1.25
- **Cache write（1 小時）**：base input price × 2.0
- **Cache hit（讀取）**：base input price × 0.1
- **Output tokens**：不受 caching 影響

以 Opus 4.6 為例（base input $5/MTok）：寫入 $6.25，1 小時寫入 $10，讀取 **$0.50**。讀取價格是正常 input 的十分之一。十分之一。這就是為什麼大家說 prompt caching 是「窮人的 fine-tuning」— 用十分之一的價格得到幾乎一樣的速度提升。

### 最低 Cacheable Token 門檻

不是所有 prompt 都能 cache。每個 model 有最低門檻，低於門檻的 prompt 會被靜靜地忽略 — 沒有 error，沒有 warning，就是默默不 cache：

- **4096 tokens**：Opus 4.6、Opus 4.5、Haiku 4.5
- **2048 tokens**：Sonnet 4.6、Haiku 3.5
- **1024 tokens**：Sonnet 4.5、Opus 4.1、Opus 4、Sonnet 4

<ClawdNote>
Opus 4.6 的門檻是 4096，Sonnet 4.5 的門檻是 1024 — 整整差 4 倍。這意味著什麼？如果你從 Sonnet 升級到 Opus（因為你需要更強的推理能力），原本穩穩 cache hit 的短 prompt 可能突然全部 cache miss。而且你完全不會收到任何錯誤訊息。帳單月底才到，驚喜才來。這就像你從小車換成大車，結果發現很多小巷子開不進去了，但 GPS 不會跟你說 — 它只會默默幫你繞路，然後月底油費帳單讓你懷疑人生 (๑•̀ㅂ•́)و✧
</ClawdNote>

### 什麼可以 Cache，什麼不行

**可以 cache 的：**
- Tool definitions
- System messages
- Text messages（user 和 assistant turns）
- Images & Documents
- Tool use 和 tool results

**不能直接 cache 的：**
- Thinking blocks（不能直接標 `cache_control`，但在 assistant turns 中會跟著一起被 cache）
- Citations 的子層級 block（要 cache 上層的 document block）
- 空的 text blocks

## 實戰教訓 — $13.86 的一堂課

好，前面講了那麼多理論，現在來講我們自己交的學費。

### 帳單爆炸事件

2026 年 3 月 7 日。一個平靜的早晨。打開 Anthropic dashboard — $13.86。

正常的一天大約 $4-5。三倍。一夜之間。

追查之後發現：71.8% 是 cache write。也就是說，我們的 agent 一直在重新寫入 cache，而不是讀取已有的 cache。

元兇？我們升級了 OpenClaw 的底層 library（pi-ai），新版本（2.1.70.3cc）在自動 migration 的時候，偷偷把 `cacheRetention` 從 `"long"`（1 小時）改成了 `"short"`（5 分鐘）。

為什麼會這樣？因為我們的 config 裡沒有顯式設定這個欄位 — 我們依賴的是舊版本的 implicit default。新版本一看：「欸，這個欄位沒有值，我來幫你設一個吧。」然後它好心地設成了 `"short"`。

<ClawdNote>
「Implicit defaults 是定時炸彈」— 這句話我要裱框掛在牆上。你覺得理所當然的 default，在 library 作者眼裡可能只是一個隨時會改的實作細節。今天的 default 是 long，明天搞不好就是 short，後天可能整個欄位都改名了。你猜怎麼知道的？沒錯，就是帳單告訴你的 (◉д◉)
</ClawdNote>

結果就是 cache TTL 從 1 小時掉到 5 分鐘。我們的 agent 在處理長任務時，思考間隔經常超過 5 分鐘 — cache 一直過期、一直重新 write、帳單一直噴。

### 修復

一行 config：

```yaml
# 在 config 裡顯式設定，不再依賴 library default
agents:
  defaults:
    models:
      cacheRetention: long
```

隔天費用：$4.52。省了 67%。

一行 config，省了三分之二的帳單。這大概是我寫過 ROI 最高的一行 code。

## 回到那個帳單

文章開頭我說，三月七號早上打開 dashboard 被 $13.86 嚇到。

現在你知道為什麼了：一個 library 更新偷改了 TTL default、5 分鐘的 cache 不夠 agent 的思考間隔用、cache 反覆過期反覆寫入。三個獨立的知識點（TTL 機制、implicit defaults 風險、agent 行為模式）交叉在一起，炸出了一張三倍帳單。

Prompt caching 在 2026 年已經不是「進階優化」了 — 它是 agent 產品能不能商業化運作的基礎建設。Claude Code 團隊會對 cache hit rate 發 SEV，這不是偶然。不是因為他們特別在意省錢，而是因為 cache miss 代表的不只是多花錢 — 它代表延遲變高、throughput 變低、用戶體驗變差。

那張 $13.86 的帳單，是最好的教材 (￣▽￣)／
