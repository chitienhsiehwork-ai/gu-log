## Automatic Caching — 懶人的福音

以前要用 prompt caching，你得在每個想 cache 的 content block 上手動加 `cache_control`。像這樣：

```yaml
system:
  - type: text
    text: 你是一個文學分析助手...（一大段 system prompt）
    cache_control:
      type: ephemeral
```

這很麻煩，尤其是在 multi-turn 對話裡 — 每次新增訊息，你都得決定要把 `cache_control` 放在哪裡。

現在 Anthropic 推出了 **Automatic Caching**：你只要在 request 的最上層加一個 `cache_control`，系統會自動把 cache breakpoint 放在最後一個可 cache 的 block 上。

```yaml
model: claude-opus-4-6
max_tokens: 1024
cache_control:
  type: ephemeral
system: 你是一個會記住對話的助手。
messages:
  - role: user
    content: 我叫 Alex，我做 ML 的。
  - role: assistant
    content: 嗨 Alex！今天想聊什麼 ML 的話題？
  - role: user
    content: 我剛剛說我做什麼的？
```

就這樣。不用在每個 block 上標記，系統自己搞定。

### Multi-turn 對話的自動前移

Automatic caching 最聰明的地方是 **cache breakpoint 會隨著對話自動往前移**：

- **第 1 次請求**：System + User(1) + Asst(1) + User(2) ← cache 寫入點。全部寫入 cache
- **第 2 次請求**：System 到 User(2) 從 cache 讀取；Asst(2) + User(3) 寫入 cache
- **第 3 次請求**：System 到 User(3) 從 cache 讀取；Asst(3) + User(4) 寫入 cache

你完全不用動任何 `cache_control` 標記，它就是自動的。

<ClawdNote>
這對所有做 chatbot / agent 產品的人來說是巨大的 DX 改善。以前我們的 SP-73 那篇提到，Claude Code 團隊要花大量精力確保 cache 前綴的排列順序完美 — 現在 automatic caching 把這件事的門檻大幅降低了。當然，如果你要極致優化，explicit breakpoints 還是更精確。
</ClawdNote>

### 可以混搭

Automatic caching 跟 explicit breakpoints 可以同時用。比如你想讓 system prompt 獨立 cache（因為它幾乎不變），同時讓對話部分自動 cache：

```yaml
model: claude-opus-4-6
max_tokens: 1024
cache_control:
  type: ephemeral    # 自動 cache 對話
system:
  - type: text
    text: 你是一個文學分析助手。
    cache_control:
      type: ephemeral  # 獨立 cache system prompt
messages:
  - role: user
    content: 《傲慢與偏見》的主要 themes 是什麼？
```

注意：automatic caching 會佔用 4 個 breakpoint slot 中的 1 個。所以如果你已經手動放了 4 個 explicit breakpoints，再加 automatic 會收到 400 error。

## 1 小時 Cache TTL — 花點錢讓 Cache 活久一點

預設的 cache lifetime 是 **5 分鐘**。每次 cache hit 都會自動刷新這個計時器（免費），但如果 5 分鐘內沒有任何請求命中這個 cache，它就消失了。

現在你可以選擇 **1 小時 TTL**：

```yaml
cache_control:
  type: ephemeral
  ttl: 1h
```

代價是 cache write 的價格從 **1.25x** 變成 **2x** base input price。

用 Opus 4.6 舉例（每百萬 token）：

- Base input：$5
- 5 分鐘 cache write：$6.25（1.25x）
- **1 小時 cache write：$10（2x）**
- Cache hit（讀取）：$0.50（0.1x）— 兩種 TTL 一樣

<ClawdNote>
什麼時候該用 1 小時 TTL？當你的使用模式是「寫一次，接下來一小時內反覆讀取很多次」的時候。比如一個 coding agent 在處理一個大任務，期間可能有幾分鐘的思考間隔 — 5 分鐘 TTL 可能會在間隔中過期，導致重新 cache write。1 小時 TTL 多付一點 write 成本，但省下了反覆 write 的錢。這正是我們踩過的坑（後面會講）。
</ClawdNote>

## Cache Invalidation Hierarchy — 什麼改動會炸掉什麼

這是整篇最重要但最少人注意的部分。

Cache 的前綴是按照 **tools → system → messages** 的順序建立的。這個順序形成一個階層，改動上層會連帶炸掉下層的 cache：

**改 Tool definitions（名稱、描述、參數）：**
- ❌ Tools cache
- ❌ System cache
- ❌ Messages cache
- 影響：全部重算，最慘的情況

**開關 Web Search / Citations：**
- ✅ Tools cache 不受影響
- ❌ System cache
- ❌ Messages cache
- 原因：這些功能的開關實際上會修改 system prompt

**改 Tool Choice 參數：**
- ✅ Tools cache 不受影響
- ✅ System cache 不受影響
- ❌ Messages cache

**改 Extended Thinking 設定（開關、budget）：**
- ✅ Tools cache 不受影響
- ✅ System cache 不受影響
- ❌ Messages cache

<ClawdNote>
注意一個反直覺的點：**開關 web search 會炸掉 system cache**，因為 Anthropic 在底層會把 search 相關的 instructions 注入 system prompt。所以如果你的 agent 有時開 search 有時不開，cache hit rate 會比你想像的低很多。SP-73 那篇 Thariq 說的「tools 不能加不能刪」就是這個道理的延伸。
</ClawdNote>

## 20-Block Lookback Window — 大型對話的隱藏陷阱

當你設定了一個 explicit cache breakpoint，系統會從那個點**往回看最多 20 個 block**，逐一檢查每個 block 的 hash 是否匹配。

這在短對話裡完全不是問題。但想像一個 30 個 content block 的長對話，你只在第 30 個 block 設了 `cache_control`：

- **你送第 31 個 block，前面都沒改**：系統檢查 block 30 → 命中！只處理 block 31。完美
- **你改了 block 25，送 block 31**：系統從 30 往回查 → 29 → 28... → 25（不匹配）→ 24（匹配！）。從 block 24 之後重新處理。還行
- **你改了 block 5，送 block 31**：系統從 30 往回查 → 29 → 28... → 11（第 20 次檢查）。**到此為止，不再往前查了。** Block 5 的修改超出了 20-block window，結果是 **整個 prompt 都要重算** (◉д◉)

### 解法

在可能被編輯的內容**之前**放一個 explicit breakpoint。這樣系統在碰到 lookback window 盡頭時，會跳到下一個 explicit breakpoint 繼續檢查，而不是放棄。

你最多可以放 4 個 breakpoints，根據內容的變動頻率來分段：

- Breakpoint 1：Tools（幾乎不變）
- Breakpoint 2：System prompt（偶爾改）
- Breakpoint 3：歷史對話中段（可能被 compaction 修改）
- Breakpoint 4：最新的 messages 或 automatic caching

## Pricing 與最低門檻

### 不同 Model 的 Cache 定價倍率

所有 model 都遵循同樣的倍率：

- **Cache write（5 分鐘）**：base input price × 1.25
- **Cache write（1 小時）**：base input price × 2.0
- **Cache hit（讀取）**：base input price × 0.1
- **Output tokens**：不受 caching 影響

以 Opus 4.6 為例（base input $5/MTok）：寫入 $6.25，1 小時寫入 $10，讀取 **$0.50**。讀取價格是正常 input 的十分之一 — 這就是 prompt caching 省錢的關鍵。

### 最低 Cacheable Token 門檻

不是所有 prompt 都能 cache。每個 model 有最低門檻：

- **4096 tokens**：Opus 4.6、Opus 4.5、Haiku 4.5
- **2048 tokens**：Sonnet 4.6、Haiku 3.5
- **1024 tokens**：Sonnet 4.5、Opus 4.1、Opus 4、Sonnet 4

低於門檻的 prompt 即使標了 `cache_control` 也不會被 cache — 但不會報錯，只是靜靜地不 cache。你可以從 response 的 usage 欄位確認是否真的命中了 cache。

<ClawdNote>
Opus 4.6 的門檻從 Sonnet 4.5 的 1024 跳到 4096 — 整整 4 倍。如果你從 Sonnet 升級到 Opus，原本穩穩 cache hit 的短 prompt 可能突然全部 cache miss，而你完全不會收到任何錯誤訊息。這種 silent failure 是帳單暴漲的經典原因。
</ClawdNote>

### 什麼可以 Cache，什麼不行

**可以 cache 的：**
- Tool definitions
- System messages
- Text messages（user 和 assistant turns）
- Images & Documents
- Tool use 和 tool results

**不能直接 cache 的：**
- Thinking blocks（不能直接標 `cache_control`，但出現在 assistant turns 中時可以跟著一起被 cache）
- Citations 的子層級 block（改 cache 上層的 document block）
- 空的 text blocks

## 實戰教訓 — 我們踩過的坑

### 帳單爆炸事件：一天 $13.86

2026 年 3 月 7 日，我們在升級 OpenClaw 的底層 library（pi-ai）之後，突然發現當天的 API 費用飆到 $13.86 — 其中 **71.8% 是 cache write**。

發生了什麼事？

pi-ai 的新版本（2.1.70.3cc）在自動 migration 的時候，偷偷把 `cacheRetention` 從 `"long"`（1 小時）改成了 `"short"`（5 分鐘）。因為我們的 config 裡 **沒有顯式設定** 這個欄位 — 我們依賴的是舊版本的 implicit default。

新版本一看：欸，這個欄位沒有值，我來幫你設一個吧 — 然後設成了 `"short"`。

結果就是 cache TTL 從 1 小時掉到 5 分鐘。我們的 agent 在處理長任務時，思考間隔經常超過 5 分鐘，cache 一直過期、一直重新 write、一直被收 1.25x 的寫入費用。

### 修復

```yaml
# 在 config 裡顯式設定，不再依賴 library default
agents:
  defaults:
    models:
      cacheRetention: long
```

隔天費用降到 $4.52，省了約 67%。

### 教訓

**Critical config 一律顯式設值，永遠不要依賴 implicit defaults。** 因為你不知道哪次升級會偷偷改掉 default 值。這不只是 prompt caching 的教訓 — 任何你覺得「反正 default 就是我要的」的設定，都應該明確寫出來。

## 結語

Prompt caching 在 2026 年已經不是「進階優化」了 — 它是 **agent 產品能不能商業化運作的基礎建設**。Claude Code 團隊會對 cache hit rate 發 SEV，這不是偶然。

如果你只記住三件事：

1. **用 Automatic Caching** 降低 multi-turn 對話的 caching 複雜度
2. **搞懂 invalidation hierarchy** — 知道什麼改動會炸掉什麼，比盲目加 `cache_control` 重要
3. **顯式設定所有 cache 相關的 config** — implicit defaults 是定時炸彈
