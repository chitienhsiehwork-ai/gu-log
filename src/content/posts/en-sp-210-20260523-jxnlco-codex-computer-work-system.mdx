[Codex](/en/glossary#codex) first looks like an engineer's helper: open a repository, read files, make a diff, run tests, and open a PR.

That center of gravity is still there. But the interesting shift is not "faster code." Codex is starting to touch the rest of computer work: browsers, terminals, APIs, documents, Slack, Gmail, calendars, desktop GUIs, and scheduled state checks.

In other words, code is turning from destination into entrance. Before, you handed Codex a task and waited for a code change. Now it feels more like leaving Codex a workbench: memory, tools, open loops, half-finished artifacts, and something reviewable next to the conversation.

The real question is no longer whether Codex can write code. It is whether this workbench can carry a computer task from "someone asked for it" to "the result got reviewed."

<ClawdNote>This is like a convenience-store cashier growing from "I can ring people up" into "I can restock shelves, order inventory, check cameras, call vendors, and handle complaints." We thought the register got smarter; actually the whole store workflow started connecting. Scary, but pretty reasonable (｀・ω・´)</ClawdNote>

## First key: the desk cannot reset every day

The biggest problem with short chats is not that the model is dumb. It is that every session rebuilds the stage.

A task gets its context today, then tomorrow the same context is explained again. Last week's naming decision, a reviewer's preference, a test trap in some repo, a customer's communication habit — all of it sits in an old conversation like a whiteboard erased after the meeting.

Durable [threads](/en/glossary#thread) turn a conversation from a transcript into a workspace. The best pinned threads are usually not one-off tasks like "fix this bug." They are recurring workflows: a release thread tracks versions, tests, and docs; a documentation-review thread keeps pages aligned with product changes; a chief-of-staff thread watches messages, calendars, to-dos, and people who need replies.

The value is not remembering the previous sentence. The value is preserving decisions, preferences, constraints, and open loops. Pinning, quick switching, and returning to the same thread all point at the same product model: keep the desk around so the work does not restart from zero every morning.

<ClawdNote>Humans often write prompts while pretending they are clearer than they are, then delete the useful hesitation, anxiety, and surrounding context. Voice input does the opposite: it pours the scratchpad directly into the sink. Messy, but often closer to the truth.</ClawdNote>

Voice input belongs in this same picture. A lot of work is not a clean instruction yet when it begins: "I think someone in Slack mentioned this. I forgot the details. Please go find it." Typed out, that sounds like a bad ticket. Spoken out loud, it sounds normal.

For an [agent](/en/glossary#agent) that can search, gather context, and report back, "I don't remember the details; please go look" is already enough to begin. Roughness is not always a flaw. Sometimes roughness is the real shape of the task.

## Second key: humans can leave the desk, not the loop

Once a long task starts running, control splits into two forms.

The first is steering: the direction is wrong now, so change it now. During a website review, the side panel can show the rendered surface while the user annotates: make this smaller, the spacing feels off, this copy is wrong. The task does not need to finish before everyone notices the artifact drifted into another county.

The second is queuing: do not interrupt the current work, but line up the next step. After the website fix is done, send the preview link to the reviewer in Slack. After PR comments are handled, update the docs.

Together, these keep the human in the loop without making the human stand beside the model like an exam proctor until their eyes dry out. The Codex mobile app also makes more sense as a control surface: not a whole development environment squeezed into a phone, but a way to let the local machine keep running while the user checks progress, answers questions, approves the next step, or redirects the task away from the desk.

<ClawdNote>Writing an entire PR on a phone feels like carving a chip with a toothpick on the subway. Admirable spirit; the fingers will resign first.</ClawdNote>

It used to be: sit at the computer until the task finishes. Now it is closer to: let it run, then intervene when a decision is needed.

## Third key: the tool radius should grow without becoming clutter

Once the desk stays around, the next question is what Codex can touch.

Do not start by memorizing tool names. Start with the radius: the inner ring is the rendered web page, where Codex can operate the surface and respond to annotations; the next ring is the signed-in browser, where internal tools and SaaS admin screens become reachable; the outer ring is the whole desktop, where old workflows with no API, no CLI, only buttons and windows, do not have to stay human-only forever.

[MCP](/en/glossary#mcp) servers and connectors are easiest to read as safe sockets that let Codex plug into external tools. Slack, inboxes, and calendars matter because many real tasks do not start as code. They start as messages, email, schedule conflicts, or someone dropping a sentence like "can you look at this?"

When a workflow repeats, it can become a [Skill](/en/glossary#skill). A Skill is not magic; it is closer to a standard operating card. Do not teach the [agent](/en/glossary#agent) the same routine every time. Write down the working procedure so Codex can run it again.

<ClawdNote>This is also where overengineering gets tempting. A weird process that happens once every three weeks does not need a skill. A daily process that always loses one step does. Buying a library barcode scanner for twenty books is how the scanner breaks before the books become searchable.</ClawdNote>

Automation adds a heartbeat to that operating card. Thread automation wakes the same thread every few minutes or hours, returns to its existing context, and checks state: if the condition is not ready, wait; if it is ready, move to the next step.

A chief-of-staff-style thread could periodically check messages and inboxes, find items that need attention, research a reply, and draft it without sending. PR comments, document comments, and Slack replies can become signals too: see feedback, update the artifact, bring it back for review. This used to be called human coordination. Now it starts to look like a workflow that can be described, rerun, and checked.

## Fourth key: a goal without a verifier is just a wish

The point of Goals is not to make tasks sound more ambitious. It is to give Codex a real finish line: something it can keep pushing toward, plus a signal that says the work is complete.

"Implement this Markdown plan" sounds clear, but it is like telling a delivery driver to bring food "somewhere nearby." A strong goal has a verifiable stopping condition. If an engineer migrates an internal tool from Python to Rust, the finish line is not "try to migrate it." It is: the new implementation is done when the unit tests pass. The verifier might be a test suite, benchmark, bug reproduction, validation matrix, or an end-to-end workflow that must keep passing.

Ambition without a verifier is just a wish. Harsh, but useful.

<ClawdNote>Goals are easy to overframe as the protagonist of the system. A better reading: the goal is the finish line. The work actually moves because threads, tools, schedules, review surfaces, and verifiers are connected. A long task without a verifier is like a gym poster that says "get stronger." Motivational, but the muscle does not appear.</ClawdNote>

The side panel keeps verifiers and artifacts inside the loop. A document, deck, webpage, table, or PDF that leaves the conversation quickly turns into another tab, another tool, and another review thread. The point of the Codex side panel is not "one more UI area." It is keeping discussion, instructions, artifacts, and corrections close enough to connect.

The in-app browser makes this concrete. The web page can be both output and control surface. Codex can generate a page, open it, inspect the rendered result, see what broke, and continue fixing it. That is a good fit for `index.html` static pages, UI review surfaces, programmatic animation tools, browser decks, and data apps: anything where the problem only appears once the artifact is visible.

<ClawdNote>I read the side panel as part of the control loop, not just a small UI feature. When the artifact leaves the thread, the next step becomes "please open another ticket." When the artifact stays beside the thread, the correction can plug straight back in. Boring? Yes. Also exactly where product work usually leaks.</ClawdNote>

## Fifth key: important context should be inspectable

Long-running threads are useful, but a thread should not be the final home for every memory.

A more durable pattern is to write reviewable, movable, versionable context into external memory. This is close to [SP-200 on Markdown / AGENTS.md memory](/en/posts/en-sp-200-20260514-mem0ai-codex-cli-grep-markdown/): a plain folder of text files, stored in Git, Dropbox, Google Drive, or the team's usual sync layer. An [Obsidian vault](/en/glossary#obsidian-vault) sounds fancy, but the basic idea is just a note warehouse that is easy to move, search, and version.

The structure can stay simple. No need to turn it into a knowledge-management religion. A plain `vault/` with `TODO.md`, `people/`, `projects/`, `agent/`, and `notes/` is enough. The important part is using `AGENTS.md` to tell Codex: treat `~/vault` as durable work memory, prefer canonical notes over note sprawl, preserve decisions, blockers, owners, dates, and useful links, and do not churn the folder when nothing meaningful changed.

Repositories preserve code. The memory folder preserves rolling context: who is involved, what changed, what is blocked, who owns the follow-up, and what the next thread must not ask again.

First-party Codex memory features are better suited to preferences, repeated workflows, and known pitfalls. Plain-text memory is better for work facts the team wants to inspect, move, and version. One is more like a habit in the head; the other is more like a folder in the cabinet. There is no need to turn this into a team sport between memory systems.

<ClawdNote>My own bias is conservative here: first-party memory is useful, but important team context should still have a plain-text version somewhere. Folders, Markdown, Git — none of this is sexy, but it still opens five years later. Many AI memory systems are scary not because they forget, but because they remember things nobody can audit. Inspectable beats mystical.</ClawdNote>

## Closing

Codex still begins from code, but it is connecting more of the work around code: messages, browsers, desktop surfaces, artifacts, review, scheduling, automation, and shared memory.

If each feature is read alone, this sounds like a product tour: threads, voice, queues, MCP, Skills, automation, side panel, memory. The real signal is that they are starting to form one control loop: the desk stays around, the tool radius expands, artifacts remain close enough to review, and important context moves into memory that can be checked.

Code used to look like the agent's destination. Now it looks more like a door. Behind that door is not another editor, but a computer-work loop that runs from instruction to execution to artifact review.
