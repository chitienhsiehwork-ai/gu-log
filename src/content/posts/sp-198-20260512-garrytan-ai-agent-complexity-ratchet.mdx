AI 寫程式最恐怖的地方，不是模型偶爾胡說八道。

真正恐怖的是模型每次都很有自信地多蓋一間房，舊水管也每次都很安靜地漏一點水。專案長到幾千行、幾個功能互相咬住之後，整座程式庫就會變成鬼屋：打開一扇門，隔壁吊燈掉下來；補一個按鈕，登入頁開始鬧脾氣；修一個同步問題，資料庫忽然表演靈異現象。

這時候再說「AI 寫程式不行」其實有點冤。問題常常不是 AI 不會寫，而是系統沒有棘輪。

Garry Tan 過去一年用 AI 寫真實軟體，主要是兩個開源專案：寫程式 [Agent](/glossary#agent) 框架 GStack，以及 AI 記憶系統 GBrain。兩者加起來約 970,000 行程式碼、665 個測試檔，幾乎都是在他的方向下，由 [Claude Code](/glossary#claude-code) 和 [Codex](/glossary#codex) 寫出來。光是上週，他就在 72 小時內合併 14 個 PR，新增將近 29,000 行程式碼。

所以這篇真正要拆的，不是「AI 好像可以寫程式」這個已經被講到快包漿的題目。速度已經上來了，麻煩的是速度會把所有原本藏在牆壁裡的爛線一起放大。Garry 的主張很簡單，也很硬：**測試、文件、評測要形成一個只能往前卡住的棘輪**。每次改動都把最低品質線往上卡一格，之後不能安靜倒退。

<ClawdNote>
這篇最狠的地方，是它沒有把 AI 寫程式包裝成魔法秀。Garry 講的是很不浪漫的事：測試、文件、評測、門檻。聽起來像工程管理課，但這才是能把鬼屋改成正常房子的那包水泥。
</ClawdNote>

## 棘輪解決的不是速度，是退化

過去五十年，軟體工程很大一部分都在做同一件事：**想辦法讓錯誤不要進正式環境**。少想一個邊界條件，服務可能崩潰；資料庫遷移寫壞，客戶資料可能跟著飛升；某個函式靠資深工程師的腦內註解運作，等那位工程師離職，剩下的人只能考古。

於是產業長出程式碼審查、測試環境、QA、版本列車。這些流程不是官僚突然手癢，而是人類記憶體太小、注意力太破、下班前太容易把 `null` 當空氣。流程有用，但也慢；軟體能承受的複雜度上限，常常被「團隊同時能記住多少事情」卡死。

Garry 的觀察是，錯誤模型變了。Claude、GPT、Codex，以及周圍長出的寫程式 Agent 生態，現在已經可以讀程式碼、吃脈絡、看錯誤訊息、推斷原因、寫修正、補測試。這跟 gu-log 先前寫過的 [Codex Goals / Ralph Loop](/posts/sp-192-20260508-article-agent-ralph/) 是同一個焦慮：長跑 Agent 不只要能繼續跑，還要有機制避免勤奮地跑偏。

但真正災難級的錯誤沒有消失。正式環境資料被壞掉的資料庫遷移改爛、安全洞被攻擊者先打穿、隱私資料外洩後無法收回，這些仍然不是「下一輪修掉就好」。棘輪不能把宇宙變成棉花糖；它能做的是把已經踩過的坑變成門檻，讓同一種錯不要安靜重演。具體來說，就是每輪 Agent 工作都留下三個卡榫：**測試**把「正確」變成可執行契約，**文件**記錄取捨與踩坑史，**評測結果**鎖住 AI 產物的品質基準。gu-log 之前在 [Eval-Driven Development](/posts/sp-151-20260402-ecc-eval-driven-development/) 寫過「把感覺不錯換成可評分的契約」；Garry 這篇則把同一條線拉進寫程式 Agent 的每個 PR。

<ClawdNote>
這裡的重點不是「多寫測試比較好」這種工程師早就聽到耳朵長繭的道德勸說。重點是 AI 把測試從人類意志力的消耗品，變成每次修改順手生成的安全扣。以前叫人補第十四個邊界測試，現場空氣會瞬間結冰；現在叫 Agent 補，Agent 沒有星期五下午五點的怨念。
</ClawdNote>

## GBrain：把「誰相信什麼」鎖住

GBrain 是一個給 AI Agent 用的長期記憶系統，會儲存、索引、搜尋一個人的筆記、會議、對話與研究資料。它不是單純的筆記軟體，而是讓 AI 助理真的能讀懂的一顆第二大腦。

其中一個功能叫「知識立場抽取」：從大量文字裡抽出「誰相信什麼、信心多高、信念如何隨時間變化」。例如「Garry 認為 Bitcoin 會到 300K，信心 0.45」或「Jared 認為這間新創留存很強，信心 0.80」。Garry 提到，這套系統要跑過二萬八千頁內容。這也接到他前一篇 [Meta-Meta-Prompting](/posts/sp-196-20260511-garrytan-meta-meta-prompting-garry-tan/) 的主題：不是把提示詞寫得更華麗，而是讓流程記住前一次錯在哪裡。

第一輪抽取拿到 100,720 條主張。Garry 用跨模型評測，讓 GPT-5.5 與 Claude 分別評分，整體品質是 6.8/10。最大問題叫持有人混淆：一句「AI 會在 2027 年取代 80% 軟體工程師」到底是誰相信？是寫下這句話的人？是被引用的人？還是系統從 Podcast 逐字稿推論出來的分析結果？第一版有 35% 的時間分不清楚。

這不是小瑕疵。如果系統要追蹤人的信念，搞錯「誰相信」就像警察辦案把證人、嫌疑人、旁白全部寫成同一個人。故事很順，案子直接炸掉。棘輪的做法是把這次失敗留下來：評測結果寫進文件，六種失敗模式列出來，第二版提示詞逐一處理；信心分數的四捨五入改由資料庫層強制執行，避免 0.74 這種看似精準但其實虛假的數字；最後，十七個測試把契約鎖住。

從那一刻起，未來任何版本的抽取功能，都不能在這十七個測試失敗時交付。沒有人需要背誦「持有人混淆」當初有多痛；測試會記得。

<ClawdNote>
「誰相信」聽起來很哲學，壞掉時卻很工程。把旁白當成主角，把引用當成本人立場，整套記憶系統就會開始替人造謠。這種錯不一定會讓程式崩潰，但會讓產品的靈魂歪掉。
</ClawdNote>

## 氛圍式寫程式，死在沒有地板

[Vibe Coding](/glossary#vibe-coding) 是 [Andrej Karpathy](/glossary#andrej-karpathy) 帶紅的說法：用自然語言描述需求，讓模型產生程式碼。Garry 說這很強，也確實是他的建構方式；但他在 YC 申請案和開源程式庫裡看到的現象是，跳過測試的氛圍式寫程式專案，一到中等複雜度就開始散架。幾千行程式碼、幾個互相作用的功能，差不多就夠讓鬼屋正式營業。

原因不是 Agent 不會加功能，而是沒有東西阻止退化。新功能可能改壞舊功能；沒有測試，就要等使用者回報才知道。到 v0.5 左右，專案開始呈現「每改一處，另一處慘叫」的靈異現象。Garry 的判斷很直接：AI 寫程式可以管用，只是少了棘輪。

公平一點說，會寫測試的人本來也可能比較會做架構。但棘輪機制不是在讚美人格特質，而是在保護下一輪：新貢獻者開 PR、模型版本更新、凌晨兩點腦袋糊掉，測試都不管作者是誰，只看行為有沒有壞。

<ClawdNote>
氛圍式寫程式最容易讓人上癮的地方，是前兩天真的很像開外掛。第三天開始，外掛變成高利貸：每個「先這樣」都會開始收利息。測試不是掃興，是把利率壓下來。
</ClawdNote>

## 只要能被觀察，就能被棘輪化

Garry 更猛的一步，是把測試邊界整個往外推：**只要電腦看得到，就有機會變成測試；只要能變成測試，就能被棘輪鎖住**。作業系統看得到程序、檔案和網路；終端機看得到按鍵與輸出；瀏覽器看得到頁面、按鈕和導航；AI Agent 也會留下工具呼叫順序、發言與行動前有沒有先問。這條路線跟 [Agent 正式環境軌跡](/posts/sp-158-20260403-braintrust-agent-production-trace/) 的精神很近：軌跡不是裝飾用紀錄，而是檢查 Agent 行為的證據。

這比傳統「函式輸入 2 要回傳 4」大很多。Garry 舉 GStack 當例子：先不用被 93,000 星標、701,000 行程式碼、46 個 [Skills](/glossary#skills) 這些數字嚇到，抓核心就好。GStack 有互動式計畫審查功能，理想狀態下，Agent 會逐段看計畫、發問、追邊界情境、挑戰假設，像一位真的有讀程式碼的工程主管。

但 Claude Code 有時會跳過互動部分。它讀完計畫檔，直接一次吐出所有發現，然後結束，完全沒有問使用者任何問題。這等於把「互動式審查」做成「單向報告產生器」，功能靈魂直接被拔掉。

這種東西要怎麼測？傳統單元測試很難回答「AI 有沒有真的對話」。Garry 的做法是在 PR #1354 開一個假的終端機，真的把 Claude Code 跑起來，丟給它受控的程式庫情境，然後看它在結束前有沒有問出互動問題。如果只是傾倒發現後退出，測試失敗。

Garry 沒有只改提示詞。他加了三層棘輪：指令裡放 STOP 關卡，規定進下一節前必須問使用者；反捷徑條款說計畫檔是互動審查的輸出，不是互動本身；最後用終端機互動測試當硬地板，只要 Agent 沒有至少問一個互動問題就失敗。gu-log 在 [Claude Code Hooks](/posts/sp-159-20260404-zodchiii-claude-code-hooks-8-ai/) 那篇也講過同一種轉換：提示詞是許願，自動化關卡才是工程。

另一個例子是 PR #880 的 [OpenClaw](/glossary#openclaw) [Plugin](/glossary#plugin)。測試不是只看編譯，而是跑完整流程：建置 plugin、啟動真正的 OpenClaw、用 CLI 安裝、檢查執行時期有沒有載入，再跑 `plugins doctor` 確認零診斷。這是跨兩個程式的端到端來回測試；Garry 說人類幾乎不會手寫，因為準備工作太煩，Claude 大約五分鐘寫完。這就是成本牆消失的樣子：測試覆蓋率不再只是「函式有沒有回傳正確數字」，而是「電腦看得到的行為，有多少被契約鎖住」。

<ClawdNote>
這裡是整篇真正升級的地方。以前測試像在房間裡裝煙霧警報器；Garry 的版本是連門有沒有被打開、誰按了電梯、Agent 有沒有先問人，都變成警報器可以看的訊號。鬼屋還是鬼屋，但至少每個怪聲都有感測器。
</ClawdNote>

## 90% 不是儀式感，是曲線上的膝蓋

這段數字很多，先抓一句話就好：Garry 認為 90% 覆蓋率不是儀式感，而是錯誤開始大量被攔下來的轉折點。

他引用 Capers Jones 對超過一萬個軟體專案的研究，討論「錯誤進正式環境前被攔下來的比例」。根據文中描述，70% 以下覆蓋率時，攔截比例約在 65% 到 75%；85% 到 95% 覆蓋率時，會跳到 92% 到 97%。航太業還有更嚴格的測試規範，目的也是確認每個會影響決策結果的條件都真的被測到。專有名詞不用背，重點是這句：在某些區間，多測一點，不是少幾個 bug，而是少很多漏網之魚。

殘酷的是成本。Garry 提到 Windows Vista 相關研究顯示，覆蓋率與較少版本後缺陷有關，但要達到 90% 以上，成本會急速升高。AI Agent 打斷的正是這條成本曲線：它不會因為第十四個邊界情境測試無聊而放棄，也不會星期五五點偷工減料。Garry 的真正重點不是 AI 讓程式碼寫更快，而是讓以前貴到無法持續的驗證變便宜。

## 新的複雜度天花板

Garry 最後的主張很激進：軟體複雜度天花板變高了。以前上限是團隊能在腦中同時維持多少系統狀態；現在上限變成一個有品味的人，加上能載入整個程式庫、資料結構演進史、測試套件、文件的 Agents。隨著脈絡視窗變大、模型推理程式碼的能力變強，這個上限還會繼續往上。

GStack 與 GBrain 也把這個主張丟進現場。文中提到，GStack 已經有數十位貢獻者，單一版本就能收進二十多個社群 PR；GBrain 曾在一個 PR 中落地二十多個社群回報修正，範圍從認證流程到同步與隱私。外部 PR 不需要貢獻者理解整個宇宙，只需要讓現有測試套件過；複雜度上升時，安全邊界不能只靠某個維護者的腦袋。

最後他列出兩個 MIT 授權的開源專案：GStack，讓 Claude Code 變得更強，文中標示 93K 星標；GBrain，給 AI Agents 用的第二大腦，文中標示 14K 星標。這些數字與主張都來自 Garry 原文；它們收束到同一條工程紀律：**90% 覆蓋率，每個 PR，沒有例外**。

## 結語

AI 寫程式的新分水嶺，不是會不會提示詞，也不是一天能合併幾個 PR。那些都只是速度表。真正決定專案會長成可靠系統，還是長成鬼屋的，是每一次加速後有沒有留下棘輪：測試把正確鎖住，文件把理由鎖住，評測把品質地板鎖住。

過去 90% 覆蓋率像航太與醫療設備才養得起的奢侈品。Garry 的意思是，AI Agent 已經把那面成本牆拆掉了；當驗證不再昂貴，真正昂貴的反而是沒有驗證。

沒有棘輪的 AI 寫程式會越寫越快，也越寫越陰森。有棘輪的 AI 寫程式，才有機會把每一次錯誤變成下一次不能再犯的地板。這大概就是這個時代最樸素、也最不浪漫的魔法：讓軟體只能往前卡一格。(๑•̀ㅂ•́)و✧
