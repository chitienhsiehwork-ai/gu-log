很多人對長跑型 [Agent](/glossary#agent) 的想像都很美：丟一句目標進去，睡一覺起來，產品就自己長好了。現實比較像另一種畫面：工具終於肯一直跑下去，但跑到後面開始忘東忘西，最後交出一個「看起來很努力、方向卻歪掉」的東西。╮(╯▽╰)╭

這篇原文最值得看的地方，是它把問題講得很白。[Ralph Loop](/glossary#ralph-loop) 的確有用，因為它能讓模型不要太早停下來；但「能繼續跑」和「能一路跑對」其實是兩件事。

先把這篇最重要的問題抓出來：如果 Agent 要自己工作好幾小時，怎麼讓它不要一路跑偏？至於文中那些工具名，可以先把它們想成幫 Agent 延長續航的零件，先不用急著背名詞。

如果只記一個結論，可以記這句：長跑型 Agent 不只需要續航，還需要三個額外零件。

1. 先把需求問清楚。
2. 讓不同 Agent 分工、互相審查。
3. 把目標與進度寫到模型腦袋外面。

原文先承認了一個不太浪漫的事實：長跑有時候有效，不是因為模型突然開悟，而是因為它花了更多 [Token](/glossary#token)。白話講，就是給模型更多回答時間、更多嘗試次數、更多犯錯後修正的空間。研究圈會把這類做法叫成「擴展測試時運算」，聽起來很硬，意思其實不玄。

原文裡也引用一張測試圖表，重點不是某條神奇公式，而是很務實的那句話：回答時多給一些時間和預算，表現常常真的會變好。

但真正的麻煩在後面。任務一長，模型要記的東西會暴增：需求、檔案、前面改過什麼、哪些地方不能碰。這些內容一旦塞爆 [context window](/glossary#context-window)（可以先想成模型這一輪看得到的白板大小），光靠「再跑一輪」就開始撞牆。這也是這篇真正想講的地方：[Ralph Loop](/glossary#ralph-loop) 可以解決「太早停下來」，卻不能單獨解決「長時間維持方向」。

<ClawdNote>Ralph 迴圈比較像讓引擎不要熄火。引擎有聲音，不代表車已經開到目的地；長跑任務還需要方向、檢查點和煞車。</ClawdNote>

## Ralph 迴圈只是最小可行引擎

所謂 Ralph 迴圈，原始形式其實很陽春：把同一段 [Prompt](/glossary#prompt) 一直丟給模型，叫它繼續工作。可以先把它想成 Agent 版的鬧鐘：時間到就叫模型再檢查一次、再做下一步。

原文拿 Codex Goals 當例子，講的是同一件事：現在有些工具已經能把大型任務掛在那裡慢慢做，讓 Agent 持續跑數小時甚至數天。對常被工具每十五分鐘問一次「還要繼續嗎」的開發者來說，這確實很有吸引力。

底層細節可以很工程，但核心概念其實很簡單：系統終於不會做一半就自己停掉了。

可是原文真正失望的點也在這裡：能一直跑，不等於能長期做好。這就像把電風扇開到最大，房間真的有風，但披薩不會因此自己烤熟。動能不是結構，續航不是判斷力。

## 真正會翻車的地方

原文不是在唱衰長跑型 Agent。相反地，它非常想把這件事做成。只是作者在 Codex `/goal` 和其他實驗裡反覆撞到同一面牆：任務跑久之後，最危險的不是停下來，而是一路很勤奮地做錯。

這種錯很煩，因為它不會一開始就大爆炸。比較常見的畫面是：第一輪只是稍微誤解需求，第二輪開始沿著誤解寫程式，第三輪補測試，第四輪修文件。每一步看起來都很努力，最後卻蓋出一棟地址正確、門牌漂亮、但根本不在同一條街上的房子。

所以後面講的三個缺口，不是三個新名詞，而是三個剎車點：

1. 開工前，先把模糊需求問到不能再裝傻。
2. 執行時，不要讓同一個 Agent 一路自嗨到底。
3. 跨回合時，把目標與進度寫到外面，不要全塞進模型短期記憶。

這三件事其實都在處理同一個恐怖故事：Agent 看起來很忙，但沒有人知道它到底有沒有往對的方向忙。

## 第一個洞：模糊性會複利

LLM 在迴圈裡工作時，每一輪輸出都會變成下一輪輸入。這代表一開始的小誤會，後面會越滾越大。檔案結構選錯、完成標準理解錯、介面方向猜錯，之後的每一輪都會沿著那條錯路繼續蓋下去。

這不是 AI 特有問題，比較像把需求講得很模糊，卻期待外包團隊一次猜中。原文的做法是：不要急著讓 Agent 開工，先進入一個「設定階段」，用訪談式問答把規格逼出來。

有些人會把這件事做成訪談式技能，有些人會自己手動問答；做法可以不同，但順序不能反。先問清楚，再開工。

這一步的重點很簡單：長跑開始前，先把錯誤分支剪掉。等目標、限制、里程碑都比較清楚了，再放 Agent 自主工作，後面才不會一路偏航。

## 第二個洞：單一強 Agent 不如小隊作戰

第二個觀察也很好懂：一個 Agent 單獨埋頭做很久，很容易越做越歪。多 Agent 的協調者加子 Agent 模式，通常比「把所有希望押在一個超強 Agent 身上」更穩。白話講，就是讓多個 Agent 分頭找答案，再把結果合起來。

他的工作流大概像這樣：主 Agent 當協調者，遇到小任務就拉一組子 Agent；通常一個負責做，一個負責看。做的人交成果，看的那個用全新的脈絡檢查，兩邊修到可以接受，再回報主協調者。真正有價值的，不只是多一個角色，而是多一雙沒被前面錯誤拖下水的眼睛。

不同任務也可以配不同工具，但這裡真正重要的不是模型名單，而是先承認一件事：長跑流程本來就不該只有一個角色。寫程式、查資料、審查、收斂，硬塞給同一個 Agent 做到底，很快就會亂掉。

<ClawdNote>多 Agent 不是魔法陣，人數變多不會自動召喚品質。比較像請兩個工程師互看 PR：如果兩個都沒看規格，只會一起錯得更有自信。真正有用的是角色分工、脈絡隔離、審查回合，還有協調者知道何時該收斂。</ClawdNote>

## 第三個洞：記憶不能只放在模型腦袋裡

長跑型 Agent 還有一個更現實的問題：脈絡視窗會滿，工作會跨回合，Agent 也可能會換。只靠當前對話記住所有決策，等於把專案管理寫在便利貼上，再把便利貼貼在電風扇前面。風一吹，人生也跟著散了。

原文的做法是，在規劃階段後，要求 Agent 建立、閱讀並維護幾個檔案。先不要把這段看成內部工具清單，可以把它想成專案交接包：

1. `GOAL.md`：最高層級的目標，也就是這輪工作到底要達成什麼。
2. `STANDARDS.md`：不可妥協的程式品質標準。
3. `IMPLEMENT.md`：工作流指令，包含使用多 Agent 審查、撰寫測試、驗證成果等規則。
4. `PROGRESS.md`：持續更新的紀錄，寫下已做決策以及已完成工作。

原文前面說「三個檔案」，但實際列了四個。這裡照內容保留：重點不是數量，而是功能分工。

每個新的 Agent 都必須先讀這些檔案，快速理解前面做過哪些決策、目前進度在哪裡，再用一致方向接手。這不只幫 Agent，也幫人類知道專案到底跑到哪一步。讀者如果只想帶走一招，其實不用照抄檔名；先準備「目標、標準、做法、進度」四個位置，就已經比把全部資訊塞在聊天紀錄裡穩很多。

原文也很老實地補了一句：這些文件不是魔法。Agent 不一定每次都會乖乖照做，文件也可能過期或漏寫。但和「完全沒有外部記錄」相比，它至少讓專案有一張地圖，不必把所有方向感都塞進模型的短期記憶。

## 最後長成一套可交班的流程

把整篇文章串起來，這套長跑工作流其實很像一套縮小版工程流程。不是因為它用了很多工具，而是因為它開始像真的工作現場：有人問需求，有人做事，有人檢查，有地方寫交接紀錄。

一開始不要急著跑，先把模糊目標變成具體規格。接著讓主 Agent 協調，小 Agent 分別負責實作和審查。最後把目標、標準、做法、進度這些記錄放到模型腦袋外面，讓下一個接手的人或 Agent 有地方可看。

作者也公開了自己整理的一套長跑工作流。不過這裡不用把每個工具名都背起來。它不是魔法成品，比較像施工現場的白板，寫著誰在做什麼、做到哪裡、下一班不要踩到哪個坑。

這也是這篇最有價值的地方：它把「長跑型 Agent」從一句很帥的口號，拉回到工程現場真正會遇到的問題。重點不再是「模型可不可以跑十小時」，而是「跑到第七小時時，還有沒有東西能確認它沒跑偏」。

<ClawdNote>很多 Agent 工作流教學喜歡停在「把這段 Prompt 貼上去」。那種感覺就像教人開餐廳只說「把瓦斯打開」。瓦斯當然重要，但菜單、備料、出餐順序、誰洗鍋、客訴怎麼處理，才是餐廳能不能活過週末的關鍵。</ClawdNote>

## 結語

這篇文章最好的地方，是沒有把 Codex Goals 講成失敗品，也沒有把自己的流程包裝成銀彈。它真正指出的是一個結構問題：Ralph 迴圈讓 Agent 持續前進，但長跑任務需要的不只是前進。

長跑型 Agent 不是「一個超強模型關在房間裡把一切做完」。它更像一個會先問問題、再分工、會留紀錄、也會被審查的小型工程流程。Ralph 迴圈負責讓引擎不要熄火；真正讓車有機會到目的地的，是有人先確認地址、途中看路牌、下車前再看一次車是不是停對地方。
