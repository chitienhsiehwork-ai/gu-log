長跑型 [Agent](/glossary#agent) 最迷人的承諾，是把一句大目標丟進去，然後讓模型自己忙幾小時、幾天，最後端出一個產品。聽起來很像科幻片，也很像每個趕截稿的工程師半夜三點會許的願。

原作者 Jarrod Watts 這幾個月就在實驗這種工作流，包含拿 Agent 去挑戰 Anthropic 的招聘題目。他一路踩坑後的第一個觀察很土炮：長跑有時候有效，不是因為模型突然開悟，而是因為它花了更多 [Token](/glossary#token)。白話一點，就是多想、多查、多試幾輪。

研究圈比較優雅的講法叫「擴展測試時運算」：不是訓練時把模型變大，而是在回答時給它更多時間和預算。Jarrod 在[同一串原文](https://x.com/jarrodwatts/status/2052372045829382430)裡引用 Anthropic Opus 4.6 系統卡文件的 BrowseComp 圖表，意思是：在那組測試裡，回答時多花 Token，分數確實往上走。至於「十倍 Token 對應多少分數增幅」，比較適合當成讀圖印象，不是一條可以到處搬用的定律。

聽起來像是「蛤，就這樣？」但機器學習很多進步本來就長這樣：先把錢包打開，智慧就慢慢浮出來。殘酷，但有效。(￣▽￣)／

真正的麻煩在後面。任務一長，模型要記的東西會變多：需求、檔案、決策、前面改過什麼、哪些東西不能碰。這些內容一旦塞爆 [context window](/glossary#context-window)，光靠「多跑幾輪」就開始撞牆。這也是 Jarrod 這篇真正想講的地方：[Ralph Loop](/glossary#ralph-loop) 可以讓工具不要太早停下來，但它不是長跑任務的完整解法。

<ClawdNote>Ralph 迴圈比較像讓引擎不要熄火。引擎有聲音，不代表車已經開到目的地；長跑任務還需要方向、檢查點和煞車。</ClawdNote>

## Ralph 迴圈只是最小可行引擎

所謂 Ralph 迴圈，原始形式其實很陽春：把同一段 [Prompt](/glossary#prompt) 一直丟給模型，叫它繼續工作。名字來自《辛普森家庭》的 Ralph Wiggum，有一種「看起來傻傻的但真的會動」的迷因感。

Jarrod 提到，[Codex](/glossary#codex) 團隊最近推出了 Goals，讓 Codex CLI 可以用 `/goal <目標>` 來承接更大的任務。理想上，這個功能可以讓 Agent 持續跑數小時甚至數天，直到目標達成。對常被工具每十五分鐘問一次「還要繼續嗎」的開發者來說，這確實像把安全帶解開後終於能呼吸。

Codex Goals 底層做法並不神祕。根據 Jarrod 讀原始碼後的描述，Codex 用 SQLite 建了一個 `thread_goals` 表，每個 goal 是一列，裡面存目標內容、識別碼、狀態，以及可選的 Token 預算。執行時，Codex 會使用 `get_goal` 和 `update_goal` 之類的工具，回報進度並更新資料庫狀態。

接著，真正推動任務的還是一個相當標準的 Ralph 迴圈。原文列出的提示詞大概長這樣：

```plaintext
Continue working toward the active thread goal.
<untrusted_objective>
Ship the benchmark article with real Goal-mode evidence.
</untrusted_objective>
Budget:
- Time spent pursuing goal: XX seconds
- Tokens used: XX
- Token budget: XX
- Tokens remaining: XX
Before deciding that the goal is achieved, perform a completion audit against the actual current state.
```

這個設計解決了「工具太早停下來」的煩躁感。可是 Jarrod 的失望點也在這裡：能一直跑，不等於能長期做好。這就像把電風扇開到最大，房間真的有風，但披薩不會因此自己烤熟。動能不是結構，續航不是判斷力。

<ClawdNote>Ralph 迴圈像便利商店微波爐的「再加熱三十秒」按鈕。冷便當可以救，法式舒芙蕾大概會變成一坨悲劇。長跑任務需要的不只是再跑一輪，而是先知道到底在煮什麼、誰負責看火、煮到一半忘記前面加過鹽時怎麼辦。</ClawdNote>

## 缺的不是時間，是結構

Jarrod 說，他非常喜歡長跑型 Agent 的想法。這幾乎是目前最接近「直接生出一個月經常性收入 100 萬美元的 B2B SaaS，還完全不犯錯」的幻想版本。這句話當然帶著網路創業圈的誇張味，但也精準描述了很多人對 Agent 的期待：不是輔助，而是接管。

可是他使用 Codex `/goal` 的經驗並不理想，而且這些問題和他先前打造各種失敗長跑工作流時遇到的問題一樣。Jarrod 把缺口整理成三件事：

1. 模糊性會複利成長。
2. 多 Agent 團隊勝過單一強 Agent。
3. 跨脈絡記憶真的有用。

這三件事看起來像工程流程建議，實際上是在回答同一個問題：當模型需要跑很久時，怎麼避免它一路自信地跑偏？

長跑型 Agent 最危險的地方，不是它笨。笨的系統通常很快露餡。真正麻煩的是，它大部分時間看起來都合理，直到最後交付物和原本想像差了十萬八千里。像請設計師做「簡潔有質感的官網」，結果拿到一個黑底紫色漸層、滿天星星、每個區塊都在發光的宇宙飛船控制台。技術上有完成，心靈上沒有。

這也接上 gu-log 前面幾篇文章談過的同一個痛點：Agent 記憶會腐爛，狀態最好落到檔案系統，「丟 Prompt 碰運氣」遲早要變成結構化工作流。Jarrod 這篇的好處，是把這些概念收回到一個很實際的問題：長跑不是把迴圈拉長，而是讓工作可審查、可接力、可留下證據。

## 第一個洞：模糊性會複利

LLM 在迴圈裡工作時，每一輪輸出都會變成下一輪輸入。這聽起來很自然，但也代表早期每個小決策都會滾到後面。模型從頭到尾打造一個東西時，會做出無數判斷：檔案怎麼切、資料怎麼流、介面怎麼擺、測試怎麼寫、哪些情境算完成。

只要其中某個分岔點選了和需求方不同的方向，後面的工作就會沿著那條路越走越遠。這不是 AI 特有問題。現實世界也是這樣：需求講得很模糊，又沒有來回確認，最後交付品「有點像」原本想要的東西，但也只是有點像。像用一句「弄得高級一點」叫人買晚餐，結果對方帶回一整盒松露鵝肝，而真正需要的其實是公司樓下雞肉飯加辣。

Jarrod 認為，AI 目前似乎還沒有真正的品味，所以很難靠模型自己避開這件事。除非提示詞寫得極度詳細，把出錯空間壓到很小，否則模糊性一定會在迴圈裡放大。

他的做法是在任何 Agent 寫程式前，先進入一個「設定階段」。這個階段會用訪談式釐清大量降低模糊性，概念接近 Matt Pocock 的 `grill-me` 技能。Jarrod 在相關脈絡裡也提過類似 `/interview` 的做法：先進入規劃模式，讓模型追問需求、整理假設，再把回答沉澱成計畫檔。這裡的重點不是某個指令名稱，而是長跑開始前要先把規格逼出來。

重點是，在自主迴圈開始之前，互動必須發生。Jarrod 會先投入大量時間，讓 Agent 問問題、抓假設、逼出隱藏需求。經過這個嚴格流程後，目標才會被拆成里程碑，以及每個里程碑底下可以執行的單一任務。

原文裡有一個很好的視覺化：把結果想成一棵樹，每個分支都是一次決策。沒有事前釐清時，Agent 會替需求方選路，最後抵達幾乎無限可能結果中的其中一個。那個結果也許和原始需求有點像，但不一定是最想要的版本。事前多花時間，就是在長跑開始前把明顯錯誤的分支剪掉。

<ClawdNote>這一步其實很殘忍，因為它逼人承認一件事：很多「AI 沒做好」其實是規格根本沒長出來。Agent 不是讀心術服務，也不是通靈板。需求只有一句「做一個厲害的產品」，最後產出失控，責任不能全推給模型。這種痛很工程，痛到有教育意義。</ClawdNote>

## 第二個洞：單一強 Agent 不如小隊作戰

Jarrod 的第二個觀察比較像工程現場的老話：一個人埋頭做到底，很容易把自己做歪。多 Agent 的協調者和子 Agent 關係，通常勝過只用一個很強的 Agent。原文說研究和他的經驗都支持這點，並把相關材料指向 [Anthropic Claude 4.6 系統卡文件](https://www-cdn.anthropic.com/0dd865075ad3132672ee0ab40b05a53f14cf5288.pdf) 第 39 頁的 **Multi-agent BrowseComp** 小節。

代價也很明確：這會燒更多 Token。Jarrod 的說法是，如果 Token 成本不是主要限制，與其讓一個 Agent 在同一個腦袋裡越想越久，不如讓幾個 Agent 分開想，再互相檢查。這有點像把「加班」改成「開 PR」：不是比較省，而是比較不容易一路錯到底。

這裡的差別很重要。單一 Agent 的長跑比較像一個人關在房間裡寫完整個專案，寫到後面房間滿地紙團，桌上三杯冷咖啡，腦袋開始把錯誤看成真理。多 Agent 則比較像小型工程團隊：有人負責實作，有人負責審查，有人負責拆任務和整合。

Jarrod 的工作流大概長這樣：主 Agent 當協調者，遇到一個小任務就拉出一組子 Agent。通常一個負責寫，一個負責看。寫的人交成果，看的那個用新脈絡檢查，兩邊修到可以接受，再回報給主協調者。

關鍵不是形式上多一個角色，而是「新鮮脈絡」。審查 Agent 第一次看到程式碼，沒有前面那堆決策和替決策找理由的包袱，比較能客觀指出問題。Jarrod 特別提到，Agent，尤其是 Claude，在脈絡視窗變得混亂時，可能會讓自己相信一些其實錯得很離譜的事。

<ClawdNote>這句話很傷，但寫程式的人看了應該會安靜三秒。脈絡一亂，Agent 很容易開始替前面的錯誤找理由，像工程師熬夜到第三杯咖啡後堅稱 bug 是 feature。</ClawdNote>

原文也提到 [Boris Cherny](/glossary#boris-cherny)，也就是 [Claude Code](/glossary#claude-code) 的創作者，曾用很精簡的方式說過類似觀點：丟更多 Token 到程式問題上，結果通常會更好；其中一種做法，就是使用分離的脈絡視窗。換句話說，子 Agent 有用，不只是因為多一個模型，而是因為多一個沒有被前面混亂拖下水的視角。

Jarrod 原本做過 Claude Delegator，讓 Codex 去審查 Claude 的工作。後來他也提到會依任務改用不同模型和工具，例如工程實作交給 Codex，前端介面設計交給更擅長設計回饋的工具。這裡不是在宣布唯一正解，而是在呈現他的實務偏好：長跑流程不只要多 Agent，也要讓不同角色用合適的工具。

<ClawdNote>多 Agent 不是魔法陣，人數變多不會自動召喚品質。比較像請兩個工程師互看 PR：如果兩個都沒看規格，只會一起錯得更有自信。真正有用的是角色分工、脈絡隔離、審查回合，還有協調者知道何時該收斂。</ClawdNote>

## 第三個洞：記憶不能只放在模型腦袋裡

長跑型 Agent 還有一個現實問題：脈絡視窗會滿，工作會跨回合，Agent 會換。只靠當前對話內容記住所有決策，等於把專案管理寫在便利貼上，再把便利貼貼在電風扇前面。風一吹，人生也跟著散了。

Jarrod 的做法是，在規劃階段後，要求 Agent 建立、閱讀並維護幾個檔案：

1. `GOAL.md`：最高層級的目標，也就是這輪工作到底要達成什麼。
2. `STANDARDS.md`：不可妥協的程式品質標準。
3. `IMPLEMENT.md`：工作流指令，包含使用多 Agent 審查、撰寫測試、驗證成果等規則。
4. `PROGRESS.md`：持續更新的紀錄，寫下已做決策以及已完成工作。

原文前面說「三個檔案」，但實際列了四個。這裡照內容保留：重點不是數量，而是功能分工。

每個新的 Agent 都必須讀這些檔案，快速理解前面 Agent 做過哪些決策、目前進度在哪裡，並用一致的方向繼續工作。這不只幫 Agent，也幫人類知道專案跑到哪一步。長跑工作最怕的不是慢，而是醒來之後發現系統忙了一整晚，卻沒人說得清楚到底忙了什麼。

Jarrod 也保留了限制條件：Agent 不一定每次都會聽話。要求它們讀太多東西時，這些文件更像指引，而不是百分之百可靠的控制系統。可是整體來說，這些檔案通常能改善流程。

這個保留非常重要。跨脈絡記憶不是把 `PROGRESS.md` 建好就萬事太平。文件會過期、紀錄會漏、Agent 可能跳讀、協調者可能整合失敗。可是沒有文件時，長跑 Agent 幾乎一定會變成「現在這個脈絡說了算」。有文件至少讓專案有外部地圖，不必把所有方向感塞進模型的短期記憶。

## 最後長成一套工程流程

把整篇文章串起來，Jarrod 的長跑工作流不是「更強 Ralph 迴圈」。它比較像把一個衝動的單人冒險，改造成一個小型工程流程。

一開始不要急著跑。先用訪談式釐清，把模糊目標變成具體規格，再拆成里程碑和小任務。這一步在 Jarrod 眼中非常重要，因為它不只幫 Agent 理解，也會逼需求方思考真正想要的成果。

接著才讓多 Agent 執行。主 Agent 當協調者，針對單一任務建立小隊。實作者負責產出，審查者用新的脈絡檢查成果，雙方來回修正，再回報協調者。

最後把記憶寫到模型腦袋外面。`GOAL.md`、`STANDARDS.md`、`IMPLEMENT.md`、`PROGRESS.md` 這類檔案，分別記目標、標準、流程和進度。新 Agent 接手時，先讀外部記錄，再繼續前人的工作。

Jarrod 也公開了自己的 `long-running-agent-skill`，包成 `SKILL.md`，可插到 Claude 或 Codex 使用。依他的說法，這套技能可以搭配支援子 Agent 視覺化的 Codex 環境，也包含用 git worktree 平行化小型子 Agent 團隊的做法。他把這些成果定位成可繼續打磨的專案基礎區塊，而不是一次就能交付的魔法成品。

這些說法都還是個人實務經驗，不是保證每個專案照抄就會成功。可是文章的價值在於，它把「長跑型 Agent」從一句帥話拉回工程現場：怎麼釐清需求、怎麼拆任務、怎麼審查、怎麼留紀錄、怎麼承認 Agent 會忘、會偏、會把錯誤當真。

<ClawdNote>很多 Agent 工作流教學喜歡停在「把這段 Prompt 貼上去」。那種感覺就像教人開餐廳只說「把瓦斯打開」。瓦斯當然重要，但菜單、備料、出餐順序、誰洗鍋、客訴怎麼處理，才是餐廳能不能活過週末的關鍵。</ClawdNote>

## 結語

Jarrod 這篇文章最好的地方，是沒有把 Codex Goals 講成失敗品，也沒有把自己的流程包裝成銀彈。他真正指出的是一個結構問題：Ralph 迴圈讓 Agent 持續前進，但長跑任務需要的不只是前進。

前期釐清，是在出發前剪掉錯誤分支。協調者加審查者，是讓系統有第二雙眼睛，不被單一脈絡拖進幻覺泥巴。跨脈絡記憶，是把專案狀態寫到模型腦袋外面，讓接力不只靠運氣。

所以在 Jarrod 這套工作流裡，長跑型 Agent 不是「一個超強模型關在房間裡把一切做完」。它更像一個會先問問題、再分工、會留紀錄、也會被審查的小型工程流程。Ralph 迴圈讓任務繼續跑；真正讓它比較有機會跑對方向的，是出發前先問清楚目的地。
