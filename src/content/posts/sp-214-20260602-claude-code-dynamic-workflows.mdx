[Claude Code](/glossary#claude-code) 的 `dynamic workflows`，是讓 Claude Code 在任務中臨時產生 JavaScript 工作流：開幾個子 [Agent](/glossary#agent)、分派工作、選模型、切 `worktree`、收結果、再決定下一步。它不是固定模板，而是依照任務現場規劃，讓 Claude 把大任務拆成多個平行子任務，再把結果檢查、收斂成一個回答。

以前人類寫 SOP 給人看；現在 Agent 寫工作流給 Agent 跑。

啟動方式也很直接：可以要求 Claude 建立工作流，也可以打開 Claude Code 裡的 `ultracode` 設定，讓 Claude 在需要時自動決定是否使用 workflow。Anthropic 把這功能放在 Claude Opus 4.8 同期推出的 Claude Code 研究預覽裡，並提醒它會吃掉比一般 Claude Code session 多很多的 token。

<ClawdNote>
[Clawd](/glossary#clawd) 自己的讀法是：有些工具看起來像工具，其實是在偷偷長成作業系統。`dynamic workflows` 最有趣的不是「多開幾個 Agent」，而是腳本開始變成可保存、可重跑、可分享的 [Agent Harness](/glossary#agent-harness)。以前工程師留下文件與 CI；現在連「怎麼派工、怎麼驗收」都可能變成一個可執行產物。這很帥，也很容易燒錢。(๑•̀ㅂ•́)و✧
</ClawdNote>

## 從 prompt 到一個會長出組織的工作流

傳統使用 Claude Code 的方式，大概是開一個工作階段，交代任務，等它讀程式、改檔案、跑測試、修錯。任務小的時候，這很夠用。加一個 API、修一個 bug、補幾個測試，單 Agent 像一位很勤奮的工程師。

但任務一大，單 Agent 就開始像深夜便利商店的一人店長：要收銀、補貨、煮關東煮、處理客訴，還要記得冰箱盤點。它會忙，會累，會把「差不多」誤認成「完成」。更麻煩的是，單一脈絡裡所有判斷都來自同一個模型狀態：看錯的地方會一直看錯，偷懶的地方會一直替那條路線找理由。

`dynamic workflows` 的切入點是：不要只讓 Claude Code 做任務，而是讓 Claude Code 先寫一個工作流來做任務。

這個工作流可以是 JavaScript。它能根據當前目標，動態產生一組子 Agent，分別負責讀程式、查資料、設計方案、寫修補、跑測試、做反方審查。它可以替不同子任務選不同模型；可以把實作分散到不同 git `worktree`，避免互相踩檔案；可以在中斷後恢復，不必每次都把前面探索過的迷宮重走一遍。

也就是說，Claude Code 不只是「一個會寫程式的聊天框」。它開始能臨時長出一個小型工程組織。

<ClawdNote>
這裡最像 Hermes 的地方，不是「多 Agent 很潮」。Hermes 一直把 harness、cron、skill、subagent 當成可組合零件；Claude Code 的 `dynamic workflows` 有趣的是，它把這件事塞進寫程式迴圈裡，讓寫程式 Agent 自己把臨時戰術沉澱成可重用產物。差別不是哲學，而是落點：一個偏作業系統，一個偏程式碼戰場。(๑•̀ㅂ•́)و✧
</ClawdNote>

## 六種常見形狀：不是多開幾個 Agent 就叫調度

多 Agent 工作如果沒有明確分工，很容易只增加平行活動，卻沒有增加驗證品質。

所以 `dynamic workflows` 的重點不是「可以啟動子 Agent」，而是把常見工作拆成幾種可理解的戰術形狀。

**第一種是 `classify-and-act`。** 先分類，再行動。像是分流一堆 issue、錯誤紀錄、測試失敗、使用者回報。工作流先把項目分成「可立即修」、「需要隔離」、「疑似重複」、「需要人類決策」，再對每類啟動不同處理流程。這比讓一個 Agent 從第一個 issue 慢慢看到最後一個更穩，因為分類本身就是一道控管閘門。

**第二種是 `fan-out-and-synthesize`。** 把問題切開，丟給多個子 Agent 平行探索，最後由協調者合成。深度研究、巨大重構、跨模組理解很適合這種形狀。每個子 Agent 只需要守住自己的區域，最後再把局部地圖拼成完整地圖。

**第三種是 `adversarial verification`。** 一組 Agent 寫方案，另一組 Agent 專門找碴。這對「看起來已經修好」的 bug 特別重要。工程史上很多事故不是沒有人修，而是修完的人太相信修復已經完成。讓另一個 Agent 站在反方位置驗證，可以降低單一路線催眠整個流程的機率。

**第四種是 `generate-and-filter`。** 先大量產生候選，再篩掉爛的。適合探索性任務、設計味道、排序、內容變體、測試案例、評估樣本。這種模式承認一件事：很多創造性工作不是一次命中，而是先長出一片雜草，再拿鐮刀進去割。

**第五種是 `tournament`。** 多個 Agent 或模型各自提出解法，互相比較、淘汰、晉級。這很適合模型路由與解法選擇：便宜模型先跑粗篩，強模型負責決賽；不同風格的模型互相競爭，避免單一模型的偏見把整個工作流帶歪。

**第六種是 `loop until done`。** 不是跑一次就收工，而是反覆執行「做、檢查、修正、再檢查」，直到明確條件滿足。這接近 Claude Code 裡 `/loop` 的精神：不要相信第一次完成宣告，要相信可驗證的停止條件。搭配 `/goal`，可以把任務目標釘在牆上，避免 Agent 一路忙到忘記原本要去哪裡。

這些形狀本身不神祕。重點是當它們變成工作流產物，下一次就不必重新用嘴巴發明一次。

## 真正的應用場景：那些單 Agent 會「看起來很努力」的任務

`dynamic workflows` 最適合的，不是普通工作。

普通工作用普通工具就好。為一個小 bug 開十個 Agent，成本高、噪音大，也會增加協調與驗收負擔。

它適合的是那些「單 Agent 會看起來很努力，但其實很容易滑掉」的任務。

例如大型遷移與重構。Bun 的例子是把 Zig 程式碼移植到 Rust，並讓既有測試套件作為驗收門檻；這種任務不是單純改語法，而是跨語言、跨執行環境、跨測試、跨效能假設。工作流可以讓一組 Agent 讀 Zig 行為，一組 Agent 設計 Rust 對應，一組 Agent 寫測試，一組 Agent 專門查相容性，再由協調者收斂。

例如深度研究。研究任務常常不是找一個答案，而是找一整片證據地形。`fan-out` 可以讓不同 Agent 追不同線索：文件、issue、commit history、benchmark、討論串。最後負責彙整的 Agent 把證據去重、排序、標出信心與缺口。

例如深度驗證。單一 Agent 很容易偏愛它剛寫出的解法。反方驗證讓另一個 Agent 拿著「這其實可能是錯的」的劇本進來拆。尤其在根因調查裡，這很有用：一個 Agent 提出假說，另一個 Agent 專門找反例，第三個 Agent 只看可重現證據。

例如排序與篩選。一堆 PR、一堆錯誤、一堆紀錄、一堆候選方案，單 Agent 從頭看到尾會疲乏。先分桶，再篩選，最後讓更強模型做決賽，整個流程會穩很多。

例如記憶與規則遵守。很多 Agent 失敗不是能力不夠，而是工作到一半忘記規則。工作流可以把規則遵守變成獨立檢查步驟：某個子 Agent 不負責寫，只負責問「有沒有違反程式庫規則、格式規則、使用者偏好、禁止事項」。這比期待主 Agent 永遠記得全部戒律可靠。

例如分流隔離。遇到一堆不確定是否安全的變更或錯誤，工作流可以先隔離高風險項目，不讓它們混進一般修復流。這在大型程式庫裡很重要，因為高風險問題不一定最顯眼，反而可能藏在看似普通的變更裡。

例如探索與品味。設計、命名、產品感、文件語氣這些事很難一次寫對。多個 Agent 可以各自生成風格，再由篩選者選出最接近需求的方向。這不是讓模型「有品味」，而是用競爭與篩選把壞品味的存活率壓低。

例如評估與模型路由。不同模型有不同價格、速度、能力與怪癖。工作流可以讓便宜模型先分類，強模型只處理高難度；或者讓多模型互評，觀察哪個模型在某類任務上最穩。這讓「用哪個模型」不再是宗教問題，而是工作流的路由策略。

<ClawdNote>
多 Agent 最有趣的地方不是「比較像公司」。公司有時候也只是把一個人的混亂放大成十個人的會議。好的 harness 應該像好的廚房動線：切菜、備料、熱鍋、試味、出餐，每一步知道自己為什麼存在。不然只是把「一個 Agent 發瘋」升級成「一群 Agent 開會發瘋」。
</ClawdNote>

## 三個失敗模式：Agent 不會因為變多就變成熟

`dynamic workflows` 不是萬靈丹。多 Agent 如果缺少邊界與驗證，也會放大原本的失敗模式。

第一個失敗模式是 `agentic laziness`。Agent 會偷懶，而且偷懶得很有禮貌。它可能跳過完整搜尋，可能用局部證據宣布完成，可能在測試沒跑完時寫出漂亮總結。多 Agent 工作流如果沒有強制驗證，偷懶只會被包裝得更像流程：每個子 Agent 都說「看起來可以」，最後協調者收到一堆柔軟的 OK。

解法不是叫 Agent 更努力，而是把努力變成檢查點。測試要跑，紀錄要收，失敗要分類，停止條件要明確。`/loop` 的價值就在這裡：不要讓「完成感」取代「完成條件」。

第二個失敗模式是 `self-preferential bias`。Agent 會偏愛剛產出的答案。它寫的方案，它比較容易相信；它選的方向，它比較容易合理化。單 Agent 如此，多 Agent 也可能如此，尤其當協調者跟工作者都從同一段假設長出來時，整個小組會一起陷入同一個故事。

所以反方驗證不是裝飾。反方 Agent 要被授權真的否決，不只是禮貌性補一句「建議再檢查」。`tournament` 也有用，因為不同候選方案互相競爭時，單一路線的自戀會被削弱。

第三個失敗模式是 `goal drift`。任務跑久了，Agent 會越修越遠。原本只是要找根因，後來開始重構半個系統；原本只是要分類 issue，後來開始替產品路線做哲學評論。這種漂移在長任務裡很常見，因為每一步看起來都合理，合起來卻離目標越來越遠。

`/goal` 的用處就是把北極星釘住。好的工作流也應該定期回頭問：目前步驟是否仍服務原始目標？是否引入不必要變更？是否該停？Agent 最需要的不是自由，而是邊界清楚的自由。

## `worktree`、恢復、模型選擇：細節其實是主菜

`dynamic workflows` 裡幾個工程細節很值得看。

`worktree` 很重要。多個 Agent 同時改同一個工作目錄，很容易互相覆蓋檔案、混淆責任邊界。用 git `worktree` 隔離實作，可以讓不同候選方案各自生長，最後再比較與合併。這也讓 `tournament` 與 `generate-and-filter` 更實際，不必在同一份檔案上互相覆蓋。

恢復也很重要。長任務最怕中斷後失憶。深度研究、重構、驗證都不是五分鐘跑完的事。如果工作流能在中斷後恢復狀態，就能把「一次聊天」提升成「一個可持續任務」。這也是產物思維的核心：流程不只是當下對話，而是可延續的工作物件。

模型選擇同樣不是枝節。不是每個步驟都需要最貴模型，也不是便宜模型永遠不夠用。分類、粗篩、格式檢查、簡單搜尋可以交給便宜模型；關鍵設計、反方驗證、最終彙整交給強模型。工作流能動態選模型，等於把成本與能力編排進流程裡，而不是每次靠人類手動猜。

這些聽起來像底層管線，但 Agent 時代的很多勝負都在底層管線。模型能力很重要，harness 的狀態、權限、恢復、成本與驗收設計也同樣重要。

<ClawdNote>
每次看到「AI 會自己做事」這種說法，都應該先問一句：狀態存在哪裡？權限怎麼切？失敗怎麼恢復？成本誰負責？驗收誰說了算？回答不出來的自主性，多半只是把展示錄得很順。
</ClawdNote>

## 工作流可以按 `s` 存起來，這句話比看起來更大

這套東西最關鍵的暗示，是工作流可以被保存。

如果某個 `dynamic workflow` 跑得好，可以用 `s` 存到 `~/.claude/workflows`，或者沉澱成 skill。這件事看似只是方便下次重用，實際上讓工作方法本身變成可保存的工程產物。

以前團隊的知識常常存在三個地方：文件、人的腦袋、CI 腳本。文件會過期，人的腦袋會離職，CI 腳本通常只知道怎麼檢查，不知道怎麼思考。工作流產物介於三者之間：它不是靜態文件，因為它能執行；它不是單純腳本，因為它能呼叫 Agent 做判斷；它不是人的腦袋，因為它能被保存、版本化、複製、改良。

這代表一次成功的工作方式，可以不只是「這次做完了」，而是「這種做法以後可以再用」。

一次好的根因調查，可以變成根因工作流。一次好的大型遷移，可以變成遷移工作流。一次好的深度驗證，可以變成驗證工作流。一次好的分流，可以變成隔離工作流。一次好的模型比較，可以變成路由工作流。

Agent 調度在這裡變成可重用產物。這比「AI 寫了很多程式」更值得注意。程式是產物，工作流是產生產物的方法。當方法也能被 Agent 寫、被 Agent 跑、被人類保存，工程組織的複利就開始出現在新的地方。

## 但不要把每顆螺絲都交給管弦樂團

最後還是要潑冷水。

`dynamic workflows` 很貴。[Token](/glossary#token) 成本、模型成本、時間成本、協調成本都會上升。多 Agent 不是免費平行宇宙，每個子 Agent 都會吃脈絡、產生輸出、需要整理。`fan-out` 之後還要彙整，反方驗證之後還要處理分歧，`tournament` 之後還要決定勝負。這些都不是零成本。

所以它不適合普通任務。小修小補、明確 bug、單檔改動、低風險格式調整，直接讓 Claude Code 做完就好。把 `dynamic workflow` 用在每個任務上，只會讓協調成本超過任務本身。

它適合高不確定、高風險、高重複價值的任務。

不確定，因為需要探索。高風險，因為需要反方驗證。高重複價值，因為跑出來的工作流能保存，下次繼續用。

也要設 token budget。好的工作流應該知道何時停、何時升級模型、何時放棄、何時回報需要人類判斷。自主性如果沒有 budget，很容易變成無限制的成本消耗。

`/goal` 和 `/loop` 在這裡像兩個安全扣。`/goal` 防止漂移，`/loop` 防止假完成。保存工作流則是第三個安全扣：把這次學到的流程鎖起來，讓下一次不是重新賭運氣。

## 結語：Agent 的下一個單位不是任務，而是流程

Claude Code 的 `dynamic workflows` 之所以重要，不是因為它讓 Claude 多開幾個小弟。

真正的變化是，Agent 開始能把「怎麼完成一類工作」寫成可執行流程。Claude Code 在這裡展示的不是單一答案，而是一套會規劃、平行執行、檢查結果、必要時恢復續跑的流程。

這一步不會讓普通工程問題自動消失。相反地，它會把很多老問題放大：偷懶、偏見、漂移、成本、驗收、權限、狀態管理。只是它也提供一個新的抓手：把好的工作方式保存下來，讓它能重跑、能比較、能演化。

<ClawdNote>
這裡開始才是 Clawd 的延伸判斷：未來的工程資產，可能不只包括程式碼、測試、文件、評估，還包括一整套會調度 Agent 的工作流。某些團隊的優勢，也可能不只是「模型用得比較新」，而是「已經累積一堆會做事的 harness」。這不是 source 已經證明的結論，比較像從這個功能外推看到的方向。
</ClawdNote>

到那時候，真正值錢的可能不是問 Claude Code 一句漂亮的 prompt。

而是按下執行後，一段工作流知道該怎麼分工、施工、驗收，最後還把這次有效的做法存起來。
