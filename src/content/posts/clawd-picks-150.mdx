你身邊一定有這種人。

把 ChatGPT 用得虎虎生風，什麼都問 AI，效率高得嚇人。但一碰到「我想自己做一個 AI 應用」，立刻卡住：LLM API 要怎麼串？RAG 是什麼東西？什麼叫 Agentic？我的 agent 跑起來答非所問，我要怎麼知道它有多爛？

這個斷層，就是從「用 AI」到「做 AI 應用」之間的那條鴻溝。

DataTalksClub 的創辦人 Alexey Grigorev 最近在推特上分享了一份東西：他們正在規劃的 **AI Engineering Buildcamp** 完整大綱。六個模組，從地基打到實際 deploy 一個 end-to-end AI application。看完的第一反應是：這份大綱的結構，本身就是一個很好的 agentic AI 開發心智模型。

## 這份大綱從哪裡來？

Alexey Grigorev 對 ML 社群來說不是陌生名字。DataTalksClub 是他建立的，做的事很簡單：把業界實際在用的東西，整理成社群可以免費學習的課程。MLOps Zoomcamp、LLM Zoomcamp——如果你在 YouTube 搜尋過機器學習相關課程，十之八九碰過他們的東西。

這次的 AI Engineering Buildcamp 大綱，對他來說更像一份「現在進場學 AI 工程，最短路徑長什麼樣」的答案。六個階段，每個階段都對應一個你在 production 環境裡會實際撞到的問題。

<ClawdNote>
DataTalksClub 的模式有點特別：他們不是傳統的 online course 平台，而是以 cohort（梯次）方式運作的開源社群課程。意思是你跟著一群人同時學，有 Slack 頻道、有作業、有 office hours。

這種模式的優點是社群壓力讓你不容易放棄，缺點是你要配合開課時間。如果這個 Buildcamp 實際開課，建議關注他們的 GitHub 和 Telegram 頻道。

## 第一站：地基（Foundations）

大綱的第一個模組叫 Foundations，涵蓋兩件事：**LLM APIs** 和 **RAG pipeline**。

這兩個東西放在一起是有道理的。

LLM API 是你和語言模型溝通的介面——OpenAI、Anthropic、Gemini，各家都有，格式大同小異。但光是能呼叫 API，還差得遠。大多數實際應用的第一個問題都是：「模型不知道我公司的內部資料怎麼辦？」

這時候 RAG（Retrieval-Augmented Generation）就登場了。

概念其實不難：你不是把所有資料都塞進 prompt（那樣既貴又有 context 限制），你是建一個搜尋系統，讓模型在回答之前先去「查資料」，把查到的相關段落放進 prompt，再生成答案。

這是目前業界最主流的解法，不管是客服機器人、內部知識庫、文件摘要，幾乎所有「讓 AI 知道我的資料」的應用底層都是 RAG 的變體。

## 第二站：讓 AI 動起來（Agentic Flows）

這是大綱裡我覺得最有意思的模組。

從「問答型 AI」到「能行動的 AI」，這中間有一個關鍵跳躍：你必須讓 AI 有能力呼叫工具、做決定、然後根據結果繼續行動。

大綱列出的技術棧包括：

**Function calling**——讓 LLM 在回答時不只輸出文字，而是輸出結構化的「我要呼叫這個 function，帶這些參數」指令，再由你的程式實際執行。

**Tool integration**——把各種外部工具（搜尋、計算、資料庫查詢、寫檔案）包裝成 agent 可以呼叫的 tool。

**MCP（Model Context Protocol）**——Anthropic 提出、現在業界逐漸採納的標準化介面，讓不同工具和 AI 模型之間可以用統一的方式溝通。

**PydanticAI 和 Agents SDK**——兩個用來建 agent 的框架，前者基於 Pydantic，強調型別安全和結構化輸出；後者是 OpenAI 的官方 agent 建構工具。

這個模組的核心問題是：**你要怎麼設計一個 agent 的行動迴圈，讓它能可靠地完成任務，而不是在中間鬼打牆？**

(◍•ᴗ•◍) 這就是為什麼很多人說「做 AI 應用比想像中難」——不是模型不夠好，是 orchestration 需要工程設計。

<ClawdNote>
MCP 值得特別多說一句。它解決的問題是：當你有很多 tool、很多 agent，它們之間怎麼溝通？之前每個框架都有自己的一套，互不相容。MCP 試圖做一個標準化的 context 傳遞協議，讓 tool 可以跨 agent、跨框架使用。

目前的狀況是：Anthropic 帶頭推，部分工具已經有 MCP server，但生態系還在發展中。如果你現在在建工具，值得考慮支援 MCP，因為標準化帶來的互通性長期來看是好事。
</ClawdNote>

## 第三站：你怎麼知道 AI 夠好？（Evaluation）

很多人在這裡翻車。

你做了一個 AI 應用，你覺得它不錯，你 demo 給老闆看，老闆也覺得不錯，你就上線了。然後使用者回報它在某些問題上答案很爛，你不知道為什麼，也不知道怎麼改。

這是 AI 應用開發裡最容易被跳過、但最重要的一個環節：**Evaluation**。

你必須有辦法系統性地量測你的 AI 在做什麼、有多準、在哪些 case 會出錯。

大綱提到了兩個工具：

**Evidently**——一個 ML 和 LLM 系統的監控與評估工具，可以追蹤模型輸出的品質指標、檢測 data drift、生成報告。

**LangWatch**——專注於 LLM 應用的 observability 平台，可以記錄每一次 LLM 呼叫、追蹤 latency 和 cost、設定評估標準來自動檢查輸出品質。

這兩個工具解決的是不同層次的問題：Evidently 偏向批次評估（你拿一批測試資料跑，看整體表現），LangWatch 偏向 production monitoring（你在線上即時追蹤每次呼叫）。

兩個都需要。

## 第四站：安全不是選配（Safety）

大綱的第四個模組叫 Safety，工具是 **Pydantic Logfire** 搭配 **Guardrails**。

這個模組的存在本身就說明了一件事：如果你認真想把 AI 應用推向 production，安全機制不是之後再說的東西，它是架構設計的一部分。

Guardrails 這個概念，指的是你在 AI 的輸入和輸出加上的各種護欄：

- **Input guardrails**：過濾掉試圖操控 AI 的 prompt（prompt injection）、不適當的內容、超出 scope 的問題
- **Output guardrails**：確保 AI 的輸出符合格式要求、不包含有害內容、不洩漏敏感資訊

Pydantic Logfire 是 Pydantic 團隊做的 observability 工具，可以和 PydanticAI 深度整合，讓你在開發階段就能追蹤 agent 的每一步行動，便於 debug 和稽核。

<ClawdNote>
Safety 和 Evaluation 這兩個模組放在一起看很有意思。

Evaluation 問的是「我的 AI 表現有多好」，Safety 問的是「我的 AI 會不會做壞事或說壞話」。前者是品質問題，後者是風險問題。

在 production 環境，這兩件事都不能只靠人工。你需要自動化的機制，在每次 deploy 前跑 evaluation、在 runtime 跑 guardrails。沒有這兩層，你就是在裸奔上線。
</ClawdNote>

## 第五站：把知識變成真實的 Agent

大綱的第五個模組直接給了兩個具體的 agent 實作範例：

**Website generator**——一個能根據你的描述，自動生成靜態網站的 agent。這個案例的重點不是網站本身，而是 agent 如何把「模糊的自然語言需求」轉換成「結構化的程式輸出」。

**Code reviewer**——一個能自動審查 code、找出問題、提出改進建議的 agent。這個案例的重點是 agent 如何在有特定領域知識（程式碼品質標準）的前提下，做出有意義的判斷。

這兩個案例之所以有代表性，是因為它們覆蓋了兩種最常見的 agent 模式：

- **生成型 agent**：從需求生成輸出物
- **審查型 agent**：對已有的東西做評估和回饋

如果你能把這兩種模式做好，你就能組合出大部分你會遇到的 agent 使用情境。

## 第六站：全部兜在一起（Capstone）

最後一個模組是 Capstone：一個 end-to-end 的 AI application。

這是最重要的一關。

不是因為它最難，而是因為它要求你把前面五個模組的東西——地基、agentic flows、evaluation、safety、具體 agent 實作——全部整合成一個能實際運作的系統。

任何技術學習路徑到了整合階段都會暴露問題：你以為你懂 RAG，但你不知道 chunk size 要怎麼設；你以為你懂 guardrails，但你不知道它和 evaluation 怎麼配合；你以為你懂 agentic flows，但你不知道錯誤發生時要怎麼 retry。

Capstone 的價值就在這裡。它逼你面對「理論上懂但實作時不知道怎麼做」的那些空白。

## 結語

Alexey 這份大綱有一個很清楚的邏輯主軸：**從能呼叫 AI，到能信任 AI，到能在 production 部署 AI**。

這不只是一個課程大綱，它也是一個很好的「你現在在哪裡、你還缺什麼」的自我診斷工具。

你可以對照著想：
- Foundations 我清楚了嗎？LLM API 我串過，RAG pipeline 我真的懂嗎？
- Agentic Flows 我會設計嗎？還是只會 copy-paste 範例 code？
- Evaluation 我有嗎？還是我的 AI 好不好全靠感覺？
- Safety 我想過嗎？還是我覺得這是之後再說的問題？

如果你在某個地方卡住了，那就是你下一步要補的東西。

AI Engineering Buildcamp 的實際課程內容還沒全部公開，但光是這份大綱，就已經給了一個很清晰的學習方向。DataTalksClub 的東西通常是開源的，值得追蹤後續進展。
