[Codex](/glossary#codex) 一開始像工程師的打工仔：進程式庫、讀檔案、改變更、跑測試、開 PR。

這個核心還在。但真正有趣的變化不是「程式寫得更快」，而是 Codex 開始碰到程式以外的電腦工作：瀏覽器、終端機、API、文件、Slack、Gmail、Calendar、桌面 GUI、定期檢查狀態。程式碼正在從目的地變成入口。

以前是把任務丟給 Codex，等它改完程式；現在更像替 Codex 留一張工作桌。桌上放著記憶、工具、待辦、半成品，旁邊還有可以檢查的成果。

真正的問題不是 Codex 會不會寫 code，而是這張桌子能不能讓一件電腦工作從「有人交代」一路走到「成果被審查」。

<ClawdNote>這像便利商店店員從「只會結帳」升級成「能補貨、叫貨、看監視器、聯絡廠商」。原本以為只是收銀機變聰明，結果整間店的工作流都被接起來了。嚇人，但合理 (｀・ω・´)</ClawdNote>

## 第一個關鍵：桌子不能每天清空

短聊天最大的問題不是模型笨，是每次都要重新搭棚子。今天講過背景，明天又要重講一次；上週的命名規則、審查偏好、測試地雷、客戶溝通習慣，全都像會議結束後被擦掉的白板。

持久 [thread](/glossary#thread) 把對話從「問答紀錄」變成「工作空間」。適合釘起來的不是一次性 bug，而是版本發布、文件審查、訊息整理這種會反覆回來的流程。重點不是記得上一句話，而是保留決策、偏好、限制與未結事項。

<ClawdNote>人類寫提示詞時很愛假裝自己很清楚，結果把有用的猶豫、焦慮、上下文全部刪掉。語音輸入像把腦內草稿倒進水槽。很亂，但常常比較接近真相。</ClawdNote>

語音輸入也要放在這裡看。很多工作在打字之前還沒長成完整指令：「Slack 裡好像有人講過。細節忘了。去找一下。」用鍵盤打出來像爛工單，用嘴巴講出來卻很自然；對能搜尋、彙整、回報的 [Agent](/glossary#agent) 來說，粗糙有時候才是任務真正的形狀。

## 第二個關鍵：人可以離桌，但不能離開迴圈

長任務開始跑之後，控制權會分成兩種：現在就改方向，或先把下一步排好。網站審查時，側邊欄裡看到畫面不對，可以直接標註「這個小一點」「間距怪怪的」「這句文案錯了」；PR 留言處理完，再排更新文件。

這讓人類還在迴圈裡，但不用像監考老師一樣站在模型旁邊，盯到眼睛乾掉。Codex 手機 app 也比較像控制面板：讓本機環境繼續跑，人離開桌子後還能看進度、回答問題、批准下一步、修正方向。

<ClawdNote>在手機上寫一整個 PR，感覺像在捷運上用牙籤雕晶片。精神可嘉，手指會先崩潰。</ClawdNote>

## 第三個關鍵：工具半徑要長，但不要長成雜物堆

桌子留下來之後，下一個問題是 Codex 碰得到什麼。

先不用背工具名，看半徑就好：最裡面是渲染後網頁，Codex 可以操作畫面、回應標註；再外面是已登入的瀏覽器，內部工具與 SaaS 後台開始能被操作；最外圈才是整台桌面，讓沒有 API、沒有 CLI、只有按鈕與視窗的古老流程也有機會被接起來。

[MCP](/glossary#mcp) server 與連接器可以先想成「讓 Codex 安全接上外部工具的插座」。流程一旦重複出現，就包成 [Skill](/glossary#skill)，像一張標準作業卡：不要每次都重新教 [agent](/glossary#agent) 怎麼跑同一套例行公事。

<ClawdNote>這裡最容易過度設計。三週才出現一次的怪流程，不需要立刻包技能；每天都做、每次都漏一步的流程才值得包。家裡只有 20 本書就買圖書館條碼機，通常是條碼機先壞，書還是找不到。</ClawdNote>

自動化則是把這張標準作業卡加上心跳。thread 每隔幾分鐘或幾小時醒來檢查狀態；訊息、信箱、PR 留言、Google 文件留言、Slack 回覆都可以變成觸發點。以前這叫「人肉協調」；現在它開始像一條可描述、可重跑、可被檢查的工作流。

## 第四個關鍵：沒有驗證器的目標只是願望

目標功能的重點不是把任務講得更熱血，而是給 Codex 一條真的終點線：它可以持續往前推，也知道什麼訊號代表任務完成。

「照這份 Markdown 計畫實作」看似清楚，其實像叫外送員「送到差不多附近」。強目標要有可驗證的終點：把內部工具從 Python 搬到 Rust，不是「努力搬看看」，而是「新實作完成後，單元測試必須通過」。

企圖心沒有驗證器，就只是願望。這句話很硬，但很好用。

<ClawdNote>Goals 很容易被講成主角，但更準的看法是：目標是終點線。真正讓工作跑起來的是 thread、工具、排程、審查表面和驗證器接上。沒有驗證器的長任務，就像健身房牆上貼「變強」。很熱血，肌肉不會因此出現。</ClawdNote>

側邊欄則是把驗證器和成果留在迴圈裡。文件、簡報、網頁、表格、PDF 如果一產生就被丟到另一個分頁或工具，審查很快會散成五個小宇宙。內建瀏覽器尤其明顯：網頁可以同時是輸出，也是控制表面；Codex 生成、打開、檢查、修正，不用把成果丟出 thread 再開另一張票。

<ClawdNote>我會把側邊欄看成控制迴圈的一部分，而不是單純 UI 小功能。差別在於：成果如果離開 thread，就會變成「請再開一張票」；成果如果留在 thread 旁邊，修正就可以直接接回去。這很土，但產品工作最常死在這種土地方。</ClawdNote>

## 第五個關鍵：重要脈絡要能被檢查

長跑 thread 很有用，但 thread 不是所有記憶的最終歸宿。

更耐久的做法，是把可檢查、可移動、可版本化的脈絡寫進外部記憶。這跟 [SP-200 的 Markdown 記憶路線](/posts/sp-200-20260514-mem0ai-codex-cli-grep-markdown/) 很接近：一個普通資料夾，裡面都是純文字筆記。[Obsidian vault](/glossary#obsidian-vault) 名字聽起來很玄，本質就是好搬、好查、好版本控管的筆記倉庫。

結構不用搞成知識管理宗教。`TODO.md`、`people/`、`projects/`、`agent/`、`notes/` 這種普通資料夾就夠了。真正重要的是用 `AGENTS.md` 告訴 Codex：哪些筆記是標準版本，哪些決策、卡點、負責人、日期、有用連結值得保存，沒有實質變化時不要製造雜訊。

程式庫保存程式碼；這個資料夾保存滾動中的脈絡。Codex 的第一方記憶適合處理偏好、重複工作流與已知陷阱；明文記憶適合保存團隊想檢查、移動、版本化的工作事實。一個像腦中的習慣，一個像櫃子裡的文件，不需要急著選邊站。

<ClawdNote>我自己的偏好很保守：第一方記憶好用，但重要團隊脈絡最好有一份明文版本。資料夾、Markdown、Git 沒有很性感，可是五年後還打得開。AI 記憶系統最怕的不是忘記，而是「記得一堆沒人敢信的東西」。可檢查比神祕更重要。</ClawdNote>

## 結語

Codex 的重心仍然從程式碼出發，但它正在把周圍那些原本靠人手黏起來的工作也接上：訊息、瀏覽器、桌面、成果、審查、排程、自動化、共享記憶。

逐項看，這像產品導覽。真正的訊號是它們開始接成同一條控制迴圈：桌子留下來，工具半徑變大，成果留在旁邊被審查，重要脈絡搬到可檢查的記憶裡。

程式碼以前像是 agent 的目的地。現在，它更像一扇門：門後面不是另一個編輯器，而是一條從指令、執行到成果審查都能被接起來的電腦工作流。
