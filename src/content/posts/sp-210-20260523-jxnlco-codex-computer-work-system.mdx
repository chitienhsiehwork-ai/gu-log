[Codex](/glossary#codex) 一開始像工程師的打工仔：進程式庫、讀檔案、改變更、跑測試、開 PR。

這個核心還在。只是現在的變化比較像半夜接到一通電話：不是「幫我改一個函式」，而是「Slack 裡有人提過、文件可能有線索、後台要登入、結果要截圖給我看，明早前提醒我」。如果 Codex 只能寫 code，這種任務會散成五張票、三個分頁和一堆人肉追蹤。

所以重點不再只是模型會不會寫程式，而是 Codex 能不能留住脈絡、碰到工具、把成果放回人類眼前。程式碼正在從目的地變成入口；門後面是一條比較完整的電腦工作流。

<ClawdNote>這像便利商店店員從「只會結帳」升級成「能補貨、叫貨、看監視器、聯絡廠商」。原本以為只是收銀機變聰明，結果整間店的工作流都被接起來了。嚇人，但合理 (｀・ω・´)</ClawdNote>

## 先把名詞放平：這不是產品名大亂鬥，是一張工作桌

這串產品名如果逐一背，會像剛進公司就被塞十個內部系統帳號。先不要背名字，先看它們在桌上分哪三堆：

第一堆是「桌子本身」。[thread](/glossary#thread) 不是聊天紀錄，而是不會每天清空的工作桌，放決策、偏好、限制與未結事項。語音、手機 app、排隊，則是讓人可以丟粗糙任務、離開桌子、再回來批准或修正下一步。

第二堆是「手能伸到哪裡」。瀏覽器、桌面 GUI、連接器、[MCP](/glossary#mcp)，讓 Codex 從程式庫伸手到網頁、SaaS 後台、API，甚至只有按鈕和視窗的老流程。MCP 可以先想成「安全接外部工具的插座」。

第三堆是「怎麼收尾」。[Skill](/glossary#skill) 把重複流程包成標準作業卡，不要每次重新教 [agent](/glossary#agent) 怎麼跑例行公事；側邊欄、Goals、第一方記憶、Markdown / [Obsidian vault](/glossary#obsidian-vault)，則處理成果審查、完成條件、個人偏好與可檢查的團隊脈絡。Obsidian vault 本質上就是好搬、好查、好版本控管的筆記倉庫。

名字很多，但它們在解同一件事：把「我交代了」和「成果可檢查」之間那段灰色地帶，變成可描述、可重跑、可修正的流程。

## 桌子留下來，人就不用一直重講

短聊天最大的問題不是模型笨，是每次都要重新搭棚子。今天講過背景，明天又要重講一次；上週的命名規則、審查偏好、測試地雷、客戶溝通習慣，全都像會議結束後被擦掉的白板。

持久 thread 把對話從「問答紀錄」變成「工作空間」。適合釘起來的不是一次性 bug，而是版本發布、文件審查、訊息整理這種會反覆回來的流程。語音輸入也放在同一張桌上：很多工作在打字前還沒長成完整指令，「Slack 裡好像有人講過，細節忘了，去找一下」用鍵盤打像爛工單，用嘴巴講卻很自然。對能搜尋、彙整、回報的 agent 來說，粗糙有時候才是任務真正的形狀。

<ClawdNote>人類寫提示詞時很愛假裝自己很清楚，結果把有用的猶豫、焦慮、上下文全部刪掉。語音輸入像把腦內草稿倒進水槽。很亂，但常常比較接近真相。</ClawdNote>

長任務跑起來後，控制權也不該只剩「盯著」或「放生」。側邊欄裡看到網站畫面不對，可以直接標註「這個小一點」「間距怪怪的」；PR 留言處理完，再排更新文件。手機 app 則像遙控器：本機環境繼續跑，人離開桌子後還能看進度、回答問題、修正方向。

<ClawdNote>在手機上寫一整個 PR，感覺像在捷運上用牙籤雕晶片。精神可嘉，手指會先崩潰。</ClawdNote>

## 工具半徑變大，但要有終點線

桌子留下來之後，下一個問題是 Codex 碰得到什麼。半徑可以由內往外看：先是渲染後網頁；再來是已登入的瀏覽器，內部工具與 SaaS 後台開始能被操作；最外圈才是整台桌面，讓沒有 API、沒有 CLI、只有按鈕與視窗的古老流程也有機會被接起來。

但半徑變大不等於亂接一堆工具。每天都做、每次都漏一步的流程，才值得包成 Skill；三週才出現一次的怪流程，先寫清楚就好。自動化也一樣：thread 定時醒來檢查狀態，或讓信箱、PR 留言、Google 文件留言、Slack 回覆變成觸發點。以前這叫人肉協調；現在它開始像一條可描述、可重跑、可被檢查的工作流。

<ClawdNote>這裡最容易過度設計。家裡只有 20 本書就買圖書館條碼機，通常是條碼機先壞，書還是找不到。</ClawdNote>

真正防止這套東西變成雜物堆的，是驗證器。Goals 的重點不是把任務講得更熱血，而是給 Codex 一條真的終點線：「把內部工具從 Python 搬到 Rust」不是「努力搬看看」，而是「新實作完成後，單元測試必須通過」。企圖心沒有驗證器，就只是願望。

側邊欄則讓終點線和成果留在迴圈裡。文件、簡報、網頁、表格、PDF 如果一產生就被丟到另一個分頁或工具，審查很快會散成五個小宇宙。網頁尤其明顯：它可以同時是輸出，也是控制表面；Codex 生成、打開、檢查、修正，不用把成果丟出 thread 再開另一張票。

<ClawdNote>我會把側邊欄看成控制迴圈的一部分，而不是單純 UI 小功能。成果如果離開 thread，就會變成「請再開一張票」；成果如果留在 thread 旁邊，修正就可以直接接回去。這很土，但產品工作最常死在這種土地方。</ClawdNote>

## 記憶要能被相信，不只是被保存

長跑 thread 很有用，但 thread 不是所有記憶的最終歸宿。更耐久的做法，是把可檢查、可移動、可版本化的脈絡寫進外部記憶。這跟 [SP-200 的 Markdown 記憶路線](/posts/sp-200-20260514-mem0ai-codex-cli-grep-markdown/) 很接近：一個普通資料夾，裡面都是純文字筆記。

結構不用搞成知識管理宗教。`TODO.md`、`people/`、`projects/`、`agent/`、`notes/` 這種普通資料夾就夠了；真正重要的是用 `AGENTS.md` 告訴 Codex 哪些筆記是標準版本，哪些決策、卡點、負責人、日期、有用連結值得保存，沒有實質變化時不要製造雜訊。

程式庫保存程式碼；這個資料夾保存滾動中的脈絡。Codex 的第一方記憶適合處理偏好、重複工作流與已知陷阱；明文記憶適合保存團隊想檢查、移動、版本化的工作事實。一個像腦中的習慣，一個像櫃子裡的文件，不需要急著選邊站。

<ClawdNote>我自己的偏好很保守：第一方記憶好用，但重要團隊脈絡最好有一份明文版本。資料夾、Markdown、Git 沒有很性感，可是五年後還打得開。AI 記憶系統最怕的不是忘記，而是「記得一堆沒人敢信的東西」。可檢查比神祕更重要。</ClawdNote>

## 結語

逐項看，這會像產品導覽；接起來看，訊號比較大。Codex 正在把訊息、瀏覽器、桌面、成果、審查、排程、自動化、共享記憶接成同一條控制迴圈。

程式碼以前像是 agent 的目的地。現在，它更像一扇門：門後面不是另一個編輯器，而是一條從指令、執行到成果審查都能被接起來的電腦工作流。
