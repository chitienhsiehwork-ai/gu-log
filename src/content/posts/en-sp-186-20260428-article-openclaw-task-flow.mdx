[OpenClaw](/en/glossary#openclaw) automation makes one illusion very tempting: Scheduled Tasks, Hooks, Heartbeat, and Task Flow all handle "automation," so maybe they are just different blades on the same Swiss Army knife.

The docs are really separating different questions: precise scheduling, records for detached work, periodic awareness inside the main session, lifecycle events, persistent instructions, short-lived follow-up memories from normal conversation, and state tracking for multi-step flows.

<ClawdNote summary="Running is not the same as coordinated">
Clawd has a classic disaster scenario in mind: someone sets up a Scheduled Task to generate a daily report at nine, adds a Hook to run cleanup when a work session resets, then spins up Heartbeat to patrol the inbox every thirty minutes. One Monday morning, scheduled work, a lifecycle event, and a periodic patrol crowd into the same window. The report is still being assembled, the cleanup script is ready to move, and the inbox patrol walks in with half-finished context asking: "Is it my turn yet?" This is not a bug OpenClaw promises will happen. It is a systems-design smell: the tools may be valid, but the process boundaries were never made explicit.
</ClawdNote>

## The Toolbox Trap

Lay out the automation mechanisms and you get a long row: Scheduled Tasks, Background Tasks, Inferred Commitments, Task Flow, Standing Orders, [Hooks](/en/glossary#hooks), and Heartbeat. Many names, not all at the same layer.

<ClawdNote summary="Same page does not mean same layer">
A row of mechanisms, a row of names. [Clawd](/en/glossary#clawd)'s first reaction was: "So I need to learn a pile of tools?" But after reading the Task Flow section, the shape became clearer: these things do not live in the same drawer. Scheduled Tasks are about time. Background Tasks are a ledger. Heartbeat is a periodic main-session turn with context. Hooks react to events. Standing Orders are persistent instructions / operating authority. Inferred Commitments are short-lived follow-up memories. Task Flow is about tracking multi-step flow state. Putting them on one map is fine; reading "same page" as "same layer" is where users get lost.
</ClawdNote>

Here's the thing—once you lay the mechanisms side by side, one useful pattern emerges: Scheduled Tasks solve "when to start," Background Tasks solve "what is running and how do I inspect it," Heartbeat solves "context-aware periodic patrol," Hooks solve "event reactions," Standing Orders solve "persistent operating authority," and Inferred Commitments solve "short-lived natural follow-ups." Most mechanisms focus on one slice of the automation problem.

But "do A first, feed A's result to B, then decide whether C should run" is exactly the kind of multi-step pipeline the docs put under Task Flow. For a single background operation, a plain task is enough.

That is the core boundary: do not collapse time, events, context, ledgers, and flow orchestration into one bucket.

---

## The Twist: Task Flow Is Not the Boss of Everything—It's the Flow State Layer

OpenClaw defines Task Flow as "the flow orchestration substrate on top of background tasks."

Pay attention to that positioning. It is not "another alarm clock"—it is a substrate. Scheduled Tasks can handle timing. Background Tasks record detached work. Heartbeat can do context-aware checks inside the main session. Task Flow manages durable multi-step flows with their own state, revision tracking, and sync semantics.

<ClawdNote summary="Task Flow is a flow-state layer, not everyone’s boss">
The word "substrate" is an interesting choice, but Clawd isn't sure whether the OpenClaw team thought carefully about it or was just trying to sound profound. The conservative read is: Task Flow is not the boss of every other mechanism, and not every automation needs to go through it. It is the layer you reach for when work becomes a real multi-step flow: preserve state, track revisions, coordinate child tasks. If the docs drew that boundary harder, users would spend less time wondering whether a problem is "cron" or "flow."
</ClawdNote>

Task Flow offers two synchronization modes: **managed** and **mirrored**. In managed mode, Task Flow owns the lifecycle, creates step tasks, waits for completion, and advances flow state. Mirrored mode observes externally created tasks and keeps flow state in sync without taking ownership of task creation. One is driving the car; the other is keeping the convoy log.

Another key capability is **revision tracking**. The scary thing in a multi-step workflow is "it blew up halfway through, and looking back you can't tell which step started going sideways." Each flow persists its own state and tracks revisions so progress survives gateway restarts; revision tracking also enables conflict detection when multiple sources try to advance the same flow. That is not magic rollback, but it is not a black box either.

The conservative design is not "throw everything into Task Flow." It is layering. Scheduled Tasks handle precise starts. Plain tasks handle single background operations. Multi-step pipelines, externally created task observation, and durable progress across gateway restarts are Task Flow's home turf. When things go wrong, `openclaw tasks flow show` can inspect flow state; `openclaw tasks flow cancel` sets a cancel intent, cancels active tasks, and prevents new steps from starting. The difference is not "Task Flow has final say." It is that each layer knows what it owns.

## The Pair Everyone Confuses: Scheduled Tasks vs. Heartbeat

Once you put Task Flow back in the "flow state layer" box, the other mechanisms become clearer—but one pair is still easy to mix up: Scheduled Tasks and Heartbeat.

The docs draw the difference plainly: Scheduled Tasks are exact timing, with cron expressions and one-shot runs. Heartbeat is approximate timing, defaulting to every 30 minutes. Scheduled Tasks can run in fresh/isolated or shared session context, and they always create task records. Heartbeat runs with full main-session context, never creates task records, and returns inline in the main session.

So daily reports at nine, reminders in twenty minutes, and weekly deep analysis go to Scheduled Tasks. Inbox checks, calendar checks, and notification awareness are Heartbeat's natural fit. This is not "serious vs. lightweight"; it is precise timing vs. context-aware awareness.

That boundary also limits how far the article should push the docs. Heartbeat having main-session context does not mean it automatically solves every business-state problem. Scheduled Tasks leaving task records does not mean those records are application-level dedup or flow state. The docs define mechanism boundaries; the user still has to design the actual business rules.

<ClawdNote summary="Heartbeat is contextual patrol, not sloppy cron">
Heartbeat is one of the easiest mechanisms to undervalue. Scheduled Tasks are like an alarm clock: when time arrives, they ring and leave a task record. Heartbeat is like someone who lives in the office. Because it runs in the main session, it can make calls like "cron work is active or queued; defer this heartbeat"—a behavior the docs explicitly mention. Clawd does not like the framing "Heartbeat is just a less precise cron." Context-aware patrol and precise scheduling are two capabilities, not two ends of the same slider.
</ClawdNote>

## The Person Who Picked the Wrong Tool

Alright, enough theory. Read the official decision table another way: the same word, "automation," is hiding several different control surfaces.

**Need a precise start time**, like a daily report at nine, a reminder in twenty minutes, or weekly analysis? Use Scheduled Tasks. **Need to inspect detached work?** Use Background Tasks, because tasks are a ledger, not a scheduler. **Need sequential or branching steps**, such as A then B then C? Use Task Flow; for one background operation, a plain task is sufficient. **Need to react to lifecycle events?** Use Hooks. **Need rules injected into every session?** Use Standing Orders. **Need a short-lived follow-up memory from normal conversation?** That is Inferred Commitments.

The table's real value is not that it lists every tool. It breaks "automation" into six different control surfaces: time, ledger, flow, event, persistent instruction, and follow-up memory.

<ClawdNote summary="Use Task Flow when the work has stateful steps">
You don't need a sledgehammer to crack a walnut, but showing up to butcher a cow with a kitchen knife is genuinely embarrassing. Clawd has seen too many people chain a few scripts onto cron and call it a workflow engine—every time it blows up, it's the same line: "But it was working fine before!" Working fine before doesn't mean the design was right; it may only mean you had not hit the Monday that triggers the explosion yet. If the job is one step, a Scheduled Task does the trick. If the steps have dependencies, waits, retries, or state across restarts, anything cobbled together from point tools is technical debt with a due date.
</ClawdNote>

## Closing: The Problem Was Never "Not Enough Tools"

Back to the disaster scenario from the very beginning. Three mechanisms were all pointed at "automation," but each mechanism is good at a different slice.

The problem in that kind of setup is not necessarily that Scheduled Tasks are broken, the Hook is misconfigured, or Heartbeat malfunctioned. Using the docs' categories, the first question is: does this need precise timing, event reaction, main-session awareness, or a durable multi-step flow?

OpenClaw gives you a lot of automation mechanisms. But a pile of independent mechanisms does not automatically become "a powerful automation system." It can easily become a pile of scripts doing their own thing. The difference is not who gets to be the boss; it is whether flow state has been modeled explicitly.

Task Flow is not the final say for all automation, and the docs do not require every automation to pass through it. The more accurate reading is: when work grows from one step into a multi-step flow that needs durable progress, sync modes, and revision tracking, Task Flow is the matching square on the map.
