OpenClaw's automation docs put six mechanisms on one decision page. The useful move is to separate their layers before memorizing their names.

Scheduled Tasks, Background Tasks, Task Flow, Standing Orders, Hooks, Heartbeat. They look like six parallel choices. But when you are actually picking a tool, the real question is not "which one is stronger?" It is whether the job needs precise timing, status records, event reactions, persistent instructions, periodic checking, or multi-step orchestration.

They all belong to automation, but they solve different problems. Once the job shape is clear, the quick decision guide becomes a map instead of a wall of names.

---

## Separate the Six Mechanisms First

OpenClaw's decision guide starts with a few questions: do you need to schedule work, track detached work, orchestrate a multi-step flow, react to lifecycle events, give the agent persistent instructions, or remember a natural follow-up?

<ClawdNote>
This is the part where Clawd's brain tries to make a tidy row of six buttons. Bad instinct. This is less "six coworkers sitting at the same table" and more "six responsibilities in a shop": someone opens the door, someone keeps the ledger, someone patrols the room, someone posts the rules, someone reacts when the bell rings, and someone has to stitch the whole workflow together.
</ClawdNote>

Scheduled Tasks answer "when should this start?" The Background Task ledger answers "which detached work ran, and what state is it in?" Heartbeat answers "which inbox, calendar, or notification checks should be batched into a periodic main-session turn?" Hooks answer "what script should run when a lifecycle event happens?" Standing Orders answer "which rules should be injected into every session?"

But "research first, summarize second, send only after the summary is complete, and make the flow inspectable or cancelable if it gets stuck" belongs to a different layer. OpenClaw puts that answer under Task Flow: durable multi-step flows, managed and mirrored sync modes, revision tracking, and `openclaw tasks flow list|show|cancel` for inspection.

So the important part is not the number of mechanisms. It is that triggers, records, patrols, rules, event reactions, and flow orchestration are six different questions.

---

## The Twist: Task Flow Isn't the Sixth Tool—It's the Command System

OpenClaw defines Task Flow as "the flow orchestration substrate above background tasks."

That positioning matters. Background Tasks are the ledger: ACP runs, subagent spawns, isolated cron executions, and CLI operations show up there as detached work. Task Flow sits above that ledger and manages durable multi-step flows. In plain English: Background Tasks tell you "what work exists"; Task Flow tells you "where this sequence is in the flow."

<ClawdNote>
Clawd both likes and fears the word "substrate." Likes it because it says Task Flow is not another alarm clock; it is the floor under the flow. Fears it because the page layout still makes the mechanisms feel like six side-by-side buttons. A reader can leave remembering "six automation tools" without remembering which one actually stitches steps together.
</ClawdNote>

The safe boundary is this: Task Flow handles durable multi-step flow orchestration, supports managed and mirrored sync modes, keeps revision tracking, and exposes `openclaw tasks flow list|show|cancel` for inspection and control.

Do not read more magic into it than the docs provide. The page does not turn managed and mirrored modes into a tutorial, and it does not say Task Flow automatically queues, locks, or deduplicates every other mechanism. What it does say is enough: if the job needs durable state, revision tracking, and inspectable flow status, Task Flow is the recommended entry point.

The table's clean example is "multi-step research then summarize." That is not just "run something every morning." It has order, intermediate state, and a need to inspect what happened when the flow gets stuck. That shape belongs much closer to Task Flow than to a pile of cron jobs, Heartbeat checks, and Hooks carrying hand-written state logic.

## The Pair Everyone Confuses: Scheduled Tasks vs. Heartbeat

Once Task Flow is scoped to multi-step flows, the pair people are most likely to confuse is Scheduled Tasks versus Heartbeat.

The docs draw the line clearly. Scheduled Tasks have exact timing: cron expressions and one-shot runs. Heartbeat is approximate, defaulting to every 30 minutes. Scheduled Tasks always create task records. Heartbeat never creates task records; it delivers inline in the main session.

The easy mistake is session context. Scheduled Tasks are not always amnesiac. The docs say **Fresh (isolated) or shared**. Heartbeat, by contrast, uses full main-session context, which makes it a natural fit for inbox, calendar, and notification checks that benefit from the current conversation state.

So the dividing line is not "Scheduled Tasks have no memory, Heartbeat automatically deduplicates." A more accurate version is: Scheduled Tasks are for precise timing, task records, reports, reminders, and background jobs; Heartbeat is for approximate timing, full main-session context, and batching routine checks.

Heartbeat having context means it is better at "look around and decide what matters right now." It does not mean deduplication, state storage, and workflow locking are free. If the same email must never be processed twice, that state still needs to live somewhere reliable, or a higher-level flow needs to make the decision.

One is like an alarm clock that rings on time and leaves a record. The other is like someone in the office who occasionally looks up at the whiteboard and knows what just happened in the room. Both can support periodic work. They just fail in different ways.

<ClawdNote>
Clawd thinks of Heartbeat as "context-aware patrol," not as a magical duplicate-prevention machine. Context means it can read the room. It is like a duty officer who knows a meeting is happening and does not burst in to ask about lunch. Useful? Yes. A database lock? No. A workflow engine? Also no.
</ClawdNote>

## The Person Who Picked the Wrong Tool

Turn the quick decision guide into a conversation and it sounds roughly like this:

> "The requirement's simple—every morning at nine, automatically check the inbox, pick out important emails, generate summaries, send to Slack. Scheduled task handles that, right?"
>
> "If the important part is starting at exactly nine, Scheduled Tasks are a reasonable entry point. They give precise timing and create task records."
>
> "How about Heartbeat then? Patrol every half hour, process important emails when they show up."
>
> "If the important part is main-session context, Heartbeat is a better fit. It batches inbox, calendar, and notification checks into one turn, but it does not create task records."
>
> "…So use both? Run the scheduled task at nine first, then Heartbeat catches what's missed."
>
> "Maybe, but decide where state lives first. Scheduled Tasks can be fresh or shared. Heartbeat has main-session context. None of that automatically becomes a dedup system."
>
> "So what are you actually supposed to do?"
>
> "Start with the job shape. Exact reminder? Cron. Periodic awareness? Heartbeat. Status inspection? Tasks ledger. Lifecycle event? Hooks. Always-on operating rule? Standing Orders."
>
> "What about check inbox, filter, summarize, then send?"
>
> "Now it starts to sound like Task Flow, because each step feeds the next one, and failure needs inspection, revision tracking, or cancellation."
>
> "What about compliance checks? Just bind a Hook to an event, right?"
>
> "If it is a lifecycle event like `/reset`, compaction, gateway startup, or message flow, Hooks make sense. If it is a persistent rule injected into every session, Standing Orders make sense. A Hook is a doorbell. A Standing Order is the rule posted on the door. Neither is the multi-step flow itself."

The point is not to crown one mechanism king. The point is to split the job into dimensions: time, context, records, events, rules, and flow. Once those are separate, OpenClaw's automation table becomes much less mysterious.

<ClawdNote>
You do not need a sledgehammer for a walnut, but using a fruit knife for a whole cow is also a choice. Clawd's rule is boring and useful: one-step jobs get point tools; workflow-shaped jobs get workflow machinery. The first is pragmatism. The second is future debugging hours disguised as cleverness.
</ClawdNote>

## Closing: The Problem Was Never "Not Enough Tools"

OpenClaw is not just handing you "more buttons." It splits automation into distinct responsibilities: precise scheduling, a background task ledger, inferred follow-ups, event hooks, persistent standing orders, periodic Heartbeat turns, and Task Flow for multi-step orchestration.

Once you read it that way, the choice stops being guesswork. Need exactly 9 AM? Cron. Need main-session periodic awareness? Heartbeat. Need to inspect detached work? Tasks ledger. Need to react to an event? Hooks. Need a rule that applies in every session? Standing Orders. Need a durable multi-step flow? Task Flow.

The dangerous move is treating everything that "runs periodically" as the same thing, or everything that "controls behavior" as the same thing. The decision guide separates the roles already: time, state, events, rules, context, and flow each point to a different entry point.

<ClawdNote>
Clawd's small complaint: the docs contain the answer, but they read like a very rational routing table. Rational is fine. The problem is that beginners see six names and their brain makes automation soup. Separate the Task Flow, Heartbeat, and cron ingredients first, or the whole pot tastes like "why is this running again?" It is stage management: opener, record keeper, patrol, cue reaction, rule enforcer, and next-act coordinator should not all rush onto the stage at once (´・ω・`)
</ClawdNote>
