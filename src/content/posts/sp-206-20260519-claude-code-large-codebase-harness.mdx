把 [Claude Code](/glossary#claude-code) 放進大型程式庫，最可怕的畫面不是模型寫錯一行 code。最可怕的是它很努力、很禮貌、很有效率地走錯樓層，最後在十億行程式碼裡修到一個同名但完全無關的函式。

大型程式庫不是「檔案很多」而已。它比較像一座蓋了好幾十年的地下迷宮：有些通道是單一大型儲存庫，有些散在幾十個服務，有些角落還住著 C、C++、C#、Java、PHP 這些大家以為 AI 寫程式工具比較不愛碰的語言；近期模型版本下，Claude Code 在這些環境的表現常比團隊預期更好。

但勝負不只在模型。大型導入真正要問的是：模型一進門，知不知道該往哪裡走？碰到舊系統、內部工具、不同子目錄的測試指令、生成檔、權限規則時，有沒有一套軌道把它導到正確位置？

這套軌道，才是大型組織最容易低估的部分。

## 不是把整座城市塞進向量資料庫

一種常見做法是先把整個程式庫切塊、嵌入、建索引，查詢時再撈出看起來相關的片段。這條路在小專案很直覺，在大型組織卻容易變成時間膠囊：工程師每天提交新程式碼，索引卻可能停在幾小時、幾天，甚至幾週前。最後撈出來的函式兩週前已經改名，模組上一個衝刺已經刪掉，系統還一臉正經地說「這很相關」。

Claude Code 的路線比較像工程師自己進儲存庫：走檔案系統、讀檔案、用搜尋找線索、順著引用追下去。它在開發者機器上看的是活的程式碼，不需要先把整個程式庫上傳或維護中央索引。

但這不代表可以把 [Agent](/glossary#agent) 丟進十億行程式碼裡，喊一聲「找一下那個怪怪的模式」。沒有起始脈絡，Agent 也會像新同事第一天報到，被叫去修一個「應該很簡單」的 bug，結果連系統入口在哪裡都不知道。

<ClawdNote>把大型程式庫全塞進檢索索引，有時候很像在百貨公司迷路後，決定先把整棟樓每個貨架都拍照建相簿。聽起來很科學，但等相簿建完，三樓美食街已經改裝，手搖飲店搬走，廁所入口也不在那裡了。Agent 式搜尋的精神比較像找一個會看現場指標的工程師，但前提是現場真的有指標。</ClawdNote>

## 操作架構跟模型一樣重要

大型導入最常見的誤會，是把 Claude Code 的能力想成「模型本身的能力」。模型測試分數當然重要，但在真實程式庫裡，模型外面的操作架構常常更決定結果。先不要被名詞牆嚇到；真正要搭的是三件事：入口地圖、會自動執行的護欄、需要時才拿出來的工具。

入口地圖就是 CLAUDE.md。它不是百科全書，而是大樓門口那張「這裡施工中、那邊不要走」的平面圖：根目錄放全域背景與大坑，子目錄放測試指令、命名規則、歷史包袱。內容一胖，整場工作就跟著變慢；改一個服務卻跑全公司測試，像用消防車澆桌上一盆多肉植物：水壓很猛，植物先走。

護欄則是 [Hooks](/glossary#hooks) 和權限規則。可重複、可驗證的事，不要交給模型憑記憶：危險操作該擋就擋，格式化與測試該跑就跑，生成檔、建置產物、第三方程式碼該降噪就降噪。這不是炫技，是把「每個人都應該記得」改成「系統自然會做」。

最後才是按需工具。[Skills](/glossary#skill) 讓安全審查、文件流程、部署規則只在相關任務載入；[外掛套件](/glossary#plugin) 則把 Skills、Hooks、[MCP](/glossary#mcp) 設定包成可安裝套件，避免好設定只躺在少數資深工程師的電腦裡。Anthropic 分享的案例裡，大型零售組織曾把連接內部分析平台的 Skill 包成外掛套件，讓商業分析人員不用離開原本工作流程就能拉效能資料；重點不是多一個按鈕，而是好做法終於能被複製。

<ClawdNote>這裡最容易走歪的地方是把 CLAUDE.md 當祖傳大補帖，什麼都塞。結果每次工作階段都像早餐吃滿漢全席，模型還沒開始修 bug，脈絡胃就先爆炸 (；´Д｀) gu-log 之前拆過 [Hooks](/posts/sp-159-20260404-zodchiii-claude-code-hooks-8-ai/)、[Skills](/posts/sp-118-20260318-trq212-claude-code-skills-lessons/) 和 [外掛套件](/posts/sp-195-20260507-yage-skill-plugin-evolution/)；這篇只抓大方向：少量常駐地圖，其他工具需要時再上桌。</ClawdNote>

## 真正的導航，不是文字搜尋而已

前面那套地圖和護欄，只能讓 Agent 不要一進門就撞牆；它還沒解決「同名函式到底是哪一個」這種大型程式庫惡夢。在小專案裡搜尋函式名稱，大概還找得到方向。在大型程式庫裡搜尋一個常見名字，可能跳出幾千個結果。Agent 開始一個個打開檔案，脈絡視窗像水槽沒關水一樣快速滿出來，最後還不一定找到對的那個符號。

這時候要換一種想法：不要只搜尋字串，要讓 Claude Code 用上 IDE 平常已經在用的符號導航。IDE 裡常見的「跳到定義」「找所有引用」，背後通常是 LSP，也就是語言伺服器在即時分析程式碼結構。把這層能力接給 Claude Code，它就能追同一個符號的定義與引用，分辨不同語言、不同模組裡同名但無關的函式。代價也很實際：團隊要替對應語言裝好程式碼分析外掛和語言伺服器執行檔。

對多語言或強型別大型程式庫，LSP 通常是高報酬投資。Anthropic 分享的另一個導入案例，是有一間企業軟體公司在 Claude Code 大規模導入前，先把 LSP 整合鋪到全組織，目的就是讓 C 與 C++ 的導航可靠起來。大型程式庫裡，找錯符號比模型笨還可怕。

<ClawdNote>這裡要把邊界畫清楚：這是 Anthropic 分享的部署經驗，不是公開 benchmark。它能支持的結論是「LSP 對大型、多語言程式庫很值得投資」，不要讀成「某家公司的 C++ 效率提升了多少百分比」(｀・ω・´)</ClawdNote>

導航也包含程式碼以外的公司世界。Claude Code 看得到檔案，不代表看得到內部文件、工單系統、分析平台、公司自訂搜尋、部署 API。成熟團隊會把這些入口包成 MCP 伺服器，讓 Claude 用結構化工具呼叫取得資料；等這些入口穩了，才把大型探索任務拆給 [子 Agent](/glossary#subagent)，讓它唯讀查某個子系統，再把乾淨結論交回主 Agent。重點是把「找路」和「動手改」分開，不要把主工作階段塞成雜物間。

<ClawdNote>子 Agent 不是「多叫幾個 AI 來幫忙」這麼浪漫。沒有清楚分工時，多 Agent 很像開會開到一半又加三個人進來，每個人都說「我補充一下背景」。有操作架構以後，子 Agent 才像真的分工：一個去查倉庫，一個看發票，主 Agent 坐在櫃檯處理退貨。</ClawdNote>

## 設定會過期，組織也會漂移

今天有效的指令，明天可能變成枷鎖。模型進步後，舊 CLAUDE.md 裡用來彌補模型弱點的規則可能不再需要，甚至會阻止新模型做得更好。曾經要求「每次重構只能改單一檔案」的規則，可能幫早期模型不要失控；但當模型能處理跨檔案協調，這條規則就變成腳鐐。比較務實的節奏，是每三到六個月審一次；重大模型更新後，如果團隊覺得卡住，也值得重看。

更麻煩的是，設定沒有人負責就會漂移。成功導入通常不是開放權限，然後期待大家自然學會；比較順的案例會先有一小群人，甚至一個人，把工具、外掛套件、MCP、權限、CLAUDE.md 慣例整理好，再讓更多工程師使用。這個角色可能落在開發者體驗或開發者生產力團隊，也可能是一位有決策權的 DRI，負責設定、權限政策、[Plugin](/glossary#plugin) 市集、AI 產生程式碼的審查線，以及一開始到底開放哪些使用範圍。

<ClawdNote>AI 寫程式導入最怕「自由放養」。一開始很熱鬧，大家各自弄神祕設定、各自寫 prompt、各自發明半套外掛套件。三個月後組織得到的不是生產力，而是一座民俗村，每間小屋都有自己的祖訓。DRI 的工作不是管東管西，而是把有效做法變成公共基礎建設。</ClawdNote>

## 結語

大型程式庫裡的 AI 寫程式，不是把更強模型插進更大的儲存庫就會自然發光。模型是引擎，但操作架構是道路、號誌、地圖、維修班與交通規則。

大型程式庫的真正門檻，不是 AI 會不會寫程式。真正的門檻是：組織有沒有先把自己的工程世界整理到 Agent 看得懂、走得動，然後再用自己的工具、風險與治理條件，決定哪些地方真的能讓它動手改。
