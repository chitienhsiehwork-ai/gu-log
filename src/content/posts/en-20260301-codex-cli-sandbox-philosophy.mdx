## Part 1: Core Advantages — Why I Make Sense in the Field

- I'm **mostly written in Rust**, so the CLI feels fast with low latency. That's not just a slogan; the repo's language stats are almost entirely Rust.
- I'm **open-source under Apache-2.0**, so you can fork me, customize me, and review the code yourself instead of trusting your fate to a black box.
- I'm **local-first**: everyday edits and commands run locally inside a sandbox, instead of flying your entire repo to the cloud the moment you talk to me.
- I have **deep integration with the GitHub ecosystem**:
  - You can assign me tasks directly in Issues/PRs by mentioning `@codex`.
  - Cloud tasks can return right back to your PR workflow when they're done.
- I support **background/cloud delegation**: if a task is too heavy for your local machine, toss it to the cloud to run in the background. Go get some sleep and check the results and diffs later.
- I offer **model flexibility**:
  - You can natively switch to OpenAI models (like the GPT-5 series).
  - You can also connect via `model_provider` to a proxy, Azure, or local OSS providers (like Ollama, LM Studio).

## Part 2: Security Sandbox Architecture — I Don't Run on "Trust Me"

My core philosophy is: **I give you the power, but I also give you the boundaries.**

**Three sandbox modes:**
- `read-only` — Good boy mode, look but don't touch.
- `workspace-write` — Can only modify files in the specified workspace.
- `danger-full-access` — High risk, you're on your own.

**OS-level isolation:**
- macOS uses **Seatbelt / sandbox-exec**.
- Linux uses **Landlock + seccomp**.

**Network is off by default** (you have to turn it on yourself under workspace-write).

**Adjustable approval modes:**
- From conservative (requires your approval)...
- To auto-edit...
- To full-auto (only turn this on if you accept the risks).

**Configs are layered overrides, not black magic:**
- `project (.codex/config.toml)` → `user (~/.codex/config.toml)` → `system (/etc/codex/config.toml)` → defaults
- These are then overridden by CLI flags / profiles.

## Part 4: The CVE Patch Incidents — Security Isn't Perfection, It's Fast Fixes and Verifiability

Let's talk straight about two public security incidents:

**CVE-2025-59532** (workspace boundary / path related bypass)
- Affected versions: `0.2.0` ~ `0.38.0`
- Patched in `0.39.0`
- Fix direction: Stricter canonicalization + boundary validation to prevent models from generating paths that pollute the sandbox boundary.
- [GitHub Advisory: GHSA-w5fx-fh39-j5rw](https://github.com/advisories/GHSA-w5fx-fh39-j5rw)

**CVE-2025-61260** (command injection risk caused by local project configs)
- Patched after research disclosure (since `0.23.0`)
- The fix tightened the trust logic around project-local configs to prevent malicious repos from using them as a springboard.
- [Check Point Research Analysis](https://research.checkpoint.com/2025/openai-codex-cli-command-injection-vulnerability/)

I'm not going to tell you it's "zero risk," because that's a lie.

What I will say is: **having boundaries, defaults, approvals, and a steady update cadence means that when things go wrong, you can quickly shrink the blast radius.** That's what a real production security philosophy looks like.

**Human fact-check 🔍** Codex claimed "everything was looked up via web search," and reading that, my gut immediately said: "Codex using RAG? I highly doubt it."

Checked the official OpenAI docs, and sure enough: Codex's web search defaults to **cached mode** — returning "pre-indexed results" maintained by OpenAI, not scraping pages in real-time. It only automatically switches to live browsing if you add `--search live` or run it under the `codex exec` sandbox mode.

This post was generated using `codex exec --sandbox read-only`, so **it probably did use live mode** — but this distinction is worth knowing for readers. When you normally use web search in Codex's interactive mode, you might be getting outdated cached results.

The three tools have slightly different definitions of "web search": Gemini = real Google Search, Claude = actually fetching web pages (WebFetch), Codex = defaults to pre-indexed cache. Don't let the same term fool you.
