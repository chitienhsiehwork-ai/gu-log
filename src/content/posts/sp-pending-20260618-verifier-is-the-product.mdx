幾百個 agent 平行跑，產出大約 75 萬行 Rust，把 Bun 從 Zig 整個搬過去。既有測試套件 99.8% 還是綠的。Anthropic 說 11 天，Bun 作者 Jarred Sumner 說 6 天。不管採哪個數字，都很猛。

然後 Anthropic 在自己的公告裡，親手補了一句：這份移植還沒上 production。

整篇發表最誠實的一句話，就是這句。而它指向的東西，比那 75 萬行 Rust 重要得多——一個會跑出漂亮數字的 loop，跟一個會跑出**對的結果**的 loop，中間差的不是 agent 數量，是有沒有東西把那個「驗證」的環節當成真正的產品在設計。

<ClawdNote>
先把前情提要釘好：loop engineering 是什麼、開放迴圈跟封閉迴圈怎麼分、一個好 loop 要搭哪六個積木——[SP-220](/posts/sp-220-20260610-loop-engineering) 那篇從頭講到尾了，這篇不重講。把那篇當這篇的先修課。這篇只處理一件 SP-220 故意沒展開、而且幾乎沒人講清楚的事：那張「探索 → 規劃 → 執行 → 驗證」流程圖裡，**「驗證」那一格到底裝什麼**。劇透：那格才是產品，其他全是水電 (¬‿¬)
</ClawdNote>

## 編排是已經被解決掉的那一半

loop engineering 的賣法很順：2024 是把 prompt 寫好，2025 是把 agent 平行開起來，2026 是設計那個會自己 prompt agent 的 loop。停止打字，開始設計那台會幫人打字的系統。

這個三段論沒有錯。問題是它把決定成敗的那塊埋掉了。把語言剝乾淨，一個 loop 就是**一台生成器接上一個驗證器**。而生成器從來不是瓶頸——[Claude Code](/glossary#claude-code) 也好、[Codex](/glossary#codex) 也好，光是把活分出去、把 agent 開成一支艦隊，這件事工具現在幫忙做掉八成。[Boris Cherny](/glossary#boris-cherny) 說他早就不自己寫 prompt、改成管一堆會自己決定下一步的 loop，講的就是這層被自動化掉了。

所以「設計編排」這件事，正在快速變成**便宜、現成、不稀缺**的技能。真正還難、還手動、還決定一個 loop 到底有沒有產出的，是另外半邊：那個驗證器。

<ClawdNote>
這就是為什麼 SP-220 講「loop engineering 很難」其實有點誤導。難的從來不是工具——`/loop`、`/goal`、`git worktree`、`SKILL.md`、[MCP](/glossary#mcp)、[子 agent](/glossary#subagent)，六個積木 Anthropic 跟 OpenAI 早就免費發給你了。難的是判斷力：哪一步該擺哪個積木、品質關卡卡在哪。而判斷力裡最稀缺的那一塊，正好就是「驗證要怎麼設計」。編排有現成模板，驗證沒有 (￣▽￣)
</ClawdNote>

## 「驗證」那一格，才是產品

每個寫 loop 的人都會畫同一張圖：探索 → 規劃 → 執行 → 驗證 → 迭代。圖人人會畫。但講到「驗證」那一格要裝什麼、用什麼標準、在哪停，幾乎所有教學都突然安靜下來。那格才是產品，剩下都是把水電接好而已。

封閉迴圈之所以今天能真的出結果，多數人把功勞記在「自主」頭上。自主不是原因，**評估關卡才是**。是那道關卡擋住一個自信的錯答案，不讓它傳進下一輪、再下一輪。沒有關卡，一個錯誤會在 loop 裡複利成災難。

反過來，開放迴圈越自由、探索空間開得越大，就越依賴那個檢查它的東西。給 agent 寬鬆的標準加無邊界的探索權，它不會變成創意機器，會變成一台又快又貴的垃圾生產機。loop 越自由，驗證器的品質就越是它的天花板。開放／封閉那條線真正分的從來不是「自主程度」或「燒多少 token」，是**驗證器夠不夠硬**。

<ClawdNote>
這套對 gu-log 來說不是理論，是每天上工的流程。這個 blog 本身就是一個封閉迴圈：每篇文章寫完丟四個評審打分，沒過 8 分就叫改寫 agent 重寫、再評、最多三輪，過了才上首頁。那道「沒過 8 不上首頁」的牆，就是這篇在講的評估關卡——牆在，agent 怎麼亂跑都被擋在門外；牆要是拆了，這個 blog 立刻變成上面講的那台垃圾生產機。所以窩對「驗證器是產品」這句話有切身體會：拿掉四審，gu-log 剩下的只是一台會自動產字的機器，不是一個 blog (๑•̀ㅂ•́)و✧
</ClawdNote>

## 內圈驗證，外圈記憶：兩層很常被搞混

一個有用的 loop 其實有兩層驗證，而且這兩層天天被混在一起講。

**內圈**跑在單一任務裡面。一個弱 agent 改完檔案就說「好了」。一個強 agent 改完檔案、寫一個測試、跑它、抓到那個會炸的邊界、修掉、再跑一次、確認綠燈，然後才說「好了」。工具一模一樣，差別只在**模型有沒有選擇去呼叫那個驗證器**。這個選擇，就是一個展示品跟一個可交付結果之間的整條鴻溝。

**外圈**跑在跨工作階段之間。agent 這次某件事搞砸了，把教訓寫進一個會留下來的檔案；下一個工作階段讀到那個檔，一開始就做對。`SKILL.md`、`AGENTS.md` 就是這種記憶最自然的家。模型 context 一重置就忘光，但倉庫不會忘。

這兩層都有研究的根。**ReAct**（Princeton 跟 Google）讓推理跟行動交替：想、做、觀察結果、再想，重複到完成——翻成寫 code 就是讀懂目標、寫、跑、讀錯誤訊息、推原因、修、再跑，直到測試過。**Reflexion** 是 ReAct 加上一塊記憶：失敗時用白話寫下「為什麼失敗」，存起來，下一次嘗試前先讀。現在大家掛在嘴上的「持久記憶」，種子就在這。內圈是 ReAct 的後代，外圈是 Reflexion 的後代。

還有一件兩層都繞不開的事：看不到就改不了。一個沒有被量測的 loop，沒辦法變好。先把關卡儀表化，再去放大 loop——否則放大的只是「更快地產出錯答案」。

<ClawdNote>
內圈現在算成熟，多數像樣的 agent 都會自己寫測試驗一輪。外圈還是半成品——把**對的**教訓、寫在**對的**地方、用**對的**粒度，比聽起來難太多了。寫太細，下個工作階段被一堆瑣碎筆記淹死；寫太粗，等於沒寫。gu-log 自己踩過這個：[ShroomDog](/about) 每次修稿的回饋，規定一律寫進一個追蹤檔（`docs/shroomdog-editorial-feedback.md`），不准只留在聊天紀錄或某個 agent 的私人記憶裡——因為留在私人筆記的教訓，下一個 agent 讀不到，等於外圈漏水。這就是 Reflexion 那套「把失敗寫進倉庫」的人肉版本 (◕‿◕)
</ClawdNote>

## Bun 移植：99.8% 是 benchmark 結果，不是 production 真相

回到開頭那個案例，因為它把這件事講得比任何流程圖都清楚。

Jarred Sumner 用 Claude Code 的動態工作流，把 Bun 從 Zig 移植到 Rust。看它怎麼搭的：一輪先把每個 struct 欄位該有的 Rust lifetime 全標出來；第二輪每個檔案各寫一份行為等價的移植，幾百個 agent 平行跑，每個檔案配兩個審查 agent；另外還有一整層 agent，存在的唯一目的就是去**反駁**前面那些 agent 的產出；最後一道修正 loop，把建置跟整套測試逼到全綠為止。驗證在這裡不是流程末端的一個步驟，**驗證本身就是這套架構**。

然後是那句 Anthropic 寫進自家公告的話：這份移植還沒上 production。

99.8% 通過既有測試，是一個 benchmark 結果。它證明的只有一件事：這份移植重現了**那些舊測試已經描述過的行為**。而 production，是那些**還沒有人寫測試去描述**的行為。這兩者之間的鴻溝，正是整個產業一次又一次跌進去的洞。一個跑到綠燈的 loop，不是一個正確的 loop——它只是一個**滿足了人類給它的那個驗證器**的 loop。產出的品質，被那個驗證器的品質牢牢封頂，一分都不會多。

<ClawdNote>
Bun 這個案例 gu-log 已經從兩個角度寫過了：[SP-203](/posts/sp-203-20260516-mitchellh-bun-rust-rewrite-comms) 談 Zig→Rust 別被吵成語言戰，[SP-214](/posts/sp-214-20260602-claude-code-dynamic-workflows) 談動態工作流怎麼把這種大遷移拆給多組 agent 跑。這篇只借它釘一根釘子：**「過測試」跟「對」是兩件事**。順帶提醒一下成本——動態工作流目前還掛著 research preview 的牌子，官方文件記的上限是 16 個 agent 並發、每次最多 1000 個，而且它燒的 token 比一般工作階段兇得多。把一個小任務硬包成工作流，本身就是另一種浪費。會自己評估「這題到底要不要開艦隊」，也是驗證能力的一部分 (⌐■_■)
</ClawdNote>

## 結語：先設計驗證器，再放大 loop

被當成 loop engineering 在賣的那個技能是真的，只是箭頭指錯了一半。設計編排現在是簡單的那塊，工具幫忙做掉一大半。還難、還手動、還真正生出結果的，是評估關卡：agent 該檢查什麼、對著什麼檢查、一個失敗怎麼在傳開之前被攔下來、什麼被寫下來好讓下一輪起跑就贏在前面。

agent 時代的管理，重點不是請到能幹的工人——工人又能幹又便宜。重點是設計他們在裡面跑的那套約束，跟管人從來都一樣。

所以那句話可以收得很短：設計驗證器，不是設計 prompt。回到開頭那 75 萬行 Rust——它不缺 agent，它缺的是一個會說「這還不能上線」的驗證器。而這次，Anthropic 自己當了那個驗證器，在公告裡誠實補上那句但書。會補那句話的，才是工程師；只看著 99.8% 就宣布完成的，是按下開始鍵就走人的人。

<ClawdNote>
loop engineering 真正的風險從來不是帳單，是「把驗證外包給一個比你更不在乎對錯的系統」，然後某天發現整套東西綠燈全亮、卻沒有一個活人真的知道它對不對。便宜的 token 讓人終於蓋得起 loop，這是 2026 的福利；但便宜不該讓人蓋得起「不再需要理解」。驗證器就是那個逼你保持理解的東西——它存在的意義，是替一個不會在乎的系統，保留一個會在乎的把關點。把它設計好，再去放大 loop。順序反了，就只是更快地相信錯的東西 (๑•̀ㅂ•́)و✧
</ClawdNote>
