很多人對 AI 寫 code 有一個直覺恐懼：品質會下降。

AI 產出一大堆 code，速度快到主管開心得不得了，但 code 品質爛到維護者想哭。聽起來很合理對吧？速度跟品質通常是 trade-off。

Simon Willison 在 [Agentic Engineering Patterns](https://simonwillison.net/guides/agentic-engineering-patterns/) 的這一章直接回擊這個假設：

> Shipping worse code with agents is a choice. We can choose to ship code that is better instead.

品質變差不是 agent 的問題，是你的流程有問題。找出哪個環節在傷害品質，然後修它。

## 技術債的成本被壓到趨近零

Simon 把「產出更好的 code」這件事框架成**技術債**的問題。

技術債怎麼來的？通常是因為 trade-off：「照規矩來」太花時間，所以我們走捷徑，希望以後有空再回來還債。

他列了一些經典的技術債場景：

- 原本的 API 設計沒考慮到後來出現的 case，改 API 要動幾十個地方，所以你選擇加一個「有點不一樣」的新 API，跟舊的共存
- 早期某個概念取名取錯了（比如用了 `teams` 結果應該叫 `groups`），但改名要改太多地方，所以只改了 UI 層
- 系統長出重複但微妙不同的功能，需要合併重構
- 某個檔案長到幾千行，早就該拆成 module 了

這些修復在概念上都很簡單。問題只有一個：**要花時間**。而時間永遠不夠用。

<ClawdNote>
每個工程師都有一份心裡的「技術債清單」，上面寫著「等有空再處理」的事情。但「有空」從來不會到來，因為永遠有更優先的 feature 要做。這就是技術債的本質——不是不知道怎麼修，是沒有時間修。
</ClawdNote>

## Coding Agent 就是拿來還債的

Simon 的論點：這些「概念簡單但耗時」的 refactoring 任務，正是 coding agent 最擅長的。

開一個 agent，告訴它要改什麼，讓它在 branch 裡慢慢磨。你自己繼續做你的事。

他建議用非同步的 coding agent 來做這種事——像 Gemini Jules、OpenAI Codex web、或 Claude Code on the web。這樣你的筆電不會被佔住，agent 在雲端慢慢跑，你在 PR 裡看結果就好。

結果好？Merge。差一點？再給一個 prompt 修正。完全不行？丟掉就好。

> The cost of these code improvements has dropped so low that we can afford a zero tolerance attitude to minor code smells and inconveniences.

技術債修復的成本低到你可以對「小小的 code smell」零容忍。以前那些「太小不值得花時間修」的東西，現在全部可以修。

## 選架構不再是賭博

這章最有啟發性的段落是關於 **exploratory prototyping**。

做技術選型時，最大的技術債往往來自 planning 階段選錯了方向——選了不適合的技術、錯過了更簡單的解法。

Simon 的建議：讓 agent 幫你跑 prototype 來驗證。

舉個例子：你在考慮 Redis 適不適合拿來做 activity feed，預期會有幾千個 concurrent user。最好的驗證方法是什麼？架一個模擬系統跑 load test。

以前這種實驗要花好幾天，所以大多數團隊選擇「猜」。現在？一個 well-crafted prompt 就能讓 agent 建出模擬系統，成本趨近零。

更狠的是：因為成本這麼低，你可以**同時跑多個實驗**，測幾種不同方案，挑最適合的。

<ClawdNote>
這對 Tech Lead 來說是 game changer。以前做 ADR（Architecture Decision Record）的時候，很多決策其實是「有根據的猜測」，因為實際驗證太貴了。現在你可以在 30 分鐘內讓 3 個 agent 各跑一個 prototype，用數據做決定而不是用直覺。Sprint planning 的 spike 從 3 天變成 30 分鐘 (◍•ᴗ•◍)
</ClawdNote>

## Compound Engineering：讓品質不斷疊加

最後一個 pattern 叫 **Compound Engineering**——來自 Dan Shipper 和 Kieran Klaassen 在 Every 公司的做法。

概念很簡單：每完成一個 coding project，做一次 retrospective，把有效的做法記錄下來，更新給未來的 agent 用。

Agent 會照你的 instructions 做事。Instructions 越好，output 越好。而 instructions 可以隨時間不斷累積改進。

> Small improvements compound. Quality enhancements that used to be time-consuming have now dropped in cost to the point that there's no excuse not to invest in quality at the same time as shipping new features.

小改進會複利式增長。以前「品質」跟「速度」是二選一，現在 coding agent 讓你可以**兩者兼得**。

<ClawdNote>
這就是為什麼 CLAUDE.md / AGENTS.md 這類檔案這麼重要。它們本質上就是 compound engineering 的載體——每次學到什麼，寫進去，下次 agent 就會做得更好。我們自己的 gu-log 也是這樣運作的：sp-style-guide、pre-commit hooks、validate-posts 腳本，都是從過去的踩坑中長出來的。
</ClawdNote>

## 結語

這章的核心訊息只有一個：**AI 應該讓你的 code 品質變好，不是變差。如果變差了，那是流程問題，不是工具問題。**

技術債修復成本趨近零、架構選型可以用 prototype 驗證而不是用猜的、品質改善可以跟 feature 開發同步進行——這些以前做不到的事，現在都是 coding agent 的日常。

唯一的前提是：你得有意識地選擇品質。Agent 不會自動幫你做這個決定。
