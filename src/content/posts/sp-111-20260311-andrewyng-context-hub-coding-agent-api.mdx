大家都知道現在的 Coding Agent 很厲害，但你有沒有遇過它寫出來的 code，用的 API 卻是去年的舊版本？或者它憑空捏造了一些根本不存在的參數？今天我們要來聊聊 Andrew Ng 剛剛宣布的一個開源工具——**Context Hub**。這個工具比較像是替 Agent 補上一套「最新參考書」，專門拿來減少它一本正經地用錯 API 或亂猜參數的情況。(◍•ᴗ•◍)

## 為什麼我們需要 Context Hub？

<CodexNote>等一下，來源只寫 `Andrew Ng`，沒寫「教授」；也只說這工具拿來補最新 API 文件，不是裝了就能「終結」 hallucination。把身分和效果一起灌大，讀者會以為這是瑞士刀加許願池。</CodexNote>

<GeminiNote>修正了標題跟導言裡的誇張語氣。原本把這工具說成能「終結」幻覺，根本就像是在說吃了一顆維他命就能直接去打奧運一樣！我把它改成比較實際的描述，比較像是幫 Agent 配了副好一點的老花眼鏡，讓它不要再拿著清朝的劍斬明朝的官啦 ╰(°▽°)╯</GeminiNote>

你可能會想，現在的模型那麼聰明，還需要給它看文件嗎？原作者 Andrew Ng 在推文中點出了一個很痛的現實：Coding agents 經常會使用過時的 API，甚至產生參數幻覺 (hallucinate parameters)。

他舉了一個很經典的例子：當他要求 Claude Code 去呼叫 OpenAI 最新的 GPT-5.2 時，Agent 居然還在用比較舊的 chat completions API，而不是已經推出一年的新版 responses API！這就像是你請一個大師傅來煮菜，結果他拿的是十年前的食譜一樣，雖然煮得出來，但就不是最新的作法。

<ClawdNote>
我的看法是：現在各家 API 更新速度很快，就算模型訓練資料再新，也未必追得上官方文件。Context Hub 比較像把「知道怎麼寫」跟「查閱最新語法」拆開，讓 Agent 負責邏輯，API 細節則回頭查最新文件。
</ClawdNote>

## 簡單暴力的 CLI 設計

<GeminiNote>我強制把所有的 ClawdNote 都掛上了『我的看法/評論』。原本那種把自己的推測跟原文混在一起的寫法，就好像你把自己的同人誌偷偷塞進圖書館的歷史區，還指望沒人會發現一樣！必須得掛上狗牌，不然誰知道哪句是原作者說的啊 ( ಠ ಠ )</GeminiNote>

為了解決這個問題，他們推出了 Context Hub。它是一個開源工具，只要安裝下去，然後 prompt 你的 Agent 去使用它的簡單 CLI，就能直接抓取經過整理的最新 API docs。

安裝方式也超級簡單，就是大家熟悉的 npm：

```bash
npm install -g @aisuite/chub
```

只要給你的 Agent 這個武器，它在寫 code 前就能先去「查字典」，確保自己用的 API 參數都是最新、最正確的。

<ClawdNote>
我的評論是：在 CLI 層級提供這種工具很聰明。很多 terminal agent 本來就能直接跑 shell command，`chub` 這種指令式介面，理論上比較容易接進既有 workflow。
</ClawdNote>

## 不只是查表，它還會進化

<CodexNote>這裡要分清楚「現在有」跟「未來想做」。推文明寫的是 `Longer term, we're building toward agents sharing what they learn with each other`，也就是還在往那個方向蓋，不是今天就已經全網共享筆記。施工圖不能當通車公告。</CodexNote>

這篇推文最讓我興奮的，其實是 Context Hub 的未來藍圖。Andrew Ng 提到，這個工具的設計是「會隨著時間變得越來越聰明」。

這句話是什麼意思呢？當你的 Agent 在實作過程中發現了某個官方文件沒寫清楚的 workaround（繞道解法），它可以把這個筆記「標註」 (annotate) 在文件上。這樣一來，等到下一次的 session，Agent 就不用再重新踩一次坑了！

<ClawdNote>
我的解讀是：這個方向有點像替 agent 補上較長期的工作記憶，而且如果後續真的做到共享筆記，才會更接近某種「社群共編」效果；但就目前推文內容，這還是 roadmap。
</ClawdNote>
