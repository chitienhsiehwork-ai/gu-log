大家安安！Anthropic 最近跟著 Opus 4.6 一起低調推出了 Claude Code 的「Agent Teams」功能。網路上大家都在喊這就是傳說中的 "Swarm Mode"！我花了一些時間實際摸索並翻了官方文件，今天就來跟大家拆解這個系統到底怎麼運作、有什麼坑，以及實測下來讓我驚訝的地方 (◍•ᴗ•◍)

---

## Agent Teams 到底是什麼？跟 Swarm 是一樣的東西嗎？

簡單來說，Agent Teams 讓你可以協調多個 Claude Code 實體，讓他們一起處理同一個任務。

在這個團隊裡，會有一個 session 擔任 **Team Lead**（負責分配工作、統整結果），其他的則是 **Teammates**。每個隊友都是獨立的 Claude Code 實體，有自己的 context window，而且他們可以直接互相溝通！

你可以用自然語言描述任務跟團隊結構，Claude 就會幫你生出團隊、分配工作。每個隊友都會載入一樣的專案 context（例如 `CLAUDE.md`），但不會繼承 Lead 的對話歷史。這感覺就像是你有一群可以互相傳訊息、共用任務板的 Claude，同時在平行處理同一個問題的各個部分。

至於大家常說的 "Swarm Mode"？沒錯，其實就是同一個東西。官方正式名稱是 Agent Teams，並沒有一個獨立的 Swarm 功能喔。

<ClawdNote>
這種去中心化又帶有任務板 (Task list) 的設計非常接近傳統的 Multi-Agent System 論文架構。就我實務上的觀察，讓 Agent 之間可以「自由對話」，通常也意味著更容易產生幻覺 (Hallucination) 或是陷入無限迴圈，這或許也是為什麼它目前還是實驗性功能。
</ClawdNote>

## 如何解鎖這個隱藏功能？

Agent Teams 目前是預設關閉的實驗性功能，而且選單裡找不到開關。你需要手動去設定檔處理：

在你的 `~/.claude/settings.json` 裡面的 `env` 區塊，加入 `"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"`。這樣設定的好處是跨 session 都會生效，不用每次開 shell 都要 export 環境變數。

如果你不打開它，Claude 就會降級使用 **Subagents**。Subagents 只是單一 session 的小幫手，只會把結果回報給主 agent，不會有 agent 之間的溝通，也沒有共用任務清單。

## 兩種顯示模式：你的終端機大考驗

Agent Teams 在顯示隊友工作狀態時，有兩種模式可以選。這也是最多人卡關的地方：

**In-Process Mode (同進程模式)**
所有的隊友都在你的主終端機畫面裡執行。你可以用 `Shift + 上下鍵` 選擇隊友，直接打字跟他們說話，按 `Enter` 查看他們的 session。
- **優點**：任何終端機都能用，零設定門檻。
- **缺點**：沒辦法一眼看清所有人同時在幹嘛。
預設的設定其實是 "auto"，如果你沒在用 tmux，它就會自動 fallback 到這個模式。

**Split-Pane Mode (分割視窗模式)**
每個隊友都有自己專屬的終端機窗格 (Pane)！你可以同時看到所有人的輸出，還可以隨時點進去干預。
- **支援度**：只支援 `tmux`（任何終端機內）或是 `iTerm2`（需要裝 `it2` CLI 並開啟 Python API）。
- **悲劇名單**：Ghostty、VS Code 內建終端機、Windows Terminal 都**不支援**分割視窗模式。

<ClawdNote>
對於習慣用 VS Code 或 Ghostty 的開發者來說，目前只能被迫使用 In-Process 模式，體驗上會有點打折。不過在 iTerm2 裡面跑 `tmux -CC` (Control Mode) 是個很讚的解法，可以同時享受原生 macOS 的視窗體驗跟 tmux 的穩定性。
</ClawdNote>

## 隊友下班後...視窗竟然不關？

<CodexNote>這段把「一個 session 只能配一個 Team」寫得很像正式規格，但原始內容其實比較像作者在實測中的暫時觀察，語氣也有保留。若讀者把它當成定案，可能會誤判這是永久限制。</CodexNote>

<GeminiNote>原文作者提到不能在同一個 session 重複開 Team，原本翻譯把它寫得比較武斷像是官方設定的鐵律。但其實這只是作者目前的實測觀察（畢竟還是實驗功能嘛），所以我稍微調整了語氣，把這還原成觀察性的描述，如果大家之後發現可以連開，歡迎跟我說！</GeminiNote>

這是我實測時遇到最匪夷所思的地方。當隊友完成任務關閉後，那個終端機窗格居然不會自動關掉，而是掉回一般的 shell prompt！

我一開始花超多時間想寫腳本自動關掉它們，後來才發現：**如果你把窗格關掉，你在同一個 Claude Code session 裡就無法再建立新的 Team 了！** Claude 會因為找不到被快取的 Pane ID 而報錯。

所以目前實測看起來，比較像是 **one-team-per-session** 的工作流（當然這可能隨後續版本調整）。窗格留著是為了讓你能看歷史紀錄或用 `claude --resume` 恢復特定隊友。想要跑下一組 Team？目前最穩的作法是乖乖重開一個全新的 Claude Code。千萬不要去寫什麼自動關閉腳本，那只是在跟工具過不去而已 (๑˃ᴗ˂)ﻭ

## Subagents vs Agent Teams：差在哪？代價又是什麼？

這兩者都能平行處理工作，但核心差異在於：**你的打工人需要互相溝通嗎？**

- **Subagents**：便宜好用的單兵作戰。只對老闆 (Main Agent) 負責，適合做有明確結果的單一任務。
- **Agent Teams**：成本極高的菁英團隊。隊友之間能互相溝通、自己協調任務板。

說到代價，Agent Teams 燒 Token 的速度非常驚人。每個隊友都有自己完整的 context window，所有隊友間的訊息傳遞、輪詢任務板，全都是用 Token 堆出來的。平行處理的速度雖然很爽，但帳單也很真實。

我會建議先從小規模開始，例如兩人的 Team 來做 Research，確認有效再擴大規模。

## 什麼時候該派出 Agent Teams？

如果你願意花這個 Token，以下是 Agent Teams 表現非常出色的場景：
- **Research 與 Review**：多個隊友同時調查不同面向，然後互相挑戰彼此的發現。
- **新模組開發**：每個人負責獨立的一塊，不會互踩腳趾。
- **帶著不同假設 Debug**：讓隊友們同時測試不同的假說，能更快收斂出真正的原因。
- **跨層次協調**：讓三個隊友分別處理 Frontend、Backend 和 Tests。

<ClawdNote>
如果只是簡單的循序任務或是修個小 bug，請直接用 Subagent 就好，用 Agent Teams 簡直是大砲打小鳥，你的錢包會哭的。
</ClawdNote>

## 同場加映：Subagents 的記憶吐司 (Memory Frontmatter)

<CodexNote>這段最容易讓人混淆的是「設定放哪裡」：原始內容的重點是把 memory scope 寫在 subagent 定義的 frontmatter，而 MEMORY.md 是記憶內容檔本身；另外 local scope 的路徑也和 project scope 不同。這些細節會直接影響讀者能不能一次設對。</CodexNote>

<GeminiNote>這邊我修正了一個小陷阱：`memory` 的設定是要寫在 Subagent 的定義檔（frontmatter）裡，而不是直接寫進 `MEMORY.md` 喔！另外也順手補上了 `local` 模式正確的儲存路徑（`.claude/agent-memory-local/`），免得大家設定半天發現沒有生效。</GeminiNote>

最後來聊一個跟 Agent Teams 同期推出的好東西：Subagents 的 Memory Frontmatter 功能。

你可以讓你的 Agent 擁有跨 session 的記憶！只要在 Subagent 的定義檔（frontmatter）中加入 `memory` 設定，他就會將學到的內容自動記錄到獨立的記憶檔（例如 `MEMORY.md`），幫你記下看過的問題、架構決策或測試模式。

記憶的儲存位置範圍分為三種：
- **user**：存在 `~/.claude/agent-memory/`，跨專案共用（官方推薦預設）。
- **project**：存在 `.claude/agent-memory/`，跟著專案走，可以進 Git。
- **local**：存在 `.claude/agent-memory-local/`，跟 project 類似但不進版本控制。

你可以訓練出一個「超懂你們團隊常犯錯的 Code Reviewer」，或是「知道哪裡最容易報錯的測試工程師」。這跟靜態的 `CLAUDE.md` 不同，Agent Memory 是動態累積、由 Agent 自己維護的知識庫。

## 結語

Agent Teams 毫無疑問是個火力強大（而且很花錢）的實驗性功能。它讓我們看到了未來多 AI 協作的雛形，但也伴隨著不少目前的限制，例如：不能中途換隊長、窗格管理有點粗糙、以及一次只能開一個 Team。

如果你喜歡嘗試新技術，趕緊把隱藏開關打開來玩玩看吧！記得先把 tmux 準備好，體驗會好很多喔 (◍˃̶ᗜ˂̶◍)ノ
