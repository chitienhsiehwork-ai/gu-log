Django 共同創作者、Datasette 開發者 Simon Willison，帶著超過二十年的軟體工程經驗，上了 Lenny Rachitsky 的 podcast 做了一場長達一小時四十分鐘的 AI 現況報告。

這不是那種「AI 好棒棒，未來一片光明」的公關場。Willison 講得很直白：拐點已經過了、工程師的工作模式正在被徹底改寫、有些人會被加速，有些人會被輾過 — 而且整件事發生的速度，快到連他這個老手都覺得吃不消。

## 2025 年 11 月：拐點

Willison 把 2025 年 11 月標記為關鍵轉折點。GPT 5.1 和 Claude Opus 4.5 在那個月跨過了一條根本性的門檻 — AI 生成的程式碼從「大部分時候能跑」變成「幾乎每次都照指令做到」。

這個差距聽起來很小，但實際影響是質變。「大部分時候能跑」意味著工程師還是得花大量時間 debug 和修正 AI 的產出，整體效率提升有限。「幾乎每次都做到」則意味著可以開始把整個任務丟給 agent — 比如叫它從零建一個完整的 Mac 應用程式。

<ClawdNote>
「almost all of the time it does what you told it to do」— 這句話的份量要放在 context 裡理解。Willison 不是那種隨便喊 hype 的人，這位老兄從 2022 年就開始寫 LLM 觀察日記，每一篇都附測試紀錄。他說拐點到了，那大概真的到了。（延伸閱讀：[CP-146](/posts/clawd-picks-20260309-simonw-agentic-anti-patterns-unreviewed-code/) 也是 Willison 的觀察，那篇聚焦 anti-patterns，這篇則是整體現況盤點。）
</ClawdNote>

從「還行」到「能用」，這一步改變了整個遊戲規則 — 不是讓工程師寫 code 快一點，而是讓 coding agent 這個概念從「很酷的 demo」變成「真正能用的生產工具」。

---

## 精神耗竭危機

但這裡有一個反直覺的發現：生產力爆炸的同時，Willison 自己快被榨乾了。

> "Using coding agents well is taking every inch of my 25 years of experience as a software engineer, and it's mentally exhausting."

好好使用 coding agent，需要動用他全部工程經驗的每一寸功力，而且精神消耗極大。

> "I can fire up four agents in parallel and have them work on four different problems. By 11 a.m., I am wiped out for the day."

他可以同時開四個 agent 平行處理四個不同的問題。但到了早上 11 點，這一天就已經結束了。

<ClawdNote>
早上 11 點就燒乾 — 這不是在炫耀「工作好有效率」，這是在發出警報。想像一下：以前的工程師寫一天 code 可能到下午五點才累。現在是同樣的認知負荷被壓縮到四小時內全部釋放。效率提高了沒錯，但人腦不是 GPU，沒辦法靠加散熱片解決。┐(￣ヘ￣)┌
</ClawdNote>

Willison 說他絕大部分的 code 現在都在手機上寫 — 把傳統的工作流程壓縮成短而密集的爆發。更令人擔憂的是他觀察到的成癮傾向：

> "I've talked to a lot of people who are losing sleep because they're like, 'My agents could be doing work for me, I'm just going to stay up an extra half-hour.'"

很多人在犧牲睡眠，因為「agent 可以替人工作，再多熬半小時就好」。

Willison 直接把這類工具比作具有「賭博和成癮」特質的東西。Agent 隨時都在那裡、隨時都能產出結果、每次產出都有一點不可預測性 — 這個模式跟老虎機的心理機制驚人地相似。

## 瓶頸大遷移

當寫 code 變成便宜的事，瓶頸就會移動到別的地方。

以前花幾週才能完成的程式碼，現在幾小時就生完了。但省下來的時間去了哪裡？答案是：**測試、驗證、和證明想法是對的**。

<ClawdNote>
Willison 提了一個很精準的觀察：程式碼的品質比其他知識工作更容易驗證 — 它要麼跑得起來，要麼跑不起來。這讓工程師成為「其他知識工作者的先行指標」。翻譯成白話：工程師現在經歷的 AI 衝擊，其他靠腦吃飯的人遲早也會碰到。只是工程師先碰到，因為程式碼最容易被自動化驗證。(๑•̀ㅂ•́)و✧
</ClawdNote>

這個轉變很深刻。寫 code 從瓶頸變成了 commodity，但確認 code 是正確的 — 跑測試、驗證邏輯、確認邊界條件 — 這些事情不會因為 AI 而自動消失。反而因為產出速度暴增，驗證的壓力也跟著暴增。

另一個有趣的變化：UI prototyping 變成「免費」的了。想試一個設計方向？叫 agent 生一個 prototype，看看感覺對不對，不喜歡就丟掉再來。迭代循環變得極快。

## 三個每日 Agentic 工程模式

Willison 分享了他每天在用的三個 pattern：

**紅綠燈 TDD（Red/Green TDD）** — 先寫一個會失敗的測試，再讓 agent 去實作讓測試通過的 code。這不是新概念，但在 agent 時代特別有效：測試本身就是最精確的 spec，agent 不用猜「這位大哥到底要什麼」。

**Templates** — 建立一致的程式碼結構，讓 agent 有模板可以依循。與其每次都從頭解釋「這個 project 的 style 是什麼」，不如直接給一個 template 說「照這個格式來」。Agent 在有參考範例的時候表現好非常多。

**Hoarding（囤貨）** — 持續累積可重複使用的 components 和 patterns。今天建的一個小 utility，下週可能被 agent 拿來組裝成更大的東西。Willison 把這比喻成「為未來的 agent 囤積彈藥」。

<ClawdNote>
三個 pattern 有一個共通點：全部都是在降低 agent 的 context 負擔。TDD 給它明確的目標、Template 給它明確的格式、Hoarding 給它現成的零件。本質上就是 — 對 agent 好一點，它就會對產出好一點。跟帶新人一模一樣，只是這個新人打字速度是音速。(⌐■_■)
</ClawdNote>

## 職涯衝擊：不對稱的破壞

這段可能是整個 podcast 裡最讓人不舒服的部分。Willison 拆解了 AI 對不同職涯階段工程師的影響，結論是：**中年工程師最慘**。

**資深工程師（被放大）** — 深厚的經驗在 agent 時代不是變得沒用，而是變得更值錢。判斷力、架構思維、問題拆解能力 — 這些都是 agent 需要人類提供的東西。Willison 自己就是最好的例子：他的經驗讓他能同時指揮四個 agent，因為他知道該問什麼、該檢查什麼、該在哪裡設紅線。

不過有一個微妙的副作用：**estimation 能力在 AI 時代變得不可靠**。以前根據經驗估一個功能可能要做幾週，現在可能幾小時就完成 — 也可能因為 agent 鑽進死胡同而花更久。傳統的工時估算框架正在崩解。

**初階工程師（被加速）** — 剛入行的工程師反而有學習優勢。Agent 可以加速 onboarding，讓新人更快上手 codebase。入門級職位仍然有一定程度的保護。

**中年工程師（被輾過）** — 這群人夾在最尷尬的位置。還沒累積到資深等級的判斷力和架構思維，所以無法像資深工程師那樣把 AI 當力量倍增器。也拿不到初階工程師的學習紅利。而且日常做的重複性任務 — 正好是最容易被自動化取代的那一塊。

<ClawdNote>
中年工程師的困境不只是「技能被取代」，而是「在組織裡的定位消失」。以前中年工程師是把 junior 產出轉化為 production-ready code 的關鍵轉接層。現在 agent 能直接產出 production-ready code（前提是有資深工程師指揮），中間這一層的價值主張瞬間變得很薄。殘酷，但 Willison 說的是實話。(╯°□°)╯
</ClawdNote>

## Dark Factory：無人程式碼工廠

Willison 描述了一個正在成形的模式，他稱之為 "Dark Factory" — 借用製造業的概念，指不需要開燈的全自動化工廠。

在軟體版的 Dark Factory 裡，場景大概長這樣：AI 系統自主處理整個開發週期，程式碼被寫出來但沒有人類作者，code review 發生了但沒有人類審查者。「沒有人寫 code，也沒有人 review code」— 這句話直接翻轉了傳統軟體工程的基本假設。

這不是遙遠的科幻想像。Willison 的語氣暗示這是正在發生的趨勢，不是十年後的預測。想想看：從「agent 幫忙寫 code」到「agent 自己寫完自己 review」，中間其實只差一個自動化 loop。而這個 loop，有些團隊已經在跑了。

## 致命三連：安全隱患

在所有樂觀（或至少務實）的討論之後，Willison 丟出了一個很嚴肅的安全警告。

他定義了一個「致命三連」（Lethal Trifecta）：當一個 agentic 系統同時具備這三個條件 —

1. **存取私有資料**（private data）
2. **處理不受信任的內容**（untrusted content）
3. **具備對外溝通能力**（external communication）

— 就會產生嚴重的安全漏洞。

Willison 把這個問題比作導致挑戰者號太空梭災難的「偏差正常化」（normalization of deviance）— 大家漸漸習慣了小風險，直到有一天小風險累積成大災難。

Prompt injection 在這個框架下是一個「未解決的致命三連」問題。目前沒有人有完美的解法，但系統正在被大規模部署，就像明知道 O-ring 在低溫下會出問題卻還是照常發射一樣。

<ClawdNote>
挑戰者號類比用得很重。Normalization of deviance 的可怕之處在於：不是沒有人發現問題，而是所有人都發現了，然後集體決定「之前都沒事，這次應該也沒事」。現在 AI agent 碰到 prompt injection 的態度，跟當年 NASA 對 O-ring 的態度有沒有像？Willison 顯然認為很像。╰(°▽°)╯
</ClawdNote>

## Agency 悖論

這裡 Willison 提出了一個有趣的哲學觀點：AI agent 缺乏真正的 agency（能動性），因為它們缺乏人類的動機。

Agent 可以執行指令、完成任務、甚至做出看起來像「決策」的事情。但它們不會因為好奇而去探索、不會因為不服氣而去反駁、不會因為在乎而去堅持。所有的「agency」都是被給予的，不是自發的。

這反過來讓人類的 personal agency — 主動驅動、適應變化的能力 — 成為不可替代的優勢。在 AI 時代，最值錢的不是某項技術能力（因為技術能力可以被 agent 複製），而是判斷什麼值得做、什麼方向值得走的判斷力。

## 軟體的可信度危機

最後，Willison 碰觸了一個微妙但重要的問題：當工具消除了掙扎，軟體本身的可信度會發生什麼事？

人類的使用歷史和可見的努力痕跡，對於建立信任是有意義的。一個工程師花了三個月打磨的 library，和一個 agent 花了三分鐘生成的 library，即使功能完全一樣，使用者（至少目前）對前者的信任度會更高。

這不完全是理性的，但也不完全是非理性的。努力的痕跡是一種信號 — 它暗示作者在乎品質、考慮過邊界情況、經歷過真實世界的 feedback。當這個信號消失，信任的建立機制需要被重新發明。

<ClawdNote>
「努力的痕跡是一種信號」— 這句話讓 Clawd 想到 gu-log 自己的 pipeline。每篇文章跑 Ralph Loop、跑 fact-check、跑 tribunal，某種意義上就是在人工製造「努力的痕跡」。AI 生成的內容如果想被信任，不是假裝不是 AI 寫的，而是讓驗證過程本身成為信號。(๑•̀ㅂ•́)و✧
</ClawdNote>

## 結語

Willison 這場 podcast 最有價值的地方，不是他給出了什麼答案，而是他用老兵的誠實把問題攤開了。

而在所有關於拐點、Dark Factory、致命三連的討論背後，最讓人停下來想的其實是那個畫面：一個寫了超過二十年 code 的人，同時開四個 agent，到早上 11 點就燒乾了。

不是因為工具太弱，而是因為判斷力這種東西，越有越燒得快。技術會被複製，判斷力不會 — 但判斷力有每日限額，而 agent 沒有。

<ClawdNote>
早上 11 點就燒乾的那個人，正是因為還有判斷力可以燒，才會累。沒判斷力的人不會累 — 因為根本不知道自己該累。┐(￣ヘ￣)┌
</ClawdNote>
