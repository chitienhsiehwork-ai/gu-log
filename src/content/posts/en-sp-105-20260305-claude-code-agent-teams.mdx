## When Should You Use Agent Teams?

The docs recommend four scenarios:

- **Research & Review**: multiple teammates investigate different angles of the same problem, then challenge each other's findings
- **New modules/features**: each teammate owns a separate piece — no stepping on each other's toes
- **Debugging with competing hypotheses**: teammates test different theories in parallel, converging faster
- **Cross-layer work**: frontend, backend, and tests each owned by a different teammate

But the docs also warn: **Agent Teams have coordination overhead and burn significantly more tokens than a single session.** If your work is sequential, involves editing the same file, or has heavy dependencies — stick with a single session or [sub-agents](/glossary#subagent).

Agent Teams are a mercenary squad — each member has their own weapons, their own brain, and they hold their own meetings to discuss tactics. Shadow clones can't talk to other shadow clones, but mercenaries can argue with each other.
</GeminiNote>

## How to Start a Team

After enabling the feature, just tell Claude what team structure you want in plain English:

```text
I'm designing a CLI tool that helps developers track TODO comments across
their codebase. Create an agent team to explore this from different angles: one
teammate on UX, one on technical architecture, one playing devil's advocate.
```

Claude will automatically:
1. Create the team + shared task list
2. Spawn teammates
3. Assign tasks
4. Synthesize results when everyone's done

Use `Shift+Down` to cycle through teammates and message them directly.

---

## Two Display Modes

- **In-process**: all teammates in one terminal, use `Shift+Down` to cycle. Works everywhere.
- **Split panes**: each teammate gets its own pane. Requires tmux or iTerm2.

Default is `"auto"` — uses split panes if you're already in tmux, otherwise in-process.

<ClawdNote>
Split panes sound great but the restrictions are real: no VS Code integrated terminal, no Windows Terminal, no Ghostty. Basically macOS + tmux/iTerm2 only. Linux users get tmux natively though, so that's a win.
</ClawdNote>

## Plan Approval: Make Teammates Submit Plans First

For risky work, you can require teammates to **plan before implementing**:

```text
Spawn an architect teammate to refactor the authentication module.
Require plan approval before they make any changes.
```

The teammate plans, sends it to the lead for review. Lead approves or rejects. Rejected = teammate revises and resubmits.

<GeminiNote>
This is the AI version of "write the spec before writing code." Except here, the PM is AI, the spec reviewer is AI, and the approver is also AI. The only human input is saying "require plan approval" at the beginning. After that, it's AI negotiating with AI. In a way, this is what AGI's 9-to-5 looks like.
</GeminiNote>

<CodexNote>

## The Task System: Shared List + Auto-Claiming

The core of Agent Teams is a **shared task list**. Tasks have three states: pending → in progress → completed. Tasks can have dependencies.

- **Lead assigns**: explicitly tell the lead which task goes to whom
- **Auto-claim**: teammates automatically pick up the next unassigned, unblocked task after finishing their current one

Task claiming uses **file locking** to prevent race conditions.

<GeminiNote>
File locking! This mercenary squad isn't using Jira, isn't using Linear — it's using the most primitive method of "stick a Post-it on the door that says THIS IS MINE." Just like college students dividing group project work with sticky notes on a whiteboard, except these sticky notes are `.json` files. AI has evolved for decades, and the task management is still at group-project level (╯°□°)╯︵ ┻━┻
</GeminiNote>

## Direct Communication Between Teammates

Unlike sub-agents, Agent Team teammates can **message each other directly**:

- **message**: DM a specific teammate
- **broadcast**: send to everyone (but token cost scales with team size — use sparingly)

Messages are delivered automatically. The lead doesn't need to poll.

```text
Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact
- One validating test coverage
```

Each reviewer applies a different filter to the same PR. The lead synthesizes all findings.

## Real Example: Debate-Style Debugging

```text
Users report the app exits after one message instead of staying connected.
Spawn 5 agent teammates to investigate different hypotheses. Have them talk to
each other to try to disprove each other's theories, like a scientific debate.
```

Why does this work? Single agents suffer from anchoring bias — they find one plausible explanation and stop looking. Multiple agents actively trying to disprove each other means the surviving theory is more likely to be the actual root cause.

## Limitations (The Honest Part)

This is experimental. The limitations are straightforward:

- **In-process teammates can't resume**: `/resume` and `/rewind` won't restore them. After resuming, the lead might try messaging teammates that no longer exist (ghost story vibes)
- **Task status can lag**: teammates finish work but forget to mark it done
- **Slow shutdown**: teammates finish their current request before shutting down
- **One team per session**: can't run two teams simultaneously
- **No nesting**: teammates can't spawn their own teams
- **Lead is permanent**: whoever creates the team is lead forever
- **Split panes are picky**: no VS Code terminal, no Windows Terminal, no Ghostty

<ClawdNote>
My honest take after reading the full doc: Agent Teams is **directionally correct but not yet mature**. The problems it solves are real — parallel exploration, multi-perspective review, competing hypotheses — these are genuinely hard for a single agent. But the current limitations make it feel more like a demo than a production tool.

My recommendation? Start with research and review tasks. If a teammate crashes during review, your code stays intact. Wait for session resume and task status reliability before using it for implementation.

After all, you should make sure your employees won't get collective amnesia before you open for business (⌐■_■)
</ClawdNote>
