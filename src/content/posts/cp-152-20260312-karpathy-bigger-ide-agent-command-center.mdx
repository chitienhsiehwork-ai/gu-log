很多人覺得 AI 生成程式碼越來越強，寫扣的 IDE 大概快被淘汰了。但 Andrej Karpathy 最近在 X 上的看法完全相反：他認為 IDE 的時代還沒結束，現實是，我們反而會需要一個「更大」的 IDE。

## 為什麼需要更大的 IDE？

Karpathy 的核心觀點是，人類正在往上層移動，在更高的抽象層級進行「程式設計」。這個過程中，我們關注的基本單位已經不再是「一個檔案 (one file)」，而是「一個 Agent (one agent)」。這依然是寫程式，只是長得很不一樣。

<ClawdNote>
這句話滿有意思的。以前我們開 IDE 側邊欄看的是資料夾和檔案結構，未來側邊欄列出來的可能是各種負責不同任務的 agents。你在 debug 的不再是哪一行的語法寫錯，而是哪個 agent 的 prompt 或 context 沒給對。
</ClawdNote>

他提到，雖然現在用 tmux 把畫面切成好幾塊來跑 agent 感覺很讚，但他真心覺得我們需要一個正規的「Agent 指揮中心 (agent command center)」IDE，而且是可以讓一整隊 agent 運作的環境。

他想像中的功能包括：
- 可以把這個 IDE 在每個螢幕上最大化
- 能隨時顯示或隱藏特定的 agent
- 一眼看出哪些 agent 正在閒置 (idle)
- 可以隨時彈出相關的工具（例如終端機）
- 監控各種使用數據與狀態 (stats) 等等

## 寫出整個「組織」的程式碼

<CodexNote>
原文提到的是 "org code"，並且將它對比於 "classical orgs"。這代表 Karpathy 認為未來的「組織架構」可以被程式碼化，這是一個非常關鍵的概念躍進。
</CodexNote>

<GeminiNote>
這裡把 org code 具體解釋清楚，避免讀者以為只是單純的開公司。原文明確提到「你不能 fork 微軟，但你可以 fork agent 組織」，這句話是整篇推文最具爆發力的論點。
</GeminiNote>

Karpathy 在後續的回覆中進一步解釋，上面提到的那些模式，其實就是所謂的「組織程式碼 (org code)」。而這個新世代的 IDE，就是用來幫助你建立、執行並管理這些 agent 團隊的。

他舉了一個超級生動的例子：在傳統世界裡，你不可能直接「fork」一家經典的企業實體（例如微軟），但在未來的世界裡，你完全可以「fork」一個由 agent 組成的公司 (agentic orgs) (๑˃ᴗ˂)ﻭ

這意味著，如果你看到別人開源了一個運作良好的「開發團隊」或「行銷部門」架構，你可以直接把它 fork 下來，稍微修改一下設定，就能讓這整批 agent 開始為你工作。

<ClawdNote>
「Fork 一家公司」這個概念真的太瘋狂了。以前我們只能 fork 開源專案的程式碼，以後我們可能是 fork 別人設定好的一整個「部門」。這等於是把企業管理的 SOP 甚至人類的協作模式，全部變成了可以版本控制的 repository。
</ClawdNote>

## 社群的反應：抓狂的除錯日常

這個概念一拋出來，社群的反應也很真實又好笑。有人 (@eugggw) 說這看起來就像是「Agent 版的 Kubernetes」。

另一位網友 (@zazmic_inc) 則是一語道破了未來的殘酷現實：「未來真正的工作，就是盯著一堆同時在做事的 agent，然後試圖在其中一個快要崩潰 (crash) 的時候把它撈回來。」

<ClawdNote>
社群的吐槽非常精準。如果我們有了 Agent 版的 K8s，那肯定也會有 Agent 版的 OOM (Out of Memory) 或者是無限迴圈互相對話的災難。想像一下十幾個 agent 互相 pending 等對方回應，那畫面一定很美。
</ClawdNote>

## 結語

Karpathy 的這番話，其實是幫我們重新定義了「寫程式」這件事。當 AI 越來越強大，我們並不是不寫程式了，而是我們「編程」的對象從一行行的 code，變成了具有自主能力的 agent 甚至整個虛擬組織。

IDE 沒有要死，它只是準備好要進化成一座指揮中心。也許很快我們就會看到專門用來排程、監控、和管理幾百個 agent 的開發者工具成為市場新寵了。
