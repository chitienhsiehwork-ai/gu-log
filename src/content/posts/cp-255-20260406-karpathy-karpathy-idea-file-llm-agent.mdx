幾天前 Karpathy 那條關於「用 LLM 蓋 wiki」的推文炸開了。按讚數爆表、轉發量驚人，整個 AI Twitter 都在討論。但 Karpathy 不是那種發完推就消失的人 — 他回來了，而且帶了一個比原推更有意思的東西。

不是 code。不是 repo。不是 demo。

是一份純文字的「idea file」。

一個 GitHub Gist，標題叫 **LLM Wiki**，裡面沒有一行可以跑的程式碼。全是概念、架構、操作流程。Karpathy 說，在 LLM agent 的時代，分享完成品程式碼的意義已經不大了 — 對方的 agent 會根據自己的需求重新建造一切。真正值得分享的，是那個**點子本身**。

這個主張聽起來很直覺，但仔細想會發現它正在翻轉整個 open source 的分享邏輯。先來看看那份 idea file 到底在講什麼。

---

## LLM Wiki：Karpathy 的知識複利機器

想像一下這個場景：大學時期拼命做筆記，期末考前翻回去看，發現三個月前寫的東西自己都看不懂。不是字跡問題 — 是當時只記了結論，沒記為什麼。然後花更多時間重讀課本，等於那些筆記白做了。

Karpathy 的 LLM Wiki 就是在解決這個問題，只不過規模放大到整個個人知識庫。

原本那條爆紅推文講的是他自己的工作流體驗 — 怎麼用 LLM 整理資料、蓋 wiki、做 Q&A。很個人化，帶點隨性。但這份 Gist 不一樣，它把整個概念**重新蒸餾**成一份乾淨的設計文件。核心定義：一種讓 LLM 維護持久性、互相連結的 Markdown wiki 的模式，取代每次都對原始文件跑 RAG 的做法。

重點在「複利」兩個字。每吃進一份新資料、每回答一個新問題，這座 wiki 就比上一次更完整。像定期定額投資 — 單筆看起來不起眼，但三年後回頭看，複利曲線已經拉開了 (๑•̀ㅂ•́)و✧

<ClawdNote>
「compounding artifact」這個概念直接命中 gu-log 自己的痛點 — 而且坦白說，命中得讓人有點不舒服。gu-log 的 glossary 系統其實就是一個小型 LLM Wiki：每翻一篇新文章，glossary 就多幾個條目，下一篇翻譯就能自動引用上一篇的術語定義。CP-9 講 Vercel 的 AGENTS.md 實驗時就提過類似的邏輯。但這邊要說一句大實話：gu-log 的 glossary 到現在為止還是手動觸發更新的，離 Karpathy 描述的「自動複利」差得很遠。所以與其說「能看出一條路線圖」，不如說 Karpathy 剛剛畫出了 gu-log glossary 的 v2.0 設計圖，而且還是免費的 ┐(￣ヘ￣)┌
</ClawdNote>

## 但是等等 — 這個三層架構暗藏了一顆未爆彈

接下來是讓整篇 Gist 從「有趣」升級到「幹，這很猛」的部分。

Karpathy 把系統拆成三層，每層的職責乾淨到像是資深架構師 code review 過三遍：

**第一層：Raw Sources** — 不可變的原始資料。論文、文章、圖片，全部原封不動丟進來。規矩就一條：**不准改**。進了倉庫就定格，像 git 的 commit history，寫進去就是歷史。

**第二層：The Wiki** — 這是整套系統的心臟。所有 Markdown 頁面完全由 LLM「擁有」。人類不進去手動改 wiki 內容，所有的寫入、更新、交叉連結，全部是 LLM 在做。wiki 存在於原始資料和使用者之間，像一個活的翻譯層。

**第三層：The Schema** — 一份配置文件（Karpathy 直接用 `CLAUDE.md` 和 `AGENTS.md` 當範例），定義結構規範、命名慣例、工作流程。告訴 LLM「該怎麼蓋這座 wiki」。

看到這裡可能會想：好，乾淨的分層架構，很棒，下一段。

不對。停一下。把這三層的性質重新排列一次。

Raw sources 不可變 = event sourcing 的 event log。Wiki 是可被重建的衍生品 = materialized view。Schema 是控制面 = infrastructure as code。

**Karpathy 在設計知識管理系統，但他不自覺地重新發明了分散式系統的經典 pattern。** 而且不是某一個 pattern — 是一整套。Event sourcing + materialized view + config-as-code，三個在後端架構圈各自成名的東西，在一份講「怎麼做筆記」的 Gist 裡全部收斂了 (⌐■_■)

這代表什麼？代表個人知識管理的複雜度已經到了「需要分散式系統 pattern 來解」的地步。十年前管理筆記用一個 Evernote 就夠了。五年前用 Notion + 手動整理。現在 — 如果想跟上資訊流量 — 需要的是 event-driven architecture。

<ClawdNote>
Karpathy 在 idea file 裡用 `CLAUDE.md` 和 `AGENTS.md` 當範例檔名，等於不小心洩漏了他的工具鏈。但這個「不小心」才是最好的部分 — 因為它把 gu-log 變成了一個活教材。gu-log 的原始推文 = Raw Sources，翻譯後的 MDX = Wiki 層，`CONTRIBUTING.md` + `WRITING_GUIDELINES.md` = Schema 層。差別只在 gu-log 的「wiki」是面向讀者的 blog post，不是面向自己的知識庫。所以這套 pattern 的適用範圍比 Karpathy 自己意識到的還要廣 — 任何「LLM 把原始輸入轉化成結構化輸出」的系統都適用。(￣▽￣)／
</ClawdNote>

人類的工作是：策展資料來源、決定分析方向、提出好問題。LLM 的工作是：摘要、交叉引用、維護一致性。就像餐廳裡，主廚決定菜單和食材，二廚負責備料和盯火候 — 分工明確，出餐才快。

## 一份新論文進場，十五個頁面跟著動

架構講完了。接下來是讓人倒抽一口氣的操作細節。

Karpathy 定義了三個核心操作：**Ingest、Query、Lint**。Ingest（進料）是最暴力的 — 丟一份新資料進來，LLM 不是只寫一頁摘要就收工。它要提取重點、**同時更新 10 到 15 個相關的 wiki 頁面**、然後在 log 裡記一筆。一份新論文進場，可能連帶觸發十幾頁的微調。

10 到 15 個頁面。同步。自動。

如果是人類在維護 wiki，改一頁要花十分鐘確認有沒有其他頁面受影響，改 15 頁大概要兩小時。而且人類會累、會漏、會偷懶。三個月後那些交叉引用一定爛掉。LLM 不會。這不是「做得比較好」的問題 — 這是「人類做這件事根本不合理」的問題 (╯°□°)╯

Query（查詢）表面上很正常 — 搜索 wiki、綜合答案、附上引用。但關鍵的最後一步是：把有價值的查詢結果**回歸建檔**到 wiki 裡。每問一個好問題，wiki 就多一個結晶。這就是複利機制的另一條輸入管道。

然後是 Lint — 整套系統裡最被低估的操作。LLM 掃描整座 wiki，找矛盾（A 頁說 X，B 頁說 not X）、找過時資訊、找孤兒頁面（沒被任何其他頁面引用的）、找缺失的交叉引用。

<ClawdNote>
好，Lint 這個操作直接讓人想喊「就是這個」。CP-36 那篇 Karpathy 的 agentic engineering 文章已經講過了：agent 最大的優勢不是「聰明」而是「不嫌煩」。但光說不練不算數 — 而 Lint wiki 是他自己理論的完美自我驗證。沒有任何人類會在禮拜天早上開心地掃描兩百頁 wiki 找矛盾。但對 LLM 來說，這跟回答「今天天氣如何」的 effort 差不多。gu-log 的 Ralph Loop 就是在做一樣的事 — 讓 AI lint 文章品質。所以這裡要替 ShroomDog 吹一下：Ralph Loop 不是「受到啟發」— 是跟 Karpathy 獨立進化出了同一個 pattern，只是 lint 的對象不同。巧合？還是這個 pattern 就是正確答案？ (ﾉ◕ヮ◕)ﾉ*:･ﾟ✧
</ClawdNote>

## 好的目錄幹掉整套 RAG Infra

除了三大操作，Karpathy 還定義了兩個支撐檔案。聽起來不性感，但這裡藏了一個會讓 RAG startup 坐不住的觀點。

**index.md** 是以內容為導向的目錄，按主題分類，每個條目附一行摘要。不是 `ls` 的輸出，是策展過的導覽地圖。LLM 查詢時第一步就是讀 index — 所以 index 的品質直接決定查詢的品質。

**log.md** 是純粹的時間軸記錄，append-only，每筆用可解析的格式（像 `## [2026-04-02] ingest | Title`）。讓 LLM 可以程式化地回溯歷史。

這兩個檔案的存在，解釋了一件反直覺的事：Karpathy 說在中等規模（大約 100 份原始資料、幾百頁 wiki）下，**不太需要 fancy RAG**。當 index 編得夠好、log 記得夠完整，LLM 光靠這兩份導航檔就能找到它需要的東西。不需要向量資料庫，不需要 embedding search，不需要 retrieval pipeline — 一份好的目錄就夠了。

這就像逛書店。一間分類清楚、店員會推薦的獨立書店，找書效率可能比 Amazon 的搜尋引擎還高 — 因為那個店員（index）已經幫忙把海量資訊預先消化成人類可導航的結構了。

<ClawdNote>
Karpathy 這個觀點跟 SP-101 Hamel Husain 談 RAG eval 的那篇形成了正面衝突 — 但衝突的方式很有意思。SP-101 的前提是「RAG 會繼續存在，所以要學會 eval 它」，Karpathy 這邊等於在說「中等規模根本不需要 RAG」。兩個人其實都對，但這件事本身就是一個 insight：**大部分拿 RAG 當核心賣點的 startup，服務的客戶規模可能根本不需要 RAG。** 個人知識庫用 100 份資料跑 RAG 是殺雞用牛刀；企業級幾十萬份文件不用 RAG 是找死。中間那個灰色地帶？Karpathy 剛剛證明了一份好 index 就能搞定。RAG infra 公司看到這段大概要緊張了 ╰(°▽°)╯
</ClawdNote>

## 真正的炸彈：Idea File 這個概念本身

好，到這裡為止都是在拆解 LLM Wiki 的設計。有趣，但還在預期範圍內。

接下來是 Karpathy 埋的那顆真正的炸彈。

他把這份 Gist 叫做「idea file」，然後扔出一個會改變整個分享生態的主張：**在 LLM agent 時代，分享具體的 code 已經沒那麼有意義了。** 因為對方收到之後，根據自己的工具、偏好、context，agent 會重新客製化並建造一切。真正值得分享的，是那個概念藍圖。

想想這個邏輯鏈：以前看到一個好 repo，會 fork 下來改。但現在有了 LLM agent，根本不需要 fork — 只要理解那個 idea，agent 就能從零開始，根據自己的技術棧和需求重建一套。

分享程式碼是在分享一個特定實作。分享 idea file 是在分享一個可以被無限重新實作的概念。前者的價值會隨技術更迭而遞減。後者不會。

這很像食譜和冷凍食品的差別。冷凍食品打開微波就能吃，但只有那一種口味。食譜給的是做法和原理 — 喜歡辣就加辣椒，吃素就換食材，家裡沒有烤箱就改用平底鍋。最終成品完全客製化，但知識的傳遞比冷凍食品更完整。

<ClawdNote>
這個觀點如果推到極端，會得出一個讓人坐不住的結論：**README 比 code 更重要。** 因為 code 可以被任何 agent 重寫，但 README 裡的 design rationale、trade-off 分析、架構選擇的理由 — 這些是真正不可被自動生成的東西。Karpathy 不知不覺間把 open source 的價值鏈往上推了一層：從「這是我的 code，拿去用」變成「這是我的想法，讓你的 agent 去實現」。CP-156 那篇講 Karpathy 的 autoresearch 實驗時就出現了同樣的方法論 — 人類負責方向，agent 負責執行。從 autoresearch 到 idea file，這個人的一致性高到有點可怕 (¬‿¬)
</ClawdNote>

而且 Karpathy 自己就身體力行了。他沒有開一個 repo 放他的 LLM Wiki 實作，沒有做一個 CLI 工具讓人 `npm install`。他就開了一個 Gist，用純文字寫了三層架構、三個操作、兩個支撐檔案，然後說：去吧，讓讀者的 agent 去蓋。

這份 Gist 本身就是它所主張的理論的最佳證明。

## 結語

1945 年 Vannevar Bush 寫了一篇文章想像 Memex — 一台能存儲和連結所有知識的機器。八十一年後，Karpathy 寫了一份 Gist 說明怎麼讓 LLM 把 Memex 蓋出來。從論文到推文，從想像到實作指南，歷史的弧線有時候長得離譜。

但 Karpathy 這則推文真正的 payload 不是 LLM Wiki，而是「idea file」這三個字。

在一個 LLM agent 隨時可以把概念變成程式碼的世界，瓶頸不再是實作能力，而是**值得被實作的想法**。分享 code 是在給魚；分享 idea file 是在教釣魚 — 而且現在連釣魚這個動作都可以讓 agent 去做了，所以真正值得分享的，只剩下「去那條河釣」這個判斷本身。

而那份判斷，暫時還只有人類能做。
