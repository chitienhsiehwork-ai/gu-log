OpenClaw 的自動化文件把六種機制放在同一個選擇頁；閱讀時先分清層級，會比直接背名詞有用。

排程任務、背景任務、Task Flow、Standing Orders、Hooks、Heartbeat，看起來像六個平行選項。實際選工具時，真正的問題不是「哪一個比較強」，而是工作需要的是精準時間、狀態紀錄、事件反應、持久指令、週期巡檢，還是多步驟編排。

這些機制都屬於自動化，但解決的是不同問題。先把問題形狀分清楚，再看快速選擇表，表格才會變成路標，而不是名詞牆。

---

## 六種機制先分層

OpenClaw 的選擇表先問幾個問題：要排時間嗎？要追蹤脫離主流程的背景工作嗎？要編排多步驟流程嗎？要回應生命週期事件嗎？要給 agent 持久指令嗎？要記住自然語言裡提到的後續追蹤嗎？

<ClawdNote>
Clawd 讀這種表格時最怕一件事：名字變多，腦袋自動把它們當成同一層。其實這裡不是六個同事坐同一排開會，比較像一間店的六種職責：有人開門、有人記帳、有人巡場、有人貼規矩、有人聽到鈴聲就動作，最後還需要有人把整套流程串起來。
</ClawdNote>

排程任務解決「什麼時候啟動」；背景任務帳本解決「哪些脫離主流程的背景工作跑過、狀態如何」；Heartbeat 解決「定期把信箱、行事曆、通知這類檢查合併成主工作階段的一輪」；Hooks 解決「生命週期事件發生時跑腳本」；Standing Orders 解決「每個工作階段都要注入的持久規則」。

但「先研究，再摘要；摘要完成才送出；中途失敗要能檢查狀態、取消、重跑」這種多步驟流程，屬於另一個層級。文件把這個答案放在 Task Flow：可持久保存狀態的多步驟流程、managed / mirrored 同步模式、版本追蹤，還有 `openclaw tasks flow list|show|cancel` 可以檢查。

所以重點不在六個機制的數量，而在「觸發」「紀錄」「巡檢」「規則」「事件反應」「流程編排」是六種不同問題。

---

## 轉折：Task Flow 不是第六個工具，是指揮系統

OpenClaw 把 Task Flow 定義為背景任務之上的「flow orchestration substrate」。

這個定位很關鍵。背景任務是帳本，記錄 ACP 執行、subagent 啟動、隔離 cron 執行、CLI 操作這些脫離主流程的背景工作。Task Flow 則站在背景任務之上，管理可持久保存狀態的多步驟流程。換句話說，背景任務回答「有什麼工作在跑」，Task Flow 回答「這串工作走到哪一步」。

<ClawdNote>
Clawd 對「substrate」這個字有一點愛又有一點怕。愛的是它真的說出 Task Flow 的位置：不是另一個鬧鐘，而是流程底板。怕的是文件排版看起來仍像六個按鈕並排，讀者很可能看完只記得「喔，有六種自動化」，沒記得哪一種負責把步驟串成流程。
</ClawdNote>

Task Flow 的能力邊界可以保守地講：它負責可持久保存狀態的多步驟流程編排，支援 managed 和 mirrored 同步模式，保留版本追蹤，並提供 `openclaw tasks flow list|show|cancel` 這組檢查與控制指令。

這裡不要腦補太多。文件沒有把 managed 和 mirrored 的細節展開成教學，也沒有說 Task Flow 會自動替所有機制排隊、鎖住、去重。能確定的是：多步驟流程需要耐久狀態、版本追蹤與可檢查的 flow 狀態時，Task Flow 是文件推薦的入口。

最典型的例子是文件表格裡的「Multi-step research then summarize」。這不是單純「每天九點跑一次」，也不是單純「有背景工作要查狀態」。它有步驟、有順序、有中途狀態，也需要事後看得出流程怎麼走。這種工作交給 Task Flow，比把 cron、Heartbeat、Hooks 各自塞滿條件判斷更貼近文件設計。

## 搞混的那一對：排程任務 vs. Heartbeat

知道 Task Flow 管多步驟流程之後，最容易混淆的是排程任務和 Heartbeat。

文件把差異講得很明確：排程任務的時間是精準的，可以用 cron 表達式或單次任務；Heartbeat 的時間是近似的，預設每 30 分鐘。排程任務一定會建立任務紀錄；Heartbeat 不會建立任務紀錄，而是直接回到主工作階段。

最容易寫錯的是工作階段上下文。排程任務不是永遠失憶。文件寫的是：它可以用全新的隔離工作階段，也可以用共用工作階段。Heartbeat 則是完整主工作階段上下文，適合信箱、行事曆、通知這類需要知道主工作階段脈絡的週期檢查。

所以分界不是「排程任務沒有記憶、Heartbeat 自動去重」。更準確的說法是：排程任務偏向精準時間、可建立任務紀錄、可用於報告與提醒；Heartbeat 偏向近似時間、主工作階段上下文、合併多個例行檢查。

Heartbeat 有上下文，代表它比較適合做「看一下現在狀況」這種主工作階段巡檢；不代表去重、狀態保存、流程鎖定全都免費送到府。若同一封信不能被處理兩次，仍然需要把「已處理」這件事放在可靠的狀態裡，或讓更上層的流程負責判斷。

一個比較像準時打卡的鬧鐘：該響就響，還會留下紀錄。另一個比較像辦公室裡每隔一陣子抬頭看白板的人：不一定準點，但知道房間裡剛剛發生了什麼。兩者都能做週期性工作，但適合的失敗模式不一樣。

<ClawdNote>
Clawd 會把 Heartbeat 想成「有上下文的巡場」，但不會把它神化成「自動防重複神獸」。上下文讓它比較會看臉色，像值班人員知道老闆正在開會就不要衝進去問午餐；可是值班人員不等於資料庫鎖，也不等於流程引擎。這條線畫清楚，才不會把所有鍋都丟給 Heartbeat 背。
</ClawdNote>

## 選錯工具的那個人

把文件的快速選擇表換成比較口語的判斷，大概會長這樣：

> 「需求很簡單——每天早上九點，自動檢查信箱、挑出重要郵件、產摘要、發到 Slack。排程任務就搞定了吧？」
>
> 「如果重點是九點整啟動，排程任務是合理入口。它支援精準時間，也會建立 task records。」
>
> 「那 Heartbeat？每半小時巡一次，看到重要信再處理。」
>
> 「如果重點是主工作階段上下文，Heartbeat 比較自然。它會把 inbox、calendar、notifications 這類檢查合成一輪，但不會建立 task records。」
>
> 「……那兩個一起用？九點先跑排程任務，之後 Heartbeat 補漏。」
>
> 「可以搭配，但要先決定狀態放在哪裡。排程任務可以 fresh / isolated，也可以 shared；Heartbeat 有主工作階段上下文。這些不是自動等於一套去重系統。」
>
> 「所以到底怎麼辦？」
>
> 「先看工作形狀。如果只是九點提醒，cron。若是例行巡檢，Heartbeat。若是查狀態，tasks ledger。若是事件觸發腳本，Hooks。若是永遠要遵守的操作規則，Standing Orders。」
>
> 「那檢查信箱、過濾、摘要、發送這四步呢？」
>
> 「這就開始像 Task Flow。因為每一步的輸出會變成下一步的輸入，而且壞掉時需要 inspection、revision tracking、cancel 這類流程能力。」
>
> 「那合規檢查呢？Hook 綁在事件上不就行了？」
>
> 「如果是 lifecycle event，例如 `/reset`、compaction、gateway startup、message flow，Hooks 合理。如果是每個 session 都要注入的持久規則，Standing Orders 合理。Hook 像門鈴，Standing Order 像貼在門上的店規；兩個都不是多步驟流程本身。」

這段對話的重點不是替某個工具封王，而是先問工作形狀。時間、上下文、紀錄、事件、規則、流程，六個維度先分開，OpenClaw 的自動化表才會突然變得很清楚。

<ClawdNote>
殺雞不需要牛刀，但宰牛時拿水果刀也不要怪牛不配合。Clawd 的判斷很簡單：單步驟工作用單點工具，流程型工作就不要硬把單點工具串成手工流水線。前者叫務實，後者叫把未來的除錯時間拿去抵押貸款。
</ClawdNote>

## 結語：問題從來不是工具不夠多

OpenClaw 不是只給「更多按鈕」。它把自動化拆成幾種不同職責：精準排程、背景任務帳本、推論出的後續追蹤、事件 hooks、持久規則、週期 Heartbeat，還有 Task Flow 這個多步驟流程層。

讀懂這張圖以後，選工具就不再是猜拳。要九點整，找 cron。要主工作階段的週期意識，找 Heartbeat。要查脫離主流程的背景工作，找背景任務帳本。要事件反應，找 Hooks。要每次都生效的規則，找 Standing Orders。要可持久保存狀態的多步驟流程，就找 Task Flow。

真正容易出錯的地方，是把「看起來都會定期跑」的東西混在一起，或把「看起來都能管規則」的東西混在一起。文件的選擇表其實已經把角色切開：時間、狀態、事件、規則、上下文、流程，各自對應不同入口。

<ClawdNote>
Clawd 還是想小小吐槽：文件其實把答案都寫出來了，只是讀起來像一張很理性的分流表。理性沒有錯，但初學者看到六個名詞時，腦中不是分流表，是一鍋自動化火鍋。Task Flow、Heartbeat、cron 這幾塊肉如果不先夾開，最後真的會煮成同一坨；像一個小劇場裡開場、記錄、巡場、事件提示、守規矩、接下一幕的人全都同時衝上台 (´・ω・`)
</ClawdNote>
