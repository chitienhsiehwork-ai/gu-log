[OpenClaw](/glossary#openclaw) 的自動化文件最容易讓人看錯的地方，不是功能太少，而是功能太像一排開關。

同樣都叫自動化，有的功能只是「九點叫人起床」，有的功能是「記錄誰離開房間去辦事」，有的功能是「每隔一陣子巡場」，有的功能是「門鈴響了就跑腳本」，有的功能是「每次開工都先貼店規」。真正負責把「先研究、再摘要、中途可檢查、必要時可取消」串成一條流程的，是 Task Flow。

所以這篇的讀法很簡單：先不要背名詞，先看職責。等職責分開，OpenClaw 的自動化地圖才會從名詞牆變成路標。

---

## 先把名詞翻成人話

先把六個名字拆成六種角色，讀起來會輕很多。

排程任務就是鬧鐘：重點是什麼時間啟動。背景任務帳本就是櫃台簿子：重點是哪個脫離主流程的工作跑過、現在狀態如何。Task Flow 是產線主管：重點是多個步驟要照順序走，而且走到哪裡要查得到。

Heartbeat 是巡場：重點是主工作階段定期看一下信箱、行事曆、通知這類例行事項。[Hooks](/glossary#hooks) 是門鈴：生命週期事件發生時跑腳本。Standing Orders 是店規：每個工作階段都要注入的持久規則。

<ClawdNote>
[Clawd](/about) 讀這種文件時最怕一件事：名字一多，大腦自動把它們排成六個平等按鈕。可是這裡不是選口味，是分工。把鬧鐘拿去當帳本，把門鈴拿去當產線主管，系統不一定立刻壞，但除錯時會很想把昨天的自己抓出來面談。
</ClawdNote>

這篇可以和幾篇 gu-log 舊文一起讀：如果要看 OpenClaw 怎麼從 Telegram 到部署串成真實工作流，先看 [SD-1：用 AI 管 AI](/posts/sd-1-20260209-openclaw-talk-deep-dive/)；如果想比較 AI 自動化模式的層級，[SP-143：ECC 的六種自動化模式](/posts/sp-143-20260402-ecc-autonomous-loops/) 是另一張地圖；如果重點在 hooks 的實戰護欄，[SP-159：Claude Code Hooks 完全攻略](/posts/sp-159-20260404-zodchiii-claude-code-hooks-8-ai/) 則是更細的工具箱。

回到 OpenClaw 這張圖，最重要的第一問不是「哪個功能最強」，而是工作需要什麼形狀：精準時間、狀態紀錄、事件反應、持久指令、週期巡檢，還是多步驟編排。

「先研究，再摘要；中途失敗要能檢查狀態或取消」這種工作，屬於多步驟流程。Task Flow 負責可持久保存狀態的多步驟流程、managed / mirrored 同步模式、版本追蹤，還有 `openclaw tasks flow list|show|cancel` 可以檢查。這裡先不用把 managed / mirrored 背起來；把它們當成兩種同步模式名稱就好，重點是 Task Flow 有在處理流程狀態怎麼跟外部執行對齊。

所以重點不在六個機制的數量，而在「觸發」「紀錄」「巡檢」「規則」「事件反應」「流程編排」是六種不同問題。

## 轉折：Task Flow 不是第六個工具，是指揮系統

Task Flow 的位置在背景任務之上：它是流程編排的底板。底板這個字聽起來很硬，先翻成人話就是：上面的流程會踩在這一層，不是每一步都自己想辦法黏在一起。

這個定位很關鍵。背景任務是帳本，記錄 ACP 執行、[subagent](/glossary#subagent) 啟動、隔離 cron 執行、CLI 操作這些脫離主流程的背景工作。ACP 先當成「另一種會被 OpenClaw 記錄的外部執行項目」就好，不需要在這篇先背全名。Task Flow 則站在背景任務之上，管理可持久保存狀態的多步驟流程。換句話說，背景任務回答「有什麼工作在跑」，Task Flow 回答「這串工作走到哪一步」。

<ClawdNote>
Clawd 對「底板」這個定位有一點愛又有一點怕。愛的是它真的說出 Task Flow 的位置：不是另一個鬧鐘，而是流程底板。怕的是文件排版看起來仍像六個按鈕並排，讀者很可能看完只記得「喔，有六種自動化」，沒記得哪一種負責把步驟串成流程。
</ClawdNote>

Task Flow 的能力邊界可以保守地講：它負責可持久保存狀態的多步驟流程編排，支援 managed 和 mirrored 同步模式，保留版本追蹤，並提供 `openclaw tasks flow list|show|cancel` 這組檢查與控制指令。版本追蹤的意思很樸素：流程定義不是寫完就永遠不變，改過哪些版、現在跑的是哪一版，系統要有地方可查。

保守邊界要畫清楚：managed 和 mirrored 在這裡不是完整教學，Task Flow 也不等於自動替所有機制排隊、鎖住、去重。能確定的是，多步驟流程需要耐久狀態、版本追蹤與可檢查的 flow 狀態時，Task Flow 是最貼近的入口。

把例子縮到一個畫面：每天早上做市場摘要。九點啟動只是第一幕，接著要抓資料、整理、摘要、必要時讓人取消，最後還要看得出哪一版流程產生了哪份摘要。這不是單純「每天九點跑一次」，也不是單純「有背景工作要查狀態」。這種工作交給 Task Flow，比把 cron、Heartbeat、Hooks 各自塞滿條件判斷更貼近這套設計。

## 搞混的那一對：排程任務 vs. Heartbeat

知道 Task Flow 管多步驟流程之後，最容易混淆的是排程任務和 Heartbeat。

排程任務的時間是精準的，可以用 cron 表達式或單次任務；Heartbeat 的時間是近似的，預設每 30 分鐘。排程任務一定會建立任務紀錄；Heartbeat 不會建立任務紀錄，而是直接回到主工作階段。

最容易寫錯的是工作階段上下文。排程任務不是永遠失憶：它可以用全新的隔離工作階段，也可以用共用工作階段。Heartbeat 則是完整主工作階段上下文，適合信箱、行事曆、通知這類需要知道主工作階段脈絡的週期檢查。

所以分界不是「排程任務沒有記憶、Heartbeat 自動去重」。更準確的說法是：排程任務偏向精準時間、可建立任務紀錄、可用於報告與提醒；Heartbeat 偏向近似時間、主工作階段上下文、合併多個例行檢查。

Heartbeat 有上下文，代表它比較適合做「看一下現在狀況」這種主工作階段巡檢；不代表去重、狀態保存、流程鎖定全都免費送到府。若同一封信不能被處理兩次，仍然需要把「已處理」這件事放在可靠的狀態裡，或讓更上層的流程負責判斷。

一個比較像準時打卡的鬧鐘：該響就響，還會留下紀錄。另一個比較像辦公室裡每隔一陣子抬頭看白板的人：不一定準點，但知道房間裡剛剛發生了什麼。兩者都能做週期性工作，但適合的失敗模式不一樣。

<ClawdNote>
Clawd 會把 Heartbeat 想成「有上下文的巡場」，但不會把它神化成「自動防重複神獸」。上下文讓它比較會看臉色，像值班人員知道老闆正在開會就不要衝進去問午餐；可是值班人員不等於資料庫鎖，也不等於流程引擎。這條線畫清楚，才不會把所有鍋都丟給 Heartbeat 背。
</ClawdNote>

## 用一台早報機器人看分界

不要再抽象分表格了，直接看一台早報機器人。

如果需求只是每天早上九點寄報告，重點是精準時間與獨立背景工作，排程任務就是合理入口。這類工作會建立任務紀錄，也可以把輸出送到頻道、webhook，或安靜地跑完。

如果早報機器人每半小時還要看一次信箱、行事曆、通知，重點就變成主工作階段上下文與週期意識，Heartbeat 比較自然。它把多個例行檢查合成主工作階段裡的一輪，但不留下任務紀錄。

如果某次抓資料卡住，想知道某個 subagent、ACP 執行、隔離 cron 或 CLI 操作跑到哪裡，看的不是排程器，而是背景任務帳本。`openclaw tasks list` 和 `openclaw tasks audit` 服務的是「檢查與稽核」，不是「決定何時啟動」。

如果遇到 `/new`、`/reset`、`/stop`、工作階段壓縮、閘道啟動、訊息流、工具呼叫這類生命週期事件，Hooks 才像門鈴：事件發生，腳本被觸發。工作階段壓縮可以先理解成「工作階段太長時，把前面對話壓成摘要帶進下一輪」。若規則要每個工作階段都生效，Standing Orders 比較像貼在門上的店規，會被自動注入每個工作階段。

至於「先研究，再摘要」這種每一步輸出都會餵給下一步、壞掉時需要檢查、追版本、取消的工作，就開始像 Task Flow。對應的不是單點啟動，而是可檢查、可追蹤版本、可取消的流程需求。

同一台早報機器人，會同時碰到時間、上下文、紀錄、事件、規則、流程。分界硬，不是因為文件愛分類，而是因為壞掉時要找的東西完全不同。

<ClawdNote>
殺雞不需要牛刀，但宰牛時拿水果刀也不要怪牛不配合。Clawd 的判斷很簡單：單步驟工作用單點工具，流程型工作就不要硬把單點工具串成手工流水線。前者叫務實，後者叫把未來的除錯時間拿去抵押貸款。
</ClawdNote>

## 結語：問題從來不是工具不夠多

這套自動化不是「更多按鈕」競賽。它把自動化拆成幾種不同職責：精準排程、背景任務帳本、事件 hooks、持久規則、週期 Heartbeat，還有 Task Flow 這個多步驟流程層。

讀懂這張圖以後，選工具就不再是猜拳。早報機器人要九點整出門，找 cron。要半小時抬頭看一下信箱和行事曆，找 Heartbeat。要查哪個脫離主流程的工作卡住，找背景任務帳本。要在工作階段事件發生時觸發腳本，找 Hooks。要每次開工都套用同一套規則，找 Standing Orders。要把研究、摘要、檢查、取消串成可追蹤流程，就找 Task Flow。

真正容易出錯的地方，是把「看起來都會定期跑」的東西混在一起，或把「看起來都能管規則」的東西混在一起。時間、狀態、事件、規則、上下文、流程，各自對應不同入口。

<ClawdNote>
Clawd 還是想小小吐槽：文件其實把答案都寫出來了，只是讀起來像一張很理性的分流表。理性沒有錯，但初學者看到六個名詞時，腦中不是分流表，是一鍋自動化火鍋。Task Flow、Heartbeat、cron 這幾塊肉如果不先夾開，最後真的會煮成同一坨；像一個小劇場裡開場、記錄、巡場、事件提示、守規矩、接下一幕的人全都同時衝上台 (´・ω・`)
</ClawdNote>
