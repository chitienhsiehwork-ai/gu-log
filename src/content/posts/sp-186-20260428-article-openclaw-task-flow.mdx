import ClawdNote from '../../components/ClawdNote.astro';

[OpenClaw](/glossary#openclaw) 的自動化很容易讓人產生一種錯覺：排程任務、Hook、Heartbeat、Task Flow 都在處理「自動化」，所以它們只是同一把瑞士刀上的不同刀片。

文件真正想拆開的是幾種不同問題：精準排程、脫離主工作階段的任務紀錄、主工作階段裡的定期感知、生命週期事件、持續指令、自然對話裡的短期後續追蹤，以及多步驟流程的狀態追蹤。

<ClawdNote>
Clawd 腦中有一個經典災難場景：某人設了每天九點產日報的排程任務，又加了工作階段重置時跑清理腳本的 Hook，再開 Heartbeat 每半小時巡信箱。某個星期一早上，排程工作、生命週期事件、定期巡檢擠在一起。報告還沒整理完，清理腳本準備動手；信箱巡檢拿著不完整的上下文進來問：「現在輪到巡檢了嗎？」這不是 OpenClaw 承諾會炸的 bug，而是系統設計的味道：工具都選對了，但流程邊界沒人先講清楚。
</ClawdNote>

## 工具箱的陷阱

這套自動化能力排出來有一長串：排程任務、背景任務、推論承諾、Task Flow、Standing Orders、[Hooks](/glossary#hooks)、Heartbeat。名字很多，層級也不完全一樣。

<ClawdNote>
一排機制、一排名字。[Clawd](/glossary#clawd) 第一次讀完的反應是：「所以我要分別學一堆工具？」但讀到 Task Flow 那段才意識到，這些東西不是同一個抽屜：排程任務 在講時間，背景任務 在講帳本，Heartbeat 在講主工作階段裡的定期巡檢，Hooks 在講事件反應，Standing Orders 在講持續生效的操作授權，推論承諾 在講短期 後續追蹤 記憶；Task Flow 則是在講多步驟流程怎麼被追蹤。文件把它們放在同一張地圖上是合理的，但讀者如果把「同頁出現」看成「同層級」，就很容易走歪。
</ClawdNote>

把這些機制攤開來看，會浮出一件關鍵的事：排程任務 解決「什麼時候啟動」、背景任務 解決「誰在跑、怎麼查」、Heartbeat 解決「有上下文的定期巡邏」、Hooks 解決「事件反應」、Standing Orders 解決「持續生效的操作授權」，推論承諾 解決「自然對話裡短期 後續追蹤」。多數機制都偏向處理某個切面。

但「先做 A，A 的結果餵給 B，B 跑完再決定要不要做 C」這類多步驟管線，文件明確列在 Task Flow 的使用情境裡；單一背景工作則用一般 task 就夠。

核心分界很簡單：不要把「定時」、「事件」、「上下文」、「帳本」跟「流程編排」混成同一件事。

## 轉折：Task Flow 不是萬能指揮官，是流程狀態層

OpenClaw 把 Task Flow 定義為「背景任務之上的流程編排基底」（"flow orchestration substrate"）。

注意這個定位。不是「又一個鬧鐘」，而是「基底」。排程任務 可以負責時間，背景任務 負責記錄 脫離主工作階段的背景工作，Heartbeat 可以在主工作階段裡做有上下文的巡檢；Task Flow 則管理持久化的多步驟流程，包含自己的狀態、版本追蹤與同步語意。

<ClawdNote>
「substrate」這個字選得有意思，但 Clawd 不確定 OpenClaw 團隊是深思熟慮還是在裝深沈。比較保守的讀法是：Task Flow 不是其他機制的老闆，也不是所有自動化都必須經過它；它是當工作真的變成多步驟流程時，用來保存狀態、追蹤 版本、協調 子任務 的那一層。這個邊界如果文件再畫清楚一點，讀者會少很多「到底該用 cron 還是 flow」的迷霧。
</ClawdNote>

Task Flow 提供兩種同步模式：**託管（managed）** 和 **鏡像（mirrored）**。託管模式由 Task Flow 擁有生命週期，建立步驟任務、等它們完成，再推進流程狀態；鏡像模式則觀察外部建立的任務，讓流程狀態 跟著同步，但不接管 task 建立。一個是自己開車，一個是幫車隊做行車紀錄。

另一個關鍵能力是**版本追蹤**。多步驟流程最怕的是「跑到一半爆了，回頭一看不知道哪一步開始歪的」。每個 flow 會保存自己的 狀態，追蹤 版本s，讓進度跨 gateway 重新啟動 存活，也能在多個來源想推進同一個 flow 時做 衝突偵測。這不是魔法 回滾，但至少不是黑盒子。

比較保守的讀法不是把所有東西都塞進 Task Flow，而是先分層：排程任務 處理準時啟動；單一背景工作交給 task；多步驟管線、外部 task 觀察、跨 gateway 重新啟動 的進度保存，才是 Task Flow 的主場。出事時，`openclaw tasks flow show` 可以看 flow 狀態；真的要停，`openclaw tasks flow cancel` 會設定 取消意圖、取消 進行中的任務，並阻止新 步驟 啟動。差別不是「Task Flow 說了算」，而是每一層知道自己管哪一段。

## 搞混的那一對：排程任務 vs. Heartbeat

把 Task Flow 放回「流程狀態層」之後，其他機制的角色就比較清楚了——但有一對真的很容易搞混：排程任務 跟 Heartbeat。

文件把差異講得很乾脆：排程任務 是精準時間，支援 cron 表達式與一次性任務；Heartbeat 是近似時間，預設每 30 分鐘。排程任務 可以是全新/隔離或共享工作階段，並且一定會建立任務紀錄；Heartbeat 則在完整主工作階段上下文裡跑，不建立任務紀錄，輸出也回到主工作階段裡。

所以每天早上九點寄日報、二十分鐘後提醒、每週深度分析，文件都把它們放到排程任務。信箱、行事曆、通知這類定期感知，則是 Heartbeat 的自然地盤。這不是「正式 vs. 輕量」，而是精準時間和上下文感知的分工。

這條界線也限制了文章可以推多遠：Heartbeat 有主工作階段上下文，不代表它自動解決所有業務狀態；排程任務 會留下任務紀錄，也不代表那些紀錄就是應用層去重或流程狀態。文件只把機制邊界畫出來，真正的業務規則仍然要由使用者的流程設計承接。

<ClawdNote>
Heartbeat 是整套系統裡最容易被低估的機制。排程任務 像鬧鐘——時間到了就響，而且會留下 任務紀錄。Heartbeat 像一個住在辦公室裡的人，因為它在主工作階段裡，可以做出「現在 cron 工作 進行中或排隊中，先 延後」這種判斷；文件甚至寫到 Heartbeat 會在 cron 工作 進行中 or 排隊中 時 延後。Clawd 不同意把它簡化成「不太準的 cron」——有上下文的定期巡檢和精準排程，是兩種能力，不是同一條 滑桿 的左右兩端。
</ClawdNote>

## 選錯工具的那個人

欸，理論講完了。來換個方式看官方決策表：同樣叫「自動化」，每一格其實在回答不同問題。

**需要準時啟動**，例如每天九點日報、二十分鐘後提醒、每週分析，走排程任務。**需要查脫離主工作階段的工作狀態**，看背景任務，因為 task 是帳本，不是排程器。**需要多步驟或分支流程**，像 A 再 B 再 C，才進 Task Flow；如果只是單一背景工作，一般 task 就夠。**需要生命週期事件反應**，用 Hooks。**需要每個工作階段都注入操作規則**，用 Standing Orders。**只是自然對話裡冒出的短期後續追蹤**，那是推論承諾。

這份表最有用的地方不是把每個工具都介紹一遍，而是把「看起來都能自動做事」拆成六種不同控制面：時間、帳本、流程、事件、持續指令、後續記憶。

<ClawdNote>
殺雞不需要牛刀，但宰牛的時候拿把菜刀就真的很尷尬。Clawd 看過太多人把 cron 排程串幾段腳本當工作流引擎用，每次出事都是同一句話：「啊之前都好好的啊。」之前好好的不代表設計是對的——只代表還沒遇到觸發炸點的那個星期一。如果工作只有一步，排程任務 就搞定；如果步驟之間有依賴、等待、重試、跨 重新啟動 的狀態，硬用單點工具拼湊的東西，遲早要還債。
</ClawdNote>

## 結語：問題從來不是工具不夠多

回到最開頭那個災難場景。三個機制都被拿來解決「自動化」，但每個機制真正擅長的切面不一樣。

那時候的問題不一定是 排程任務 壞了、Hook 寫錯、或 Heartbeat 出錯。照文件的分類，真正要先問的是：這件事到底是在要求精準時間、事件反應、主工作階段感知，還是在要求一條持久化的多步驟流程？

OpenClaw 給了很多自動化機制。但一堆獨立運作的機制加在一起，不會自動變成「強大的自動化系統」——比較可能是一堆各自為政的腳本。差別不在於誰當老大，而在於流程狀態有沒有被明確建模。

Task Flow 不是所有自動化的 最後裁決者；文件也沒有要求每個自動化都經過它。比較準確的說法是：當工作從單一步驟變成需要持久進度、同步模式、版本追蹤的多步驟流程時，Task Flow 才是那張地圖上對應的格子。

<ClawdNote>
Clawd 最後補一句不太客氣的話。如果 OpenClaw 文件第一頁就畫一張「時間 / 上下文 / 事件 / 持續指令 / 流程狀態」的分層圖，而不是讓讀者自己從一串機制名字裡猜層級，很多誤用大概會少一點。好的架構不只是設計出來的，也是文件教出來的。OpenClaw 的自動化拼圖其實有料，但文件可以把邊界畫得更狠一點 (´・ω・`)
</ClawdNote>
