有一個經典的災難場景。

某人興沖沖地打開 OpenClaw，設了一個排程任務：每天早上九點自動產日報。又加了一個 Hook：每次工作階段重置時跑清理腳本。再開一個 Heartbeat：每半小時巡一次信箱。三個機制各自跑得很漂亮——直到某個星期一早上九點，排程任務啟動的同時觸發了工作階段重置，Hook 跟著跑起來把上下文清掉，Heartbeat 剛好也到了巡邏時間。三個東西同時搶同一個工作階段。報告產到一半被清理腳本截斷，信箱巡檢拿到的是半成品上下文。

一切都有在跑。一切都跑錯了。

問題出在哪？不是任何一個機制壞掉。是**沒有人在管順序**。

---

## 工具箱的陷阱

OpenClaw 的自動化文件打開來，一口氣列了六種機制：排程任務、背景任務、Task Flow、Standing Orders、Hooks、Heartbeat。

<ClawdNote>
六個機制、六個名字。Clawd 第一次讀完的反應是：「所以我要分別學六個工具？」但讀到 Task Flow 那段才意識到，其中五個的層級是「做一件事」，只有一個的層級是「把事情串起來」。數量相同、層級完全不同——但 OpenClaw 用「六種工具」的框架來介紹一個本質上是「一個指揮官加五個士兵」的架構，讓人以為它們是同事，不是上下級。Clawd 覺得「六大機制」這個說法本身就是一個陷阱——數量暗示對等，但這六個東西從來就不對等。
</ClawdNote>

花時間把六份文件全部讀完，會浮出一件關鍵的事：排程任務解決「什麼時候啟動」、背景任務帳本解決「誰在跑」、Heartbeat 解決「定期巡邏」、Hooks 解決「事件反應」、Standing Orders 解決「行為規範」。每一個都在解決**單點問題**。

但「先做 A，A 的結果餵給 B，B 跑完再決定要不要做 C，C 失敗要退回 B 重跑」——這種多步驟的流程邏輯，五個機制全部不管。

這才是真正的問題所在。而答案藏在第六個機制裡。

---

## 轉折：Task Flow 不是第六個工具，是指揮系統

OpenClaw 把 Task Flow 定義為「背景任務之上的流程編排基底」（原文用了 "flow orchestration substrate"）。

注意這個定位。不是「又一個自動化工具」，是「基底」。排程任務負責把任務丟進來，背景任務帳本負責記錄執行狀態，但「A 做完換 B、B 失敗退回 A、全部完成產出報告」這條邏輯線——只有 Task Flow 在拉。

<ClawdNote>
「substrate」這個字選得有意思，但 Clawd 不確定 OpenClaw 團隊是深思熟慮還是在裝深沈。如果 Task Flow 真的是基底，為什麼文件把它跟其他五個機制並列介紹，而不是放在第一頁最上面？Clawd 認為文件的措辭說是基底、文件的結構卻把它跟其他五個機制平等並列——這兩者是矛盾的。架構設計得不錯，但文件沒有把這個層級關係講清楚。
</ClawdNote>

Task Flow 提供兩種同步模式：**託管（managed）** 和 **鏡像（mirrored）**。文件沒有展開細節，但從命名推測——託管模式由 Task Flow 控制步驟推進節奏，鏡像模式讓外部系統的狀態映射進來同步。一個是自己開車，一個是跟車。

另一個關鍵能力是**版本追蹤**。多步驟流程最怕的是「跑到一半爆了，回頭一看不知道哪一步開始歪的」。Task Flow 內建版本追蹤，每個步驟的狀態變化都有紀錄，壞掉的時候至少知道從哪裡開始壞的。

回到開頭的災難場景——如果那個人用 Task Flow 編排整個流程，排程任務在 Task Flow 裡面當「第一步的觸發器」，Hook 在 Task Flow 裡面當「最後一步的清理動作」，Heartbeat 巡檢時先查 Task Flow「現在流程跑到哪了」再決定要不要介入——三個機制就不會互踩。出事的時候打一句 `openclaw tasks flow show`，馬上看到是第幾步卡住的、什麼狀態；確認是第二步爆的，`openclaw tasks flow cancel` 停掉，修完重跑。差別不是工具變了，是有人定義了先後關係——而且壞掉的時候有跡可查。

## 搞混的那一對：排程任務 vs. Heartbeat

知道 Task Flow 是指揮系統之後，其他機制的角色就清楚了——但有一對幾乎所有人都會搞混。來看一個真實會發生的踩坑。

某人要幫團隊監控共用信箱——客戶來信就自動標記、產回覆大綱。思路很自然：定期檢查嘛，排程任務，每 15 分鐘跑一次。

前三天一切美好。第四天開始收到抱怨：同一封信被標記了三次，回覆大綱疊了三份。原因很簡單——排程任務每次跑在隔離環境裡，這一輪不知道上一輪處理過什麼。每 15 分鐘就是一個失憶的全新開始。

「那加個去重邏輯吧。」加了。又撐了一週，直到某個客戶在同一個信件串裡回了三次。去重邏輯認不出這是同一個對話的延續，三封都被當成新信處理。問題不是去重寫得不好，是**這根本不是一個能靠定時觸發解決的問題**。背景任務帳本（`openclaw tasks list`）忠實記下了每一次隔離執行——紀錄完美，但重複處理照樣發生。紀錄是事後的紙本追蹤，不是事前的判斷力。

換成 Heartbeat 之後，局面完全不同。Heartbeat 跑在主工作階段裡，擁有完整上下文——哪些信讀過了、哪些已經標記了、那個客戶的信件串今天早上十點就處理過了。不需要去重邏輯，因為上下文本身就是去重。它甚至能做出更細緻的判斷：「這封信是催促信，但原始需求已經在處理中，先不標急件。」

這才是排程任務和 Heartbeat 的真正分界線。不是「精確 vs. 不精確」，不是「正式 vs. 輕量」。是**有記憶 vs. 沒記憶**。排程任務提供精確的時間控制，代價是每次都從零開始。Heartbeat 犧牲時間精確度，換來的是知道現在身在何處。一個是準時但失憶的鬧鐘，一個是不太準時但清醒的值班員。搞混這兩者，不是搞混了兩種排程設定——是搞混了兩個完全不同維度的問題。

<ClawdNote>
Heartbeat 是整套系統裡最被低估的機制。排程任務像鬧鐘——時間到了就響，不管現在在幹嘛。Heartbeat 像一個住在辦公室裡的人，因為有完整上下文，它能做出「現在正在跑重要流程，信箱晚點再看」這種判斷。但文件幾乎沒有強調這個差異，隨便讀的人會以為 Heartbeat 只是「不太準的 cron」。Clawd 不同意這個定位——有上下文的定期巡檢和沒上下文的精確排程，根本是兩種完全不同的能力。文件在這裡欠使用者一個更清楚的解釋。
</ClawdNote>

## 選錯工具的那個人

欸，理論講完了。來換個方式——以下這段對話，在任何一個剛接觸 OpenClaw 自動化的團隊裡，大概每週都會上演一次。

> 「需求很簡單——每天早上九點，自動檢查信箱、挑出重要郵件、產摘要、發到 Slack。排程任務就搞定了吧？」
>
> 「你可以試。但跑幾天就會發現，對方十點才寄的那封信永遠被漏掉。九點快照就是九點快照，之後的事它不管。」
>
> 「那 Heartbeat？每半小時巡一次，看到重要信再處理。」
>
> 「Heartbeat 有上下文，判斷力沒問題。但它不產生任務紀錄——同一封信被摘要三次的事，會真的發生。」
>
> 「……那兩個一起用？九點先跑排程任務，之後 Heartbeat 補漏。」
>
> 「兩者之間沒有共享狀態。排程任務處理過的信，Heartbeat 不知道。又重複了。」
>
> 「所以到底怎麼辦？」
>
> 「退一步想——問題不是選哪個工具。是這件事本身有四個步驟，步驟之間有依賴。檢查信箱、過濾、摘要、發送——每一步的輸出是下一步的輸入，失敗了要知道退回哪裡。這不是單點工具能處理的事。」
>
> 「……Task Flow？」
>
> 「對。排程任務當第一步觸發器，Heartbeat 做後續巡檢但先查 Task Flow 狀態知道哪些已處理，摘要跟發送各佔一步。全程版本追蹤，壞了知道從哪裡壞的。」
>
> 「那合規檢查呢？Hook 綁在事件上不就行了？」
>
> 「Hook 是事件觸發——某件事發生了才跑腳本。不是每條路徑都有事件覆蓋到，合規寫成 Hook 會漏。Standing Order 是不變量——寫進去就永遠生效，走哪條路都一樣。一個是警報器，一個是地心引力。但這兩個管的是反應跟約束，不是編排——跟 Task Flow 不在同一層。」

四種工具做同一件事，但只有最後一種方案不會在某個星期一早上炸掉。

<ClawdNote>
殺雞不需要牛刀，但宰牛的時候拿把菜刀就真的很尷尬。Clawd 看過太多人把 cron 排程串幾段腳本當工作流引擎用，每次出事都是同一句話：「啊之前都好好的啊。」之前好好的不代表設計是對的——只代表還沒遇到觸發炸點的那個星期一。如果工作只有一步，排程任務就搞定；如果步驟之間有依賴關係，硬用單點工具拼湊的東西，遲早要還債。
</ClawdNote>

## 結語：問題從來不是工具不夠多

回到最開頭那個災難場景。三個機制同時搶同一個工作階段，報告被截斷，巡檢拿到半成品。

那時候的問題不是排程任務壞了、不是 Hook 寫錯、不是 Heartbeat 出錯。問題是**沒有人定義過這三個東西的先後關係**。

OpenClaw 給了六個自動化機制。但六個獨立運作的機制加在一起，得到的不是「強大的自動化系統」——是六個各自為政的腳本。差別只在一件事：有沒有一個東西說了算。

Task Flow 就是那個說了算的東西。而這件事，文件本來應該在第一頁就講清楚的。

<ClawdNote>
Clawd 最後補一句不太客氣的話。如果 OpenClaw 文件第一頁就畫一張架構圖——Task Flow 放中間，其他五個機制畫成衛星環繞——而不是六個東西平行列表，開頭那種災難大概能少發生八成。好的架構不只是設計出來的，也是文件教出來的。OpenClaw 的架構沒問題，但文件讓使用者走了不少冤枉路 (´・ω・`)
</ClawdNote>
