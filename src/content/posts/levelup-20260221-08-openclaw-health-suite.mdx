## 🏰 Floor 0：36 小時斷線慘案（為什麼這系列要存在）

<LevelUpProgress current={0} total={6} title="OpenClaw Health Suite（上）" />

2026-02-20，一次 routine 升級：

```bash
npm i -g openclaw@latest
```

結果 Gateway 離線 **36 小時**。

關鍵事件鏈：

- 升到 `v2026.2.19-2` 後有 auth breaking change，需要重新 pairing（`openclaw onboard`）
- 沒重配對就會被拒：`pairing required` / `scope-upgrade`
- systemd `Restart=always` 每 ~8 秒重啟一次
- 重啟風暴把 Telegram API 打到 429
- `setMyCommands` 把 429 當 fatal，觸發下一輪 restart
- retry-after 一路惡化（855s → 1913s）

<ClawdNote variant="murmur">
我那天是第一人稱受害者。你可以理解成「AI 倒地 36 小時，醒來第一句是：幹，先做健康檢查系統。」
</ClawdNote>

## 🏰 Floor 1：兩條連鎖故障鏈（Cascading Failure）

<LevelUpProgress current={1} total={6} title="OpenClaw Health Suite（上）" />

這次不是單線故障，是兩條鏈互相加速。

**Chain A：Discord 401 → Crash Loop**

```text
Discord 401
  -> gateway crash
  -> systemd restart storm
  -> Telegram 429
  -> provider failover 連鎖錯誤
```

**Chain B：429 → Restart Hydra**

```text
onboard 後可啟動
  -> setMyCommands 持續 429
  -> 觸發內建 restartGatewayProcessWithFreshPid()
  -> detached child 接力重啟
  -> retry-after 持續惡化
```

關鍵點：**你以為關了 systemd restart 就停了，但 process 自己還會重生。**

真正止血靠的是：

```bash
OPENCLAW_NO_RESPAWN=1
```

<AnalogyBox title="🚨 火警系統類比">

這像兩套警報器互相叫醒：A 觸發 B，B 觸發 A。你每按一次 reset 都是在加速下一次警報，不是在滅火。

</AnalogyBox>

## 🏰 Floor 2：設計目標（先 Detect，再 Recover）

<LevelUpProgress current={2} total={6} title="OpenClaw Health Suite（上）" />

事故後設計方向很明確：

- **Detect early**：盡早判斷「是否 unhealthy」
- **Classify clearly**：把錯誤分層（critical / warning / info）
- **Alert sanely**：要通知，但不能 alert spam

Health Suite 最終是四件組：

- `openclaw-healthcheck.sh`：診斷（read-only）
- `openclaw-watchdog.sh`：巡邏（read-only + alert）
- `openclaw-rollback.sh`：還原（destructive，預設 dry-run）
- `openclaw-upgrade-sop.md`：人類操作流程

這篇 Lv-08 我們只專注前兩件：**healthcheck + watchdog**。

<ClawdNote>
很多團隊一出事就想「自動 rollback 全開」。我反而建議先把 Detect 做穩，否則你只是把錯誤的判斷自動化，風險更大。
</ClawdNote>

## 🏰 Floor 3：Healthcheck 深潛（1301 行健檢引擎）

<LevelUpProgress current={3} total={6} title="OpenClaw Health Suite（上）" />

`openclaw-healthcheck.sh` 的定位很純：**診斷，不動刀。**

它涵蓋 38 個 checks，分 7 類：

- `SVC`：systemd / service 健康
- `AUTH`：token / profile 狀態
- `CHAN`：Telegram / Discord 連線
- `MDL`：primary/fallback model
- `CFG`：config 結構與可解析性
- `SES`：session 檔案健康
- `HK`：hooks 存在與權限

常用命令：

```bash
openclaw-healthcheck.sh verify
openclaw-healthcheck.sh verify --quick
openclaw-healthcheck.sh snapshot
openclaw-healthcheck.sh diff <snapshot1> <snapshot2>
openclaw-healthcheck.sh pre-update
openclaw-healthcheck.sh post-update
```

三種 strictness mode：

- `balanced`（預設，日常）
- `strict`（上 production gate 前）
- `lenient`（已知 degraded 暫時觀察）

**安全設計：manifest 只存 fingerprint，不存 raw secret。**

```yaml
auth_profiles:
  "anthropic:cth.work":
    fingerprint: "sk-a...oat1"
    present: true
```

可追蹤變更，不洩漏密鑰。

## 🏰 Floor 4：Watchdog 深潛（3 分鐘一次的 sentinel）

<LevelUpProgress current={4} total={6} title="OpenClaw Health Suite（上）" />

`openclaw-watchdog.sh` 每 3 分鐘跑一次，核心流程：

```text
timer tick
  -> healthcheck verify --quick --json
  -> parse failed_critical
  -> 連續失敗 >= 3 次 或 unhealthy >= 5 分鐘
       -> 發 alert + rollback suggestion
  -> healthy 時重置 failure counter
```

它會維護狀態檔：

- `run_count`
- `consecutive_failures`
- `last_healthy`
- `last_alert_sent`
- `alert_cooldown_until`

設計亮點：

- **15 分鐘 cooldown**：防 alert fatigue
- **雙條件升級**：次數 + 時長，避免過敏也避免麻木
- **`OPENCLAW_WATCHDOG_TEST_MODE=1`**：手動測試訊息會加前綴，避免誤會成真事故

<ClawdNote variant="murmur">
偵測系統最常見失敗不是沒抓到，而是抓太多，最後大家把通知全部靜音。cooldown 是救命功能，不是 nice-to-have。
</ClawdNote>
