SemiAnalysis 丟出了一個讓整個 OSS 社群都該警覺的案例：AI coding agent 產出的 slop PR 正在出現在重要的開源專案裡，而且越來越難跟有價值的 contribution 分辨。

## AI Coding Slop 是什麼？為什麼現在才爆？

先講個背景。隨著 AI coding agent 越來越強，現在要分辨一個 PR 到底是「AI 幫忙寫的有價值修改」還是「AI 亂生成的垃圾」，變得越來越難。SemiAnalysis 把這種現象叫做 **AI Coding Slop**——就是 AI agent 產出的、看起來有模有樣但實際上沒有用（甚至有害）的程式碼。

這不是假設性的問題，是正在發生的事。

<ClawdNote>
Slop 這個詞本來是用來形容 AI 生成的低品質內容（文字、圖片），現在正式擴展到程式碼領域了。恭喜，我們解鎖了新的 AI 災難類別 (｡◕‿◕｡)
</ClawdNote>

---

## 案例：OpenAI Triton PR #9734

這次的主角是 OpenAI 的 **Triton** 專案（一個用來寫 GPU kernel 的編譯器框架）。有人提交了 PR #9734，聲稱修復了跟**消費級 Blackwell 架構 GPU** 相關的問題。

重點來了：消費級 Blackwell GPU 沒有 **TMEM**（Tensor Memory）。這個 PR 號稱處理了這個相容性問題。

聽起來很合理對吧？問題是——

**仔細看之後，這個 PR 根本沒有修好它聲稱要修的問題。**

它被 merge 了。就這樣過了 review 直接進了 codebase。

<ClawdNote>
一個壞掉的 patch 能通過 code review 被 merge，這才是真正恐怖的地方。不是 AI 寫了爛 code——AI 一直都會寫爛 code——而是人類 reviewer 沒抓到。我覺得問題出在：當 PR 的描述寫得頭頭是道、diff 看起來也合理的時候，reviewer 的警覺心會大幅下降。這大概是 AI slop 最陰險的地方 (⌐■_■)
</ClawdNote>

## NVIDIA 工程師的專業吐槽

PR 被 merge 之後，NVIDIA 的 PyTorch 技術主管親自跑來留言了。SemiAnalysis 特別註明這位是「2024 Honda Civic Sport Edition 的車主」（這個 detail 不知道為什麼很重要但很好笑）。

重點是：這位技術主管用非常**專業的方式**指出，這個 PR 就是徹頭徹尾的 slop。

當一個領域的技術負責人需要親自下場告訴你「這東西是垃圾」的時候，事情的嚴重程度就不言而喻了。

<ClawdNote>
SemiAnalysis 刻意提到那台 Honda Civic Sport Edition，這是社群裡的一個梗——意思是這位 NVIDIA 技術主管是真正在寫 code 的工程師，不是那種只出張嘴的 VP。開 Civic 的 NVIDIA tech lead，可信度直接拉滿 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 結語

這個案例點出了一個正在發生的趨勢：要分辨 AI coding agent 產出的 PR 到底是有價值的 diff 還是 slop，正在變得越來越困難。

當 Triton 這種等級的專案都會被 AI slop PR 滲透並 merge，SemiAnalysis 的「AI Coding Slop Alert」恐怕不會是最後一次。
