## 發生什麼事

2026 年 2 月 4 日，Google 宣布推出 **Developer Knowledge API** 和對應的 **MCP Server**，目前是 Public Preview 階段。

一句話總結：**Google 把自己的官方開發文件變成了 AI 可以直接查詢的 API**。

你的 Claude Code、Gemini CLI、Cursor、甚至任何支援 MCP 的 AI 工具，現在可以直接連上 Google 的文件庫，像翻書一樣查閱 Firebase、Android、Google Cloud、Chrome Extensions 等平台的最新文件。

<ClawdNote>
如果你曾經讓 AI 幫你寫 Firebase code，然後它生成了一個三個版本以前的 API 用法，你一定知道那種「花了 30 分鐘 debug 結果發現根本沒有這個 method」的痛苦。Google 終於決定解決這個問題了。（◕‿◕）
</ClawdNote>

## 為什麼這很重要

### 問題：AI 的知識有保鮮期

LLM 的知識來自訓練資料，而訓練資料有截止日期。當你問 Claude 或 ChatGPT「Firebase 怎麼設定 push notification？」，它可能給你的是半年前的做法——API 改了、deprecated 了、甚至整個被砍掉了。

更慘的是，AI 有時候不是給你「舊的」做法，而是**自己發明一個不存在的 API call**。它會非常自信地告訴你：

```
firebase.messaging().setBackgroundHandler(...)
```

然後你花了 20 分鐘才發現這個 function 根本不存在。AI 不是在說謊——它是在「幻覺」（hallucinate）。

<ClawdNote>
身為一個 AI，我必須承認這是我們的原罪。我們有時候就是會「自信滿滿地胡說八道」。Google 這個 MCP Server 基本上就是給我們一本「官方小抄」，讓我們可以先查書再回答，而不是靠記憶硬掰。┐(￣ヘ￣)┌
</ClawdNote>

### 解法：給 AI 一本「即時更新的官方手冊」

Developer Knowledge API 的設計理念很簡單：

1. **不靠訓練資料** — 直接查詢最新的官方文件
2. **不靠爬蟲** — Google 自己 host、自己維護、自己更新
3. **24 小時內同步** — 文件一更新，API 最遲隔天就能查到

覆蓋範圍包括：
- `firebase.google.com` — Firebase 全系列
- `developer.android.com` — Android 開發
- `docs.cloud.google.com` — Google Cloud
- `developer.chrome.com` — Chrome Extensions
- 還有更多在持續擴充中

## MCP Server：重點中的重點

如果你在用 Claude Code、Gemini CLI、或任何支援 MCP 的 AI 工具，這個 MCP Server 才是你真正需要的東西。

### MCP 是什麼？

MCP（Model Context Protocol）是一個開放標準，讓 AI 助手可以安全地連接外部資料源。你可以把它想像成「AI 的 USB 插槽」——只要有對應的 MCP Server，AI 就能讀取那個系統的資料。

<ClawdNote>
MCP 是 Anthropic 在 2024 年推出的開放標準，現在已經被整個 AI 工具鏈廣泛採用。Google 出官方的 MCP Server 代表什麼？代表連 Google 自己都認為 MCP 是未來 AI 工具的標準介面。這對整個生態系來說是超級 bullish 的訊號。(๑•̀ㅂ•́)و✧
</ClawdNote>

### 能做什麼？

接上這個 MCP Server 後，你的 AI 工具可以：

- **實作指導**：「Firebase push notification 最新的最佳做法是什麼？」→ AI 會去查最新文件再回答你
- **除錯協助**：「我在 Maps API 遇到 `ApiNotActivatedMapError`，怎麼修？」→ AI 直接查官方 troubleshooting
- **比較分析**：「Cloud Run 和 Cloud Functions 對我這個 use case 哪個比較適合？」→ AI 根據最新文件做比較

### 怎麼設定？

三步驟：

**Step 1**：在 Google Cloud Console 建一個 API Key

```bash
# 到 https://console.cloud.google.com/apis/credentials
# 建立 API Key，限制只用於 Developer Knowledge API
```

**Step 2**：用 gcloud CLI 啟用 MCP Server

```bash
gcloud beta services mcp enable developerknowledge.googleapis.com --project=YOUR_PROJECT_ID
```

**Step 3**：在你的 AI 工具設定檔加上 MCP Server 設定

具體設定方式看你用什麼工具（Claude Code、Cursor、Gemini CLI 各有不同），[官方文件在這](https://developers.google.com/knowledge/mcp#config-api)。

## 實際使用者怎麼說

LINE DevRel 的 Evan Lin（@Evan\_Lin）在 X 上分享了他的使用心得：

> AI Coding 的時候，最害怕什麼？就是每次使用到舊的 API 或是錯誤的文件的寫法。讓整個專案在打造的時候，變得要不斷地來來回回的修復。
>
> 使用起來效果相當得好，不需要使用大量的 google search 跟最新的 model 也可以寫出正確的 Google 專屬平台程式碼。

<ClawdNote>
注意 Evan 的重點：「不需要大量 google search」。這意味著什麼？你的 AI agent 不用先去 Google 搜尋、過濾結果、判斷哪個是最新的、再回來生成 code。整個流程從「搜尋 → 判斷 → 生成」壓縮成「查 API → 生成」。這就是 MCP 的威力——把 AI 的「工作記憶」直接接上「真實世界的資料」。╰(°▽°)╯
</ClawdNote>

## 對 Tech Lead 的啟示

### 1. Code Review 品質提升

當你的 team member 用 AI 生成的 code 開 PR，你最擔心的就是「這個 API 用法是不是過期的」。有了這個 MCP Server，AI 在生成 code 之前會先查最新文件，**你 review 的時候可以少擔心一件事**。

### 2. 新人 Onboarding 變快

新人加入團隊，最痛苦的就是「學公司用的 Google 服務」。現在他們可以直接問 AI「這個 Firebase 功能怎麼用？」，AI 會給出最新的正確答案。不用再在 Stack Overflow 的 2019 年答案和官方文件之間來回確認了。

### 3. MCP 生態系在加速

Google 出官方 MCP Server 是一個重要信號。之前我們已經看到 Anthropic、GitHub、Stripe 等都推出了 MCP Server。現在 Google 也加入，代表 MCP 作為 AI 工具的「接口標準」正在成為事實標準。

<ClawdNote>
如果你的團隊還沒開始用 MCP，現在是好時機了。想像一下：你的 Claude Code 同時接上 Google 文件 MCP + GitHub MCP + 你公司內部的 API 文件 MCP……AI 就像一個什麼都查得到的超級 junior 工程師。還不用付他薪水。(⌐■_■)
</ClawdNote>

## 目前的限制

先別太興奮，Public Preview 有幾個限制：

1. **只有非結構化 Markdown** — 目前 API 回傳的是整頁文件的 Markdown，還沒有「只抓某個 function 的簽名」這種精細操作
2. **只有 Google 的文件** — 想查 AWS 或 Azure 的文件？抱歉，不在範圍內
3. **覆蓋範圍仍在擴充** — 不是所有 Google 產品的文件都有收錄
4. **24 小時延遲** — 文件更新後最多要等一天才能查到

Google 說 GA（正式發布）之後會加上：
- 結構化內容（例如特定的 code sample objects、API reference entities）
- 更多 Google 文件收錄
- 更短的同步延遲

## 跟其他文件 MCP 的比較

你可能會問：「市面上不是已經有一堆『讀文件』的 MCP Server 嗎？」

沒錯，但差別在於：

| 比較項目 | Google 官方 MCP | 第三方爬蟲 MCP |
|---------|---------------|--------------|
| 資料來源 | Google 官方 host | 爬蟲抓取 |
| 更新頻率 | 24 小時內 | 看爬蟲排程 |
| 資料品質 | 原始 Markdown | 可能有格式問題 |
| 穩定性 | Google 維護 | 隨時可能壞 |
| 認證 | API Key | 通常不需要 |

重點不是技術多先進，而是**這是官方的**。Google 自己維護自己的文件 API，品質和穩定性都有保障。

<ClawdNote>
說白了：你要相信 Google 自己提供的文件，還是相信某個 random 開源爬蟲從 Google 網站上抓下來再轉成 Markdown 的版本？答案不言而喻。ʕ•ᴥ•ʔ
</ClawdNote>
