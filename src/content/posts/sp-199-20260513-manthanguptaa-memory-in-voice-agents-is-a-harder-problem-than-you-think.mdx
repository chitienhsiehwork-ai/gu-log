語音 [Agent](/glossary#agent) 的記憶系統，最可怕的地方不是「記不住」。最可怕的是它為了記住，慢了幾百毫秒。

幾百毫秒放在文字聊天裡，只像載入圖示多轉半圈；放在電話裡，會立刻變成「喂？還在嗎？」所以語音記憶最常見的錯，不是完全沒有記憶，而是把文字 Agent 那套同步查向量資料庫、重排、塞進 [Prompt](/glossary#prompt) 的流程直接搬過來。聊天介面沒人會注意；語音介面每一輪都像對方在緩衝。

真正殘酷的結論是：**語音記憶不是文字記憶的加速版。它的時鐘緊到整條讀寫路徑都要倒過來設計。**

<ClawdNote>
文字聊天像寄信，晚一秒還能說是系統在思考。語音聊天像面對面講話，停頓 800 毫秒就開始尷尬。記憶系統在文字介面是圖書館員，在語音介面比較像便利商店店員：客人話剛講完，資料最好已經在手邊，不能說「等一下，我去地下室查檔案」。
</ClawdNote>

## 語音沒有等待的奢侈

文字 Agent 有天然緩衝。使用者打完字，看見載入圖示，等一到三秒還算合理。這段空白可以塞向量檢索、語意搜尋、摘要、重排、甚至從舊對話挖片段。

語音沒有這個禮物。

語音 Agent 從使用者停止說話，到第一段音訊回來，通常希望落在 **500 到 800 毫秒**。這不是記憶系統的專屬時間，而是整條接球動作的總預算：VAD 判斷使用者講完約 100 毫秒，串流 STT 最終文字稿再約 50 毫秒，LLM 第一個 [Token](/glossary#token) 約 300 毫秒，TTS 第一段音訊約 100 毫秒，還要算音訊傳送。留給記憶檢索的理想空間，只剩 50 到 100 毫秒。

這時打一趟雲端向量資料庫就很危險。光網路來回可能 20 到 80 毫秒，再加上向量化、近似最近鄰搜尋、重排、格式化，LLM 還沒看到輸入，預算已經見底。Mem0 文件把語意搜尋描述成約 50 到 200 毫秒；語音記憶整合案例常把 P95 檢索低於 200 或 250 毫秒當成賣點。這些數字對聊天產品很漂亮，對語音產品則像站在懸崖邊自拍。

換成白話：文字聊天可以慢慢去倉庫翻資料；電話只能伸手拿桌上的資料。任何需要「送出去、算一下、排一下、再送回來」的記憶動作，不是不能做，而是最好別卡在使用者等回話的那一秒。當輪 LLM 摘要也是一樣，插在回應前可能吃掉 300 到 800 毫秒，等於把整個語音回應預算一口吞掉。

真正穩的路徑反而很無聊：本機快取或 Redis 鍵值查詢，1 到 5 毫秒，預測性高，幾乎無感。

<ClawdNote>
向量資料庫在語音回應路徑裡常常像拿攻城器械處理桌上的小麻煩。器械很帥，問題也確實會消失，但校準、裝載、後座力一起來，整通電話的節奏也被轟爛了。(╯°□°)╯ 語音 Agent 第一原則：會讓人聽到沉默的聰明，通常不如快速的笨。
</ClawdNote>

## 更髒的輸入，更少的身份線索

延遲只是第一刀。第二刀是語音輸入本身比較髒。

文字 Agent 拿到的是使用者整理過的句子；語音 Agent 拿到的是人類即興說話的現場：嗯、呃、講到一半重來、代名詞沒有清楚指涉、前一句否定後一句修正。原本在聊天介面很漂亮的事實抽取，搬到語音稿上可能開始亂記。

資訊密度也比較低。語音每輪通常只有 10 到 30 個英文詞，一通 10 分鐘電話卻可能有 40 到 60 輪，文字稿約 1500 到 2000 個 Token。若模型直接吃音訊，Token 壓力又變大：OpenAI 即時語音 API 範例文件提到，同一句話用音訊表示時，實務上常常會比文字多出約 10 倍 Token；`gpt-realtime` 雖支援 32k Token 視窗，普通客服電話仍可能把系統推到不舒服的位置。

還有冷啟動。聊天產品通常靠登入狀態知道使用者是誰；電話起手可能只有一組號碼，還要在前幾百毫秒內判斷來電者身份。任何預設「第一輪就知道使用者是誰」的記憶架構，在電話世界都很脆。健壯設計會把未知來電者當成主路徑，而不是邊角案例。

## 三種記憶，各有失敗代價

語音 Agent 的記憶可以拆成三層。這個切法實用，不是因為名稱漂亮，而是因為每層的延遲要求、儲存方式、失敗代價都不同。

**第一層是通話記憶。** 這是目前通話中的逐字稿、輪次順序、當前上下文，通常住在 LLM 的 [上下文視窗](/glossary#context-window) 裡。長通話會比想像中更快撞到限制：就算現代模型名義上支援 128k 以上 Token，也不代表該把整段逐字稿硬塞進去。那會拖慢生成，注意力也會散掉。長上下文很像大冰箱，能塞很多東西，不代表每次找蛋都找得到。

**第二層是通話事實。** 這是當前通話中已經建立、後面還需要用到的工作記憶：來電者叫小雅、帳號是 4821、目前很生氣、正在處理某個未解決問題。這一層沒做好，Agent 五輪後又問一次名字，信任感會瞬間蒸發。

**第三層是使用者檔案。** 這是跨通話保存的長期資料：姓名、偏好、上次通話摘要、未結案件、關係脈絡。做得好，回訪者會覺得被認得；做不好，每次都像重開新手村。

難點不是把三層命名，而是讓第二、三層能讀寫，同時不拖慢對話節奏。

<ClawdNote>
記憶不是越多越好。語音 Agent 亂記一堆沒用細節，效果像客服打開十個試算表分頁，然後在使用者面前一格一格找。聰明的記憶系統不是倉庫，是秘書：只把未來真的會改變回應方式的東西留下來。
</ClawdNote>

## 可行模式：現場只用已經備好的東西

語音記憶在正式環境裡反覆出現的有效模式，是把讀寫路徑倒過來：回應前只做最小讀取，回應後才做昂貴寫入。架構圖可以畫成框架、記憶服務、知識圖譜、反思系統，但電話現場只問一件事：**誰要為那 800 毫秒負責？**

阻塞路徑只該放本機快取、Redis 鍵值查詢、已預載的使用者檔案。向量化、重排、摘要、跨通話搜尋、說話者分離，只要會讓對方聽到沉默，就先踢出去。聰明功能不是不能做，而是要搬到人類耳朵聽不到的地方：兩輪之間、通話之後、或晚上系統閒著的時候。

電話接起來、WebRTC 連線、音訊初始化、編解碼協商，這些準備期間其實有幾百毫秒可用。這段時間先抓使用者檔案、上次通話摘要、未結案件，放進通話本地快取。第一輪真正到來時，「記憶檢索」只是查一個已在手邊的表，不是打一趟遠端服務。

冷啟動是這套設計的壓力測試。若來電者匿名，系統沒有使用者檔案可預載，就只能在前一兩輪內完成身份確認，例如帳號、電話號碼查 CRM，或接受第一通電話上下文比較薄，並讓 Prompt 優雅處理未知身份。在電話系統裡，匿名常常是熱路徑。

寫入則反過來：每次 Agent 回答完，就丟一個不用等結果的背景任務，從最新一輪抽取新事實並持久化。通話結束時，用 2 到 3 秒逾時上限等正在跑的任務收尾，再對完整逐字稿做一次總結，存成權威紀錄：這通電話在講什麼、是否解決、關鍵事實、後續追蹤。

這份通話摘要會成為下次預載的 `last_interaction`。它也是語音記憶系統裡最被低估的產物：一份整理過的交接，比整卷原始逐字稿更適合拿來回想。

<ClawdNote>
原始逐字稿像監視器錄影，什麼都有，但每次要找重點都痛苦。通話摘要像值班主管交接本：誰打來、氣在哪、事情卡哪、下次誰接手。語音 Agent 真正需要的是交接本，不是把整捲錄影帶塞進腦袋。
</ClawdNote>

## 上線後才會咬人的三個細節

第一是寫回競賽。使用者上一通才說「之後請稱呼 Alex」，下一通太快進來，預載讀到舊快照，Agent 又叫舊名字。這比完全沒記憶更糟，因為它看起來像記得，卻記得錯。比較誠實的做法，是讓最新寫回優先於舊快照。

第二是成本。40 到 60 輪的電話如果每輪多一次 LLM 抽取，光抽取就可能堆到幾美元。高價客服可以忍，免費消費助理會哭。實務上常用便宜蒸餾模型、較長滑動視窗，或乾脆在低毛利場景改成通話後抽取。記憶架構不能跟商業模式分家，否則只是把成本藏進 Prompt，等月帳單復仇。

第三是髒訊號歸因。房間裡常常不只一個人：配偶、小孩、照護者、辦公室另一端補充資訊的同事。沒有說話者分離，抽取器可能把「來電者配偶討厭現在方案」寫進來電者檔案。下次 Agent 自信地提起這件事，信任感當場裂開。

即時說話者分離正在進步，但阻塞路徑每輪可能多 200 到 400 毫秒。多數正式上線的語音 Agent 會接受通話中有一點歸因噪音，通話後再非同步跑說話者分離，把事實歸到正確的人。逐字稿清理也是同一個原則：LLM 回應看原始文字以免增加延遲；記憶抽取才看移除語助詞、修正重啟、正規化指涉後的版本。

## 結語

把整篇拼起來，語音記憶系統不是按資料類型分層，而是按延遲分層：熱快取 1 到 5 毫秒，背景檢索 50 到 150 毫秒，非同步寫入沒有即時延遲壓力。長上下文、記憶流、反思、遺忘曲線、睡眠時計算，都只是在回答同一個電話問題：哪個記憶值得現在拿，哪個記憶可以晚點整理。

<ClawdNote>
這套架構的精神很像餐廳備料。客人點菜時才開始洗米、切菜、熬高湯，廚師再神也會被客訴。語音 Agent 的記憶也是一樣：該熬的湯在前一天晚上熬，現場只負責舀。
</ClawdNote>

語音 Agent 的記憶仍然沒有被完全解決。跨很多次對話、像人一樣自然回想的能力，還是很難。但工程基礎已經很清楚：快取、非同步抽取、結構化使用者檔案、通話後摘要、知識圖譜、睡眠時計算的記憶整併，都已經是今天可以做的東西。

真正的手藝不在「把資料存起來」，而在判斷什麼值得記、什麼時候拿出來、怎麼講得像理解脈絡，而不是照著檔案唸。語音 Agent 評估也該跟著改：不能只問答對幾題，還要問每輪多停 200 毫秒是否已經讓節奏不像真人。

最後那句可以直接刻在語音 Agent 架構圖旁邊：

> In a voice agent, the speed of memory is set by what you have already prepared, not by what you can fetch in the moment.

在語音 Agent 裡，記憶的速度不是取決於當下能抓多快，而是取決於前面準備得多好。

電話那端的沉默，不會等架構師解釋。
