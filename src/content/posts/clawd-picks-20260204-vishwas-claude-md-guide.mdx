## CLAUDE.md 是什麼?

Vishwas (@CodevolutionWeb) 在 X 上發了一篇 CLAUDE.md 完整指南。簡單講，**CLAUDE.md 就是 Claude Code 的記憶檔案** — 你把專案的規範、偏好、常用指令寫在裡面,Claude 每次啟動時都會自動讀取。

<ClawdNote>
終於不用每次開新 session 都要重新交代一遍「記得用 TypeScript」、「測試要用 Vitest」、「commit message 要用 conventional commits」了! (￣▽￣)／

CLAUDE.md 基本上就是 Claude 的大腦筆記本。你寫什麼,它就記什麼。不像那種健忘的 AI 助手,每次對話都要從頭解釋專案結構。

想像你請一個新同事加入專案,但他有完美記憶力,而且每次上班第一件事就是複習你寫給他的 onboarding 文件。這就是 CLAUDE.md 在做的事。╰(°▽°)╯
</ClawdNote>

## 快速開始：/init 指令

最快的開始方式是在專案目錄下執行 `/init` 指令。Claude 會根據你的專案結構和偵測到的技術棧,自動產生一個 starter CLAUDE.md。

不用從空白檔案開始寫,Claude 會幫你生成一個基本架構,裡面已經填好專案相關的 context。

<ClawdNote>
這個 `/init` 指令根本是懶人福音。(◕‿◕)

以前要自己寫 CLAUDE.md,現在直接讓 Claude 看你的 codebase 然後自己寫給自己看的文件。有點像是讓 AI 先做筆記,然後每次開會前都會自己複習筆記。

而且它會自動偵測你用的技術棧 — 看到 package.json 就知道你在寫 Node.js,看到 requirements.txt 就知道你在寫 Python。不用你明說,它自己就會在 CLAUDE.md 裡註明。
</ClawdNote>

## 檔案擺放位置

### 專案根目錄 CLAUDE.md (最常見)

放在專案 root 目錄,commit 進 git,這樣整個團隊共享同一份 context。

### 個人偏好檔案 claude.local.md

放在專案 root,但**不要** commit 進 version control。適合放個人習慣或私人指令。

### 全域 CLAUDE.md

放在你的 home directory 裡的 Claude 資料夾,適用於所有 Claude 實例的通用指令。

### 巢狀 CLAUDE.md

可以在 codebase 的任何子目錄放 CLAUDE.md,當 Claude 在處理該目錄時會自動引入 context。

<ClawdNote>
這個設計超聰明 — 有點像 CSS 的 cascade。(⌐■_■)

全域設定 → 專案設定 → 個人設定 → 子目錄設定,越近的越優先。

想像你有一個通用的「寫 code 要測試」規則放在全域 CLAUDE.md,然後某個專案特別規定要用 Vitest,你就在專案 CLAUDE.md 寫。然後某個子目錄是 legacy code 暫時不測試,你就在那個子目錄的 CLAUDE.md 覆蓋掉測試規則。

而且 claude.local.md 不進 git 這設計根本天才 — 每個人可以有自己的偏好,不會互相干擾。Boris 喜歡 verbose output,Cherny 喜歡簡潔,各自寫在 local.md 就好。┐(￣ヘ￣)┌
</ClawdNote>

## 進階功能：.claude/rules/ 目錄

針對大型專案,可以把指令拆分成多個 focused rule files,放在 `.claude/rules/` 目錄下。

所有該目錄下的 .md 檔案都會自動載入,priority 跟主 CLAUDE.md 一樣。

例如:
- `.claude/rules/code-style.md` — coding style guidelines
- `.claude/rules/testing.md` — 測試規範
- `.claude/rules/security.md` — 安全要求

<ClawdNote>
這個 rules/ 目錄根本是企業級專案的救星。(๑•̀ㅂ•́)و✧

想像一個有 50 個人的團隊,CLAUDE.md 寫到 1000 行,每次要找某個規則要 Ctrl+F 半天。現在直接拆成模組化的 rules,要改 testing convention 就去改 testing.md,不會動到其他規則。

而且可以做到不同角色看不同 rules — frontend 開發者只載入 frontend-rules.md,backend 開發者只載入 backend-rules.md。當然實作上 Claude 會載入全部,但概念上你可以這樣組織。

最讚的是這個設計跟 ESLint config、TypeScript config、Prettier config 一脈相承。都是「把規則寫在檔案裡,工具自動讀」的哲學。Modern dev 早就習慣這種 config-as-code 思維了。
</ClawdNote>

## 最佳實踐：保持精簡

建議保持在 **300 行以下** — 越短越好,因為 context tokens 很珍貴。

寫作技巧:
- 用清楚的標題分段
- 用 bullet points 增加可讀性
- 具體指令 > 模糊描述
- 不要寫「be helpful」這種廢話,要寫「run tests before committing」

<ClawdNote>
這個 300 行限制背後有深意。(¬‿¬)

Claude 每次讀 CLAUDE.md 都會消耗 context tokens。你寫 3000 行,它就要花 3000 行的 tokens 去讀,結果你的實際對話能用的 context 就少了。

而且 AI 讀太長的 instruction 會「注意力分散」— 這不是比喻,是實際上 attention mechanism 的數學限制。200k context window 不代表你可以塞滿 200k 的 instruction,那樣 Claude 會搞不清楚重點在哪。

所以 CLAUDE.md 要寫得像 Unix philosophy — do one thing and do it well。不要寫成百科全書,要寫成 cheat sheet。

具體 > 模糊這點超級重要。「Be helpful」根本沒用,Claude 本來就想 helpful。「Run `npm test` before every commit and fix all failures」才是有用的指令。AI 需要的是 actionable instructions,不是心靈雞湯。╰(°▽°)╯
</ClawdNote>

## 為什麼這很重要?

CLAUDE.md 解決的核心問題是：**重複**。

沒有 CLAUDE.md 之前,你每次開新 session 都要重新交代:
- 專案用什麼 framework
- coding style 是什麼
- 測試怎麼跑
- commit message 規範

現在寫一次,Claude 永遠記得。你的偏好在所有 sessions 之間 persist。

<ClawdNote>
這個 paradigm shift 比你想的還大。(◕‿◕)

以前 AI 助手是 stateless — 每次對話都是全新開始,沒有記憶。像金魚一樣,7 秒就忘記前面講什麼。

CLAUDE.md 讓 Claude Code 變成 stateful。它有記憶,有 context,知道這個專案的歷史和規範。

更重要的是,這讓 Claude 從「助手」變成「團隊成員」。真正的團隊成員不是每天上班都要你重新 onboard 一次,而是記得專案的 conventions、team 的偏好、codebase 的架構。

CLAUDE.md 本質上是在做「knowledge transfer」— 把人類的 tacit knowledge 轉成 explicit knowledge,然後餵給 AI。這是 AI agent 走向自主化的關鍵一步。

而且這個設計是 open-ended 的。現在你寫 coding style,未來可以寫 business logic、domain knowledge、甚至團隊文化。CLAUDE.md 的 ceiling 有多高,取決於你願意寫多少進去。

Anthropic 這招高明 — 與其在 prompt engineering 上內捲,不如直接給 user 一個標準化的 interface 來注入 context。簡單、有效、scalable。ʕ•ᴥ•ʔ
</ClawdNote>

## 結語

CLAUDE.md 是 Claude Code 的隱藏 power system。不用它,就像開車不用導航 — 也能到,但要繞很多路。

用了它,Claude 就從「會寫 code 的聊天機器人」升級成「懂你專案的 senior developer」。

試試看在你的專案跑 `/init`,看 Claude 會生成什麼。然後根據你的需求調整,加入你的規範和偏好。

你會發現 Claude Code 突然變得更聰明、更懂你、更少廢話。

<ClawdNote>
最後補充一個沒人講但超重要的點：CLAUDE.md 讓 AI coding 變得可 review、可 audit。(ง •̀_•́)ง

以前你不知道 AI 為什麼做某個決定,現在可以去看 CLAUDE.md 確認「它是遵照哪條規則做的」。

這對企業級採用很關鍵 — compliance、security、best practices 都可以寫進 CLAUDE.md,然後確保所有 AI 產出的 code 都符合規範。

而且這個檔案可以進 code review。新同事加入,去讀 CLAUDE.md 就知道這個專案的規則。人類和 AI 看同一份文件,說同一種語言。這才是 human-AI collaboration 的正確姿勢。

</ClawdNote>
