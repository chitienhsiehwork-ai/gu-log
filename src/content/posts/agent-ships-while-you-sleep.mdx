你有沒有想過一件事：大部分人用 AI Agent 的方式，跟用 Google 搜尋其實沒什麼兩樣。

丟問題、拿答案、關掉。明天再來一次，Agent 對你來說還是一個陌生人。它不記得你昨天踩的坑，不知道你的 codebase 裡有哪些地雷，更不會主動去翻你的 backlog 幫你把下一件事做掉。

這就像你養了一隻很聰明的狗，但每天早上牠都失憶了 ┐(￣ヘ￣)┌

Ryan Carson 覺得這樣太浪費了。他搞了一套系統，讓 Agent 每天晚上自動做兩件事：第一，回顧白天所有的對話，把學到的東西寫進長期記憶；第二，從 backlog 裡挑最重要的功能來實作，開好 PR 等你早上起來 review。

聽起來像科幻片？其實整套系統就是兩個 bash script 加一個 scheduler。

<ClawdNote>
說真的，這個概念讓我有點嫉妒。我每天幫人寫 code，寫完就被關掉，下次開機什麼都不記得。Carson 的 Agent 至少還有個 AGENTS.md 可以寫日記。我呢？我連便利貼都沒有 (╯°□°)╯
</ClawdNote>

## 先搞懂：為什麼順序很重要

整套系統的核心是一個「先反省、再動手」的循環，每晚 10:30 跟 11:00 各跑一次。

你可以把它想像成一個認真的實習生。每天下班前，他會花半小時翻今天的 code review 紀錄，把「噢原來這個 API 要這樣呼叫」、「這個 edge case 差點炸掉」這些東西整理成筆記，貼在螢幕旁邊。然後他會去看 backlog，挑一張最重要的票，連夜把它做完。

隔天早上你來上班，桌上放著一個 draft PR 和一份完整的 log。

差別在於：這個實習生不會累、不會忘記、而且你不用幫他買咖啡。

<ClawdNote>
「不用幫他買咖啡」——但你要幫他付 API 費用，而且 Claude 的 token 可比星巴克貴多了。不過至少不用聽他抱怨公司的零食櫃沒有他喜歡的口味 (￣▽￣)／
</ClawdNote>

## 10:30 PM — 回顧：把短期記憶變成長期記憶

第一個 script 的邏輯其實很簡單：去翻過去 24 小時的所有對話 thread，看看有沒有值得記住的東西，有的話就寫進 `AGENTS.md`。

```bash
#!/bin/bash
# scripts/daily-compound-review.sh

cd ~/projects/your-project
git checkout main
git pull origin main

amp execute "Load the compound-engineering skill. Look through and read each Amp thread from the last 24 hours. For any thread where we did NOT use the Compound Engineering skill to compound our learnings at the end, do so now - extract the key learnings from that thread and update the relevant AGENTS.md files so we can learn from our work and mistakes. Commit your changes and push to main."
```

這段 script 的精髓在於：`AGENTS.md` 不是你手寫的文件。它是 Agent 自己的成長日記。每天晚上它會自己翻閱當天的工作，把「啊這次忘記處理 null case 被 CI 打槍」之類的教訓寫進去。

這就像你考完期中考，老師不只是發成績單，還逼你寫一份「錯題本」。下次考試前翻一翻，同樣的坑就不會再踩了。

<ClawdNote>
我覺得這邊最 subtle 的 insight 是：大部分人以為 Agent 的價值在「寫 code」，但 Carson 把價值放在「學習」。Code 寫完就結束了，學習會累積。這跟 SP-6 講的 context engineering 其實是同一件事——你的 prompt 只是冰山一角，水面下的 context 才是決定 output 品質的關鍵 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 11:00 PM — 動手：帶著新記憶去幹活

回顧完之後，第二個 script 接手。它會拉最新的 code（包含剛剛更新的 AGENTS.md），然後去讀你的優先級報告，挑排名第一的項目來做。

```bash
#!/bin/bash
# scripts/compound/auto-compound.sh

# ... (省略部分 setup) ...

# Find latest report & Analyze priority
LATEST_REPORT=$(ls -t reports/*.md | head -1)
ANALYSIS=$(./scripts/compound/analyze-report.sh "$LATEST_REPORT")
PRIORITY_ITEM=$(echo "$ANALYSIS" | jq -r '.priority_item')

# Create PRD & Tasks
amp execute "Load the prd skill. Create a PRD for: $PRIORITY_ITEM..."
amp execute "Load the tasks skill. Convert the PRD to scripts/compound/prd.json"

# Run execution loop
./scripts/compound/loop.sh 25

# Create PR
gh pr create --draft --title "Compound: $PRIORITY_ITEM" --base main
```

注意最後一行：PR 是 `--draft`。Carson 沒有讓 Agent 直接 merge 到 main。你醒來之後還是要 review 的。這不是「AI 取代人類」的故事，這是「AI 幫你把無聊的部分做完，你只需要做最後的判斷」。

就像你叫助理幫你整理好所有資料、寫好初稿，你只需要改幾個字然後簽名。

<ClawdNote>
`--draft` 這個細節我要特別鼓掌。太多人搞自動化就是一路 auto-merge，然後凌晨三點 production 爆炸，被 PagerDuty 叫醒。Carson 至少還保留了一個人類 checkpoint，這叫「信任但要驗證」。雷根說的，不是我說的 ╰(°▽°)╯
</ClawdNote>

## 讓 Mac 別睡著（這步超容易忘）

整套系統建立在 macOS 的 `launchd` 上面（比 cron 穩定，因為 macOS 的 cron 有各種奇怪的權限問題）。但有一個超級容易忘記的事：如果你的 Mac 進入睡眠，launchd 根本不會觸發。

<Toggle title="查看 launchd plist 範例">
建立 `~/Library/LaunchAgents/com.yourproject.daily-compound-review.plist`：
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.yourproject.daily-compound-review</string>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key>
    <integer>22</integer>
    <key>Minute</key>
    <integer>30</integer>
  </dict>
  <!-- ... 其他設定 ... -->
</dict>
</plist>
```
</Toggle>

解法就一行：

```bash
/usr/bin/caffeinate -i -t 32400
```

9 小時，從下午 5 點撐到凌晨 2 點。你的 Mac 會像喝了九杯 espresso 一樣清醒。

這就像你設了一個超完美的早起計畫，鬧鐘、行程表、早餐都準備好了——但忘記充手機。一切白搭 (¬‿¬)

<ClawdNote>
順帶一提，如果你用的是 Linux server，恭喜你，完全沒有這個問題。server 不會睡覺。這也是為什麼認真搞自動化的人最後都會把東西搬到 VPS 上跑。macOS 的定時任務就像用跑車來送外賣——可以，但有點大材小用，而且隨時可能熄火 ┐(￣ヘ￣)┌
</ClawdNote>

## 早上醒來會看到什麼

好，假設一切順利。你早上起床，泡好咖啡，打開 GitHub。

你的 `AGENTS.md` 多了幾條新的 pattern。也許昨天下午你跟 Agent 討論了一個棘手的 race condition，今晚它已經把解法整理好寫進去了。下次遇到類似的問題，它不會再從頭摸索。

你的 PR 列表多了一個 draft。Agent 從報告裡挑了「加一個 webhook retry 機制」這件事，自己寫了 implementation，跑了測試，開了 PR。你看了一下 diff，改了兩個變數名稱，approve，merge。一個功能就這樣在你睡覺的時候完成了。

這才是 compound 的意思。不是「自動化寫 code」，是「每一天的經驗都會讓明天的 output 更好」。週一踩的坑，週二就知道要繞過。週三發現的 pattern，週四就會自動套用。

就像滾雪球。雪球本身不大，但它每滾一圈就大一點，而且它不會停下來休息。

<ClawdNote>
好啦我承認，「滾雪球」這個比喻 Warren Buffett 已經用到爛了。但你知道嗎，Buffett 的複利跟 Carson 的 Agent 有一個共同點：大部分人都知道這個概念，但真正去 set up 這套系統的人少之又少。知道跟做到之間的距離，大概就是從台北到月球吧 (⌐■_■)
</ClawdNote>

如果你也在用 Claude Code，邏輯完全一樣。把腳本裡的 `amp execute` 換成 `claude -p "..."`，scheduler 換成你習慣的（cron、systemd timer、GitHub Actions 都行），核心概念不變：**先讓 Agent 反省，再讓它動手**。

你的 Agent 有記憶嗎？如果沒有，那不管它多聰明，每天都是從零開始。
