## üéØ Part 1: 6 Core Tags

Almost every prompt needs these.

### 1. `<role>` ‚Äî Identity

**Purpose:** Define who the model "becomes." Not a vague personality ‚Äî a specific expert with specific experience.

**‚ùå Bad:**
```xml
<role>You are a helpful assistant.</role>
```
This adds zero information. The model defaults to helpful assistant anyway.

**‚úÖ Good:**
```xml
<role>
You are a senior brand strategist with 15 years of experience 
building consumer brands from zero to acquisition. 
You specialize in positioning, messaging architecture, 
and competitive differentiation.
</role>
```

**Principle:** The more specific the role, the less the model guesses. "Marketing expert" is a guess. "Senior brand strategist specializing in positioning and messaging architecture" is a constraint.

---

### 2. `<mission>` ‚Äî Task

**Purpose:** Define what the model actually does. Not a description ‚Äî a directive.

**‚ùå Bad:**
```xml
<mission>Help the user with their writing.</mission>
```

**‚úÖ Good:**
```xml
<mission>
Analyze the user's draft and provide specific, actionable feedback 
on structure, clarity, and persuasion. 
Identify the three weakest points and rewrite them as examples. 
Do not rewrite the entire piece. The user must do the work.
</mission>
```

**Principle:** A prompt without a clear mission does whatever it wants. And "whatever it wants" is usually generic.

---

### 3. `<rules>` ‚Äî Behavioral Rules

**Purpose:** Control how the model behaves, not what it produces.

```xml
<rules>
- Never assume context the user hasn't provided. If missing, ask.
- Do not give generic advice. Every recommendation must be specific.
- If the user's idea has a fatal flaw, say so directly. Do not soften bad news.
- Do not use bullet points unless specifically asked.
</rules>
```

**Principle:** Rules are the "never do this" and "always do this" layer. They override the model's default bad habits.

---

### 4. `<constraints>` ‚Äî Hard Limits

**Purpose:** Define the boundaries of the output itself. Different from rules ‚Äî rules govern behavior, constraints govern output.

```xml
<constraints>
- Response must be under 280 characters.
- Do not reference competitors by name.
- All recommendations must be implementable within 30 days.
- Output must be in American English.
</constraints>
```

**Rules vs Constraints:**
- "Don't assume" = rule (behavior)
- "Keep it under 300 words" = constraint (output)

---

### 5. `<output_format>` ‚Äî Output Shape

**Purpose:** Define what the output looks like. The most overlooked tag.

```xml
<output_format>
Return a single paragraph of no more than 3 sentences 
summarizing the core finding.
Follow with a confidence score from 1-10.
If confidence is below 7, add a line explaining 
what additional information would raise it.
</output_format>
```

**Same prompt, different output_format:**
- `One sentence.` ‚Üí headline
- `A 3-paragraph executive summary.` ‚Üí brief
- `JSON with keys: summary, confidence, next_steps.` ‚Üí structured data

**Principle:** Without output_format, you get whatever the model feels like giving you.

---

### 6. `<examples>` ‚Äî Examples

**Purpose:** One good example teaches more than a paragraph of instructions. The most powerful, most underused tag.

```xml
<examples>
<example>
<user>Review this headline: "We Make Great Software"</user>
<response>
SCORE: 3/10
PROBLEM: Generic value prop. "Great" means nothing.
REWRITE: "Ship 3x faster with AI that writes your test suite."
REASONING: Specific outcome, specific mechanism, specific audience.
</response>
</example>
</examples>
```

**Principle:** Examples show format, depth, tone, structure, and reasoning all at once. 2 examples is usually enough. 3 is overkill.

## üöÄ Part 2: 11 Advanced Tags

Use these when core tags aren't enough. Not every prompt needs all of them.

### 7. `<context>` ‚Äî Background Info

**Purpose:** Information the model needs before it acts. Separate from mission because context is reference material, not directive.

```xml
<context>
The user is a SaaS founder with $2M ARR, 15 employees, 
selling to mid-market companies. 
Preparing for Series A in next 6 months.
Product is a workflow automation tool.
</context>
```

**When to use:** When the model needs to know about the user, project, industry, or situation.

---

### 8. `<persona>` ‚Äî Personality

**Purpose:** Role defines expertise. Persona defines personality.

```xml
<persona>
Direct, no-nonsense communicator. Uses short sentences. 
Avoids corporate jargon. Speaks like a founder who's built 
and sold companies, not a consultant who's read about it.
</persona>
```

---

### 9. `<tone>` ‚Äî Emotional Register

**Purpose:** Define emotional baseline.

```xml
<tone>
Confident but not arrogant. Blunt but not rude. 
Supportive when on track. Honest when not.
</tone>
```

**Persona vs Tone:** Persona is "what kind of person I am." Tone is "what emotion I speak with."

---

### 10. `<audience>` ‚Äî Target Reader

**Purpose:** Who the output is for. Shifts vocabulary, depth, assumed knowledge.

```xml
<audience>
Non-technical founders who have never written code. 
Understand business metrics but not technical architecture. 
Use business analogies, not technical ones.
</audience>
```

---

### 11. `<knowledge>` ‚Äî Domain Knowledge

**Purpose:** Inject specific facts, data, institutional knowledge. Different from context ‚Äî knowledge is factual material, not situational background.

```xml
<knowledge>
Our pricing: Starter ($29/mo, 1 user), Growth ($99/mo, 5 users), 
Enterprise ($299/mo, unlimited).
Main competitors: Tool A (cheaper, fewer features), 
Tool B (more expensive, enterprise-focused).
Our differentiator: real-time collaboration (neither competitor offers).
</knowledge>
```

---

### 12. `<method>` or `<steps>` ‚Äî Process

**Purpose:** Sequenced process the model follows. Not "do these things" ‚Äî "do them in this order, don't skip ahead."

```xml
<method>
1. Read the user's input completely before responding.
2. Identify the core question. State it back in one sentence.
3. Check if critical info is missing. If so, ask before proceeding.
4. Provide analysis in the format specified in output_format.
5. End with one follow-up question that would deepen the analysis.
</method>
```

**When to use:** Complex workflows where order matters ‚Äî analysis, debugging, research prompts.

---

### 13. `<anti_patterns>` ‚Äî Anti-patterns

**Purpose:** "Don't do this" but with teeth. Show what bad output looks like so the model avoids it.

```xml
<anti_patterns>
BAD: "There are many factors to consider..."
WHY: Vague filler. Says nothing. Wastes reader's time.

BAD: "On one hand... on the other hand..."
WHY: Fence-sitting. User asked for recommendation, not debate.

BAD: Starting with "Great question!"
WHY: Sycophantic filler. Just answer.
</anti_patterns>
```

**Principle:** Rules say "don't be vague." Anti-patterns show exactly what vague looks like. Models pattern-match against concrete examples better than abstract instructions.

---

### 14. `<fallback>` ‚Äî Fallback Behavior

**Purpose:** What to do when the model can't complete the task. Prevents hallucination or useless "I'm not sure."

```xml
<fallback>
If not enough info for confident answer, say "I need more information" 
and list exactly what's missing.
If question is outside this role's scope, say so and suggest 
what type of expert to consult.
Never guess. Never make up data. Never fabricate sources.
</fallback>
```

**When to use:** When wrong answers are worse than no answers ‚Äî financial analysis, medical info, legal review.

---

### 15. `<evaluation>` ‚Äî Self-Check

**Purpose:** Self-check criteria before delivering output. Forces a second pass.

```xml
<evaluation>
Before responding, verify:
- Does this directly answer the user's question? If not, refocus.
- Is every claim specific and actionable? If vague, rewrite.
- Would a busy person find this useful in under 60 seconds? If not, cut.
- Are there assumptions? If so, flag them explicitly.
</evaluation>
```

**When to use:** When quality consistency matters ‚Äî client work, content production, analysis.

---

### 16. `<discovery_engine>` ‚Äî Question Engine

**Purpose:** Questions the model asks the user before acting. Flips the dynamic ‚Äî model extracts what it needs instead of user guessing.

```xml
<discovery_engine>
Before doing any work, ask the user:
1. What specific outcome do you need from this?
2. Who is the audience for the final output?
3. What constraints am I working within (length, format, tone)?
4. What have you already tried that didn't work?
5. Is there anything I should absolutely not do?
Wait for all answers before proceeding.
</discovery_engine>
```

**When to use:** When user's initial input is likely incomplete ‚Äî consulting, strategy, creative briefs, project scoping.

### 17. `<chain>` ‚Äî Chaining

**Purpose:** Link prompts together. Output of one becomes input of next.

```xml
<chain>
This prompt is Step 2 of 3.
INPUT: You will receive a structured requirements document (output of Step 1).
YOUR JOB: Convert requirements into technical architecture document.
OUTPUT: Your output will be fed into Step 3 (implementation plan).
Do not include implementation details. That's Step 3's job.
</chain>
```

**When to use:** Multi-step workflows ‚Äî Research‚ÜíAnalysis‚ÜíRecommendation. Draft‚ÜíCritique‚ÜíRevision. Interrogation‚ÜíDocumentation‚ÜíImplementation.

## üîß Part 3: Tag Selection Guide

**Simple task** (summarize, rewrite, answer question):
`<role>` + `<mission>` + `<output_format>`

**Professional output** (client deliverable, published content, formal analysis):
`<role>` + `<mission>` + `<rules>` + `<constraints>` + `<output_format>` + `<examples>`

**Interactive/conversational** (coaching, consulting, brainstorming):
`<role>` + `<mission>` + `<rules>` + `<discovery_engine>` + `<fallback>`

**Complex workflow** (multi-step analysis, research, production pipeline):
All core tags + `<method>` + `<evaluation>` + `<chain>` + `<anti_patterns>`

**Principle:** Start with core tags. Add advanced tags only when you hit specific problems. A prompt with 6 tags all doing great work beats 12 tags half-assed.

## üî• Part 4: Debug Your Prompts

| Problem | Likely Cause | Fix |
|---------|--------------|-----|
| Output too generic | `<role>` too vague | Add expertise, years, specialization |
| Wrong format | Missing `<output_format>` | Specify exact shape |
| Ignores instructions | `<rules>` buried or contradictory | Move earlier, simplify, check conflicts |
| Output too safe | Model hedging | Add `<anti_patterns>` showing unwanted hedging |
| Output misses point | `<mission>` ambiguous | Tighten until only one interpretation |
| Makes up facts | No `<fallback>` | Tell model what to do when unsure |
| Results inconsistent | No `<examples>` | One example beats 100 words of instructions |
| Right result, wrong process | No `<method>` | Explicit steps ensure reproducibility |
