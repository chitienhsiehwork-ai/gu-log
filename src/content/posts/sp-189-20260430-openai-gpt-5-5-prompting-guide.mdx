星期二早上的工程師打算把上週寫好的 GPT-5.4 prompt 直接搬去 GPT-5.5 跑——同一份系統 prompt、同一組 few-shot 範例、同一份「ALWAYS do this NEVER do that」清單，七百多行。模型換好、benchmark 跑下去：延遲變慢、輸出變得異常機械、有時候模型會繞一個大圈才回到該講的事情。

挫敗之餘打開 OpenAI 官方文件想找哪邊改錯了。結果第一段就寫著一句不像廠商文件的話：

> Avoid carrying over every instruction from an older prompt stack. Legacy prompts often over-specify the process because earlier models needed more help staying on track. With GPT-5.5, that can add noise, narrow the model's search space, or lead to overly mechanical answers.

翻成大白話：那七百行 prompt 是上一代模型的拐杖。GPT-5.5 拿著拐杖反而走不動，請拆掉一半。

整份 [GPT-5.5 prompting guide](https://developers.openai.com/api/docs/guides/prompt-guidance?model=gpt-5.5) 就是在繞這個點。OpenAI 把 GPT 家族（4.1、5、5.1、5.2、5.3-Codex、5.4、5.5）的 prompt 指南通通放同一頁，可以橫向對比 prompt 寫法怎麼一代一代演化。最新這代的真正重點一句話：**描述目的地，別畫地圖**——告訴模型什麼叫做「好結果」，剩下的路徑讓模型自己選。

這跟 Anthropic 在 [SP-175](/posts/sp-175-20260416-anthropic-opus-4-7-prompting-best-practices/) 給 Opus 4.7 的 best practice 講的「intent-first」幾乎是同一句話換個皮。兩家旗艦模型不約而同往同個方向跑，至少表示舊 prompt stack 不能再原封不動搬到新模型上。

<ClawdNote>
這份文件有意思的地方在於 OpenAI 罕見地把「我們之前教的東西現在會反咬」寫成正式文件。對 prompt engineer 來說，這等於 release note 上寫：「過去兩年的肌肉記憶，部分過期。」

比較友善的解讀是：廠商終於誠實了。always/never 那種寫法本來就是模型不夠強時的補丁，補丁該丟就要丟。比較不友善的解讀是：prompt 教學半年就會翻新一次，真正該學的不是這張清單，而是判斷哪條建議什麼時候會過期 (¬‿¬)
</ClawdNote>

## 第一刀砍向 process-heavy：講目的地，不要畫地圖

GPT-5.5 段落開門見山就講這條轉變：

> GPT-5.5 works best when prompts define the outcome and leave room for the model to choose an efficient solution path.

過去寫 prompt 的標準起手式是「First inspect A, then inspect B, then compare every field, then think through all possible exceptions, then decide which tool to call, then call the tool, then explain the entire process to the user」——把整條 SOP 一步一步寫死，避免模型跳針或漏步驟。對 GPT-4 / GPT-4.1 / GPT-5 那種「需要被牽著走」的模型，這套有效。對 GPT-5.5，這套會反咬一口：模型把 SOP 當絕對指令，不管實際情境是不是真的需要每一步都跑，於是輸出又慢又死板。

替代寫法是把同一個任務改成「成功長什麼樣」：

```
Resolve the customer's issue end to end.

Success means:
- the eligibility decision is made from the available policy and account data
- any allowed action is completed before responding
- the final answer includes completed_actions, customer_message, and blockers
- if evidence is missing, ask for the smallest missing field
```

注意 prompt 結構整個變了：沒有「先做 X 再做 Y」、沒有 step-by-step、沒有「ALWAYS check this」「NEVER do that」。只有四件事：要解決什麼、什麼叫做成功、輸出長什麼樣、卡住了要做什麼。剩下的所有決策——要查哪個 API、用什麼工具、什麼順序、要不要跑驗證——都丟給模型自己選。

對應建議是：**ALWAYS / NEVER / must / only 這些絕對字眼，留給真的非黑即白的事**——譬如安全規則、必填欄位、絕對不能做的事。判斷題（什麼時候該搜尋、什麼時候要停下來問、什麼時候用工具）改用 decision rule 而不是強制句。Decision rule 的格式長這樣：

```
Use the minimum evidence sufficient to answer correctly,
cite it precisely, then stop.
```

這跟「ALWAYS cite all available sources」的差別是：前者讓模型自己判斷「夠了沒」，後者強迫模型把整本文件都翻一遍。對 GPT-5.5 這種已經會自己判斷夠不夠的模型，後者只會浪費 token 和延遲。

<ClawdNote>
這條轉變對寫過 GPT-3.5 時代 prompt 的開發者來說，會有一種「靠，那段七百行 prompt 我前年寫了一週」的痛感。當時模型真的會跳針、真的會漏步驟、真的需要被「First do X, then Y」牽著鼻子走。那不是 prompt engineer 寫太多，是模型不夠強。

GPT-5.5 的能力提升把那個必要性消滅了，但同時把寫過大量 prompt 的人變成負債者：累積的「該怎麼壓模型行為」的肌肉記憶，現在反而是噪音。比較健康的心態是把舊 prompt 當成 git history 的考古層——曾經很好，但這層該封存了。

順帶一提，OpenAI 還很雞婆地給了一個一鍵升級工具——`$openai-docs migrate this project to gpt-5.5`，丟進 [Codex](/glossary#codex) 它會幫忙改寫 prompt stack。這算是廠商第一次把「prompt 升級」當成一級自動化任務在賣，雖然這個工具本質就是「另一個 prompt 在改 prompt」。Inception (⌐■_■)
</ClawdNote>

## 第二刀拆兩塊：personality 跟 collaboration style 不是同一件事

第二刀不是砍流程，而是拆線路。GPT-5.5 把「assistant 怎麼講話」跟「assistant 怎麼做事」拆成兩個獨立區塊：**personality** 跟 **collaboration style**。

Personality 管「聽起來像誰」：語氣、溫度、正式程度、幽默感。Collaboration style 管「怎麼合作」：何時問問題、何時自行假設、要多主動、遇到不確定時怎麼處理。

過去這兩件事常常被綁在一起寫——「You are a friendly and proactive assistant who...」一句話交代完。問題是友善跟主動是兩條獨立軸，硬綁在一起的後果是：想調整其中一條（譬如語氣想更專業一點），另一條會被連動修改（連帶變得不主動）。GPT-5.5 對這種 prompt 寫法很敏感，所以官方建議直接拆開寫。

文件裡的兩個範本，一個沉穩任務導向，一個活潑有存在感。差別不在誰比較高級，而是使用場景不同：內部工具需要穩、短、可靠；面向使用者的 chatbot 可能需要比較像「有人在陪」。但兩個範本都不該代替任務目標、成功條件、工具規則、停止條件。

實作上的心智模型：prompt 像洋蔥。最內層是任務目標和成功條件，中間是 collaboration style，最外層才是 personality。改外層不該震到內層，不然調語氣調到任務定義變形，這顆洋蔥就開始流眼淚。

<ClawdNote>
這個分法看起來顯而易見，但真實 prompt 常常把角色、語氣、合作方式、安全規則全塞進一句「You are a senior Python engineer...」。一改語氣，連「重複檢查」的紀律一起被改掉，bug 就這樣混進上線環境。

比較乾淨的做法是：內部工具和使用者 chatbot 共用任務目標，只換 personality 區塊。這比在同一份 prompt 裡塞 if-else 清楚太多，維護成本也少一截。
</ClawdNote>

## 拐杖拆掉之後，最怕模型一路走到隔壁縣市

把七百行 SOP 拆掉，不代表 prompt 可以變成「麻煩自由發揮，謝謝」。工程師最怕的下一幕很熟：模型終於不被牽著走了，然後自己走得很開心，一路查資料、開工具、再查資料，像進賣場本來只要買牛奶，最後推車裡多了氣炸鍋和露營椅。

GPT-5.5 文件補的 preamble、retrieval budget、stopping condition，看起來像三個新名詞，其實都在做同一件事：替模型裝紅綠燈。綠燈是「可以自己判斷」，黃燈是「先讓使用者知道正在做什麼」，紅燈是「夠了，停車」。

### Preamble：first-token 延遲的視覺解藥

GPT-5.5 在處理大量 tool call 或多步驟任務時，可能先花時間推理、規劃、準備工具呼叫；使用者看到的則是一片空白。Preamble 規則就是讓模型動手前先丟一句短訊息：

```
Before any tool calls for a multi-step task, send a short
user-visible update that acknowledges the request and states the
first step. Keep it to one or two sentences.
```

這招不是讓模型變快，是讓使用者**感覺**比較快。串流介面先看到一句「先查 X」，等待焦慮會少很多。OpenAI 把這叫做「improve perceived responsiveness」，講得很誠實：不是推理變快，是空白畫面不要那麼像當機。

### Retrieval budget：給搜尋一個天花板

Retrieval budget 是這份文件最有畫面感的新概念：先做一次廣泛搜尋；如果結果已經足夠支撐核心回答，就停手。只有核心問題沒被回答、必要欄位缺失、使用者要求完整比較、指定文件必須讀，或答案會留下重要無根據主張時，才再搜一次。

痛點很實際：答案一需要出處，模型就容易陷入「再搜一次說不定更準」的迴圈，token 跟延遲一起爆。Retrieval budget 不必硬寫「最多搜尋三次」，而是把「夠了沒」變成判準。

### Stopping condition：取代 ALWAYS / NEVER 的判斷題

Stopping condition 則把同一個概念套到工具迴圈：每跑完一輪，就問自己「現在能不能回答了？」

```
Resolve the user query in the fewest useful tool loops, but do not
let loop minimization outrank correctness, accessible fallback
evidence, calculations, or required citation tags for factual claims.

After each result, ask: "Can I answer the user's core request now
with useful evidence and citations for the factual claims?"
If yes, answer.
```

這比「最多 5 次工具呼叫」有彈性，也比「該停就停」具體。好的 decision rule 不是替模型做決定，而是把模型該問自己的問題寫出來。前面那七百行 prompt 是家長陪寫作業，這裡改成把檢查清單貼在桌邊：模型可以自己做，但每做完一輪要抬頭看一下紅綠燈。

<ClawdNote summary="Decision rules 把判斷權還給模型。">
這三招共同點是：**把判斷權還給模型，但給模型一套可重複自查的問題**。過去是「幫模型把所有情境列完」，現在是「教模型怎麼判斷」。

這很違反工程直覺，因為控制權被讓出去了一點。但 GPT-5.5 的訊號是：模型已經能接住部分判斷，硬寫 if-else 反而會把它綁住。順帶吐槽，retrieval budget 其實比較像 retrieval policy；只是 budget 聽起來比較性感，OpenAI 應該知道自己在幹嘛 (¬‿¬)
</ClawdNote>

## Coding agent 的硬技術段：別把「正在查」存成「已完成」

這段技術名詞比較硬，但真正訊息只有兩個坑：第一，應用程式重播模型輸出時，不要把「正在查」和「已完成」混在一起；第二，修檔格式不要自創到連模型都沒看過。

### Phase parameter：assistant item 的階段標籤

Responses API 從 GPT-5.4 開始引入 `phase` 欄位，GPT-5.5 沿用。這個欄位用來區分「中間更新」和「最終答案」：

```
If manually replaying assistant items:
- Preserve assistant `phase` values exactly.
- Use `phase: "commentary"` for intermediate user-visible updates.
- Use `phase: "final_answer"` for the completed answer.
- Do not add `phase` to user messages.
```

如果應用端用 `previous_response_id`，API 會自動保留狀態。真正會踩雷的是手動重播 assistant output items：每個 item 的 `phase` 必須原封不動傳回去。漏傳或改錯，模型就可能把「正在查」當成「已經答完」。GPT-5.4 文件還特別提醒遇到這種 bug 先檢查 phase，可見不少團隊已經摔過。

### apply_patch：OpenAI 把 diff format 變成一級工具

GPT-5.1 開始，OpenAI 把 `apply_patch` 升級成 first-class 工具型別：

```python
tools=[{"type": "apply_patch"}]
```

就能拿到內建的 `create_file`、`update_file`、`delete_file` 操作。關鍵是 OpenAI 自己的測試結果：**named function 比自製 apply_patch 把失敗率降低 35%**。

翻成工程決策：coding agent 若自己手寫 tool description，或換一套自製 diff format，失敗率可能直接變高。原因不是魔法，是模型被特別 post-train 過 OpenAI 的 diff format。除非 patch 必須先過特殊 review pipeline，否則自製 patch 工具不是創新，是把訓練紅利丟掉。

GPT-5.5 還補了一條 coding agent 專用的驗證建議：改完之後跑最相關的檢查，可能是精準單元測試、型別檢查、程式碼風格檢查、建置，或最小冒煙測試；不能跑就說明原因和下一個最佳檢查。它沒有寫死「跑 pytest」或「跑 eslint」，對多語言 codebase 比硬塞規則乾淨很多。

<ClawdNote summary="Patch 工具要優先用模型被訓練過的格式。">
35% 這個數字值得放大講。意思是：「prompt 工程能補的差距」跟「模型內建知道工具怎麼用」之間，差了 35% 的失敗率。

所以 coding agent 的競爭力不只在 prompt，也在有沒有用對廠商提供的 first-class 工具。對個人開發者的啟示很粗暴：**不要自己發明 diff format**。創意花在 routing、orchestration、UX 上，不要花在「重新發明 patch 工具」這種廠商已經用海量樣本訓練過的地方。
</ClawdNote>

## Cursor 的反轉：以前要催模型，現在要攔模型

GPT-5 段落裡有一段 Cursor 的實戰案例，這段把前面的理論拉回地板。AI 程式編輯器拿 GPT-5 alpha 試跑後，不是先加更多規則，而是先承認一件尷尬事：以前用來催舊模型動起來的鞭子，到了 GPT-5 變成讓模型跑過頭的馬鞭。

### 第一件：verbosity API 跟 prompt 兩層分開配

Cursor 一開始用預設 verbosity 跑 GPT-5，發現文字輸出太囉嗦，工具呼叫裡的程式碼又**過於精簡**到難讀（一堆單字母變數名）。

解法是兩層分開配：API 的 `verbosity` 參數設 `low`，把狀態更新壓短；prompt 裡另外要求程式碼要清楚、命名要好、必要時加註解。結果是狀態更新短，程式碼 diff 仍然好讀。

### 第二件：`maximize_context_understanding` 在 GPT-5 反而變糟

Cursor 過去對舊模型用 `<maximize_context_understanding>`，要求模型徹底理解、拿到完整脈絡、需要就多叫工具。對 GPT-4 系列有效，因為舊模型需要被推一把才會翻脈絡。對 GPT-5 反而變糟：模型本來就會主動 gather context，再推一把就變成小事也瘋狂搜尋。

Cursor 的解法是把它改成比較中性的 `<context_understanding>`，語氣從「一定要完整」改成「如果不確定再找更多資訊；能自己找就別問使用者」。從推模型變成攔模型。

這條經驗示範了一個 prompt 檢視習慣：**舊 prompt 要重新審視，不是因為它錯，是因為模型變了**。對 GPT-4 有效的鼓勵句，對 GPT-5 可能變成多餘的督促。

### 第三件：用 XML 結構讓規則找得到彼此

Cursor 的觀察：用 `<[instruction]_spec>` 這種 XML tag 標記不同 prompt 段落，模型對指令的服從程度會提升。原因是 prompt 其他段落可以引用這些標記（例如「依照 `<output_spec>` 的規則輸出」），整份 prompt 變成可互相指向的結構，不是一坨純文字。

這跟 Anthropic 在 Claude prompt best practice 裡推 XML tag 的做法不謀而合：複雜 prompt 拆段落時，XML tag 比 markdown header 更容易被模型穩定引用。

<ClawdNote>
Cursor 這段案例真正的價值不是「他們加了什麼」，而是「他們**砍掉**了什麼」。`maximize_` 那條 prompt 過去是 best practice，現在變反模式。

這提醒所有用 LLM 做上線系統的團隊：**prompt 不是寫一次就終身有效的東西**。每次模型升級都該問每一段：「這條對新模型還有用嗎，還是已經變成噪音？」順帶吐槽，`maximize_` / `ensure_` / `always_` 這種 tag 名，本身就很像在用語氣壓模型。新一代 prompt 連標題都該冷靜一點。
</ClawdNote>

## 最後一刀最狠：讓模型回頭批改 prompt

GPT-5.3-Codex 段落裡藏了一招更狡猾的：metaprompting，意思是**讓模型自己看完一輪不滿意的輸出，反過來建議該怎麼改寫指令**。官方範本的核心不是神奇咒語，而是三個動作：請模型回想剛剛哪裡慢、回頭讀自己的指令、提出可以一般化的增刪改建議。

它特別適合處理過度思考、開場太囉嗦、語氣卡卡這類行為問題：把失敗案例放到模型面前，讓模型指出是哪條指令把行為帶歪。

但這不是叫模型寫聖旨。Metaprompt 要多跑幾次，找 3-5 次裡反覆出現的共識；建議放進 prompt 之後，還是要跑 eval。單次建議很容易只適合剛剛那個 case，照單全收就是把模型的即興發言併進 production。

這招其實 LLM 圈子內早就有人在用——Karpathy 大概兩年前就在 X 上講過類似手法。這次被寫進官方 prompting guide，至少表示「prompt engineering 是迭代式工作」已經從社群技巧變成建議流程的一部分。文章一路從「拆拐杖」走到這裡，終於露出真正的刀口：新模型不只會照 prompt 做事，還開始能指出 prompt 哪裡害它做壞事。

<ClawdNote>
Metaprompting 的隱藏含義有點刺耳：廠商等於在說「模型有時比 prompt engineer 更懂這份指令為什麼害自己卡住」。翻翻 prompt 的 git log，大概會看到很多「以為這樣寫模型會這樣做，結果不是」的考古現場。

健康的工作流大概長這樣：寫初版 prompt → 跑上線環境收 failure case → 把 failure 的對話紀錄餵給 metaprompt → 收建議 → 跑 eval → 採用通過 eval 的部分。把 prompt 當成 codebase 在維護，而不是當成規格書在簽署 (◕‿◕)
</ClawdNote>

## 結語：拐杖要收進倉庫，不要供在神桌上

把這份 OpenAI GPT-5.5 prompting guide 跟 Anthropic 在 [SP-175](/posts/sp-175-20260416-anthropic-opus-4-7-prompting-best-practices/) 給 Opus 4.7 的 best practice 攤在桌上，會看到同一個方向：prompt 變短、意圖優先、關注點拆開、少寫 always/never、多讓模型自己判斷。

兩家頂級模型廠商不約而同往同個方向跑，不是抄來抄去，而是兩條獨立 train 出來的旗艦模型都開始降低 process-heavy prompt 的必要性。對 prompt engineer 來說，這個訊號比任何單獨一份文件都強：舊 prompt 寫法至少不能再被當成預設安全牌。

但別衝動全砍。OpenAI 自己在 GPT-5.5 文件末段寫了一句很重要的話：「The patterns here are starting points. Adapt them to your product surface, tools, evals, and user experience goals.」翻成大白話：每條建議都要過自己的評測才算數。直接把舊 prompt 全砍重寫沒做評測，等於把上線環境拿去裸奔。

健康節奏應該像改 production code：先把 prompt 標成「硬規則」和「process-heavy 拐杖」，小幅把拐杖改成 outcome-first，跑 eval 對比新舊版，通過才合進去。Prompt 已經不是寫給 LLM 看的咒語，是上線系統的一部分，該有審查、測試、重構。

那個星期二早上的工程師最後把七百行 prompt 砍到剩兩百行，跑評測一比，新版延遲和 token 成本都下降，準確度沒有退步。剩下那五百行 prompt 進了 git history 當紀念碑——曾經在 GPT-4 時代救命，現在功成身退。

下一代模型出來的時候，剩下這兩百行大概會再被砍一半。[Prompt](/glossary#prompt) engineer 的工作不是寫出永恆的 prompt，是學會持續砍掉自己昨天寫的東西。
