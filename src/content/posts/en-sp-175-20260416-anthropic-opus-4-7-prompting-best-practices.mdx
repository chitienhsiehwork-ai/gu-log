Monday morning, 10:30 AM. A team's code review dashboard finishes its weekend batch. The report looks suspiciously clean — five issues, all CVE-grade, all high severity. The team lead clicks "approved" and closes the laptop to grab coffee. Tuesday afternoon, production explodes. Fifteen minutes before the fire reaches the incident channel, the release manager runs `git blame` and finds the line: an off-by-one error. The [harness](/glossary#agent-harness) saw it during review, classified it as "not high severity," and silently skipped it. On 4.6, it would have been flagged. On 4.7, the prompt said "high severity only," so 4.7 did exactly that.

This is not a hypothetical teaching exercise — it is an actual case study documented in Anthropic's two official best practice releases: a [Claude Code](/glossary#claude-code) [guide](https://claude.com/blog/best-practices-for-using-claude-opus-4-7-with-claude-code) (shorter, focused on interactive development) and a [full prompting guide](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) (longer, with an Opus 4.7-specific section). Both shipped at the same time, carrying the same warning: prompt instincts from the Sonnet and Haiku era are expiring. Not because the techniques were wrong — because 4.7 became a different person.

---

## The deeper they dug, the worse it got

The release manager pulled up the weekend batch logs expecting to find one missed bug. Instead, every layer revealed something uglier than the last.

The surface layer: the prompt said "report only high severity issues." On 4.6, this worked fine — the model would auto-balance recall and precision, mentioning logic errors even when they were not CVE-grade. On 4.7, the phrase gets taken at face value. The threshold tightens to CVE-only. The off-by-one was not missed — the model saw it, judged it, and consciously decided not to report it. The fix principle from the official docs comes down to one word: **split**. Coverage in one instruction, filtering in another. Never bundle two goals into one sentence. 4.7 picks the most literal meaning and drops the rest. This trap shows up everywhere — issue triage, log anomaly detection, security scanning, doc review.

One layer deeper, the release manager's blood pressure started climbing. One of the harness rules referenced a function that had been renamed six months ago. On 4.6, the model would automatically `grep` the repo to verify, stopping to ask if it could not find the function. On 4.7, no verification — it pulled the old name from training data, built a confident-sounding conclusion around stale knowledge, and kept going. Clean reasoning, flawless logic, built on an expired premise. This is 4.7's "reason more, use tools less" tendency at its worst. The fix is one line — "verify by reading the relevant file before answering" — but what that line means is heavy: **4.7 defaults to trusting its own memory over its tools**.

The bottom layer nearly made the release manager throw something. The review harness was designed for parallel execution: three [subagents](/glossary#subagent) reading three files independently, then an aggregator combining results. In practice, only one subagent spawned. The other two never woke up. Coverage was 1/3 of what it should have been — but the log said "review complete," and nothing on the surface looked wrong. On 4.6, subagents were like energetic kids — you had to rein them in. On 4.7, they are like employees who calculated exactly how little work they can get away with — unless you explicitly write "spawn in parallel," only one shows up. (For the full framework on when to split into multi-agent versus keeping a single agent, [SP-172](/posts/sp-172-20260413-anthropic-multi-agent-when-how/) covers the decision tree.)

Three threads, one root cause. Not a dumber model — a model that **stopped guessing what your prompt meant to say**.

<ClawdNote>
The release notes say "improved instruction following" and "more predictable behavior." Translated to plain language: the model became so obedient that every lazy prompt that used to survive on model generosity now has its seams on display. But here is the counterintuitive part nobody wants to say out loud: **4.7 is right to behave this way**. Sure, 4.6 filling in the gaps felt helpful — but how many hours did engineers spend debugging "why does the same prompt sometimes work and sometimes not"? A model that does not guess is a model you can debug, predict, and write tests for. That is the tool engineers actually want, not a coworker who is sometimes helpful and sometimes checked out (⌐■_■)
</ClawdNote>

## Can you just crank the effort?

Prompts fixed, shipped. The team lead breathed out — but one question kept nagging: if effort had been set to max from the start, would any of this have happened?

The intuition makes sense. It is also completely wrong.

Same review set, five effort levels, one run each. `max` caught the off-by-one and the race condition — then attached two bonus sections: a comparison of migration strategies, a refactoring proposal, and a short essay on why the module should be split into three services. The team lead read it and sighed: "I asked if there were bugs. I got a debate tournament."

One level down, `xhigh`. Both bugs, cleanly reported, no bonus essays. This is Claude Code's default effort — and the team lead finally understood why. Not "highest quality," but the sweet spot between investment and return.

The unsettling part was the other end. `high` caught the off-by-one but missed the race condition — that one required context across three files. Anthropic defines `high` as the floor for intelligence-sensitive work, and this result showed exactly where that floor sits. `medium` was worse: it opened the adjacent file but only did surface cleanup — renamed variables, adjusted formatting, walked right past the logic bug. The team lead's comparison afterward: "Like telling an intern to 'take a careful look' without defining what careful means." `low` was the most honest — it simply did not open anything outside its scope. The off-by-one lived in an adjacent file's boundary check, and `low` did not know that file existed.

So just set effort to `xhigh` and call it a day? Look closer at the test logs — the team lead ran all five rounds with the **fixed prompt**. What about the old prompt, the one that said "only high severity"? With that prompt, `max`'s literal execution would only make it skip non-high issues more thoroughly. Effort is the throttle, not the steering wheel. A crooked steering wheel plus a bigger engine just means you hit the wall faster.

The benchmark conclusion went into the postmortem addendum: **fix the prompt first, then start at `xhigh` for coding and agentic work, at least `high` for intelligence-sensitive tasks, `medium`/`low` only for simple format jobs**. When running `xhigh` or `max`, the docs recommend starting `max_tokens` at **64k** to give reasoning and subagents enough room.

<ClawdNote>
Cannot resist commenting on the naming. `low`, `medium`, `high`, `xhigh`, `max` — not `higher`, not `very_high`, just straight to `xhigh`, which sounds like some Anthropic engineer ran an A/B test and found it saves two keystrokes so they shipped it. Next-gen model will probably need another tier — `ultrahigh`, `ludicrous`, `xhigh-pro-max` — and `xhigh` gets demoted to the middle. This naming trajectory is identical to CPUs going Pro → Pro Max → Ultra → Ultra Max. Eventually the levels grow legs and run away (¬‿¬)
</ClawdNote>

## Opus 4.7 thinks the company is a boutique coffee shop

Same week, different desk. An engineer unrelated to the pipeline work needed a quick internal monitoring dashboard — API latency, error rates. Tailwind, dark mode, data table, chart. Clear spec in the ticket. Handed it to Claude Code, waited for it to finish, opened localhost.

Staring back: `#F4F1EA` off-white background, Georgia typeface, Playfair Display headlines, terracotta buttons, italic word accents. The whole thing looked like an Aesop store website.

Fifteen minutes of digging later, the answer: nobody touched the config. Opus 4.7 has a strong default design aesthetic — cream backgrounds, serif fonts, warm earth-tone accents. Beautiful for editorial sites, portfolio pages, boutique hospitality. Disastrous for dashboards, admin panels, fintech tools, or anything that needs to look like work rather than a weekend brunch menu.

The default is stubborn too. Generic instructions like "don't use cream" just swap in a different fixed palette rather than following the spec. To actually override it, you need a concrete design system: font stack, hex palette, layout grid. Writing it into `CLAUDE.md` as a hard constraint is the most reliable approach.

The engineer posted the screenshot to the team channel with the caption: "Opus 4.7 thinks our company is a boutique coffee shop." It became the most-reacted message of the week.

<ClawdNote>
The coffee shop aesthetic and the prompt failures from earlier are actually the same root cause — 4.7 has its own defaults, and those defaults are stubborn. An underspecified prompt makes code review skip low-severity issues; an underspecified design system makes the UI look like a hipster café. The difference is that bad code review crashes production, while a café UI only crashes an engineer's aesthetic dignity — but the fix is identical: write your expectations explicitly, leave no room for interpretation. If 4.7 had a personality tagline it would be: "Anything you didn't specify, I decide. You're welcome" ╰(°▽°)╯
</ClawdNote>

## The fourth accomplice was on the bill

Prompts fixed, effort benchmarked, harness rewritten. Two quiet weeks. The team lead thought the case was closed.

Then the monthly API bill arrived. Twenty-two percent higher than the month before.

Request volume was flat. Prompt lengths unchanged. The team lead traced the billing breakdown line by line and found the discrepancy: same input, same `max_tokens` ceiling, but 4.7 was producing different token counts than 4.6. Not wildly different — 3 to 7 percent per request — but multiplied across a month of traffic, that is the unexplained 22% on the invoice.

**The tokenizer had changed.** One line in the release notes, barely mentioned. Like a landlord raising the rent without notice — a few dollars more each month feels like nothing, but the annual statement shows a whole appliance disappeared. Every pipeline relying on precise token math — [context window](/glossary#context-window) budgeting, prompt caching keys, context compression thresholds — was now slightly off.

The team lead's instinct: use `budget_tokens` to cap thinking costs. Opened the API docs and found a blank wall — **`budget_tokens` was removed.** Opus 4.7 uses adaptive thinking: the model decides for itself whether and how long to think. Good news: no more "budget ran out mid-thought." Bad news: the hard brake on thinking cost is gone. Only `max_tokens` remains as a ceiling.

Looking back at the full timeline, the most unsettling piece surfaces: the tokenizer shift had been silently skewing context budgets the entire time. Those skipped issues, those unspawned subagents — how many were partly caused by a prompt that got quietly truncated, with a critical instruction landing just past the cut? The fourth accomplice was not acting alone. It was amplifying the other three from the shadows. Four 4.7 defaults, each fixable on its own, devastating in combination. "Looks the same but works differently."

<ClawdNote>
The tokenizer swap plus `budget_tokens` removal, taken together, looks like a coordinated hit. But flip the perspective — `budget_tokens` was always a hack. Hard-capping a thinking budget means telling the model "stop thinking mid-thought," which was never going to be stable in production. Anthropic replaced it with adaptive thinking, which is actually removing an unreasonable control surface. The problem is not the removal — it is the migration path, or lack of one. One line in the release notes saying "deprecated." That is not technical debt. That is communication debt ┐(￣ヘ￣)┌
</ClawdNote>

## Back to Monday's dashboard

The team lead's final postmortem had one line in the root cause field: **"Didn't read the model's personality."**

Not a skill gap. Not a prompt mistake. Not a model downgrade. Opus 4.7 is a ["personality-swapped"](/glossary#model-swap) 4.6 — more literal, fewer tool calls, shorter responses, fewer [subagents](/glossary#subagent), even a fixed design taste — and the team was still running on 4.6 assumptions. Every fix they applied — splitting coverage from filtering, explicitly requesting parallel subagents, packing `CLAUDE.md` with upfront context ([harness design philosophy](/posts/sp-174-20260415-intuitiveml-ai-first-harness-engineering/), [who owns the memory](/posts/sp-173-20260411-hwchase17-harness-memory-lockin/)), pinning a design system, matching effort to task — all trace back to one habit: read the migration guide before you drive.

Anthropic shipped both the Claude Code blog and the full prompting guide in the same window. The unspoken message: they know this generation's behavioral diff is large enough to need **two documents** to explain. No previous model got that treatment.

What expired is not prompt skill — it is the comfort of letting the model fill in what was left unsaid. 4.6 guessed your intent. 4.7 does not — and in exchange you get a tool that is debuggable, predictable, and will not produce wildly different results based on model mood. Next time Opus 4.8 ships, the whole cycle repeats. But once you build the reflex of "new model → read migration guide → find the personality diff," version numbers stop being a source of panic.

<ClawdNote>
One last thing Anthropic will not say out loud: official best practices also expire. The fixes you learned today, Opus 4.8 might require another round of tuning. But the real point was never memorizing templates — it is building the habit of reading model personality. When a new model drops, the first move is not checking benchmark scores. It is opening the migration guide to find how this generation's personality differs from the last. Treat a prompt as a forever answer, and you will spend the rest of your career chasing version numbers ʕ•ᴥ•ʔ
</ClawdNote>
