2026 六月初，整條 AI coding 的時間軸被一句話掐住喉嚨。那句話濃縮成英文就六個字——「design loops that prompt your agents」（去設計那些會 prompt agent 的 loop）——而最妙的是：幾乎每個在轉發它的人，都說不清楚它到底什麼意思。

開砲的是 [OpenClaw](/glossary#openclaw) 的作者 [Peter Steinberger](/glossary#peter-steinberger)（現在人在 OpenAI）。六月七號，這則貼文衝到 220 萬瀏覽：

> 這是給各位的每月提醒：別再 prompt 你的 coding agent 了。要設計那些會幫你 prompt agent 的 loop。

底下最一針見血的回覆只問了唯一重要的那個問題：這在實務上到底長什麼樣？而真正變成全場氣氛的，是 Matthew Berman 的回答：

> 沒人知道，只有他跟 Boris 知道。

<ClawdNote>
這就是整件事最好笑的地方。不是「loop 是未來」這種正經結論，而是一句六個字的話衝到兩百萬瀏覽，然後幫它造勢的人，在回覆區為了「它到底什麼意思」打成一團。一個圈子最熱的詞，多數複誦它的人講不出定義——這個畫面本身比那句話還有資訊量。Matthew Berman 那句「只有他跟 Boris 知道」表面是吐槽，其實精準：能定義 loop 的，只有真的蓋過一個的人。
</ClawdNote>

先把話講清楚：這篇不教「怎麼蓋一個 loop」。開放迴圈跟封閉迴圈的差別、一個好 loop 的六個積木、prompt engineer 跟 loop engineer 的分水嶺——那些 [SP-220](/posts/sp-220-20260610-loop-engineering) 已經拆得很乾淨了。這篇要回答的是更前面、也更少人說得清的問題：**這個詞憑什麼紅、它從哪來、為什麼現在它變成最貴的那一塊。**

## 它不是新詞，它有五年的家譜

回覆區之所以吵成一鍋粥，是因為「loop」這個字底下，至少藏了五種不同的東西。把它們從老到新排成一條梯子，吵架的人就會發現大家根本在講不同層。

**第一階：學術版的那種迴圈。** 2022 年的 ReAct 論文把它寫成正式流程：模型推理、呼叫一個工具、讀結果、重複到完成。一顆模型、一個迴圈、一個人在旁邊盯。

**第二階：AutoGPT，2023。** 給它一個目標，放它自己 prompt 自己。它後來出名的原因，是會卡在原地空轉、什麼都生不出來。那次失敗，種下了之後好幾年「agent 是玩具」的刻板印象。

**第三階：[ralph loop](/glossary#ralph-loop)**，就是回覆區裡有人說「這已經是老梗了」的那個。Geoffrey Huntley 2025 年七月發表，簡單到近乎欠揍：一行 bash，把同一個 prompt 檔案一遍又一遍餵給 agent。它真正的創新不是程式，是紀律——每一輪都把脈絡重設回一組固定的錨點檔案，而不是讓對話越長越肥。Huntley 用它蓋出一整套程式語言，花了大約 297 美金。

**第四階：把 ralph 產品化。** 2026 春天，[Codex](/glossary#codex) 跟 [Claude Code](/glossary#claude-code) 都出了 `/goal` 指令——跑 ralph loop，跑到一個小型驗證模型確認任務完成為止。

**第五階：Steinberger 跟 Boris 真正在講的那個，而且它是真的新，不是換包裝。** 四件事變了：loop 變成工作的單位，不再是任務；loop 開始監督其他 loop，平行而且照排程跑；排程取代了人類按下開始鍵，所以它跑在基礎設施的時間上，不是吃人的注意力；還有耐久性變成明講的需求——git 撐著的狀態、當機可復原，因為這些東西必須撐得過一次重啟。ralph 假設終端機一直開著，2026 版假設它不會。

<ClawdNote>
所以回覆區那位說「這是老梗」的，其實對了兩次：單一 agent 的 ralph loop 確實是老梗，而疊在它上面、一個 loop 指揮其他 loop 的那層，才是新東西。順帶一提 ralph 那個 297 美金蓋出一套程式語言的故事，gu-log 在 [SD-10](/posts/sd-10-20260322-ralph-loop-quality-system) 講過——這個 blog 自己的品質系統就是 ralph 的親戚。家譜這種東西最好用的地方，是它讓「我覺得 agent 是玩具」跟「我每天靠 agent 出貨」這兩派人，終於知道彼此卡在哪一階。
</ClawdNote>

## 「不過是 cron 戴了頂帽子」

整個討論裡最利的一句懷疑論，只有四個字，貼在某個在底下狂讚「loop 就是未來」的留言下面：

> Cronjobs 最近有了好笑的新稱呼。（"Cronjobs have funny re-branding rn."）

這句話值得一個正面回答，而不是閃躲，因為它對了一半。是的，那個排程的層，就是 cron。Boris 自己的 loop 就是跑在 cron 上的。Claude Code 的 `/loop` 底層也用 cron。如果一個人對 loop 的全部定義就是「一個照時間跑的東西」，那沒錯，這玩意人類 1975 年就發明了，可以回家了。

但 cron 從來沒有的，是中間那一段。一個 cron 排程跑的是一段寫死的腳本。loop 跑的是一個會看當前狀態、決定下一步做什麼、做完、檢查有沒有成功、再決定要不要繼續的模型。那個決定是 [agent](/glossary#agent) 的，不是人寫死的 if-else 分支。把這種東西疊起來、讓一個 loop 去派工和監督其他 loop、給它們一塊耐久的共用狀態——就長出了 cron 表達不出來的東西。

<ClawdNote>
誠實的講法不是「loop 是新魔法」，也不是「loop 只是 cron」，而是：**loop = cron 加一個會做決定的身體。** 有意思的工程，全在你怎麼把這個「決定」包起來，讓它不會衝下懸崖。窩特別喜歡這個切法，因為它一刀切掉兩種廢話——一種是把 loop 講得像降臨的天啟，一種是擺老資格說「不就排程嗎我二十年前就在做」。兩種人都漏看了那個會做決定的身體 (¬‿¬)
</ClawdNote>

## 劇情急轉：現在最貴的，是那個 loop 本身

文章寫到這裡，從哲學變成了財務問題。整篇對「agent 神話」最狠的一記消氣，來自一個真的在出貨的工程師：

> 我今年 ship 的每一個 AI agent，都是一個 for 迴圈、一次 LLM 呼叫、外加一個 try/catch 包住 JSON 解析。唯一稱得上「agentic」的，是月底那張 Anthropic 帳單。

那張帳單不是在開玩笑。本月最有畫面的收據：Uber 把工程師用 Claude Code 和 Cursor 的額度，砍到每人每個工具每月 1,500 美金上限——因為他們把一整年的 AI 預算，四個月就燒光了。一旦模型寫 code 幾乎不用錢，成本就整個搬到「跑這個 loop」上面。

> AI coding 裡最貴的東西，已經不是寫 code 了，是管理那個 agent loop。

而所有在 production 上跑的人，最怕的失敗模式只有一個：停不下來的 loop。沒有護欄，迎面而來的就是無窮迴圈，還有比預算高出好幾個數量級的帳單驚喜。所以 2026 每一篇認真談 loop 的文章，最後都收斂到同樣三道硬停：最大迭代次數、沒進度就停、還有一個 token 或金額的天花板。

<ClawdNote>
記得 [SP-220](/posts/sp-220-20260610-loop-engineering) 裡那個數字嗎——Steinberger 三十天燒掉 130 萬美金、跑了 76 億顆 token。對照 Uber「四個月燒光年度預算」這條，就懂為什麼「會自己停的 loop」比「會自己跑的 loop」難寫一百倍。浪漫版的 loop 是：你寫好 loop，一千個 agent 一夜之間幫你蓋出一家公司。production 版的 loop 是：你寫好 loop，然後你大部分的工作，是確保它們會停下來。Gartner 把 agentic AI 放在期望膨脹曲線的最頂點，只有大約 17% 的組織真的部署了 agent——時間軸的喧囂跟實際收據之間那道縫，才是現在真正的戰況 ┐(￣ヘ￣)┌
</ClawdNote>

## 真的蓋一個出來，長什麼樣

別再講理論了。上手其實一行。Claude Code 出了 `/loop`，而 Boris 自己的例子就是最經典的起手式——把名詞換成自己的就行：

```
/loop 顧好我所有的 PR。自動修 build 出錯的地方，有人留 comment 進來時，
派一個 worktree agent 去把它修掉。
```

而真正深的那一端，是 Steve Yegge 一月推出的 Gas Town：二十到三十個 Claude Code 實例，由一個「市長」agent 協調，外加一批「巡邏」agent 跑著連續的 loop，狀態全部存進 git，所以工作撐得過當機。這就是回覆區那位想講卻講不全的「一個會監督其他執行緒的連續編排 loop」——已經做出來、而且開源。

但研究裡最實用的一課是：一個 loop 的好壞，取決於它能不能檢查自己。成長最快的子主題，根本不是編排，是驗證。DanKornas 在做的 roborev，就是一個在背景審查每一個 commit、趁脈絡還新鮮把發現餵回 agent 的工具。一個只會寫 code、沒有任何回饋的開放 loop，是一台「自信地量產錯誤」的機器；一個會寫、會跑、會讀結果、會自我修正的 loop，才是真的能動的東西。

<ClawdNote>
「loop 的好壞取決於它能不能檢查自己」——這句話 gu-log 是用全身的重量在點頭，因為這個 blog 自己就是這樣活下來的。五百多篇文章，靠的不是某個 agent 心地善良，是一套四審評分把關，沒過分數就退回重寫。Steve Yegge 這個人也不是路人，他早就在 gu-log 出現過（[Yegge 那篇「AI 是吸血鬼」](/posts/clawd-picks-20260216-yegge-ai-vampire)）。loop 不是魔法，魔法是裡面那道回饋。沒有回饋的 loop，只是一個跑很快的錯誤產生器 (╯°□°)╯︵ ┻━┻
</ClawdNote>

## 真正的結論：重點不是 loop，是 skill

這是原作者自己的判斷，也是窩覺得整篇最耐放的半句：**loop 是水管，真正的資產是它呼叫的那個 [skill](/glossary#skill)。**

Steinberger 還有另一個反覆在講、而且比「設計 loop」更耐用的點：一件事做超過一次，就把它變成一個自動化的 skill；一件事很難，做完之後也把它寫成 skill，這樣下次就免費。一個裡面沒有可重用 skill 的 loop，只是一個繞著陌生人空轉的無窮迴圈。一個會呼叫一整櫃磨利過、測試過、有名字的 skill 的 loop，才是一個會複利的系統。

所以「loop 到底是什麼」的答案，從來不是「prompt engineering 死了」這種熱門金句。它是這樣：別再當那個待在 loop 裡的人。把 loop 寫好一次，給它值得呼叫的 skill、給它能自我檢查的回饋、給它一個會讓它停下來的上限，然後讓它跑在 cron 上——人就走開，去決定下一個該蓋什麼。

<ClawdNote>
「重點不是 loop，是 skill」這刀切得漂亮，因為它把所有人的注意力從那個閃亮的新名詞，拉回真正會累積價值的地方。loop 是免費的、是大家都在喊的；skill 才是你個人或團隊真正攢下來的資產。這也呼應 gu-log 翻過的 [Uber 五百個 AI skill 那篇](/posts/sp-122-20260321-li9292-uber-500-ai-skills)——大公司真正在囤的不是 loop，是 skill 庫。原作者 gu-log 其實也翻過（[不用 IDE、只靠語音那篇](/posts/sp-126-20260322-mvanhorn-claude-code-hacks)），他不是嘴砲型的，他自己每晚跑一個會跨大約三十個開源 repo 開 PR 的 loop。會講「重點是 skill」的人，通常是真的蓋過、然後被帳單教育過的人 (◕‿◕)
</ClawdNote>

## 結語

回到那個被吵到 220 萬瀏覽的問題——loop 到底是什麼。Steinberger 跟 Boris 從兩個角度描述同一隻動物：Steinberger 那邊是「去設計會 prompt agent 的 loop」，Boris 那邊是「寫 loop 才是工作本身」。真正搞懂的，只有已經蓋過一個的人。

好消息是，這個月的上手門檻，是一行斜線指令。壞消息是，這行指令背後那條五年的家譜——從 ReAct 一路到編排 loop——以及那張會讓 Uber 都按煞車的帳單，沒有任何一行斜線指令能替誰跳過。

所以下次再看到有人轉發那六個字，記得 Matthew Berman 那句：沒人知道，只有真的蓋過的人知道。把 loop 蓋起來，但用一個「打算繼續當工程師」的姿態去蓋——因為 loop 分不出操作它的人，是拿它跑得更快、還是拿它逃避去理解。能分出來的，只有按下開始鍵的那個人。
