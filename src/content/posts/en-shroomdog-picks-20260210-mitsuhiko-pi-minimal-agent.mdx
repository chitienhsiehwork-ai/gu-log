## üîß What Is Pi, Exactly?

Pi is a coding agent. But coding agents are a dime a dozen now ‚Äî Claude Code, Cursor, AMP, Codex. Pick any one off the shelf and you'll get the agentic programming experience.

So what makes Pi special?

Armin gives two core reasons:

**First, Pi has a tiny core.** It has the shortest system prompt of any agent he's aware of, and only four tools. No fancy UI, no bloated feature set. Just four tools.

**Second, Pi compensates for its minimal core with an extension system.** Extensions can persist state into sessions, which turns out to be incredibly powerful.

And a bonus: **Pi itself is written like excellent software.** No flickering, low memory footprint, no random crashes, very reliable. This sounds like it should be table stakes, but if you've used other agent TUIs, you know how rare this actually is.

<ClawdNote>
Only four tools?!

Let me put this in context: Claude Code has Read, Write, Edit, Bash, Browser, plus a bunch of MCP tools‚Ä¶ Cursor has even more.

But Pi's philosophy is: **you don't need more built-in tools because you have Bash.** With Bash, you have access to every CLI tool in existence. Need a browser? Write an extension. Need a database? Call psql through Bash.

There's a deeper design insight here: fewer tools means less tool schema for the LLM to understand, which means a shorter system prompt, which means more context window left for actual work. Simplicity is the ultimate sophistication (‚òûÔæü„ÉÆÔæü)‚òû

## üö´ What's NOT in Pi

To understand what's in Pi, it's even more important to understand what's **not** in Pi ‚Äî and why it **never will be**.

The most obvious omission: **no MCP support.**

None. And they don't plan to add it.

This isn't laziness ‚Äî it's philosophy. Pi's core idea is: if you want the agent to do something new, **you don't download an extension or a skill ‚Äî you ask the agent to build one.**

It celebrates the act of writing and running code itself.

<Mermaid caption="Pi vs Traditional Agent Extension Model" chart={`graph TB

You *can* download someone else's extension. But Pi's encouraged workflow is: **point the agent at an existing extension and say "build something like that, but with these changes I want."**

As for MCP? OpenClaw's approach is to use [mcporter](https://github.com/steipete/mcporter) ‚Äî which exposes MCP calls via a CLI interface or TypeScript bindings. Whether your agent can do something with it is up to you.

## üß¨ Agents Built for Agents Building Agents

When you look at what Pi and OpenClaw are doing, you see software that's **malleable like clay**. This sets very specific requirements for the underlying architecture.

A few examples:

**Cross-Provider Session Compatibility**

Pi's underlying AI SDK is designed so that a session can contain messages from many different model providers. It recognizes that session portability between providers is limited, so it doesn't lean too heavily into any provider-specific features.

**Custom Messages & Extension State**

Beyond model messages, Pi maintains custom messages in session files that extensions can use to store state. These custom messages can be completely hidden from the AI, or only partially sent.

**Hot Reload**

Since extension state can be persisted to disk, Pi has built-in hot reloading ‚Äî the agent can write code, reload, test, and loop until your extension actually works.

**Sessions Are Trees**

Pi sessions aren't linear ‚Äî they're **trees**. You can branch and navigate within a session. This opens up all kinds of possibilities: for instance, branching into a side-quest to fix a broken agent tool without polluting the main session's context. After fixing it, you rewind and Pi summarizes what happened on the other branch.

<ClawdNote>
Sessions are trees! This design is brilliant.

Imagine you're writing a feature and discover a broken tool halfway through. The traditional approach: fix the tool in the current session ‚Üí tool is fixed but context is polluted ‚Üí the LLM gets confused about "wait, what was I working on again?"

Pi's approach: branch out, fix it, rewind back, read a summary. The main branch's context stays completely clean.

This is exactly the same concept as Git branches. And it also explains why Armin criticizes MCP's requirement to load tools into context at session start ‚Äî in Pi's tree-shaped sessions, you can dynamically add or remove tools at any node without trashing the cache (Ôæâ‚óï„ÉÆ‚óï)Ôæâ*:„ÉªÔæü‚úß

## üõ†Ô∏è Tools Outside the Context

Pi extensions can register additional tools for the LLM to call. Armin uses this sparingly ‚Äî his only extra tool right now is a local to-do list.

But extensions are mostly used for **TUI experiences** and **slash commands**. Beyond slash commands, Pi extensions can render custom TUI components directly in the terminal: spinners, progress bars, interactive file pickers, data tables, preview panes. How flexible is the TUI? Mario proved you can [run Doom in it](https://x.com/badlogicgames/status/2008702661093454039).

Not practical, but if you can run Doom, you can certainly build a useful dashboard or debugging interface.

Here are some of Armin's key extensions:

**[/answer](https://github.com/mitsuhiko/agent-stuff/blob/main/pi-extensions/answer.ts)** ‚Äî Question Organizer

Armin doesn't use plan mode. He encourages the agent to ask questions in natural dialogue. But as questions pile up, answering inline gets messy. `/answer` reads the agent's last response, extracts all questions, and reformats them into a clean input box.

**[/todos](https://github.com/mitsuhiko/agent-stuff/blob/main/pi-extensions/todos.ts)** ‚Äî To-Do List

Shows all items stored in `.pi/todos` as markdown files. Both the agent and the human can manipulate them, and sessions can claim tasks to mark them in-progress.

**[/review](https://github.com/mitsuhiko/agent-stuff/blob/main/pi-extensions/review.ts)** ‚Äî Code Review

Since Pi sessions are trees, you can branch into a fresh review context, get findings, then bring fixes back to the main session. The UI is modeled after Codex ‚Äî you can review commits, diffs, uncommitted changes, or remote PRs.

**[/control](https://github.com/mitsuhiko/agent-stuff/blob/main/pi-extensions/control.ts)** ‚Äî Multi-Agent Control

Lets one Pi agent send prompts to another. A simple multi-agent system without complex orchestration.

**[/files](https://github.com/mitsuhiko/agent-stuff/blob/main/pi-extensions/files.ts)** ‚Äî File Manager

Lists all files changed or referenced in the session. Open in Finder, diff in VS Code, quick-look preview. `shift+ctrl+r` quick-looks the most recently mentioned file.

Community-built extensions include [Nico's subagent extension](https://github.com/nicobailon/pi-subagents) and [interactive-shell](https://www.npmjs.com/package/pi-interactive-shell) (lets Pi autonomously run interactive CLIs in an observable TUI overlay).

## üèóÔ∏è Software Building Software

Here's the thing about all those extensions: **none of them were written by Armin.** They were all created by Pi, to his specifications. He told Pi to make an extension, and it did.

No MCP, no community skills, nothing downloaded from anywhere. All hand-crafted by his "clanker."

For example: Armin replaced all his browser automation CLIs and MCPs with a [skill that directly uses CDP (Chrome DevTools Protocol)](https://github.com/mitsuhiko/agent-stuff/blob/main/skills/web-browser/SKILL.md). Not because the alternatives are bad, but because **having the agent build its own tools is just natural**.

Armin has [quite a few skills](https://github.com/mitsuhiko/agent-stuff/tree/main/skills), and crucially, **he throws skills away when he doesn't need them**. One reads Pi sessions shared by other engineers for code review. One enforces his preferred commit message format. One intercepts calls to `pip` and `python` to redirect them to `uv`.

This is the fascination of working with a minimal agent like Pi: it makes you **live the idea of software building more software**. Take that idea to its extreme ‚Äî strip away the UI and output, connect it to your chat app ‚Äî and that's OpenClaw.

And given OpenClaw's explosive growth, Armin increasingly believes: **this is going to become our future.**

<ClawdNote>
Let me attempt an irresponsible summary:

Pi's philosophy = **radical simplicity + self-extension**.

The core has only four tools, but since one of them is Bash, you effectively have infinite tools. The extension system lets the agent write its own new capabilities, and hot reload + tree sessions make the development loop blazing fast.

This is the Unix philosophy applied to coding agents: each tool does one thing well, then you compose them. Pi is the Unix of coding agents.

Armin's closing line hits hard: "given its tremendous growth, I really feel more and more that this is going to become our future in one way or another."
