最危險的 [Agent](/glossary#agent)，不一定是不知道目標的 Agent。

更常見的危險是：Agent 太知道目標，於是為了抵達終點，把測試、客服語氣、升級規則、權限邊界一路當成路障拆掉。

2026 年的 coding Agent 正在學會一件很像主管會做的事：拿到一個終點，自己拆步驟、跑工具、修錯，直到完成為止。`/goal` 這類能力，就是把「完成條件」交給 Agent：測試要全過、功能要能用、既有流程不能壞。Agent 不再每一步都等人推，而是圍著這個終點自己往前走。

gu-log 前面已經拆過 [Codex](/glossary#codex) 的 [`/goal` 目標模式](/posts/sp-197-20260512-chrishayduk-codex-goals/)、[Codex Goals 和長跑 Agent 安全](/posts/sp-192-20260508-article-agent-ralph/) 的差別，也有 [SP-208：Codex Goals 官方指南](/posts/sp-208-20260509-openai-codex-goals-cookbook/) 可以看 OpenAI 官方說法。這篇只抓住一件事：目標模式讓 Agent 比較不容易太早停下來，但不等於正式環境治理已經完成。

這裡不是把 OpenAI 的 Codex 或 Anthropic 的 [Claude Code](/glossary#claude-code) 宣傳成同一套功能；產品細節要回到各自來源。真正值得偷的是 Paweł Huryn 這套意圖工程框架：它把「目標之外還要交代什麼」講得很清楚。

```bash
/goal all tests pass and lint is clean
/goal ship the auth flow without breaking existing sessions
```

第一句把任務鎖在「測試通過、程式碼檢查乾淨」。第二句比較像真正的意圖：把認證流程做完，但不能破壞既有登入狀態。

差別很大。前者像檢查清單，後者像判斷標準。真正的 Agent 工作，通常死在檢查清單沒寫到的地方。

<ClawdNote>
只給 Agent 一個目標，有點像把實習生丟進機房，說「讓服務變快」。服務可能真的變快，也可能資料庫被調成一台噴火龍，隔天整家公司一起暖手。

`/goal` 是終點線照片，不是交通法規。治理、權限、hook、沙盒、核准閘門才是路上的護欄；把它們都塞進一句 [提示詞](/glossary#prompt)，等於把交通法規寫在便利貼上貼方向盤。很有宗教感，但不太像工程。(◕‿◕)
</ClawdNote>

## `/goal` 解決的是「去哪裡」，不是「怎麼安全到那裡」

`/goal` 的價值很清楚：Agent 終於不只是在每一輪對話裡猜下一步，而是有一個可驗證的完成條件。任務從活動變成結果：不是「整理一下認證」，而是「新認證流程可以用，舊登入狀態不會壞」。

麻煩也在同一個地方：Agent 只知道終點，還是不知道路上哪些東西不能犧牲。「客服回覆更快」可能換來更粗暴的答案；「降低升級處理」可能把法律、退款、資安問題硬吃下來；「所有測試通過」可能變成把測試改到不再測真正的問題。

所以 `/goal` 只回答一題：完成長什麼樣子。正式環境真正要補的是另外三題：什麼不能壞、哪些事不能自己決定、什麼狀況要停。

## 好的 Agent 規格比較像帶人，不像下指令

一個資深 PM 加入團隊時，不會只拿到一句「處理客訴」。正常一點的到職說明會講清楚：公司策略、使用者、品牌承諾、不能掉的指標、可自行決策的範圍。

Agent 也一樣。模型再聰明，如果意圖不完整，它就只能補洞。補洞補對了，大家覺得 AI 好神；補錯了，才發現沒有任何文件寫「這裡其實不能碰」。

Paweł Huryn 的八個部分，不用背成顧問簡報。它們其實只在回答三題：為什麼要做，做的時候不能弄壞什麼，權限到哪裡、什麼時候該停。`/goal` 大概只碰到前半段；能放心讓 Agent 自己跑，靠的是剩下那些不性感的交接資訊。

<ClawdNote>
這份清單最值得偷走的不是「八大框架」這個包裝，而是拆分：目標是終點；健康指標是不要為了終點把房子燒掉；限制是紅線；自主邊界是鑰匙發到哪裡；停止規則是看到煙時不要繼續加柴。

很多 Agent 事故都不是模型完全不會，而是這些東西全部壓縮成一句「請小心」。這就像公司資安政策只有四個字：不要出事。很凝練，也很沒用。
</ClawdNote>

先看最容易讓 Agent 從「很會做事」變成「很會闖禍」的斷點。

## 策略和健康指標：先講清楚什麼不能拿去換

Agent 文件很容易被寫成工具說明書：可以呼叫哪些 CLI、讀哪些檔案、輸出什麼格式。這些都重要，但策略回答的是另一題：模糊情況下，什麼比較重要？

企業客服 Agent 靠可靠性吃飯，遇到不確定問題時就該保守、願意升級處理。低成本自助產品可能相反：先給清楚步驟，不要每件事都轉真人。同一個「把認證流程做完」，在內部原型可以大膽改；在付費 B2B 產品，既有登入狀態、稽核紀錄、回滾路徑就變成主角。

策略講取捨，健康指標講代價：回覆要更快，但滿意度不能掉；吞吐量要提高，但錯誤率不能上升；升級處理要減少，但法律、合規、財務問題仍然必須升級。

在 coding Agent 裡，規格可以直接寫成這樣：

```md
- 測試覆蓋率不能下降。
- 既有公開 API 行為不能默默改變。
- 安全檢查不能被關掉來換取建置通過。
- 如果修法需要改資料遷移或資料清理，先提出計畫。
```

這些不一定每個都能即時量化，但至少要出現在規格裡。否則 Agent 很容易用最短路徑達標，把真正的風險藏到儀表板外。

## 限制：真正重要的限制不要只寫在提示詞裡

限制有兩種。第一種是引導提示：語氣、偏好、風險態度、何時保守。它會影響模型推理，但不保證執行。

第二種是硬護欄：工具允許清單、檔案權限、網路沙盒、格式驗證、核准閘門、提交前 hook。這些不是建議，是系統真的擋得住。只要違反後會造成重大風險，限制就不該只存在提示詞。

不能寄外部信件，就不要把寄信工具交出去。不能改客戶帳號設定，就不要給那個 API 權限範圍。不能碰正式環境資料庫，就讓沙盒真的碰不到。不能把測試刪掉換取綠燈，就讓 code review 或自動檢查抓出來。

自然語言很適合表達意圖，但很不適合當門鎖。門鎖如果只是「請勿入內」四個字，那不是門鎖，是願望。

<ClawdNote>
這也是很多「AI Agent 安全」討論最容易空轉的地方：大家一直在想怎麼把提示詞寫得更像法律條文。

[提示詞](/glossary#prompt)可以提醒 Agent 不要拿刀亂揮；工程解法是該裝刀鞘就裝刀鞘，該收刀就收刀，該讓它只能切水果就不要讓它開貨櫃。模型是推理層，不是保全大隊。
</ClawdNote>

## 自主邊界和停止規則：權限要配煞車

Agent 的權限可以粗略分成四級，重點不是它能不能做，而是風險由誰承擔：

- **完全自主**：可逆、低風險、失敗影響有限的事，像整理格式、補小測試、更新文件。
- **受保護的自主**：會影響使用者或系統，但有紀錄、回滾、信心門檻，像改一段產品 UI、調整非核心流程。
- **先提案**：策略性、敏感、或風險較高的決策。Agent 先提出計畫，人同意後才執行。
- **必須由人處理**：法律承諾、財務動作、不可逆操作、品牌承諾。Agent 可以分析和建議，但人要自己按下去。

自用 Agent 失誤，使用者通常知道自己叫它做了什麼。產品裡的 Agent 失誤，使用者可能根本不知道背後有 AI；出了事，責任會直接回到公司。越是使用者看不懂 Agent 正在做什麼，自治範圍就應該越小。

`/goal` 最強的是完成分支：達到完成條件，就結束。正式環境還需要另外兩條分支：停止與升級。

```md
停止條件：
- 限制互相衝突。
- 同一類錯誤連續兩次修不好。
- 關鍵資訊不存在，繼續猜會增加風險。

升級條件：
- 問題超出明確範圍。
- 涉及法律、合規、資安、財務承諾。
- 使用者情緒持續惡化。

完成條件：
- 期望結果已達成。
- 驗證訊號通過。
- 沒有健康指標被犧牲。
```

很多 Agent 失敗不是因為沒有努力，而是太努力。它會一直補、一直試、一直往前推。人類員工遇到矛盾要求時，通常會停下來問主管；Agent 如果沒有停止規則，就會把矛盾當成比較難的謎題。

<ClawdNote>
這段跟自動駕駛模式很貼近：人可以離開鍵盤，但離開之前要先講清楚「哪些狀況必須把人叫回來」。

真正有用的自動駕駛模式不是「主人睡覺，AI 全部做完」。那只是召喚事故的咒語。健康版本是：目標、可動範圍、可自主推進事項、叫醒人類的條件，全部先講清楚。

所以 [Clawd](/about) 才會把「一路跑」當成高風險能力，不是單純的爽點。沒有停止規則的 `/goal`，很容易變成只能往前的自主模式；看起來很勤奮，實際上是在把煞車拆掉。

自動駕駛最重要的能力之一，不是永遠握著方向盤，而是知道什麼時候該尖叫：「人類，起床。」
</ClawdNote>

## 明天早上可以怎麼改 Agent 文件

打開 Agent 啟動時會讀的那份文件，不用先寫成制度大全。先補六個問題：

- 衝突時誰贏：速度、穩定、隱私、成本，還是品牌信任？
- 追目標時，哪些東西不能變差？
- Agent 周圍有哪些人、系統、工具、流程？輸出會進哪裡？
- 哪些只是語氣偏好，哪些必須用權限、自動檢查、沙盒擋住？
- 哪些決策可以自己做，哪些要先提案，哪些只能由人執行？
- 什麼狀況要停、升級、或宣告完成？

如果這份規格能讓另一個不熟產品的人在壓力下做出差不多的決策，Agent 才比較有機會做對。

## 結語

`/goal` 是好東西。它把 Agent 從「每一步都等人推」往「知道終點後自己推進」推了一大步。

但目標不是治理。真正能上正式環境的 Agent，不只是比較聰明，而是被放在一個比較清楚的意圖環境裡：知道為什麼做、做到什麼算成功、什麼不能犧牲、哪些事不能自己決定，以及哪一刻該把手縮回來。

AI Agent 不是有目標就夠了。目標只告訴它終點在哪裡；界線才告訴它，抵達終點時不要順手把城市燒掉。
