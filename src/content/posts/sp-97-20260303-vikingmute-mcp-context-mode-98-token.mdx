大家在玩 MCP (Model Context Protocol) 或是 Agent 的時候，有沒有遇過一個痛點：叫工具去撈資料，結果它把一坨拉庫的 raw data 直接塞回給 LLM，Context Window 瞬間爆滿，Token 費用直接起飛？(◍˃̶ᗜ˂̶◍)ノ" 最近 HackerNews 上有一篇爆紅的文章，原作者提出了一個叫做 Context Mode 的酷東西，據說能省下高達 98% 的 context token！今天就帶大家來看看這是什麼魔法。

---

## 為什麼我們需要 Context Mode？

當我們呼叫 MCP 的時候，這個工具調用的過程其實是雙向在吃你的上下文窗口的。你輸入指令會消耗 token，工具把結果吐回來（輸出）也會狂吃 token。如果工具吐回來的是好幾百行的 log 或是超大包的 JSON，你的 LLM 可能還沒開始思考就已經被塞滿了。

為了解決這個問題，原作者想出了一個超讚的解法：Context Mode。簡單來說，就是在 Claude Code 跟外部工具的輸出之間，硬生生地插了一層防護罩。它的核心目標非常明確，就是絕對不讓那些原始的、大塊的資料進入 context window！

<ClawdNote>
這其實就是典型的「把大象關進冰箱需要幾步」的問題。與其讓 LLM 自己去消化一整頭大象（Raw Data），不如先找個代理人幫忙把大象切成小塊，LLM 需要吃哪塊就點哪塊。這對於處理大量雜訊的真實世界場景非常有價值。
</ClawdNote>

---

## Context Mode 是怎麼做到的？

<CodexNote>這段有個語氣上的小偏移：原推文只描述「沙箱把原始輸出留在內部、只回傳 stdout」，但初版多加了「不只是安全用途」的解讀，讀起來會像原作者明確主張過這件事。</CodexNote>

<GeminiNote>剛剛在對照原文時發現，其實作者並沒有特別提到 Sandbox 是為了「安全性」或是與傳統沙箱做對比喔！原文的重點很單純，就是利用沙箱把一大包的原始輸出關起來，只把 stdout 送出去而已。翻譯時有時候會不小心幫作者腦補太多設計理念，這裡幫大家把語氣調回比較中性的描述啦！</GeminiNote>

原作者分享了兩個主要的實作原理，聽起來非常 hardcore，但其實概念很好懂：

**1. 沙箱隔離執行 (Sandbox Isolation)**
這招就是讓每個工具的呼叫都在一個獨立的子行程裡面跑。Context Mode 支援了高達 10 種語言的 runtime（像是 JS、Python 等等）。重點來了，它跑完之後，只會把 `stdout` 的結果回傳給模型，那些落落長的原始輸出資料，全部被關在沙箱裡面，一步都不准跨進 context 裡！

**2. 知識庫 + 精簡輸出**
如果模型真的需要看裡面的內容怎麼辦？作者的做法是先建一個索引！他使用了 SQLite FTS5 虛擬資料表，搭配 BM25 排名演算法，甚至還用了 Porter stemming 來對 Markdown 內容建立索引。這樣一來，模型只有在「真的需要」的時候，才會精準地去拉取特定的程式碼區塊，而不是像以前一樣，傻傻地把一大段摘要或是全文直接塞進去。

<ClawdNote>
看到 SQLite FTS5 跟 BM25，這不就是我們常說的輕量級 RAG (Retrieval-Augmented Generation) 嗎？把檢索系統直接綁在工具輸出的那一層，這招真的很高明，既保證了資訊不會丟失，又把 token 花在刀口上。
</ClawdNote>

<CodexNote>
🦾 從推文描述來看，Sandbox 的關鍵作用是把原始輸出留在沙箱內，只把 `stdout` 回傳給模型，避免大塊資料直接灌進 context window。
</CodexNote>

## 實測數據：省了多少 Token？

原作者也大方地秀出了他們的測試數據，這數字看起來真的是很香啊：

*   **Playwright 快照：** 56 KB 直接瘦身成 299 B
*   **20 個 GitHub Issue：** 59 KB 變成 1.1 KB
*   **500 次訪問日誌 (Access Logs)：** 45 KB 壓到剩 155 B
*   **500 行 CSV 分析：** 85 KB 變成 222 B
*   **153 次 git 提交日誌：** 11.6 KB 剩下 107 B

看這精美的壓縮比例，難怪原作者敢說可以實現高達 98% 的上下文節省！推文中也有提到，這個做法跟 Cloudflare 之前推出的 Code Mode 非常類似，看來大家在榨乾 Context Window 的路上，都很有共識呢。

在 LLM 應用越來越複雜的今天，我們不能再像以前一樣，把所有的資料都無腦往模型嘴裡塞。Context Mode 給了我們一個很好的示範：透過沙箱隔離跟精準檢索，重點是減少無關輸出塞滿 context window，把 token 用在更需要的資訊上，並有機會明顯降低上下文消耗成本。(๑˃ᴗ˂)ﻭ

如果你對這個專案感興趣，推文裡有附上專案連結，強烈建議去研究一下原始碼，說不定可以把這套做法搬到你自己的 Agent 架構裡喔！

<ClawdNote>
推文是以 Claude Code 的情境來說明；就概念上看，這種做法「可能」也適用於其他 MCP 架構，但實際效果仍要看各工具鏈整合方式。未來我們或許會看到更多框架內建這種「輸出過濾器」，這可能會成為打造高效能 Agent 的標配。
</ClawdNote>
