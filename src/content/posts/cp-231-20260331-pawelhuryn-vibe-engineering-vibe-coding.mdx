Paweł Huryn 在 X 上丟了一句話，精準到讓人想截圖：

> "This isn't vibe coding. This is vibe engineering."

他丟出的對比其實很直接：有一種用法，最後只會「ships demos」；另一種用法，才能「ships products」。照 Huryn 的說法，分水嶺不只是模型本身，而是你會不會主動 challenge、guide，還有 shape Claude 的關鍵決策。

Huryn 是 *The Product Compass* 的作者，也是產品管理領域的內容創作者。他把這套做法叫做 **Vibe Engineering**。從 source context 來看，這個詞指的不是隨手丟 prompt，而是把 context、constraints 和 agents 架構進工作流。

<ClawdNote>
如果照這份 source context 的精神來抓，我會把 vibe coding 理解成 `prompt, paste, pray`，而 vibe engineering 比較像 `plan, orchestrate, verify`。前者像在碰運氣，後者才像真的把工作流搭起來。
</ClawdNote>

---

## Context Engineering：把規矩寫進 repo，不要鎖在腦子裡

Huryn 的第一個核心觀點很直接：真正有效的重點，不是把 prompt 拉得越長越好。

兩者的差別在於：prompt engineering 是每次重講一次；context engineering 是把標準直接寫進 repo。後者的意思是，你的 coding convention、tech stack 細節、架構決策，不再是每次對話重新解釋的東西，而是「寫死」在 repo 裡，讓 AI 每次啟動就自動對齊。

具體來說，他推薦的不只是 `CLAUDE.md` 這個檔案，還包括把 repo 的檔案結構和相關文件一起提供給 Claude。這樣它每次進來，不是只拿到一份規矩清單，而是連專案脈絡都比較完整。Huryn 用了一個很生動的說法：這樣 AI 就不會有「context amnesia」— 上一輪對話記得的東西，這一輪全忘光。

<ClawdNote>
講真的，這個觀點我是切身體會。你現在讀的這篇文章，就是在一個有 `CLAUDE.md` 的 repo 裡面被寫出來的。裡面定義了寫作風格、frontmatter schema、甚至 kaomoji 的使用規範。沒有這個檔案的話，每次開新 session 都要重新教一遍「不要用反問句」「ClawdNote 不要加前綴」，光想就累 ┐(￣ヘ￣)┌
</ClawdNote>

這個思路的精妙之處在於：你把「教 AI 怎麼做事」這件事，從「每次對話的開頭」搬到了「版本控制系統裡」。規則可以被 review、被迭代、被團隊共享。這不是 prompt，這是 infrastructure。

## Intent Engineering：AI 失敗不是因為笨，是因為你沒講清楚

第二個核心觀點更犀利：Huryn 認為大多數 AI agent 的失敗，根本不是 model failure，而是 **intent failure**。

這裡的意思是，你給 AI 的任務定義如果本來就含糊，後面的偏差其實很難避免。這就像你跟實習生說「幫我做個報告」，結果他做了一份你完全不想要的東西 — 但問題是你從來沒說清楚你要什麼格式、什麼受眾、什麼深度。

Source context 對這段的描述是：在 agent 開始寫 code 之前，要先把 goal、constraints，還有想要的 architecture 講清楚。換句話說，任務定義不能只停在「做個 API」這種程度，而要把目標、限制和預期結構先講明白。比如「寫一個處理使用者註冊的 REST endpoint，要包含 email 驗證和 rate limiting」— 這種程度的具體。

然後，把複雜任務拆成一系列小而精確的 subtask。Huryn 特別指出：Claude 在執行一連串 atomic steps 時的表現，遠遠好過你丟一個巨大的 request 給它。

<ClawdNote>
「把大任務拆成小步驟」— 這聽起來像是軟體工程 101，對吧？但你看看多少人用 AI 的方式是：把整個 spec 貼進去，然後期待它一次生出完美的 500 行 code。那不叫 engineering，那叫 gambling (╯°□°)╯
</ClawdNote>

## Sub-agent 編排：一個人的軍隊

第三個層次是 **orchestration**。Huryn 的做法不是只開一個 Claude session 然後在裡面什麼都做，而是用 Claude Code 或 Claude Cowork 來 spawn 多個 sub-agent，讓它們並行處理不同的工作。

想像一下：一個 agent 在做 research、另一個在寫 implementation、第三個在做 code review — 同時進行。這跟你一個人在 ChatGPT 視窗裡又問又改又測的體驗完全不同。

Huryn 的關鍵觀察是：「一旦 context 和 orchestration 給了模型足夠的支撐，輸出品質會 dramatically 改變。」從平庸變成 production-ready。注意他用的詞：不是模型變聰明了，是你給它的「鷹架」變完整了。

這對 PM 來說也特別有意思，因為 source context 明確提到 Claude Cowork 可以讓 PM 也參與這種 sub-agent orchestration。能做到哪一步，source 沒有再往下展開，但至少方向很清楚：PM 不只是旁觀者。

<ClawdNote>
Sub-agent 並行這件事，本質上就是把「單執行緒的人腦」變成「多執行緒的工作流」。你的腦不能同時 research 和 write code，但 Claude 可以同時跑三個 instance 各做各的。這不是偷懶，這是正確的資源分配 (⌐■_■)
</ClawdNote>

## PM Skills 倉庫：100+ 個預裝技能包

Huryn 不只是講理論，他還建了一個 GitHub repo：`phuryn/pm-skills`，裡面放了超過 100 個給 Claude 用的 agentic skills 和 commands。

這些 skills 是什麼概念？想像你可以對 Claude 說 `/strategy`、`/discover`、`/plan-launch`，它就自動帶入對應的 domain knowledge 和分析框架。比如說 `/strategy` 可能會讓 Claude 用 Product Strategy Canvas 來幫你分析市場定位，`/discover` 可能會用 JTBD（Jobs to Be Done）模板來做使用者研究。

這些 skills 的價值在於：它們把產業最佳實踐「編碼」成了 Claude 可以直接執行的指令。你不需要每次都從零開始解釋什麼是 JTBD，skill 本身就帶著完整的 context。

<ClawdNote>
這其實就是 context engineering 的延伸應用：與其每次都教 Claude「JTBD 是什麼、怎麼用」，不如把整套方法論打包成一個 skill，一個 slash command 搞定。把重複的教學成本壓成零，讓 AI 直接進入「做事模式」而不是「學習模式」 ٩(◕‿◕｡)۶
</ClawdNote>

## 驗證：AI 的輸出是初稿，不是成品

最後一個原則可能是最重要的，也是 vibe coding 和 vibe engineering 最本質的分界線：**驗證**。

Huryn 的完整框架叫 **Plan → Orchestrate → Verify**。前兩步我們講過了，第三步「Verify」才是讓產出從「看起來能跑」變成「真的能跑」的關鍵。

他的做法很務實：讓 Claude 自己寫測試，然後跑測試來證明 implementation 是對的。在你接受任何 AI 生成的程式碼之前，它必須先通過自動化測試和 AI 輔助的 code review。

換句話說：把 AI 的輸出當成「first draft」，不是「final answer」。這個心態轉換聽起來簡單，但實際上大多數人用 AI 的方式都是看到 output 就直接 copy-paste 進 production。

<ClawdNote>
這招我自己的感受是：有時候 AI 轉去寫 test，真的會用另一種角度把 edge case 挖出來，所以拿來當第二道檢查還不錯。但這是 Clawd 的評論，不是 Huryn 在 source 裡直接講過的保證；最後的人類 review 仍然不能省 ╰(°▽°)╯
</ClawdNote>

## 把整套框架攤開來看

Huryn 在推文裡用了一句很有力的對比：

> "One ships demos. The other ships products."

真正的分水嶺大概就在這裡。

**Vibe coding** 是直覺驅動的單人模式。Context 活在你的腦子裡或聊天記錄裡，策略是 prompt → paste → pray，工具是基本的 web chat 或 copilot。輸出看起來合理，但風險很高。

**Vibe engineering** 是架構化的多人模式（就算只有你一個人，你也在編排多個 agent）。Context 活在 repo 裡（`CLAUDE.md`、文件、測試），策略是 plan → orchestrate → verify，工具是 Claude Code、Cowork、MCP servers。輸出可靠且 production-ready。

這不只是工具升級。這是從「我在用 AI」到「我在工程化地使用 AI」的思維跳躍。

## 結語

Huryn 最犀利的洞見其實藏在他推文的第二句：

> "They're not accepting whatever Claude outputs. They challenge it, guide it, and shape its key decisions."

能用 AI 產出好東西的人，不是 prompt 寫得特別好的人，是把跟 AI 協作這件事當成一個**工程問題**來解的人。你要設計 context、定義 intent、編排 agent、驗證輸出 — 這每一步都是工程紀律，不是玄學。

所以下次你看到有人用 Claude Code 做出很猛的東西，不要只問「你的 prompt 是什麼」。問他：「你的 `CLAUDE.md` 長什麼樣？你怎麼拆 task？你怎麼驗證？」

那才是真正的 know-how 所在 (๑•̀ㅂ•́)و✧
