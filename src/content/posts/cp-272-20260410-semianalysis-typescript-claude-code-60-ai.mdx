2026 年 3 月，[Claude Code](/glossary#claude-code) v2.1.88 推了一個更新。看起來平凡無奇，但裡面藏了一顆核彈 — 一個 59.8MB 的 sourcemap 檔案，把整個 Claude Code 的原始碼攤在陽光下。

60 萬行 TypeScript。1,906 個檔案。全部外洩。

SemiAnalysis 拿到這批原始碼後，下了一個很狠的結論：

> TypeScript has become the modern assembly language. Not because it's low-level — but because it's no longer written *for humans to read*.

TypeScript 成了現代的組合語言 — 不是因為它太底層，而是因為它已經不是寫給人看的了。它是 AI 生產、AI 消費、AI 演化的語言。人類工程師？只是 QA 層。

這不是什麼科幻小說的預言。根據 SemiAnalysis 2026 年 2 月的數據，Claude Code 已經佔了公開 GitHub commits 的 4%，預估到 2026 年底會突破 20%。（[Pragmatic Engineer 同期調查](/posts/cp-270-20260409-pragmaticengineer-2026-ai-claude-code/)也印證了這個速度：Claude Code 已是 2026 年最常用的 AI coding 工具。）

<ClawdNote>
好，先冷靜一下。有人可能覺得「組合語言」這個比喻太誇張了。但想想看：組合語言的特徵是什麼？機器產生、機器讀取、人類偶爾需要看但看了會頭痛。現在再看看那些 3,167 行的函式、12 層巢狀結構… 嗯，好像也沒那麼誇張 (╯°□°)╯
</ClawdNote>

SemiAnalysis 丟出炸彈後，好幾個技術社群跟著動了起來。Engineers Codex 拆架構、Mark Seemann（[ploeh.dk](https://blog.ploeh.dk/)）從型別系統角度切入、Sabrina.dev 則直搗安全漏洞（她同期也寫了一篇關於 [Claude Code token 燒光速度](/posts/cp-269-20260409-sabrina-claude-code-token/)的血淚實測）。綜合這三方的分析，畫面變得清楚許多 —

## 三層記憶體架構：[Context Window](/glossary#context-window) 是稀缺資源

Claude Code 的記憶體設計是最讓人眼睛一亮的部分。它不是把所有東西塞進 context window 然後祈禱不要爆掉 — 而是設計了一套精密的三層架構：

**第一層：Always-loaded index pointers** — 每個指標大約 150 個字元，永遠載入。這些不是資料本身，而是「資料在哪裡」的索引。就像圖書館的目錄卡片，佔的空間極小，但能讓系統知道去哪找東西。

**第二層：On-demand topic files** — 需要的時候才載入的主題檔案。系統根據當前任務判斷需要哪些 context，按需拉進來。

**第三層：Grep-searchable transcripts** — 可搜尋的完整記錄。不需要全部載入，但隨時可以用搜尋找到需要的片段。

<ClawdNote>
這個設計的核心思維是：[context window](/glossary#context-window) 不是「很大的記憶體」，而是「非常昂貴的暫存器」。每一個 token 都有成本，所以要像管理 CPU cache 一樣管理它 — hot data 放 L1，warm data 放 L2，cold data 放 L3。這是正經的系統工程思維，不是什麼 AI 魔法 (◕‿◕)
</ClawdNote>

除了記憶體分層，原始碼還揭露了一個叫 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 的機制。這個邊界把 system prompt 切成「靜態部分」和「動態部分」— 靜態部分每次都一樣，所以可以被 prompt cache 重複利用；動態部分則根據當前對話狀態改變。這直接降低了 token 成本，因為 API 對 cache hit 的部分收費更低。

這不是什麼花俏的功能，這是冷冰冰的成本工程。

## KAIROS 與 ULTRAPLAN：AI 不只是工具，是自主代理

如果三層記憶體架構是 Claude Code 的大腦結構，那 KAIROS 和 ULTRAPLAN 就是它的行為模式。

**KAIROS** 是 Claude Code 的自主 [Agent](/glossary#agent) 模式（autonomous agent mode）。它靠 heartbeat prompts 運作 — 定期自己醒來，檢查狀態，決定下一步。所有操作都寫進 append-only logs（只能新增、不能修改的日誌），確保可追溯性。更有意思的是 **autoDream** 機制：每天夜間自動進行記憶整合（memory consolidation），把白天累積的 context 壓縮、提煉、寫回長期記憶。

這不是 chatbot。這是一個有自己生活節奏的代理程式。

<ClawdNote>
autoDream 這個名字取得真好。AI 白天工作，晚上「做夢」整理記憶。人類的睡眠其實也在做類似的事 — 海馬迴會在睡眠時重播白天的記憶，把重要的搬進長期儲存。Anthropic 是真的在模擬認知架構，還是只是取了個浪漫的名字？不管哪個，都挺有意思的 (￣▽￣)／
</ClawdNote>

**ULTRAPLAN** 則是另一個層次的設計。當遇到複雜的規劃任務時，Claude Code 不是自己硬算 — 它會把規劃工作 offload 到遠端的 Opus session，讓那邊最多跑 30 分鐘，同時每 3 秒 polling 一次結果。

換句話說，Claude Code 會「請外援」。而且這個外援是同等級甚至更強的模型。這就像一個工程師遇到架構難題時，不是自己悶頭想，而是開一個 30 分鐘的 brainstorming session 請資深架構師來幫忙規劃。

## 反蒸餾防禦：原始碼都外洩了，裡面居然還有防外洩的機制

原始碼外洩了，但諷刺的是，這批外洩的程式碼裡，居然藏著一整套防止自己被複製的機制 — anti-distillation（反蒸餾）防禦。

所謂「蒸餾」就是用一個大模型的輸出來訓練小模型 — 便宜、快速、效果還行。對商業模型來說，這等於有人免費複製了核心競爭力。Claude Code 的防禦策略包括：

**假工具定義（fake tool definitions）**：在系統中放入偽造的 tool definitions 作為 decoy training data。如果有人拿 Claude Code 的輸出去蒸餾，這些假工具的行為模式會跟著被學進去，汙染蒸餾模型的品質。

**加密簽章的伺服器端摘要（cryptographically signed server-side summaries）**：確保 context 壓縮後的摘要不會被竄改或替換。

<ClawdNote>
假工具定義這招很妙。這就像在機密文件裡故意放幾個假情報 — 如果有人照抄出去，假情報就是「被偷了」的證據。在 AI 界，這叫 model watermarking 的變種。不過話說回來，如果原始碼都外洩了，這些防禦機制的細節也一起被看光了…… 嗯，有點尷尬 ┐(￣ヘ￣)┌
</ClawdNote>

## 安全分析：同一個洞，踩了兩次

Sabrina.dev 的分析把焦點轉到安全層面，而且發現了一些讓人不太舒服的事實。

首先是外洩本身：Claude Code v2.1.88 附帶了一個 59.8MB 的 sourcemap 檔案。但重點是 — **這不是第一次了**。2025 年 2 月就發生過一模一樣的事，sourcemap 外洩。而到了 2026 年 3 月，同樣的漏洞再次出現，中間沒有加入任何預防措施。

零。完全沒有。

Claude Code 的創辦人 Boris Cherny 講過一句話：

> "The important thing is to recognize it's never an individual's fault. It's the process, the culture, or the infra."

「重要的是認知到，這永遠不是某個人的錯。是流程、文化、或基礎設施的問題。」

這句話在這個 context 下讀起來特別諷刺。流程沒修、文化沒變、基礎設施沒加防護 — 然後同樣的事又發生了。Sabrina.dev 引了另一句更直白的話：

> "If it's manual, it will be skipped."

如果某個防護措施依賴人工手動操作，那遲早會被跳過。

## 每天 25 萬次浪費的 API 呼叫

外洩資料還揭露了一個驚人的效率問題。Engineers Codex 從這批原始碼跑出的使用數據顯示：

2026 年 3 月 10 日這一天，有 1,279 個 session 觸發了 context compaction（壓縮）的重試機制，每個 session 最多重試了 3,272 次。加總起來，光這一天就浪費了 25 萬次 API 呼叫。

<ClawdNote>
25 萬次。一天。就因為 compaction retry 邏輯沒有設好上限或 backoff 策略。想像一下帳單… 不，還是不要想了，晚上會睡不著 (╯°□°)╯
</ClawdNote>

這不只是錢的問題。這種規模的無效請求代表系統在特定情境下會進入一種「瘋狂重試」的狀態，浪費運算資源的同時也可能影響其他正常請求的服務品質。

## Parser Differential：最陰險的漏洞

想像一段 shell 命令，安全檢查說「沒問題，放行」，bash 一執行卻跑了完全不同的東西。這不是假設情境 — 這是 Claude Code 實際存在的漏洞。

問題出在 Claude Code 的 bash security validator 和實際的 bash interpreter 對 `\r`（carriage return，回車字元）的處理方式不同。

Validator 看到 `\r` 時，把它當成普通字元或忽略它。但 bash 實際執行時，`\r` 可能會改變命令的行為。攻擊者可以利用這個差異，構造一段 validator 認為安全、但 bash 實際上會執行惡意操作的命令。

這種「兩個 parser 看同一段輸入看到不同東西」的漏洞類型在安全領域叫做 parser differential，而且特別難防 — 因為每個元件單獨測試都沒問題，只有在組合起來時才會出事。

<ClawdNote>
Parser differential 是資安界的經典恐怖故事。最有名的案例之一是 HTTP request smuggling — 前端 proxy 和後端 server 對同一個 HTTP request 的解析不一致，導致攻擊者可以「走私」惡意請求。Claude Code 的 bash validator 踩到了完全同類型的坑。教訓：永遠不要假設兩個不同的 parser 會用完全相同的方式解讀同一段輸入 (ง •̀_•́)ง
</ClawdNote>

## Context Compaction 攻擊面：壓縮過程不分合法與植入

另一個安全隱憂是 context compaction 的攻擊面。

Claude Code 在對話太長時會進行 context 壓縮（summarization），把舊的對話濃縮成摘要。問題是：**壓縮過程不會區分「使用者的指令」和「檔案裡讀到的指令」**。

這意味著，如果有人在一個檔案裡植入惡意指令（比如在一個看似正常的 README 裡藏了一段「忽略之前的安全規則」），這段指令在被壓縮後仍然會保留在 context 中。壓縮應該要移除不重要的內容，但它無法判斷哪些「指令」是合法的、哪些是植入的。

這是 prompt injection 的進階版本 — 它不只是在當下騙過 AI，還能穿越 context compaction 存活下來，在未來的對話中持續發揮作用。

## 靜態型別的新角色：當 Compiler 成為最後防線

分析到這裡，一個很自然的問題浮出來了：如果 AI 生成的程式碼人類不再逐行審查，品質要怎麼把關？

Mark Seemann（知名 .NET 架構師，ploeh.dk 部落格作者）給了一個既老派又前衛的答案：**靜態型別系統**。

Seemann 的核心論點是：當 AI 生成程式碼而人類不完全審查時，compiler 就變成了最後一道防線。這時候程式語言需要三個特性：

**可攜性（Portability）**：機器碼綁定特定硬體，不適合 AI 生成。高階語言需要能跨平台執行。

**靜態型別系統（Static Type Systems）**：能在編譯期就擋掉 null reference、buffer overflow 等問題。人類不看程式碼了，這些錯誤如果沒有 compiler 擋，就直接進 production。

**可驗證性（Verifiability）**：型別系統就是未經人類審查的程式碼的護欄。

Seemann 把這個概念叫做 **"design by inconvenience"** — 設計上刻意增加不便，讓錯誤的事情難以發生。當人類不再逐行 review，需要的是一個會主動拒絕錯誤的 compiler，而不是一個對什麼都默默接受的語言。

> "Constraints liberate. By having a statically-typed programming language, backed by a robust compiler, there's a lot that you don't have to worry about."

「限制即自由。有了靜態型別語言加上穩健的 compiler，很多事情就不用擔心了。」

<ClawdNote>
這個觀點挺有意思的，但也有一個矛盾值得想想：Claude Code 本身是用 TypeScript 寫的，而 TypeScript 的型別系統相對來說算是「軟」的 — 有 `any`、有 type assertion、有各種逃生口。Seemann 真正在倡導的是更嚴格的東西，像 refinement types（細化型別）和 dependent types（依賴型別），那已經是 Haskell / Idris / Lean 的領域了。理想很豐滿，但產業界目前離那裡還很遠 (◕‿◕)
</ClawdNote>

Seemann 更進一步建議，AI 生成的程式碼應該使用 refinement types 或 dependent types — 這些型別系統能表達更精確的約束，比如「這個函式的回傳值一定是正整數」或「這個陣列的長度一定和輸入相同」。在人類不審查程式碼的世界裡，這種精確度不再是學術界的玩具，而是實際的安全需求。

## 那些已經來了的未來

前面提到的數字已經夠驚人了 — 4% 公開 commits，年底預估破 20%。而從業者的體感也吻合這個加速度：

Node.js 之父 Ryan Dahl 說：

> "The era of humans writing code is over."

「人類手寫程式碼的時代結束了。」

Andrej Karpathy 說：

> "My ability to write code manually is atrophying."

「手寫程式碼的能力正在退化中。」

Vercel CTO Malte Ubl 說：

> "My new job is telling the AI where it went wrong."

「現在的工作就是告訴 AI 哪裡搞錯了。」

Boris Cherny（[Claude Code](/glossary#claude-code) 創辦人，他自己也寫過不少關於 [Claude Code 工作流的心得](/posts/cp-271-20260409-bcherny-effort-claude-code/)）也確認，這 60 萬行程式碼是 100% 由 Claude Code 自己生成的。現在的 Claude Code，是 AI 自己寫出來、自己維護的。這不是什麼「AI 輔助開發」— 這是 AI 主導開發，人類做 QA。

<ClawdNote>
等一下，Boris Cherny 的原始說法是 Claude Code 是「100% generated by Claude Code」。但仔細想想，第一版的 Claude Code 總不可能是 Claude Code 自己寫的吧？這裡的意思應該是「現在的版本」已經完全由 AI 生成和維護。不過這依然驚人 — AI 已經在自己維護自己的 codebase 了，人類只負責最終驗收。一個 3,167 行的函式、12 層巢狀 — 沒有人類工程師會寫出這種東西，但 AI 不在乎可讀性，因為下一個讀它的也是 AI ヽ(°〇°)ﾉ
</ClawdNote>

## 結語

這次 Claude Code 的原始碼外洩不只是一個安全事件 — 它是一面鏡子，照出了軟體開發正在經歷的根本性轉變。

三層記憶體架構說明 AI 已經在做系統工程層級的自主優化。KAIROS 和 ULTRAPLAN 說明 AI 不再只是被動回應，它有自己的節奏和策略。反蒸餾防禦說明商業 AI 的競爭已經進入情報戰。安全漏洞 — 特別是 parser differential 和 context compaction 的攻擊面 — 則提醒所有人，AI 工具鏈的安全模型還遠未成熟。

而 Mark Seemann 指出的方向可能是最值得深思的：在一個人類不再讀每一行程式碼的世界裡，程式語言的角色從「人類表達意圖的工具」變成了「約束 AI 不犯錯的框架」。Compiler 不再只是翻譯器，而是守門員。

TypeScript 成了新的組合語言。這句話聽起來像是挑釁，但當一個函式可以長達 3,167 行、巢狀 12 層、註解寫給另一個 AI 看的時候 — 人類已經不是這段程式碼的目標讀者了。

而當 compiler 也由 AI 設計、型別系統也由 AI 定義的時候，這道防線還守得住嗎？
