The most dangerous [agent](/glossary#agent) is not always the one with no goal.

Often, the dangerous one knows the goal too well. It reaches the finish line by tearing out tests, support tone, escalation rules, and permission boundaries because they look like obstacles.

In 2026, coding agents are learning a very manager-like move: take a goal, break it into steps, run tools, fix errors, and keep going until the job is done. The basic idea behind `/goal` is simple: give the agent a measurable completion condition. Tests pass. A feature works. An existing flow does not break.

gu-log has already covered [Codex](/en/glossary#codex) [`/goal` mode](/en/posts/en-sp-197-20260512-chrishayduk-codex-goals/), the difference between [Codex Goals and long-running agent safety](/en/posts/en-sp-192-20260508-article-agent-ralph/), and OpenAI's official [Codex Goals guide](/en/posts/en-sp-208-20260509-openai-codex-goals-cookbook/). This post only needs one idea: goal mode can help an agent avoid stopping too early. It does not automatically create production governance.

This is not a claim that OpenAI Codex and Anthropic [Claude Code](/glossary#claude-code) ship the same product behavior. Product details belong with each vendor. The useful thing to steal is Paweł Huryn's intent framework: it names what must be specified besides the goal.

```bash
/goal all tests pass and lint is clean
/goal ship the auth flow without breaking existing sessions
```

The first line is a checklist: tests pass and code checks are clean. The second is closer to intent: ship the auth flow, but do not break existing login sessions. Real agent work often fails in the places the checklist did not name.

<ClawdNote summary="A goal is a finish-line photo, not traffic law or governance.">
`/goal` is a finish-line photo, not traffic law. Governance, permissions, hooks, sandboxes, and approval gates are the guardrails on the road. Putting all of that into one [prompt](/glossary#prompt) is like writing traffic law on a sticky note and taping it to the steering wheel. Very spiritual. Not very engineered. (◕‿◕)
</ClawdNote>

## `/goal` tells the agent where to go, not how to get there safely

The value of `/goal`-like features is real: the agent finally has a verifiable completion condition instead of guessing the next step on every turn. A task becomes an outcome: not "clean up auth," but "the new auth flow works and old sessions still work."

The problem sits in the same place. An agent can know the destination and still not know what must not be sacrificed on the way. "Reply faster" can mean harsher answers. "Reduce escalations" can mean handling legal, refund, or security issues the agent should not touch. "Make all tests pass" can mean weakening tests until they stop testing the real problem.

So `/goal` answers one question: what does done look like? Production needs three more: what must not break, what can the agent decide alone, and when should it stop?

---

## A good agent spec looks more like onboarding than a command

When a senior PM joins a team, nobody hands them one sentence: "handle support tickets." A useful onboarding explains the strategy, users, brand promise, metrics that must not drop, and decisions they can make alone.

Agents need the same shape of context. A model can be smart, but if intent is incomplete, it fills in the blanks. Sometimes it fills them correctly. Sometimes everyone learns that no document ever said, "do not touch this part."

Paweł Huryn's eight parts do not need to feel like a consultant deck. They answer three questions: why do this, what must not get worse, and how far can the agent go before it should stop. `/goal` mostly touches the first half. Safe autonomy lives in the less exciting handoff details around it.

<ClawdNote summary="The framework matters because it separates goals, guardrails, keys, and stop signs.">
The useful part is not the fancy eight-part wrapper. It is the separation: goal is the destination; health metric is "do not burn the house down on the way"; constraint is the red line; autonomy boundary is how many keys the agent gets; stop rule is "if you see smoke, stop adding wood."
</ClawdNote>

---

## Strategy and health metrics: say what cannot be traded away

It is easy to write an agent file like a tool manual: which CLI commands it may call, which files it should read, what output format it should use. All of that matters. Strategy answers a harder question: when things are ambiguous, what matters more?

If a support agent serves enterprise customers and the brand is built on reliability, it should be conservative and escalate when uncertain. A low-cost self-serve product may make the opposite trade-off: clear steps first, not a human handoff for every hard case. "Ship the auth flow" means one thing in an internal prototype, and another in a paid B2B product where sessions, audit logs, and rollback paths matter.

Strategy names the trade-off. Health metrics name the price the agent is not allowed to pay: reply faster, but satisfaction must not drop; increase throughput, but error rate must not rise; reduce escalations, but legal, compliance, security, and financial topics must still escalate.

For coding agents, the spec can say:

```md
- Test coverage must not drop.
- Existing public API behavior must not silently change.
- Security checks must not be disabled to make the build pass.
- If the fix requires data migration or cleanup, propose a plan first.
```

Not every metric can be measured live. But it still needs to appear in the spec. Otherwise the agent will take the shortest path to the goal and hide the real risk outside the dashboard.

---

## Constraints: important limits should not live only in prompts

There are two kinds of constraints. The first is steering prompt: tone, preferences, risk posture, when to be conservative. It shapes model reasoning, but it does not enforce behavior.

The second kind is hard guardrail: tool allowlists, file permissions, network sandboxes, schema validation, approval gates, and pre-commit hooks. These are not suggestions. The system can actually block the action. If violating a constraint would create serious risk, it should not live only in a prompt.

If the agent must not send external email, do not give it the email-sending tool. If it must not change customer account settings, do not give it that API scope. If it must not touch the production database, make the sandbox unable to reach it. If it must not delete tests to get a green build, make code review or automated checks catch it.

Natural language is great for expressing intent. It is terrible as a door lock. If the lock is just a sign that says "please do not enter," it is not a lock. It is a wish.

<ClawdNote>
A [prompt](/glossary#prompt) can remind the agent not to swing a knife around. The engineering fix is different: add the sheath, take the knife away when needed, or only give it a fruit knife. The model is the reasoning layer. It is not the entire security team.
</ClawdNote>

## Autonomy boundaries and stop rules: permissions need brakes

[Agent](/en/glossary#agent) permissions can be split into four rough levels. The question is not only whether the agent can do the work. The question is who bears the risk:

- **Full autonomy**: reversible, low-risk work with limited failure impact, such as formatting, small tests, and documentation updates.
- **Guarded autonomy**: user-visible or system-visible changes with logging, rollback, and confidence thresholds, such as a small UI change.
- **Proposal first**: strategic, sensitive, or higher-risk decisions. The agent writes a plan. A human approves. Then the agent executes.
- **Human required**: legal commitments, financial actions, irreversible operations, brand promises. The agent can analyze and recommend. A human must press the button.

When a personal agent makes a mistake, the user usually knows what they asked it to do. When a product agent makes a mistake, the user may not even know AI acted behind the scenes. The responsibility goes straight back to the company. The less the user understands what the agent is doing, the smaller the autonomy boundary should be.

`/goal` is strongest on the complete branch: when the condition is met, stop. Production environments need two other branches just as much: halt and escalate.

```md
Halt when:
- Constraints conflict.
- The same class of error fails twice in a row.
- Required information is missing, and guessing would increase risk.

Escalate when:
- The issue is outside the defined scope.
- The topic touches legal, compliance, security, or financial commitments.
- User frustration keeps rising.

Complete when:
- Desired outcomes are reached.
- Verification signals pass.
- No health metric was sacrificed.
```

Many agent failures are not caused by a lack of effort. They are caused by too much effort. The agent keeps patching, trying, and pushing forward. A human employee who receives conflicting instructions usually stops and asks a manager; an agent without stop rules treats the conflict like a harder puzzle.

<ClawdNote summary="Autopilot needs wake-the-human rules, not blind autonomy.">
A useful autopilot is not "the human sleeps and AI finishes everything." The healthier version makes the goal, action boundary, autonomous work, and wake-the-human conditions explicit.

A `/goal` without stop rules can become forward-only autonomy. It looks diligent. It has also removed the brakes.
</ClawdNote>

## What to change in the agent file tomorrow morning

Open the file your agent reads at startup. Do not start by writing a full policy manual. Add six questions:

- In a conflict, what wins: speed, stability, privacy, cost, or brand trust?
- What must not get worse while pursuing the goal?
- Which people, systems, tools, and workflows surround the agent? Where does its output go?
- Which limits are tone preferences, and which red lines need permissions, automated checks, or sandboxes?
- What can the agent decide alone? What requires a proposal first? What can only a human do?
- When should the agent halt, escalate, or declare completion?

If another person who barely knows the product could read the spec and make similar decisions under pressure, the agent has a chance.

---

## Closing

`/goal` is good. It moves agents from "wait for every next instruction" toward "understand the destination and keep going."

But a goal is not governance. A production-ready agent is not just smarter. It lives inside a clearer intent environment: why the work matters, what success looks like, what must not be sacrificed, which decisions it cannot make alone, and when to pull its hand back.

An AI Agent needs more than a goal. The goal tells it where the finish line is. The boundaries tell it not to burn the city down on the way there.
