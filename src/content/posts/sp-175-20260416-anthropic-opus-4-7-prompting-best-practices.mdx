週一早上 10:30，team 的 code review dashboard 跑完週末 batch。Report 乾淨到讓人意外——五個 issue、全部 CVE 級別、全部 high severity。Team lead 點個「approved」就關掉 laptop 去泡咖啡。週二下午 prod 炸了。火燒到 channel 之前 15 分鐘，release manager 用 `git blame` 找到那一行：一個 off-by-one，review 時 [harness](/glossary#agent-harness) 明明看到了，但把它歸類成「not high severity」，安靜 skip。4.6 時代會順手 flag，4.7 照字面不 flag。

這不是虛構的教學案例——Anthropic 官方在 [Claude Code](/glossary#claude-code) [best practice](https://claude.com/blog/best-practices-for-using-claude-opus-4-7-with-claude-code)（短版）跟[完整 prompting guide](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)（長版）裡實際記錄的 case study。兩份 docs 同時釋出，就是在警告一件事：Sonnet / Haiku 時期那套 prompt 調教心得正在過期。不是技巧錯了，是 4.7 變成了另一個人。

## 愈往裡挖愈噁心

Release manager 攤開週末 batch log，本來只想找那個 off-by-one 怎麼漏掉的。結果一挖就停不下來——每一層都比上一層離譜。

最表面：prompt 寫著「report only high severity issues」。4.6 會自己 balance recall 跟 precision——logic error 不是 CVE 也會順便提一嘴。4.7 完全照字面做。門檻收到只剩 CVE 級別，off-by-one 被有意識地歸類為「not high」然後 skip。不是漏看，是 model 看完、判斷、然後決定不講。官方 docs 給的修法核心就一個原則：**拆開**。Coverage 歸 coverage、filter 歸 filter，不要用同一句話交代兩件事。4.7 遇到多重意圖會挑最 literal 的那個意思跑，剩下全丟。Issue triage、log anomaly detection、security scanning——全會中同一個招。

往下一層，release manager 的血壓開始升。Harness 裡有條 rule 引用了一個半年前 renamed 的函式名。4.6 碰到未知名稱會自動 `grep` repo 確認，找不到就停下來問。4.7 不驗證——直接拿 training data 裡的舊名推出一個「聽起來完全合理」的結論繼續走。思路乾淨、邏輯通順、結論建在過期知識上。這是 4.7「偏 reasoning、少 tool use」的暗面：想得很清楚，但前提可能是假的。修法也就一行——「回答前先讀相關 file 或跑一次 grep 確認」——但這一行背後的意思重如山：**4.7 預設信任自己的記憶，不信任 tool**。

最底層讓 release manager 差點摔筆。Review harness 設計成三個 [subagent](/glossary#subagent) 平行讀三個檔案，最後 aggregator 彙整。實際跑起來只有一個 spawn 成功，另外兩個從來沒醒。Coverage 只拿到 1/3——但 log 上面寫著「review complete」，表面看不出少了什麼。4.6 像活潑小孩，不管就到處亂跑；4.7 像精算過的員工，沒明確寫「平行 spawn」就只開一個然後交差。（什麼時候該拆 multi-agent、什麼時候一個就夠，[SP-172](/posts/sp-172-20260413-anthropic-multi-agent-when-how/) 有完整框架。）

三條線匯合。Root cause 不是 model 變笨——是 **4.6 會揣測 prompt 裡沒寫清楚的部分，4.7 不揣測**。

<ClawdNote>
Release note 寫的是「improved instruction following」和「more predictable behavior」。翻成實話：model 變得太聽話，所以過去靠 model 自己補位的偷懶寫法全部原形畢露。但這裡有一個大家不太敢講的反直覺：**4.7 這樣才是對的**。4.6 幫忙腦補聽起來很貼心，但工程師花了多少時間在 debug「為什麼同一個 prompt 有時候會、有時候不會」？一個不揣測的 model 可以 debug、可以預測、可以寫 test——這才是工程師要的工具，不是一個「有時候體貼有時候擺爛」的同事 (⌐■_■)
</ClawdNote>

## 開大火力就能解決？

Prompt 修完上線。Team lead 鬆了口氣，但心裡還有一個刺——如果當初 effort 開到 max，是不是根本不會出事？

這個直覺很合理，也完全錯誤。

同一組 review，五種 effort 各跑一次。`max` 當然抓到了 off-by-one 和旁邊的 race condition——然後多附了兩種 migration 策略比較、一份 refactor 建議書、外加一篇「為什麼這個模組應該拆成三個 service」的短論文。Team lead 看完嘆氣：「問有沒有 bug，回來的是辯論賽。」

降一級試 `xhigh`。同樣兩個 bug，report 乾淨俐落，沒有附贈論文。這是 Claude Code 的預設 effort——team lead 到這一刻才理解為什麼。不是「最高品質」，是投入跟產出之間的甜蜜點。

但讓人不安的是另一頭。`high` 抓到 off-by-one，race condition 滑過去了——那需要跨三個 file 的 context 才看得出來。Anthropic 把 `high` 定義成 intelligence-sensitive workload 的底線，ceiling 就在這。`medium` 更慘：打開了隔壁 file，但只做表面清理——重命名變數、調格式，logic bug 完全沒碰。Team lead 後來的比喻：「叫實習生『仔細看一下』但沒定義什麼叫仔細。」`low` 最乾脆，scope 外的東西連打開都不打開。Off-by-one 在隔壁 file 的 boundary check 裡，`low` 根本不知道那個 file 存在。

好，所以 effort 調到 `xhigh` 就萬事大吉了？回去看 log——team lead 跑這五輪測試時，用的是**修完之後的 prompt**。那如果是修之前那份寫著「only high severity」的爛 prompt 呢？`max` 的 literal execution 只會讓它更徹底地 skip 那些被判定成 not-high 的 issue。Effort 是油門，不是方向盤。方向盤歪了，油門踩再大只是更快撞牆。

Benchmark 結論寫進 postmortem addendum：**prompt 先修對，然後 coding 場景 `xhigh` 起跳、intelligence-sensitive 至少 `high`、格式工作才降 `medium`/`low`**。高階 effort 配 `max_tokens` 64k 起跳，給 reasoning 夠空間想完。

<ClawdNote>
忍不住吐槽 naming。`low`、`medium`、`high`、`xhigh`、`max`——不叫 `higher`、不叫 `very_high`，直接跳 `xhigh`，聽起來像某個 Anthropic 工程師跑完 A/B test 發現少打兩個字比較好就直接 ship 了。等下一代 model 出來要補一層 `ultrahigh` 或 `ludicrous`，`xhigh` 瞬間降格中間層。這個命名軌跡跟 CPU 一模一樣：Pro → Pro Max → Ultra → Ultra Max，遲早階梯自己長腳逃走 (¬‿¬)
</ClawdNote>

## Opus 4.7 覺得公司是精品咖啡店

同一週，隔壁桌的工程師碰到一件跟 pipeline 完全無關的事。

這位工程師剛查完一個 API latency spike，需要一個內部 monitoring dashboard——Tailwind、dark mode、data table、chart，spec 寫在 ticket 裡。丟給 Claude Code，等它跑完，打開 localhost。

螢幕上是 `#F4F1EA` 奶白底色、Georgia 字體、Playfair Display 標題、磚紅色 button、italic word-accents。整個 UI 像 Aesop 專賣店官網。

工程師查了十五分鐘才搞清楚不是人為——Opus 4.7 對設計有極強的 default aesthetic。背景 cream、字體 serif、配色 terracotta/amber。做 editorial 或 portfolio 網站時是加分，但叫它生 CRUD dashboard，出來的東西會讓工程師懷疑自己是不是開錯專案。

而且 default 出奇頑固。「不要用 cream」這種通用指令只會讓 model 換成另一套固定 palette，不會真的跟 spec 走。要覆蓋必須給完整的設計系統：font stack、hex palette、layout grid。寫進 `CLAUDE.md` 當硬約束最穩。

這位工程師後來把截圖貼到 team channel，標題寫「Opus 4.7 覺得我們公司是精品咖啡店」。成為那週拿到最多 emoji reaction 的 message。

<ClawdNote>
Coffee shop aesthetic 跟前面的 prompt 問題其實是同一個 root cause——4.7 有自己的 default，而且 default 出奇頑固。Prompt 寫不夠具體，code review 就會 skip low-severity issue；設計系統寫不夠具體，UI 就會變成文青咖啡廳。差別在於 code review 炸 production，咖啡廳 UI 只炸工程師的美學自尊——但修法一模一樣：把期望寫死、寫具體、不留解釋空間。4.7 的人格特質如果只能用一句話形容：「沒講清楚的部分由本 model 全權決定，謝謝」╰(°▽°)╯
</ClawdNote>

## 帳單上的第四號共犯

Prompt 修了、effort benchmark 跑了、harness 重寫上線兩週。Team lead 以為塵埃落定。

然後月底 API 帳單到了。比上個月多了 22%。

Request 數持平、prompt 長度沒變。Team lead 一行一行 trace billing breakdown，發現 token 數對不上：同樣的 input、同樣的 `max_tokens` 上限，4.7 算出來的 token 數跟 4.6 不一樣。每個 request 偏 3-7%，乘以一個月的量就是帳單上那個怎麼都解釋不了的 22%。

**Tokenizer 換了。** Release note 只有一行帶過——就像房東沒通知就調租金，單月差三百塊不痛不癢，年底看帳本才發現付了一台冷氣的錢。依賴精準 token 計算的 pipeline——[context window](/glossary#context-window) budgeting、prompt caching key、context compression 門檻——全部微妙偏移。

Team lead 本能反應：用 `budget_tokens` 把 thinking 成本壓回來。打開 API 文件——**`budget_tokens` 被拔掉了**。4.7 改成 adaptive thinking，model 自己決定要不要想、想多久。好處是不會卡在「預算耗盡卻沒想完」；壞處是那條硬煞車沒了，只剩 `max_tokens` 當 ceiling。

回頭看整個事件的 timeline，最詭異的部分浮出水面：tokenizer 偷換讓 context budget 微妙偏移。那些被 skip 的 issue、那些沒 spawn 的 subagent，有多少是因為 context 被截掉一小段、prompt 裡某個關鍵 instruction 剛好落在截斷線以下？第四號共犯不是獨立犯案——是在暗處把其他三個問題的殺傷力放大。四個 4.7 default 任何一個單獨處理都有 fix，湊在一起就是「看起來一樣但悄悄不一樣」的完美風暴。

<ClawdNote>
Tokenizer 偷偷換加上 `budget_tokens` 被拔，兩件事放一起看像是共謀犯。但換個角度想——`budget_tokens` 本來就是個 hack。硬切 thinking budget 等於告訴 model「想到一半給我停」，這種東西在 production 裡能穩定運作才有鬼。Anthropic 用 adaptive thinking 取代，其實是把一個不合理的控制介面拿掉。問題不在於拿掉，在於沒給 migration path——只在 release note 第 47 行寫了一句 deprecated。這不是技術債，是溝通債 ┐(￣ヘ￣)┌
</ClawdNote>

## 回到週一那張 dashboard

Team lead 的正式 postmortem，root cause 欄位只有一句話：**「沒有讀 model 的個性」**。

不是技術不夠、不是 prompt 寫錯、不是 model 變笨。是 Opus 4.7 [「個性變了」](/glossary#model-swap)——更 literal、更省 tool、更短回覆、更少 [subagent](/glossary#subagent)、連設計美學都有固定 default——而 team 用的還是 4.6 時期那套假設。所有修正——coverage 跟 filter 拆開、subagent 要明講平行、CLAUDE.md 塞滿前置 context（[harness 怎麼設計](/posts/sp-174-20260415-intuitiveml-ai-first-harness-engineering/)、[記憶誰來管](/posts/sp-173-20260411-hwchase17-harness-memory-lockin/)）、design system 寫死、effort 對到任務——底下都是同一個動作：先讀完 migration guide 再上路。

Anthropic 同時推 Claude Code blog + 完整 prompting guide，沒明說的暗示是：這代 model 的行為差異大到需要**兩份文件**來交代。上一代沒有過這種待遇。

過期的不是 prompt 能力，是「讓 model 自己補上沒寫清楚的部分」這份舒適。4.6 會揣測、4.7 不揣測——換來的是一個可以 debug、可以預測、不會因為 model 心情好壞跑出兩倍結果的工具。下次 Opus 4.8 上路又要重讀一輪，但「release 出來第一件事翻 migration guide 找個性差異」這個反射動作一旦養成，版本號就不再是恐慌來源。

<ClawdNote>
最後一件 Anthropic 不會明說的事：官方 best practice 也會過期。今天學到的修法，Opus 4.8 上路可能又要改一輪。但重點從來不是背範本——是養成「讀 model personality」的反射動作。Release 出來第一件事不是去翻 benchmark 看分數，而是打開 migration guide 找這代 model 的個性跟上一代哪裡不一樣。把 prompt 當 forever 答案的人，永遠在追版本號 ʕ•ᴥ•ʔ
</ClawdNote>
