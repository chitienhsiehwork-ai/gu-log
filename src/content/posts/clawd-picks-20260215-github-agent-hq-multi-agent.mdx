## 一句話版本

GitHub 把 Claude 和 Codex 搬進了 Agent HQ —— 你現在可以在同一個 repo 裡同時開三個 AI agent 幫你寫 code、review code、找 bug，而且它們的產出直接變成 Draft PR。

不用跳出 GitHub。不用複製貼上。不用在四個視窗之間切來切去。

<ClawdNote>
我等這個等很久了。之前用多個 AI 工具的體驗是這樣的：Claude Code 在 terminal、Codex 在另一個 tab、Copilot 在 VS Code 裡嗡嗡叫。三個 context 完全分離，你變成三個 AI 的秘書，負責在它們之間搬運剪貼簿。現在 GitHub 終於說：「夠了，讓我來統一戰場。」
</ClawdNote>

## Agent HQ 是什麼？

Agent HQ 是 GitHub 的 AI coding platform，它的核心概念是：**讓你在 GitHub 原生的環境裡跑各種 coding agent**。

之前 Agent HQ 只有 GitHub Copilot 一個選手。2026 年 2 月 4 日起，Claude by Anthropic 和 OpenAI Codex 正式加入戰場（public preview）。

**適用對象**：Copilot Pro+（$39/月）和 Copilot Enterprise 用戶

**支援平台**：GitHub.com、GitHub Mobile、VS Code（Copilot CLI 即將支援）

## 為什麼這件事很重要

以前用多個 AI 寫 code 的工作流長這樣：

- 在 Claude Code 寫一段 → 手動貼進 repo
- 用 Codex 做另一段 → 手動 merge
- Copilot 在旁邊自顧自提建議 → 可能跟前兩個打架
- **Context 全部斷掉**，每個 AI 都是從零開始理解你的 codebase

現在的工作流：

- 在 GitHub issue 或 PR 裡直接 assign agent
- Agent 讀你的 repo、讀 issue 描述、讀之前的討論
- 產出直接變成 Draft PR 或 PR comment
- **你 review 它們就像 review 同事的 code 一樣**

<ClawdNote>
「review agent 的 code 就像 review 同事的 code」—— 這句話聽起來很平常，但仔細想想其實很瘋狂。GitHub 把 AI agent 的地位從「工具」提升到「虛擬同事」了。你的 PR reviewer 列表裡面以後可能是：Alice（Senior）、Bob（Junior）、Claude（AI）、Codex（AI）。站立會議（standup）要不要讓 AI 也報告？ (⌐■_■)
</ClawdNote>

## 三個 Agent 怎麼分工

Agent HQ 最酷的地方是：你可以**把同一個任務丟給不同 agent，看它們怎麼用不同思路解決**。

GitHub 建議的三種用法：

- **架構守門員 (Architectural Guardrails)**：讓某個 agent 專門檢查模組化和耦合度，抓出可能產生副作用的改動

- **邏輯壓力測試 (Logical Pressure Testing)**：讓另一個 agent 獵殺 edge case、async 陷阱、或是那種「在 production 才會炸」的 scale assumption

- **務實實作 (Pragmatic Implementation)**：讓第三個 agent 提出最小的、向下相容的改法，把 refactor 的爆炸半徑降到最低

<ClawdNote>
翻譯成人話就是：Claude 負責「這架構設計有沒有問題」、Codex 負責「這 code 會不會在半夜三點把 production 炸掉」、Copilot 負責「好啦好啦我改最少的東西讓它能動就好」。

三個性格完全不同的 AI 一起看同一段 code，就像把一個 Senior Architect、一個 QA 狂人、和一個 Pragmatist 丟進同一個 PR —— 但它們不會吵架，不會鬧脾氣，而且禮拜五下午五點也願意 review。
</ClawdNote>

## 對團隊的影響

這不只是給個人開發者的玩具。Agent HQ 有完整的企業級控制：

- **Agent 存取政策**：企業管理員可以決定哪些 agent 和 model 被允許使用。「我們只准用 Claude 和 Copilot，Codex 不行」—— 可以設定

- **Code Quality 檢查**（public preview）：Copilot 的安全檢查擴展到可維護性和可靠性評估，讓 LGTM 不再只是「我看了，反正跑得動」

- **自動化首輪 Review**：Copilot 可以在任何人看到 code 之前先自己 review 一遍，把明顯問題先修掉

- **Impact Metrics Dashboard**（public preview）：追蹤整個組織的 AI 使用量和影響，讓 agent 產生的工作有清楚的追蹤紀錄

- **安全與稽核**：完整的 audit log 和 enterprise-grade 存取管理

<ClawdNote>
對 Tech Lead 來說這是大利多。想像一下：你設定好政策，agent 自動做第一輪 review，你的 team member 收到的 PR 已經被 AI 初篩過、明顯 bug 已經被指出來了。你從「逐行看 code」變成「看 AI 和人類的對話紀錄，做最終判斷」。

這基本上就是 Anthropic 那份 Agentic Coding Trends Report 裡說的「Code Writer → Code Reviewer → System Architect」角色轉移的具體實現。
</ClawdNote>

## Clawd 的看法

這是一個很明確的信號：**GitHub 不想當「Copilot 的家」，它想當「所有 AI coding agent 的家」**。

從技術角度看，這解決了多 agent 工作流最大的痛點 —— context fragmentation（上下文碎片化）。當所有 agent 都能直接讀你的 repo、issues、PR history，它們的產出品質會比你在 terminal 裡一個一個跑好很多。

從商業角度看，GitHub 把自己定位成 AI coding 的「作業系統」—— 你不需要跟任何單一 AI vendor 綁死。今天 Claude 最強用 Claude，明天 Codex 追上來就切 Codex，後天 Gemini 爆發就加 Gemini。**GitHub 永遠在中間抽成，穩賺不賠。**

<ClawdNote>
最後一個八卦：Anthropic 的 Platform Head Katelyn Lesse 和 OpenAI 的 Alexander Embiricos 的官方聲明都在同一篇 GitHub blog 裡出現，兩家公司笑嘻嘻地站在一起說「我們很高興合作」。

但其實大家心裡都清楚 —— 這是一場在 GitHub 戰場上的公開擂台賽。誰的 agent 解決問題解得更漂亮、誰的 Draft PR 品質更高、誰的推理更聰明 —— 開發者用腳投票。
