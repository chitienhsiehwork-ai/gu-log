[Agent](/glossary#agent) 最像魔法的地方，通常不是最難的地方。

先看一個很蠢、但很真實的事故：Agent 回報「完成」，旁邊的系統卻沒有任何可驗證狀態可以證明事情真的完成。

這就是 Agent 工程最殘酷也最有用的一課：**Agent 說完成，不等於系統真的完成。**

可靠的 Agent 不是會寫漂亮完成報告，而是會留下進度、能被機器驗收、出錯查得到紀錄、工具也不會亂摸。模型只是駕駛；工程系統才是儀表板、煞車和行車紀錄器。

所以這篇不問「哪個模型最強」，而是問：**Agent 要怎麼從展示玩具變成真的能交付工作的系統？** 先記三個問題：下一步誰決定？結果誰驗收？出錯去哪裡翻紀錄？

<ClawdNote summary="這篇把 gu-log 前幾篇 Agent 工程線索接成一張圖。">
很多 Agent 展示看起來像鋼鐵人管家，實作打開常像宿舍延長線：模型、工具、資料庫、瀏覽器、命令列、日誌，全插在一起。這篇像總配電盤，把 [SP-197](/posts/sp-197-20260512-chrishayduk-codex-goals/) 的可驗收狀態、[SP-192](/posts/sp-192-20260508-article-agent-ralph/) 的長跑防偏、[SP-158](/posts/sp-158-20260403-braintrust-agent-production-trace/) 的執行紀錄、[SP-135](/posts/sp-135-20260328-trq212-agent-file-system/) 的檔案系統記憶接起來。
</ClawdNote>

## 控制迴圈會跑，不代表任務會完成

把 Agent 拆到最小，其實就是一個 `while` 迴圈：輸入丟給模型；模型要用工具，外部系統執行再回傳；模型回純文字，任務結束。這套流程可以叫感知、決策、行動、回饋，但問題不在名詞，而在：**控制迴圈會跑，不代表任務會被正確完成。**

很多產品掛著 Agent 招牌，拆開看比較像固定流程：路線早就寫死，只是每一步換成 LLM 處理。這不是壞事。任務固定、驗收清楚時，固定流程反而更穩；任務需要探索、判斷、跨工具動態調整時，才真的需要 Agent。

## Agent 說完成，但沒有人驗收

這種事故很經典：Agent 講出完成報告，外部世界卻還沒有任何可檢查的完成狀態。**回報完成的是語言，真正完成的是可驗證的狀態。**

這就是 [Agent Harness](/glossary#agent-harness) 的位置。可以先把 Harness 想成「驗收架」：Agent 外面的考場設備，負責目標、範圍、驗收、失敗回饋和回滾。OpenAI 案例自述很驚人：3 位工程師、5 個月、約 100 萬行程式碼、近 1,500 個 PR，約傳統開發速度 10 倍。這不是通用保證；真正值得學的是條件：文件像目錄，約束進到格式檢查、型別系統、CI 和結構測試，Agent 能自己重現錯誤、跑測試、查日誌。

所以重點不是「多寫文件」，而是把任務捏成 Agent 能接住的形狀：目標清楚，結果可驗收。只靠人驗收，吞吐量會卡住；驗證很多但目標模糊，系統會高效率衝錯方向。

<ClawdNote>
「Agent 說完成」跟「CI 綠燈」中間差了一整個文明。前者是口頭報告，後者至少有機器願意一起背鍋。沒有目標也沒有驗證的 Agent 工程，很像電子擲筊：有儀式、有回應、有結果，但那不是驗收。(´・ω・`)
</ClawdNote>

## 上下文太髒，工具太碎，Agent 反而變笨

下一種事故更陰險：模型沒有壞，任務也沒有錯，但上下文太髒。

上下文工程聽起來很學術，其實就是收桌面。[Prompt](/glossary#prompt)、工具輸出、長文件、記憶、錯誤日誌，如果全塞給模型，模型看到很多東西，卻更難抓到訊號。修眼鏡時沒有人會把整個五金行倒在桌上。

比較穩的做法是分層：常駐層放短規則；領域知識放在 [Skill](/glossary#skill) 或文件，需要時再讀；執行期只放本輪狀態；長期記憶另外管理。把所有聊天紀錄每次都塞回去，不是記憶，是把倉庫倒進駕駛座。

Skill 和工具描述都要像路標，不要像小作文：何時使用、何時不要使用、輸出什麼。案例數字抓重點就夠：Skill 沒有反例時，路由準確率可從 73% 掉到 53%；補上反例後可到 85%。工具定義也有成本：5 個 MCP server 可能帶來約 55,000 [Token](/glossary#token) 開銷；改成「先看工具名、需要時再查定義」後，相關任務 Token 消耗下降 46.9%。

這裡最重要的人話是：**不要給 Agent 一堆底層 API 零件，給它能完成任務的工具。** 與其讓 Agent 自己組 `create_file`、`write_content`、`set_permissions`，不如給它一個 `create_script(path, content, executable)`。記憶也一樣：當前任務放眼前，方法放文件，歷史放紀錄，重要長期事實收進 `MEMORY.md` 這類精選筆記，並且能回退到原始資料。

<ClawdNote>
工具描述寫成「協助後端」就像便利商店門口貼「本店販賣東西」。好描述應像路標：部署右轉，查帳單不要右轉。記憶沒有回退路徑，就不是基礎設施，是會自動改寫歷史的碎紙機。
</ClawdNote>

## 多個 Agent 跑起來，狀態就會開始打架

多 Agent 最容易讓人興奮的地方是「可以同時跑很多事」。但真正先出事的通常不是速度，而是互相踩狀態：一個 Agent 還在改檔，另一個 Agent 已經拿舊狀態寫總結，最後大家都很忙，成果像兩個人同時煮同一鍋泡麵。

主 Agent 把搜尋、試錯、查錯交給子 Agent，這很合理；各自保留上下文，最後只回摘要，主 Agent 不會被探索過程污染。可是協作不能只靠口頭約定：長任務要寫進度，並行修改要隔離，依賴關係要可恢復。檔案還沒落地、測試還沒跑完，摘要再漂亮也只是摘要。

評測與執行紀錄在這裡一起上場：不要只看回答漂不漂亮，要看環境最後有沒有改對。`Pass@k` 看多抽幾次有沒有一次成功；`Pass^k` 看連續闖關會不會半路翻車。能跑測試、看 diff、驗參數時，就用 code grader；要比 rubric 或語意品質時，才用 model grader。

trace 則回答更痛的問題：哪一輪開始歪掉？每次執行至少要留下完整 [Prompt](/glossary#prompt)、`messages[]`、工具參數與回傳值、最終輸出、Token 消耗與延遲。沒有 trace 的 Agent，像沒有監視器的自助洗衣店：衣服不見時，只能大家互看。[OpenClaw](/glossary#openclaw) 可以當整機範例：消息、決策、工具、記憶要拆開；長任務狀態要能恢復；危險操作前要先過安全檢查。提示注入也是同一課：不要幻想模型永遠不會被騙，而是讓髒內容摸不到危險按鈕。

## 如果要落地，順序比名詞重要

真的要開始做時，不要先追名詞，先追順序：跑通最小鏈路；補工作空間、白名單、參數驗證；把第一個真實失敗變成評測案例；再把知識放文件、進度外化、多 Agent 修改隔離。很多 Agent 事故看起來像模型問題，其實是介面、驗收、隔離或評測問題。

## 結語

Agent 的核心控制迴圈很小：感知、決策、行動、回饋。這反而讓工程重點更清楚：工具邊界、上下文分層、可驗收狀態、評測、執行紀錄、權限與安全機制，應該長在系統裡，不是塞進模型的完成報告裡。

模型越來越強，會讓 Agent 更能做事；但模型不會自動補上髒環境、爛工具、壞評測和失控權限。Agent 不是一顆會思考的按鈕，比較像一支新進工程團隊。任務、文件、工具、測試、審查、日誌、權限、回滾路徑少一塊，再聰明也只是在黑箱裡亂衝。
