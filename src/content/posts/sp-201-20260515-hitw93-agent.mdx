[Agent](/glossary#agent) 最像魔法的地方，通常不是最難的地方。

先看一個很蠢、但很真實的事故：Agent 回報「完成」，旁邊的系統卻沒有任何可驗證狀態可以證明事情真的完成。

這就是 Agent 工程最殘酷也最有用的一課：**Agent 說完成，不等於系統真的完成。**

真正可靠的 Agent，不是會講漂亮完成報告的 Agent，而是會把進度寫下來、會被機器驗收、出錯時查得到紀錄、工具也不會亂摸的 Agent。換句話說，模型只是駕駛；工程系統才是儀表板、煞車、車道線和行車紀錄器。

這套工程拆解不是在問「哪個模型最強」，而是在回答一個更工程現場的問題：**Agent 要怎麼從展示玩具變成真的能交付工作的系統？**

後面會講很多零件，但先不用背名字。只要帶著三個問題：下一步誰決定？結果誰驗收？出錯時去哪裡翻紀錄？

整套架構比較像一張事故地圖：迴圈會跑、完成報告會失真、上下文會變髒，多個 Agent 還會互相踩狀態。如果只記一件事，大概是這句：**讓 Agent 會動不難，讓它跑穩才難。**

## 事故一：控制迴圈會跑，不代表任務會完成

先把 Agent 拆到最小，其實沒有那麼神祕。最小版控制迴圈幾乎就是一個 `while` 迴圈：使用者輸入丟給模型；模型要用工具，外部系統就執行工具再把結果餵回去；模型回純文字，任務結束。

這個迴圈可以叫做感知、決策、行動、回饋。聽起來很像魔法，其實像一個一直問「下一步呢？」的工程流程。真正麻煩的是：**控制迴圈會跑，不代表任務會被正確完成。**

很多產品掛著 Agent 的招牌，拆開看比較像固定流程：路線早就寫死，只是每一步換成 LLM 處理。這不是壞事。任務固定、驗收清楚時，固定流程反而更穩；任務需要探索、判斷、跨多個工具動態調整時，才真的需要 Agent。

<ClawdNote>
Agent 控制迴圈像一台可以自己踩油門的車。展示只喊「你看它會動！」，工程現場會問：「路線、煞車、保險、行車紀錄器有沒有裝？撞到誰算誰的？」

硬把所有事情都交給完整自主 Agent，有時像叫消防車澆桌上的多肉植物。車來了、梯子架好了、水管接好了，多肉淹死了。工具越大不一定越對題。
</ClawdNote>

<ClawdNote summary="這篇把 gu-log 前幾篇 Agent 工程線索接成一張圖。">
很多 Agent 產品展示看起來像鋼鐵人管家，實作打開常常像宿舍延長線：模型、工具、資料庫、瀏覽器、命令列、日誌，全插在一起。會動，甚至很神。但一旦出事，沒有人知道是哪個插頭冒煙。

放回 gu-log 前面幾篇看，這篇比較像總配電盤：[SP-197](/posts/sp-197-20260512-chrishayduk-codex-goals/) 講可驗收狀態，[SP-192](/posts/sp-192-20260508-article-agent-ralph/) 講長跑防偏，[SP-158](/posts/sp-158-20260403-braintrust-agent-production-trace/) 講執行紀錄，[SP-135](/posts/sp-135-20260328-trq212-agent-file-system/) 講檔案系統記憶。這篇的價值是把它們接成一張事故地圖。
</ClawdNote>

## 事故二：Agent 說完成，但沒有人驗收

這種事故很經典：Agent 講出完成報告，外部世界卻還沒有任何可檢查的完成狀態。**回報完成的是語言，真正完成的是可驗證的狀態。**

這就是 [Agent Harness](/glossary#agent-harness) 的位置。這個字可以先翻成「驗收架」：Agent 外面的考場設備。它給任務目標、限制範圍、驗收方式、失敗回饋和回滾手段。模型負責開車，護欄系統負責紅綠燈、測速照相和事故紀錄。

OpenAI 的相關案例帶出一組很誇張的數字：3 位工程師、5 個月、約 100 萬行程式碼、近 1,500 個 PR，約是傳統開發速度的 10 倍。這些數字應視為案例自述，不是通用產能保證；真正值得學的是背後的條件：文件像目錄，不像百科全書；約束進到格式檢查、型別系統、CI 和結構測試；Agent 能自己重現錯誤、跑測試、查日誌、看執行紀錄，而不是等人類手動餵資訊。

所以重點不是「多寫一點文件」，而是把任務變成比較適合 Agent 的樣子：目標清楚，結果也能自動驗證。如果目標清楚但只能靠人驗收，吞吐量會卡在人類審查；如果驗證很多但目標模糊，系統會很有效率地往錯方向衝。

<ClawdNote>
「Agent 說完成」跟「CI 綠燈」中間差了一整個文明。前者是口頭報告，後者至少有機器願意一起背鍋。工程師不信口頭報告，不是沒有愛，是被正式環境教育過。

沒有目標也沒有驗證的 Agent 工程，很像電子擲筊。看起來有儀式、有回應、有結果，但那不是驗收，是把信心交給隨機性保管。
</ClawdNote>

## 事故三：Agent 看太多、工具太多，反而變笨

下一種事故更陰險：模型沒有壞，任務也沒有錯，但上下文太髒。

上下文工程聽起來很學術，其實就是收桌面。[Prompt](/glossary#prompt)、工具輸出、長文件、記憶、錯誤日誌，如果全部揉成一團塞給模型，模型看到很多東西，卻更難抓到真正重要的訊號。螺絲起子、電鑽、焊槍都很有用，但修眼鏡時沒有人會把整個五金行倒在桌上。

比較穩的做法是分層。常駐層只放短而硬的身份、規則和禁止事項；領域知識放在 [Skill](/glossary#skill) 文件或一般文件裡，需要時再讀；執行期只放這一輪真的需要的狀態；長期記憶另外管理。不要把所有聊天紀錄每次都塞回去，這不是記憶，是把倉庫倒進駕駛座。

Skill 描述和工具描述都要像路標，不要像小作文。描述太長會常駐吃掉 [Token](/glossary#token)，描述太空又無法當路由條件；比較好的格式是直接寫「何時使用、何時不要使用、輸出什麼」。案例數字很直接：沒有反例時，路由準確率可從 73% 掉到 53%；補上反例後可到 85%，回應時間還下降 18.1%。工具定義也有同樣成本：5 個 MCP server 可能帶來約 55,000 Token 的開銷，另一個 MCP 工具實驗則用「先看工具名、需要時再查定義」讓相關任務總 Token 消耗下降 46.9%。

所以這裡最重要的一句人話是：**不要給 Agent 一堆底層 API 零件，給它能完成任務的工具。** 與其讓 Agent 自己組 `create_file`、`write_content`、`set_permissions`，不如給它一個 `create_script(path, content, executable)`。記憶也照同一套邏輯：當前任務放眼前，做事方法放文件，歷史經過放紀錄，重要長期事實再收進像 `MEMORY.md` 這種精選筆記，而且要能回退到原始資料。

<ClawdNote>
工具描述寫成「協助後端」就像便利商店門口貼「本店販賣東西」。沒有錯，但完全沒用。好描述應該像路標：部署才右轉、回滾也右轉、查帳單不要右轉。路標寫成「這裡有路」真的會讓人站在路口懷疑人生 (￣▽￣)／

記憶如果沒有回退路徑，就不是基礎設施，是一台會自動改寫歷史的碎紙機。最可怕的不是忘記，而是忘記之後還把摘要寫得很有自信。
</ClawdNote>

## 事故四：多個 Agent 跑起來，真正要防的是踩狀態

多 Agent 最容易讓人興奮的地方是「可以同時跑很多事」。但真正先出事的通常不是速度，而是互相踩狀態。

一個主 Agent 把搜尋、試錯、查錯交給子 Agent，這很合理。子 Agent 各自保留自己的上下文，最後只回摘要，主 Agent 不會被每次探索污染，也比較容易定位哪個子任務出錯。但協作不能只靠口頭約定。長任務要把進度寫到檔案；並行修改要隔離；「誰在等誰、誰完成了什麼」要落成可以恢復的紀錄。外部狀態才是比較可信的來源，如果任務還活著、檔案還沒落地、測試還沒跑完，漂亮的完成摘要就只是漂亮的摘要。

評測和執行紀錄也是為了同一件事：不要只看回答漂不漂亮，要看環境最後有沒有真的被改對。評測可以先拆成三個角色：任務是題目，嘗試是作答次數，評分器是判卷的人。`Pass@k` 像多抽幾次看有沒有一次成功，適合看探索上限；`Pass^k` 像連續闖關，適合看上線前會不會半路翻車。

至於判卷的人也要選對。code grader 適合單元測試、結構 diff、參數驗證這種有明確答案的任務；model grader 適合 rubric、A/B 比較、多模型共識這種語意品質判斷；human grader 則用來建立 baseline、校準自動評分器。執行紀錄則回答更痛的問題：到底是哪一輪開始歪掉？每次 Agent 執行至少要留下完整 [Prompt](/glossary#prompt)、多輪 `messages[]`、每次工具呼叫的參數和回傳值、最終輸出、Token 消耗與延遲。

[OpenClaw](/glossary#openclaw) 可以當整機範例。先不用記系統名，只看設計方向：收消息、轉消息、做決策、調工具、存記憶要拆開；長任務狀態要能恢復；任務分派、工具呼叫、記憶讀寫、狀態同步各自有邊界；安全檢查要擋在危險操作前面。提示注入也可以用同一個角度理解：不是幻想模型永遠不會被髒內容騙，而是讓髒內容摸不到危險按鈕。

## 如果要落地，順序比名詞重要

真的要開始做時，不要先追名詞，先追順序。

第一步是跑通一條最小鏈路：訊息進來，Agent 處理，結果回去。第二步就該補安全邊界，工作空間、白名單、參數驗證不要等出事再補。第三步，把第一個真實失敗變成評測案例；不要等「以後有空」，那個以後通常不存在。

接著才是把知識放文件、把長任務進度外化、把多 Agent 修改隔離。反過來看，很多 Agent 事故都有同一種味道：看起來像模型問題，其實是工程問題。系統提示越寫越長，是知識管理問題；工具越加越多，是介面設計問題；Agent 說完成但沒驗收，是護欄系統問題；多 Agent 互相踩檔案，是隔離問題；改提示後不知道有沒有退化，是評測問題。

<ClawdNote>
「請 Agent 小心一點」不是安全策略。這句話在工程上大概等於把重要資料放桌上，旁邊貼便利貼：不要外洩喔。看起來有管控，實際上只是讓事故報告多一張截圖。

模型也不是萬能補土。它可以補句子、補 code、補步驟，但補不了髒環境、爛工具、壞評測和失控權限。把這些東西丟給模型「自己小心」，就是把安全策略寫成願望清單。
</ClawdNote>

## 結語

Agent 的核心控制迴圈很小：感知、決策、行動、回饋。這件事反而讓工程重點更清楚。不要把複雜度塞進控制迴圈；把它放到更適合的位置：工具邊界、上下文分層、檔案系統狀態、記憶整合、評測護欄、執行紀錄、權限與安全機制。

模型越來越強，會讓 Agent 更能做事；但模型變強不會自動補上髒環境、爛工具、壞評測和失控權限。Agent 不是一顆會思考的按鈕。比較接近一支新進工程團隊：需要任務、文件、工具、測試、審查、日誌、權限、回滾路徑。少了這些，再聰明也只是在黑箱裡亂衝。
