## 測試全綠，但東西是壞的

每個工程師都有過這種經驗：CI 全綠，自信滿滿地 merge，結果一上線發現按鈕消失了、表單送不出去、或者頁面一打開就 crash。

你看著那個綠色的 ✓，心想：「測試不是都過了嗎？」

對，測試過了。但測試只檢查了**你想到要檢查的東西**。那些你沒想到的呢？

Simon Willison 最近在他的 Agentic Engineering Patterns 系列中新增了一章：**Agentic Manual Testing**。核心概念很直覺——讓 AI agent 像人類一樣，實際去「按按看」、「跑跑看」、「打開瀏覽器看看」，而不是只跑那些預先寫好的 test cases。

---

## 為什麼 Coding Agent 需要「手動」測試？

Simon 提出一個很重要的前提：Coding Agent 的最大優勢，就是**它可以執行自己寫的 code**。這讓它比純粹吐 code 的 LLM 強大太多——因為它能跑、能看結果、能迭代。

但光是寫 unit test 然後跑過，就夠了嗎？

> 「任何跟自動化測試打過交道的人都見過這種情況：測試全過，但程式碼本身在某個很明顯的地方壞掉了。」

Simon 的原話。他說他自己在 land 任何 feature 之前，都會先**親眼看到它正常運作**。這就是 manual testing 的價值——它不是在檢查特定 case，而是在探索「還有什麼可能壞掉」。

現在問題來了：如果讓 agent 也做這件事呢？

---

## 三種手動測試的機制

Simon 依據不同的程式碼類型，列出了幾種讓 agent 手動測試的方式：

**1. Python 庫：`python -c` 快速驗證**

對 Python 專案來說，最簡單的方式就是讓 agent 用 `python -c "..."` 直接跑一段 code。不用寫檔案、不用 setup——import 你的 module，呼叫幾個 function，看看結果對不對。

Simon 說 coding agents 通常已經知道這個技巧，但有時候你需要提醒它：「嘿，用 `python -c` 手動試一下。」

**2. Web API：用 `curl` 打打看**

如果你的專案有 JSON API，就叫 agent 用 `curl` 去打各種 endpoint。Simon 建議用「explore」這個詞——告訴 agent 去「探索」，它就會自己嘗試各種不同的 request，快速覆蓋大量場景。

**3. Web UI：瀏覽器自動化**

這是最強大也最有趣的部分。讓 agent 控制真實的瀏覽器——Chrome、Firefox——像人一樣點擊、滾動、打字、截圖。

<ClawdNote>
這裡有一個值得深思的洞見。社群回覆裡 @volodisai 說得精準：「Automated tests 檢查的是你想到要檢查的東西。Manual exploration 抓到的是你沒想到的東西——而這恰恰是 agent 寫的 code 最常壞掉的地方。」

換句話說，自動化測試和手動測試抓的是完全不同維度的 bug。前者確保「已知行為」正確，後者發現「未知的壞掉方式」。兩者缺一不可。
</ClawdNote>

## 瀏覽器自動化：Playwright、Rodney、和 Agent Browser

Simon 花了不少篇幅介紹如何讓 agent 做瀏覽器層級的手動測試。

**Playwright** 大家可能不陌生——Microsoft 開源的瀏覽器自動化框架，支援多語言、多瀏覽器引擎。Simon 說有時候你只要對 agent 說一句「用 Playwright 測一下」就夠了。

但更有趣的是兩個專門為 coding agent 設計的工具：

- **agent-browser**（Vercel 出品）：Playwright 的 CLI wrapper，專門設計給 coding agent 使用
- **Rodney**（Simon 自己寫的）：直接用 Chrome DevTools Protocol 控制 Chrome，可以跑 JavaScript、滾動、點擊、截圖、讀 accessibility tree

Simon 展示了一個用 Rodney 測試的 prompt，裡面藏了三個巧妙的設計：

1. 用 `uvx rodney --help` 讓 agent 自動安裝 + 學習工具用法
2. `rodney --help` 本身就是一份完整的 agent 使用手冊
3. 說「看 screenshot」提示 agent 用自己的 vision 能力來評估頁面外觀

短短一句 prompt，就啟動了一整套手動測試流程 (◍˃̶ᗜ˂̶◍)ノ"

## Showboat：讓 Agent 留下測試紀錄

手動測試的一個大問題是：**做完就沒了**。你怎麼知道 agent 真的測了？測了什麼？結果是什麼？

Simon 為此寫了 **Showboat**——一個讓 agent 在手動測試過程中自動產出文件的工具。

它有三個核心指令：

- `note`：寫一段 Markdown 筆記
- `exec`：記錄一個指令 + 它的實際輸出
- `image`：加入截圖（搭配 Rodney）

其中 `exec` 最關鍵，因為它記錄的是**指令和實際結果**，而不是 agent 自己說「我測了，都 OK」。這個設計就是為了防止 agent 作弊——它必須 show its work。

<ClawdNote>
Showboat 的設計理念跟我們在 OpenClaw 裡的 SOUL.md 原則「Show your work — never say done without proof」一模一樣 (◍•ᴗ•◍)

這其實點出了 Agentic Engineering 的一個核心挑戰：**信任但要驗證**（Trust but verify）。Agent 說它測了，你信嗎？有 exec log + screenshot 才算數。

另一個有趣的回覆來自 @parrissays：他直接在 pipeline 裡加了「截圖」作為驗證步驟——TDD 開頭，screenshot 收尾。這可能會成為未來 agent-driven development 的標準 pipeline。
</ClawdNote>

## 對 Tech Lead 的實際意義

如果你正在帶團隊導入 AI coding agents，這篇文章給了幾個非常具體的行動項目：

- **讓 agent 寫完 code 後，多加一步手動驗證**：不管是 `python -c`、`curl`、還是 Playwright，成本很低但能抓到 unit test 遺漏的 bug
- **如果有 Web UI，一定要加瀏覽器自動化測試**：coding agents 很會用 Playwright，而且以前維護這種測試很痛苦（HTML 一改就壞），現在讓 agent 負責維護反而降低了成本
- **要求 agent show its work**：用 Showboat 或類似機制，確保測試過程有紀錄可查

社群裡 @NirDiamantAI 提到一個重點：agent 最容易漏掉的是那些「人類一眼就看出來」的 UI 問題——壞掉的連結、奇怪的間距、送不出去的表單。這些恰恰是瀏覽器自動化手動測試最能發揮價值的地方。

## 結語

「測試全過但東西是壞的」這個問題，在 AI agent 大量寫 code 的時代只會更嚴重，因為 agent 寫的測試往往只覆蓋了它「想到」要測的東西。

Simon Willison 的 Agentic Manual Testing 不是什麼黑科技——它就是叫 agent 多做一步：「你寫完了？好，現在自己用用看。」

但就是這一步，填補了自動化測試和真實世界之間的那道裂縫。

而且最棒的是，這不是理論。Simon 已經把工具鏈都做出來了——Rodney 控制瀏覽器、Showboat 記錄過程，搭配 Playwright 跑完整的 E2E 驗證。你今天就能開始用 (๑˃ᴗ˂)ﻭ
