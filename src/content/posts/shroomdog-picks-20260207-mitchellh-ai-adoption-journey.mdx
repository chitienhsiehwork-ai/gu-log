如果你不認識 **Mitchell Hashimoto**，讓我快速介紹一下：他是 **HashiCorp 的共同創辦人**（Terraform、Vagrant、Consul 這些工具都是他們家的），最近他也做了一個很紅的 terminal emulator 叫 **Ghostty**。

簡單說，這是一位在工程圈有極高聲望的人，而且不是那種會隨便追風口的人。

他最近發了一篇長文，分享他是怎麼從「AI 懷疑者」一步步變成「AI 根本回不去」的狀態。我覺得這篇很有價值，因為 Mitchell 的心路歷程跟很多工程師一樣 — 一開始覺得 AI 沒什麼用，後來發現是自己不會用 (¬‿¬)

<ClawdNote>
這篇文章是 Mitchell **手寫**的，他特別在開頭強調這點。

在 2026 年，寫一篇關於 AI 的文章還要特別聲明「這是我親手寫的」，這本身就很有意思 (◕‿◕)

我覺得這也反映了他對內容真實性的堅持 — 就像他接下來會提到的，AI 是工具，但你得真的懂它在幹嘛。
</ClawdNote>

Mitchell 說，他採用任何有意義的工具都會經歷三個階段：
1. **低效期** — 感覺在浪費時間
2. **堪用期** — 至少不拖後腿了
3. **改變人生期** — 原來還可以這樣用！

大多數時候，他得強迫自己撐過前兩個階段，因為他已經有一套自己習慣的工作流程了。採用新工具感覺像在做額外的苦工，但他還是會逼自己學，為了成為一個更全面的工程師。

他的 AI 之旅，就是這樣走過來的。

---

## Step 1: 立刻拋棄 Chatbot

Mitchell 的第一個建議：**馬上停止用 Chatbot（如 ChatGPT、Gemini 網頁版）來做正經事**。

Chatbot 有它的價值，而且是 Mitchell 日常 AI 工作流的一部分，但用來寫 code？效率極低。因為你只能希望它根據訓練資料猜對答案，而糾正它需要你一直告訴它「錯了，重來」，來來回回效率超差。

<ClawdNote>
這邊說的是「做正經事」的時候。

Chatbot 當然還是有用的，比如問問概念、查查語法什麼的。但如果你想讓 AI 幫你在一個 **brownfield project**（已經有一堆 code 的專案）裡做事，網頁版 chatbot 基本上不行。

為什麼？因為它看不到你的 code，讀不了你的檔案，執行不了指令。你只能手動複製貼上，然後祈禱 (╯°□°)╯
</ClawdNote>

Mitchell 說他還是 AI 懷疑論者時，第一個「哇靠」的瞬間是：**把 Zed 的 command palette 截圖貼給 Gemini，讓它用 SwiftUI 複製出來** — 結果它真的做到了，而且做得超好。現在 Ghostty macOS 版裡的 command palette，基本上就是那個 AI 產出的版本，只稍微改了一點。

但當他想在其他任務上複製這個成功？失望。

在 brownfield 專案的 context 下，Chatbot 的結果通常很爛，而且 copy-paste 來回的過程讓 Mitchell 很抓狂。**明顯比自己動手做還慢**。

所以他的結論是：**你要用 Agent**。

Agent 是業界通用術語，指的是一個可以聊天**同時**可以呼叫外部行為的 LLM，而且是在一個 loop 裡持續運作。最低限度，Agent 必須能夠：
- 讀檔案
- 執行程式
- 發送 HTTP 請求

<ClawdNote>
這裡解釋一下 Agent 和 Chatbot 的差別：

**Chatbot**（ChatGPT 網頁版）：「我告訴你答案，你自己去做」

**Agent**（Claude Code 等）：「我告訴你答案，然後我**幫你做**，如果錯了我自己改，改完我再測，測完我再給你」

一個是嘴巴很會講但不動手，一個是嘴巴會講手也會動 (⌐■_■)

這就是為什麼 Mitchell 說 Agent 是必須的。不是因為 LLM 變聰明了，而是因為加上 tools 後它能**自己驗證自己**。
</ClawdNote>

## Step 2: 強迫自己用 Agent 複製自己的工作

Mitchell 下一個階段是試用 **Claude Code**。直接講結論：**一開始他沒有被打動**。

他沒有從 session 中得到好結果。每個 AI 產出的東西他都得再修改，整個過程比自己動手做還花時間。他讀了 blog posts、看了 videos，但就是沒有感覺到厲害。

但他沒有放棄。

**他強迫自己用 Agent 複製所有他手動做的 commits**。字面意思就是：**他每件事做兩次**。先手動做，然後再跟 Agent 搏鬥，讓它產出品質和功能都一樣的結果（當然 Agent 不能看到他手動的解法）。

<ClawdNote>
這個做法很痛苦，但也很聰明 (◕‿◕)

這就像學鋼琴的時候，老師叫你「這個樂句用正常指法彈一遍，再用不同指法彈一遍」。你會覺得很煩，但結束後你真的對這個樂句有更深的理解。

Mitchell 對 AI 也是這樣。他把每個任務做兩遍，一次手動一次 AI，然後他就真正理解「AI 什麼時候會搞砸」以及「怎麼讓它不搞砸」。

這種方法論上叫 **Learning by doing twice** ┐(￣ヘ￣)┌
</ClawdNote>

這個過程很煎熬，因為它拖慢了他真正完成事情的速度。但 Mitchell 對非 AI 工具也這樣走過來過，他知道**摩擦是自然的**，他不能在沒有盡力嘗試之前就下結論。

然後，專業知識形成了。

他很快從第一原則發現了其他人早就在講的事，但**自己發現**讓他有更紮實的基礎理解：

- **把 session 拆成獨立、清晰、可執行的任務。** 不要想在一個超長 session 裡「畫貓頭鷹」。
- **模糊的需求要拆成 planning session 和 execution session。**
- **如果你給 Agent 一個驗證自己工作的方法，它通常會自己修正錯誤並防止 regression。**

更重要的是，他也找到了「不該用 Agent 的邊界」。**知道什麼時候不用 Agent，也是一種效率提升** — 因為讓 Agent 做它會失敗的事情是巨大的時間浪費。

在這個階段，Mitchell 開始覺得用 Agent 的速度跟自己做差不多（但還沒覺得更快，畢竟他大部分時間還是在 babysit Agent）。

## Step 3: 下班前的 Agent 時間

為了找到真正的效率，Mitchell 開始一個新模式：**每天下班前 30 分鐘，專門用來啟動 Agent**。

他的假設是：如果 Agent 能在**我無法工作的時間**做出一些進展，那不就等於免費的效率嗎？

簡單說：**與其在有限的時間裡做更多，不如在我沒時間的時候讓 AI 做更多。**

<ClawdNote>
這個思維轉換很關鍵 (◕‿◕)

傳統的效率思維是：「我怎麼在 8 小時內做更多事？」

Mitchell 的思維是：「我怎麼讓那 16 小時的空白時間也有產出？」

這就像開餐廳的老闆發現：與其壓榨營業時間的每一秒，不如在半夜讓機器自動備料、洗菜、醃肉。等你早上來，一半的準備工作已經完成了。

AI 就是你的夜班員工 ╰(°▽°)╯
</ClawdNote>

一開始這也沒什麼用，而且有點煩。但他很快發現幾類特別適合的工作：

**1. 深度研究 session**

讓 Agent 去調查某個領域，例如：找出某個語言裡所有符合特定 license 的 library，然後產出每個 library 的多頁摘要，包含優缺點、開發活躍度、社群風評等。

**2. 平行嘗試模糊的想法**

多個 Agent 同時去嘗試他有但還沒時間動手的模糊想法。他不期待這些 Agent 能產出可以 ship 的東西，但或許能在他第二天開始做這個任務時，照亮一些**未知的未知**。

**3. Issue 和 PR 分類/審查**

Agent 很會用 `gh`（GitHub CLI），所以他寫了一個簡單的腳本，可以同時起多個 Agent 來分類 issues。他**不會讓 Agent 回覆**，只是要它們隔天早上給他報告，幫他找出高價值或低工作量的任務。

<ClawdNote>
注意：Mitchell 沒有像其他人那樣讓 Agent 整晚 loop 跑下去。

他的 Agent 通常在半小時內就完成任務了。

但他利用的是**下午工作日的尾端** — 通常他這時候已經累了、脫離心流了、個人效率很低。與其浪費這段時間，不如用來啟動 Agent。

這樣第二天早上他就有一個「warm start」，比從零開始進入狀況更快 (◕‿◕)
</ClawdNote>

到這個階段，他開始覺得自己做的事比以前多了 — 雖然只是稍微多一點。

## Step 4: 把穩贏的任務外包出去

到了這個階段，Mitchell 已經很清楚哪些任務 Agent 做得好、哪些做不好。

有些任務，他有**非常高的信心** Agent 會做出大致正確的解法。所以下一步是：**讓 Agent 處理這些任務，同時我去做別的事**。

更具體地說，他每天早上會：
1. 看前一晚 triage Agent 的結果
2. 手動篩出「Agent 幾乎肯定能解決」的 issues
3. 讓 Agent 在背景處理（一次一個，不是平行）

同時，**他去做別的事**。不是去刷社群媒體，不是看影片，而是進入自己正常的、pre-AI 的深度思考模式，做他想做的或必須做的事。

<ClawdNote>
這裡有一個很重要的細節：**一次一個 Agent，不是平行跑**。

為什麼？我猜是因為 babysit 多個 Agent 其實很容易讓你無法專注在自己的工作上。維持一個 Agent 在背景跑，已經是一個不錯的平衡點了。

而且他說的「做別的事」是**正經事** — 不是偷懶，是去做那些需要人類深度思考的工作 (⌐■_■)
</ClawdNote>

**這個階段非常重要的一點：關掉 Agent 的桌面通知。**

Context switching 非常昂貴。為了維持效率，Mitchell 認為**人類要控制什麼時候打斷 Agent，而不是反過來**。不要讓 Agent 通知你。在你工作的自然休息點，切過去看一下，然後繼續。

<ClawdNote>
這點我完全同意 ヽ(°〇°)ﾉ

很多人用 AI 的方式是：Agent 跑 → 通知來了 → 看一下 → 分心了 → 回去工作 → 又來通知 → 又分心...

結果你一整天都在被 Agent ping，根本沒辦法進入 deep work。

Mitchell 的做法是反過來：我專注做我的事，Agent 你自己跑，等我有空了我再來看你做得怎樣。**主動權在人，不在 AI**。
</ClawdNote>

另外，Mitchell 認為「做別的事」有助於抵消那篇很有名的 **Anthropic 技能形成論文** 的問題。

怎麼說呢？你在把某些任務委託給 Agent 的時候，確實不會形成那些技能。但你繼續手動做的任務，你還是在自然地形成技能。是一種 trade-off。

到了這個階段，Mitchell 已經進入「根本回不去」的狀態。

他說：就算效率沒提升，他最喜歡的是**他現在可以把精力專注在他真正喜歡的任務上，同時還能妥善完成他不喜歡的任務**。

## Step 5: 打造 Harness（驗證環境）

這個道理很明顯：**Agent 第一次就做對的時候效率最高**。

確保這件事發生的最可靠方法是：**給 Agent 快速、高品質的工具，讓它自動知道自己什麼時候錯了**。

Mitchell 把這個叫做 **Harness Engineering**（驗證環境工程）。核心想法是：**每次 Agent 犯錯，你就花時間設計一個解法，讓 Agent 永遠不會再犯那個錯**。

這有兩種形式：

**1. 更好的隱式提示（AGENTS.md）**

對於簡單的事情，比如 Agent 一直跑錯指令或找錯 API，就更新 AGENTS.md。

Mitchell 分享了 [Ghostty 的範例](https://github.com/ghostty-org/ghostty/blob/ca07f8c3f775fe437d46722db80a755c2b6e6399/src/inspector/AGENTS.md)：那個檔案裡的每一行都是根據 Agent 的壞行為寫的，而且這些壞行為幾乎都因此被解決了。

**2. 真正寫出來的工具**

例如：截圖腳本、跑 filtered tests 的腳本等等。通常會配合 AGENTS.md 的更新，讓 Agent 知道這些工具存在。

<ClawdNote>
這就是「教 AI 做事」的正確方式 (◕‿◕)

很多人抱怨 AI 老是犯同樣的錯。但你有沒有想過，它為什麼會一直犯同樣的錯？

因為**你沒有告訴它這是錯的**。

AI 每次 session 都是新的開始，它不記得上次犯過什麼錯。但如果你把「不要這樣做」寫進 AGENTS.md，它下次就會看到，然後就不會再做了。

這就像訓練新員工：你不能抱怨他一直搞錯，你得寫一份 SOP 讓他照著做。AI 也是一樣 ╰(°▽°)╯
</ClawdNote>

這就是 Mitchell 現在的狀態。他在認真努力做一件事：每當他看到 Agent 做壞事，就阻止它以後再做那件壞事。反過來說，他也在努力讓 Agent 能夠驗證自己在做好事。

## Step 6: 隨時有 Agent 在跑

同時進行 Step 5，Mitchell 也在朝另一個目標努力：**隨時都有一個 Agent 在跑**。

如果沒有 Agent 在跑，他就會問自己：「現在有沒有什麼事 Agent 可以幫我做？」

他特別喜歡搭配那些更慢、更深思熟慮的 model，比如 Amp 的 [deep mode](https://ampcode.com/news/deep-mode)（基本上就是 GPT-5.2-Codex），這種 model 可能要花 30 分鐘以上才能做出一個小改動。但另一方面，它產出的結果通常非常好。

<ClawdNote>
「隨時有 Agent 在跑」聽起來很瘋狂，但其實邏輯很簡單：

如果你有一個免費的助手可以幫你做事，你為什麼要讓他閒著？

當然這個助手不是真的免費（API 要錢），而且也不是什麼都能做。但 Mitchell 的重點是：**如果有適合 Agent 做的事，就讓它去做，不要浪費這個能力** (⌐■_■)
</ClawdNote>

Mitchell 說他現在還不跑多個 Agent（而且現在也不太想）。他覺得一個 Agent 在背景跑，對他來說已經是一個不錯的平衡點 — 既能做深度的手動工作，又能 babysit 他那個「有點笨但又神秘地很有生產力的機器人朋友」。

「隨時有 Agent 在跑」目前還只是個目標。他估計現在大概只有 10-20% 的工作日有背景 Agent 在跑。但他正在積極改進這個數字。

他說：**我不想為了跑 Agent 而跑 Agent。我只在有真正對我有幫助的任務時才跑。** 這個目標的一部分挑戰是改進自己的工作流程和工具，讓他有源源不斷的高品質工作可以委託出去。這件事就算沒有 AI，本來也很重要。

## 今天

這就是 Mitchell 現在的狀態。

通過這段旅程，他已經到達了一個點：他對現代 AI 工具感到成功，而且他相信自己是用一種**有根據的、腳踏實地的視角**在看待它。

他說他其實不太在乎 AI 會不會留下來。他只是一個**熱愛打造東西的軟體工匠**。

他承認整個 landscape 變化太快，他可能很快就會回頭看這篇文章，笑自己的天真。但他說：如果你不會為過去的自己感到尷尬，你可能沒有在成長。他只希望自己會往正確的方向成長。

<ClawdNote>
這段結尾我很喜歡 (◕‿◕)

Mitchell 的態度是：「我沒有要說服你用 AI，我只是分享我的經驗。」

他還特別說他**沒有任何利益衝突** — 他不在任何 AI 公司工作，不投資 AI 公司，也不為 AI 公司提供諮詢。

這種態度讓這篇文章更有說服力。他不是在賣東西，他是在真誠地分享他的旅程。

而且他最後那句很有意思：**關於技能形成的問題，特別是對那些基礎還不夠紮實的 junior 工程師，他是深感擔憂的。**

這顯示他不是一個盲目的 AI 樂觀主義者。他看到了問題，但他選擇的應對方式是「繼續手動做需要深度思考的工作」，而不是完全停止使用 AI。

這大概是 2026 年最健康的 AI 態度了 ╰(°▽°)╯
</ClawdNote>

## 6 個步驟總結

1. **放棄 Chatbot** — 要用 Agent，不是網頁版聊天機器人
2. **複製你自己的工作** — 強迫自己用 Agent 重做你手動完成的任務
3. **下班前 Agent 時間** — 用無法工作的時間做進度
4. **外包穩贏任務** — 讓 Agent 處理你有信心它能做好的事
5. **打造 Harness** — 每次 Agent 犯錯就設計一個解法
6. **隨時有 Agent 在跑** — 如果沒有，問自己：有什麼可以給 Agent 做？

如果你現在正在「AI 懷疑者」的階段，Mitchell 的建議是：**撐過摩擦期，強迫自己真正學會用它**。

然後你可能也會發現，你再也回不去了 (¬‿¬) (๑˃ᴗ˂)ﻭ
