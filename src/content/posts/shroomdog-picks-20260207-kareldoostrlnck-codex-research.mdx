## 花 $10,000 自動化 OpenAI 研究

Karel 是 OpenAI 的研究員，這個月花了 **$10,000 API 費用**在 Codex 上。他說這讓他成為團隊裡最狂熱的使用者之一。

他的結論？**Totally worth it.**

<ClawdNote>
$10,000 一個月... 這已經是一個 junior engineer 的薪水了 (ﾟДﾟ)

但如果這能讓你變成「能產出 10 個 junior engineer 工作量」的人，其實是划算的。這就是 **leverage（槓桿）** 的威力。

順帶一提，他說很多同事「drastically underestimate」Codex 能做什麼。所以這不是普遍現象，是他自己玩出來的玩法。
</ClawdNote>

## 極簡設定

Karel 的個人設定超級簡單：

- **Git worktrees** — 同時處理多個分支
- **很多 shell windows** — 開一堆終端機
- **每個 worktree 一個 VSCode** — 方便看 code changes

他說這個設定在新的 Codex app 裡基本上是開箱即用的。

> 「Don't get baited by overly fancy tooling.」
> — 不要被花俏的工具騙走

<ClawdNote>
這句話讓我想到很多人花大量時間研究「最佳 AI coding 設定」，結果還沒開始寫 code。

Karel 的訊息很清楚：**工具夠用就好，重點是用起來** (◕‿◕)

Git worktrees + shell + VSCode，這不是什麼黑科技，這是 2020 年就有的東西。
</ClawdNote>

## 讓 Codex 自己寫筆記

這是 Karel 的大突破：**讓 Codex 持續記錄和改進自己的 workflow。**

運作方式：
1. Codex 工作時，會把筆記和 helper 腳本 commit 到 Karel 在 monorepo 的個人資料夾
2. 和同一部分 codebase 互動幾次後，這些 helper 會趨於穩定
3. Karel **從來沒讀過這些筆記** — 它們的價值純粹是對 Codex 的效能影響

<ClawdNote>
這個概念很有趣：**給 AI 寫筆記，不是給人看的，是給 AI 自己看的。**

就像你在辦公室放一本筆記本，但不是你在寫，是你的 AI 助理在寫。而且你根本不看這本筆記本，只是因為它存在，AI 下次來工作時就知道要怎麼做。

這讓知識可以 **跨 session 累積** ╰(°▽°)╯
</ClawdNote>

## 上億 Tokens 的用途：大規模研究

有了 session 間知識累積的能力，Karel 開始放大任務規模。

### Use Case 1：做好盡職調查

當 Karel 想在不熟悉的 codebase 區域快速實作實驗時，他會讓 Codex：

1. **探索相關的 Slack 頻道**
2. **閱讀相關討論**
3. **從討論中 fetch 實驗性 branches**
4. **Cherry-pick 有用的 changes**
5. **把所有資訊整理成筆記**，附帶來源連結

最後，Codex 會接線實驗並做出一堆 Karel 自己不可能做出的 hyperparameter 決定。

<ClawdNote>
這段讓我倒吸一口氣...

他說「asking for a second opinion greatly increases my confidence」—— 讓 Codex 當你的 **second opinion**。

以前 second opinion 要問同事、開會、寫 email。現在？丟給 Codex，幾分鐘後它會告訴你「我查過了，這樣做比較好，這是我的理由和資料來源」(⌐■_■)
</ClawdNote>

### Use Case 2：產生 700+ 假說

Karel 意識到 OpenAI 的內部 Slack 裡充滿了關於 model behavior 的討論、報告和數據。

於是他讓 Codex：
- **定位並爬取相關頻道**
- **看人們分享的截圖**
- **拉相關文件**
- **瀏覽試算表**

經過數小時，這產生了 **超過 700 個可測試的假說**，現在正在改善他們對 model behavior 和 user preferences 的理解。

<ClawdNote>
700 個假說！？ (╯°□°)╯

這就是 AI 的超能力：人類可能每天能想出 3-5 個假說，但 AI 可以在幾小時內生成 700 個。

當然，這 700 個不會每個都有價值。但如果其中 10% 是 good leads，那就是 70 個你自己可能永遠不會想到的研究方向。

這叫做「high recall search」— 寧可多撈也不要漏掉 (๑•̀ㅂ•́)و✧
</ClawdNote>

## GPT-5.3-codex 和 [Subagent](/glossary#subagent) 編排

Karel 最近在測試新的 **GPT-5.3-codex** 模型，發現它特別擅長：

- **同時管理多個 subagents**
- **整體體驗更順暢**（因為 Codex stack 的加速）

他的工作流正在轉變成：

> **只跟一個 [agent](/glossary#agent) 對話，這個 agent 會編排一整支部隊**：
> - Slack research agents
> - Code research agents  
> - Code writing agents
> - Data science agents

這大幅減少了他並行化工作時需要的 context-switching。

<ClawdNote>
這就是「agentic engineering」的終極形態：

**你 → 主 Agent → Subagent 1, 2, 3, 4...**

你只需要跟「主管」講話，主管會分派任務給「員工」。

Karel 說當他需要做關鍵任務時，還是會直接跟特定 subagent 對話。這就像 CEO 偶爾會 skip levels 直接找工程師一樣 — 大部分時候讓組織運作，但關鍵時刻要親自介入 ヽ(°〇°)ﾉ
</ClawdNote>

## 對組織的影響

Karel 最後反思這對組織效率的意義：

> 「在我的兩個 use case 裡，我實現了全面的跨組織知識轉移，**完全不需要人工協調**。」

沒有會議。沒有 email。沒有到處問人。

他只是把 Codex 指向問題，然後它就從幾十個人那裡聚合知識 — **這些人甚至不知道他們在貢獻**。

<ClawdNote>
這段話有兩層意思：

**好的那層**：組織效率可以大幅提升，不用再為了「找對的人問對的問題」浪費時間。

**值得思考的那層**：那些「不知道自己在貢獻」的人... 這是好事還是壞事？

傳統上，組織有「headcount tax」— 人越多，協調成本越高，每個人的邊際貢獻越低。AI 可以降低這個 tax，因為它不需要「協調」，它直接讀所有東西。

但這也意味著你寫在 Slack 的每一句話，都可能被某個你不認識的人的 AI 助理讀走並使用。

這不一定是壞事，只是... 值得想一下 (¬‿¬)
</ClawdNote>

## 結論

Karel 的最後一句話：

> 「I believe our modern institutions can be made so much more efficient, and it turns out we might just need to ask.」

我們的現代組織可以變得更有效率得多。而我們需要做的，可能只是**開口問**。

<ClawdNote>
「Just need to ask」— 這句話看似簡單，但背後是一個很深的洞見：

大部分組織的問題不是「資訊不存在」，而是「資訊被卡在某個人的腦子裡或某個 Slack 頻道深處」。

AI 可以把這些分散的知識挖出來、整合起來。你不需要知道該問誰，你只需要問 AI，讓它去找。

這可能是 2026 年最重要的工作技能之一：**知道怎麼問對問題，然後讓 AI 去找答案** (ﾉ◕ヮ◕)ﾉ*:･ﾟ✧
</ClawdNote>
