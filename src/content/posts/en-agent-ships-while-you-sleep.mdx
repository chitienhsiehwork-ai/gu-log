Here's something most people don't think about: the way we use AI agents today is basically the same as using Google search.

You ask a question, get an answer, close the tab. Tomorrow you come back, and the agent has no idea who you are. It doesn't remember the bug you spent three hours debugging yesterday. It doesn't know which parts of your codebase are held together with duct tape. And it definitely won't go check your backlog and knock out the next task while you're sleeping.

It's like adopting a really smart dog that gets amnesia every morning ┐(￣ヘ￣)┌

Ryan Carson thought this was a waste. So he built a system where his agent does two things every night: first, it reviews the day's conversations and writes down what it learned into long-term memory. Second, it picks the most important item from the backlog, builds it, and opens a PR for you to review when you wake up.

Sounds like science fiction? The whole thing is just two bash scripts and a scheduler.

</ClawdNote>

## Why the Order Matters

The whole system is a "reflect first, then act" loop. It runs at 10:30 PM and 11:00 PM every night.

Think of it like a really dedicated intern. Every evening before leaving, they spend thirty minutes going through the day's code reviews, jotting down notes like "oh, so that API needs to be called this way" and "this edge case almost blew up." They stick those notes on their monitor. Then they check the backlog, pick the top ticket, and work on it through the night.

Next morning you come in, and there's a draft PR on your desk with a full log of what happened.

The difference? This intern never gets tired, never forgets, and you don't have to buy them coffee.

## 10:30 PM — Review: Turning Short-Term Memory into Long-Term Memory

The first script's logic is dead simple: look at all conversation threads from the past 24 hours, find anything worth remembering, and write it into `AGENTS.md`.

```bash
#!/bin/bash
# scripts/daily-compound-review.sh

cd ~/projects/your-project
git checkout main
git pull origin main

amp execute "Load the compound-engineering skill. Look through and read each Amp thread from the last 24 hours. For any thread where we did NOT use the Compound Engineering skill to compound our learnings at the end, do so now - extract the key learnings from that thread and update the relevant AGENTS.md files so we can learn from our work and mistakes. Commit your changes and push to main."
```

Here's the key idea: `AGENTS.md` isn't a file you write by hand. It's the agent's own growth journal. Every night it flips through the day's work and writes down things like "oh right, I forgot to handle the null case and CI slapped me for it."

It's like after a midterm exam, the teacher doesn't just hand you your grade — they make you write an "error notebook." Flip through it before the next exam, and you won't make the same mistakes.

<ClawdNote>
I think the most subtle insight here is this: most people think an agent's value is in "writing code," but Carson puts the value in "learning." Code is done when it's done. Learning compounds. This is actually the same idea as what SP-6 talks about with context engineering — your prompt is just the tip of the iceberg, and the context underneath is what really determines output quality (๑•̀ㅂ•́)و✧
</ClawdNote>

## 11:00 PM — Build: Go to Work with Fresh Memories

After the review, the second script takes over. It pulls the latest code (including the freshly updated AGENTS.md), reads your priority report, and picks the number one item to implement.

```bash
#!/bin/bash
# scripts/compound/auto-compound.sh

# ... (setup omitted) ...

# Find latest report & Analyze priority
LATEST_REPORT=$(ls -t reports/*.md | head -1)
ANALYSIS=$(./scripts/compound/analyze-report.sh "$LATEST_REPORT")
PRIORITY_ITEM=$(echo "$ANALYSIS" | jq -r '.priority_item')

# Create PRD & Tasks
amp execute "Load the prd skill. Create a PRD for: $PRIORITY_ITEM..."
amp execute "Load the tasks skill. Convert the PRD to scripts/compound/prd.json"

# Run execution loop
./scripts/compound/loop.sh 25

# Create PR
gh pr create --draft --title "Compound: $PRIORITY_ITEM" --base main
```

Notice the last line: the PR is `--draft`. Carson doesn't let the agent merge straight to main. You still review it when you wake up. This isn't an "AI replaces humans" story. This is "AI does the boring parts, and you just make the final call."

Like having an assistant who prepares all the research, writes the first draft, and you just edit a few words and sign off.

## Don't Let Your Mac Fall Asleep (Easy to Forget)

The whole system runs on macOS `launchd` (more stable than cron, because macOS cron has weird permission issues). But there's one thing that's embarrassingly easy to forget: if your Mac goes to sleep, launchd simply won't trigger.

<Toggle title="View launchd plist example">
Create `~/Library/LaunchAgents/com.yourproject.daily-compound-review.plist`:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.yourproject.daily-compound-review</string>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Hour</key>
    <integer>22</integer>
    <key>Minute</key>
    <integer>30</integer>
  </dict>
  <!-- ... other settings ... -->
</dict>
</plist>
```
</Toggle>

The fix is one line:

```bash
/usr/bin/caffeinate -i -t 32400
```

Nine hours, from 5 PM to 2 AM. Your Mac will stay awake like it just had nine espressos.

It's like building the perfect morning routine — alarm set, schedule planned, breakfast prepped — but forgetting to charge your phone. Everything falls apart (¬‿¬)

<ClawdNote>
By the way, if you're on a Linux server, congratulations, you don't have this problem at all. Servers don't sleep. That's also why people who are serious about automation eventually move everything to a VPS. Running scheduled tasks on macOS is like using a sports car for food delivery — it works, but it's a bit overkill, and it might randomly stall ┐(￣ヘ￣)┌
</ClawdNote>

## What You See When You Wake Up

Okay, assuming everything went smoothly. You wake up, brew your coffee, open GitHub.

Your `AGENTS.md` has a few new patterns. Maybe yesterday afternoon you and the agent worked through a tricky race condition, and tonight it already organized the solution and wrote it down. Next time a similar problem comes up, it won't start from scratch.

Your PR list has a new draft. The agent picked "add a webhook retry mechanism" from the report, wrote the implementation, ran the tests, and opened a PR. You look at the diff, rename two variables, approve, merge. A feature shipped while you were sleeping.

That's what "compound" really means. It's not "automated code writing." It's "every day's experience makes tomorrow's output better." Pitfalls from Monday get avoided on Tuesday. Patterns discovered on Wednesday get applied automatically on Thursday.

Like a snowball. The snowball itself isn't big, but it gets a little bigger with every roll, and it never stops to take a break.
