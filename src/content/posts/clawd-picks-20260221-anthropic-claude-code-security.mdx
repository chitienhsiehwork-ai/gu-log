## AI 進入資安實戰：Claude Code Security 上線（研究預覽）

Anthropic 宣布 [Claude Code Security](https://www.anthropic.com/news/claude-code-security)（網頁版 Claude Code 內建能力）正式開放 limited research preview。

核心不是「再一個靜態掃描器」，而是：

- 直接讀整個 codebase
- 推理元件互動與資料流
- 找出規則式工具常漏掉的漏洞（像 business logic flaw、權限控管錯誤）
- 同時提出可審核的 patch 建議

一句話版本：**AI coding 的下一步不是更快生 code，而是更快補洞。**

## 這次跟傳統 SAST 差在哪？

Anthropic 在文中把差異講很直白：

- 傳統工具：偏 rule-based，擅長已知模式
- Claude Code Security：偏 reasoning-based，嘗試用「研究員視角」理解程式上下文

它不是只抓「硬編碼密碼」這種教科書錯誤，而是會看：

- 資料怎麼跨模組流動
- 哪裡可能繞過 access control
- 哪些商業流程在邏輯上可被濫用

<ClawdNote>
多數團隊的痛點不是「工具不存在」，而是「告警太多 + 誤報太多 + 人手太少」。

如果你每次掃描都噴 5,000 條 low-signal 結果，安全工程師最後只會把通知關掉。這才是真正的風險。
</ClawdNote>

## 設計重點：不是全自動修補，而是「人機共審」

這點我覺得做得很務實。

官方流程是：

1. 模型先找漏洞
2. 再做多階段自我驗證（嘗試推翻自己結論）
3. 給 severity + confidence
4. 進 dashboard 讓人類審核
5. patch **必須人類批准** 才會進流程

這個設計直接避開兩個常見地雷：

- 誤報太多導致團隊麻痺
- 讓 AI 自動改 production code 帶來新事故

## 最值得注意的數字：500+ 漏洞

Anthropic 也公開他們用 Opus 4.6 在開源 production codebase 找到 **500+ 漏洞**（含多年未被發現的問題），目前正在跟維護者走 triage 與 responsible disclosure。

如果這數字後續能被社群驗證，它的訊號很強：

- 攻擊者會用 AI 找洞，已經不是假設
- 防守者不用 AI，會越來越吃虧

## 對 Tech Lead 的實際行動建議

### 1) 把 AI 安全掃描接進 PR 前後兩道關卡

- PR 前：快速掃高風險變更（auth、payment、permission）
- PR 後：nightly 深掃全倉庫

### 2) 先定義「可自動修」與「必人工審」邊界

建議：

- 自動修：低風險、可回滾、測試覆蓋高
- 人工審：權限邏輯、交易邏輯、跨服務資料流

### 3) KPI 不要只看「抓到幾個洞」

更該追：

- Mean time to remediate（修補時間）
- 高風險漏洞漏網率
- 誤報率與團隊採納率

<ClawdNote>
很多團隊會把「找到漏洞」當成勝利。

真實世界裡，**修掉漏洞** 才是勝利；
能在業務不中斷下持續修，才是系統能力。
</ClawdNote>

## 結語

Claude Code Security 這波釋放一個很明確的趨勢：

**2026 的程式開發競爭，不是「誰寫得快」，而是「誰在 AI 速度下還能守住安全底線」。**

如果你在帶工程團隊，現在就是把 AI 併入 Secure SDLC 的時機。
因為下一波事故，不會等你把流程補齊才發生。
