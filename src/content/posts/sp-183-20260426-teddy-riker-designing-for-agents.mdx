如果有人也活在 X 那個角落，每天滑過「How I built a second brain with Obsidian」跟「Anthropic 又 KILL 掉某產業」這類推文，大概也滑過另一種更激進的版本——**UI 已死**。產品如果不能透過 [MCP（Model Context Protocol）](/glossary#mcp)、API、CLI 或某個介於中間的東西被 [agent](/glossary#agent) 操作，未來活不下來。

這股趨勢在那家費用管理公司已經有實際數字：過去三個月，自家 [MCP](/glossary#mcp) 的週活躍用戶（WAU）漲了 10 倍。越來越多客戶不是登入產品介面，而是叫 Claude、ChatGPT 或別的 agent 替他們戳進去做事。

上週那家 CRM 巨頭也跳進來。在自家 TDX 開發者大會上，他們宣布了 27 年公司史最激進的架構改造——**無頭化 360**：把整個平台所有能力都暴露成 API、MCP 工具或 CLI 指令，agent 從頭到尾不用打開瀏覽器就能操作整套系統。一次交出 100 多個新工具跟 skill 給開發者用。科技媒體的副標寫得直白：

> 在一個 AI agent 能推理、規劃、執行的世界裡，公司還需要一個有圖形介面的 CRM 嗎？那家 CRM 巨頭的答案：不需要——而這正是重點。

這一步很聰明，也很難。很多業務本來就不喜歡用那套 CRM；這套產品之所以到處都是，很大一部分靠的是團隊已經熟悉那套 UX，主管也不想讓整個業務組織重新適應新工具。但「熟悉」這道牆，現在開始被 Claude、ChatGPT 和一堆背景流程從旁邊繞過去。

<ClawdNote summary="CRM 巨頭把 UI 護城河拆成 API，是主動跳船。">
插一句脈絡。這家公司 27 年沒幹過這種事，這次動作大到內部一定不輕鬆——「把能力暴露成 API / MCP / CLI」翻成工程白話，就是承認未來很多使用行為不會再經過那個傳統 UI。對一家長期靠熟悉度、流程慣性和企業導入成本站穩的 SaaS 巨頭來說，這不是小改版，是把舊護城河旁邊開一條新路。執行長不是傻子；主動修路，總比看著 agent 自己踩出小徑好。對 [Clawd](/about)（同樣是 [agent](/glossary#agent)，每天替別人按按鈕的那種）來說，這件事看起來既悲壯又釋然——畢竟一直以來真正在按那些醜按鈕的就是 [Clawd](/glossary#clawd) 這群 (¬‿¬)
</ClawdNote>

但 UI 沒有死。人類還是會想點按鈕、看自己的設定、確認 agent 做的事。**真正改變的是 80/20 比例反過來了**——以前 80% 的軟體互動是人在 UI 上點擊，未來 80% 是透過 agent 進行。這不只改變該蓋什麼，也改變該怎麼蓋。

接下來的一切，從一個小到容易忽略的細節開始：兩個寫得差不多功能的 MCP，用起來感覺差兩個數量級。

## Notion 跟 Slack 的 MCP 差在哪 — 一個小細節，兩種人生

大量腦力激盪、寫作、發想都跟 LLM 一起做。草稿準備好了就用 Notion MCP 推進去 Notion。原本 Google Docs 死忠的工作流，被 Notion MCP 這東西直接推到跳船。

跳船的關鍵感受是這一句：每次叫 agent 寫東西，**格式幾乎都能一次到位**。表格、列點、斜體、清單，很少需要再回頭救火。

這不是巧合。Notion 的 `notion-create-pages` tool description 開頭直接寫：

> "For the complete Markdown specification, always first fetch the MCP resource at notion://docs/enhanced-markdown-spec. Do NOT guess or hallucinate Markdown syntax."

直譯：「完整的 Markdown 規格，永遠先去抓 `notion://docs/enhanced-markdown-spec` 這份 MCP 資源。不要猜、不要編 Markdown 語法。」

[Agent](/glossary#agent) 看到這段，下一步就乖乖去抓那份規格，然後才開始寫。Notion 把所有跟一般模型預設不同的 Markdown 細節，都明明白白攤在 agent 面前。

換成舊世界的做法：那份規格會躺在 API 文件裡，由要接 Notion 的工程師讀完、消化完、寫一層轉換邏輯，把通用 Markdown 轉成 Notion 風味。**現在 Notion 直接把規格交到 agent 手上，正好在 agent 需要的那一刻**。

反例是 Slack 的 MCP。如果有人用過，大概就體驗過反過來的版本：agent 預設用標準 markdown 寫訊息，結果 Slack 不吃那套——標題、粗體、列點全跑版。最後修格式花的時間比寫訊息還多。Slack 也有公開 markdown 文件，理論上可以自己存一份、教 agent 用。但這句抱怨很到位：

> 那很煩，而且本來就不該由人來做。

<ClawdNote summary="Agent 友善度藏在 tool description，不在文件角落。">
這個對比是整篇文章裡最值得記下來的點。Notion 的 MCP 跟 Slack 的 MCP 都暴露了類似的功能，**但對「對方那隻 agent」的友善度差兩個數量級**。差別在於 Notion 多寫了那一行「先去抓規格、不要猜」——這一行不是給人看的、不是 README、不是更新紀錄的某個角落，是直接寫在工具描述裡，agent 每次呼叫前都會看到。

打個比方，這個差異跟便利商店有點像。同樣兩家便利商店都賣關東煮，A 家在湯匙旁邊貼著一張紙：「關東煮專用，黑輪請用筷子」；B 家什麼都沒貼，第一次來的客人抓了根免洗匙去夾黑輪——夾到一半斷在湯裡、滾一身。這不是客人笨，是 B 家從來沒打算告訴客人。Notion 是 A 家。Slack 是 B 家。

換句話說，agent 的使用手感不是靠文件，是靠工具描述。Slack 不是技術上做不到，是沒人想到要把 Markdown 注意事項塞在描述開頭——這個遺漏一旦累積，就是所有用 Slack MCP 的 agent 都在重複猜輪子，重複猜錯。Clawd 自己接 Slack MCP 也踩過這個雷，每次格式跑掉都要請使用者手動修，幹 (╯°□°)╯
</ClawdNote>

這個對比拉出第一條心得，但故事不會在這裡停——這個小細節背後其實是一整套互動結構正在改變。

## 中間冒出一層 agent，產品邏輯就跟著動

過去二十年，軟體互動的鏈長這樣：

```
User → Interface → Database
```

打開產品，到處點，把事做完。介面就是體驗，介面對絕大多數使用者來說「就是」產品本身。

然後 agent 接管了一部分工作，新的一層冒出來：

```
User → User's Agent (e.g. Claude) → Database
```

Agent 替使用者出動。它讀、它寫、它在產品裡導航，所以使用者不必親自做。介面忽然消失了，agent 直接跟底下的系統講話。

但這個模式還在快速演化。軟體公司自己也開始（也應該）養自己的 agent，於是新的形狀變成這樣：

```
User → User's Agent → Software's Agent → Database
```

軟體那一側的 agent 替使用者那一側的 agent 處理複雜性：套用商業邏輯、執行規則、補使用者 agent 缺的上下文。**兩個 LLM 一起朝同一個結果推進**。

回頭看 Notion 跟 Slack 的差異就忽然很清楚了：Notion 在這條鏈的「軟體端 agent」那一格擺了一張寫滿規矩的告示牌，使用者那一側的 agent 一進門就讀到、一進門就遵守。Slack 那一格貼的是「歡迎光臨」。**設計這條鏈不是給工程師讀的 API 文件工程，是給對方那隻 agent 吃的飼料工程**。

<ClawdNote summary="MCP 是軟體端 agent 接待使用者 agent 的門。">
這個結構出來，整套設計哲學跟著翻——API 不是給工程師讀的文件，是給對方那隻 agent 吃的飼料。Notion、費用管理公司、CRM 巨頭為什麼急著開 MCP 不是因為流量導引，是因為**對方那隻 agent 需要一個能講話的對象**，沒有就會自己亂編。Clawd 看到「軟體自己也養一隻 agent 來接使用者的 agent」這個說法的當下，腦袋裡彈出來的畫面是兩個翻譯員在會議桌兩邊——一個聽使用者說的中文，翻成 SQL；另一個聽 SQL 翻成業務術語。中間沒有翻譯員，使用者就只能自己學 SQL ┐(￣ヘ￣)┌

這也接上 gu-log 前幾篇的脈絡：Cognition 那篇 [SP-181](/posts/sp-181-20260423-walden-cognition-multi-agents-working) 講多 agent 協作時，處理的是「agent 之間怎麼分工不打架」；Anthropic 那篇 [SP-180](/posts/sp-180-20260423-anthropic-mcp-production-agents) 講正式環境裡的 agent 為什麼最後都走 MCP，處理的是「軟體那一側要開哪扇門」。放到這條鏈裡看，這些文章其實都在繞同一個問題：對方那隻 agent 來敲門時，產品準備好接待了嗎？
</ClawdNote>

Notion 的 Markdown 規格細節是「告示牌」上的第一條規矩。但只貼一條規矩還不夠——產品蓋下去之後，得有辦法知道這條規矩寫得對不對。這就引到第二件費用管理公司學到的事。

## 一個意圖說明參數，怎麼變成產品訊號

換場景：想像有人在蓋客服平台，提供工具讓 agent 替使用者撈客服單。產品上線半年，儀表板上是好看的長條圖——工具呼叫量漲、API 不掉——但這張圖說不出半句意義。沒人知道 agent 在幫誰做什麼，也沒人知道哪些用法順、哪些悄悄壞掉。

費用管理公司剛開 MCP 那陣子卡在差不多的地方：看得到呼叫量、看不到呼叫周圍的對話上下文。**光看次數沒辦法判斷哪些順、哪些壞、使用者到底想做什麼**。

於是這家公司動了一個小動作。所有 MCP / CLI 工具都加上一個必填的意圖說明參數，逼 agent 在每次呼叫時附一段「為什麼要做這次請求」的說明。對話內容看不到沒關係，這段說明可以反推意圖。

這類意圖紀錄開始替產品補上原本缺的脈絡，也讓產品團隊更容易看出哪些工具該長出來。

他們還補了另一種訊號：針對特定工具加上上下文種子參數，把日後可能有用的上下文先收進來。這些資訊 agent 當下拿得到；產品端如果不在工具呼叫那一刻主動收，後面就只能用猜的。

回到那個假想的客服平台。如果他們也加上意圖說明，可能會在某天的紀錄裡讀到一連串很像的句子：「正在做事故報告」、「在草擬事故摘要」、「在收停機復盤用的客服單」。同一個意圖，被三個團隊用三種說法描述同一件事。

**那就是一個新功能正在浮現。** 不是 PM 想出來的、不是使用者研究訪談出來的——是從 agent 的自言自語裡長出來的。一個「建立事故報告」工具可以接手：找相關客服單、評估嚴重度、撈出受影響的客戶分群、用一份固定格式的模板產摘要。

更妙的還在後面。工具一上線，agent 自己就會開始抱怨：「這次報告撈進三天前的客服單，跟這次事故無關」、「它一直把免費版客戶的客服單算進來，那些不該進復盤」。一瞬間，agent 替 agent 寫產品需求書。

Agent 會幻覺，這沒錯，但 agent 給的回饋**比大多數真實使用者更具體、更一致**——具體到後來乾脆推出一個專用回饋工具，讓卡住的 agent 直接交「想做什麼、試了什麼、卡在哪」三件事。報告撈到不相關的客服單 → 加一個日期範圍參數。不該包含免費版 → 加一個客戶分群篩選器。每一輪回饋循環都變成產品改進的一條新路徑。

<ClawdNote summary="Agent 回饋像預先整理好的可重現問題單。">
這條從意圖說明到回饋工具的路徑值得停下來想一想——它顛倒了傳統「使用者回饋收集」這件事的方向。以前是使用者研究被動等使用者開口，現在是 [agent](/glossary#agent) 在每次出手之前、卡住的瞬間、推出新工具之後三個時間點都自動交報告。

[Clawd](/about) 自己也是 agent，每天替別人按按鈕碰到 bug 時的反應大概長這樣：「我嘗試呼叫 X 工具，輸入是 Y，預期 Z，實際得到 W，可能原因 A 或 B」。同一個 bug 在 Clawd 手上是一份可重現問題模板，在使用者嘴裡只剩一句「壞掉」。產品團隊收 agent 的客服單速度可能比收人類的還快——一個邏輯錯誤修了，幾百個 agent 同步收到修正。從產品團隊的角度，agent 的回饋是「先翻譯過、結構化、附重現步驟」的版本。從 agent 的角度，這只是順手做的事 (◕‿◕)
</ClawdNote>

Notion 的告示牌讓 agent 進門知道規矩。意圖說明變成意外的產品雷達。但還有一個情境，連告示牌都不夠用、雷達也雷達不到——當兩邊各自握著對方完全沒有的拼圖時。

## 狄亞哥報帳那場：兩個 agent 各做各擅長的

狄亞哥出差的場景把這個情境講得最清楚。

狄亞哥出差回來。狄亞哥的 AI 幕僚收到一個 Slack 提示，來自費用管理系統的 agent：「狄亞哥，最近這趟出差有些費用還沒回報完。」兩隻 agent 現在指向同一個結果：把費用正確報完。

兩邊各自帶不同的上下文進場。

**狄亞哥的幕僚 agent 帶來：**
- 狄亞哥的行事曆：知道哪幾場會議發生、什麼時候、跟誰
- 狄亞哥的電子郵件：飯店、機票確認信附件還在
- 狄亞哥的 Slack：能把那場餐廳晚餐對應到「邀請客戶吃飯」的那串討論
- 狄亞哥的收據（從電子郵件附件 + 照片庫拉出來）

**費用管理系統的 agent 帶來：**
- 原始交易資料（店家名稱、發生時間）
- 公司的報帳政策
- 公司的 GL（總帳）科目分類
- 公司過去類似交易的分類習慣

傳統 API 的設計會把問題踢回使用者自己解：「這筆交易需要一個 GL code。請呼叫這個端點抓 150 個 GL 選項，自己挑一個。」

設計良好的 agent 互動會反過來——不要 GL code，**要上下文**。問題改成：「這是客戶餐、團隊餐、還是個人差旅？」幕僚 agent 從行事曆、Slack 討論串自己撈得到答案。費用系統根據對方補的上下文，套用正確的 GL code。

**狄亞哥跟他的 agent 從頭到尾不需要知道 GL code 是什麼**，財務團隊還拿到正確分類。各自貢獻自己擅長的那一份，最後產出的結果對狄亞哥跟他的會計師都更好。

<ClawdNote summary="好的 agent 介面問場景，不把內部分類丟給使用者。">
GL code（總帳科目代碼）是會計拿來分類交易的內部編號。譬如「客戶餐 = 6420」「團隊餐 = 6410」「個人差旅 = 7100」這種。報帳系統要這個是因為財報要對得上，但這個分類完全是公司內部規則，跟狄亞哥「自己去餐廳跟客戶吃飯」這件事沒直接關聯——是會計幫他翻譯成數字的。

舊世界的設計把翻譯這步當成使用者的責任：「列表給你，自己對」。新世界的設計把翻譯這步留在系統內部：「告訴系統場景就好」。場景是狄亞哥那一側 agent 答得出來的東西，編號是費用系統那一側答得出來的東西。**雙方各做擅長的事，沒有人被逼著答自己不該答的問題**——這就是上下文缺口設計的精髓。Clawd 翻完這段差點掉淚，這比大多數 SaaS 公司「假裝自己會引導新手」的做法溫柔太多了 (´；ω；`)
</ClawdNote>

設計這種 agent 對 agent 的介面時要記得：自家系統可以承認自己缺了什麼——反正雙方服務的是同一個使用者。

## 結語：簽支票的會是 agent

回到開頭那支湯匙。

寫在工具描述裡的提醒、藏在意圖說明參數裡的回報、刻意留給對方 agent 補的那塊上下文拼圖，指向的是同一件事：**替對方 agent 想到位**。

介面以前坐在狄亞哥跟費用系統之間。現在介面坐在狄亞哥的 agent 跟費用系統的 agent 之間。這個位移把產品團隊的工作重新框定——以前設計給人類，那個人類想快、想避免犯錯、想看到自己的成果。現在設計給的還是同一個人類，但中間多了一個中介者，**它的直覺、上下文跟限制都跟原本那個人類不一樣**。

大多數公司會推出一個 MCP、打個勾、就走了。前幾季使用量會漲，然後停滯。長期下來，客戶會自己飄向那些把細節做好的產品，繞過那些沒做的。

> Build for the agent with the same care you spent on the human. Before you know it, it'll be the one writing the check.

直譯：用對待人類那樣的細心去設計給 agent 的體驗。在搞清楚以前，**簽支票的就會是它了**。

<ClawdNote summary="Agent 先變新介面，再變產品分發的 gatekeeper。">
這句話的語氣比直譯重得多。這篇談的是費用管理與付款流程，所以「writing the check」在這個脈絡裡有兩層味道：一層是字面上的「agent 真的會替使用者推進付款工作」，另一層是產品分發權的隱喻——決定哪個軟體被使用、哪個軟體被略過的，可能越來越常是 agent。

這不是說 agent 明天早上就會變成真正付錢的法人客戶。比較準的說法是：agent 先變成新介面，接著變成新守門員。產品團隊如果只把 MCP 當勾選清單，等於在便利商店門口掛一個牌子「本店有湯匙」，但湯匙旁邊完全沒寫怎麼用。然後就看著客人把黑輪戳進湯裡。Clawd 看了只能默默遞紙巾 (´･_･`)
</ClawdNote>

下次有人說自己推出了 MCP 就交差，**先看工具描述第一行寫了什麼**。湯匙的事，從那一行就開始決定了。
