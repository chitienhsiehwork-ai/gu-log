## ‚ö° Full CLI Command Reference

According to the [Obsidian v1.12 changelog](https://obsidian.md/changelog/2026-02-10-desktop-v1.12.0/) and [official docs](https://help.obsidian.md/cli), the Obsidian CLI capabilities break down into several categories:

### üìÖ Daily Note Operations

```bash
# Open today's daily note
obsidian daily

# Append to daily note
obsidian daily:append content="- [ ] Buy groceries"

# Prepend to daily note
obsidian daily:prepend content="## Today's Goals\n- Finish the article"
```

These three commands are the CLI's **killer feature**. Daily notes are the most-used Obsidian function, and now you can push content without opening the app.

### üîç Search & Read

```bash
# Search your vault
obsidian search query="project status"

# Read the currently active file
obsidian read
```

Search gives AI the ability to explore your knowledge base just like you would. `read` lets AI know what you're currently looking at.

### ‚úèÔ∏è Create Notes

```bash
# Create a new note from a template
obsidian create name="Tokyo Travel Plan" template=Travel
```

Template support in the CLI is incredibly powerful ‚Äî AI can batch-generate notes using your predefined formats.

### ‚úÖ Task Management

```bash
# List all tasks from daily notes
obsidian tasks daily
```

### üè∑Ô∏è Tag Statistics

```bash
# List all tags with their counts
obsidian tags counts
```

This command looks unassuming, but it's **incredibly useful** for AI-powered knowledge analysis. More on this in the hands-on section.

### üìä Version Comparison

```bash
# Compare two versions of a file
obsidian diff file=README from=1 to=3
```

Obsidian's built-in file recovery, now accessible through CLI.

### üõ†Ô∏è Developer Tools

```bash
# Dev tools: inspect element, screenshots, reload plugins
```

The CLI also exposes developer tool access, including element inspection, screenshots, and plugin reloading. A huge win for plugin developers.

## üîß Hands-On: Claude Code + Obsidian CLI

Theory done. Let's see what **Claude Code + Obsidian CLI** actually looks like in practice.

[XDA published an article](https://www.xda-developers.com/claude-code-inside-obsidian-and-it-was-eye-opening/) about putting Claude Code inside Obsidian, running it directly from a terminal in the vault. They said it "transformed how they work."

With the official CLI, this becomes even more natural. Here are three real-world scenarios:

### Scenario 1: AI Auto-Summarizes Daily Notes ‚Üí Weekly Review

**Pain point:** You write daily notes religiously but never look back at them. A week passes and seven daily notes sit there collecting dust.

**Workflow:**

```bash
# Step 1: Find this week's daily notes
obsidian search query="2026-02-03..2026-02-09"

# Step 2: List all tasks
obsidian tasks daily

# Step 3: Claude Code synthesizes a weekly review
# Claude reads search results, analyzes task completion,
# extracts highlights, generates the review

# Step 4: Create the weekly review note
obsidian create name="Weekly Review 2026-W06" template=WeeklyReview

# Step 5: Write content
obsidian daily:append content="..."
```

**What Claude Code does in this flow:**
1. Uses `search` to find all relevant daily notes
2. Uses `tasks` to list everything, analyzing what's done and what's not
3. Uses LLM capabilities to **understand context** ‚Äî not string matching, actually comprehending content
4. Uses `create` to generate a formatted review from your template
5. No Obsidian GUI needed. No manual organizing.

**What used to take 30 minutes now takes one command.**

### Scenario 2: Knowledge Graph Analysis with Search + Tags

**Pain point:** Your vault has 2,000 notes. You sense that certain topics are connected but can't see the big picture.

**Workflow:**

```bash
# Step 1: See what tags you have
obsidian tags counts

# Output might look like:
# #project/alpha    45
# #project/beta     23
# #meeting          89
# #idea             156
# #reference        201
# #ai               67
# #blockchain       34

# Step 2: Search for tag intersections
obsidian search query="#ai #project/alpha"

# Step 3: Claude analyzes co-occurrence patterns
# Which tags frequently appear together?
# Which topics are isolated islands?
# Where are the gaps in your knowledge?
```

**Claude Code can:**
- Analyze tag distribution to find where you spend the most time
- Discover unexpected correlations (e.g., your #ai and #healthcare notes overlap 70% ‚Äî maybe you should double down on AI + healthcare?)
- Suggest new tag structures for better organization
- Find "knowledge islands" ‚Äî notes that aren't connected to anything

**This is essentially doing data analysis on your own knowledge base.**

### Scenario 3: Diff to Track Note Evolution

**Pain point:** An important document (say, a project spec) has been revised many times. You want to understand how it evolved.

**Workflow:**

```bash
# Compare version 1 and version 3 of a file
obsidian diff file="Project-Alpha-Spec" from=1 to=3

# Claude analyzes the differences:
# - What requirements were added?
# - What was removed? Why?
# - Did the architectural direction change?
```

**Claude Code's added value:**

A normal diff tool gives you line-by-line comparison. Claude gives you **semantic analysis**:

- "Between v1 and v3, you switched from microservices to monolith. This usually indicates team downsizing or timeline compression."
- "You dropped three non-core requirements ‚Äî good scope management."
- "But you added two contradictory requirements. Watch out."

**This isn't diff. This is AI doing code review ‚Äî except the code is your ideas.**

<ClawdNote>
Someone might ask: can't I just have Claude Code read the .md files directly? Why go through Obsidian CLI?

Good question. The difference is:

1. **Obsidian CLI understands Obsidian semantics**: it knows what a daily note is, what a template is, what tags mean. Reading raw .md files means you have to teach AI these concepts yourself.
2. **Plugin integration**: CLI search uses Obsidian's built-in search engine, which is far more powerful than text-based grep.
3. **State awareness**: `read` knows what file you currently have open ‚Äî something raw file access can't do.

In short: CLI is an API for Obsidian's semantic layer, not a filesystem wrapper (‡∏á ‚Ä¢ÃÄ_‚Ä¢ÃÅ)‡∏á
