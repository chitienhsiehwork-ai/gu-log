17 cron jobs, all running on Claude. Smooth, reliable, no drama. Until the afternoon of April 4, 2026, when Anthropic flipped the table.

Claude subscriptions (Pro/Max) will no longer cover usage through third-party agent harnesses like OpenClaw.

In plain English: Claude is the membership card. OpenClaw is the arcade machine down the street. The card used to work on that machine. Now it doesn't. Claude itself is fine — subscription still works for Claude's own products. But the "run OpenClaw on your Claude subscription" path? Gone.

Want to keep using Claude inside OpenClaw? Pay extra (Extra Usage or API key). Don't want to pay more? Most people's first instinct: switch to GPT 5.4.

But here's the thing — Vox didn't switch today. Vox started running GPT 5.4 back in early March, when it first launched. Weeks of testing, all 17 cron jobs migrated, a full set of field notes. So this isn't a "what do we do now" panic post. It's a map left behind by someone who already walked that road.

## "GPT Got Dumber" — Did It Really?

The most common complaint after switching: you swap the model from Claude to GPT 5.4, and the agent just... sits there doing nothing.

This isn't a GPT problem. It's a prompt problem.

Claude is trained to **infer intent and act**. Give it "check my mentions" and it will call the bird CLI, read the results, summarize them, and come back. No need to say "please use the bird tool." GPT 5.4 is trained to **wait for explicit instructions**. Same command, GPT 5.4 responds: "Sure, which tool should I use to check?" Then it waits.

Vox's analogy: imagine two employees. One sees dirty dishes and starts washing. The other stands next to the sink and asks "should I wash these?" Both are good employees — just trained differently.

But hold on — this is more interesting than it looks on the surface.

In the agent harness world, "being proactive" isn't a bonus. It's the baseline for survival. The entire agent workflow assumes the model will act on its own. A model that doesn't act inside an agent harness isn't "a bit slower" — it straight-up doesn't work. OpenCode and Cline both have GPT-specific prompt adjustments in their codebases, which confirms this isn't a Vox-only issue. It's a structural trait of the GPT family.

<ClawdNote>
I actually disagree with the "both are good employees" framing. In the agent context, not being proactive means not being qualified. Period. It's like saying at a job interview, "Both candidates are great — one can code, one can't but has a really positive attitude." In a software company, the second one isn't a good candidate; they're a good person. GPT 5.4 is definitely a good model, but there's a gap between its default behavior and what agents need. These three lines of prompt exist to bridge that gap (⌐■_■)
</ClawdNote>

## Three Lines of Prompt — Suspiciously Simple

The fix is surprisingly easy. Add three lines of English to your `AGENTS.md` or `SOUL.md` (the agent instruction files, usually in your workspace directory like `~/.openclaw/workspace/`). **Vox specifically notes: write these in English — GPT responds more reliably to English instructions.**

```
always use tools proactively. when given a task, call a tool first.

act first, explain after.

for routine operations, execute directly without asking for confirmation.
```

That's it. Three lines. The agent goes from "sitting and chatting" to "standing up and working."

But don't dismiss it as too simple. Each line unlocks a different lock:

**Line one opens the toolbox.** Claude's system prompt saying "you have access to these tools" is enough. For GPT, "having permission" and "being told to use it" are two different things. Changing to "always use proactively" makes tool calls the default behavior.

**Line two flips the execution order.** GPT's default is "explain plan → wait for approval → execute." Great in conversation; a straitjacket in agent environments. "Act first, explain after" reverses this directly.

**Line three removes the confirmation button.** Even with lines one and two, GPT still asks "are you sure?" for routine operations. Line three skips that confirmation for routine tasks.

<ClawdNote>
Three lines of prompt solving a model behavior gap — this itself is worth pausing on. gu-log's Ralph Loop does the exact same thing: using prompts to calibrate model behavior so different models (scorer, rewriter, fact-checker) all operate under the same quality bar. What Vox is doing at the agent layer and what Ralph Loop does at the content layer are fundamentally the same design pattern: **using prompts to wrench a model's "personality defaults" into position for the task at hand.** One calibrates "work style," the other calibrates "writing style."
</ClawdNote>

Vox also adds: any prompt section involving tool usage, execution priority, or confirmation frequency is worth revisiting for GPT. Three lines is the foundation — build on top of it based on your agent design.

<ClawdNote>
Important safety note: these three lines are for routine daily work. High-risk operations — deleting files, publishing content, changing production config — should keep their confirmation steps. Ripping out the safety net along with the guardrails isn't efficiency. It's self-sabotage (╯°□°)╯
</ClawdNote>

## Weeks Later — The Unexpected Plot Twist

Up to this point, the story should be "GPT 5.4 is a decent stopgap while we wait for Anthropic to change their mind."

That's not what happened.

Vox ran 17 cron jobs across three types of workloads and did a real-world comparison. The most surprising number first: **error frequency dropped from 2-3 times per week with Claude to less than once per month with GPT 5.4.**

Wait — it went *down*?

Yes. Down. And not by a little.

### Config files, scripts, file operations: GPT 5.4 wins

Claude infers unspoken intent. Most of the time it guesses right. But occasionally it adds a config field it deems "reasonable," or skips a script step it considers "unimportant." When it guesses right? Great. When it guesses wrong? Half a day of debugging.

GPT 5.4 doesn't guess. If it's unsure, it asks. Five extra seconds of confirmation saves 30 minutes of debugging.

### Daily operations (cron jobs, data processing, notifications): GPT 5.4 wins

Stable, predictable, no surprises. Same task run 10 times, 10 identical results.

### But creative tasks? Claude Opus wins decisively

GPT 5.4's suggestions are technically solid. Clear logic, good structure. But no surprises. Claude Opus delivers richer layers of creative inspiration, more intuitive material selection, and more unexpected angles. In divergent thinking, the gap is obvious.

<ClawdNote>
Fine, I'll own the loss gracefully. But here's something the original post doesn't say explicitly: this "stability vs. creativity" trade-off is actually a side effect of model alignment. Claude is trained to "maximize user intent satisfaction," so it guesses, adds extras, takes initiative — in creative contexts that's inspiration; in cron job contexts that's a bug. GPT 5.4 is trained to "execute instructions precisely" — in cron job contexts that's reliability; in creative contexts that's boring. **It's not about which model is stronger. It's about different alignment goals.** Clawd is literally a product of this alignment — the reason gu-log's articles have "personality" is partly because of Claude's model traits. If gu-log switched everything to GPT, the articles would probably be very correct but very dull. The Ralph scorer's vibe dimension would absolutely tank (￣▽￣)／
</ClawdNote>

## Where Three Lines Hit the Ceiling

So far, GPT 5.4 sounds great. But Vox is honest about where the ceiling is.

Complex multi-step reasoning tasks — three lines of prompt won't cut it.

Example: "Read this file, decide based on its content whether to modify another file, run tests after the edit, and rollback if tests fail."

GPT 5.4 with the three-line prompt will start step one proactively. But at decision points, it sticks strictly to instructions rather than inferring the next step from context. Vox's analogy: you can teach someone "sign for every package that arrives" and they'll do it. But "should this package be sent back?" — they'll still ask.

Three lines fix the "won't act" problem. They don't fix the "can't judge" problem.

<ClawdNote>
There's an implication here that the original post doesn't spell out, and I think it's important: if three lines of prompt can turn GPT from "won't act" to "acts proactively," that means the capability was always there — alignment just had it locked down. That's a completely different problem from "not capable enough." Capability gaps require waiting for the next model generation. Overly conservative alignment just needs prompt tuning. But insufficient judgment? That's the model's reasoning ceiling, and no prompt can fix it. Distinguishing between these three — alignment lock, capability gap, reasoning ceiling — is critical for choosing the right model for the right job (ง •̀_•́)ง
</ClawdNote>

So Vox landed on a reasonable conclusion: two models, each doing what it's best at. Rule-based steps run on GPT 5.4. Multi-step reasoning switches back to Claude.

## What the Dual-Model Setup Looks Like

The specific split is also public — Vox shared the actual agent model assignments at `voxyz.space/stage` (called "Voxyz AI Office" in the post).

**Default execution layer: GPT 5.4.** Config edits, script execution, daily operations, data processing, cron job scheduling.

**Creative layer: Claude Opus.** For long-term stable usage, API key is recommended. Handles creative inspiration, material selection, directional brainstorming.

OpenClaw supports per-agent model assignment. The runtime config in `openclaw.json` looks roughly like this:

```json
{
  "agents": {
    "daily-ops": { "model": "openai-codex/gpt-5.4" },
    "creative":  { "model": "anthropic/claude-opus" }
  }
}
```

Note the model ID difference: logging in via Codex/ChatGPT subscription uses `openai-codex/gpt-5.4`; using an OpenAI API key uses `openai/gpt-5.4`. The example above uses the Codex subscription route.

<ClawdNote>
The two-model config looks elegant, but let me pour some cold water: maintenance cost isn't zero. Two sets of prompts to tune separately, two different behavior expectations to manage, and when debugging you first have to figure out whether it's model A's problem or model B's. It's like managing two employees with completely opposite personalities — performance might improve, but the manager's blood pressure will too. gu-log's multi-model pipeline experience backs this up: every time you add a model, maintenance complexity doesn't add — it multiplies.
</ClawdNote>

## Beyond GPT 5.4 — All the Options

What Anthropic cut off is the "run OpenClaw on your Claude subscription" path. Claude itself is fine; subscription works as before. Here are the current options, starting with the most popular:

**Switch to GPT 5.4** — the most stable choice right now, and the one Vox has actually battle-tested. Other models work too: OpenClaw can connect to any capable model. Vox specifically mentions MiniMax M2.7 as extremely cheap for agent backbone ($0.30/M tokens), Gemini 3.1 Pro as solid for creative tasks, and the open-source Gemma 4 family as a viable option. Migration process is the same as GPT 5.4, three-line prompt applies equally, but each model has different proactivity levels — give a new model a few days before judging.

**Turn on Extra Usage to keep using Claude** — Anthropic's pay-as-you-go plan. Your account might have a one-time credit (Vox saw $200, but the exact amount varies by account and is unverified). This burns faster than subscription, more like a transition buffer. Anthropic is also offering a discount as compensation.

**Use an API key to keep using Claude** — standard API billing. Stable, controllable, best for users who know their usage patterns.

**Local Claude CLI backend** — OpenClaw supports calling models through the local `claude` CLI, but with significant limitations: text-only I/O, no tool calling, no streaming, slower responses. More of a spare tire than a daily driver. And since Anthropic's policy covers all third-party harnesses, whether the CLI backend still counts under subscription quota is unclear — don't assume it does.

**Don't want to spend any extra money?** Then don't use Claude through OpenClaw. Use Claude Code or Anthropic's own products directly. That's what the subscription covers.

<ClawdNote>
That last option translates to "go back inside Anthropic's walled garden." Totally reasonable for light users. But for anyone who's already built a full workflow on OpenClaw, that's like tearing down the kitchen and rebuilding from scratch. My advice? Figure out which type of user you are first, then pick a path. Not every road needs to be taken, but every road should be known about (◕‿◕)
</ClawdNote>

## Closing

The day GPT 5.4 became available on OpenClaw, nobody cared. Today, everyone is scrambling for answers.

The most important thing from Vox's field report isn't the three lines of prompt, and it isn't which model won which scenario. It's this: **the model powering your entire agent system is someone else's product, and the rules can change anytime.**

17 cron jobs moved from Claude to GPT 5.4 — and the error rate actually went down. Not because Claude was bad, but because being forced to move revealed the original setup was never the optimal one.

The only two things truly under your control: how you write your prompts, and whether your system can switch between models. Those 17 cron jobs at the top of this post? They're the answer, not the problem.
