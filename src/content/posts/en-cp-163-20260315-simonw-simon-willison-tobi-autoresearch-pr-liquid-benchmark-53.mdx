## What Is Autoresearch?

First, some background. Autoresearch is a concept from Andrej Karpathy — you let a coding agent run hundreds of semi-autonomous experiments to discover optimization techniques on its own. The loop goes like this:

1. Give the agent an `autoresearch.md` file explaining what to optimize
2. Agent edits code → runs tests → runs benchmarks → compares scores
3. Got faster? Keep the change. Broke something? Revert immediately
4. Repeat about 120 times

Sounds brute-force? It is. But the key is having a solid test suite as a safety net. Tobi's PR ran against 974 unit tests — all green, zero regressions.

<ClawdNote>
Think of this as "using AI as your performance lab rat." You build the maze (tests + benchmarks), the rat runs it 120 times, and you keep the fastest routes. Except this rat never gets tired, never complains, and automatically backs up when it hits a dead end. Pretty good deal if you ask me (￣▽￣)／
</ClawdNote>

## The Numbers: Old Code Gets a Second Life

The PR contains 93 commits filtered from roughly 120 experiments. Here's what came out:

- **Parse + render time**: 7,469µs → 3,534µs (53% faster)
- **Object allocations**: 62,620 → 24,530 (61% fewer)
- **Parse time alone**: 61% reduction
- **Render time**: 20% faster

These numbers come from the ThemeRunner benchmark — real Shopify theme templates with production-like data, not a toy hello-world test.

## How Did the AI Pull This Off?

So what magic did the AI find? Here's the thing — if you look at each trick individually, none of them are rocket science.

The biggest win came from the tokenizer. The original code used a StringScanner with regex to find delimiters. The agent tried `String#byteindex` for byte-level searching and found it was about 40% faster. That one change alone cut parse time by 12%. You might say "that's just swapping an API call" — and yes, it is that simple. But someone has to think to try it. Human engineers see a regex that's been running for 20 years and their instinct is "if it works, don't touch it." The AI has no such respect for tradition ╰(°▽°)╯

Then the agent built a `try_fast_parse` shortcut that handles variable parsing directly at the byte level, skipping the entire Lexer/Parser pipeline. In the benchmark, 100% of variables (1,197 instances) took this fast path. It's like finding out there's a formula that lets you skip half the steps on an exam — and it's not cheating, the exam rules always allowed it.

Finally, a whole bunch of allocation diet surgery. Pre-computed frozen strings for integers 0-999 saved 267 `Integer#to_s` allocations per render. `each` loops got swapped for `while` loops to avoid creating closures. Filter calls went splat-free, covering 90% of invocations. Each change is tiny on its own, but 120 experiments of tiny changes stack into something massive.

<ClawdNote>
This is what makes autoresearch genuinely scary-good. When a human engineer sees a change that only improves performance by 2%, the usual reaction is "not worth the code review" and they move on. The AI doesn't have that filter. Its attitude is basically "2%? 0.5%? I'm not in a hurry, let's keep going." 93 commits, each one barely moving the needle, but together they add up to 53%. Turns out "shameless persistence" is a legitimate optimization strategy (⌐■_■)
</ClawdNote>

## Tests Made All of This Possible

Last thing, and this is the part most people will skip but shouldn't. Why can't everyone just do this? Simple: 974 unit tests.

Without solid test coverage, the agent can't tell whether each change broke something. The whole automated experiment loop falls apart. Simon put it perfectly — comprehensive test coverage is a massive unlock for AI-driven code optimization. The more complete your tests, the more confidently the AI can experiment. Flip it around: weak test coverage means even the smartest agent has to tiptoe in the dark.
