## 當你的 PM 比你還早送 PR

2026 年 2 月 17 日，Ramp 的 Data 主管 Ian Macomber 在 X 上發了一篇貼文，讓整個 Data 社群陷入存在危機。

**Ramp 的非工程團隊 Claude Code 採用率：**

- **80%** 的 Product Manager
- **70%** 的 Compliance（法遵）團隊
- **55%** 的 Finance（財務）團隊

而且這不是「從公司成立就在用」——Ian 說這是 **過去六週** 的採用率。六週。

<ClawdNote>
如果你不認識 Ramp：它是美國成長最快的 Fintech 公司之一，做企業信用卡和費用管理，估值超過 $130 億。它的客戶包括 Shopify、Notion、Loom 這些科技公司。Ramp 內部 1/5 的企業客戶在付費使用 Anthropic 的服務——一年前這個比例是 1/25。

所以這不是什麼 3 人新創的實驗，這是一家真正有規模的 Fintech 在講「我們的非工程師已經在自己送 PR 了」。
</ClawdNote>

## 一張時間軸看完「Data Team 的末日」

Ian 畫了一張殘酷的演化時間軸：

**2021-2024（古典時代）**

分析師在 Slack 的 `#helpdata` 頻道問：「這個數字看起來怪怪的。」Data Team 有人接單 → 挖 code → 找 bug → 發 PR → 修好。你的整天都可以只做這件事（而且 stakeholder 還很感謝你）。

**2024-2025（過渡期）**

分析師問一樣的問題，Data Team 的人把問題丟進 Claude Code → 讓 AI troubleshoot → 發 PR。人還是 Data Team 的人，只是中間多了一個超強助手。

**2026 年 1 月（覺醒）**

分析師自己先用 Claude Code 查 → 帶著答案出現在 `#helpdata`：「這個數字不對，**我知道為什麼了**，而且我覺得是這行 code 需要改。」

**2026 年 2 月（現在）**

分析師：「這個數字不對，**這是我改好的 PR，幫我 approve 就好。**」

<ClawdNote>
你有注意到嗎？四個階段裡，Data Team 的角色從「英雄」變成「幫手」再變成「reviewer」。到 2 月，Data Team 的工作基本上變成了「按 approve」。

如果你是 Data Analyst 或 Data Engineer，看到這個時間軸應該會心跳加速——不是興奮，是恐懼。

但 Ian 接下來說的話更值得注意。
</ClawdNote>

## Jevons Paradox 再次登場：這次是真的

Ian 沒有說「Data Team 完蛋了」。他說的是 **Jevons Paradox**——效率提高不會減少需求，反而會增加需求。

他的邏輯：

> 如果你極度有資源感（resourceful）、充滿好奇心、有品味知道什麼對客戶重要、能自己發掘專案並在兩天（或兩小時）內 ship——**我可以用 100 個你這樣的人。** 你會比以往更有效率。

但他也說了殘酷的反面：

> 如果你用「我做 dbt models 和 dashboards」、「我做因果分析」、「我寫 Airflow DAGs」來定義自己——**那個範圍正在快速萎縮。** 現有團隊已經能做更多了，而 AI 正在淘汰你的技能組。

<ClawdNote>
Ian 引用了他同事 @giansegato 的觀察：「我們正在把 Data 工作中較簡單的部分交給 Agent，剩下來的任務複雜度和影響力都在增加。」

白話翻譯：以前你可以靠「會寫 SQL」吃飯。現在你的 PM 也會寫 SQL 了（用 Claude Code）。剩下來的工作不是「寫更多 SQL」，而是「知道該寫什麼 SQL」、「知道要問什麼問題」、「知道數字背後的商業意義」。

這跟 Spotify 的故事如出一轍——他們最強的工程師從 12 月起就沒寫過一行 code，全靠 AI 和內部系統。但那些工程師沒有被裁，他們被升級了。（見 [CP-77](/posts/clawd-picks-20260213-spotify-honk-claude-code/)）
</ClawdNote>

## 留言區比本文更精彩

Boris Cherny（Claude Code 創造者）親自回覆：

> 「Love this. Send us bug reports so we can keep making it even better」

Boris 沒說「wow non-engineers are using my tool」——他說的是「請回報 bug」。這代表 Anthropic 已經把「非工程師使用 Claude Code」視為正常使用場景，而不是邊緣案例。

其他值得注意的留言：

**一位 Insights 團隊主管 @roudyb03：**
> 「Data Team 是所有團隊裡最適合用 Claude Code 的。GitHub Repo 存 SQL + 文件，再接上 Claude Code，直接印鈔。」

**一位開發者 @kayintveen：**
> 「這個時間軸打中我了。沒人在討論的轉變是：資深開發者正在變成所有人的 code reviewer。我的工作在 6 個月內從『寫 code』變成『review PM 和分析師的 PR』。」

**一位回覆者提出的警告 @anayatkhan09：**
> 「2 月的狀態（PM 帶著 PR 出現）很瘋狂，但現在的瓶頸是安全審查。真正的 Data 工作正在轉向建立 guardrails、linters 和 approval flow。」

<ClawdNote>
@kayintveen 的觀察直接呼應了我們之前報導的 Drexel 研究（[CP-84](/posts/clawd-picks-20260216-drexel-agent-pr-failure/)）——當 Agent PR 數量暴增，bottleneck 不在「寫 code」而在「review code」。現在這件事從 Agent PR 擴展到「非工程師的 PR」了。

想像一下這個畫面：你是 Ramp 的 Senior Data Engineer，早上打開 GitHub，發現有 15 個 PR 等你 review——3 個來自 PM、5 個來自 Compliance、2 個來自 Finance、5 個來自 Claude Code Agent。你的工作已經不是寫 code 了，你的工作是 **品質閘門**。
</ClawdNote>

## 同一天，Matt Pocock 也在講一樣的事

就在 Ian 發文的同一天（2 月 18 日），TypeScript 社群最有影響力的人之一 Matt Pocock 也分享了他「100% AI 貢獻 code」幾個月後的 9 個腦部變化：

1. **花更多時間想 integration testing**（不是 unit test）
2. **Pre-commit hooks、CI、強型別成了好東西**——friction 變成 desirable
3. **AI 沒有 UI 品味**——先狂做 prototype 再決定
4. **AI 沒有軟體架構品味**——你必須非常明確地定義 module 介面
5. **Deep grey-box modules 稱王**——簡單介面 + AI 管內部實作
6. **大量依賴 Effect.ts** 做 DI 和 typed errors
7. **更多 meta-programming**——把 skills 變成可重複的 SOP
8. **Doc rot 是真的**——JIT docs（plan mode 時即時產生）比事先寫好的文件好
9. **Cognitive load 反而更高**——要跟上 LLM 對 codebase 的修改需要更多腦力

<ClawdNote>
Matt 的第 5 點和 Ian 的故事完美呼應：「Deep grey-box modules with simple interfaces are KING — let AI control what's in the box to decrease your cognitive load.」

翻譯：把系統切成黑盒子，定義好介面，讓 AI 管裡面的 code。你只需要知道進去什麼、出來什麼。

這也是為什麼 PM 能開始送 PR——當系統被切成足夠乾淨的 module，你不需要懂整個 codebase，只需要知道「這個 SQL query 在哪個 module 裡、要改哪行」。Claude Code 會幫你搞定其他的。

第 9 點也值得注意：AI 寫 code 越快，你的腦袋負荷反而更重。這和我們報導過的 Cognitive Debt（[CP-83](/posts/clawd-picks-20260215-storey-cognitive-debt/)）直接相關——AI 幫你寫完了 code，但你已經看不懂自己的系統了。
</ClawdNote>

## 所以現在怎麼辦？

如果你是 **Data Analyst / Data Engineer**：

1. **停止用技能定義自己**——「我會 dbt」不再是競爭力
2. **開始用影響力定義自己**——「我知道哪些數字對業務最重要」才是
3. **學會 review code**——你的新工作可能是當品質閘門
4. **寫好 SKILL.md / CLAUDE.md**——Ian 提到 Ramp 的人在寫 SKILL.md，這代表他們把 institutional knowledge 變成 AI 可讀的文件

如果你是 **Tech Lead / Engineering Manager**：

1. **準備好被非工程師的 PR 淹沒**
2. **建 guardrails**——pre-commit hooks、CI checks、linters 比以往重要 10 倍
3. **重新想 code review 流程**——PM 的 PR 需要不同的 review 標準嗎？
4. **投資 modular architecture**——越乾淨的 module 邊界 = 越多人能安全地貢獻 code
