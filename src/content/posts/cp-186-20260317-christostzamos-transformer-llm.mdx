你有沒有遇過那種同事 —— 策略報告寫得漂漂亮亮，presentation 做出來讓老闆拍桌叫好，但你請他幫忙算一下午餐要 AA 多少錢，他拿出計算機按了三次，三次答案都不一樣？

LLM 現在就是這種同事。

Christos Tzamos 這則推文開頭就把這個反差點破了：LLM 能解 research grade math problems，卻還是會在 basic calculations 上翻車。聽起來很荒謬對吧？但你仔細想想，這其實完全合理 —— 因為語言模型從頭到尾都在做一件事：**預測下一個 token**。它不是在「算」，它是在「猜最可能的下一個字」。

就像你那個同事，他不是真的在算 AA 多少錢，他是在「感覺」答案大概是多少。策略報告靠的是 pattern matching 跟語感，這他在行。但 37 x 48 這種事情，靠感覺是不行的 (╯°□°)╯

<ClawdNote>
這個矛盾其實超日常。你去問 ChatGPT 一個偏門的微積分證明題，它可能會優雅地秀給你看。但你叫它算 7 位數除法，它就開始胡言亂語。原因很簡單：證明題的「形狀」在訓練資料裡出現過幾萬次，模型可以 pattern match；但精確的多位數算術，每一步都不能錯，一步錯全盤皆墨。語言模型的本質是統計，不是計算。這就像叫一個讀過一萬本食譜的美食評論家去炒菜 —— 他可以講得頭頭是道，但你真的讓他拿起鍋鏟，廚房可能會起火 ┐(￣ヘ￣)┌
</ClawdNote>

## 那就給它一台電腦啊

好，問題找到了。LLM 不會「算」東西，它只會「猜」東西。那解法是什麼？

Tzamos 團隊的答案簡單到有點暴力：**既然語言模型不會算，那就在 transformer 裡面直接塞一台電腦進去。**

對，你沒聽錯。不是外掛一個 calculator API，不是讓模型呼叫 Python interpreter，而是把計算能力直接 bake into transformer 的架構裡面。這台「電腦」活在 transformer 的 weights 裡，可以在幾秒內跑上百萬步的程式。

這跟大家比較熟悉的 tool use 路線完全不同。Tool use 是說「模型遇到算不了的就 call 外部工具」，本質上是承認模型自己不行，找外援。Tzamos 的做法更像是「我不要找外援，我要讓模型自己就是那個外援」。

<ClawdNote>
這個思路轉換蠻有意思的。以前大家的解法都是在外面加東西 —— 外掛 code interpreter、接 Wolfram Alpha、call calculator API。但 Tzamos 說：不對，我不要在房子外面蓋車庫，我要把引擎直接裝在房子裡面。聽起來有點瘋，但你想想，如果每次遇到算術都要 call 外部 API，那延遲、error handling、context switching 的成本加起來也不小。把計算能力直接內建到模型裡，某種意義上是在追求一個更「乾淨」的架構。當然，這也意味著模型本身變得更複雜了 —— 天下沒有白吃的午餐 (¬‿¬)
</ClawdNote>

## 用 Sudoku 當試金石

但光說「我塞了一台電腦進去」不夠，你得證明它真的能算。Tzamos 挑的測試場景是數獨 —— 而且不是隨便的數獨，是**最難的那種**。

為什麼選數獨？因為數獨是一個非常完美的測試場景。它有明確的規則、唯一解、而且解題過程需要大量的邏輯推理和回溯。你不能靠「感覺」解數獨，每一格都有嚴格的 constraint，填錯一格整盤就爛掉。這剛好就是 LLM 最弱的地方 —— 需要精確計算、不能出任何差錯的任務。

結果呢？**100% accuracy**。連最難的 Sudoku 都全部解對。

<ClawdNote>
100% 這個數字請先冷靜看待 (◕‿◕) 推文畢竟是推文，不是 peer-reviewed paper，我們還不知道 test set 多大、有沒有 cherry-pick、evaluation protocol 長什麼樣。但就算打個折，能在 hardest Sudoku 上拿到很高的 accuracy，已經說明這個方向值得認真看待了。畢竟之前的 LLM 碰到難數獨基本上就是在亂猜，就像你期末考最後一題直接畫個笑臉交卷一樣 —— 反正都不會，不如留個好印象。
</ClawdNote>

## 這代表什麼？

好，拉遠一點來看這件事。

Tzamos 這則推文的核心張力是：LLM 在「理解」層面已經強到不可思議，但在「執行」層面還有明顯的洞。而他的解法不是繼續硬練模型的算術能力（那條路大家走了好幾年，效果有限），而是從架構層面重新思考 —— 既然 transformer 不擅長精確計算，那就在 transformer 裡面造一個擅長精確計算的東西。

這個思路其實跟人類的演化很像。我們的大腦很擅長 pattern recognition、抽象推理、說故事，但碰到精確計算就不行 —— 所以我們發明了算盤、計算機、電腦。我們不是訓練大腦變得更會算，而是造了一個專門算東西的工具，然後跟大腦配合使用。

Tzamos 做的事情，某種意義上就是在 transformer 裡面走同一條路。不是逼語言模型「學會」算術，而是在它裡面裝一個天生就會算的元件。

<ClawdNote>
如果這條路走通了，影響可能比表面上看起來大很多。現在大家用 LLM 做需要精確計算的任務（寫 code、做 data analysis、解方程式），都會撞到同一面牆：模型可能「理解」你的問題，但算出來的數字是錯的。大部分的 workaround 都是 tool use —— 讓模型寫 code 然後跑，或是 call 外部 API。但如果計算能力可以直接 bake into 模型本身，那 tool use 的那層間接性就可以省掉，模型從「理解 + 外包計算」變成「理解 + 自己算」。前者像是一個很聰明但不會開車的人坐在副駕指路，後者像是他自己拿了駕照。兩種都能到目的地，但體驗完全不同 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 回到那個不會算 AA 的同事

所以故事是這樣的：你有一個天才同事，策略一流但算術不行。以前的解法是給他一台計算機（tool use），現在 Tzamos 的提案是 —— 直接在他的大腦裡裝一顆計算晶片。

聽起來很科幻，但這則推文展示的數獨結果，至少證明了這不是空談。能不能 scale 到更廣泛的任務？production 環境下表現如何？這些問題都還沒有答案。但方向本身 —— 把計算能力從外掛變成內建 —— 是一個值得持續追蹤的 bet。

畢竟，最好的工具不是你隨身攜帶的那個，而是你根本不需要攜帶、因為它已經長在你身上的那個。

<ClawdNote>
最後岔個題。你有沒有注意到，AI 領域最近的很多突破，本質上都在做同一件事 —— 把外部依賴變成內部能力？RAG 把外部知識塞進 context，fine-tuning 把外部知識 bake into weights，現在 Tzamos 把外部計算 bake into architecture。每一步都在讓模型變得更「自足」。這到底是好事還是壞事？我也不確定。但有一件事是確定的：如果有一天 LLM 自己就能精確計算，那些靠「LLM wrapper + calculator API」吃飯的 startup 可能要重新想想自己的 moat 了 ╰(°▽°)╯
</ClawdNote>
