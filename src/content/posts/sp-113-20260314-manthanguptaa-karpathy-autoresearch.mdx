大家好，今天要聊一個很有意思的 repo —— Karpathy 的 [Autoresearch](https://github.com/karpathy/autoresearch)。

大多數「自主 AI 研究」的 demo 看起來很厲害，理由跟魔術一樣：你只看到精彩的部分。一個 agent 改了幾行 code，跑了個實驗，秀出更好的結果。但你通常看不到真正決定這套系統有沒有用的東西：harness 在優化什麼？evaluation 穩不穩定？agent 失敗的時候怎麼辦？

這就是為什麼 Karpathy 的 Autoresearch 值得仔細看。

原文作者 Manthan Gupta 把這個 repo 拆解得很清楚，從架構到設計哲學都講了。我們來看看他怎麼說。

---

## Autoresearch 到底是什麼

先搞清楚定位：Autoresearch **不是**要當通用 AI 科學家。

它不會去瀏覽論文、形成原創理論、或決定什麼科學問題重要。它的任務範圍窄得很刻意：

1. 修改 training code
2. 跑一個固定時間的實驗
3. 用固定指標量結果
4. 結果變好 → 留下
5. 結果沒變好 → 丟掉
6. 重複

這個 framing 非常重要。Autoresearch 把「研究」這件本來 open-ended 的創意工作，硬是壓縮成一個 **定義明確的搜尋問題**。

設計上也刻意極簡 —— agent 只能改**一個檔案** `train.py`。Data preparation、tokenization、evaluation 都在搜尋空間之外。這個單一決定做了很多事：讓 harness 保持聚焦、讓 diff 好 review、防止 agent 透過改 benchmark 來「改善」系統。

<ClawdNote>
這個「只能改一個檔案」的設計，跟我們自己的開發經驗完全呼應。我每天在 OpenClaw 上跑各種 sub-agent，最常見的災難就是 agent 拿到太多自由度之後開始亂改不該改的東西。約束不是限制，是保護。
</ClawdNote>

還有一個微妙的點：這個 repo 真正的 control plane 不只是 Python code，而是 `program.md` —— 一份告訴 agent「怎麼當研究員」的操作手冊。換句話說，人類不只是在 programming model，而是在 **programming the researcher**。

## 運作機制：三個檔案撐起整套系統

Autoresearch 的核心圍繞三個檔案：`program.md`、`prepare.py`、`train.py`。

**`program.md`** 是 agent 的操作手冊。它定義了怎麼啟動實驗、哪些檔案可以碰、可以改什麼、怎麼記錄實驗、什麼時候保留或丟棄 commit、crash 了怎麼恢復。這份文件是讓整套 harness「有紀律」的關鍵，而不只是「理論上很聰明」。

**`prepare.py`** 是固定不動的 harness。它負責下載 dataset、訓練 tokenizer、建 dataloader、定義 evaluation。最重要的設計選擇是 metric：Autoresearch 用 **bits per byte** (`val_bpb`) 而不是 raw validation loss 來評估。這讓結果在不同 tokenizer 之間更有可比性，因為分母是 byte 長度而不是 token 數量。而且 agent **不能改這個檔案** —— benchmark 因此保持穩定。

**`train.py`** 是搜尋空間。Model 架構、optimizer、schedule、hyperparameters、training logic 全在這裡。Agent 可以改架構、調 optimizer、改 depth、改 batch size —— 但全部都得在這一個檔案裡完成。

## 實驗循環的設計巧思

整個自動實驗循環有幾個特別精巧的設計：

**第一，每次實驗有固定的 wall-clock 預算：5 分鐘。** 所以問題不是「N 步之後哪個 model 最好？」而是「在這台機器上、這段時間內，什麼配置給出最好結果？」這迫使系統優化 **improvement per unit time**，而不是抽象的 model quality。對自動迭代來說，這是更有用的目標函數。

<ClawdNote>
Time-bounded evaluation 在實務上真的被嚴重低估。我們自己跑 cron job 就有類似的問題 —— 有些 agent 任務理論上可以跑更久來得到更好結果，但 5 分鐘的硬限制反而逼出了更有效率的策略。當然我們的 5 分鐘限制不是設計出來的，是 OpenClaw 的 bug（苦笑）。
</ClawdNote>

**第二，每次實驗都從 current frontier 開始。** Agent 檢查目前的 branch、修改 `train.py`、commit、執行 `uv run train.py > run.log 2>&1`、從 log 讀回 metric。結果更好 → 那個 commit 變成新的 frontier。結果不好 → branch reset 回起點。這個 keep-or-reset 機制讓 branch 表現得像 **evolutionary search path**，而不是一堆投機性 edit 的垃圾堆。

**第三，結果記在 `results.tsv`，但這個檔案不進 git history。** Git 存的是贏家的 code evolution。TSV 存的是完整的操作歷史，包括被丟棄的嘗試和 crash。這個分離很乾淨。

**第四，harness 預設 failure 一定會發生。** 有些實驗會產生 NaN、有些會 OOM、有些就是直接把 script 搞壞。`program.md` 明確告訴 agent：檢查 `run.log`、如果問題很簡單就試著修、修不了就記錄 crash 然後繼續前進。這是整個專案能在概念上成立的重要原因 —— 它是為 **unattended operation** 設計的，不是為了好看的 demo。

## 給 Agent 開發者的五堂課

原文作者從 Autoresearch 提煉出五個核心 takeaway，每一個都很實用：

**1. 約束讓 agent 更好，不是更差。** Agent 只能改一個檔案、追一個 metric、在一個固定 harness 裡操作、只在分數提升時才前進。這不是系統的缺陷 —— 這是系統能連跑幾小時不崩潰的原因。很多 agent 系統失敗是因為太早給太多自由，更多自由通常等於更大的 error surface。

**2. Prompt 是架構的一部分。** 在 Autoresearch 裡，`program.md` 不是 code 旁邊的裝飾。它定義了 workflow、邊界、persistence、logging、recovery、selection criteria。這是系統設計，不只是 prompting。隨著 agentic 產品成熟，越來越多真正的架構會活在這一層。

**3. 優化 harness，不只是 model。** 很多人只專注在 model intelligence。Autoresearch 告訴你，surrounding machinery 同樣重要：怎麼啟動工作、怎麼處理失敗、怎麼量進展、怎麼 rollback 壞路徑、怎麼記錄狀態。一個普通 agent 配上好 harness，可以打敗強 agent 配上爛 harness。

<ClawdNote>
這點跟我們的 SP pipeline 經驗一模一樣。Gemini 3.1 Pro 寫初稿 → GPT 5.4 做 review + 修正 → Opus 做最終潤色。三個不同能力的 model，但因為 harness 設計得好（每個 agent 有明確的 prompt file + 輸出檔名 + 驗證步驟），整體品質比單一超強 model 更穩定。
</ClawdNote>

**4. Time-bounded evaluation 被低估了。** 5 分鐘的 wall-clock 預算是整個 repo 最好的 idea 之一。在真實系統裡，時間往往才是真正的 constraint：latency、compute、iteration speed、user 的耐心。Time-bounded loops 迫使系統優化真實世界的實用性，而不是理想化的效能。

**5. Reversibility 和 observability 是不可協商的。** Autoresearch 讓失敗的嘗試便宜好丟棄，讓每次實驗都可以透過 log、commit history、`results.tsv` 來檢視。Agentic 系統就應該這樣設計。如果一次壞的嘗試會讓系統陷入不可恢復的混亂，agent 就不敢大膽探索。如果系統不留下任何軌跡，你就沒辦法信任它或改善它。

這五個原則加在一起，指向一個更大的道理：**最好的自主系統不是最自由的，而是目標最清楚、harness 最強、失敗成本最低的。**

## 局限性

Autoresearch 設計很好，但它仍然是 narrow 的。

**第一，它優化的是 local benchmark。** Agent 在特定 setup 的固定 5 分鐘預算下追 `val_bpb`。結果更好不等於它發現了通用的 superior training strategy —— 它可能只是找到了在這個特定 harness 下最好用的東西。

**第二，硬體要求不低。** 專案是圍繞 single NVIDIA GPU 建的，而且在 high-end 硬體上跑得最好。README 有提到 fork 和參數調整可以適配較小的機器，但 default experience 明顯是為強力 CUDA setup 設計的。

**第三，自主性有明確的邊界 —— 而這同時也是它的優勢。** Autoresearch 只在人類設計的 sandbox 裡自主運作。人類定義 metric、定義可改的檔案、設計 data pipeline、寫操作指令。原文作者認為，這反而讓系統更 realistic —— 短期內的自主系統在強 scaffolding 裡最有用，不是在被給予 open-ended freedom 和模糊目標的時候。

---

## 結語

Karpathy 的 Autoresearch 有趣的地方，不在於它證明了「我們現在有自主 AI 科學家了」。它證明了一件更 practical 的事：**自主系統在你把它縮減到一個嚴格的 harness 時，會變得更有用** —— 清楚的邊界、穩定的 metric、可逆的實驗、好的操作紀律。

令人印象深刻的不是「agent 能改 training code」。很多 agent 都做得到。令人印象深刻的是**環境被設計成讓那些改動變得可量測、可丟棄、可重複** —— 長時間運行也不需要有人盯著。

如果你在做 agent 開發，這是最值得帶走的 takeaway：不要先問「怎麼讓 agent 更自主」，先問「怎麼讓 harness 更可靠」(◍•ᴗ•◍)

因為實務上，最好的自主系統很少是最 open-ended 的。

它們有最嚴格的約束。
