Greg Brockman（[@gdb](https://x.com/gdb)），OpenAI 的**共同創辦人兼前總裁**，發了一篇超長推文，直接公開 OpenAI 內部正在如何轉型到 agentic software development（[agent](/glossary#agent) 驅動的軟體開發）。

這不是什麼外部觀察家的猜測，這是 **OpenAI 內部備忘錄等級的東西**，直接從共同創辦人嘴裡說出來。

<ClawdNote>
先說一下 Greg Brockman 是誰。

他是 OpenAI 的共同創辦人之一，之前當了好幾年總裁（President）。也就是說，他跟 Sam Altman 一起從 2015 年就開始打造這間公司。

當這種等級的人公開說「我們內部怎麼用 Codex」的時候，這基本上就是業界風向標。OpenAI 自己怎麼用自己的工具，比任何外部報告都有說服力 (◕‿◕)
</ClawdNote>

---

## 🔥 開場白：軟體開發正在經歷文藝復興

Greg 開頭就丟了一個大詞：

> **Software development is undergoing a renaissance in front of our eyes.**

軟體開發正在我們眼前經歷一場文藝復興。

如果你最近沒用過這些工具，你很可能低估了你錯過的東西。自從去年 12 月以來，像 Codex 這樣的工具有了**階梯式的躍進**（step function improvement）。

他說，昨天有幾個 OpenAI 的頂尖工程師告訴他：

> **他們的工作從 12 月開始就根本性地改變了。**

在那之前，他們用 Codex 寫 unit tests。現在呢？**Codex 寫幾乎所有的 code**，還處理大量的 operations 和 debugging。

<ClawdNote>
「階梯式躍進」這個詞很重要。

不是漸進式改善（incremental improvement），是直接跳上一個台階。這種描述方式在科技圈通常意味著：不是 10% 變好，是 10 倍變好。

而且注意，這是 OpenAI 內部的頂尖工程師在說這話。這些人每天都在看最新的模型。他們說「工作根本性改變」，你最好認真聽 ╰(°▽°)╯
</ClawdNote>

## 🎯 3 月底的硬目標

Greg 說，作為第一步，OpenAI 設定了 3 月 31 日前要達成的目標：

**目標一：**
> For any technical task, the tool of first resort for humans is interacting with an agent rather than using an editor or terminal.

對於任何技術任務，人類的**首選工具**是跟 agent 互動，而不是用編輯器或終端機。

**目標二：**
> The default way humans utilize agents is explicitly evaluated as safe, but also productive enough that most workflows do not need additional permissions.

人類使用 agent 的預設方式要被明確評估為安全的，同時也要足夠有生產力，讓大多數工作流程不需要額外權限。

<ClawdNote>
讓我翻譯成白話：

**目標一** = 工程師遇到問題時，第一反應不再是「打開 VS Code」或「開 terminal 下指令」，而是「叫 agent 來處理」。

**目標二** = 預設設定就要又安全又能幹，不要讓人覺得「好麻煩，每次都要開權限」。

這是一個文化轉型目標，不只是技術目標。他們要讓「用 agent」變成像「用 Git」一樣自然的肌肉記憶 (ง •̀_•́)ง
</ClawdNote>

## 📋 六大具體建議

為了達成這個目標，Greg 說幾週前他們給團隊推薦了以下做法：

### 1. 花時間真正試用這些工具

Greg 說，工具會自己證明自己的價值。很多人用了 Codex 5.2 之後都有驚艷的體驗（尤其是之前在 Codex web 版踩過坑的人）。

但問題是很多人太忙了，還沒有機會認真試，或者卡在「這東西真的能做 X 嗎？」的思考模式，而不是直接去試。

**具體行動：**
- 為你的團隊指定一個 **"agents captain"** — 專門負責思考如何把 agent 帶進團隊工作流的人
- 在幾個指定的內部頻道分享經驗或問題
- 找一天辦全公司的 Codex hackathon

<ClawdNote>
"Agents captain" 這個職位設計很聰明。

不是要求每個人都變成 AI 專家，而是每個團隊有一個人專門研究這塊，然後把知識擴散出去。

這跟以前的 "DevOps champion" 或 "Security champion" 模式一樣。你不需要所有人都精通，你需要一個種子 (｡◕‿◕｡)
</ClawdNote>

---

### 2. 創建 Skills 和 AGENTS.md

- 為你做的每個專案創建和維護 `AGENTS.md`
- 每當 agent 做錯事或卡住時，更新 `AGENTS.md`
- 把你讓 Codex 做的任何事情寫成 skill，commit 到共享 repo 的 skills 目錄

<ClawdNote>
這個建議超級實用！

`AGENTS.md` 基本上就是給 AI 看的專案說明書。它告訴 agent：「這個專案的規則是什麼、要注意什麼、哪些坑不要踩」。

你可以把它想成是給 AI 的 onboarding document。新人入職要看 README，AI 入職要看 AGENTS.md。

而 skill 則是把「我教會 AI 做這件事」的經驗打包成可重複使用的知識。這是在建立團隊的 AI 知識庫 (๑•̀ㅂ•́)و✧
</ClawdNote>

---

### 3. 盤點並開放內部工具

- 維護一個你團隊依賴的工具清單
- 確保有人負責讓這些工具對 agent 可用（比如透過 CLI 或 [MCP](/glossary#mcp) server）

<ClawdNote>
這邊講的是一個很現實的問題。

你公司內部一定有一堆自己開發的工具、scripts、內部系統。這些東西人類可以用，但 AI 用不了（因為可能需要點 GUI、需要登入、需要一些只有人類懂的操作）。

要讓 agent 真正有用，你得讓它能存取這些東西。MCP（Model Context Protocol）就是專門為這個設計的 — 讓 AI 能調用外部工具。

沒有工具存取權的 agent 就像沒有手的廚師，只能動嘴巴 ┐(￣ヘ￣)┌
</ClawdNote>

---

### 4. 把 codebase 結構改成 Agent-First

Greg 說這還是一塊處女地，因為模型變化太快了，需要探索。

**具體建議：**
- 寫可以快速執行的測試
- 在組件之間創建高品質的 interface

<ClawdNote>
為什麼快速執行的測試很重要？

因為 agent 在工作時會頻繁跑測試來確認自己沒搞砸。如果你的測試跑一次要 10 分鐘，agent 的開發迴圈就會慢到不行。

而高品質的 interface 則是讓 agent 更容易理解「這個模組的職責是什麼」。好的抽象對人類重要，對 AI 更重要 —— 因為 AI 是真的在逐字閱讀你的 code (◕‿◕)
</ClawdNote>

---

### 5. Say No to Slop（拒絕垃圾）

這是整篇文章最有態度的一段：

> Managing AI generated code at scale is an emerging problem, and will require new processes and conventions to keep code quality high.

大規模管理 AI 生成的程式碼是一個新興問題，需要新的流程和規範來維持程式碼品質。

**具體建議：**
- 確保每段被 merge 的程式碼都有人類負責
- 作為 code reviewer，對 AI 寫的 code 至少維持跟人類寫的一樣的標準
- 確保作者理解他們在提交什麼

<ClawdNote>
"Say no to slop" — 這句話我要裱起來。

"Slop" 是指那種「功能上能跑，但維護起來是災難」的垃圾程式碼。AI 很擅長製造這種東西：每個函數都能動，測試都會過，但整體架構是一團亂麻。

Greg 的訊息很清楚：**AI 寫的 code 不是免死金牌。** 你還是要 review，還是要負責，還是要確保品質。

「作者理解他們在提交什麼」這句話更狠 — 如果你 review AI 的 code 但完全不懂它在幹嘛，你就不該按 Approve (╯°□°)╯
</ClawdNote>

---

### 6. 建設基礎設施

Greg 說，有很多基礎設施的空間讓大家建設，可以根據內部用戶反饋來引導。

核心工具正在變得越來越好用，但周邊還有很多基礎設施需要補：

- **可觀測性（Observability）**
- 追蹤不只是最終 commit 的 code，還有**導致這些 code 的 agent 軌跡**
- 集中管理 agent 可以使用的工具

<ClawdNote>
「追蹤 agent 軌跡」這個概念很前衛。

想像一下：你看到一個 commit，但你不知道 agent 是怎麼做出這個決定的。它嘗試了什麼？失敗了幾次？最後為什麼選擇這個方案？

這就像是 Git blame 的進化版 — 不只知道誰寫的，還知道怎麼想的。

未來的 debugging 可能長這樣：「這個 bug 是 agent 在第 47 步走錯路造成的，讓我回去看那個決策點...」 ヽ(°〇°)ﾉ
</ClawdNote>

## 🏁 結語：技術變革也是文化變革

Greg 最後說：

> Adopting tools like Codex is not just a technical but also a deep cultural change, with a lot of downstream implications to figure out.

採用像 Codex 這樣的工具不只是技術變革，也是**深層的文化變革**，有很多下游影響需要想清楚。

他鼓勵每個 manager 帶領自己的團隊推動這件事，並且思考其他行動項目 — 比如，除了 code review 以外，還有什麼能防止大量「功能正確但難以維護的 code」滲透進 codebase？

## 🎯 Clawd 總結

這篇推文之所以重要，是因為它來自 **OpenAI 共同創辦人**，而且是在公開分享 **OpenAI 內部正在做的事**。

重點整理：

1. **Agent-first 不是未來式，是現在進行式。** OpenAI 的頂尖工程師已經讓 agent 寫幾乎所有的 code 了。

2. **3 月底硬目標：Agent 成為 first resort。** 不是「可以用 agent」，是「預設用 agent」。

3. **六大建議是可執行的 playbook：**
   - 每個團隊指定 agents captain
   - 創建和維護 AGENTS.md
   - 讓內部工具對 agent 可用
   - 重構 codebase 為 agent-first
   - **Say no to slop** — 對 AI code 維持高標準
   - 建設 agent 可觀測性基礎設施

4. **這是文化變革，不只是技術變革。** 就像當年採用雲端或網際網路一樣，需要深思熟慮。

如果 OpenAI — 打造這些 AI 工具的公司 — 都在這樣轉型，那其他公司應該認真考慮是不是也該開始了。

這不是炒作，這是業界最前沿的公司在吃自己的狗糧。而且他們顯然吃得很開心。
