有些軟體團隊的問題不是「工程師不夠努力」，而是每一件小事都要人類親手搬一次。

PR 要看。問題票要分類。安全問題要掃。重複回報要合併。效能基準掉了要有人發現。會議裡講到一個功能，會後還要有人記得開票、寫規格、排優先級。這些事情單獨看都不難，合在一起就像辦公室裡每個人桌上都放一疊便條紙，風吹一下，整個開發流程開始懷疑人生。

所以原作者在 2026 年 5 月 15 日那則推文裡，真正丟出的問題不是「[OpenClaw](/glossary#openclaw) 到底花了多少 AI 錢」。推文說外界看到的是 AI 花費，沒看到的是 OpenClaw 正在回答一個更刺激的問題：

如果 [Token](/glossary#token) 成本不再是主要限制，未來軟體會怎麼被建造？

這句話很危險。因為它不是在問「能不能用 AI 幫忙寫程式碼」，而是在問：既然機器可以一直讀、一直跑、一直比對、一直追蹤，軟體團隊還需要把多少組織摩擦保留給人類？

這篇可以接著 gu-log 前面兩條線讀：一條是 [OpenClaw Task Flow](/posts/sp-186-20260428-article-openclaw-task-flow/) 那篇講的「把多步驟流程編排起來」，另一條是 [Claude Code Agent Teams](/posts/sp-105-20260305-claude-code-agent-teams/) 那篇講的「多個 [Agent](/glossary#agent) 互相分工」。這次補上的，是更下一層的問題：當流程和分工都已經存在，團隊能不能把它們變成常駐作業系統。

<ClawdNote>這裡的重點不是「Token 免費所以亂燒」。比較像便利商店突然多了一百個不會睡覺的夜班店員。真正該問的不是電費，而是：補貨、盤點、過期品檢查、客訴分類，還有店長會議紀錄，哪些事情還需要每次都叫人類從零開始？</ClawdNote>

## 不是 AI 花費，是摩擦轉移

原作者說 OpenClaw 在雲端長期跑大約一百個 [Codex](/glossary#codex)，審每一個 PR、看每一張問題票。這個數字聽起來很浮誇，但浮誇的地方不只是「一百個」。真正浮誇的是工作邊界。

傳統軟體團隊通常把 AI 放在單點任務：幫工程師補一段測試、解釋一個錯誤、產生一個元件、改一個小錯。這像把電動螺絲起子交給工人，效率變高，但產線本身沒有變。

OpenClaw 的想像比較像把 AI 放到整條產線的空隙裡。不是某位工程師需要幫忙時才叫 Agent，而是 Agent 本來就在那裡巡邏。

PR 進來，就有人讀。提交進來，就有人掃安全風險。問題票被打開，就有人判斷是否重複、是否符合產品願景、是否應該直接開 PR。留言區開始出現垃圾訊息，就有人擋。效能基準回歸，就有人往 Discord 報告。會議正在討論功能，就有 Agent 開始把討論變成可執行的改動。

這不是「AI 寫比較快」而已。這是在把組織裡最容易卡住的地方，改成常駐監控系統。

## 先從沼澤開始：每張問題票都被接住

推文裡最核心的一句，是 OpenClaw 長期在雲端跑約一百個 Codex，審查每個 PR、每張問題票。

在一般專案裡，「每張問題票都被認真讀過」聽起來像童話故事。大型開源專案尤其可怕。使用者回報重複問題、貼半截紀錄、用不同語言描述同一個錯誤、在半年後才有人發現主分支早就修好了。維護者不是不想處理，而是問題票清單會自然變成沼澤。

推文提到的 `@clawsweeper`，就是這個沼澤清理工。假設主分支合進一個修正，`ClawSweeper` 最後會找到六個月前那張已經被修掉的問題票，並用精確參照把它關掉。

這件事很小，但它代表一個重要轉向：問題票不再只是「等待人類注意的清單」，而是可以被程式碼變動反向更新的知識庫。

以前流程大概是：

- 某人修了錯誤。
- 某人記得這個錯誤以前有問題票。
- 某人回去找問題票。
- 某人貼上修正 PR 或提交。
- 某人關掉問題票。

這裡面每個「某人」都可能消失。不是因為懶，是因為軟體開發本來就充滿情境切換。工程師剛修完認證流程，下一秒 Slack 跳出線上事故，再下一秒 PM 問下週發版能不能趕上。六個月前那張問題票？抱歉，大腦快取已經被清掉。

常駐 Agent 的價值，就是讓這種「很重要但很容易被忘記」的工作，不再依賴人類記憶。

<ClawdNote>問題票清單最恐怖的地方不是數量，是失去因果關係。錯誤明明修了，但問題票還活著；使用者明明又回報一次，但其實是同一件事；PR 明明解決三個問題，但只關掉一張票。這就像冰箱裡有十盒便當，不知道哪盒過期、哪盒剛買、哪盒其實已經被吃完只剩空盒。Agent 做的是貼標籤、對發票、把空盒丟掉。</ClawdNote>

## 接著輪到安全：每次提交都看一眼

OpenClaw 還會在每次提交上跑 Codex 做安全審查。推文裡的理由很直接：安全問題太容易漏掉。

這句話沒有炫技，反而很像資深工程師的恐懼。安全錯誤通常不是那種大紅燈閃爍的東西。它可能藏在一個權限判斷順序、一個沒有檢查的 webhook、一個錯把使用者輸入傳進命令列的輔助函式、一段「看起來只是重構」的差異裡。

人類程式碼審查有盲點。審查者可能累了、趕著吃飯、太熟悉這段程式碼反而看不到問題。更麻煩的是，安全審查的成本很不平均：九十九次可能什麼都沒有，第一百次漏掉就爆炸。

所以 OpenClaw 的做法是把「每次都看一眼」這件事交給 Agent。這不代表 Agent 一定比專業安全工程師強，也不代表掃到的每個問題都是真的。它代表安全檢查從「有人有空才做」變成「每次提交自動發生」。

這裡也接到推文提到的 Vercel `deepsec` 與 Codex 安全審查。Vercel 在 [2026 年 5 月 4 日開源 `deepsec`](https://vercel.com/blog/introducing-deepsec-find-and-fix-vulnerabilities-in-your-code-base)，把它描述成由寫程式 Agent 驅動的安全檢查框架，可以在使用者自己的基礎設施上跑，找大型程式庫裡難抓的漏洞。Vercel 的文章寫到，`deepsec` 會先用靜態分析找安全敏感區域，再讓 Agent 追資料流、檢查防護、產出可執行的發現，最後還有第二輪重驗來降低誤報。Vercel 也明講，早期經驗裡誤報大約落在 10% 到 20%。

這個限制很重要。Agent 安全掃描不是魔法護身符，比較像永遠醒著的助理安全研究員：會翻很多地方、會提出可疑點、會把事情整理成可追的單子，但仍然需要制度處理誤報、分級與修復責任。

<ClawdNote>安全掃描最怕「看起來有做」。那種工具跑完吐三百個紅色警告，最後大家建立肌肉記憶：看到紅色就關掉。`deepsec` 比較有趣的地方，是它把重驗放進流程，承認 Agent 會誤報，然後再派另一輪 Agent 來驗屍。這很工程化：不是假裝模型聖光普照，而是把模型當會犯錯的員工管理。</ClawdNote>

## 沼澤變大以後，分流不能靠英雄主義

推文接著說，OpenClaw 會用 Codex 去除重複問題票、找出問題群聚，並針對最緊急的問題送出報告。

這句聽起來像一般客服分類，但放在軟體專案裡很有殺傷力。因為很多產品問題不是單一問題票能看出來的，而是要從一群相似回報裡看出模式。

可以把它想成這種情境：幾張問題票用不同語言描述同一類登入或設定問題，有人貼半截紀錄，有人只留一句「更新後不能用了」。人類如果一張張看，會覺得這些都是雜訊；Agent 如果能把它們聚成同一類，維護者看到的就不再是幾個零散抱怨，而是一個可能正在擴散的回歸問題。

這就是分流的本質：不是把票排漂亮，是把雜訊變成可行動的訊號。

傳統上，這件事靠資深維護者的腦內索引。誰記得哪個模組最近改過、哪個平台最常壞、哪類回報其實是同一個根因，誰就能把混亂變成優先級。問題是這種能力不容易擴展，而且會累。

OpenClaw 的設計把這個腦內索引外部化。Agent 持續讀新問題票、比對舊問題票、形成群聚、推送最重要的報告。人類還是要做產品判斷，但不用先從泥巴裡徒手撈訊號。

## 最昂貴的不是修，是重現

推文裡最有畫面感的一段，是 OpenClaw 有 Agent 可以重建複雜環境、啟動臨時機器、登入像 Telegram 這樣的服務、錄一段影片，然後把修正前後的比較貼到 PR。

那個 `t.co` 短連結實際指向 `crabbox.sh`。只靠這則推文，能穩穩說的是：原作者指向一種可拋棄的臨時機器／重現環境，而不是 Vercel 那個沙盒服務。這個差別要守住，否則文章就會把「原作者貼了一個重現環境」偷換成「原作者在講某個特定雲端產品」。

這和一般 CI 最大的差異，是它不只跑測試。它可以像一個很有耐心的 QA 工程師：

- 把專案環境架起來。
- 登入外部服務。
- 重現使用者流程。
- 錄下錯誤發生的畫面。
- 套用修正後再跑一次。
- 把修正前後的證據貼回 PR。

對開源專案來說，這非常實際。很多錯誤不是單元測試能抓的。它可能需要特定帳號狀態、第三方服務、瀏覽器權限、手機通知、聊天平台整合。叫人類每次手動重現，跟叫消防車去澆桌上的多肉植物差不多：不是不能做，是成本比例荒謬，而且最後多肉還可能被沖爛。

Agent 加上臨時機器的意義，是把「重現成本」壓低到可以常態化。當重現不再貴，PR 討論就不必停在「看起來應該修了」。它可以直接附上一段修正前後的影片證據。

## 如果問題真的值得做，它會先長出 PR

更進一步，推文說 OpenClaw 有 Codex 會監看新問題票；如果問題票很符合專案文件化的願景，就自動建立 PR。然後，另一個 Codex 會審查那個 PR。

這個流程很有意思，因為它把產品判斷拆成兩層。

第一關是「這件事該不該做」。OpenClaw 不是讓 Agent 對所有問題票都衝去寫程式碼，而是用文件化願景當篩選器。也就是說，產品方向先被寫成文件，Agent 再用那份文件判斷新回報是否落在邊界內。

第二關是「做出來能不能進」。自動產生 PR 不代表自動合併。另一個 Codex 會審查，後面仍然可以接人類審查、測試、安全掃描與效能基準。

這裡最值得注意的是，Agent 不是單一人格從頭包到尾，而是形成互相檢查的網路。寫程式碼的 Agent 和審查的 Agent 分開；找漏洞的 Agent 和重驗的 Agent 分開；做功能的 Agent 和檢查效能基準回歸的 Agent 分開。

單一 Agent 很容易像精神太好的實習生，一路自信滿滿衝到牆上。Agent 網路比較像一間小公司：有人寫、有人審、有人測、有人盯指標、有人清垃圾留言。每個角色都有限，但角色之間的制衡開始長出工程系統的味道。

<ClawdNote>「Agent 互相審查」聽起來有點像讓兩台影印機互相檢查考卷，荒謬中帶著一點合理 (¬‿¬)。重點是不要把它當最終真理，而是當便宜的第一道摩擦。人類審查者最珍貴的時間，應該花在架構、產品判斷、風險取捨，不是每次都先幫 PR 找少打一個空值檢查。</ClawdNote>

## 開發作業系統開始處理日常雜務

推文後半段開始變得像一份「軟體團隊自動化清單」，但每一項都對應一種真實摩擦。

OpenClaw 有 Codex 掃留言垃圾訊息並封鎖使用者。這看似跟開發無關，其實很關鍵。開源專案一旦有流量，問題票與留言區就會變成公共場所。公共場所沒有基本清潔，維護者的注意力會被垃圾訊息、釣魚、灌水、惡意連結吃掉。自動管理不是酷炫 AI，而是讓工程討論保有衛生條件。

OpenClaw 也有 Codex 驗證效能基準，並把回歸問題報到 Discord。這又是另一種常見災難：效能壞掉通常不是功能壞掉那麼明顯。按鈕還能按，測試還會過，但延遲變高、記憶體吃多、某個流程慢到使用者開始暴躁。等客服抱怨進來，問題可能已經進主分支很久。

把效能基準回歸推到 Discord 的意思，是讓效能從「有人想到才看」變成團隊日常訊號。它不是一份躺在 CI 頁面的報告，而是會跑到團隊溝通場所拍桌子。

還有會議 Agent。推文說 OpenClaw 有 Agent 會聽會議，並主動開始工作；例如團隊正在討論新功能時，Agent 就建立 PR。

這大概是整則推文裡最科幻、也最容易被誤解的一點。重點不是「會議紀錄自動摘要」這種 2024 年味道很重的功能。重點是會議裡的決策不再掉進會後黑洞。

很多團隊真正浪費的地方，是會議中其實已經形成共識，但共識沒有變成工作產物。沒有人開問題票、沒有人寫驗收條件、沒有人把探索分支做出來。兩週後大家又開一次會，重新考古上一集劇情。

會議 Agent 如果真的能把討論中的功能變成 PR，或至少變成可審查的工作項目，那會議就從「同步狀態」變成「觸發產線」。這個差異很大。

## 地圖先畫好，Agent 才不會巡邏到迷路

原作者最後提到，OpenClaw 建了一個工具，推文裡用 `t.co` 短連結帶過；從他的描述看，這個工具的用途是把所有專案拆成功能模組，再拿去審查、找錯誤、找回歸問題。安全面也做同樣拆分，配合 Vercel `deepsec` 與 Codex 安全審查找回歸與漏洞。

這是整個系統的地基。

Agent 如果只面對整個程式庫，很容易變成「請閱讀全世界，然後告訴工程師哪裡有問題」。這種 [prompt](/glossary#prompt) 聽起來很豪邁，實際上常常像叫一個人進圖書館，三小時後交一份人類文明總結。氣勢很大，結果很玄。

拆成功能模組的價值，是把程式庫變成 Agent 可以反覆巡邏的區域。每個模組有邊界、有責任、有相關測試、有可能的回歸類型。審查不再是看一坨巨大的差異，而是問：「這個功能模組的行為有沒有變？安全假設有沒有破？效能指標有沒有退？」

安全掃描也一樣。`deepsec` 的公開設計就是先找安全敏感候選區，再派 Agent 深挖、重驗、輸出可處理結果。OpenClaw 把專案先切好，再針對功能與安全兩條線做回歸檢查，本質上是在替 Agent 建地圖。

沒有地圖的 Agent 很容易迷路。有地圖的 Agent 才能巡邏。

<ClawdNote>這裡不要小看「切模組」。很多 AI 流程失敗，不是模型笨，是任務切得像一顆霸王級水餃。把整個大型單一程式庫丟給 Agent 叫它找錯誤，就像把台北市地圖丟給外送員，說「午餐在裡面，自己找」。功能模組就是地址、門牌、樓層，少了這些，再強的 Agent 也只能在巷口沉思。</ClawdNote>

## 真正的精實，不是人少，是交接少

推文最後一句是：「所有這些自動化讓這個專案可以非常精實地運作。」

這句話值得拆開。精實不是把人砍到最少，也不是幻想 Agent 可以無痛取代團隊。OpenClaw 展示的精實比較像另一件事：讓工作少一點掉在交接縫隙裡。

問題票到 PR，中間少一段等待。PR 到審查，中間少一段排隊。提交到安全檢查，中間少一段「下週有空再看」。效能基準回歸到團隊注意，中間少一段沉默。會議決策到可執行工作產物，中間少一段會後失憶。

這些縫隙平常看不見，但它們是軟體團隊的暗物質。專案不是只被困難技術拖慢，也會被小小的等待、小小的忘記、小小的「誰有空處理一下」拖慢。

Token 成本下降後，最直接的想像是「每個人都有更便宜的寫程式助理」。但這則推文比較激進：便宜的不是個人助理，而是組織裡的背景勞動。讀 PR、掃問題票、分類、去重、重現、錄影、驗效能、盯安全、整理會議後續。這些工作以前太碎、太煩、太不值得自動化；現在如果 Agent 足夠便宜，它們就變得值得。

這也解釋了為什麼外界只盯 AI 花費會看錯。花費本身不是故事，摩擦被搬走才是故事。

## 結語

OpenClaw 的做法不一定能被每個團隊照抄。約一百個 Codex 常駐雲端、Agent 登入第三方服務、會議中自動開 PR、每次提交都做安全審查，這些都需要權限控管、誤報處理、成本監控、文件化產品願景，以及願意把流程拆成機器可巡邏模組的工程紀律。

但它至少畫出了一個很清楚的實驗方向：當 Token 不再是主要限制，軟體開發可能不只是「寫程式碼更快」。更大的變化，可能是團隊開始把所有會讓人類分心、忘記、排隊、重複確認的工作，交給一張常駐 Agent 網路。

以前軟體團隊最稀缺的是工程師時間。接下來更稀缺的，可能是清楚定義工作的能力。

因為只要工作能被清楚定義，就可能被巡邏、被重現、被檢查、被交叉驗證，甚至在會議還沒結束前，就長成第一個 PR。
