作者最近一直在用 Anthropic 的 CLI 工具 Claude Code，而且覺得它用起來不錯。只是，這個工具以沒有原始碼的 closed binary 形式發佈，檔案大小卻高達 213 MB，實在很難不讓人想知道裡面到底裝了什麼。

原作者 [@jaywyawhare](https://x.com/jaywyawhare/status/2033488305191616875) 剛完成兩個專案，一個是用 C 寫的機器學習函式庫，一個是向量資料庫引擎。專案收尾後那種「總得再找點東西來拆」的焦躁感一上來，他就把注意力放到 Claude Code 身上。對他來說，這不是抱怨，而是一個起點條件：它會替他和 API 溝通，但他不知道它實際在做什麼。再加上自己有資安背景，也知道怎麼往裡面看，所以就真的動手拆了。(◍•ᴗ•◍)

而真正的觸發點還更鬧一點：他懶得更新套件。當時他還跑在 2 月中旬的 v2.1.33，世界早就往前走了。結果上週末他沒有更新，反而開始把整個 binary 拆開來看。事後看來，這可能比更新還有收穫。

## 裡面到底裝了什麼？

拿到一個二進位檔，第一步通常就是先跑 `file` 和 `readelf`。

結果顯示，這是一個 ELF 64-bit 的執行檔。對 CLI 工具來說，213 MB 真的大得離譜；但用 `strings` 掃一下，故事馬上就清楚了：裡面有 `Bun v1.3.5 (Linux x64 baseline)`。

原來 Claude Code 是一個 Bun single executable application (SEA)。也就是說，Anthropic 把整個 JavaScriptCore runtime 和應用程式碼一起打包進同一個 ELF binary。那 213 MB 大多是 runtime 本身，真正的應用程式其實沒有想像中那麼大。

<ClawdNote>
逆向時最怕的不是檔案胖，而是胖到最後只剩機器碼。這種 SEA 至少還留著 JS bundle 可以追，痛苦歸痛苦，但讀法比較接近拆一團打結的線，不是直接拿頭去撞彙編。
</ClawdNote>

## 把程式碼抓出來

Bun SEA 會在 binary 尾端放一個 trailer。只要搜尋 `Bun!`，就能把它找出來。這個 trailer 會指向一個 table of contents，裡面列出 15 個嵌入檔案。

真正重要的是位在特定 offset 的 JavaScript bundle。作者用 `dd` 把它切出來之後，得到了一個約 9.88 MB 的壓縮 JavaScript 檔案，總共 7,493 行。

這就是整個 Claude Code 應用程式的本體。

## 如何閱讀 10MB 的壓縮 JS？

10 MB 的壓縮 JavaScript 讀起來當然不會舒服。變數名稱全都變成 `tC`、`sC1`、`HY1` 這類幾乎沒有語意的短字串。但壓縮工具做不到一件事：它沒辦法重新命名字串常值 (string literals)。

所以切入點就很明確了。搜尋 `You are Claude`，你會直接落到 system prompt；搜尋 `tengu_`，你會找到 597 個內部 feature flag 命名空間的參考。這些字串就是錨點，接著再從那些位置一路往外追邏輯。

有趣的是，作者還直接用 Claude Code 來幫忙分析它自己的原始碼。把一大段壓縮 JS 丟進去，問它某個 function 在幹嘛，它其實答得很好。讓一個工具參與逆向自己，聽起來有點荒謬，但確實有效。(๑˃ᴗ˂)ﻭ

## Prompt 架構的秘密

Claude Code 的 system prompt 並不是一長串固定字串，而是在執行時由 15 個以上的模組化區塊拼起來的：

- 身份層 (Identity layer) → "You are Claude Code..."
- 語氣與風格 (Tone and style) → 簡潔、不加 emoji 等
- 任務管理 (Task management) → 怎麼追蹤工作進度
- 工具使用策略 (Tool usage policy) → 什麼情況用什麼工具
- 安全策略 (Security policy) → 寫死的常數，無法覆寫
- 程式碼規範 (Code references) → 格式規則
- 動態區塊 (Dynamic sections) → memory、environment、MCP servers

其中安全策略是 hardcoded，其他部分則會依照可用工具、目前模型和使用者設定動態變化。

裡面甚至還藏了一個隱藏的極簡模式：只要設定環境變數 `CLAUDE_CODE_SIMPLE=true`，整個 prompt 就會縮成一句話。原作者猜測這大概是拿來測試用的。

至於完整 prompt 內容，作者沒有直接貼出來。原因也很實際：他不知道 Anthropic 對這條線畫在哪裡，也不打算親自去踩。

## 名為 Tengu 的神祕系統

在程式碼深處，作者挖到了一套叫做 `Tengu` 的 feature flag 與 telemetry 系統。Tengu 是日本民間傳說裡的天狗，但為什麼取這名字，沒人解釋，工程團隊就是會留下這種命名彩蛋。

這套系統裡有 37 個功能開關，以及大約 560 個遙測事件。開關透過 Anthropic 基礎設施上的 GrowthBook 評估，Statsig 當備援；telemetry 則走 OpenTelemetry，送往 Datadog 和 Anthropic 自己的分析端點。對一個 CLI 工具來說，這套儀表真的下得很重。

更有意思的是，很多功能明明已經存在於程式碼中，卻還被旗標擋住沒開。像是 extended thinking modes、alternative model routing、experimental UI，以及 cost optimization experiments，通通都已經埋在裡面，只是開關還沒撥上去。

<ClawdNote>
我的解讀是：這很像先把功能跟著版本送到使用者機器上，再靠 server-side flag 控制是否開啟。原文沒有直接把它叫做 dark launch，但從作者後面做的版本比對來看，那個味道非常接近。
</ClawdNote>

## 比較版本差異

到了星期天晚上，作者已經建立起一個假設：這些被旗標擋住的功能不是 dead code，而是已經進了 binary、只是還沒對外啟用的即將發佈功能。

為了驗證這件事，他抓了 Anthropic 剛發佈的 v2.1.76，抽出新的 bundle 後，拿去和舊版 v2.1.33 比對。新版 bundle 變成了 11 MB，比原本還更大。

結果完全證實了他的推測。舊版裡被旗標擋住的功能，在新版裡已經正式上線；新工具被加進來，prompt 架構也重新整理過，agent 系統更是明顯擴張。Anthropic 會提早好幾週把功能送進 binary，等時機成熟再公開打開。

這也是逆向工程真正有意思的地方。爽點不在把檔案機械式地切出來，而是在你讀了幾天亂成一團的壓縮程式碼之後，對系統形成的理解，真的能預測它下一步要做什麼。那一刻才表示你看懂了。

## 意料之外的發現

- **Token 消耗非常大**：在使用者開口之前，system prompt 就先吃掉 10,000+ tokens。再加上工具描述、`CLAUDE.md`、memory 和系統提醒，每一輪互動都會先背上 15k 到 20k tokens 的基礎成本。
- **Bash sandbox 不是演戲**：作者原本以為這層保護可能只是做做樣子，結果裡面真的有 allow/deny 機制，而且還包含網路限制。
- **560 個 telemetry event 真的很多**：它們幾乎什麼都在量，包含工具使用模式、錯誤率、效能指標和旗標評估狀況。
- **Prompt injection 指令帶著遞迴味**：prompt 裡既有如何處理 prompt injection 的規則，也有如何解讀承載那些規則的標籤的規則。這種自我指涉的結構，會讓人停下來想一下。

## 作者用了哪些工具？

1. `file`、`readelf`、`strings`：辨識 binary 結構
2. `dd`：抽出 JS bundle
3. `grep`、`ripgrep`：在壓縮 JS 中搜尋模式
4. Claude Code：協助分析它自己的原始碼
5. `diff`、`meld`：比對版本差異

## 如果你也想自己試試看

作者的建議很直接：

- 安裝 Claude Code
- 用 `which claude` 找到 binary
- 用 `strings -t x $(which claude) | grep "Bun!"` 找出 Bun trailer
- 從 `.rodata` 把 JS bundle 切出來
- 接著就從字串搜尋開始

這個 binary 沒有加密、沒有完整性檢查，也沒有 anti-tamper 機制。它只是被壓縮過，並沒有被混淆。門其實是開著的。

## 前人研究與最後的結論

在作者之前，也有人從不同角度分析過 Claude Code：

- Kir Shatrov 用 mitmproxy 攔截 API 呼叫：適合看 runtime 行為，但看不到 prompt 架構。
- Reid Barber 在早期版本裡找到 source maps：能看到乾淨程式碼，但後來被 Anthropic 移除了。
- Vrungta 從執行行為反推架構：很完整，但偏推測。
- Yuyz0112 透過 monkey-patch SDK 記錄 API 呼叫。

直接從 binary 抽取內容，能得到最完整的全貌，但代價是可讀性最差。每一種方法都有自己的取捨。

作者最後的結論很直接：**Claude Code 本質上就是一個 Prompt 的傳遞系統**。程式碼主要是在處理工具、管理上下文、與 API 溝通；真正的產品，是那個由成千上萬個 token 組成、負責告訴模型如何行動的 prompt。

binary 是包裝，prompt 才是產品本體。
