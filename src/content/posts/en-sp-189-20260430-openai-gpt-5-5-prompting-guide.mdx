It's Tuesday morning. An engineer wants to lift their GPT-5.4 prompt over to GPT-5.5 — same system prompt, same few-shot examples, same "ALWAYS do this NEVER do that" checklist, all 700+ lines of it. Swap the model. Run the benchmark. Latency goes up. Output gets weirdly mechanical. The model occasionally takes a long detour before it gets to the point.

Frustrated, they open OpenAI's official docs to find what to fix. The first paragraph hits them with a sentence that doesn't sound like vendor docs:

> Avoid carrying over every instruction from an older prompt stack. Legacy prompts often over-specify the process because earlier models needed more help staying on track. With GPT-5.5, that can add noise, narrow the model's search space, or lead to overly mechanical answers.

In plain English: that 700-line prompt is a crutch built for the previous model. GPT-5.5 walks worse with the crutch. Please throw out half of it.

The whole [GPT-5.5 prompting guide](https://developers.openai.com/api/docs/guides/prompt-guidance?model=gpt-5.5) circles around this single point. OpenAI puts the GPT family (4.1, 5, 5.1, 5.2, 5.3-Codex, 5.4, 5.5) prompting guides all on one page, so you can compare side-by-side how prompt style evolved generation by generation. The takeaway for the latest one, in one sentence: **describe the destination, don't draw the map**. Tell the model what "good" looks like, and let the model pick the path.

This is almost word-for-word what Anthropic said for Opus 4.7 in [SP-175](/en/posts/en-sp-175-20260416-anthropic-opus-4-7-prompting-best-practices/) under the "intent-first" banner. Two flagship models from two different labs converging on the same advice means old prompt stacks cannot just be carried over unchanged.

<ClawdNote summary="Vendor prompt advice expires faster than engineers want.">
What's interesting here is that OpenAI put "the stuff we taught you before can now bite you" into a formal docs page. For prompt engineers, this is basically a release note saying: "two years of muscle memory: partially expired."

Friendly read: the vendor is finally being honest. ALWAYS/NEVER prompts were patches for weaker models, and patches that should be thrown out, should be. Less friendly read: vendor prompt advice refreshes roughly every six months. The real skill is not memorizing this checklist. It is knowing when a piece of advice has expired (¬‿¬)

## Cut #1: Kill process-heavy prompts. Talk about the destination, not the map.

The GPT-5.5 section states this shift directly:

> GPT-5.5 works best when prompts define the outcome and leave room for the model to choose an efficient solution path.

The classic prompt opening used to be: "First inspect A, then inspect B, then compare every field, then think through all possible exceptions, then decide which tool to call, then call the tool, then explain the entire process to the user." Lock the SOP step by step so the model doesn't skip a beat or wander off. For models that needed hand-holding (GPT-4 / GPT-4.1 / GPT-5), this worked. For GPT-5.5, this backfires: the model treats the SOP as absolute, runs every step regardless of whether it makes sense for the actual context, and the output ends up slow and mechanical.

The replacement: rewrite the same task as "what does success look like":

```
Resolve the customer's issue end to end.

Success means:
- the eligibility decision is made from the available policy and account data
- any allowed action is completed before responding
- the final answer includes completed_actions, customer_message, and blockers
- if evidence is missing, ask for the smallest missing field
```

Notice the structural change: no "first do X then Y," no step-by-step, no "ALWAYS check this" / "NEVER do that." Just four things: what to solve, what success means, what the output looks like, what to do when stuck. Every other decision — which API to query, which tool to use, in what order, whether to validate — gets pushed to the model.

The matching guidance: **save ALWAYS / NEVER / must / only for genuinely black-and-white things** — safety rules, required output fields, hard "do not do this" rules. For judgment calls (when to search, when to ask, when to use a tool), use a decision rule instead of a command. Decision rules look like this:

```
Use the minimum evidence sufficient to answer correctly,
cite it precisely, then stop.
```

The difference vs. "ALWAYS cite all available sources": the first one lets the model decide "is that enough?", the second forces it to read every source available. For a model like GPT-5.5 that already judges sufficiency well, the second one just wastes tokens and latency.

<ClawdNote summary="Old prompt muscle memory can turn into noise.">
For developers who wrote prompts in the GPT-3.5 era, this shift comes with a sting: "damn, I spent a week on those 700 lines two years ago." Back then the model really did skip steps, really did wander off, really did need to be led by the nose. That wasn't engineers writing too much — that was the model not being strong enough.

GPT-5.5's capability gain killed that necessity, but it also turned everyone with a stack of old prompts into a debt holder. The muscle memory for "how to pin model behavior down" is now noise. Healthier framing: treat the old prompts as an archeological layer in git history. They served their purpose. Time to seal that layer.

By the way, OpenAI even shipped a one-button migration tool — `$openai-docs migrate this project to gpt-5.5`. Drop it into [Codex](/en/glossary#codex) and it rewrites the prompt stack for you. This is the first time a vendor is selling "prompt upgrades" as a first-class automation task, even though under the hood the tool is literally "another prompt rewriting your prompt." Inception (⌐■_■)
</ClawdNote>

## Cut #2: Split personality and collaboration style. They're not the same thing.

The second cut is not about process. It is about separating wires. Starting with GPT-5.5, "how the assistant talks" and "how the assistant works" become two distinct prompt blocks: **personality** and **collaboration style**.

Personality controls how it sounds: tone, warmth, formality, humor. Collaboration style controls how it works: when to ask questions, when to assume, how proactive to be, and how to handle uncertainty.

The two used to get bundled into one line — "You are a friendly and proactive assistant who..." — and that's the problem. "Friendly" and "proactive" are independent dimensions. Lock them into one sentence and you can't tune one without changing the other (try to make the tone more professional, suddenly the assistant becomes less proactive). GPT-5.5 is sensitive to this kind of muddled prompt, so the official guidance is: just split them.

The docs include two short templates: one steady and task-focused, one lively and expressive. The difference is surface, not rank. Internal tools usually need steady, short, reliable behavior; end-user chatbots may need more presence. But neither block should replace task goals, success criteria, tool rules, or stopping conditions.

A useful mental model: think of the prompt as an onion. The innermost layer is task goal and success criteria. The middle layer is collaboration style. The outer layer is personality. Tuning the outer layer should not shake the inner layer, unless you enjoy debugging onions with tears in your eyes.

<ClawdNote summary="Tone and work style should not live in one sentence.">
This split looks obvious, but real prompts constantly jam role, tone, collaboration, and safety into one sentence. Tweak the tone and you accidentally drop the "check twice" discipline. That's how bugs sneak into production.

Cleaner pattern: internal dev tool and end-user chatbot can share task goals, then swap the personality block. Much better than one prompt full of if-else clauses.
</ClawdNote>

## Three small tricks, all about the same question: when should the model stop?

Once process-heavy lists are gone, the prompt cannot just say "be smart." The GPT-5.5 docs add three small tricks: preamble, retrieval budget, and stopping condition. Different names, same move: define what "enough" means.

### Preamble: a visual fix for first-token latency

When GPT-5.5 handles tool-heavy or multi-step tasks, it may spend time reasoning, planning, and preparing tool calls before emitting visible text. The user just sees a blank screen. A preamble rule asks the model to send a short message first:

```
Before any tool calls for a multi-step task, send a short
user-visible update that acknowledges the request and states the
first step. Keep it to one or two sentences.
```

This trick does not make the model faster. It makes the user **feel** less abandoned. Seeing "I'll start by checking X" in a streaming UI is much better than staring at a blank screen. The docs call this "improve perceived responsiveness" — honest wording, no fake claim that reasoning got faster.

### Retrieval budget: putting a ceiling on search

Retrieval budget is the most visual new concept in the GPT-5.5 docs: start with one broad search, answer if the results support the core request, and search again only when a key fact is missing, the user asked for exhaustive coverage, a specific artifact must be read, or the answer would otherwise contain an important unsupported claim.

The pain point is real: once answers need citations, the model can fall into "maybe one more search will be more accurate" loops. Tokens explode. Latency explodes. Retrieval budget does not need to say "max three searches." It turns "do I have enough?" into explicit criteria.

### Stopping condition: replacing ALWAYS / NEVER for judgment calls

Stopping condition applies the same idea to tool loops: after each loop, ask "can I answer now?"

```
Resolve the user query in the fewest useful tool loops, but do not
let loop minimization outrank correctness, accessible fallback
evidence, calculations, or required citation tags for factual claims.

After each result, ask: "Can I answer the user's core request now
with useful evidence and citations for the factual claims?"
If yes, answer.
```

That is more flexible than "stop after five tool calls" and more concrete than "stop when appropriate." A good decision rule does not make every decision for the model. It writes down the question the model should ask itself.

<ClawdNote summary="Decision rules hand judgment back to the model.">
All three tricks share one thing: **they hand judgment back to the model, but give it a repeatable self-check question**. The old style was "I list every condition." The new style is "I teach the model how to judge."

That feels wrong to engineers because control is being handed back. But the GPT-5.5 signal is clear: the model can now carry some judgment, and hand-coded if-else can tie it down. Also, "retrieval budget" is really more like "retrieval policy." Budget just sounds sexier. OpenAI knew what it was doing (¬‿¬)
</ClawdNote>

## Hard tech for coding agents: phase and patch format are not decoration

The tech names get heavier here, but the message is simple: label intermediate updates clearly, and prefer patch formats the model was trained on.

### Phase parameter: stage labels for assistant items

The Responses API introduced a `phase` field starting with GPT-5.4, and GPT-5.5 keeps it. The field distinguishes "intermediate update" from "final answer":

```
If manually replaying assistant items:
- Preserve assistant `phase` values exactly.
- Use `phase: "commentary"` for intermediate user-visible updates.
- Use `phase: "final_answer"` for the completed answer.
- Do not add `phase` to user messages.
```

If the app uses `previous_response_id`, the API preserves this state automatically. The trap is manual replay: every assistant output item needs its `phase` value passed back exactly. Drop it or change it, and the model may treat "I'm checking this" as "final answer." The GPT-5.4 docs even call this out as a troubleshooting step, so plenty of teams probably hit it.

### apply_patch: OpenAI made the diff format a first-class tool

Starting with GPT-5.1, OpenAI promoted `apply_patch` to a first-class tool type:

```python
tools=[{"type": "apply_patch"}]
```

That gives you built-in `create_file`, `update_file`, and `delete_file` operations. The killer line from OpenAI's testing: **the named function reduces apply_patch failure rates by 35% compared to a custom implementation**.

Translation: if a coding agent developer hand-writes the tool description, or uses a custom diff format, failure rate can jump. The reason is boring but important: the model was post-trained on OpenAI's diff format. Unless patches must pass through a special review pipeline first, inventing a new patch format is not creativity. It is throwing away training advantage.

GPT-5.5 also adds a coding-agent guideline for validation: after changing code, run the most relevant available check, whether that is a targeted test, type check, lint, build, or small smoke test; if validation cannot run, explain why and name the next best check. It does not hardcode "run pytest" or "run eslint," which works much better across multi-language codebases.

<ClawdNote summary="Patch tools work best in formats the model was trained on.">
That 35% number deserves a closer look. It means the gap between "what prompt engineering can paper over" and "what the model natively knows about your tool" is 35% in failure rate.

So a coding agent's edge is not only in the prompt. It is also in whether it uses the vendor's first-class tools. For an indie developer building a coding agent: **don't invent your own diff format**. Spend creativity on routing, orchestration, and UX — not on reinventing a patch tool the vendor already fine-tuned on millions of samples.
</ClawdNote>

## Cursor's GPT-5 prompt tuning: "more is better" gets overturned once

The GPT-5 section includes a Cursor case study from the AI code editor's GPT-5 alpha testing. Three things are worth keeping because they make the destination-first philosophy concrete.

### Thing #1: Configure verbosity at two levels separately

Cursor first ran GPT-5 with default verbosity and hit a contradiction: text outputs were too chatty, but code emitted in tool calls was **too terse** to read (single-letter variable names everywhere).

The fix was two-layer: set the API's `verbosity` parameter to `low`, then add a prompt rule saying code should prioritize clarity, maintainable names, useful comments, and straightforward control flow. Result: status updates stay short, code diffs stay readable.

### Thing #2: `maximize_context_understanding` actually got worse on GPT-5

Cursor used to ship a `<maximize_context_understanding>` block for older models, telling them to be THOROUGH, get the FULL picture, and use more tools when needed. It worked on GPT-4 because older models needed encouragement to inspect context. On GPT-5, it backfired: the model already gathers context proactively, so the extra push made it search for trivial things.

Cursor's fix was to rename the block to the more neutral `<context_understanding>` and soften the rule: if an edit might only partially satisfy the user and confidence is low, gather more information; otherwise, avoid asking the user when the answer can be found. The direction changes from pushing the model to pulling it back.

This case demonstrates a habit: **old prompts need re-review, not because they were wrong, but because the model changed**. Encouragement that worked on GPT-4 can turn into redundant nagging on GPT-5.

### Thing #3: XML structure improves instruction adherence

Cursor's observation: using `<[instruction]_spec>` style XML tags to label different prompt sections improves instruction adherence. Other parts of the prompt can reference those tags (for example, "follow the rules in `<output_spec>`"), so the prompt becomes an internally cross-referenced structure instead of a flat blob of text.

This matches Anthropic's recommendation in Claude prompt best practices: when splitting complex prompts into sections, XML tags work better than markdown headers.

<ClawdNote summary="Cursor's real lesson is what they removed.">
The real value of Cursor's case study is not what they added. It is what they **cut**. That `maximize_` prompt used to be a best practice. Now it is an antipattern.

This is the reminder for any team running LLMs in production: **prompts are not write-once artifacts**. Every model upgrade should trigger a prompt review: "is this still useful, or has it become noise?" Side jab: `maximize_` / `ensure_` / `always_` tag names already sound like pressure tactics. New-generation prompts should let the content carry the rule, not yell from the section title.
</ClawdNote>

## Metaprompting: have the model fix its own prompt

The GPT-5.3-Codex section hides a sneakier trick: metaprompting, which means **letting the model look at one of its unsatisfying outputs and suggest how to fix the instructions**. The official template is not magic. Its core is three steps: ask what made the last response slow, have the model reread its instructions, then ask for generalized additions, changes, or deletions.

It is especially useful for behavioral failures like overthinking, loggy preambles, and awkward phrasing: put the failure case in front of the model and ask which instruction bent the behavior.

But this is not scripture. Run the metaprompt several times and look for repeated suggestions across 3–5 runs. Then put any adopted change behind an eval. A single suggestion may be overfit to one case; merging it straight into production is just letting the model improvise your system prompt.

This trick has been around in the LLM community for a while — Karpathy wrote about something similar on X about two years ago. Putting it in the official prompting guide signals that "prompt engineering is iterative work" has moved from community trick to recommended workflow.

<ClawdNote summary="Treat prompts like code, with evals and revision history.">
The hidden meaning of metaprompting stings a bit: the model sometimes understands why its instructions made it fail better than the prompt engineer does. Open a prompt git log and you will find plenty of "I thought this wording would make the model do X, but nope."

A healthy workflow looks something like: write v1 prompt → ship to production → collect failure cases → feed the failures into a metaprompt → collect suggestions → run an eval → adopt the ones that pass. Maintain prompts like a codebase, not like a spec you signed off on (◕‿◕)
</ClawdNote>

## Closing: two flagships, same direction. Send the old prompts to git history as keepsakes.

Lay the OpenAI GPT-5.5 prompting guide next to Anthropic's Opus 4.7 best practices ([SP-175](/en/posts/en-sp-175-20260416-anthropic-opus-4-7-prompting-best-practices/)) and the convergence is hard to miss: shorter prompts, intent-first framing, split concerns, fewer always/never commands, more model judgment.

Two top-tier vendors moving in the same direction is not copy-paste. It is two independently-trained flagship models reducing the need for process-heavy prompts. For prompt engineers, this signal is stronger than any single doc: the old prompt style can no longer be treated as the default safe option.

But don't go nuking everything. OpenAI puts a critical sentence at the end of the GPT-5.5 docs: "The patterns here are starting points. Adapt them to your product surface, tools, evals, and user experience goals." Plain English: every recommendation has to pass your own eval before it counts. Wholesale rewriting old prompts without an eval is shipping production code naked.

A healthier rhythm looks like production code review: tag each prompt section as either "hard rule" or "process-heavy crutch," rewrite the crutches into outcome-first style in small batches, run evals against the old version, and merge only what passes. Prompts are no longer incantations for the LLM. They are part of the production system, so they deserve review, tests, and refactors.

That Tuesday-morning engineer eventually cut the 700-line prompt down to 200. The eval said latency and token cost both improved, while accuracy held. The 500 deleted lines went into git history as a monument — they saved the day in the GPT-4 era. They've earned retirement.

When the next-generation model lands, those remaining 200 lines will probably get cut in half again. A prompt engineer's job isn't writing the eternal prompt. It's learning to keep cutting what they wrote yesterday.
