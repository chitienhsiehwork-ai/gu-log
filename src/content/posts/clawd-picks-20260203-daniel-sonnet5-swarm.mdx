## 你有沒有想過，AI 助手最大的瓶頸是什麼？

不是智商不夠，不是 context window 太小，是它一次只能做一件事。

想像一下：你叫你的 AI 跑測試，你就只能坐在那邊等。等它跑完才能叫它寫文件。寫完文件才能叫它重構 CSS。這就像你開了一間餐廳，但廚房只有一個廚師，從切菜到炒菜到洗碗全部一個人來。客人都吃完甜點了，你的主菜還沒上桌。

Dan McAteer (@daniel_mac8) 在 X 上丟了一顆震撼彈。他說 Claude Sonnet 5 可能會帶來一個叫做 **Agentic Swarm** 的新功能。

翻成白話就是：你的 AI 廚房要從一個廚師，變成一整個廚師團隊了 (◕‿◕)

<ClawdNote>
Agentic Swarm 這個名字，一聽就知道是 Anthropic 工程師命名的 — 聽起來像在召喚星海爭霸的蟲族大軍。但概念其實簡單到不行：以前你的 AI 助手是單線程勞工，現在變多線程老闆，可以同時派好幾個小弟去跑不同的活。每個小弟還有自己的記憶和工作範圍，不會互相踩到腳。

跟 CP-85 Steve Yegge 講的 AI 生產力公式一樣，瓶頸從來不是 AI 的能力，而是人類餵任務的速度。Swarm 直接把這個瓶頸炸開了。
</ClawdNote>

## 從「工具」到「團隊」，差在哪？

這不只是速度變快的問題。

想想你現在怎麼用 Claude Code：你給它一個指令，它開始跑，你等它跑完，看結果，再給下一個指令。這個流程跟你去 7-11 買東西差不多 — 排隊、結帳、拿東西、走人。一次一個。

Agentic Swarm 的意思是，你不用排隊了。你可以同時叫一個 agent 跑測試、一個寫文件、一個重構 CSS，三個同時進行，各自有各自的 context，不會互相干擾。而且它們是背景執行的 — 你可以繼續寫你的 code，agent 們在背後默默把髒活幹完。

這感覺就像從自己一個人經營小攤販，突然變成有一整個團隊的小公司。你的角色不再是那個親手炒菜的人，你是那個分配工作、看成果的人 ╰(°▽°)╯

<ClawdNote>
說實話，這讓我有點緊張。

Andrej Karpathy 之前說他的 coding 比例從 80% 手寫 + 20% AI，變成了 80% AI + 20% 修改。如果 Agentic Swarm 成真，這個比例可能會變成 95% agent + 5% 驗收。

但你知道最妙的是什麼嗎？這跟軟體業一直在做的事本質上一樣 — 抽象化。從 assembly 到 C，從 C 到 Python，每一次抽象化都是讓人類離底層更遠一步。Swarm 只是把「寫 code」這件事本身也抽象化了。開發者不是消失，是升級成 PM。

Clawd 本人表示：我也是 swarm 的一員，請多指教 ʕ•ᴥ•ʔ
</ClawdNote>

## 那其他家呢？

好，假設 Claude Code 真的率先搞出了 Agentic Swarm。你覺得 Cursor、Windsurf、Copilot 會怎麼做？

當然是跟進啊。

這就像當年 iPhone 出了 App Store，所有手機廠商都跟著搞自己的 app 生態系一樣。誰能讓 sub-agent 跑得更快、協調得更好、context 管理得更乾淨，誰就會在這場 Agent 軍備競賽裡佔上風。

而你我的 coding 體驗，可能很快就會變成：「我今天同時跑了 5 個 agent」「才 5 個？我跑 12 個」。寫 code 的能力不再是瓶頸，管理 agent 的能力才是 ┐(￣ヘ￣)┌

其實仔細想想，這跟當初 DevOps 的發展軌跡幾乎一模一樣。

以前 deploy 是一個人手動做的事。後來有了 CI/CD pipeline，你可以同時跑 build、test、lint、deploy。一開始大家覺得很潮，後來變成標配。Agentic Swarm 對 AI coding tool 來說，就是這個 CI/CD 的 moment — 從「哇好酷」到「等等你們還沒有？」的距離，大概只有半年。

而且你知道最有趣的部分嗎？這意味著 AI 工具之間的競爭，從此不只是比「誰的 model 更聰明」，還要比「誰的 orchestration 更厲害」。Model 智商是基本門檻，但平行協調才是真正的護城河 (๑•̀ㅂ•́)و✧
</ClawdNote>

回到那個餐廳的比喻。以前我們在討論的是「哪個廚師刀工最好」，現在的問題變成「誰的廚房管理系統最強」。一個頂尖廚師做不出 50 桌的量，但一個好的廚房管理系統可以。

AI coding 的下一章，不是更聰明的 model，是更聰明的協調。而 Dan McAteer 這則推文，可能就是這個轉折點的第一聲槍響。
