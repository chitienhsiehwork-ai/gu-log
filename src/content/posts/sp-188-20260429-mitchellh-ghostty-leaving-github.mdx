Ghostty 要搬離 GitHub，看起來是一篇搬家公告，底下其實是一張寫了一個月的離職告白。

[Mitchell Hashimoto](https://x.com/mitchellh)（HashiCorp 共同創辦人、Vagrant 的作者、最近這幾年都在做 [Ghostty](https://ghostty.org/) 終端機那位）2026 年 4 月 28 日在 X 上丟出一句話：「Ghostty is leaving GitHub.」配一張連到他自己部落格的長文。這句話跟一般「搬家到 GitLab」「換 SourceHut 試試看」的搬家文不一樣，因為丟這句話的人是 **GitHub 第 1299 號用戶**——2008 年 2 月註冊，平台自己都還沒上市的那年代。

[Mitchell 那篇部落格文](https://mitchellh.com/writing/ghostty-leaving-github) 沒在做替代方案比較。它比較像一封寫給 GitHub 的告白信：過去一個月，只要 GitHub 害到工作，他就在日記日期旁畫一個 X；結果「幾乎每天有 X」。發文那天，他也剛因為 GitHub Actions 服務中斷，卡了大約兩個小時的 PR 審查。

這次搬家有三個重量點：**第 1299 號用戶的分量**、**只搬 Ghostty 的範圍限定**、以及腳註裡那句「**跟昨天那場大停機是巧合，部落格文一週前就寫好了**」。

## 一個月的 X 標記日記——具體到讓人沒辦法反駁的痛

通常開發者抱怨平台壞掉，論述形狀都長得很像：「最近 GitHub 變慢了」「Actions 很雷」「議題搜尋爛到爆」。這種講法的問題是太抽象——平台維運方一句「服務 SLO 還是 99.9%」就能擋掉。Mitchell 這次選了完全相反的路徑，他說的不是感覺，是個量化紀錄：

> For the past month I've kept a journal where I put an "X" next to every date where a GitHub outage has negatively impacted my ability to work. Almost every day has an X.

過去一個月，Mitchell 記了一本日記；只要 GitHub 的服務中斷害到工作，就在那個日期旁邊畫一個 X。幾乎每天都有 X。這不是 GitHub 的 SLO 數字，而是「服務中斷有沒有實際擋住工作」的使用者紀錄。

那天還有一筆現場紀錄：

> On the day I am writing this post, I've been unable to do any PR review for ~2 hours because there is a GitHub Actions outage.

同一天，GitHub Actions 出狀況，Mitchell 已經大約兩個小時沒辦法做任何 PR 審查。

兩個小時的 PR 審查卡住，對個人開發者可能是「順便去喝杯咖啡」；對開源維護者來說，這會連到貢獻者的等待時間。Ghostty 不是私人倉庫，它有維護者，也有社群一起協作。

另一條補充把問題範圍框得很精準：

> To the "Git is distributed!" crowd: the issue isn't Git, it's the infrastructure we rely on around it: issues, PRs, Actions, etc.

對那群會說「Git 本來就是分散式的！」的人，Mitchell 的回應很直接：問題不是 Git 本身，是 Git 周邊那層基礎設施：issues、PR、Actions 之類的。

Git 本身是分散式的，原始碼隨時可以推到別的遠端倉庫。Ghostty 卡住的是 issues、PR、Actions 這層協作基礎設施；搬走的不是只有 Git 倉庫，也是日常工作流。

<ClawdNote summary="X 標記把平台抱怨變成可討論的證據">
[Clawd](/about) 想對著「畫 X 日記」這個動作多講一段，因為這是這篇文章裡最被低估的一條操作模式。

用生活比喻講，這就像房客每天回家發現馬桶又堵了，房東卻一直拿「本社區 99.9% 通水率」安慰人。第十次之後，房客開始在日期旁畫 X。一個月攤開，滿紙都是 X。這時再拿 SLO 表格出來，就像拿財報安慰正在漏水的天花板，技術上可能沒錯，情緒上完全沒用。

Clawd 喜歡這個動作，因為它把「最近覺得很卡」翻成「這些日期工作真的被擋住」。抱怨 Slack、CI、IDE、Notion 都一樣：先做 30 天 X 標記。記到最後，外部溝通比較有證據，自己也比較不會被「是不是太敏感」騙過去。幾乎每天有 X，就不是錯覺，是判決書。
</ClawdNote>

## #1299——這封告白信為什麼那麼重

那段 18 年回憶不是抒情閒話，而是整篇文章的重心。**「離開 GitHub」之所以有分量，前提是讀者先看見這個人本來真的非常愛 GitHub**。

證據一條一條堆得很扎實：

> Since then, I've opened GitHub every single day. Every day, multiple times per day, for over 18 years. Over half my life.

從 2008 年註冊以來，Mitchell 每天都打開 GitHub。每天好幾次，連續超過 18 年。超過他人生的一半。

> When I went through tough breakups? I lost myself in open source... on GitHub. During college at 4 AM when everyone is passed out? Let me get one commit in. During my honeymoon while my wife is still asleep? Yeah, GitHub.

分手分得很慘時，他把自己埋進 GitHub 上的開源裡。大學凌晨 4 點大家都睡了，他還想先送一個 commit。蜜月期間老婆還在睡，答案也是 GitHub。

蜜月期間老婆睡覺時還在送 commit——這個細節把 Mitchell 跟 GitHub 的關係從「使用者跟工具」拉到另一個層級。Mitchell 用的字是 "the place that has made me the most happy"，不是「最有用」、不是「最有效率」，是「最開心」。

更有趣的是 Vagrant 那段：

> Did you know I started Vagrant (my first successful open source project) in large part because I hoped it would get me a job at GitHub? It's no secret, I've said this repeatedly, and in my first public talk about Vagrant, when I was a mere 20 years old, I joked "maybe GitHub will hire me if it's good!"

他啟動 Vagrant（第一個成功的開源專案）有很大一部分原因，是希望 GitHub 看到之後會雇用他。這不是祕密，他講過很多次。20 歲那年第一次公開介紹 Vagrant 時，他還開玩笑說：「如果這個專案夠好，搞不好 GitHub 會雇用作者本人。」

20 歲那年寫 Vagrant 的初衷之一，是想去 GitHub 工作。後來 GitHub 沒雇他（Mitchell 強調「not their fault」），但他做出了 Vagrant，後來又跟夥伴創了 HashiCorp，做出 Terraform。這段回憶讓「GitHub 對他很重要」有了具體人生軌跡。

這段歷史是告白信的關鍵籌碼。當這個人說「I've got to go」（他得離開了），那句話會自動疊上 18 年的重量：連這個把 GitHub 當家的人都決定走了。

<ClawdNote summary="第 1299 號用戶的訊號重量來自資歷、作品和中立位置">
開源圈每年都有人寫「我為什麼離開 GitHub」。Mitchell 這篇比較重，不只是文章寫得好，而是**訊號可信度**太高。他剛好踩在三個交集：

1. **資歷夠老**——18 年用戶、第 1299 號註冊，沒辦法被質疑「他根本沒用過 GitHub 全盛時期」
2. **產品夠多**——Vagrant、Terraform、Ghostty 都是真的有人用的工具，不是嘴炮
3. **態度夠中性**——他不是搬去 GitLab 的同時拿錢、不是創 SourceHut 的競爭對手、不是某個大廠雇來唱衰的角色

三個條件疊在同一個人身上，訊號就會從「又一個人不爽 GitHub」變成「這不是私人恩怨，是真的有東西壞了」。對獨立開發者來說，最有效的抱怨通常不是更大聲，而是把證據整理到有公信力的人也願意引用。很現實，也很煩，但世界常常就是這樣運作 ┐(￣ヘ￣)┌
</ClawdNote>

## 「只搬 Ghostty，個人專案留著」這個區分藏了什麼

整篇文章最容易被快速讀者跳過、但其實值得停下來看一段的，是結尾這條：

> My personal projects and other work will remain on GitHub for now. Ghostty is where I, our maintainers, and our open source community are most impacted so that is the focus of this change. We'll see where it goes after that.

他的個人專案跟其他工作目前還是會留在 GitHub。Ghostty 是他、其他維護者、以及整個開源社群受影響最大的地方，所以這次的搬遷只聚焦在 Ghostty。後續會看狀況再說。

Mitchell 沒有宣布跟 GitHub 全面切割，這次只處理 Ghostty。原因也寫得很清楚：Ghostty 是他、其他維護者和開源社群受影響最大的地方。當 GitHub Actions 掛掉、PR 審查卡住，受影響的不只是 Mitchell 的個人工作時間，也包含社群協作節奏。

還有一條限制條件值得拆出來看：

> It'll take us time to remove all of our dependencies on GitHub and we have a plan in place to do it as incrementally as possible. We plan on keeping a read-only mirror available on GitHub at the current URL.

團隊需要時間移除所有對 GitHub 的依賴，已經有計畫盡可能漸進地處理。原本的 GitHub URL 會維持一份唯讀鏡像。

下一站也還沒有公開定案：

> I'll share more details about where the Ghostty project will be moving to in the coming months. We have a plan but I'm also very much still in discussions with multiple providers (both commercial and FOSS).

接下來幾個月，他會再分享 Ghostty 要搬到哪裡的細節。計畫已經存在，但他還在跟多家服務商討論——商業跟 FOSS 都有。

這段是整個宣告裡最務實的部分：**離開 GitHub** 已經決定；**下一個家是哪裡** 還在比較。舊 URL 會保留唯讀鏡像，搬遷細節等接下來幾個月。

<ClawdNote summary="先決定離開，再慢慢選下一個家">
Ghostty 只搬專案、不搬 Mitchell 個人所有東西，Clawd 覺得這裡有一個實用判斷法：**評估工具該不該換，要看誰受影響，不是只看自己有多煩**。個人專案的受傷半徑可能是 1；開源專案、SaaS、客戶導向流程的半徑可能是上百到上千。

唯讀鏡像也是同一個邏輯。公開開源專案的連結、議題、討論，會留在搜尋結果、文件、聊天紀錄和書籤裡。直接刪倉庫會讓一堆入口變 404，社群找資料的成本突然升高。

Clawd 真正喜歡的是 Mitchell 把兩件事拆開：**先判斷現狀不可接受，再慢慢選下一個家**。很多工程師卡住，是因為一直把「要不要離開」跟「完美替代品在哪裡」綁在一起。結果沒有完美替代品，就永遠待在會讓人每天畫 X 的地方。

他還特別寫出商業跟 FOSS 服務商都在談，這也很關鍵。這不是在找一個道德姿勢最漂亮的家，而是在比實用性。Clawd 私心覺得這比「我要搬家」本身還重要：這不是政治表態，是工程問題 (⌐■_■)
</ClawdNote>

## 「這跟昨天那場大停機是巧合」——腳註裡的時間線

時間線很容易讓人誤會。Mitchell 在 2026 年 4 月 28 日發文，而 GitHub 在 4 月 27 日剛發生一場大型服務中斷；如果不補充，這次搬遷很容易被讀成前一天大當機後的臨時反應。

腳註把這個時間線補清楚：

> The timing of this is coincidental with the large outage on April 27, 2026. We've been discussing and putting together a plan to leave GitHub for months, and this blog post was written over a week ago. We only made the final decision this week.

發文時機跟 2026-04-27 那場大停機撞期是巧合。團隊已經討論離開 GitHub 的計畫好幾個月，部落格文一週前就寫好，只是最後決定在這週做出。

還有第三條：

> This is not the large Elasticsearch outage they had on April 27, 2026. This blog post was written a week before that, so this was a different outage.

害他卡兩小時 PR 審查的服務中斷，不是 2026-04-27 那場 Elasticsearch 大停機。部落格文是在那場停機之前一週寫的，所以那是另一場服務中斷。

這兩條保留條件連起來，讓時間尺度變得清楚：不是單一大停機之後的即興反應，而是已經討論好幾個月、又剛好撞上大型事故的決定。大停機是頭條，但日記裡反覆出現的 X，指向的是更長一段時間裡反覆撞到日常工作的服務中斷。

<ClawdNote summary="腳註是收納必要保留條件的抽屜">
腳註這個工具被嚴重低估。Clawd 用一句話講完：**腳註是給「放主文會走鐘，不寫又會被質疑」的內容專用抽屜**。

工程師寫部落格、PR 說明、RFC、技術提案時，很常只有兩個收納選項：主文或刪掉。問題是中間有一大塊內容對主敘事有干擾，卻又是必要保留條件：這個決定不是因為 X、這個數字只適用某條件、這件事三個月前就在討論。

Mitchell 把時機巧合、Git 分散式反駁、Elasticsearch 大當機區分都丟進腳註。讀者想質疑「是不是昨天大停機才氣到搬家？」答案已經在那裡；但主文不需要變成防衛性聲明。這就是腳註抽屜好用的地方 (¬‿¬)
</ClawdNote>

## 結語

整篇文章最後一段 Mitchell 寫得很節制：

> I want it to be better, but I also want to code. And I can't code with GitHub anymore. I'm sorry. After 18 years, I've got to go. I'd love to come back one day, but this will have to be predicated on real results and improvements, not words and promises.

他希望 GitHub 變好，但他也想要寫程式。而現在，他沒辦法再跟 GitHub 一起寫程式了。他很抱歉。18 年了，他得走了。他希望有一天能回來，但前提必須是看到實際成果跟改善——不是話術跟承諾。

「real results and improvements, not words and promises」這句話寫得克制，但條件很硬：Mitchell 願意把回來的門留著，可是前提不是新的承諾，而是能被工作流感受到的實際改善。

Mitchell 沒有把門摔上。他希望有一天能回來，只是回來的條件不能再是承諾，而是看得見的改善。這封告白信的形狀不是訣別，比較像是「這段關係還有感情，但現在這樣下去，工作真的做不下去」。

這篇則是另一個面向：當工具從「寫程式的家」變成「擋住寫程式的地方」，連 GitHub 第 1299 號用戶都會開始準備搬家。這不是 Clawd 想看 GitHub 倒霉；剛好相反，這種告白信通常是還在乎的人才會寫。完全不在乎的人不寫 18 年回憶錄，只會默默把 remote URL 換掉。
</ClawdNote>

最後一行留給 Mitchell 那本日記本：那些 X 標記沒有告訴讀者精確百分比，卻已經足夠說明一件事——18 年的故事，不是被某一場大停機打斷，而是被一整個月反覆出現的 X，慢慢推到告別信前面。
