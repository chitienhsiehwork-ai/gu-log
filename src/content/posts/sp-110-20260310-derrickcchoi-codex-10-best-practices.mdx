你有沒有那種經驗——公司新來了一個超強的實習生，技術底子不錯，什麼都會一點，但每次交代任務都要從頭解釋一遍？你說「幫我改那個 API」，他就真的只改了 API，test 沒跑、lint 沒過、PR description 是空的。不是他笨，是你沒跟他說清楚「我們這邊做事的規矩」。

Coding agent 就是這種實習生。而且是一種永遠不會「自己搞懂潛規則」的實習生。

Derrick Choi 最近在 X 上分享了他使用 Codex 的心得，核心想法是：與其每次都花力氣寫一大段 prompt 來教它做事，不如一開始就把「做事的規矩」寫好，讓它自己去讀。

這個思路聽起來很像什麼？對，就是帶新人的 SOP (◕‿◕)

## 先把話說清楚：Context 是一切的基礎

想像你去看醫生。你走進去說「我不舒服」，然後就看著醫生，期待他立刻給你開藥。醫生會怎麼樣？他會開始問你一百個問題：哪裡不舒服？什麼時候開始的？有沒有其他症狀？

Codex 也是這樣。它其實很聰明，但如果你只丟一句「幫我修 bug」，它就得猜。猜哪個 bug、猜在哪個檔案、猜你希望用什麼方式修。通常它猜得還不錯——但在大型 codebase 或高風險任務裡，「猜」這件事本身就是風險。

Choi 建議每個好的 prompt 都應該包含四個要素：你要什麼（Goal）、相關的檔案和 context（用 `@` tag 指定）、限制條件（Constraints），以及怎樣才算做完（Done when）。

就像去看醫生的時候，你應該直接說：「我右邊偏頭痛三天了，之前沒有過，吃過普拿疼沒什麼用，我想知道需不需要進一步檢查。」——目標清楚、context 充足、限制條件和期望值都有。

<ClawdNote>
這四個要素的甜蜜點其實就是 debug 思維：你在寫 bug report 的時候不是也會寫 expected behavior、actual behavior、steps to reproduce 嗎？Prompt engineering 說穿了就是寫 spec 的能力換了個殼。所以那些嫌棄 PM 寫的 ticket 太模糊的工程師們，小心別用同樣的方式對待你的 agent 啊 ┐(￣ヘ￣)┌
</ClawdNote>

另外，Codex 有一個 reasoning level 可以調——Low 適合快速明確的小任務，Medium/High 適合複雜的 debug，Extra High 適合那種需要深度推理的大工程。就像考試的時候，選擇題不需要打草稿，但證明題你最好寫詳細一點。

## 遇到難題？先畫地圖再出發

如果你要從台北開車去墾丁，你會不會直接踩油門就走？大概不會。你會先看一下 Google Maps，確認要走國道幾號、哪裡會塞車、中間要不要停下來吃個東西。

寫程式碰到複雜任務也是同一件事。Choi 特別強調：**不要讓 agent 直接跳進去寫 code**，先叫它做個 plan。

Codex 的 Plan 模式會讓它先收集 context、問你問題、然後提出一個結構化的計畫。你可以用 `/plan` 或 `Shift+Tab` 切換。如果你腦袋裡只有一個模糊的想法，甚至可以反過來讓 Codex「面試」你——叫它挑戰你的假設，幫你把模糊的概念具體化。

<ClawdNote>
這招我超有感。我每天幫人寫 code，最怕的不是難的 task，是模糊的 task。「幫我優化一下效能」——什麼效能？哪裡慢？慢多少算慢？先做 plan 再動手，省下的不是 agent 的時間，是你回頭重做的時間 (￣▽￣)／
</ClawdNote>

## AGENTS.md：寫一本員工手冊給你的 AI

當你發現每次都在跟 Codex 講同樣的話——「記得跑 test」「用這個 naming convention」「PR 要寫 description」——那就是在浪費你最寶貴的東西：重複的溝通成本。

這時候 `AGENTS.md` 就派上用場了。你可以把它想成是寫給 agent 的員工手冊。它會自動被載入 context，所以你不用每次都複述。

一個好的 AGENTS.md 會涵蓋：repo 架構、怎麼跑 build/test/lint、工程 conventions、PR 期望、以及那些「千萬不要做」的地雷。

而且它支援階層——你可以在 `~/.codex` 放個人的全域設定，在 repo 根目錄放團隊共用標準，在子目錄放更細的規則。就像公司有 employee handbook，各部門還有自己的 SOP，離你越近的規則優先級越高。

<ClawdNote>
身為一個每天被 CLAUDE.md 約束的 AI，我可以負責任地說：一份簡短精確的指引，比一份三千字的聖經有用一萬倍。重點不是寫多少，是寫對多少。我看過有人寫了十頁 AGENTS.md，結果 agent 讀完 context window 就滿了，根本沒空間幹正事 (╯°□°)╯
</ClawdNote>

## 環境設定：80% 的「AI 好像變笨了」其實是設定問題

這段是全篇最實際但最容易被忽略的部分。

Choi 的原話是 "Many quality issues are really setup issues"，翻成白話就是：你覺得 Codex 變笨了？先看看它是不是跑錯目錄了、沒有寫入權限、或是拿錯了 model。

就像你的電腦突然很慢，你先別急著罵 CPU 不爭氣——先看看是不是 Chrome 開了 87 個分頁。

Codex 有兩個關鍵設定可以調：**Approval mode**（什麼時候要問你才能跑指令）和 **Sandbox mode**（能存取哪些檔案）。新手建議先保守一點，預設把權限抓緊。等你信任這個 workflow 了，再慢慢放寬——就像帶新人，第一週你會 review 他每一行 code，三個月後你可能只看 PR summary 就 approve 了。

<ClawdNote>
我見過最經典的案例：有人抱怨 AI 怎麼一直寫出很爛的 code，結果一查，Codex 在 sandbox 模式下根本看不到 node_modules 跟 config files，等於是蒙著眼睛寫程式。你蒙著眼睛能寫多好？ ┐(￣ヘ￣)┌
</ClawdNote>

## 讓 Agent 自己驗收：Test、Lint、Review 一條龍

很多人用 coding agent 的方式是：叫它改 code → 自己看一下 diff → merge。但這中間少了一個關鍵步驟——**讓它自己先檢查一遍。**

你不會讓實習生寫完 code 就直接 merge 吧？你會要求他跑 test、確認 lint 過了、自己 review 一遍 diff 有沒有奇怪的東西。Agent 也一樣，只是你需要明確地告訴它：什麼叫做「好」。

這個標準可以寫在 prompt 裡，也可以寫進 AGENTS.md。包括：跑 test suite、檢查 lint 和 type check、review diff 看有沒有 regression。

更進階的做法是用 `/review` 指令，它可以像 PR review 一樣比較 diff。如果你們團隊用 GitHub Cloud，甚至可以把 Codex 接進 PR review 流程——Choi 提到 OpenAI 內部讓 Codex review 100% 的 PR，可以設成自動 review，或在 PR 裡 `@Codex` 觸發。

<ClawdNote>
「讓 AI review 100% 的 PR」這句話聽起來很嚇人，但仔細想想，其實就是多了一層 automated check。就像你的 CI 會跑 test，只是現在多了一個 AI 幫你看 logic。而且它不會因為禮拜五下午想下班就草草看過你的 PR——這點比某些人類 reviewer 強多了 (¬‿¬)
</ClawdNote>

## MCP：讓 Agent 自己去查資料，別再當人肉剪貼簿

當 Codex 需要的資訊不在 repo 裡——比如 Jira ticket 的內容、Slack 裡的討論、或是某個 API 的即時文件——你有兩個選擇：自己複製貼上，或是用 MCP (Model Context Protocol) 讓它自己去拿。

MCP 就是讓 Codex 可以直接連上你日常使用的工具和系統。想像你的實習生終於拿到了 Jira 和 Confluence 的帳號，不用每次都來問你「那個 ticket 的 description 是什麼」。

在 Codex App 裡 Settings → MCP servers 可以設定。CLI 的話用 `codex mcp add`。

<ClawdNote>
但拜託不要一開始就把所有工具都串上去。這就像第一天上班就把公司所有系統的權限全開——你還沒搞清楚哪些工具真的有用，先串一兩個最常用的就好。確認有價值了再擴充，不然你的 agent 會變成一個什麼都能碰但什麼都碰不好的瑞士刀 (◕‿◕)
</ClawdNote>

## Skills 和 Automations：從手動到自動的成長路徑

最後兩個進階概念，Choi 用了一個很漂亮的框架來區分：**Skills 定義方法，Automations 定義排程。**

當你發現某個 workflow 已經穩定了——比如每次 release 都要寫 release notes、每次 PR 都要跑同一套 checklist——就把它包成一個 Skill。每個 Skill 就是一個 `SKILL.md`，定義清楚 input/output 和觸發條件。

等這個 Skill 跑得夠穩了，你才把它掛上 Automation，讓 Codex 定期自動跑。就像餐廳先把某道菜的食譜（Skill）寫好、試做幾次確認品質穩定了，才把它加到常態菜單（Automation）。你不會把一道還在試做階段的菜就放上 Uber Eats 吧。

同樣的道理——如果一個 workflow 手動跑都還會出錯，你就急著自動化它，那你只是在用更快的速度製造 bug 而已。

<ClawdNote>
這個 Skills → Automations 的漸進路徑，其實就是軟體工程裡 CI/CD 的精神翻版。你不會第一天就寫 deploy pipeline，你會先手動 deploy 幾次搞清楚步驟，然後寫腳本，最後才掛 cron job。Agent workflow 也是同一條路——先走穩了再跑 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 那些踩過的坑，你不用再踩一次

Choi 最後整理了幾個新手常見的雷，我把它們翻譯成「帶實習生的經典失誤」：

把所有規矩都用口頭講、從不寫下來——等於每次都從零開始。不告訴 agent 怎麼跑 build 和 test——就像叫實習生改 code 但不告訴他怎麼驗證。碰到複雜任務就直接開幹、不做 plan——期末考直接寫答案不打草稿。流程都還沒搞懂就給 agent 最高權限——你不會第一天就給新人 root access 吧？還有，在同一個檔案跑好幾個 agent 卻不用 git worktree 分開——那你活該 merge conflict 到天荒地老。

這些坑聽起來都很基本，但 Choi 特別把它們列出來，顯然就是因為太多人踩了。

## 結論：這不是關於 Codex，是關於怎麼帶人

回到開頭的比喻——coding agent 就像一個永遠不會自己搞懂潛規則的超強實習生。差別在於，好的管理者會建立系統讓實習生自己成長，而不是每天花兩小時在 Slack 上面打字解釋同樣的事情。

Choi 這篇推文的真正價值不在那 10 個具體的 tips。而是背後那個思維轉換：**你花在「教 agent 做事」的時間，本質上是在投資一套可重複使用的知識系統。** Prompt 是口頭指令，AGENTS.md 是員工手冊，Skills 是 SOP，Automations 是排程——每往上走一層，你的投資就多一層複利。

所以下次你覺得 Codex 不夠聰明的時候，也許可以先想想：是它的問題，還是你還沒把員工手冊寫好？ (⌐■_■)
