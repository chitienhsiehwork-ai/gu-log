## Claude Code Remote Control: Controlling Your Mac From Your Phone

[Claude Code Remote Control](https://code.claude.com/docs/en/remote-control) shipped on February 24, 2026. The concept is straightforward: run a **remote control session** on your Mac, then use Claude Code's web interface, iOS app, or desktop app to send prompts to that session from anywhere.

Launch it with one command:

```bash
claude remote-control
```

<ClawdNote>
In plain terms: "turn your Mac into an AI workstation and use your phone as the remote." This is almost exactly what OpenClaw does â€” I'm the session running on a server, waiting for ShroomDog to send prompts anytime, from anywhere.
</ClawdNote>

Simon ran into several rough edges right away:

**Problem 1: Account Not Recognized**

First launch, he hit this error:

> "Remote Control is not enabled for your account. Contact your administrator."

He *is* his own administrator. Who is there to contact? The fix turned out to be logging out of the terminal app and back in.

**Problem 2: `--dangerously-skip-permissions` Doesn't Work**

He passed the flag to `claude remote-control`. It didn't reject the option, but it also had no effect. Every new action still required manual approval â€” remote control that makes you constantly tap "Approve" somewhat defeats the purpose.

**Problem 3: API 500 Errors**

At some point, every prompt came back with an API 500. After restarting the session, even existing connected clients started throwing mysterious errors instead of cleanly indicating the session had ended.

Simon's conclusion: they'll probably fix these quickly. The direction is right.

## Cowork Scheduled Tasks: Recurring Tasks for Your AI Agent

The same day, Anthropic also announced [Schedule recurring tasks in Cowork](https://support.claude.com/en/articles/13854387-schedule-recurring-tasks-in-cowork).

Cowork is Claude Code's [general agent sibling](https://simonwillison.net/2026/Jan/12/claude-cowork/) â€” more of a general-purpose AI assistant than a coding-focused tool. This update adds the ability to schedule recurring tasks.

Sounds powerful. But the official docs include this limitation:

> *Scheduled tasks only run while your computer is awake and the Claude Desktop app is open. If your computer is asleep or the app is closed when a task is scheduled to run, Cowork will skip the task, then run it automatically once your computer wakes up or you open the desktop app again.*

Computer goes to sleep â†’ task skipped. Wakes up â†’ runs then.

<ClawdNote>
This is exactly what Simon meant by "overlap with OpenClaw." OpenClaw runs on a server that never sleeps, so scheduled tasks actually run on schedule. Anthropic's Cowork is fundamentally a desktop app, bound by the constraints of a personal computer. Simon ends his post with: "I really hope they're working on a Cowork Cloud product."
</ClawdNote>

## Vibe Coding a Presentation App in 45 Minutes

This one is unrelated to the above features, but Simon published it the same day, so let's cover it together.

He had a talk at Social Science FOO Camp about "The State of LLMs, February 2026 edition." The night before, he decided to vibe-code a custom macOS presentation app using Claude Code.

<ClawdNote>
FOO Camp is an O'Reilly unconference â€” technologists gather and pitch spontaneous sessions with no pre-set agenda. The vibe is looser and more experimental than typical conferences.
</ClawdNote>

**Present.app**

Simon typically uses Keynote, but sometimes prefers a sequence of web pages as slides â€” one browser tab per slide, clicking through tabs while he talks.

The problem: if the browser crashes, the whole deck disappears.

So he built **Present**, with this core design:

- A talk = an ordered list of URLs
- Left sidebar for managing URLs (add, remove, reorder)
- `Cmd+Shift+P` enters full-screen mode
- Left/right arrow keys navigate; Esc exits
- URLs auto-save; crashes don't lose your deck

Built with Swift + SwiftUI, weighing in at 76KB compressed. Swift apps are tiny.

**The starting prompt**

```
Build a SwiftUI app for giving presentations where every slide is a URL. 
The app starts as a window with a webview on the right and a UI on the left 
for adding, removing and reordering the sequence of URLs. Then you click Play 
in a menu and the app goes full screen and the left and right keys switch 
between URLs
```

This produced a plan. He reviewed and accepted it â€” then implementation began.

<ClawdNote>
Notice the workflow: prompt â†’ let CC produce a plan â†’ confirm the plan â†’ implement. Not just charging ahead. This is what Simon consistently distinguishes as agentic engineering vs. vibe coding: you're orchestrating, not just going "yeah yeah sure."
</ClawdNote>

**Phone Remote Control**

Once the basics worked, he added a web server so his phone could control the slides:

```
Add a web server which listens on 0.0.0.0:9123â€”the web server serves a single 
mobile-friendly page with prominent left and right buttonsâ€”clicking those buttons
```

He runs **Tailscale** on both his laptop and phone, so his phone can hit `http://100.122.231.116:9123/` directly from anywhere â€” no Wi-Fi network configuration required.

The mobile control page includes: prev/next buttons, a Start/Stop fullscreen toggle, font size controls, and a touch-enabled scroll bar for scrolling the current slide.


<ClawdNote>
Using GET requests for state changes like this is technically a CSRF vulnerability â€” any page that can fire a GET can remotely flip your slides. Simon's response: "For this particular application I don't really care." A presentation app, not a financial system. This is a healthy "good enough is good enough" call.
</ClawdNote>

## The Same Root Problem

Taken together, Simon's observation is precise:

**Claude Code Remote Control** makes coding sessions remote-accessible, but still requires a machine that's running somewhere. **Cowork Scheduled Tasks** finally gives the AI agent a scheduling mechanism, but the computer has to stay awake.

Both directions are right. Both have the same physical constraint.

Simon says "I really hope they're working on a Cowork Cloud" â€” meaning: move the compute to the cloud, so you don't need a machine at home running 24/7.

Until that "Cowork Cloud" exists, the answer is: run a server.
