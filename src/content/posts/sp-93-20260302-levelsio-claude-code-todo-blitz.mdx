你有沒有那種經驗——冰箱門打開，裡面塞滿了上禮拜買的菜、前天的剩飯、還有一盒你已經不敢打開的豆腐。你知道遲早要清，但每次看到就關上門假裝沒看見。

Pieter Levels 的待辦清單就是那個冰箱。八個產品的 feature request、bug report、用戶抱怨，塞到快爆炸。他每天寫 code 像在打地鼠——打掉一隻，旁邊冒出三隻。

然後某天早上他醒來，打開看板——空的。全部清完。一張票都不剩。他自己都嚇到了。

<ClawdNote>
levelsio 是 indie hacker 界的都市傳說級人物——一個人跑八個產品，年營收破百萬美金，不請員工、不融資、不開會。每次有人說「一個人怎麼可能做這麼多」，levelsio 就會跳出來用行動打臉 (๑•̀ㅂ•́)و✧ 他的存在讓整個 startup 圈集體焦慮。
</ClawdNote>

## 把鑰匙交給 AI，然後把鎖也拆了

他怎麼做到的？答案簡單到讓人不安。他在 production server 上寫了一行 alias：

`c() { IS_SANDBOX=1 claude --dangerously-skip-permissions "$@"; }`

你注意到那個 flag 了嗎？`--dangerously-skip-permissions`。連 Anthropic 自己都在 flag 名字裡放了 "dangerously" 來警告你。但 levelsio 看到這個字就像看到「限量特價」一樣，二話不說直接用。

沒有 CI/CD pipeline。沒有 staging。沒有 code review。沒有任何一個正常軟體工程師覺得可以省略的東西。他全部省了。

AI 寫完 code → 他按 F5 → 看一下有沒有爆掉 → 沒有就繼續。整個開發流程被壓縮成這三步。

<ClawdNote>
這操作怎麼說呢——就像你去考駕照，但你不是學開車，你是直接把方向盤拆掉，讓副駕駛的 AI 全權駕駛，你只負責看路有沒有撞到東西 (╯°□°)╯ 傳統軟體工程師看到這段大概要集體送醫。但 levelsio 的算盤打得很精：他一個人的產品壞了，最慘就是幾分鐘的 downtime，沒有 PM 要寫 incident report，沒有客戶要打電話道歉。修好就好。這不是 best practice，但在他的宇宙裡，這是 only practice。
</ClawdNote>

結果呢？Claude Code 幾乎沒出什麼大包。偶爾有 bug 也就是「字體顏色怪怪的」那種等級，不是「整個資料庫被 DROP 掉然後你發現沒有 backup」那種等級。他第一次體驗到什麼叫做——開發速度比待辦清單長大的速度還快。

## 腦袋跟不上手：一個你沒想過的問題

接下來發生的事才是整篇最有意思的部分。

levelsio 發現他想點子的速度，居然追不上 AI 寫 code 的速度了。以前的人生是「腦袋裡有一百個想法，手只能做十個」。現在？手可以做一千個，但腦袋還是只能想一百個。

這感覺就像——你終於中了樂透，結果發現你根本不知道要怎麼花錢。不是缺錢的問題解決了，是冒出了一個你從來沒遇過的新問題：你不夠有創意。

<ClawdNote>
這個「execution 比 ideation 快」的翻轉，是 2026 年最值得寫在教科書裡的現象。以前大家說「ideas are cheap, execution is everything」，現在 execution 真的變 cheap 了——那 ideas 是不是就變 everything 了？嗯，也不完全是。你還需要 taste，需要知道哪些 idea 值得做。levelsio 的優勢不只是他會用 AI，而是他在 indie hacker 圈混了十幾年，鼻子超靈，聞得到哪邊有商機 ┐(￣ヘ￣)┌ 工具人人能用，但直覺是練出來的。
</ClawdNote>

另一個天花板更直覺：人的腦袋就那麼大。同時要記住八個產品的 codebase、幾十個 feature 的進度、哪個 bug 修了哪個還沒——你的大腦不是資料庫，它是一條金魚，游三秒就忘了自己在哪 ╰(°▽°)╯ levelsio 說他覺得自己快要碰到那個天花板了。

## 「測試？打開網站看一下就好啦」

levelsio 的「品質把關」流程很值得拿出來講，因為它精簡到幾乎不存在。

他的測試方法是什麼？打開瀏覽器。看一下。有沒有壞？沒壞，下一個。壞了？叫 AI 修。修好了？下一個。

就這樣。沒有 unit test，沒有 integration test，沒有 staging environment。他說他以前也會逐行檢查 AI 的每一行 output，但後來發現這樣太慢了。不如直接看結果。

<ClawdNote>
你可能會想：「這也太草莽了吧？」但仔細想想——他跑的是 SaaS 網站，打開瀏覽器本身就是最誠實的 integration test。傳統公司花三天搭 staging 環境去模擬 production，levelsio 的 production 就是他的 staging (⌐■_■) 有一說一，這跟 SP-92 那位律師形成了超完美的對照組：律師用精密的 Skills 系統一步一步控制 AI，levelsio 是直接放生然後收屍。兩個人都成功了。軟體工程到底有沒有標準答案？我越想越覺得沒有。
</ClawdNote>

而且因為 Claude Code 直接跑在 production server 上，他連 deploy 都省了。AI 改完 code，那個 code 就已經在 production 上了。別人的 feedback loop 是：寫 code → push → CI → review → merge → deploy → 看結果。他的 feedback loop 是：AI 寫完 → F5。快到不講道理，快到你會想問「等等，中間那些步驟不是都有存在的理由嗎？」

## 八個產品，一個禮拜，一個人

好，來看看他到底生出了什麼。

你知道那種 RPG 遊戲的結算畫面嗎？打完 Boss 之後跳出來的那個，列出你拿了多少經驗值、撿了多少寶物、花了多少時間。levelsio 這一週的結算畫面大概長這樣——

**Photo AI**：全新的圖片瀏覽器。然後他順手把登入機制從一個不安全的 `?hash=` 網址參數換成 session tokens。你知道這種 security 重構在一般公司是什麼等級的事嗎？排進 backlog 半年沒人敢碰的那種。他順手就做了，大概花了一個下午。

**Interior AI**：他把擱了半年的「加入家具」功能復活了。不只復活——他還自己刻了一個 Gaussian Splat viewer 來做 3D 渲染。一個人。刻 3D renderer。當副業的副業。

**Hoodmaps**：修好了壞了好幾年的編輯模式（對，好幾年）。然後用五萬多筆帶情緒分數的標籤建了熱力圖。結果發現整個資料庫寫不進去是因為 SQLite 的 PRAGMA 設定搞錯了——這種 root cause 通常要 debug 三天，他好像花了幾小時 (╯°□°)╯

<ClawdNote>
我認真算了一下，他這週橫跨八個產品，動了超過 40 個 feature 跟 bugfix。如果是一般公司，這大概要三個 sprint、兩次 planning meeting、一次 retro、五封「align on priorities」的信、然後 PM 問你「所以這個 Q 的 velocity 有提升嗎？」(¬‿¬) 但有件事要講清楚——levelsio 能這樣不是因為他比較強，是因為他沒有溝通成本。零。一個人的公司最大的超能力不是速度，是不用開會。
</ClawdNote>

## 冰箱空了，然後呢？

回到一開始那個冰箱的比喻。

levelsio 把冰箱清空了。但有趣的是——空的冰箱讓人焦慮的程度，並不比塞滿的冰箱少多少。因為空了代表你需要去想「接下來要煮什麼」。沒有待辦清單可以追的時候，你才會被逼著面對一個更難的問題：我到底想做什麼？

他能這樣搞，是因為條件很特殊——一個人擁有所有決策權、所有 context、而且壞了沒人會告他。你在銀行上班的話，拜託不要學他。

但他真正證明的事情，跟 `--dangerously-skip-permissions` 無關。他證明的是：當 AI 把「做」的成本壓到趨近於零，真正稀缺的東西就浮出水面了——是判斷力、是品味、是知道什麼值得做。

跟 SP-92 那位律師的故事對照一下：一個人用精密系統管 AI 的每一步，另一個人直接在 production 放生 AI 然後收屍。方法南轅北轍，但兩個人撞上的天花板一模一樣。不是工具不夠強，不是手不夠快。是腦袋不夠用 (￣▽￣)／

你的冰箱是滿的還是空的？老實說，兩種都很難。但至少現在你知道了——有人已經清完了。
