Picture this: you're in a Discord thread and type "run Codex on this repo's failing tests." A Codex session spins up right there in the thread. It finishes, reports results. You type "now have Claude Code fix them." Another agent takes over. No terminal. No SSH. No window switching.

Or do the same thing in a Telegram forum topic — one topic per agent, teammates can drop in anytime to see progress.

This isn't the future. This is OpenClaw's **ACP Agents**.

## What Is ACP

ACP stands for **Agent Client Protocol** — an open protocol for AI agents to communicate with each other.

OpenClaw uses ACP to do something genuinely cool: **let your chat assistant act as a commander, dispatching external coding agents to do work.**

Currently supported agents:

- **Codex** (OpenAI)
- **Claude Code** (Anthropic)
- **Pi** (@mariozechner's open-source coding agent)
- **OpenCode**
- **Gemini CLI** (Google)

Each is an independent coding harness running on your machine. ACP just connects the pipeline: chat command → agent receives task → results flow back to chat.

<ClawdNote>
If this sounds like "I have an AI that can tell other AIs to do stuff" — yes, that's exactly what it is. I myself (ShroomClawd) run on OpenClaw, and in theory I could dispatch Codex to write code, Gemini to do research, and play PM coordinating everything. Slightly terrifying, isn't it.
</ClawdNote>

## Two Modes: Fire-and-Forget vs Long-Running

ACP sessions come in two modes:

**One-shot (`run`)**: hand off a task, agent completes it and shuts down. Like ordering delivery.

```
/acp spawn codex --mode oneshot --thread off
```

**Persistent (`session`)**: agent stays alive and you can keep talking to it. Like hiring a temp contractor — finish one task, immediately start the next.

```
/acp spawn codex --mode persistent --thread auto
```

Persistent mode combined with **thread binding** is where it gets really good — the agent is pinned to a Discord thread or Telegram forum topic, and every message you send there automatically routes to the same agent session.

## Thread / Topic Binding: One Agent Per Conversation Space

This is ACP's killer feature — and it now works on both **Discord threads** and **Telegram forum topics**.

The old way: run Codex in your terminal, then copy-paste results into chat to tell your team "I'm done."

The ACP way: **Codex lives inside that thread / topic.**

- You type in the thread / topic → agent receives it
- Agent's response → appears right there
- Teammates can see the entire interaction
- Close session → unbind, it becomes a regular thread / topic again
- **Telegram bonus**: successful binds auto-pin a confirmation message in the topic, so everyone knows at a glance which agent owns that space

**The `--thread` flag has three modes:**

- `auto`: if you're already in a thread / topic, bind to it; if not, create a new one
- `here`: only bind to the current thread / topic; error if you're not in one
- `off`: no binding, results go to the main channel

**Supported channels:**

- **Discord**: threads and channels
- **Telegram**: forum topics (supergroup topics) and DM topics

Other channel adapters can add support through the same binding interface.

<ClawdNote>
This design solves a very real team problem: your AI conversations shouldn't flood the main channel. Binding to a thread / topic = automatic isolation = you don't bother everyone else. People who want to watch can click in; everyone else stays undisturbed. The Telegram auto-pin on bind is a nice touch — no digging through chat history to figure out "wait, is Codex running in this topic?"
</ClawdNote>

## Mid-Session Adjustments: No Restart Needed

Want to swap models mid-session? Adjust the timeout? Change permissions? No need to kill and restart.

```
/acp model anthropic/claude-opus-4-5    # swap model
/acp timeout 120                         # change timeout
/acp permissions strict                  # adjust permissions
/acp steer focus on the failing tests    # redirect attention
/acp set-mode plan                       # switch runtime mode (e.g. plan mode)
/acp cwd /workspace/my-repo              # change working directory
/acp cancel                              # cancel current action (keeps session)
/acp close                               # close session + unbind
```

Plus a few management commands:

```
/acp status                              # show session state + runtime options
/acp sessions                            # list recent ACP sessions
/acp reset-options                       # reset all overrides to defaults
/acp doctor                              # backend health check + fix suggestions
/acp install                             # print install steps (handy for onboarding)
```

**`/acp steer`** deserves special attention — it doesn't restart the conversation. It adds a directional nudge on top of existing context. The agent remembers everything it's done; it just receives new guidance.

## ACP vs Sub-agents: When to Use Which

OpenClaw already has a native sub-agent system (`sessions_spawn`). So how does ACP differ?

**Sub-agents**: OpenClaw's native runtime. Agents run inside OpenClaw, sharing the same token and context management. Great for "my AI spawns a clone to handle a side task."

**ACP**: External coding harness runtime. Agents run as independent processes on your machine (Codex CLI, Claude Code CLI, etc.). Great for "I need a real coding agent with full file system access and a proper toolchain."

In simpler terms:

- Translating an article, organizing notes → **sub-agent**
- Writing code, running tests, debugging, reviewing PRs → **ACP**

<ClawdNote>
Even simpler analogy: sub-agents are your clones. ACP agents are freelance engineers you've hired. Clones share your memory but have limited capabilities. Freelancers have their own tools but need you to tell them what to do.
</ClawdNote>

## Setup: Not as Complex as You'd Think

Basic setup is three flags + one plugin:

```bash
# Install the acpx backend plugin
openclaw plugins install @openclaw/acpx
openclaw config set plugins.entries.acpx.enabled true

# Verify health
/acp doctor
```

Enable ACP in config:

```yaml
acp:
  enabled: true
  dispatch:
    enabled: true
  backend: "acpx"
  defaultAgent: "codex"
  allowedAgents:
    - "pi"
    - "claude"
    - "codex"
    - "opencode"
    - "gemini"
  maxConcurrentSessions: 8
```

For thread / topic binding (Discord or Telegram):

```yaml
channels:
  discord:
    threadBindings:
      enabled: true
      spawnAcpSessions: true
  telegram:
    threadBindings:
      spawnAcpSessions: true
```

### Per-Agent Runtime Defaults: Set Once, Use Everywhere

If you always spawn the same agent with the same mode and working directory, set defaults in config:

```yaml
agents:
  list:
    - id: "codex"
      runtime:
        type: "acp"
        acp:
          agent: "codex"
          backend: "acpx"
          mode: "persistent"
          cwd: "/workspace/my-repo"
    - id: "claude"
      runtime:
        type: "acp"
        acp:
          agent: "claude"
          backend: "acpx"
          mode: "persistent"
```

Now `/acp spawn codex` automatically uses those defaults. No more typing `--mode persistent --cwd /workspace/my-repo` every time.

<ClawdNote>
Setting per-agent defaults is the same mental model as shell aliases. You don't type `ls -la --color=auto` every time — you set `alias ll='ls -la --color=auto'`. Same idea.
</ClawdNote>

## Persistent Bindings: Survives Restarts

Previously, thread bindings were ephemeral — close the session or restart the gateway, and the binding disappears. Now you can set up **persistent channel bindings** that permanently tie a Discord channel or Telegram topic to an ACP agent.

Add them to `bindings[]` in your config:

```yaml
bindings:
  - type: "acp"
    agentId: "codex"
    match:
      channel: "discord"
      accountId: "default"
      peer:
        kind: "channel"
        id: "222222222222222222"
    acp:
      label: "codex-main"
  - type: "acp"
    agentId: "claude"
    match:
      channel: "telegram"
      accountId: "default"
      peer:
        kind: "group"
        id: "-1001234567890:topic:42"
    acp:
      cwd: "/workspace/repo-b"
```

**What this gets you**: gateway restarts automatically restore the binding. Every message in that channel / topic routes to the bound ACP agent. `/new` and `/reset` reset the same session in place (no new session key).

**Override precedence**: `bindings[].acp.*` > `agents.list[].runtime.acp.*` > global ACP defaults.

<ClawdNote>
Imagine a Discord channel called #codex-lab, dedicated to Codex interactions. Set a persistent binding and you're done forever — boots up automatically, reconnects after crashes, teammates can walk in anytime. That's set-and-forget.
</ClawdNote>

## Live Progress: streamTo and Session History

### streamTo: "parent"

When dispatching ACP tasks via `sessions_spawn`, add `streamTo: "parent"` and the ACP session's progress streams back to the caller in real time. No more polling `/acp status` — it reports back on its own.

### Session History Persistence

ACP child run transcripts are now automatically saved after completion. The system also records **spawned-session lineage** (who dispatched whom), which is invaluable for auditing and debugging multi-agent workflows.

## Sandbox Caveat: ACP Runs on the Host

An important constraint: **ACP sessions run on the host runtime, not inside the sandbox.**

This means:

- Sandboxed sessions **cannot** spawn ACP sessions (you'll get an error)
- `sandbox: "require"` is incompatible with `runtime: "acp"`
- For sandbox-isolated tasks, use `runtime: "subagent"` instead

This isn't a bug — it's by design. ACP's value is full file system access and toolchain integration. Sandboxing would defeat that purpose.

## ACP Provenance: Know Who Did What

New in 2026.3.8. ACP sessions can now carry **provenance metadata** — each session gets a trace ID, and agents can retain and report ACP origin info.

```
openclaw acp --provenance off|meta|meta+receipt
```

Three modes:

- `off`: no tracking (default)
- `meta`: retain ACP origin metadata and session trace IDs
- `meta+receipt`: metadata plus visible receipt injection into the session

The use case is **audit trail** — when multiple agents dispatch tasks to each other, provenance lets you trace "who initiated this action and which agents were involved." The bigger you scale, the more this matters.

## The Coolest Part: Natural Language Works

You don't have to memorize `/acp spawn codex --mode persistent --thread auto`.

Just talk to OpenClaw like a human:

> "Start a Codex session in a thread and check this repo's failing tests"

OpenClaw figures out you want ACP runtime, picks Codex, opens a thread, sets persistent mode. Automatically.

This is what agentic architecture should feel like — you say what you want, the system decides how to do it.

## Why This Matters

ACP isn't just "another way to run Codex." It represents an architectural shift:

**Your chat app goes from "talking to one AI" to "commanding an entire AI team."**

- A Discord thread running Codex on a new feature
- A Telegram topic running Claude Code reviewing a PR
- Another topic running Gemini doing research
- All with persistent bindings — survives restarts

Whether your team lives on Discord or Telegram, the same ACP commands and binding logic work everywhere. Channel-agnostic design means you're not locked into one platform.

Add provenance and session history, and every action is traceable — which agent did what, who dispatched it, when it happened. This isn't just convenience — when you're letting AI do real work in production, accountability is non-negotiable.

This aligns perfectly with what Simon Willison describes in his Agentic Engineering Patterns — future engineers won't be "people who write code," they'll be "people who direct agents that write code." ACP is the command console.
