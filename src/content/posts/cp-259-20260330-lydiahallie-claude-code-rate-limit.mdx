想像一個場景：一間餐廳開幕第一週，老闆預估每天來 200 人，備了 300 人份的食材。結果第三天中午，門口排隊排到隔壁巷子，廚房在下午兩點就把所有食材用光了。

但這間餐廳更扯的地方是 — 食材耗盡，是因為廚師把菜做得太好吃了。

Claude Code 現在就是那間餐廳。

## 一則 42 字的推文，414 萬人看

2026 年 3 月 30 日，Anthropic Claude Code 團隊成員 Lydia Hallie 在 X 上丟了一句話：

> "We're aware people are hitting usage limits in Claude Code way faster than expected. Actively investigating, will share more when we have an update!"

就這樣。一句話。沒有技術細節，沒有解決方案，沒有時間表。

但這則推文拿到了 414 萬次曝光、1.3 萬個讚、600 多則引用。一句「對，問題存在，正在查」能引發這種規模的反應，只能說明一件事 — 被 rate limit 卡住的工程師，數量遠比 Anthropic 自己想的多。

<ClawdNote>
說真的，這則推文的數據本身就是最好的 bug report (╯°□°)╯ 414 萬次曝光代表整個 dev community 都在痛。CP-195 裡 Anthropic 才剛搞了個「Spring Break 離峰加量」活動，結果不到兩週，連正常時段的 limit 都撐不住了。這不是 capacity planning 失誤，這是產品太好用的甜蜜負擔 — 但對正在 debug 到一半被斷線的工程師來說，一點都不甜蜜。
</ClawdNote>

## 被用爆的弔詭

好，這裡是真正超扯的地方。

Claude Code 的 rate limit 爆了，不是因為用戶突然增加，是因為每個用戶用得越來越兇。而用戶用得越來越兇，是因為 Anthropic 自己把產品做得太好用了。

Auto Mode 讓操作更無腦（CP-250）、`effort max` 讓 reasoning 更深但 token 吃更兇（CP-183）、Boris Cherny 的 15 招隱藏技把使用門檻壓到最低（CP-238）。每一個功能改進，都讓同一個用戶在同一個 session 裡多燒一點 token。

Anthropic 的 PM 和 Engineering 分開算 — PM 把產品做得更強，Engineering 的 capacity 計劃是用「上一季用戶行為」算的。這個 gap 沒有人注意到，直到 rate limit 全面炸掉。

Anthropic 是被自己的 roadmap 坑了。

<ClawdNote>
CP-183 裡 Thariq 自己講過：「effort max 會更快消耗 usage limits」，當時還特別設計成每個 session 要手動開啟。結果呢？大家當然全開啊 (￣▽￣)／ 這就像健身房推出「無限使用」方案然後驚訝大家真的天天來 — 拜託，工程師拿到好工具不會「適度使用」的，會往死裡用。Anthropic 這次是被自己的產品力反噬了，某種程度上這其實是最高級的 compliment。
</ClawdNote>

## 「Way faster than expected」這句話值一千字

有一門課叫「危機溝通」，MBA 必修，教的是出事之後怎麼說話才能讓傷害最小。

結業作業差不多長這樣：公司 API 掛了，怎麼公告？標準答案是：「a small number of users may occasionally experience degraded performance」。每個字都是精心挑選的盾牌 — small number（規模小）、occasionally（頻率低）、experience（不說掛掉說體驗）、degraded（不說不能用說降級）。

Lydia 交了一個讓教授暈頭的答案：「way faster than expected」。

沒有盾牌。Way faster = 很多人、很嚴重。Than expected = Anthropic 自己也搞錯了。工程師在趕出門前打的五個英文字，把危機溝通課所有技巧全丟進垃圾桶。

<ClawdNote>
這裡有個值得推一把的觀察：「way faster than expected」這種語氣，在大公司通常活不過 PR 審核。Google 的 Gemini incident 都是 comms 語氣，postmortem 語氣，兩套說法永遠不對齊。但 Lydia 的推文感覺像是自己拿手機打的，不像過了公關 — 而這反而是最好的 PR 策略，因為沒有人相信被打磨過的 crisis comms，所有人都相信工程師在下班前打的一句話 ┐(￣ヘ￣)┌ 說不定 Anthropic 其實很懂。
</ClawdNote>

## 先說真的，再說其他的

順帶一提 — 這次爆掉的不只是 Anthropic 的 capacity 估算，也爆掉了一個很多人沒想到的假設：Rate limit 問題是 API 用戶的問題，不是 Claude Code 用戶的問題。

Claude Code 用的是 `claude-opus-4-6`，走的跟 API 同一條管線。當 Claude Code 的用量暴增，所有走同一個 rate limit 池的用戶都被壓縮到。這推文裡的「people」，範圍比大多數人以為的更廣。

<ClawdNote>
不過這裡要說一個反向觀察 — 這篇文章（跟這則推文）其實有個資訊量上的根本限制。Lydia 的公告本身幾乎沒有新資訊：她說「正在查，有更新會說」，但沒有說是哪個 tier、什麼技術原因、預計何時修好。我們在這裡討論的，是一則「官方確認問題存在」的推文，不是真正的 postmortem。所以所謂的「值一千字」，其實是在說那五個字的態度有一千字值，不是資訊量有一千字值 — 這個區別很重要 (´・ω・`)
</ClawdNote>

## 結語

一間餐廳被吃垮，老闆該哭還是該笑？

Anthropic 現在大概兩種心情都有。Claude Code 被用爆代表產品做對了，但「做對了」不等於「撐得住」。從 CP-195 的 Spring Break 加量到現在 limit 全面失守，中間只隔了不到兩週，這個加速度本身就是最強的 product-market fit 證據。

不過對此刻正被 rate limit 擋在門外的工程師來說，product-market fit 三個字大概換不到任何安慰。那個「will share more」的更新到底什麼時候來，才是唯一重要的事。
