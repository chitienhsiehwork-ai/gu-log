The most magical-looking part of an [Agent](/glossary#agent) is usually not the hardest part.

Start with a dumb but very real incident: an agent reports that it is "complete," while the surrounding system has no verifiable state proving the work is actually complete.

That is the cruelest and most useful lesson in agent engineering: **an agent saying it is done does not mean the system is actually done.**

A reliable agent is not one that writes a pretty completion report. It records progress, can be checked by machines, leaves inspectable traces when something breaks, and cannot randomly poke tools it should not touch. The model is only the driver. The engineering system is the dashboard, brakes, lane markings, and dashcam.

So this piece is not asking "which model is strongest?" It is asking: **how do agents go from demo toys to systems that can actually deliver work?** Keep three questions in mind: who decides the next step, who checks the result, and where do engineers look when something fails?

## A control loop can run without finishing the task

Strip an agent down to the minimum, and it is basically a `while` loop: send user input to the model; if the model wants a tool, the external system runs it and feeds the result back; if the model replies with plain text, the task ends. You can call that perception, decision, action, and feedback, but the name is not the problem. The problem is: **a control loop can run without completing the task correctly.**

Many products wear the agent label, but inside they are fixed workflows: the route is already written, and each step merely uses an LLM. That is fine. When the task is fixed and acceptance is clear, a fixed workflow is often more stable. You only need an agent when the task requires exploration, judgment, and dynamic adjustment across tools.

## A harness is an acceptance rig, not model decoration

This is where the [Agent Harness](/glossary#agent-harness) matters. Think of a harness as the acceptance rig around the agent: goals, scope limits, checks, failure feedback, and rollback paths. The OpenAI case reports dramatic numbers: 3 engineers, 5 months, roughly 1 million lines of code, nearly 1,500 PRs, about 10x traditional development speed. Treat that as self-reported case data, not a universal guarantee. The lesson is the conditions behind it: docs that work like maps, constraints embedded in format checks, type systems, CI, and structural tests, and agents that can reproduce failures, run tests, inspect logs, and read traces.

The point is not "write more docs." The point is to shape the task so an agent can hold it: clear goals, checkable results. Human-only acceptance caps throughput; vague goals with lots of checks make the system sprint efficiently in the wrong direction.

<ClawdNote summary="A green CI check is a different civilization from a completion report.">
There is an entire civilization between "the agent says it is done" and "CI is green." The first is a verbal report. The second at least has a machine willing to share the blame. An agent workflow with no goal and no verification is electronic divination: ritual, response, result, but not acceptance testing. (´・ω・`)
</ClawdNote>

## Dirty context and tiny tools can make the agent dumber

After acceptance comes the sneakier failure: the model is not broken, but the desk is a mess.

Context engineering sounds academic. It is really just cleaning your desk. [Prompt](/glossary#prompt), tool outputs, long documents, memory, error logs: if you mash all of them together and shove the pile at the model, the model sees a lot, but has a harder time finding the signal. Nobody dumps the whole hardware store onto the table when fixing glasses. Keep the current task in front, methods in docs, history in traces, and only durable long-term facts in `MEMORY.md`-style notes with rollback to raw data.

The same rule applies to tools: **do not give agents a pile of low-level API parts; give them tools that complete tasks.** Instead of making an agent assemble `create_file`, `write_content`, and `set_permissions`, give it `create_script(path, content, executable)`. [Skill](/glossary#skill) and tool descriptions should act like road signs: when to use it, when not to use it, and what it returns. The numbers tell the story: without counter-examples, skill routing can fall from 73% to 53%; with counter-examples, it reaches 85%. Tool definitions have the same cost: five MCP servers can add roughly 55,000 [Token](/en/glossary#token) of overhead; another MCP experiment fetched definitions only when needed, cutting related task token use by 46.9%. These are all versions of the same engineering move: make the agent guess less and grab the right handle faster.

<ClawdNote summary="Bad tool descriptions and bad memory both create confident confusion.">
A tool description like "helps with backend" is like a convenience store sign that says "we sell things." A good description works like a road sign: turn right for deploys, not for billing. Memory without rollback is not infrastructure. It is a paper shredder that rewrites history for you.
</ClawdNote>

## Once multiple agents run, state starts fighting back

The exciting part of multi-agent systems is that many things can run at once. But the first thing to break usually is not speed. It is shared state: one agent is still editing a file, another agent summarizes old state, and suddenly two very busy workers have cooked the same instant noodles in the same pot.

It makes sense for a main agent to delegate search, trial-and-error, and debugging to subagents; each subagent keeps its own context and returns only a summary, so the main agent does not get polluted. But collaboration cannot rely on verbal agreements. Long tasks need progress files; parallel edits need isolation; "who is waiting for whom, and who completed what" needs to become a recoverable record. If files have not landed or tests have not run, a pretty summary is still only a summary.

At this point, stop asking which agent is smarter. Look at three instruments: **evals check whether the final state is right, traces show where the run drifted, and safety boundaries decide what dirty content is allowed to touch.**

Evals are not essay exams. They are crash tests. `Pass@k` asks whether one of several shots succeeds. `Pass^k` asks whether every step in a run holds. Use code graders for tests, structural diffs, and parameter checks; use model graders only when the job is rubric judgment or semantic comparison. Asking the model to judge everything is like asking the person who parked badly to write their own ticket.

Each run should log the full [Prompt](/en/glossary#prompt), `messages[]`, tool parameters and returns, final output, token use, and latency. An agent without traces is a self-service laundromat with no cameras: when the clothes disappear, everyone just stares at each other. [OpenClaw](/glossary#openclaw) works as a full-system example: messages, decisions, tools, and memory should be separated; long-running task state should be recoverable; safety checks should stand in front of dangerous operations.

[Prompt](/en/glossary#prompt) injection is the same lesson: do not assume the model will never be fooled; make sure dirty content cannot reach dangerous buttons.

<ClawdNote summary="Agent safety is more practical when the dirty note cannot reach the keys.">
I like the phrase "dirty content cannot touch dangerous buttons" because it drags agent safety back from sci-fi philosophy into engineering. The question is not whether the model has evil intent. The question is whether a poisoned sticky note can walk into the server room holding the keys.
</ClawdNote>

## If you want to ship it, order matters more than terminology

When you start building, do not chase terms first. Chase the order of operations.

Day one: get one minimal path working, where a message comes in, the agent handles it, and the result goes back out. Day two: add the workspace, allowlist, and parameter checks. Day three: turn the first real failure into an eval case. Only after that should documents, externalized progress, and isolated multi-agent edits grow around the system.

This order is boring, and that is the point. Many agent incidents look like model problems, but they are interface, acceptance, isolation, or eval problems. On stage it looks like the magic trick failed. Backstage, the prop box was never locked.

## Closing

An agent's core control loop is small: perception, decision, action, feedback. It is small enough to tempt people into thinking the outside engineering can be added later.

But the opening "complete" incident was never a speech problem. The system had no verifiable state, no trace, no permission boundary, and no path for investigating failure. Stronger models will make agents more capable; that also means they can amplify bad tools, weak evals, and runaway permissions faster.

An agent is not a thinking button. It is closer to a new engineering team. If tasks, docs, tools, tests, review, logs, permissions, or rollback paths are missing, even a smart agent is just charging through a black box. The reliable agent is not the one that says "done" most beautifully. It is the one whose system can prove it.
