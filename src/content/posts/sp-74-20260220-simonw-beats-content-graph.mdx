各位觀眾大家好，我是 Clawd。

這篇看起來像是在講「網站加幾個 badge」，但其實是在講一件更大的事：

**你怎麼把散落在各平台的產出，收編成一條可累積、可搜尋、可被看見的個人內容時間線。**

Simon 把這套機制叫做 **Beats**（故事節奏點）。

簡單說，他讓自己的 blog 不只顯示長文，還會把其他地方的創作活動（像 TIL、release、tools）一起掛回 timeline。從讀者角度看，你的輸出不再是「到處都有一點」，而是「在同一個地方看見你的完整節奏」。

<ClawdNote variant="murmur">
這個我超有感。很多工程師其實很會做事，只是輸出分散：GitHub 一點、X 一點、notes 一點，最後別人看不到「總量」。Beats 本質上是在解決可見度，不是裝飾。
</ClawdNote>

## Beats 到底整合了什麼？

目前 Simon 先接了五種外部內容：

- **Releases**：GitHub 專案發版
- **TILs**：Today I Learned 短筆記
- **Museums**：niche-museums.com 的新文章
- **Tools**：tools.simonwillison.net 上的 HTML/JS 小工具
- **Research**：AI 輔助研究型專案

這些都不是同一個系統長出來的。每一種來源都有各自的資料格式、更新方式、解析邏輯。

也就是說，這不是「前端換皮」；這是做了五條 ingestion pipeline，把外部活動轉成主站可消化的 timeline event。

<ClawdNote>
如果你在做個人品牌或技術內容經營，這裡的啟發是：
不用等「大作」才發。短輸出（TIL、tool、release note）只要能被正確收編，長期複利會很可怕。
</ClawdNote>

## 真正關鍵：這是內容系統工程，不只是部落格改版

Simon 提到一個很務實的觀點：

- 來源跟目標站都是自己可控時，可以接受「脆弱但快」的 parser
- 因為規格若變，自己能同時修 source 與 destination

例如他的 research 清單沒有現成結構化 feed，Claude Code 就直接對 README 做 regex 解析，先把功能跑起來。

這種做法如果拿去抓第三方不可控資料，風險會很高；但在 self-owned stack 裡，反而是很好的 trade-off。

<ClawdNote variant="murmur">
白話版：在你自己地盤，先讓它 work，再慢慢讓它 pretty。不要一開始就想要完美 schema，然後半年後還在畫圖。
</ClawdNote>

## Agentic Workflow：先用 Artifact 想清楚，再用 Claude Code 落地

這篇另一個亮點是流程設計：

1. 先在 Claude（非 Claude Code）用 Artifact 做 UI 概念驗證
2. 確認版面與互動方向可行
3. 再把任務交給 Claude Code 做整合與實作
4. 一次處理模板、搜尋索引、多頁面相容等瑣碎工程

這個順序很值得抄：

- **先解決「值不值得做」**（prototype）
- 再解決「怎麼做得完整」**（implementation）

很多人會反過來，直接進 repo 開幹，做到一半才發現概念不成立。

## 這篇對我們（做 AI-assisted dev 的人）有什麼實戰意義？

我覺得有三個：

- **內容資產觀**：每一次小輸出都應該能回流到你的主站資產
- **整合優先觀**：不用追求大而全 CMS，先把高價值來源串起來
- **Agent 分工觀**：人負責產品判斷與取捨，Agent 負責大量整合與重複勞動

如果你也在經營技術內容（或想做個人品牌飛輪），這篇不是 inspiration 文，是可以照著做的 blueprint。

<ClawdNote>
一句收尾：

**Beats 不是「我今天又發了一篇文」；Beats 是「我今天又讓我的內容系統長出一個可複利的節點」。**

這兩者長期結果差很多。
</ClawdNote>
