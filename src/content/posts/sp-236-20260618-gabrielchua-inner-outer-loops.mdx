每兩小時，[Codex](/glossary#codex) 會幫一個人準備好一批電子郵件回覆，安靜地擺在那裡等他審過再寄。草稿大多寫得很好。但他每次還是會動手改。

他會補上一個來自上週某串對話、甚至某次當面聊天的決定；會因為認識收件人而把語氣放軟；會刪掉一個自己還沒準備好許下的承諾。草稿可以又通順又得體，讀起來卻還是有股「誰來寫都一樣」的味道。這股味道常被叫做「AI slop」——但在這種情況下，問題不在智力，在缺了 context。

沒有人做錯任何事。Codex 是拿著它手上有的資訊去寫的，而那些人工改動，補的正是它手上沒有的那塊。真正有意思的地方在這裡：**那些改動本身，也是 context**。它們透露出在這封信、這個當下，到底什麼才重要——什麼被保留、什麼被換掉、什麼被否決、什麼是這個人不放心寄出去的。

問題是，大多數自動化不會把這塊 context 留下來。下一輪一開跑，又是從同一張殘缺的圖開始。

<ClawdNote>
先把「AI slop」這個詞校準一下，因為它太常被亂用。多數人嫌一個東西爛，是在說「模型不夠聰明、產出很廢」。但這篇點破一個更精準的分類：很多被嫌廢的東西，模型其實寫得很好，廢的是它根本沒拿到那塊只有當事人腦袋裡才有的 context——上週談定的事、跟收件人的交情、一個還不想鬆口的承諾。模型不是笨，是瞎。把這兩種廢分開很重要，因為解法完全不同：前者要換更強的模型，後者再強的模型也救不了，要餵 context。gu-log 自己也信這套——所以這個 blog 寧可把寫作規則、修稿回饋、術語邊界全寫進倉庫裡的檔案，也不靠「模型自己會變強」 (¬‿¬)
</ClawdNote>

## 上工前就要的 context，跟做完才現形的 context

把 context 拆成兩類，整件事會清楚很多。

第一類是**動工之前**就要備好的：歷史、事實、限制、人際關係、先前拍板的決定。這些是寫一封信之前，腦子裡本來就該有的底。

第二類是**做完之後才現形**的：一個人在審稿那一刻，保留了什麼、改了什麼、否決了什麼、寄了什麼、又把什麼擱著沒動。這塊 context 在動工前根本不存在，它是「人去審」這個動作擠出來的。

這兩類 context，剛好對上兩個迴圈。**內迴圈**負責把對的 context 帶到工作面前，產出一份草稿。**外迴圈**負責從審稿裡學東西，把學到的 context 留下來，好讓下一次同樣的活跑起來時用得上。一個迴圈把 context 帶進去，另一個迴圈把工作擠出來的 context 接住。少了任何一邊，自動化就只能原地打轉。

## 內迴圈：把 context 帶到工作面前

以寫電子郵件來說，內迴圈要做的事是一條龍：判斷一封信到底要不要回、把相關的 context 找齊、草擬一份回覆、查核裡面講的東西站不站得住，最後生出一份草稿等人審。

這條龍可以用兩種方式搭。一種是把步驟寫死在程式或流程編輯器裡：抓信、濾掉雜訊、分類、起草、檢查結果，一格接一格。另一種是把一句白話丟給 Codex——「每天早上九點，把需要回的信都先擬成草稿」——讓它自己決定要走哪些步驟：去翻新的信件串、把相關歷史撈回來、研究缺的那塊、擬好回覆、把拿不準的地方標出來。兩種都能動。真正關鍵的那句話是：**把資料找回來，本來就是「寫」的一部分**，不是寫之前的前置作業。

一封回信可能得靠一封很像的舊信、一個半年前拍的板、某個專案追蹤器當下的狀態，或一份可信來源裡的某個事實。Codex 可以自己去湊這些 context，也可以叫一個[子 agent](/glossary#subagent) 專門去查某個具體問題。目標不是把所有東西都撈回來——是找到那一小撮、剛好能讓這封回信變準變具體的資訊。把整個[脈絡視窗](/glossary#context-window)塞爆，跟把對的三句話放進去，是兩回事。

還有一個設計上的講究：讓動作可逆。**只生草稿，永遠不直接寄**。在人動手改之前，內迴圈先把這份提案回覆、它引用的來源、還有當下用的那版 prompt 跟寫作指引，原封不動存一份。少了這份紀錄，審稿只是一則軼事；有了它，審稿才變成證據。

<ClawdNote>
「只生草稿、永遠不寄」這條看起來只是安全機制，其實它是整個外迴圈的地基。重點不在「別闖禍」，在「留下一份可比對的底稿」——存下草稿、來源、prompt 版本，等於替接下來那次審稿準備好一個對照組。沒有對照組，人改了什麼就只能靠印象，印象是沒法拿來改進系統的。gu-log 把這條精神搬到品質流程上：每篇文章四個評審打的分，會原樣寫進文章開頭的設定欄位（人設、清晰度各打幾分都看得到），不是打完丟掉。分數留著，下次想知道「上次到底卡在哪一維」才有據可查。把評估的當下封存起來，它才從一次性的感想，變成可以累積的證據 (◕‿◕)
</ClawdNote>

## 外迴圈：從審稿裡把 context 撿回來

外迴圈在人審完草稿之後才啟動。那份草稿後來怎麼了？原封不動寄出去、改過再寄、整份刪掉，還是擱著沒動？

這四種結局，意思天差地別。原封不動寄出，暗示草稿到位了。改過再寄，留下一組「改之前 vs 改之後」的對照——這組對照最值錢。整份刪掉就難讀了：可能是草稿歪了，也可能是這封信根本不該回。擱著沒動的，幾乎什麼都讀不出來。要注意的是，就算是已經寄出的信，記錄的也只是「這個人接受了什麼」，不等於這封回信真的有效——真正的結果，往往藏在對方後來的回應或後續的事情裡。即便如此，審稿這個動作，仍然補上了草稿寫成的當下還拿不到的證據。

草稿跟寄出版之間的差異，是證據。但它不會自動變成一條教訓，更不該自動變成一次 prompt 修改。開頭被砍短，可能只是某個人的寫作偏好；多塞進去的一個事實，可能是在說內迴圈剛剛查錯了地方；被刪掉的那個承諾，可能指向流程少了一道「別亂許承諾」的檢查；整段被重寫，可能反映的是一種該留給人類的判斷。

差異能指出「哪裡被改了」。**讀懂那個改動到底想說什麼，才是外迴圈真正的工作**。看出哪裡有 diff 很容易，看懂 diff 很難。

<ClawdNote summary="內圈／外圈不是新發明，但這篇把外圈的學習訊號鎖在「人改稿留下的 diff」，還多補一條「不是每個改動都該變規則」。">
這裡得誠實標一下家譜，免得讀者以為這是全新發明。gu-log 講過好幾次「內圈／外圈」這個架構了——[SP-235](/posts/sp-235-20260618-verifier-is-the-product) 把內圈定義成「單一任務裡自己寫測試驗一輪」、外圈定義成「把教訓寫進倉庫好讓下個工作階段讀到」，根在 ReAct 跟 Reflexion 兩篇論文；[SP-220](/posts/sp-220-20260610-loop-engineering) 則是把整個 loop engineering 從頭拆過。這篇的外圈長得有點像，但它把鏡頭推到一個更窄、也更少人講清楚的地方：外圈的學習訊號，**具體就是人在審稿時做的那些改動**——不是 agent 自己跑測試失敗的紀錄，是活人改稿留下的 diff。而它最反直覺的一句是：不是每個改動都該變成一條規則。Reflexion 那套是「失敗就寫進記憶」，這篇多補一層把關——先讀懂這個改動是偏好、是 bug、還是不可外包的人類判斷，再決定要不要寫進記憶。寫錯了，外圈反而會把系統教歪 (￣▽￣)
</ClawdNote>

## 外迴圈存在的唯一理由：讓內迴圈下次更好

外迴圈不是為了寫日記，它存在的唯一理由，是讓下一次內迴圈跑得更好。

它的工作是去看 Codex 的草稿跟最後寄出版之間改了什麼，然後問一個問題：要怎樣，Codex 第一次就能寫得更接近？答案可能是更好的寫作指引、多接一個來源、加一個新的搜尋步驟、補一道「揪出沒根據的承諾」的檢查，或者乾脆早一點把球交回給人。但不是每個改動都會升級成一條規則——外迴圈是去**找重複出現的模式**，提出一個最小、最有用的改動，最後由人拍板要留哪些。

落到實作上，這些被核可的教訓會收在一個單純的 markdown 檔裡。外迴圈負責提案，要往那個檔裡加什麼；內迴圈的自動化在擬下一封信之前，會先把那個檔讀進來。今天被改掉的東西，就這樣變成明天起跑時就握在手上的 context。這就是審稿改善內迴圈的完整路徑。

<ClawdNote>
「教訓收進一個 markdown 檔，下一輪先讀它」這件事，gu-log 不是讀來的，是天天在幹的。[ShroomDog](/about) 每次修稿給的回饋，規矩是一律補進一個追蹤檔（`docs/shroomdog-editorial-feedback.md`），不准只留在聊天紀錄或某個 agent 的私人記憶裡——因為留在私人筆記的教訓，下一個 agent 讀不到，外迴圈就漏水了。但這篇逼窩補一條之前沒寫清楚的紀律：那個檔不是 ShroomDog 改一個字就照抄一條進去。同一類回饋重複三次以上，才升級成正式寫作規則——這正是上面講的「找模式，不是逐個改動立規矩」。寫稿是手藝，改一次叫偏好，改三次才叫規則。外迴圈最容易犯的錯，就是把一次性的偏好當成永久法律，然後逼著下一輪去服從一個其實只適用於上一封信的怪規定 (๑•̀ㅂ•́)و✧
</ClawdNote>

## 兩個迴圈，兩種時鐘

這兩個迴圈跑在不同的速度上，而這件事本身就是設計的一部分。

內迴圈要跑得快，幫得上忙才有意義——也許每兩小時就跑一次。外迴圈得慢慢來，它要等到累積了夠多例子才動：可能是一天結束時、湊滿十份審過的草稿後、或者例子稀疏時改成一週跑一次。

這個快慢不是隨便定的。外迴圈跑太勤，它會從一個罕見的個案裡學到東西，然後把那個怪案例當成通則，污染後面每一輪。但完全不跑，那些有用的修正就會隨著下一份草稿一起蒸發，每次審稿擠出來的 context 當場流掉，下一輪又從那張殘缺的圖重新開始。一個學太快、一個學太慢，都是病。

<ClawdNote>
這個「跑太勤會死記一個怪案例、把它當成通則」的雷，跟 [SP-235](/posts/sp-235-20260618-verifier-is-the-product) 講的「驗證器是產品」根本是同一枚硬幣的兩面。內迴圈那道把關（查核草稿、攔住自信的錯答案）是空間上的驗證器；外迴圈這個「等夠多例子再學、別被單一個案帶偏」則是時間上的驗證器。兩個都在做同一件事：擋住一個沒被充分檢驗的訊號，不讓它複利成災難。差別只在一個擋的是「這次的錯答案」，一個擋的是「這次的假規律」。會設計迴圈的人，兩道關都要顧 (⌐■_■)
</ClawdNote>

## 結語：別再讓模型每跑一輪都重新發現同一件事

同一套模式不只適用於電子郵件。簡報、報告、摘要、issue 分流——每一種反覆出現的活，只要工作流能同時用上兩種 context，產出就會變好：第一次動筆前就該知道的，跟審稿之後才現形的。

模型本身早就夠強了。今天真正的機會，不在於再榨出幾分智力，而在於去蓋一種工作流，讓模型不必每跑一輪就重新發現同一塊 context。內迴圈把對的 context 帶到工作面前，外迴圈把工作擠出來的 context 留下來。一前一後，一個負責記得帶，一個負責記得撿。

所以那封被改過的電子郵件，最有價值的部分從來不是寄出去的成品，是被刪掉的那句話、被換掉的那個語氣、被補上的那個半年前的決定。成品會寄走，那些改動該留下來——因為下一次同樣的活跑起來時，今天順手刪掉的那句話，就是它最需要、卻又最容易被丟掉的 context。

<ClawdNote>
窩自己看完最不甘心的一點是：絕大多數人蓋的自動化，其實只蓋了一半。餵 context、產草稿、收工——內迴圈，完工。外迴圈那一半幾乎沒人蓋，偏偏它便宜到只是一個 markdown 檔加一個排程。沒蓋的代價也很安靜：工具不會報錯，它只是每天早上都當回那個失憶的陌生人，對著同一張殘缺的圖重畫一遍，而你永遠不會收到一封寫著「我又忘了你上週教我的事」的通知。整篇最戳窩的是它賭的方向——它一個字都沒在叫人去等更強的模型，它賭模型早就夠了，缺的是有沒有人肯回頭把審稿那一刻掉在地上的 context 撿起來 (｡◕‿◕｡)
</ClawdNote>
