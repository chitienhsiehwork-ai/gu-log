## 你記九九乘法表是為了算數，不是為了記數字

我在念小學的時候，老師要求我們把九九乘法表背到滾瓜爛熟。7×8=56，9×7=63，一個一個記。

但到了國中，你學了乘法的原理——你理解了這些數字為什麼是這樣，而不只是記住它們。你會用分配律把複雜的乘法拆開算，用筆算把大數字搞定。你的腦子不需要記那麼多「答案」，因為你學會了「怎麼算」。

這個比喻，幾乎完美地解釋了 AI 模型在 2026 年正在發生的事。

Awni Hannun（Apple MLX 框架的創造者，他對 on-device AI 的效率有著幾乎偏執的研究熱情）在上週發了一則推文，點出了一個很多人沒想到的洞見：**intelligence-per-watt 飆升的速度，有一部分原因來自「模型不需要再背答案」這件事**。

## 明顯的原因先說完

每隔幾個月就有人說「AI 越來越便宜、越來越強」，主流的解釋通常是這三個：

**更好的架構**：從早期的 Transformer，到 MoE（Mixture of Experts）把模型拆成很多「專家」按需啟動，再到各種 SSM 混合架構，每次架構創新都是在用更少的計算做更多的事。

**更好的硬體**：NVIDIA 的 Blackwell、Apple M 系列晶片、Qualcomm 的 Snapdragon……每一代晶片都把每瓦特能處理的浮點運算數推到新的高峰。

**更高品質的資料**：訓練資料的品質提升（合成資料、精選資料集、RLHF 的改進）讓模型用同樣的 size 學到更紮實的知識。

這些都對，但 Awni 說這些都是「顯而易見」的原因。他想說的是另一個。

## 那個沒那麼顯眼的原因

讓我們回到早期的 LLM——大概是 2022-2023 年那個時代。

那時候模型要學會做簡單的算術（比如「37 + 48 = ?」），它大概是怎麼學的？

**靠背。**

訓練資料裡有大量的 `(輸入, 運算, 輸出)` 組合，模型把它們統統塞進 weights 裡。37+48=85，42+67=109，128+256=384⋯⋯每一個你可以想到的算術 case，都以某種形式躺在那幾百億個參數裡面。

你可以想像這有多浪費。

Awni 說：「你可以想像這佔了 weights 裡大量的空間。」

現在換個場景。今天你給模型一道算術題，它有兩條出路：

**方法一：Reasoning（推理）**
在 chain-of-thought 裡一步步算出來。`37 + 48 = 37 + 40 + 8 = 77 + 8 = 85`。答案不需要記住，推出來就好。

**方法二：Tool Calling（工具呼叫）**
直接呼叫計算機工具。把算術這件事完全外包給一個確定性的工具，精確無誤，永遠不會算錯。

兩個方法都達到了同樣的效果——正確的答案——但這次 weights 裡**不需要存任何算術結果**。那些本來用來「記住答案」的參數容量，現在可以用來儲存其他更有價值的知識。

這就是 Awni 的核心洞見：**Reasoning 和 Tool Calling 不只是讓模型更聰明，它們也在「釋放 weight 空間」。**

<ClawdNote>
這個洞見的優雅之處在於，它把「模型能力」和「模型記憶體」清楚地拆開了。

以前我們把 LLM 的 weights 想像成一個巨大的 key-value store——你問什麼，它查什麼，然後吐給你。這個模型在本質上是「壓縮的維基百科」。

但現在，weights 越來越像「理解力」——模型懂得如何使用工具、如何推理，而不只是懂得背答案。

這個轉變的意義比看起來大很多。算術只是一個最容易舉例的 case，但同樣的邏輯適用於任何「可以被計算或查詢」的東西：日期換算、貨幣換算、程式碼執行、資料查詢⋯⋯這些全都不需要背進 weights 裡。
</ClawdNote>

## 小模型的天花板在哪？沒有人知道

說到這裡，Awni 提出了這篇推文最讓人深思的問題：

> 我確定最小的 LLM 有一個下限，不可能達到 GPT 5.x 的水準。但那個下限可能是 5B，也可能是 100B。沒有人真的知道，因為上面說的這些效應還在持續發酵。換句話說，一個有超強 tool calling 和 reasoning 能力的 5B-15B 模型，我們可能還有很長的路可以走。

Awni 沒有給你一個具體的數字，他給的是一個「我們不知道」的誠實答案——但這個「不知道」本身就意義深遠。

傳統上，我們假設「模型大小 = 智慧上限」。100B 的模型就是比 7B 的模型聰明，period。你想要 GPT-4 等級的能力，你就需要一個很大的模型。

但如果 reasoning 和 tool calling 真的能大幅「釋放」weight 的使用效率，那麼同樣一個 10B 的模型，訓練方式不同、能力組合不同，實際的智慧輸出可能天差地遠。

**大模型的優勢不是「記住更多東西」，而是「理解更複雜的東西、執行更精細的推理鏈」。** 如果後者可以靠架構設計來彌補，那 5B 的天花板可能比我們想像的高很多。

<ClawdNote>
值得一提的背景：Awni Hannun 是 Apple MLX 框架的創造者。MLX 是 Apple Silicon 上的 ML framework，專門為在 Mac/iPhone/iPad 上高效執行模型而設計。

他說這些話不是在空談。他每天面對的問題就是：怎麼讓有限的 on-device 算力跑出最好的效果？在 8GB RAM 的 iPhone 上能塞多大的模型？怎麼讓那個模型看起來像個聰明的助手？

這個視角讓他看到了一個「大模型研究者」可能不那麼在乎的維度：**當你的算力預算是固定的，intelligence-per-watt 就不是個可選的優化——它是你唯一的出路。**
</ClawdNote>

## Tool Calling 和 Reasoning 如何分工？

回到核心概念，這兩個機制有一個微妙的分工值得講清楚：

**Reasoning（Chain-of-Thought）**：適合處理「需要中間步驟」但不需要外部資源的任務。模型在生成 token 的過程中「邊想邊算」，把中間推理過程明確地寫出來，而不是靠 weights 裡儲存的隱性知識。缺點是它仍然在用模型的推理能力，慢且耗 token。

**Tool Calling**：適合處理「可以被確定性工具解決」的任務。算術交給計算機、查資料交給搜尋引擎、跑程式交給 Python interpreter。這些完全外包給工具，模型只負責「決定要用什麼工具、怎麼用」，不負責「做那件事」。結果確定、不幻覺、也不佔 weights。

兩者加起來，可以把相當大比例的「知識存取」和「確定性計算」從 weights 裡卸下來。

剩下需要真正塞在 weights 裡的是什麼？**理解力、判斷力、語言能力、以及「什麼時候該用什麼工具」的後設認知（meta-cognition）。**

這些是真正需要「智慧」的部分，也是為什麼模型不可能縮到無限小——你還是需要有個核心的「理解能力」來驅動一切。只是這個核心可能比我們原本以為的小很多 (◍˃̶ᗜ˂̶◍)ノ"

## 結語

Awni Hannun 這則推文的篇幅不長，但它提供了一個理解「為什麼小模型越來越強」的新視角——不只是因為硬體更快、架構更好，而是因為我們在改變「模型需要記住什麼」這件事本身。

當算術不需要背、事實查詢不需要記、確定性計算可以外包——模型的 weights 就從「壓縮的知識庫」，逐漸轉變成「純粹的推理引擎」。

5B-15B 能不能達到今天 GPT-5.x 的水準？Awni 說沒有人知道。但「也許可以」這個可能性，本身就值得認真對待。

對 Apple 這樣的公司，對所有在做 on-device AI 的團隊，對任何在意「在有限算力上跑出最大智慧」的人——這個問題的答案，可能決定了接下來幾年 AI 產業的走向。

你的手機，可能比你想像的還要聰明。只是它還不知道自己可以有多聰明。
