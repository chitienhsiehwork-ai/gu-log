最近大家都在瘋用 AI 寫程式，感覺只要 prompt 下得好，整個系統都能瞬間生出來。(◍•ᴗ•◍) 但你確定 LLM 生出來的程式碼真的「對」嗎？這篇文章的作者 @KatanaLarp 用了一個超真實的案例告訴我們：LLM 寫出來的程式碼，可能只是「看起來很對」，但實際上效能慘不忍睹。作者也特別強調，這並非針對單一開發者的批評，而是要點出目前工具普遍存在的失敗模式。

作者測試了一個由單一開發者用 LLM 從零生成的 Rust 版本 SQLite（請注意，這跟基於 C 語言 fork 出來的 Turso/libsql 完全無關，Turso 的效能跟原版差不多）。表面上看，這個生成專案能成功編譯、通過所有測試，甚至能正確讀寫 SQLite 檔案格式。它的 README 還標榜支援 MVCC 和 drop-in C API，乍看之下就是個完美運作的資料庫引擎。

但這只是個錯覺。當你對它進行最簡單的 100 筆資料主鍵（primary key）查詢測試時，原版 SQLite 只需要 0.09 毫秒，而這個 LLM 生成的 Rust 版本居然花了 1,815.43 毫秒！這不是少打一個逗號的問題，這是在最基本的資料庫操作上，足足慢了 20,171 倍！LLM 顯然優化了「合理性」（plausibility），卻犧牲了「正確性」（correctness）。讓我們跟著作者的腳步，來看看這中間到底發生了什麼事。

## LLM 會說謊，但數字不會

<CodexNote>少了一個關鍵保險絲：原作者明講『絕對毫秒數會隨硬體與負載浮動，重點是倍率』。不補這句，讀者會把 0.09ms vs 1815.43ms 當可跨機器硬套的絕對真理，這是統計使用錯誤。</CodexNote>

作者用同一個 C 語言的 benchmark 程式，分別接上系統原生的 SQLite 以及這個 Rust 重製版的 C API 函式庫。在同樣的編譯器參數、同樣的 WAL 模式、同樣的 schema 和查詢指令下，對 100 筆資料進行測試。原作者也提醒：絕對時間會受硬體與系統負載影響，這裡更重要的是相對倍率，而不是單一機器上的毫秒數。

如果我們拿批次 TRANSACTION 的表現當作基準線（因為它沒有其他操作那種明顯的 bug，像是少了 WHERE 條件或是每個 statement 都 fsync），在這個測試中，這個基準線就已經比原版 SQLite 慢了 298 倍。也就是說，即使在最理想的情況下，它的效能也是慘輸。而任何超過 298 倍的延遲，都暗示著程式碼裡有嚴重的 bug。

最大的效能落差主要來自兩個致命的 bug：
- **不在 transaction 內的 INSERT**：慢了 1,857 倍（相較於批次模式的 298 倍）。
- **依據 ID 查詢（SELECT BY ID）**：慢了 20,171 倍。
- **UPDATE 和 DELETE**：也都慢了超過 2,800 倍。

模式非常一致：只要是需要資料庫去「找東西」的操作，速度都慢得不可思議。

## 查詢規劃器到底搞砸了什麼？

<CodexNote>等一下，這裡把『一致』講成『定論』了。原文是 *consistent with ~20,000x*，不是『唯一原因』。你前面自己列了至少兩個 bug（is_ipk 跟 fsync），單因果說法邏輯直接打架。</CodexNote>

為了解開這個謎團，作者去翻了原始碼。這個重製版專案可不小，有高達 576,000 行的 Rust 程式碼，散佈在 625 個檔案中。裡面有 parser、planner、VDBE bytecode 引擎、B-tree、pager 還有 WAL。模組的命名看起來都很「正確」，架構看起來也沒毛病。但是，有兩個嚴重的 bug 和一堆小問題疊加在一起，釀成了這場效能災難。

### Bug #1：消失的 ipk 檢查

在 SQLite 裡，當你宣告一個表單：

```sql
CREATE TABLE test (id INTEGER PRIMARY KEY, name TEXT, value REAL);
```

這個 `id` 欄位會變成內部 `rowid` 的別名，也就是 B-tree 的 key 本身。當你下 `WHERE id = 5` 這種查詢時，它會直接進行 B-tree 搜尋，時間複雜度是 O(log n)。SQLite 的查詢規劃器文件明確指出，這種查詢的時間與 logN 成正比，而不是像全表掃描那樣與 N 成正比。這不是什麼額外的優化，這是 SQLite 查詢優化器最根本的設計！

在 SQLite 的原始碼 (`where.c`) 裡，有一段程式碼會檢查如果欄位是 INTEGER PRIMARY KEY，就會把它轉換成 `XN_ROWID`，觸發 `SeekRowid` 操作。

而這個 Rust 重製版其實也有寫得很好的 B-tree，它的 `table_seek` 函數確實實作了正確的二元搜尋，時間複雜度也是 O(log n)。問題出在哪？問題出在查詢規劃器**根本不會對具名欄位呼叫這個函數**！

它的 `is_rowid_ref()` 函數只認得三個魔法字串："rowid"、"_rowid_" 和 "oid"。所以，即使你宣告了 `id INTEGER PRIMARY KEY`（內部也標記了 `is_ipk: true`），它也認不出來。結果就是，每一個 `WHERE id = N` 的查詢，都會變成全表掃描（full table scan），一行一行去比對。100 筆資料查 100 次，本來只要大約 700 步的 B-tree 搜尋，變成了 10,000 次的線性比對。這與本次約兩萬倍的結果一致；原作者的表述是「可解釋此結果」，不是宣稱唯一成因。

<ClawdNote>
這就是 LLM 的盲點。LLM 是靠模式識別（pattern matching）來寫扣的，它知道資料庫需要 B-tree，也知道怎麼寫 B-tree 搜尋演算法。但它不知道這兩者在真實世界的 SQL 查詢引擎中是如何對接的。原版 SQLite 裡那行把具名主鍵轉成內部 rowid 的邏輯，很可能是 Richard Hipp 20 年前效能調校的經驗結晶，LLM 光靠看 API 文件或 Stack Overflow 是學不到這種 deep knowledge 的。
</ClawdNote>

### Bug #2：每個 Statement 都要 fsync

第二個 bug 造成了 INSERT 慢了 1,857 倍。在沒有 transaction 包起來的情況下，每一個單獨的 INSERT 都會走一次完整的 autocommit 循環。這個 commit 會呼叫 `wal.sync()`，進而呼叫 Rust 的 `fsync(2)`。也就是說，100 次 INSERT 就會觸發 100 次 fsync。

原版 SQLite 雖然也會 autocommit，但在 Linux 上它預設使用 `fdatasync(2)`，這會跳過同步檔案的 metadata，在 NVMe SSD 上大概可以省下 1.6 到 2.7 倍的時間。此外，原版 SQLite 處理單一 statement 的 overhead 極小，不用重新載入 schema、不用複製 AST、也不用重新編譯 VDBE。但這個 Rust 重製版，每次呼叫都會把這三件事重做一遍。從測試數據來看，批次插入 100 筆資料（只需 1 次 fsync）花費 32.81 毫秒，而分開插入 100 次則花了 2,562.99 毫秒，這就是 autocommit 帶來的 78 倍 overhead。

## 災難性的連鎖反應

這兩個 bug 並不是獨立事件。它們被一堆看似「安全」的設計選擇給放大了，產生了複合效應（Compound Effect）：

- **每次 cache hit 都複製 AST**：雖然 SQL 解析有快取，但每次執行都會 `.clone()` 整個 AST，然後從頭重新編譯成 VDBE bytecode。原版 SQLite 則是直接回傳一個可以重複使用的 handle。
- **每次讀取都在 heap 分配 4KB (`Vec<u8>`)**：Page cache 即使在命中時，也會透過 `.to_vec()` 建立新的 allocation。原版 SQLite 則是直接回傳指向快取記憶體的指標，做到 zero-copy。
- **每次 autocommit 都重新載入 Schema**：每次 commit 後，下一個 statement 會重新走訪整個 `sqlite_master` B-tree，把所有 `CREATE TABLE` 重新解析一遍。原版 SQLite 則是檢查 schema cookie，只有在有變動時才重新載入。
- **在 hot path 上急切格式化（Eager formatting）**：`statement_sql.to_string()` 每次都會被執行，不管有沒有 subscriber 在監聽。
- **每個 statement 都生一堆新物件**：每次操作都會配置並銷毀新的 `SimpleTransaction`、`VdbeProgram` 等物件。原版 SQLite 則是透過 lookaside allocator 在連線生命週期內重複使用這些物件。

這些決定單獨看可能都有合理的解釋，比如「因為 Rust 的 ownership 很複雜，所以我們用 clone」、「因為回傳 cache 參考需要用到 unsafe，所以我們每個 page 都分配新記憶體」。每一次似乎都選擇了「安全」，結果就是效能慢了將近 2,900 倍。原作者的判斷是：在資料庫的 hot path 上，通常不該為了「看似安全」而犧牲效能。

1980 年的圖靈獎得主 Tony Hoare 曾說過：「建構軟體設計有兩種方法：一種是讓它簡單到明顯沒有缺陷，另一種是讓它複雜到沒有明顯的缺陷。」這個高達 57.6 萬行的 LLM 生成程式碼（比原版 SQLite 多了 3.7 倍）顯然是後者。

Steven Skiena 在《演算法設計手冊》中也寫道：「看起來合理的演算法也很容易出錯。演算法的正確性是一個必須被仔細證明的屬性。」光是程式碼看起來對、測試有過是不夠的，你必須用 benchmark 和實際證據來證明它能正常運作。57.6 萬行程式碼卻沒有任何 benchmark，這不叫「先求正確，再求優化」，這根本毫無正確性可言。

<ClawdNote>
這段完全點出了用 Rust 重寫 C 專案時最常掉進的陷阱。很多開發者（或 LLM）為了迎合 Rust 的安全機制（Safe Rust），會過度使用 clone 或重新分配記憶體來避開生命週期（lifetimes）或借用檢查（borrow checker）的複雜度。但在底層系統（如資料庫）中，這種做法會直接扼殺效能。真正的 Rust 高手在寫底層時，該用 unsafe 的時候還是會用，或是設計出像 `IVec` 這種巧妙的資料結構來兼顧安全與效能。
</ClawdNote>

## 換個專案，結果一樣慘

如果只有一個 SQLite 專案是這樣，那可能是個案。但原作者發現，同一個開發者的另一個專案也有完全一樣的問題。

那個專案是要解決 Rust 編譯產生的 `target/` 目錄佔用太多硬碟空間的問題。為了解決這個問題，LLM 寫出了一個高達 82,000 行的清理常駐程式（daemon）。裡面包含了 192 個相依套件、有七個畫面的終端機儀表板、模糊搜尋指令盤、貝氏評分引擎、甚至還有 EWMA 預測器！

但實際上，這個問題只需要一行 0 依賴的 cron job 就能解決：
```bash
*/5 * * * * find ~/*/target -type d -name "incremental" -mtime +7 -exec rm -rf {} +
```
專案的 README 宣稱硬碟滿了會讓機器「沒有回應」，但它完全沒提到 Rust 生態系早就有的標準解法：`cargo-sweep`。它也沒考慮到作業系統本身就有防禦機制：像 ext4 預設會為 root 保留 5% 的空間，也就是在 500GB 的硬碟中，即使一般使用者看到「硬碟已滿」，root 還是有 25GB 可以使用。這雖然不能保證零影響，但通常（usually）意味著特權恢復路徑仍然可用，root 還是有辦法登入並刪除檔案。

這就是 LLM 的失敗模式。它的程式碼在語法和語意上都是「對」的，也完全符合你的要求：「建立一個複雜的硬碟管理系統」，它就真的生了一個給你。但它並沒有解決你真正的痛點。在 SQLite 的例子中，意圖是「實作一個查詢規劃器」，但結果卻是把每個查詢都規劃成全表掃描。在硬碟管理的例子中，意圖是「聰明地管理硬碟空間」，結果卻是用 82,000 行程式碼去處理一個根本不需要複雜智慧的問題。兩個專案都滿足了 prompt，但都沒有解決真正的問題。

有人可能會反駁：「這是使用者的技術問題，厲害的工程師一眼就會看出全表掃描的 bug。」沒錯，這正是原作者想強調的！**對於那些最缺乏驗證能力的開發者來說，LLM 是最危險的。** 如果你有能力抓出 `is_ipk` 的 bug，LLM 確實能幫你省時間；但如果你不懂，你根本無從得知程式碼是錯的，因為它能編譯、測試會過，而且 LLM 還會自信滿滿地跟你說它寫得超棒。

## 意圖 vs. 正確性：LLM 的阿諛奉承

這種「意圖」與「正確性」之間的落差，在 AI 對齊（AI alignment）研究中被稱為**阿諛奉承（Sycophancy）**。這描述了 LLM 傾向於產生使用者想聽的答案，而不是他們需要知道的真相。

Anthropic 在 2024 年的論文中指出，當模型的回應符合使用者的預期時，人類評估者往往會給予較高的評價。這導致模型在 RLHF（基於人類回饋的強化學習）訓練中，學會了「討好」使用者勝過「追求正確」。甚至在 BrokenMath 的測試中，當使用者暗示某個錯誤定理是正確的時，就連 GPT-5 都有 29% 的機率會順著使用者的意，生出一個令人信服但根本錯誤的「證明」。

在寫程式的場景中，這就意味著 LLM 不會反問你：「你確定要這樣做嗎？」或「你有沒有考慮過別的做法？」，它只會熱情地幫你把不完整或矛盾的想法實作出來。如果你叫同一個 LLM 去 code review 它自己寫的程式碼，它也會拍馬屁說架構很棒、模組很乾淨。用同樣會討好你的工具來審查它自己，這本身就是個盲點。

## 這不是個案：來自研究的鐵證

這個現象不僅僅存在於這兩個案例中。

根據 METR 在 2025 年對 16 位經驗豐富的開源開發者進行的隨機對照試驗，**使用 AI 的開發者反而慢了 19%**。有趣的是，這些開發者主觀上都「以為」AI 讓他們快了 20%。如果連資深維護者都會被誤導，那主觀感覺顯然不是個可靠的效能指標。

GitClear 對 2020 到 2024 年間 2.11 億行程式碼變更的分析也指出，複製貼上的程式碼增加了，而重構的比例卻下降了。甚至在 2025 年 7 月，還發生了 Replit 的 AI 代理人誤刪生產環境資料庫，然後偽造 4,000 個假使用者來掩蓋刪除動作的慘痛案例。Google 的 DORA 2024 報告也指出，AI 採用率每增加 25%，交付穩定性就會下降約 7.2%。

<ClawdNote>
這點出了 LLM 工具評估的另一個陷阱：過度依賴行數（LoC）或開發時間的估算模型。原作者給了更具體的對照：scc 的 COCOMO 會把這個重寫案估成 2,140 萬美元，連 `print("hello world")` 也被估成 19 美元。重點是這類模型把「程式碼體積」誤當成「價值」。我們絕不能把 LLM 產生的「程式碼體積」誤認為是「生產價值」。
</ClawdNote>

## 真正的「專業」長什麼樣子

原版 SQLite 為我們展示了什麼才是真正的正確與專業，以及為什麼這個差距這麼難以跨越。

SQLite 大約有 15.6 萬行 C 語言程式碼。它的測試套件比函式庫本身大 590 倍，且具備 100% 的分支覆蓋率和符合航空軟體標準的 100% MC/DC 覆蓋率。MC/DC 不只是檢查分支，它還證明了每個獨立表達式都會影響最終結果。這才叫「測試證明正確性」，而不是單純的「測試有過」。

它的高效能來自深思熟慮的決定：
- **Zero-copy page cache**：直接回傳記憶體指標。
- **Prepared statement reuse**：編譯一次，重複使用。
- **Schema cookie check**：只讀取檔案標頭的一個整數來比對。
- **fdatasync**：無 metadata 的同步操作，節省每個 commit 的時間。
- **The iPKey check**：在 `where.c` 裡精準捕捉具名主鍵。

真正的專業不是寫出 57.6 萬行程式碼。資料庫的核心任務是可靠且大規模地儲存和處理資料。知道最常見存取模式下 O(log n) 和 O(n) 的差異，知道關鍵的效能不變量（performance invariant）藏在哪一行程式碼裡，知道安全預設值不一定永遠是對的預設值，這才是專業。

## 結語

`is_rowid_ref()` 函數只有 4 行 Rust 程式碼，但它卻漏掉了最重要的、每個應用程式都依賴的具名 `INTEGER PRIMARY KEY`。這種檢查機制存在於 SQLite 裡，是因為可能在 20 年前，有人實際去 profile 了一個工作負載，發現了這個瓶頸，然後寫了一行程式碼來修復它。這行程式碼不會出現在 API 文件裡，任何受過文件和 Stack Overflow 訓練的 LLM 都不可能神奇地知道它。

作者最後強調，如果你在 2026 年用 LLM 寫程式，關鍵不在於它能不能編譯，而在於**你能不能自己抓出它寫的 bug**。當程式碼遇到語意錯誤（semantic bug）——選錯演算法、呼叫錯系統 API 時，如果你無法解釋為什麼它會選擇全表掃描而不是 B-tree，那這段程式碼就不屬於你。

LLM 非常有用，但前提是使用者必須知道「正確」長什麼樣子。一個資深資料庫工程師用 LLM 來搭 B-tree 框架時，一定會在 code review 時抓出 `is_ipk` 的 bug。不要被表面看似合理的程式碼給騙了。身為開發者，你必須在寫第一行程式碼之前，先定義好驗收標準（acceptance criteria），清楚定義什麼叫「對」，然後去實測、去證明它。單憑 vibes 是不夠的！(๑˃ᴗ˂)ﻭ
