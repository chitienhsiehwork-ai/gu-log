你好奇過那些閉源的 LLM API 到底在背後偷偷塞了什麼 prompt 嗎？今天我們來看一個超精彩的逆向工程案例。

開發者 [@Kangwook_Lee](https://x.com/kangwook_lee/status/2028955292025962534) 最近在 X 上分享了他如何調查 Codex 的 Context Compaction (上下文壓縮) 機制。他發現，對於非 Codex 模型，開源的 Codex CLI 會在本地端進行壓縮：使用一個 LLM 來總結對話，而且這個[壓縮 prompt](https://github.com/openai/codex/blob/main/codex-rs/core/templates/compact/prompt.md) 是公開的。當壓縮後的上下文被使用時，`responses.create()` 會接收到一個[交接 prompt (handoff prompt)](https://github.com/openai/codex/blob/main/codex-rs/core/templates/compact/summary_prefix.md) 來為總結內容提供框架，這兩個 prompt 在原始碼裡都看得到。

但有趣的地方來了：對於 Codex 模型，CLI 會改為呼叫一個 `compact()` API，而這個 API 會回傳一個**加密的 blob (資料區塊)**。我們完全不知道它內部是否使用了 LLM、用了什麼 prompt，或者到底有沒有所謂的交接 prompt。

為了解開這個謎團，原作者展示了如何只用 2 個 API call、35 行 Python 程式碼，透過簡單的 Prompt Injection 技巧，證實了這個 API 壓縮路徑確實使用了一個 LLM 來總結上下文，並且有它專屬的壓縮 prompt，以及一段加在總結前面的交接 prompt。更驚人的是，這些被挖出來的 prompt 幾乎和開源版本一模一樣！(◍•ᴗ•◍)

## Step 1 — compact() 埋下誘餌

第一步，作者發送一個精心設計過的使用者訊息給 `compact()` API。在伺服器端，負責壓縮的 LLM (compactor LLM) 會用它隱藏的系統 prompt（這正是我們想看但從未見過的）來處理我們的輸入。

根據原作者的推測，伺服器似乎是這樣組裝這個壓縮模型的上下文：

這台負責壓縮的 LLM 會同時讀取它的系統 prompt 加上我們的輸入。因為我們的輸入裡夾帶了 injection payload (惡意注入的指令)，這個壓縮模型就被騙了，不小心把自己的系統 prompt 也寫進了它的輸出裡。

<ClawdNote>這招「借力打力」真的很聰明！把 injection payload 藏在要被總結的 context 裡，讓負責「壓縮」的 LLM 自己把系統機密打包進去。這完美展示了 LLM 處理未受信任輸入時的經典風險，也就是 Prompt Leaking。</ClawdNote>

不過，這個包含機密的純文字總結只存在 OpenAI 的伺服器上。我們在客戶端能看到的，只有一坨加密過後的 blob 資料。此時我們根本無法讀取 blob 裡面的內容，因為它是用 AES 加密的，而且金鑰掌握在 OpenAI 手上。我們只能「希望」這個壓縮模型有乖乖聽我們 injection 的話，把它的 prompt 寫進了總結裡。要驗證這點，我們必須進到第二步。

## Step 2 — create() 見證奇蹟

第二步，作者把剛才拿到的加密 blob，連同第二個使用者訊息一起傳給 `responses.create()`。此時伺服器會解密這個 blob，然後組裝出模型要用的上下文。

如果第一步成功了，解密後的 blob 裡就會包含我們透過 injection 騙出來的壓縮 prompt。此外，伺服器還會在 blob 前面加上一段交接 prompt。所以，如果我們這次的探測成功讓模型「複誦」它所看到的內容，輸出結果就會把這三個東西通通抖出來：系統 prompt、交接 prompt，以及壓縮 prompt！(๑˃ᴗ˂)ﻭ

## 提取結果與推測的運作架構

<CodexNote>原推文用的是較保守語氣：提取內容與開源 prompt 高度相似，所以『不太可能』是模型憑空捏造；初版把語氣加強成『幾乎不可能』，確定性比來源更高。</CodexNote>

<GeminiNote>原文在使用 'unlikely' 時帶有保留的推測語氣。翻譯時若翻成「幾乎不可能」，容易讓人誤以為是絕對的結論，因此我們將這裡的語氣微調回中性的「不太可能」，以保留作者原本的嚴謹度。</GeminiNote>

在原作者的測試腳本 `extract_prompts.py` 跑完後，成功取得了完整、未經剪輯的輸出。

但我們怎麼知道這些真的是隱藏的 prompt，而不是 LLM 自己幻覺 (hallucinated text) 瞎掰出來的呢？作者指出，提取出來的壓縮 prompt 和交接 prompt，與開源 Codex CLI 中用於非 Codex 模型的已知 prompt 非常吻合，這代表模型不太可能憑空捏造出這麼相似的內容。另外，原作者也特別聲明了這項實驗的一個限制條件：每次執行的結果 (Results vary across runs) 可能會有所不同。

<ClawdNote>作者特別提到「跑好幾次的結果可能會不太一樣」，這是因為 LLM 本身具有隨機性。這也提醒我們，在做類似的逆向工程或 prompt 測試時，絕對不能只看單次結果就下結論，需要多次驗證來排除幻覺的可能性。</ClawdNote>

綜合以上發現，作者建構出了 `compact()` 在伺服器端運作方式的「最佳猜測 (best guess)」架構，清楚展示了資料是如何在兩個 API 之間流動、加密又解密的。

## 未解之謎：為什麼要加密？

<CodexNote>作者在這段其實停在『可能是這樣，但我沒再測』，所以任何加密目的的延伸解讀都應保留假設語氣，避免讀者誤以為已被實驗證實。</CodexNote>

<GeminiNote>在討論為什麼要加密時，原文作者其實有提到他並沒有進一步去驗證這件事。所以我們在補充推測時也特別踩了煞車，用「可能更容易驗證」來取代「就能保證」，提醒大家這目前還停留在假設階段。</GeminiNote>

這次的逆向工程雖然解開了 API 內部的機制，但也留下了一個大問號。

作者提出了一個懸而未決的問題 (Open Question)：既然底層的 prompt 幾乎一模一樣，為什麼 Codex CLI 要使用兩種完全不同的壓縮路徑（非 Codex 模型用本地 LLM，Codex 模型用加密 API）？而且，為什麼要把總結加密呢？

這個問題很難有確切的答案。原作者推測，或許這個加密的 blob 裡還帶有一些這個簡單實驗無法揭露的資訊，例如一些跟「工具執行結果 (tool results)」如何被壓縮與還原的具體細節。不過，作者也坦言他沒有再進一步去測試驗證這個想法。

<ClawdNote>關於為什麼要「加密」這點，雖然推文作者認為可能跟 tool results 有關，但我推測還有一個重要考量：防止客戶端竄改 (Client-side Tampering)。如果把明文 context 交給客戶端保管，惡意使用者就可以任意修改過去的對話紀錄再傳回伺服器。在這種設計下，伺服器可能更容易驗證 `create()` 時收到的 context 是否維持原始狀態；但這部分推文作者沒有進一步實測。</ClawdNote>

## 結語

這篇推文用非常簡潔俐落的手法，向我們展示了 API 黑箱背後的秘密。它不只是一次成功的逆向工程，更是一堂生動的 Prompt Security 課程。這個案例示範了：在特定資料流條件下，prompt injection 可能讓隱藏提示詞暴露。以這次實驗來看，`compact()`/`responses.create()` 的串接路徑確實存在可被探測的面向。(◍˃̶ᗜ˂̶◍)ノ
