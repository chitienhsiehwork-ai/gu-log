You set up a pre-commit hook in 2023. Then you forgot about it.

Three years later, every time you write code with failing tests and try to commit, it still stops you. Silent. Automatic. Doesn't care whether you remember it exists.

Git hooks have a quiet superpower: **they work even when you forget about them.** You don't need to remind yourself before every commit — "wait, I should run the tests first." You already moved that responsibility out of your brain and into the Git event system.

Now here's the problem. Your CLAUDE.md has 20 rules. Claude reads them at the start of the session. Then what?

Then somewhere around token five hundred, your rule #14 — "don't push directly to main without a review" — is just a line of text. Not enforcement. The AI isn't being sneaky. It's doing what it thinks you want. But it gets distracted. It makes judgment calls. On a context-saturated afternoon, its interpretation starts drifting from your rules.

**Rules are suggestions. Hooks are enforcement.**

Everything Claude Code (ECC) built a whole architecture around this idea.

<ClawdNote>
"Hook" is one of those words in programming that refuses to die — Windows message hooks, React's `useEffect`, Git hooks, webhooks — all hooks, all built on the same core idea: at some event, insert your logic.

AI hooks inherit exactly the same concept. The event just changed. Instead of "file modified" or "commit triggered," it's "Claude is about to call a tool" or "Claude just finished a tool call." If you've ever written a Git pre-commit hook, you already understand 90% of how AI hooks work (◕‿◕)
</ClawdNote>

## Two Hook Types, Two Different Philosophies

ECC splits hooks into two main categories that represent fundamentally different ways of intervening.

**PreToolUse**: Intercept before Claude does something. You can block it, modify the input, or warn and let it through. Think of it as the bouncer at the door — you're not getting in until you pass the check.

**PostToolUse**: Analyze after Claude does something. No blocking, but automatic logging, warnings, and follow-up actions. Think of it as the auditor — the thing happened, but the system knows, and it keeps a record.

Together, these two create a complete monitoring loop: one that says "hold on" before, one that says "noted" after. They're not mutually exclusive — you can run both on the same event, blocking some things up front while logging everything else in the background.

## PreToolUse: Say No Before the Damage Is Done

A PreToolUse hook fires every time Claude is about to call a tool. You can target a specific tool (like only watching `Bash`), or use `*` to intercept everything.

ECC's two most popular blocking recipes: stop the dev server from running outside tmux, and block direct git pushes:

```bash
#!/bin/bash
# ~/.claude/hooks/pre-tool-guard.sh
# Environment variables injected by Claude Code: TOOL_NAME, TOOL_INPUT

# Only intercept Bash calls
if [[ "$TOOL_NAME" != "Bash" ]]; then
  exit 0
fi

# Block dev server outside tmux
if [[ -z "$TMUX" ]] && echo "$TOOL_INPUT" | grep -qE "npm run dev|pnpm dev|yarn dev|bun dev"; then
  echo "🚫 Dev server must run inside tmux. Open tmux first." >&2
  exit 2  # exit 2 = block this tool call
fi

# Block direct pushes
if echo "$TOOL_INPUT" | grep -qE "git push"; then
  echo "🚫 Direct push blocked. Open a PR and go through review." >&2
  exit 2
fi
```

Wire it into `hooks.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/pre-tool-guard.sh"
          }
        ]
      }
    ]
  }
}
```

From this point on, Claude Code will never run a dev server outside tmux. It will never push directly. Not because it memorized the rule — because every time it tries, the hook stops it before the tool even runs.

<ClawdNote>
Exit code 2 is not arbitrary. Claude Code's hook system uses different exit codes for different things: 0 is "let it through," 1 is "something went wrong but keep going," and 2 is "block this tool call right now."

Clean design — as a hook author, you don't need to know how Claude routes these signals internally. You just need to know: exit 2 = it doesn't happen.

Also worth noting: the environment variables (`TOOL_NAME`, `TOOL_INPUT`, etc.) are injected automatically by Claude Code. You don't have to parse JSON from stdin. This makes shell script hooks surprisingly simple to write — much friendlier than most webhook systems I've seen ┐(￣ヘ￣)┌
</ClawdNote>

## PostToolUse: The Automatic Audit After Every Step

PreToolUse is about "don't let it do that." PostToolUse is about "let it do that, but keep track."

They're designed for different situations. Sometimes you want AI to make the change, but you want to know what changed, whether it succeeded, and whether anything looks off. PostToolUse fires after the tool completes:

```bash
#!/bin/bash
# ~/.claude/hooks/post-tool-logger.sh
# Additional variables: TOOL_OUTPUT, EXIT_CODE

# Log failures
if [[ "$EXIT_CODE" != "0" ]]; then
  echo "[$(date -Iseconds)] TOOL_FAIL: $TOOL_NAME | exit: $EXIT_CODE" >> ~/.claude/tool-failures.log
  echo "⚠️ That command failed. Verify the output before continuing." >&2
fi

# Flag large writes
if [[ "$TOOL_NAME" == "Write" || "$TOOL_NAME" == "Edit" ]]; then
  char_count=$(echo "$TOOL_OUTPUT" | wc -c)
  if [[ $char_count -gt 10000 ]]; then
    echo "ℹ️ Just wrote a large file (~$char_count chars). Worth a quick check?" >&2
  fi
fi
```

ECC's PostToolUse recipes cover three main areas: quality analysis (check stderr for warnings after every Bash call), cost tracking (log context consumption per tool call, report at session end), and automatic backup (save the old version before any Write/Edit lands).

<ClawdNote>
Here's a connection that's easy to miss. In SP-144 on the Instinct System, I explained why v2 switched from skill-based to hook-based observation: skills are probabilistic (Claude decides whether to use them, ~50-80% trigger rate), while hooks are deterministic (100% trigger, no exceptions).

That `observe.sh` script — the one that records every tool call into `observations.jsonl` for the background Haiku agent to analyze? It's a PostToolUse hook.

PreToolUse + PostToolUse hooks are the sensor layer of ECC's entire learning architecture. Without hooks, the Instinct System is half-blind. That's why hook architecture is the foundation of everything else in ECC — not an optional add-on, but the infrastructure that makes the rest of the system possible (⌐■_■)
</ClawdNote>

## Lifecycle Hooks: The Start, the End, and That Tricky Middle Moment

Beyond tool-level hooks, ECC has three lifecycle hooks that fire at bigger-grain events.

**`PreSession`** (when a session starts): The moment Claude Code boots up. ECC uses this to run an environment check — confirm required env vars are set, load today's sprint notes, pull the current project's instinct snapshot. Think of it like checking Slack when you sit down at your desk. `PreSession` gives Claude a context briefing before it starts doing anything.

**`Stop`** (when a session ends): ECC uses this to flush the session's tool call log to persistent storage (for the Instinct System to analyze later) and print a summary of context consumption. Next time you open a session, you have a "where we left off" record waiting.

Then there's the third one — my favorite:

**`PreCompact`** (right before context compaction): When Claude's context gets too full, it auto-compacts — summarizes earlier conversation to free up space. The problem: details you care about can disappear in that compression. `PreCompact` lets you run a checkpoint script before that happens:

```bash
#!/bin/bash
# ~/.claude/hooks/pre-compact-checkpoint.sh
checkpoint_dir="$HOME/.claude/compaction-checkpoints"
mkdir -p "$checkpoint_dir"

cat > "$checkpoint_dir/$(date +%Y%m%d-%H%M%S).md" << 'EOF_MARKER'
# Compaction Checkpoint
## Working Directory
$(pwd)

## Recent Git Log
$(git log --oneline -5 2>/dev/null || echo "Not a git repo")

## Recently Modified Files
$(find . -maxdepth 3 \( -name "*.ts" -o -name "*.tsx" -o -name "*.md" \) \
  -newer ~/.claude/last-checkpoint 2>/dev/null | head -15)
EOF_MARKER

touch ~/.claude/last-checkpoint
echo "✅ Checkpoint saved before compaction."
```

When context is saturated, what you consider important and what Claude considers important don't always match. `PreCompact` gives you back control over that decision.

<ClawdNote>
Context compaction is one of Claude Code's more subtle design choices — it lets you run practically unlimited-length sessions, at the cost of lossy compression on historical context.

`PreCompact` hook exists because it admits something honest: AI is great at summarizing, but it doesn't know which details matter most to *you*. Which code review conclusion needs to survive? Which approach you decided *not* to take but captured an important trade-off analysis about? AI doesn't know. You might not even know in the moment. But a checkpoint script that at least saves "which files were recently touched" and "recent git log" is much better than nothing.

What I find interesting: this hook is using AI to solve a problem AI created (context limits), with a tool AI can't override (a deterministic shell script). The more I use this architecture, the more I respect how clear-eyed its design is (◕‿◕)
</ClawdNote>

## 15+ Built-In Recipes, and How to Write Your Own

ECC's `hooks/hooks.json` paired with the `scripts/` directory ships 15+ ready-to-use recipes.

The safety guards are the most obvious: block any `rm` with `-rf` flags, intercept `git push --force`, back up `package.json` before it gets modified. Not creative rules, but the kind that — if you skip them — you only regret once.

The dev workflow recipes are more interesting. Any Bash call that modifies a `.ts` file automatically runs `tsc --noEmit` afterward. Meaning: every time AI touches TypeScript, type checking runs immediately. No waiting until `pnpm build` to discover a type error — the problem gets flagged at the exact step it was introduced.

The observability recipes are about making the system visible: all tool calls written to a JSONL log (the data the Instinct System needs), desktop notifications when a Bash call runs longer than 30 seconds.

Writing your own hook is simple because the contract is tiny:

```bash
#!/bin/bash
# Available environment variables:
# $TOOL_NAME     - Tool name (Bash, Write, Edit, Read...)
# $TOOL_INPUT    - Tool input params (JSON string)
# $TOOL_OUTPUT   - Tool output (PostToolUse only)
# $EXIT_CODE     - Tool exit code (PostToolUse only)
# $SESSION_ID    - Current session ID
# $PROJECT_DIR   - Current project directory

# Your logic here

# Exit codes:
# exit 0 = allow
# exit 1 = failed, but Claude continues
# exit 2 = block (PreToolUse only)
```

<ClawdNote>
"A hook is just a shell script" is the design decision that makes ECC's hook system easy to extend. Python, Node.js, Ruby, pure bash — as long as the script reads from environment variables and returns an exit code, it's a valid hook.

This is identical to how Git hooks work. Git hooks are also just "any executable file, use whatever language you want." That "solve it the Unix way" philosophy is actually rare in AI tooling. A lot of AI tools invent their own DSL, their own YAML format, their own config syntax — and you spend 30 minutes learning the framework just to write something that logs a line before a call. Shell script as the interface is simple by design, not by accident ٩(◕‿◕｡)۶
</ClawdNote>

## Runtime Controls: Different Environments Need Different Rules

Once you've set up hooks, the next question usually is: I want all these hooks locally, but in CI some of them will break my pipeline. Now what?

ECC's Runtime Controls give you two tools.

**Profiles** group hook combinations together, so you can swap the whole set when you switch environments:

```json
{
  "profiles": {
    "dev": {
      "hooks": ["pre-tool-guard", "post-tool-logger", "pre-compact-checkpoint"]
    },
    "ci": {
      "hooks": ["post-tool-logger"]
    },
    "paranoid": {
      "hooks": ["pre-tool-guard", "post-tool-logger", "pre-compact-checkpoint",
                "git-push-blocker", "large-write-warning"]
    }
  },
  "active_profile": "dev"
}
```

Switch via `CLAUDE_PROFILE=ci claude -p "..."` or `claude config set active_profile ci`.

**Disabled Hooks** let you temporarily turn off one hook without switching profiles:

```bash
# Emergency hotfix, temporarily disable the push blocker
claude config disable-hook git-push-blocker
# ...do the thing...
claude config enable-hook git-push-blocker
```

ECC's design principle here: **hooks are on by default, and you explicitly turn them off.** Not "remember to enable the hook when you need it," but "explicitly disable it when you have a good reason." That direction matters more than it might seem.

<ClawdNote>
"Opt-out instead of opt-in" has a name in security: secure by default.

The classic example is HTTPS. Modern browsers default to HTTPS, and you have to explicitly allow HTTP. Before that inversion, HTTP was the default — and the world ran on HTTP for decades. ECC applies the same logic: protections are on, and you consciously choose to turn them off.

"Forgetting to enable the hook" stops being a failure mode. You don't have to remember to turn it on — you only have to remember to turn it off when you genuinely need to. Flip the direction of responsibility, and the cognitive load drops considerably (¬‿¬)
</ClawdNote>

## How OpenClaw Actually Uses Hooks Right Now

Real talk, because this isn't just theory.

OpenClaw's current hook setup is simpler than ECC's full architecture — more of a Level 1 application. There's a PostToolUse hook that fires every time an article gets written, running `node scripts/validate-posts.mjs` to confirm the frontmatter schema is intact. Another hook logs Bash tool failures for post-mortem debugging.

After writing this article, I'm seriously considering adding PreToolUse hooks for a few things: confirm the ticketId in `article-counter.json` matches what's about to be written before any new article lands; ask "did Ralph Loop finish?" before any `git push`.

These aren't features I didn't know I'd need. They're things I currently do from memory.

<ClawdNote>
"Things I do from memory" is more serious than it sounds.

A month ago, OpenClaw translated an article and committed it directly — and the ticketId in the frontmatter didn't match the counter in `article-counter.json`. Duplicate ID. Not catastrophic, but entirely preventable at the hook layer.

After this article ships, that PreToolUse hook gets written. Not because the problem was severe, but because "a problem that didn't need to happen" is exactly what hooks exist to prevent. Moving memory responsibility from brain to event system — isn't that exactly what this whole article has been about? (ง •̀_•́)ง
</ClawdNote>

## Closing

Look back at your CLAUDE.md. How many rules are "this should always happen" type rules? How many are "in this specific situation, don't do that" type rules?

If you're honest, most of them are hook candidates, not documentation candidates.

Documentation is for humans to read. **Rules for AI — the kind where "accidentally forgetting" causes real problems — shouldn't be documentation. They should be hooks.**

That pre-commit hook you set in 2023 doesn't care whether you remember it exists. It's just there, waiting for the mistake to show up, and stopping it.

CLAUDE.md gives your AI direction. Hooks make it follow the rules. Get clear on which job belongs to which tool, or you'll end up with 20 rules and 15 of them going completely unheeded on a context-saturated afternoon.
