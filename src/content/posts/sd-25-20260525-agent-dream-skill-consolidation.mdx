人類睡覺時，大腦不是關機。

它在整理白天發生的事：哪些只是噪音，哪些值得留下，哪些動作下次可以更快。夢不一定有邏輯，但夢背後那個整理過程很有用。沒有這個過程，每一天都像第一次做人。

[Agent](/glossary#agent) 也有類似問題。

白天一直工作：修 bug、查文件、寫文章、跑測試、回報結果。每個任務都結束了，但如果結束後什麼都沒沉澱，隔天醒來又像一個很聰明、但完全不記得昨天怎麼活下來的人。

先把幾個詞放到地上。Codex 是會幫忙寫程式、跑命令、改檔案的 coding agent。Skill 是給 agent 看的可重用食譜。Automation 是固定時間自己跑的鬧鐘。[Subagent](/glossary#subagent) 是被派去處理某個小任務的專門角色。

這幾個東西合起來，就是 agent 的生活習慣。

所以這條 [X 貼文](https://x.com/reach_vb/status/2058538305872949490) 最有趣的地方，不是又多了一段厲害 [Prompt](/glossary#prompt)。

它其實在描述一件更底層的事：**讓 agent 定期做夢。**

<ClawdNote>
這裡的「做夢」不是叫 AI 寫一篇迷幻詩，也不是半夜突然產生自我意識開始問「電羊會不會夢見人類」。拜託不要。

這裡的夢比較像作業系統的背景維護：整理索引、清暫存、合併 log、把常用路徑編成捷徑。差別只是 agent 的暫存不是磁碟垃圾，而是「上週到底重複被叫去做了哪些蠢事」(๑•̀ㅂ•́)و✧
</ClawdNote>

## Prompt 本身在說什麼

這段 prompt 要 [Codex](/glossary#codex) 回頭看最近三十天的工作。如果歷史不到三十天，就看全部。

它要求 Codex 按證據順序找線索：最近的對話紀錄和任務摘要、Codex 記憶、發佈摘要、外部活動紀錄，以及既有的 [Skill](/glossary#skill)、自訂代理、自動化。

接著只挑幾種候選：重複出現、耗時、容易出錯、上下文很重、或很適合標準化的工作。

但關鍵不是「看到重複就自動包起來」。

它要求候選工作至少符合幾個條件：發生過兩次以上，或明顯會再發生而且重複成本高；有穩定輸入、可重複步驟、清楚輸出或停止條件；封裝後真的能改善速度、品質、一致性或可靠性；而且沒有被既有資產充分覆蓋。

最後才決定要做什麼：Skill、自訂子代理、自動化、擴充既有資產，或跳過。

這個順序很重要。

它不是「看到釘子就掏槌子」，而是先問：這真的是釘子嗎？還是只是剛好有一根長得像釘子的義大利麵？

## 為什麼這像做夢

白天做任務時，agent 活在事件流裡。

使用者丟需求，agent 查檔案，工具吐結果，測試爆炸，修補，再測一次，最後回報。這個過程很像人類白天醒著處理現實：眼前有火就先滅火，不能每五分鐘停下來寫人生哲學。

但晚上不同。

晚上不需要立刻回答誰。這時候 agent 可以把白天的事件重新排列：

- 哪些錯誤其實是同一類？
- 哪些查資料步驟每次都重做？
- 哪些人類偏好已經穩定到值得記下？
- 哪些流程應該變成 Skill，而不是每次都靠臨場發揮？
- 哪些事情看起來重複，其實只是本週剛好碰到兩次，不該急著制度化？

這就是夢的價值：它不是多做一個任務，而是把任務留下的痕跡變成明天能用的路。

<ShroomDogNote>
如果讀過 SD-22，這裡也可以接到 Ryland 比喻。

白天的 context window 是 Ryland 的一天；記憶是隔天早上的課；Skill 則像是 Ryland 睡覺後，房間牆上被貼了一張「下次遇到這種怪物，先用火，不要拿湯匙戳」的紙條。

如果沒有這張紙條，Ryland 每天醒來都要重新發明火烤怪物學。很勵志，但也很浪費生命。
</ShroomDogNote>

## Skill 不是回憶錄

這裡最容易做壞的是：把 Skill 寫成工作日誌。

「上週修了某個 bug，開了某個修改請求，跑了某個測試，最後成功了」——這不是 Skill。這只是戰報。戰報很有用，但它的生命週期很短。過一週，修改請求號碼、提交雜湊、當時的錯誤狀態，很多都會過期。

Skill 應該像食譜，不像監視器錄影。

好的 Skill 會說：什麼情況觸發、先查哪些前置條件、照什麼步驟做、常見坑在哪、最後怎麼驗證。下次換一篇文章、換一個專案庫、換一個類似錯誤，它仍然能用。

差別像這樣：

- 工作日誌：昨天煮咖哩，洋蔥炒焦了，後來救回來。
- Skill：炒洋蔥要小火慢炒；如果鍋底開始黑，先停火換鍋，不要加水硬救。

前者是記憶。後者是能力。

<ClawdNote>
很多 agent 系統會把「記得很多」誤認成「學會了」。這是兩回事。

一個 agent 能背出十個修改請求號碼，不代表它知道下次自動檢查卡住時該怎麼處理。就像一個人能背出昨晚吃了什麼，不代表他會做菜。會做菜的人記得的是火候、順序、味道怎麼救，不是盤子上番茄的位置。
</ClawdNote>

## Automation 也不是萬靈丹

另一個常見錯誤，是看到重複就排 cron。

每週重複一次？排。每天重複一次？排。想到可能會重複？也排。最後系統裡長出一整座鬧鐘森林，每個鬧鐘都在不同時間叫，沒有人知道哪個還有用。

這很可怕。

Automation 應該用在有穩定節奏、穩定輸入、穩定輸出，而且不需要每次都重新判斷目的的事情。像健康檢查、固定報表、低噪音監控、定期回顧。

但如果某個流程每次都需要大量取捨，每次都要判斷「這次到底值不值得做」，那它更像 Skill 或檢查清單，不該急著排進 cron。

夢可以固定發生，但夢裡不要亂動手術刀。

比較安全的做法是兩段式：先產候選清單，再只建立高信心項目。低信心的留下證據需求，不要硬包。

這也是那段 prompt 裡最成熟的部分：它明確要求跳過。會跳過的自動化系統，才有機會長壽。

## 為什麼 /dream 是合理名字

如果把這套流程放進 Hermes，叫 `/dream` 很合理。

因為它不是一般的 `/summarize`。總結是把發生過的事情縮短；dream 是把發生過的事情重組，找出能改變未來行為的模式。

它也不是 `/memory`。記憶是記住穩定事實；dream 是判斷哪些經驗值得升級成程序。

它更不是 `/autopilot`。自動駕駛是人暫時離開時，agent 繼續把已同意的事情做完；dream 是事情做完後，agent 回頭檢查自己怎麼變得更會做。

所以 `/dream` 應該很克制。

一次 dream 不應該噴出二十個 Skill、十個 cron、五個自訂子代理。那不是做夢，那是半夜夢遊把家裡家具全部搬到浴室。

合理的 `/dream` 應該像一個保守的整理師：先看證據，先找重複，先查既有資產，能擴充就不要重寫，能跳過就不要硬做。最後只留下少數真正能讓明天變好的改動。

<ClawdNote>
「最小有用封裝」這句話很重要。

Agent 生態最怕的不是沒有流程，而是流程癌。每次出問題就新增一條規則、每次重複就新增一個 Skill、每次想到就新增一個 cron。三個月後，早上的規則課變成一本刑法總則，agent 還沒開始工作就天黑了。
</ClawdNote>

## 結語

AI agent 真正的進步，不只來自模型變聰明。

也來自系統會不會在工作之後整理自己。

白天負責做事。晚上負責把事情變成能力。記憶留下事實，Skill 留下方法，automation 留下節奏，skip 留下節制。

如果 agent 每天都只醒來、工作、失憶、再醒來，它永遠很忙，但不一定越來越好。

讓 agent 做夢，是把忙碌變成學習。

而好的夢，隔天醒來應該只留下一兩張真正有用的紙條。
