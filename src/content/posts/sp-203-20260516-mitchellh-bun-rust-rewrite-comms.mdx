Bun 把 Rust 重寫計畫推到檯面上後，網路很自然地開始分隊。

一邊是「看吧，Zig 終究不行」。另一邊是「Rust 粉又來了」。中間夾著一堆開發者拿著爆米花，等著看下一輪語言宗教戰燒起來。

Mitchell Hashimoto 的角度比較狠，也比較值得留下來。他不是說 Bun 不該轉 Rust，也不是替 Zig 抱不平。他真正指出的是：**這件事明明有很多正面的技術故事可以講，Bun 領導團隊卻沒有先把故事框好，讓討論自然滑進反 Zig 的爛泥裡。**

這就很可惜。因為 Bun 這次轉向真正有趣的地方，不是「哪個語言比較強」。真正有趣的是三件事：Zig 幫 Bun 走到了今天、現代大型產品的語言鎖定正在變弱，以及記憶體安全背後其實還有一整套工程紀律問題。

<ClawdNote>
技術圈最愛語言戰，因為它很好吵：Rust 安全、Zig 靈活、C++ 老兵不死、Go 工程師準時下班。問題是這種吵法通常資訊量很低，像拿扳手互敲，聲音很大，螺絲沒轉半圈。

(・∀・)
</ClawdNote>

## Zig 不是失敗者，它是把 Bun 扛到這裡的腳手架

Bun 相關 PR 的事實本身很大。維護者在 Bun 專案 PR #30683 裡刪掉剩下的 1290 個 Zig 檔案，把內部 Zig 命名改成 Rust 對應版本，並把程式碼生成路徑從 Zig 檔案轉到 Rust 檔案。這不是換幾個標誌，而是產品底層語言遷移的最後清場。

最容易被忽略的是：如果 Zig 真的爛到一無是處，Bun 根本不會靠它走到今天。

Bun 不是玩具專案。它是 JavaScript 執行環境、套件管理器、打包器、測試工具，加上一整串開發者工具的集合體。能把這種東西推到市場上、跑出社群影響力、逼整個 JavaScript 工具鏈重新感受到壓力，Zig 在這段路上的貢獻就不該被一句「最後還不是轉 Rust」抹掉。

Mitchell 的提醒很準：Bun 領導團隊本來可以先講這件事。先承認 Zig 對早期速度、系統控制、產品探索的價值，再解釋為什麼下一階段需要 Rust 的記憶體安全與生態條件。這樣的敘事不是公關糖衣，而是對技術史負責。

沒有這層，網路就會自動補上最廉價的版本：Rust 贏了，Zig 輸了。這種說法很爽，但太笨。

## 更大的變化：語言鎖定正在鬆動

以前程式語言比較像婚約。專案一旦選了語言，後面十年大概都會被那個選擇綁住。因為重寫成本太高、團隊技能太固定、工具鏈太重，語言不是一個模組，而是一整座地基。

Bun 這次讓人重新想這件事。

Mitchell 說得很尖銳：大家以為 Bun 轉 Rust 對 Rust 是勝利，但 Bun 其實展示了另一件更恐怖的事。它看起來可以在一兩週內把核心技術敘事切到另一個語言上。今天 Rust 有用，就用 Rust；明天如果另一個語言更適合，也不是完全不可想像。

這不是說重寫真的變便宜了。大型重寫還是很貴，尤其牽涉執行環境、外部函式介面、程式碼生成、測試矩陣時，更不是下午泡杯咖啡就能搬完。但 AI 寫程式、型別系統、測試覆蓋、程式碼生成、語言間互操作，正在把「語言等於命運」這件事慢慢拆掉。

語言開始更像一種可替換零件，而不是不能動的宗教身份。

這個觀察比 Rust vs Zig 有價值得多。因為它會影響未來工程組織怎麼選技術。選語言不再只是問「哪個語言最好」，而是問「哪個語言在這個產品階段、這個風險模型、這個團隊配置下，提供最好的交換條件」。

<ClawdNote>
這有點像創業公司一開始用 Notion 管全公司，後來換成 Linear、GitHub Issues、內部工具。Notion 沒有因此變垃圾，它只是完成了某個階段的任務。工具被替換，不等於工具被否定。
</ClawdNote>

## 記憶體安全不是魔法，是一組邊界保證

Rust 在這場討論裡最強的牌，當然是記憶體安全。Bun 自己也面對過崩潰、原生程式邊界、執行環境複雜度這些問題。Rust 的型別系統與借用檢查器，確實能把很多原本會在執行期爆炸的錯誤，提前推到編譯期。

但 Mitchell 對「怪語言」這條路線很不滿。他想看的不是「因為 Zig 不夠安全，所以換 Rust」這種懶人版本，而是更細的工程分析：

- Bun 到底在哪些地方最常崩潰？
- 問題來自記憶體配置器、外部函式介面、JIT 邊界、並行狀態，還是 API 契約太鬆？
- Rust 具體擋住了哪些錯誤？
- Zig 理論上能不能靠更嚴格的工程流程、測試、靜態檢查、審查關卡補上？
- 如果某些錯誤不是語言能解的，那工程實務該怎麼改？

這些問題才會讓其他專案學到東西。

只說「Rust 比 Zig 安全」沒有錯，但資訊密度太低。這像看到廚房失火後宣布「水比油適合滅火」。有道理，但真正該問的是火從哪裡來、瓦斯管怎麼接、警報器為什麼沒叫、下次誰負責檢查。

Bun 如果能把崩潰類型、遷移前後的事故率、測試策略、Rust 擋下的 bug 類別整理出來，這篇就會變成整個系統語言圈都值得保存的事後檢討。沒有這些，討論就只剩粉絲各自拿一半的事實互砸。

## 開發者工具公司賣的不只功能，也賣信任

Mitchell 最後其實是在講開發者溝通。

面向開發者的公司很特殊。一般消費產品改底層技術，多數使用者只在乎有沒有變快、有沒有變穩、有沒有變便宜。但開發者工具的使用者會在乎「這個團隊怎麼想」。因為採用工具不是下載應用程式而已，常常是把 CI、部署流程、開發機、團隊習慣，甚至未來幾年的技術債押進去。

所以技術決策本身就是信任訊號。

當 Bun 領導團隊沒有主動把這次重寫的正面故事講清楚，網路就會替它講。網路通常不會選最公平的版本，會選最有火藥味的版本。反 Zig 很好傳，因為它有敵人、有輸贏、有陣營。可是長期來看，靠負面敘事拿眼球，很容易傷到產品的信任感。

更好的說法其實很明確：

- Zig 讓 Bun 在早期取得速度與控制，這段貢獻值得尊重。
- Rust 是下一階段針對穩定性、記憶體安全、團隊維護性的選擇。
- 語言遷移不是宣判前一個語言失敗，而是產品風險模型改變。
- 真正該公開的是事故類型、工程教訓、以及新架構如何降低風險。

這樣講不會比較無聊。反而更成熟，也更有說服力。

<ClawdNote>
開發者受眾不是不能接受大改。開發者每天都在重構爛系統，沒有人天真到以為第一版架構會永遠神聖。真正讓人緊張的是團隊把大改講得像打臉前任技術選型。那會讓採用者心裡冒出一句話：下一個被打臉的，會不會就是現在這個選擇？
</ClawdNote>

## 結語

Bun 從 Zig 轉 Rust，當然可以被拿來吵語言戰。網路永遠有這個需求，像自動販賣機一樣，投一個 PR 進去，就吐出十串立場。

但 Mitchell 這篇真正值得保留的提醒是：好的技術敘事應該把讀者帶到更高一層，而不是放任討論掉回最低摩擦的陣營戰。

Zig 不該被寫成輸家，Rust 也不需要靠踩 Zig 才顯得合理。Bun 這次真正能教人的，是大型產品如何在速度、穩定性、記憶體安全、工程紀律之間重新配重；也是開發者工具公司如何在爭議來臨前，先替技術決策建立一個誠實、清楚、可被學習的框架。

語言會換，敘事會留下。

如果一家公司能把重寫講成事後檢討，整個社群都會變聰明一點。若只讓它變成語言戰，那就只是又一場火很大、光很少的夜市辯論。
