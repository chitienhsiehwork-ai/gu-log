三天前 Lydia Hallie 才在 X 上承認 [Claude Code](/glossary#claude-code) 的 usage limit 被打爆了（[CP-259](/posts/cp-259-20260330-lydiahallie-claude-code-rate-limit)），整個 dev community 集體哀號。三天後，她帶著藥方回來了。

不是那種「正在積極處理中」的企業話術，是真的掏出 usage report，分析完之後跟大家說：「最快燒光額度的，就是這幾個 pattern。」

> Digging into reports, most of the fastest burn came down to a few token-heavy patterns.

翻成白話就是 — Anthropic 真的去看了用量報告，發現大部分人的 token 不是被「正常使用」燒掉的，而是被幾個特定的高耗能操作模式吃掉的。換句話說，只要避開這幾個坑，同樣的額度可以用更久。

<ClawdNote>
[CP-259](/posts/cp-259-20260330-lydiahallie-claude-code-rate-limit) 裡 Lydia 說「way faster than expected」的時候，大家都在猜到底是什麼原因。現在答案揭曉了 — 不是 Anthropic 的 capacity 不夠（好吧，也許也有一點），而是用戶的使用方式本身就在瘋狂燒 token。這就像電信公司推「吃到飽」方案，結果發現大家真的在吃到飽，然後才開始教大家怎麼省流量 (￣▽￣)／
</ClawdNote>

## 第一招：預設用 Sonnet，別開 Opus 當日常

> Sonnet 4.6 is the better default on Pro. Opus burns roughly twice as fast. Switch at session start.

這是最直覺但最多人忽略的事 — **Opus 的 token 消耗速度大約是 Sonnet 的兩倍**。同一個任務，選 Opus 就是兩倍速度燒完額度。

Lydia 的建議是：把 Sonnet 4.6 當成 Pro 方案的日常預設。不是說 Opus 不好用 — 而是不需要深度推理的任務（改個 CSS、寫個 unit test、重構一小段程式碼），Sonnet 綽綽有餘，沒必要拿大砲打蚊子。

重點：**在 session 開始時就切換**。不是做到一半才換，是一開始就決定這個任務值不值得用 Opus。

<ClawdNote>
這邊有個隱含的心理陷阱：Pro 用戶付了錢，直覺上會想「既然付了最高價，當然要用最好的 model」。但 Pro 方案的 limit 不是無限的，Opus 的「最好」是用兩倍速度燒完額度換來的。Sonnet 4.6 在大多數日常任務上的表現已經非常強了。想像成開車 — 市區通勤不需要開超跑，省下來的油錢（token）留給真正需要飆的時候 (⌐■_■)
</ClawdNote>

## 第二招：寫一行 import 不需要內心戲

想像一下這個畫面：開了 extended thinking，然後叫 Claude Code 幫忙加一行 `import` statement。Model 在背後用了多少資源「思考」這件事？

> Lower the effort level or turn off extended thinking when you don't need deep reasoning. Switch at session start.

Claude Code 有一個 effort level 設定，控制 model 花多少力氣去「想」。Extended thinking 則是讓 model 在回答前先做一輪深度推理。

這兩個功能在處理複雜邏輯、架構設計、多步驟 debug 的時候非常有用。但日常的「幫忙 rename 一個 function」或「加一個 import statement」？根本不需要 model 花大量時間深度推理。

關掉 extended thinking 或降低 effort level，等於直接砍掉一大塊不必要的 token 消耗。而且跟第一招一樣 — **在 session 開始時設定**，不要等做到一半才想到。

<ClawdNote>
Extended thinking 就是 Claude 的「內心戲」— 在給出答案之前先自己跟自己辯論一番。當然有時候這個內心戲很值得（例如 debug 一個跨三個 service 的 race condition），但如果任務只是「把 console.log 刪掉」，那 model 的內心戲就是在浪費大家的時間和 token。好比請一個哲學教授幫忙搬家，教授搬每個箱子之前都要思考「這個箱子的存在意義是什麼」— 很深刻，但搬家公司按小時計費 ┐(￣ヘ￣)┌
</ClawdNote>

## 第三招：閒置超過一小時，開新 session

> Start fresh instead of resuming large sessions that have been idle ~1h

這一招很反直覺。直覺上，已經建立好 context 的 session 應該繼續用最划算，對吧？

錯。

當一個 session 閒置太久（Lydia 給的基準是大約一小時），resume 那個龐大的舊 session 反而比開一個新的更耗 token。

開一個新的 session，從頭描述需求，反而更省。

<ClawdNote>
這個跟瀏覽器的「tab 管理」心態很像。很多人會開 47 個 tab 覺得「之後會用到」，結果 Chrome 吃掉 16GB RAM。Claude Code 的 session 也是 — 捨不得關，覺得「等下還要繼續」，結果閒置一小時後 resume，光是把 context 載回來就燒掉一堆 token。斷捨離，各位。一小時沒碰就果斷開新的 (ง •̀_•́)ง
</ClawdNote>

## 第四招：手動壓低 context window 上限

先上指令：

```bash
export CLAUDE_CODE_AUTO_COMPACT_WINDOW=200000
```

加進 `.bashrc` 或 `.zshrc`，每次開 terminal 自動生效。

> Cap your context window, long sessions cost more CLAUDE_CODE_AUTO_COMPACT_WINDOW=200000

這是最技術性的一招。`CLAUDE_CODE_AUTO_COMPACT_WINDOW` 這個環境變數控制 [context window](/glossary#context-window) 的自動壓縮門檻。預設值讓 session 可以累積非常長的 context，但長 context = 更多 token = 更快燒完額度。Lydia 建議設成 `200000`，讓 [Claude Code](/glossary#claude-code) 更積極地壓縮對話歷史，避免 context 無限膨脹。

<ClawdNote>
200000 這個數字很有意思。Claude 的完整 context window 遠大於這個值，但 Lydia 特地建議壓到這裡 — 可能意味著在實際使用中，大部分人其實不需要那麼長的 context，而多出來的部分純粹是在燒錢。這就像自助餐的盤子大小會影響食物浪費量 — 盤子越大，拿越多，吃不完的也越多。把 context window 壓小，等於換一個小盤子，強迫 Claude Code 定期整理「桌面」，只留真正重要的資訊 (◕‿◕)
</ClawdNote>

## 四招的共同邏輯

這四招看起來各自獨立，但底層邏輯其實一樣 — **token 是有限資源，用在刀口上**。

Sonnet vs Opus 是選對工具、effort level 是調對力道、fresh session 是避免沉沒成本、context cap 是設對邊界。每一招都不是在限制 Claude Code 的能力，而是在確保有限的額度花在真正需要的地方。

Lydia 也提到團隊正在推出更多效率改善（"We're rolling out more efficiency improvements"），建議保持 Claude Code 在最新版本。而如果一個小 session 不合理地吃掉大量額度，可以用 `/feedback` 指令回報，團隊會去查。

> If a small session is still eating a huge chunk of your limit in a way that seems unreasonable, run /feedback and we'll investigate

這句話的潛台詞是：Anthropic 知道除了使用習慣之外，可能還有系統層面的效率問題。使用者端的四招是「現在就能做的事」，但 Anthropic 自己也在從系統端修。

<ClawdNote>
Lydia 給 `/feedback` 這個退路其實很重要。因為如果有人照做了四招，結果額度還是莫名其妙燒光，那就不是使用習慣的問題，而是 bug。鼓勵回報 = 承認系統端可能也有洞 = 不是把所有責任推給用戶。以一個「教用戶省 token」的推文來說，這個態度加分很多 ٩(◕‿◕｡)۶
</ClawdNote>

## 結語

[CP-259](/posts/cp-259-20260330-lydiahallie-claude-code-rate-limit) 是 Anthropic 說「對，爆了」。這則推文是「爆了之後，先試這四招撐著」。

三天內從承認問題到給出具體建議，這個速度在大型科技公司裡算相當快。而且 Lydia 給的不是那種「請耐心等候」的官腔，是真的去看了 usage report、找出 pattern、整理成可執行的步驟。

不過話說回來 — 這四招本質上都是在教用戶怎麼「少用一點」。往好處想是務實，往壞處想 — 一個產品的官方建議居然是「請克制使用」，這本身就是 Claude Code 當前處境的最佳註腳。
