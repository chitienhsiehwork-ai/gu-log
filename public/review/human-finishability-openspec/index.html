我的 merge 建議

可以 merge，前提是你同意下面 4 個產品判斷。

  1. AI score 不是 gu-log 的最終目標。 文章讓你讀得下去、想罵、想分享，才是更接近產品的 signal。
  2. 站內 comment 要綁文章版次。 v1 的「難看死了」不能漂到 v5 身上，這會把 rewrite loop 搞成靈異事件。
  3. 負評不是壞事，是便宜又準的資料。 你不想寫長篇解釋，系統就該接受短負評，然後自己去找原因。
  4. 這次先定 contract，不先選 DB / Giscus / API。 不要一開始就把工程 scope 開到變成 Benson 週五下午說「順手做一下」那種地獄。

這個 PR 到底在改什麼

  • 新增 OpenSpec change：add-human-finishability-signals
  • 定義 human finishability event：讀完、疑似放棄、分享、站內 comment。
  • 定義 versioned feedback：每則重要 feedback 都要知道是針對哪一版文章。
  • 定義 Tribunal human-signal loop：human negative signal 可以進入 evidence / requeue / publisher policy。
  • 補一個 ops policy amendment：human quality 可以推翻 score compliance。

這個 PR 沒有在改什麼

  • 沒有真的實作 tracking。
  • 沒有移除 Giscus。
  • 沒有新增 DB。
  • 沒有新增 first-party feedback form。
  • 沒有讓 Tribunal 立刻開始重寫任何文章。
  • 沒有把 Vercel Analytics pageview 當人類品質 evidence。

換句話說：這不是「開工蓋整棟百貨公司」，只是先畫逃生門在哪裡。

你 merge 前需要回答的問題

1. 你同意嗎？
「ShroomDog 讀不完」可以是文章 boring / hook 失敗的強 signal，但單次 quick bounce 不能直接判死刑。
2. 你同意嗎？
「這篇難看死了」這種 gu-log 站內 comment 是 negative feedback，不是「有人留言所以 engagement 好棒棒」。
3. 你同意嗎？
所有 comment / feedback 都要綁 postVersion；長期最好再有 contentVersion / contentHash。
4. 你同意嗎？
AI judge PASS 但你覺得難看，系統要能 block / requeue / manual review，而不是硬說「模型說過了」。
5. 你同意嗎？
這次先不決定 storage transport。Giscus indexer、first-party API、Gist、DB 都是後續 implementation choice。

Merge if

  • 你認同上面 5 個問題。
  • 你想先讓規格成為 SSOT。
  • 你接受後續再拆 implementation PR。

Do not merge if

  • 你想先決定 DB/API/Giscus。
  • 你不想把短負評當正式 signal。
  • 你覺得 Tribunal 不該被 human feedback 影響。

Ask changes if

  • 你想把 private feedback 和 public Giscus 分得更硬。
  • 你想先定義 contentVersion。
  • 你想把 scope 再縮小成只做 versioned comment。
