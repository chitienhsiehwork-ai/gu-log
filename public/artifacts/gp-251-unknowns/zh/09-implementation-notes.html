舊版註解不一定有 frame timestamp

計畫說的
每個註解都有 frame_ts；burn-in renderer 可以排序並把它們都放到時間軸上。

待重新評估

偏離 2/415:22

Queue 把 Date 物件來回傳成字串

計畫說的
ExportJobPayload.requestedAt: Date，傳給 worker 並與 review 的 updated_at 比較以偵測過時的匯出。
程式碼揭露的
BullMQ 把 payload JSON 序列化到 Redis，所以 worker 收到的是 ISO 字串，不是 Date。過時比較在單元測試中悄悄強制轉型通過了，但在整合測試的重新投遞 job 上失敗。
保守選擇
把 payload 型別定義為線上格式（requestedAt: string ISO-8601），在 worker 邊界的 exportConsumer.ts 解析一次。不用自訂 serializer。
待重新評估
其他 media-tasks payload 可能有一樣的潛在 bug——一個共用的 job payload zod codec 可以修掉這整類問題。

偏離 3/415:48

Zip streaming 已經存在——拿掉計畫中的 archiver 依賴

計畫說的
加 archiver 來把匯出 bundle（影片 + CSV + 縮圖）建成 zip stream 到 S3。
程式碼揭露的
packages/media/src/zipStream.ts 已經有 streaming zip 加 backpressure 處理——它驅動批次素材下載功能，從三月就上線了。
保守選擇
直接重用 zipStream.ts。少一個依賴，bundle 匯出也繼承了跟批次下載一樣測試過的 backpressure 行為。
待重新評估
zipStream 在 packages/media 但現在被兩個功能用——值得在清理時提升到 packages/shared。

偏離 4/416:20

訪客審閱者可以看 review 但不能下載素材

計畫說的
匯出權限 = review 層級 ACL：任何能打開 review 的人都能匯出。
程式碼揭露的
Workspace 層級的 guest_reviewer 角色授予 review 存取權但 can_download_assets = false——客戶用它讓外部利害關係人可以評論但不能把素材帶走。只檢查 review ACL 會讓訪客透過匯出洩漏原始影片。
保守選擇
所有格式的匯出對 guest_reviewer 回傳 403，跟 assetAccess.ts 現有的素材下載檢查一致。最嚴格的防禦解讀。
待重新評估
理論上訪客應該還是能拿到 CSV（只有註解，沒有素材）。這是產品決策，不是我的——記錄在下面。

這對第二次嘗試的改變

三行可以貼到下一份計畫或 prompt 的內容，讓下次執行不用重新發現今天的意外。

  1. 事先說明舊資料的注意事項：約 12% 的註解（migration-0087 之前）有 null frame_ts。在開始實作前決定要推算還是放 sidecar，不要在建置中途的 14:41 才決定。
