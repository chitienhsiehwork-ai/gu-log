心智模型

一張圖抵 14 個檔案：匯出請求現在的流向 vs. 之前。

在這個變更之前，「匯出 clip」意味著瀏覽器用 MediaRecorder 在客戶端拼接 frames 並上傳結果——慢、鎖住 tab、而且在 Safari 上壞掉。現在客戶端只請求匯出；一個新的 worker 從原始媒體伺服器端渲染，客戶端 poll job 直到 signed download URL 出現。

這帶來的三個不明顯行為

掃 diff 不會告訴你的部分。每一個都是刻意的——這是背後的原因。

01

匯出從原始上傳渲染，不是審閱者看的 proxy

什麼Worker 拉 media/originals/，從不用 720p review proxy。因此匯出的 clip 可能比審閱者畫註解時看到的更清晰。
為什麼剪輯師匯出 clip 是要交給客戶的；送出壓縮的 proxy 會讓他們難堪。註解座標儲存為正規化（0–1），所以可以正確重新投射到全解析度 frame。
在哪worker/export/render.ts:41 worker/export/burn_in.ts:88
02

Export job 能撐過 worker crash——透過 visibility timeout，不是重試

什麼Job row 透過設定 locked_until = now() + 10min 來認領。如果 worker 在渲染中途死掉，不會觸發重試；lock 只是過期，下一個閒置的 worker 會從頭撿起 job。
為什麼ffmpeg 渲染在中途不是 idempotent，而 retry queue 需要我們還不想要的 dead-letter 處理。Lock 過期用一個欄位就給了 at-least-once 語意。結果：crash 的 job 會顯示「處理中」最多 10 分鐘後才恢復。
在哪db/migrations/0142_export_jobs.sql worker/export/claim.ts:19
03

下載 URL 24 小時後過期——匯出檔案本身保留 7 天

什麼GET /api/exports/:id 回傳的 signed S3 URL 有效 24 小時，但底層物件要到 7 天 lifecycle rule 觸發才會刪除。重新呼叫 endpoint 會產生新的 URL，不需要重新渲染。
為什麼短 URL 限制連結被轉發到 workspace 外（匯出可能包含未發佈的素材）；較長的物件 TTL 意味著過期的 Slack 連結只需要一個 API call 就能復活，不用 90 秒重新渲染。
在哪api/exports/get.ts:57 infra/s3_lifecycle.tf:23
這依賴的現有行為

Workspace-scoped signed-URL auth。 GET /api/exports/:id 本身不做權限檢查——它重用已經守住每個 media route 的 requireWorkspaceMember middleware。如果那個 middleware 的 session 處理有變（有一張公開 ticket，BL-2214，關於 guest reviewer sessions），匯出下載也會跟著變。這個 diff 裡沒有任何東西會標記出這件事。
