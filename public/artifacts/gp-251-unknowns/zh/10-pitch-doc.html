02 賣點

審閱者在 Acme 留回饋，然後重新打字到 email、Notion、和剪輯軟體——這是流失訪談中被提及最多的缺口（41% 流失團隊）。註解匯出讓任何選擇的評論和繪圖一鍵變成 SRT、CSV、或 PDF，使用我們已經在跑的基礎設施。它已經建好、flag 包好、負載測試過 12k 註解每 session，而且一個開關就能回滾。從這裡到發佈只差下面四個簽核。

03 審閱者會問什麼

Q1這會不會讓內部評論洩漏給客戶？
不會。匯出跟審閱頁面走同一個 visibility resolver：標記為 internal 的註解會被排除，除非匯出者有 workspace:admin 並透過標示的 checkbox 明確選擇包含。有 14 個 permission-matrix 測試涵蓋。
↗ spec §4.2 — 可見性與權限
Q2超大 session（幾千個註解）會怎樣？
匯出每 500 個註解分頁並串流檔案。對 prod 最大的真實 session（一個 12,400 註解的代理商審閱）壓測 3.1 秒完成；一般 session 的 p95 是 340ms。
↗ 實作筆記，06-24 — 負載測試結果
Q3為什麼是檔案格式而不是公開 API？
SRT + CSV + PDF 涵蓋過去兩季 212 個標記為 export-request 的支援工單中的 83%。API 是正確的 phase 2——它需要 auth scoping 工作但這個不需要。檔案現在就能解放使用者，不會限制 API 設計。
↗ 指標 — 支援標籤分析，Q1–Q2
Q4這會增加我們需要維運的基礎設施嗎？
零新服務。SRT/CSV 在現有 worker pool 裡 inline 渲染（services/render-workers）；PDF 重用已經在產生週報的 report service。沒有新 queue、沒有新 datastore、沒有 schema migration。
↗ spec §5.1 — 架構
Q5匯出的客戶內容的合規故事是什麼？
每次匯出都在檔案 metadata 打上 workspace ID、請求使用者、和時間戳，並寫入 audit log（audit.annotation_export）。檔案透過到期的 signed URL（24 小時）提供，超過該時間窗口不會儲存。
↗ spec §6 — 稽核與保留

05 風險、回滾、和我需要你們的

風險與回滾

  • 回滾只要一個開關。關掉 annotation_export——沒有 schema migration，什麼都不用復原。
  • 影響範圍是有界的。匯出在共用 worker pool 跑，每 job 30 秒 timeout；一個失控的 job 不會餓死審閱播放。
  • 最壞可信故障：如果 signed-URL cleanup 延遲，會有孤立的匯出檔——但它們 24 小時後會自動過期。
  • 已知缺口：繪圖註解在 CSV 會 flatten 成 bounding boxes（完整向量只有 PDF）。產品內有文件；phase 2 項目。
