Deviation 1 of 414:41

Legacy annotations don't always have frame timestamps

What the plan said
Every annotation has a frame_ts; burn-in renderer can sort and place them all on the timeline.
What the code revealed
~12% of rows in annotations predate migration 0087 and have frame_ts = NULL — they were general review comments before per-frame annotations existed.
Conservative choice
Exclude null-timestamp annotations from the video burn-in; include them in the CSV sidecar with frame_ts left blank and a legacy_comment flag. Nothing is silently dropped from the export bundle.
Revisit
Could interpolate a timestamp from created_at offset against the review session — decide whether that's honest enough to show in the video.

Deviation 2 of 415:22

The queue round-trips Date objects as strings

What the plan said
ExportJobPayload.requestedAt: Date, passed through to the worker and compared against the review's updated_at to detect stale exports.
What the code revealed
BullMQ JSON-serializes payloads into Redis, so the worker receives an ISO string, not a Date. The staleness comparison silently coerced and passed in unit tests, but failed on a redelivered job in the integration run.
Conservative choice
Typed the payload as wire format (requestedAt: string ISO-8601), parse once at the worker boundary in exportConsumer.ts. No custom serializer.
Revisit
Other media-tasks payloads likely have the same latent bug — a shared zod codec for job payloads would fix the class of problem.

Deviation 3 of 415:48

Zip streaming already exists — dropped the planned archiver dependency

What the plan said
Add archiver to build the export bundle (video + CSV + thumbnails) as a zip stream to S3.
What the code revealed
packages/media/src/zipStream.ts already does streaming zip with backpressure handling — it powers the bulk asset download feature and has been in production since March.
Conservative choice
Reused zipStream.ts as-is. One fewer dependency, and bundle exports inherit the same tested backpressure behavior as bulk downloads.
Revisit
zipStream lives in packages/media but is now used by two features — worth promoting to packages/shared in a cleanup pass.

Deviation 4 of 416:20

Guest reviewers can view reviews but can't download assets

What the plan said
Export permission = review-level ACL: anyone who can open the review can export it.
What the code revealed
Workspace-level guest_reviewer role grants review access with can_download_assets = false — clients use it to let external stakeholders comment without taking the media offsite. A review-ACL-only check would let guests exfiltrate source video via export.
Conservative choice
Export returns 403 for guest_reviewer across all formats, mirroring the existing asset-download check in assetAccess.ts. Strictest defensible reading.
Revisit
Arguably guests should still get the CSV (annotations only, no media). That's a product call, not mine — logged below.

What this changes about attempt #2

Three lines to paste into the next plan or prompt, so the next run doesn't rediscover today's surprises.

  1. State the legacy-data caveat up front: ~12% of annotations (pre-migration-0087) have null frame_ts. Decide interpolate-vs-sidecar before implementation starts, not at 14:41 mid-build.
  2. Spec queue payloads as wire types: BullMQ round-trips through JSON, so payload interfaces should use ISO strings, never Date — and a shared zod codec for media-tasks payloads would retire the whole bug class.
  3. Inventory existing utilities before adding dependencies: zipStream.ts and useProgressChannel() both existed and both beat the plan's version. Add a “search for prior art” step to the plan template, and settle the guest-reviewer export policy before the next feature touches ACLs.
