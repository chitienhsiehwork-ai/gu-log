## Context

`TableOfContents.astro` 的桌面版在 `min-width: 1280px` 時直接成為 `position: fixed`，頁面載入當下就佔住右側視覺欄位。捲動追蹤同時會預設第一個段落標題為作用中，因此首屏看起來已進入段落導覽，和文章優先的層級衝突。

手機版目錄是文章流內的展開控制，這次不改。桌面版目錄的導引線、作用中標記、平滑捲動與段落追蹤也都保留。

## Goals / Non-Goals

**Goals:**

- 桌面版首屏不顯示目錄。
- 以文章標頭是否滑出固定導覽起點作為顯示門檻。
- 回到頂端時可逆地收起；雜湊導覽、捲動位置還原與視窗縮放後狀態正確。
- 顯示過場輕量，並尊重 `prefers-reduced-motion`。

**Non-Goals:**

- 不改手機版展開控制。
- 不改目錄內容、標題深度、作用中段落算法或平滑捲動偏移。
- 不重新設計桌面版導引線、顏色或字級。

## Decisions

### D1：門檻綁定 `.post-header` 與桌面版目錄的計算後 top

顯示條件採 `.post-header.getBoundingClientRect().bottom <= 桌面版目錄計算後的 top`。CSS 的 `top` 是唯一偏移量 SSOT，JS 不再重複硬編 `96px`。這表示文章標頭滑出目錄的起始閱讀線後，段落導覽才接手。相較單純 `scrollY > N`，這個門檻會隨根字級、文章標題、中繼資料與來源列的實際高度自然調整。

### D2：共用畫面更新排程，覆蓋捲動、位置還原、雜湊與視窗縮放

`setupScrollHighlight()` 已有被動捲動監聽與 `requestAnimationFrame` 節流。目錄顯示與作用中段落在同一輪更新計算，避免兩套捲動狀態。

同一個排程器 SHALL 由以下事件觸發：

- 初始化後的畫面更新；
- 被動 `scroll`；
- `pageshow`（涵蓋重新載入、前後頁快取與較晚發生的捲動位置還原）；
- `hashchange`（涵蓋同頁深層連結）；
- 節流的 `resize`（涵蓋跨越現行桌面版斷點與文章標頭重新排版）。

每次更新都從目前 DOM 幾何位置重新推導狀態，不保存可能漂移的捲動門檻。

### D3：用 `data-visible` 控制透明度、可見性、指標事件與位移

標記初始為隱藏，避免首次繪製閃出完整目錄。顯示時只做短距離透明度與位移過場，不改版面。隱藏狀態同步使用 `visibility: hidden`、`pointer-events: none`、`aria-hidden="true"` 與原生 `inert`；顯示時全部切回可互動狀態。`inert` 與 `visibility` 共同確保看不到的連結不會進入鍵盤依序焦點，不能只依賴 `aria-hidden`。

`prefers-reduced-motion: reduce` 下移除過場與位移。

## Risks / Trade-offs

- **[風險] JavaScript 未執行時桌面版目錄維持隱藏** → 目錄原本的作用中追蹤與平滑捲動都依賴客戶端程式；文章正文與標題錨點仍完整可讀。這次優先避免首屏閃現，之後若要支援無 JavaScript 的桌面版目錄，可另做明確的漸進增強契約。
- **[風險] 極短文章標頭讓目錄太早出現** → 門檻綁定標頭底部與目錄計算後的 top，而非固定捲動距離，且只在桌面版斷點生效。
- **[風險] 顯示動畫干擾對動態敏感的讀者** → `prefers-reduced-motion` 下完全停用過場。
